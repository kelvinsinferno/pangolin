### A1. Signing approach — GPG-signed SHA-256 manifest, NOT Authenticode

**Decision:** P12-1 ships an **unsigned** `pangolin-cli.exe`
+ `chaincli.exe` alongside a **GPG-signed** `SHA256SUMS`
manifest (the `.asc` detached signature). The user verifies
the manifest signature with `gpg --verify` against Kelvin's
published public key, then verifies each binary's SHA-256
matches the manifest entry. **No Authenticode certificate is
acquired.**

**Rationale:**

1. **Master-plan §3.7 P12-1 names Authenticode for Windows**
   (`Authenticode on Windows / GPG-signed AppImage on Linux`).
   The locked answer (Kelvin, recorded in this PLAN's spec
   — top-of-file Locked answers) **promotes the GPG-signed-
   manifest pattern from Linux to Windows** for the PoC. The
   master plan's vocabulary reads as a per-OS *minimum*; GPG
   over the manifest is the same trust anchor with one fewer
   external dependency.
2. **Authenticode certificates cost USD 200-700/year + an
   identity verification cycle.** For a PoC release that
   targets ~10 evaluating developers, the cost is
   disproportionate. Authenticode acquisition is deferred to
   MVP-1's packaging cycle (master plan MVP-1 issue 1.4-pkg
   or its successor; specific issue ID locked when MVP-1's
   issue list is opened).
3. **GPG-signed-manifest matches the existing Linux pattern.**
   The master plan §3.7 P12-1 says "GPG-signed AppImage on
   Linux"; using the same trust anchor on Windows means a
   user evaluating both Linux + Windows binaries verifies via
   the same `gpg --verify` ritual. Operational consistency
   for the user.
4. **Windows SmartScreen heat is acknowledged.** An unsigned
   `.exe` may trigger SmartScreen "unrecognized publisher"
   warnings on first run. Documented in `POC_README.md` (§A5
   antivirus-whitelist note) and in `docs/RELEASE.md`. Users
   accept the warning at their own discretion; this is a
   PoC-only friction MVP-1's Authenticode acquisition closes.

**Alternative considered:** Authenticode certificate
acquisition. **Rejected** per Kelvin direction (locked answer
top-of-file): cost, time-to-acquire, and identity
verification overhead are disproportionate for a PoC release.
The deferral is logged in the §3.9 retrospective as
"D-002-adjacent: PoC binary signing throwaway-for-PoC; MVP-1
acquires Authenticode."

**Alternative considered:** Self-signed certificate (Windows-
native, no third-party CA). **Rejected** — self-signed
certs are not chain-of-trust-verifiable and produce the
same SmartScreen warning as no signature at all. The user-
facing trust model is identical to "no signature"; the
operational cost is non-zero. GPG-signed manifest is
strictly better.

**Alternative considered:** Skip signing entirely. **Rejected**
— the §3.9 gate language ("P12 build artifact + screencast
available") is more honestly satisfied with *some*
verifiable trust anchor; GPG-signed manifest is the cheapest
honest option.

### A2. Target OS — Windows ONLY for P12; macOS/Linux/mobile deferred

**Decision:** P12-1 produces ONLY a Windows-x64 release
binary. macOS desktop and Linux desktop are deferred to MVP-1's
packaging cycle. Android and iOS mobile are deferred to MVP-3
(per the master plan's mobile shell phase).

**Rationale:**

1. **Kelvin's development host is Windows-x64** (D-013).
   Producing a cross-compiled macOS / Linux binary requires
   either (a) macOS / Linux build hosts (Kelvin doesn't
   have one set up for the PoC release), (b) Docker-based
   cross-compilation (introduces a build-environment
   dependency MVP-1 should own end-to-end), or (c) GitHub
   Actions cross-build (requires the GPG signing key to
   be uploadable as a CI secret, which extends the trust
   surface).
2. **The §3.9 gate language doesn't require multi-OS.** "P12
   build artifact" is singular; one OS satisfies the gate. The
   non-author rehearsal in P11 was performed on a single host
   (Kelvin's, in Mock mode); the gate doesn't promise any
   particular target.
3. **Mobile is a different shell layer entirely** (UniFFI
   bindings + iOS/Android packaging). The master plan §3.7
   P12 entry doesn't mention mobile at all; mobile lives in
   MVP-3 per master plan §6.

**Alternative considered:** Cross-compile a Linux x86_64
AppImage in addition to the Windows binary. **Rejected** for
the same reason as the macOS option above — adds build-
environment complexity (cross-compilation toolchains; AppImage
packager) and an alloy-rs cross-compile compatibility
verification cycle that's out of scope.

**Alternative considered:** Ship only the source repository
and direct users to `cargo build --release`. **Rejected** —
the master-plan §3.7 P12-1 names "binary" explicitly. Source-
only handoff would also fail the §3.9 gate language ("build
artifact").

### A3. Versioning — `v0.0.0-poc` for the PoC release; MVP-1 starts the 0.1.x line

**Decision:** P12 ships under the version tag `v0.0.0-poc`
(matching the existing workspace-root `Cargo.toml`'s
`workspace.package.version = "0.0.0"` per the workspace-
relative pin). The release zip is named
`pangolin-poc-v0.0.0-windows-x64.zip`; the GitHub Release tag
is `v0.0.0-poc`; the `docs/RELEASE.md` runbook documents the
versioning convention. MVP-1's issue 0.x cycle starts the
0.1.x line.

**Rationale:**

1. **`workspace.package.version` is already `0.0.0`** in
   `Cargo.toml`. Bumping to `0.1.0` for the PoC release
   would imply a "polished" release; the `0.0.0-poc` tag is
   honestly self-describing.
2. **Semver continuity.** MVP-1's first stable release is
   the natural `0.1.0`; the PoC's `0.0.0-poc` cleanly
   precedes it without colliding.
3. **GitHub Release tags need a unique value.** The `-poc`
   suffix lets MVP-1 tag a `v0.0.1-mvp1-prerelease` or any
   other tag without clashing with this one.

**Alternative considered:** `v1.0.0-poc`. **Rejected** —
overstates the maturity. PoC is pre-1.0 by every measure.

**Alternative considered:** No tag, just a release zip
attached to a GitHub Release named "PoC handoff."
**Rejected** — without a tag, future references ("the binary
shipped with v0.0.0-poc") have no anchor.

### A4. POC_README structure — Download section BEFORE Build

**Decision:** The polished `POC_README.md` (P12-2) places the
new "**Download a prebuilt binary**" section IMMEDIATELY
AFTER the "Status" callout (top of file) and BEFORE the
existing "Build" section. The "Build" section is moved to
"Build from source (alternative)" to signal that downloading
the prebuilt is the recommended path.

**Rationale:**

1. **Distribution-audience reading order.** A non-author dev
   reading the README has just downloaded the zip from a
   GitHub Release; the README's first actionable section
   should be "verify what you downloaded," not "build from
   source."

### A8. Screencast-script "no test password visible on screen" review

**Decision:** The `docs/SCREENCAST_SCRIPT.md` includes an
explicit pre-recording review step: before Kelvin records,
the script's beats are walked to confirm no beat's "what
the camera frames" line includes the literal vault password
`pangolin-poc-test-vault-do-not-reuse` (the P11 §A8 sentinel)
visible on screen. The script's recommended beats use
`echo '<password>' | pangolin-cli vault create
--password-stdin` such that the password is piped in but
the terminal does NOT echo it. If the user prefers
interactive entry, the recording protocol notes that
the terminal must be configured to NOT echo password
characters (default `getpass()` behavior is suppressed
characters; verify on Kelvin's host before recording).

**Rationale:**

1. **§A8's sentinel password is intentional public-record
   content** ("do not reuse" is part of the value); a
   viewer who freezes the video can see it. That's
   acceptable from a security standpoint (the password is
   non-secret), but it's a friction marker — the screencast
   is meant to set "this is how Pangolin works" expectations,
   and a visible test password could mis-train a viewer.
2. **Defense in depth.** Even though the test password is
   non-secret, the screencast IS the user's introduction to
   Pangolin's discipline; "no plaintext on screen" is a
   small but consistent trust signal.
3. **Cardinal Principle 2** ("no plaintext on disk; no
   plaintext leaves the device") — extending it to "no
   plaintext visible to the audience" is a graceful PoC
   handshake.

**Alternative considered:** Show the test password on
screen (it's non-secret; the audience benefits from
seeing it for memorability). **Rejected** — see rationale 1.
The script's narration can mention "type your vault password

### A13. §3.9 criterion verdict template

**Decision:** Each of the five §3.9 criteria gets a one-
paragraph verdict in the retrospective, with the format:

```
**§3.9 criterion N — <criterion text>:** VERDICT.
Evidence: <pointer to commit, DEVLOG entry, or repository
file>. Outstanding: <NONE | one-sentence description>.
```

VERDICT values:
- **CLOSED** — criterion fully met; evidence is in-tree.
- **OPEN-WITH-EVIDENCE** — criterion met but evidence is
  manual / outside-the-tree (e.g., "the screencast was
  recorded; the video is at <YouTube URL>"); reviewer
  takes the attestation on Kelvin's word.
- **NEEDS-FOLLOWUP** — criterion NOT met at P12 SIGNOFF;
  the retrospective names the follow-up item that
  closes it.

**Rationale:**

1. **Three verdict states cover all reasonable cases** —
   in-tree evidence (CLOSED), out-of-tree attestation
   (OPEN-WITH-EVIDENCE), or genuinely unmet (NEEDS-FOLLOWUP).
2. **NEEDS-FOLLOWUP is the gate-failure marker.** If any
   criterion is NEEDS-FOLLOWUP, the §3.9 gate is NOT
   closed, and the master plan §3.9's "If any item fails:
   stop, fix the PoC, do not start MVP work" applies.
3. **§3.9 criterion 4 ("contract redeployed at least once")
   is the most likely NEEDS-FOLLOWUP at P12 SIGNOFF tip**
   — see §8 Q1 below. The verdict template forces the
   BUILD agent to confront this rather than glossing.

### A15. Release-script pre-flight smoke

**Decision:** `scripts/release-windows.ps1` includes a
pre-flight smoke step BEFORE the release build:

1. `cargo fmt --all -- --check` — fail-fast on format drift.
2. `cargo clippy --workspace --all-targets -- -D warnings`
   — fail-fast on lint.
3. `cargo test --workspace --lib` — fail-fast on test
   regression.

If any of these fails, the script aborts BEFORE producing
the release zip. The user sees the failure and either fixes
the working tree or re-runs the script.

**Rationale:**

1. **Releases should not ship from a red tree.** Even
   manual releases benefit from a pre-flight gate.
2. **The pre-flight is fast (~30s for the test suite).**
   Negligible compared to the 5-minute release build it
   gates.
3. **The pre-flight reuses the §A10 baseline.** No new test
   code; just runs the existing 401-test suite against the
   working tree.

**Alternative considered:** Skip the pre-flight; trust the
operator. **Rejected** — operator-trust is a single human
attention slip away from a broken release.

**Alternative considered:** Move pre-flight to a separate
`scripts/release-preflight.ps1`. **Rejected** — separating
adds a step the operator might forget; bundling makes the
release script a single entry point.

## Threat model row

P12 likely needs **no new row** in `THREAT_MODEL.md`.
Verification walk:

- **The release script is build-tooling, not attack surface.**
  A malicious actor with write-access to the script could
  inject a backdoor into the binary; that's the threat any
  release pipeline carries. Defence: the script is
  committed; CI's existing `cargo deny` / `cargo audit`
  passes catch dependency-tree changes; the GPG signature
  is over the SHA-256 manifest, so any post-build
  tampering invalidates the signature. The threat is
  inherent to "user trusts a binary" — not new from P12.
- **GPG key compromise** is a separate threat, not
  introduced by P12. Kelvin's signing-key handling is
  documented in `docs/RELEASE.md` (§A1) but the key's
  custody discipline is operational, not architectural.
- **Antivirus / SmartScreen warnings** are a usability
  issue, not a security risk. A user who clicks "Run
  anyway" on an unsigned binary is making the same trust
  decision a Linux user makes when running `chmod +x` on a
  downloaded file. Documented in §A5; not a threat-model
  surface.
- **YouTube unlisted URL leak** (§A7) — the screencast is
  unlisted, not private; anyone with the URL can watch.
  The screencast contains no secrets (per §A8 review); URL
  leak is a discovery-rate issue, not a security issue.

**If BUILD's review surfaces a new user-facing risk** (e.g.,
"the release script writes the GPG private key to disk in
plaintext during the signing step" — which it MUST NOT;
the GPG key is held by the OS keyring + accessed by `gpg
--detach-sign`), P12-5 appends THREAT_MODEL row #29
following P11A's §A14 placeholder-numbering discipline. The
existing rows #1-#28 are not relitigated.

**Recommend NO new row** as the default position; flag at
SIGNOFF if the BUILD walk disagrees.
