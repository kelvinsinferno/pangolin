## Goal

After this issue closes, `pangolin-kdbx` is a real read-only KeePass 2.x
parser: given a `.kdbx` path + master password (+ optional keyfile per Q-b), it
decrypts the database (KDBX 3.1 AES-KDF or KDBX 4.x Argon2 / AES-or-ChaCha20 /
HMAC-SHA256 block-MAC), parses the gzip'd XML payload, walks the entry tree
(skipping the recycle bin), and yields a `Vec<KdbxEntryDraft>` — each carrying
the KeePassXC Title->display_name, UserName->a username, URL->a url,
Password->password, Notes->notes, the TOTP `otp`/`TimeOtp-*` field parsed via
`pangolin-totp` into `totp_secret`+`totp_params`, and the group path +
KeePassXC tags merged into `tags`. The 1.1-frozen `kdbx_import` FFI entry is
implemented: it opens the unlocked vault, parses the file, calls
`Vault::account_add` per entry inside the vault's transactional discipline, and
returns a `KdbxImportReport` with imported / skipped / failed counts and the
non-secret `failure_kinds`. A round-trip test (build a `.kdbx` -> import ->
assert every field) plus a 500-entry fixture import prove correctness and
scale. `pangolin-crypto`'s zero-serde HIGH-1 invariant and the uniffi-isolation
invariants are unaffected (`pangolin-kdbx` is downstream of `pangolin-ffi`).
`forbid(unsafe_code)` retained on `pangolin-kdbx`. PoC `.pvf` vaults still open
/ unlock / `account_history`; importing into a PoC vault works.

## Approach

### 1. The `pangolin-kdbx` crate (`crates/pangolin-kdbx/src/`, replacing the scaffold)

Module layout:

- `header.rs` — KDBX magic (`0x9AA2D903 0xB54BFB67`) + version dispatch (3.1 vs
  4.x); the outer-header TLV fields (cipher UUID, compression flag, master
  seed, KDBX3 transform seed/rounds, KDBX4 KDF VariantDict, encryption IV,
  KDBX3 stream-start-bytes, inner-random-stream id+key).
- `kdf.rs` — AES-KDF (KDBX3.1: AES-256-ECB transform `rounds` times over the
  32-byte composite key, then SHA-256) + Argon2d/Argon2id (KDBX4, via the
  already-vendored `argon2` crate, params from the VariantDict).
- `crypto.rs` — composite-key derivation (SHA-256 of password-SHA256 [||
  keyfile-hash]), master-key derivation, KDBX4 HMAC-SHA256 block-MAC verify +
  HMAC block-key derivation, outer-cipher decrypt (AES-256-CBC via `cbc`+`aes`,
  or ChaCha20 via `chacha20`), the KDBX3 inner-payload start-bytes check.
- `payload.rs` — gzip-inflate (`flate2`/`miniz_oxide`) the decrypted inner
  stream -> XML bytes; KDBX4 inner-header TLVs (inner-random-stream id+key,
  binaries); the inner-random-stream (Salsa20 with the fixed IV
  `0xE830094B97205D2A` for KDBX3.1, ChaCha20 for KDBX4) that XOR-protects the
  `Protected="True"` string values (Password, otp, ...).
- `xml.rs` — walk `<KeePassFile><Root><Group>...<Entry>...`; per Entry collect
  `<String>` Key/Value pairs (Title, UserName, Password, URL, Notes, + custom
  keys incl. `otp` / `TimeOtp-*`), `<Times>` (Expires, ExpiryTime), `<Tags>`,
  `<History><Entry>...` (the per-entry version history), the group `<Name>`
  chain and the recycle-bin UUID (from `<Meta><RecycleBinUUID>`). Base64-decode
  + un-XOR Protected values.
- `map.rs` — `KdbxEntry` -> `AccountIdentityDraft` per the table below; the
  `otp`/`TimeOtp-*` normalisation -> `pangolin_totp::parse_*`.
- `import.rs` — the public `import_kdbx` orchestrator.
- `tests/fixtures/` — tiny committed `.kdbx` files (KDBX3.1, KDBX4-AES,
  KDBX4-ChaCha20, one with TOTP, one with a recycle bin, one keyfile-protected
  if Q-b = support); `tests/roundtrip.rs`; `tests/scale.rs` (500-entry).

**Public API of `pangolin-kdbx`:**

- `pub struct KdbxCredentials { pub password: Zeroizing<Vec<u8>>, pub keyfile:
  Option<Zeroizing<Vec<u8>>> }` (keyfile only if Q-b = support).
- `pub struct KdbxEntryDraft` (secret-bearing; redacting `Debug`):
  `display_name: String`, `usernames: Vec<String>`, `urls: Vec<String>`,
  `notes: String`, `password: Zeroizing<Vec<u8>>`, `totp:
  Option<pangolin_totp::ParsedTotpSecret>`, `tags: Vec<String>` (KeePassXC
  `<Tags>` union group-path components), `expired: bool`, `history_passwords:
  Vec<(Vec<u8>, i64)>` (per Q-d).
- `pub struct KdbxImportSummary { entries_found: u32, drafts: Vec<KdbxEntryDraft>,
  skipped_recycle_bin: u32, skipped_expired: u32, entry_failures: Vec<(u32,
  String)> }`.
- `pub fn import_kdbx(path: &Path, creds: &KdbxCredentials) ->
  Result<KdbxImportSummary, KdbxError>` — parses + decrypts; does NOT touch a
  vault (so `pangolin-kdbx` has no `pangolin-store` dep and `roundtrip.rs`
  needs no SQLite); returns `WrongCredentials` for a bad password OR a
  bad/missing keyfile (no oracle).

**Why `import_kdbx` returns drafts rather than a `Vault::import_kdbx` doing it
all:** keeps `pangolin-kdbx` free of any `pangolin-store` dependency (L2 — the
dep arrow stays `pangolin-ffi -> pangolin-kdbx`, never into the store) and
keeps the parser independently testable. The store-side ingestion is a thin
loop in `pangolin-ffi::kdbx_import` (it has the `Arc<VaultHandle>` and the
`Vault` lock; mapping `KdbxEntryDraft -> AccountIdentityDraft` is ~20 lines).

### 2. Field-mapping table (KeePassXC -> `AccountIdentity`)

| KeePassXC source | -> `AccountIdentityDraft` slot | Edge-case handling |
|---|---|---|
| `Title` string | `display_name` | **Empty `Title`** -> synthesise `"(untitled - <UserName>)"` / `"(untitled KeePass entry)"` (the validator rejects empty/whitespace). Truncate to 256 chars with an ellipsis; per-entry note if truncated. NFC by the validator. |
| `UserName` string | one entry in `usernames` | **Empty `UserName`** -> `usernames` would be empty but `AccountIdentity` requires >=1 -> inject a single `"(no username)"` placeholder (per Q-g; alt: fail the entry). Truncate to 320 chars. |
| `URL` string | one entry in `urls` | **Empty/whitespace `URL`** -> omit (empty `urls` is valid). **Unparseable by the `url` crate** -> try prepending `https://` if host-ish; if it still won't parse, drop the URL + per-entry note (don't fail the entry). KeePass `{S:Field}` / `cmd://` placeholders -> keep verbatim if the `url` crate accepts the scheme (1.2 Q3 = any scheme), else drop+note. Cap 32 URLs, each <=2048. |
| `Password` string (Protected) | `password` (`SecretBytes`) | **Empty `Password`** -> the validator rejects empty -> per Q-g recommend **skip the entry** (counted, non-secret reason — an entry with no password isn't a credential). Cap 4096 bytes. |
| `Notes` string | `notes` | Cap 65536 chars. May be augmented with dropped-data summaries per Q-c (attachments) / Q-g (custom fields). |
| `otp` field (an `otpauth://totp/...` URI) **or** `TimeOtp-Secret-Base32` (+ `TimeOtp-Period`/`TimeOtp-Digits`/`TimeOtp-Algorithm`) | `totp_secret` + `totp_params` | Normalise the `TimeOtp-*` form into an `otpauth://`-equivalent (or feed the base32 to `pangolin_totp::parse_totp_secret` with the parsed period/digits/algorithm overriding defaults); feed the `otp` URI to `pangolin_totp::parse_otpauth_uri`. On a `TotpError` -> drop the TOTP + per-entry note (don't fail the entry). `secret_bytes` <=256. `hotp://` -> drop+note (no HOTP). |
| KeePassXC `<Tags>` (`;`/`,`-separated) **+** the entry's group path (`Root/Work/Banking` -> `["Work","Banking"]`, minus `Root` and the recycle bin) | `tags` | Merge, dedup case-insensitively (validator does this), cap 32 tags, each <=64 (truncate longer names + note). Whether the group path goes into `tags` -> see Q-f; recommend **yes** (closest analogue to KeePass folders Pangolin's flat-with-tags model has). |
| `<Times>` `Expires` + past `ExpiryTime` | `expired: bool` -> an `"expired"` tag | **Expired entries**: import them anyway (data preservation) + an `"expired"` tag so the user can filter; do NOT skip. (Q-f minor; alt: skip+count.) |
| Recycle bin (`<Meta><RecycleBinUUID>` + `RecycleBinEnabled`) | — | **Skip the recycle bin entirely** (recursively, incl. subgroups). Counted in `skipped_recycle_bin`. Entries the user already deleted in KeePass shouldn't reappear. |
| Duplicate entries / KeePass clones | — | **Import all of them** — Pangolin has no natural dedup key; clones can be intentional; silent merge risks data loss. (Q-f minor; alt: dedup by `(Title,UserName)`.) |
| `<History>` (per-entry version list, oldest->newest copies with their own `LastModificationTime`) | `history_passwords` -> prepended to the `AccountIdentity` password history | **Per Q-d.** Recommend: import the *distinct* historical `Password` values (oldest first) as Pangolin password-history entries, each timestamped from the KeePass history entry's `LastModificationTime`, originating-device = a synthetic `"imported-from-kdbx"` `DeviceId`. Implemented as `account_add` + N `account_update` calls per entry (no API change; each historical password = a real revision; cap N at ~64). Alt (Q-d=drop): just the current Password. |
| Binary attachments (`<Entry><Binary Ref="N">` + the `<Meta><Binaries>` / KDBX4 inner-header binary pool) | — | **Per Q-c.** Pangolin's 1.2 model has **no attachment slot**. Recommend: **drop, but record a redacted per-entry note** in `notes` like `"[Pangolin import: 2 attachment(s), 41 KiB total, not imported]"` (sizes only — filenames can be sensitive). Never stuff attachment *bytes* into notes. |
| Custom/extra string fields (any `<String>` Key not in {Title,UserName,Password,URL,Notes,otp,TimeOtp-*}) | — | **Per Q-g.** Recommend: **append to `notes`** as a `"--- Imported custom fields ---\n<Key>: <Value>\n..."` block (Protected custom values are secret — but `notes` is itself a recovery-class secret field, so consistent); cap the block + truncate to the 64 KiB notes limit (mark if truncated). Dropping silently loses deliberate data. |
| KDBX 3.1 vs KDBX 4.x | (transparent) | The header version word picks the path: 3.1 = AES-KDF + Salsa20 inner stream + the stream-start-bytes integrity check; 4.x = Argon2 KDF (VariantDict params) + AES-256-CBC *or* ChaCha20 outer + HMAC-SHA256 block-MAC + ChaCha20 inner stream + the inner header. Both decode to the same XML walk. KDBX 1.x/2.x -> `UnsupportedVersion`. |
| Keyfile / hardware-key protected | — | **Per Q-b.** Recommend MVP-1: **support a keyfile** (one extra hash into the composite key — raw-32/hex/XML/arbitrary-file-hash formats; ~30 lines) since "my KeePass DB has a keyfile" is common; **defer YubiKey challenge-response** (HMAC-SHA1 CR with the device — needs hardware I/O the offline CLI can't do; rare) with a clean error. If Q-b = password-only, both -> an error directing the user to remove the keyfile in KeePassXC and re-export. |

### 3. The `kdbx_import` FFI body (`crates/pangolin-ffi/src/kdbx.rs`)

Replace the `todo!()`:

1. `let mut guard = handle.lock_vault(); let vault = guard.as_mut().ok_or(FfiError::NotUnlocked)?;` — locked vault -> `NotUnlocked` (the existing `as_mut()?` pattern; importing into a locked vault makes no sense).
2. Build `KdbxCredentials { password: Zeroizing(kdbx_password.expose().to_vec()), keyfile: <None, or Some(read keyfile_path) if Q-b adds the arg> }`.
3. `let summary = pangolin_kdbx::import_kdbx(Path::new(&path), &creds).map_err(map_kdbx_err)?;` — a parse-level failure (bad file / wrong password / corrupt header) -> the mapped `FfiError`; this fails the *whole* import (nothing written yet).
4. Loop over `summary.drafts`: for each, map `KdbxEntryDraft -> AccountIdentityDraft` (synthesise placeholders for empty Title/UserName per the table; `Zeroizing<Vec<u8>>` -> `SecretBytes`; flatten `ParsedTotpSecret` -> `totp_secret` + `totp_params`); call `vault.account_add(draft)`; if Q-d = import history, follow with the `account_update`s for the historical passwords. `Ok` -> `imported += 1`. A per-entry validation `Err` -> `failed += 1; failure_kinds.push(<kind label>)` (a category like `"password_empty"`, `"url_invalid"`, `"display_name"` — never the entry's actual data). Recycle-bin / skipped-expired -> `skipped`.
   - **Atomicity:** recommend **per-entry-committed** — a 500-entry import that dies on entry 499 keeps the 498 that worked (the report says what failed); each `account_add` already commits transactionally. (One giant transaction would roll back the lot on one bad entry — worse for a migration tool. Q-f minor; recommend per-entry.)
5. Return `KdbxImportReport { schema_version: <current FFI schema const>, imported, skipped, failed, failure_kinds }`.
6. Add the `KdbxError -> FfiError` arm to `crates/pangolin-ffi/tests/error_taxonomy.rs` (exhaustive-match).

`map_kdbx_err`: `WrongCredentials` -> `FfiError::Validation { kind: "kdbx_credentials" }` (UI: "Wrong password or keyfile for the KeePass database"); `NotKdbx`/`UnsupportedVersion`/`Corrupt*`/`BlockHmacMismatch`/`Unsupported{Kdf,Cipher}`/`XmlMalformed` -> `FfiError::Validation { kind: "kdbx_format" }` (with a `failure_kinds`-style detail string); `Io` -> the existing IO-mapping; `TooManyEntries` -> `FfiError::Validation { kind: "kdbx_too_large" }`. No variant ever carries the master password or entry data.

### 8. Failure modes considered

- **Decryption / parser bug in `pangolin-kdbx` -> wrong bytes, panic, or worse.** Mitigation: `forbid(unsafe_code)` (no UB even on a hostile file); per-entry failures are non-fatal (`entry_failures` + counted, not a panic); the corrupt-input fuzz-smoke (test #6); blast-contained in the leaf crate (a bug here can't reach `pangolin-crypto`/`pangolin-core`); the round-trip fixtures (tests #1-4) catch silent mis-mapping.
- **A hand-rolled XML walk mishandles entities / nesting / `Protected` un-XOR ordering.** Mitigation: use a real XML parser crate (`quick-xml`/`roxmltree` — near-zero-cost addition); the round-trip + TOTP + recycle-bin fixtures exercise the walk; the inner-random-stream XOR is order-sensitive (a keystream consumed in document order over the `Protected` values) — test #1 with multiple Protected fields per entry, in order.
- **Imported credential lands in cleartext on disk.** Mitigation: every secret routes through `Vault::account_add` (same AEAD seal); `no_plaintext_on_disk` extended corpus (test #8); `Zeroizing` everywhere in `pangolin-kdbx`; redacting `Debug`.
- **A secret leaks into `KdbxImportReport.failure_kinds` / a log line / the CLI summary.** Mitigation: `failure_kinds` is a fixed set of category labels — never entry data; a test asserts the label set is closed; the CLI summary prints only the three counts + the labels.
- **Wrong-password timing / error oracle.** Mitigation: wrong-password and wrong-keyfile collapse to one `WrongCredentials` variant (test #5); the KDF (Argon2 / AES-KDF) runs before the failure is known, so timing is dominated by the (intentionally slow) KDF — no useful oracle.
- **A hostile `.kdbx` with a billion entries / a 10 GiB inner stream / an Argon2 dict demanding 64 GiB -> OOM / DoS.** Mitigation: `KDBX_MAX_ENTRIES` ceiling -> `TooManyEntries`; cap the inflated-payload size (refuse to inflate past ~256 MiB -> `CorruptPayload`); sanity-clamp the KDBX4 Argon2 params (reject `memory > 1 GiB` / `iterations > 100` -> `UnsupportedKdf` — KeePassXC's own defaults are ~64 MiB / a few iterations); stream the gzip inflate with a bounded buffer.
- **HIGH-1 / uniffi-isolation regression** — accidentally making `pangolin-store`/`pangolin-core` depend on `pangolin-kdbx`. Mitigation: the dep arrow is one-way by design (the store-side loop takes already-mapped `AccountIdentityDraft`s, not `pangolin-kdbx` types — or lives in `pangolin-ffi`); the `cargo tree` CI greps catch it.
- **New dep advisory / supply-chain.** Mitigation: `=`-pins + `Cargo.lock` commit; `cargo audit` scans (with the `--ignore` escape hatch per env-quirk #3); the chosen parser dep (Q-a) is picked for maintenance + RUSTSEC-clean + `unsafe`-free; the crypto closure is RustCrypto, already vendored + audited.
- **PoC `.pvf` vault import.** Mitigation: importing into a PoC vault just calls `account_add` (produces V1/V2-body revisions regardless of the vault's other rows); existing PoC entries untouched; tested in #11.
- **Per-entry `<History>` import explodes the revision count** (an entry with 200 password changes -> 200 revisions). Mitigation: cap the imported history per entry (~64); the 500-entry scale fixture uses <=1 history each so the test stays fast.

## Resolved decisions (Kelvin's rulings, 2026-05-12)

All seven open questions were put to Kelvin at the plan gate; he ruled on every
one (each = the plan's recommended option) and signed off on 1.9 as a
security-critical issue. The rulings are baked into the "Decisions locked" table
above (rows L11-L18); this section records the decisions in past tense **with the
rationale preserved** so the reasoning isn't lost. (The "Approach" body below may
still read in the original "recommend / alt" voice — the table is authoritative;
treat the body as colour.)

### Q-a — KDBX parser dependency → **HAND-ROLL** (see L12)

Decided: hand-roll the KDBX 3.1 + 4.x parser in `pangolin-kdbx`, reusing the
already-vendored RustCrypto primitives (`aes`, `chacha20`/`salsa20`, `argon2`,
`cipher`, `cbc`, `hmac`, `sha2`, `byteorder`, `base64`, `hex`) + one XML parser
(`quick-xml` *or* `roxmltree`, builder's pick, `=`-pinned) + `flate2`/`miniz_oxide`
for gzip. ~500-800 LoC. **Not** the `keepass` crate, not a fork.

Rationale (Kelvin's, on agreeing): the dangerous part — the actual KDF/cipher
crypto — isn't ours either way; we call the same audited RustCrypto crates
`pangolin-crypto` already vendors. The hand-rolled part is just the container
format (headers, VariantDict, block-MAC framing, inner-random-stream un-masking,
gzip'd XML), and a parser is an attack surface regardless of authorship —
isolating *our* parser in a `forbid(unsafe_code)` leaf crate we can fuzz beats an
opaque external one with its own transitive surface. Accepted tradeoff: more of
our own format code to get right (caught by round-trip + fixture tests + the
adversarial audit) and a longer build (~16-22 h). This makes the crate fully
`forbid(unsafe_code)`, gives the tightest dep tree, and puts the parser in scope
for fuzzing + the §16 adversarial audit.

### Q-b — Keyfile / hardware-key DBs → **PASSWORD + KEYFILE, DEFER HARDWARE CR** (see L13)

Decided: support master-password + optional keyfile (raw-32 / hex / XML /
arbitrary-file-hash, ~30 lines, common in KeePassXC); the FFI `kdbx_import`
signature grows `keyfile_path: Option<String>` (additive). Defer YubiKey /
hardware HMAC-SHA1 challenge-response — it needs token I/O an offline CLI can't
do. A hardware-CR-protected DB → a clear **"unsupported credential type"** error
(distinct from wrong-password; not a password-guessing oracle — it's "this DB
needs a hardware key we don't support", which is fine to say).

Rationale: Kelvin explicitly checked whether hardware CR seeds Pangolin's future
hardware/NFC-unlock plan — it does **not**. That plan is a separate MVP-3/4
subsystem riding FIDO2/WebAuthn/platform attestation, not legacy YubiKey
HMAC-SHA1 CR; none of the KDBX CR plumbing would be reused. So skipping it
incurs no architectural debt; revisit in its own future issue only if real users
ask.

### Q-c — Binary attachments → **DROP + SIZE-ONLY REDACTED NOTE** (see L14)

Decided: drop attachments; append a redacted note to `notes` like
`"[N attachment(s) not imported, total X KB]"`. Filenames may be included if the
builder judges them non-sensitive, but **no attachment bytes** anywhere.
Rationale: Pangolin's 1.2 model has no attachment slot (adding one is an MVP-3+
schema bump); the note tells the user something was left behind.

### Q-d — Per-entry `<History>` → **IMPORT AS PANGOLIN PASSWORD-HISTORY** (see L15)

Decided: `account_add` the current entry state, then replay the distinct
historical `Password` values (oldest→newest, dedup consecutive equals, cap ~64)
as `account_update` calls so they land in Pangolin's password-history slot; each
historical revision timestamped from the KeePass history entry's
`LastModificationTime`, originating-device a synthetic `"imported-from-kdbx"`
`DeviceId`. Not "drop", not "import the full historical entries". No FFI/API
change. Rationale: preserves the rotation trail using the lineage Pangolin
already has; the ~64 cap bounds revision blow-up on a pathological entry.

### Q-e — `pangolin-cli import <file.kdbx>` in 1.9 → **YES, MINIMAL SUBCOMMAND** (see L16)

Decided: ship a minimal `import` subcommand in 1.9 (~30 lines; `import` is
already named in `cli.rs`): prompt for the KDBX password (and accept optional
`--keyfile <path>`) on **stderr**, print the import counts (added / per-entry
failures by category) to **stdout**, never echo secrets, exit code reflects "any
failures?". This is a **deliberate exception** to the 1.2-1.8 pattern of
deferring CLI subcommands. Rationale: the MVP-1 exit criterion ("import a
500-entry KDBX") is a user flow and the CLI is MVP-1's only shell — it must be
demoable there; the subcommand is the natural completion of an already-named
intent (like 1.8's `--generate-password` rewire).

### Q-f — Minor mapping calls → **ALL THREE RECOMMENDATIONS** (see L17)

Decided: group path → `tags` (the only place Pangolin's flat model preserves
KeePass folders); expired entries → import + add an `"expired"` tag (data
preservation; user can filter) — do not skip; duplicate entries / KeePass clones
→ import all, no dedup (Pangolin has no natural dedup key; clones can be
intentional; silent merge risks data loss).

### Q-g — Custom / empty fields → **ALL FOUR RECOMMENDATIONS** (see L18)

Decided: custom `<String>` keys (any Key not in {Title,UserName,Password,URL,
Notes,otp,TimeOtp-*}) → appended to `notes` as a labelled block, truncated to the
64 KiB notes cap with a "truncated" marker; empty `Title` → synthesise a name
(from UserName / URL / host, or `"Imported entry"`); empty `UserName` → inject a
`"(no username)"` placeholder (the model requires ≥1 username); empty `Password`
→ skip the entry, counted as a per-entry failure with a non-secret reason (an
entry with no password isn't a credential).
