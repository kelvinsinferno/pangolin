# Issue P2: pangolin-store — encrypted local vault store

> **Status:** PLAN-gate, awaiting Kelvin approval per §16.3.
> **Security-critical:** BORDERLINE. No new cryptography (consumes `pangolin-crypto`'s primitives). However, this is the layer where data hits disk, so the cardinal "no plaintext on disk" invariant lives here. Plan should be reviewed before code lands.

## Spec reference

- **Master plan §0 cardinal principle 2:** *"No plaintext leaves the device. Ever. For any reason. Verified at code-review boundary on every PR that touches a network or IPC interface."* — and by extension, no plaintext written to disk.
- **Whitepaper §C1 (Encrypted Local Vault Store):**
  > "Managed entirely by Rust. Likely SQLite + encrypted blobs. Stores: encrypted vault data, revision cache, device metadata, guardian metadata, local sync checkpoints. Properties: Offline-first, fast search, corruption-resistant, no plaintext secrets at rest."
- **Master plan §3.7 EPIC: Local Store (P2):**
  > "P2-1 SQLite schema: vaults, account_identities, revisions, devices, sessions
  > P2-2 Encrypted blob storage for account payloads (encrypted by VDK, never plaintext on disk)
  > P2-3 `Vault::create` / `open` / `unlock` / `lock` / `close` API surface
  > P2-4 Corruption-safe writes (WAL mode, atomic rename for blob updates)"
- **Whitepaper §F (Identity & Key Hierarchy):** the consumer of `pangolin-crypto`'s `AuthorityKey`, `WrappedVdk`, `WrapContext`, `VdkKey` types lives here.
- **DECISIONS.md D-001:** Rust core is single source of truth. This crate IS that core for storage.

## Goal

Ship `pangolin-store` — the local vault persistence layer. After this issue:

1. A `Vault` can be **created** on disk with a user password, producing a single `.pvf` (Pangolin Vault File) containing nothing-plaintext-sensitive.
2. The same `Vault` can be **opened, unlocked, used, locked, closed** repeatedly. The vault file is a single SQLite database.
3. **No plaintext sensitive data is ever written to disk** — verified by file-content scanning in tests.
4. Multiple **`Account`** entries can be added, edited, retrieved, and deleted (tombstoned). Each one has its sensitive content (display name, username, password, url, notes, totp_secret) inside an AEAD-encrypted blob.
5. **Local revision lineage** is tracked for every account — every edit produces a new revision referencing its parent. This is the structure that P7 (chain adapter) will sync up to the deployed `RevisionLogV0` contract.
6. **WAL + atomic write discipline** prevents corruption on crash mid-write.
7. **Encrypted-at-rest verification** — a property test that creates a vault, writes accounts with passwords like `"hunter2-marker-string"`, locks the vault, scans the raw bytes of the `.pvf` file for the marker string, and asserts ZERO occurrences. This is the test that proves cardinal principle 2 holds.

After this issue, P3 (account identity model production), P4 (session policy), P7 (chain adapter), and P9 (conflict resolution) can all build on top of `pangolin-store`'s API surface.

## Approach

### File-format design — single SQLite file

A vault is a single `.pvf` file on disk. It IS a SQLite database, with this layout:

```
.pvf file
└── SQLite database
    ├── meta                   (single-row table)
    │   ├── magic              "PNGLNVT0" (8 bytes)
    │   ├── format_version     u32 (currently 0)
    │   ├── vault_id           [u8; 32]
    │   ├── created_at         i64 (unix ms)
    │   ├── kdf_params         encoded (memory_kib, time_cost, parallelism, salt[16])
    │   └── wrapped_vdk        BLOB — full WrappedVdk including ctx, nonce, ciphertext
    │
    ├── account_identities     (one row per logical account)
    │   ├── account_id         [u8; 32] (UUIDv4-style random, generated client-side)
    │   ├── created_at         i64
    │   ├── last_modified_at   i64
    │   ├── tombstoned         BOOL
    │   └── (no sensitive content here — all sensitive fields live in revisions)
    │
    ├── revisions              (append-only — every edit = new row)
    │   ├── revision_id        [u8; 32] (deterministic — keccak256 of canonical revision body)
    │   ├── account_id         [u8; 32] (foreign key to account_identities)
    │   ├── parent_revision_id [u8; 32] (0x00...00 for genesis revisions)
    │   ├── device_id          [u8; 32] (the device that authored this revision)
    │   ├── schema_version     u8
    │   ├── created_at         i64
    │   ├── enc_payload        BLOB — AEAD ciphertext of the canonical CBOR-serialized
    │   │                              account snapshot (display_name, username, password,
    │   │                              url, notes, totp_secret, custom_fields)
    │   └── (chain_anchor    optional: tx hash + block when this revision lands on chain.
    │                          NULL until P7 publishes it. Indexed for sync queries.)
    │
    ├── devices                (one row per trusted device — generated by future MVP-1
    │                           issue 1.5; stub schema in P2 with just device_id + label.)
    │   ├── device_id          [u8; 32]
    │   ├── label              TEXT (human-readable, plaintext — non-sensitive by design)
    │   ├── added_at           i64
    │   └── revoked_at         i64 NULL
    │
    └── (sessions / guardians tables deferred to P4 / MVP-3 respectively)
```

### Crucial design decisions

**1. SQLite is NOT page-encrypted.** SQLCipher would add a C dependency and license complexity. Instead:
- All **sensitive content** lives inside `enc_payload` BLOB columns, encrypted with AEAD using `VdkKey`.
- All **non-sensitive structural metadata** is plaintext in the SQL columns (account UUIDs, revision parentage, timestamps, device IDs).
- This is intentionally compatible with the on-chain format from P5-1: `enc_payload` is the same bytes that get published in `RevisionPublished` events.

This means anyone who steals a locked `.pvf` file can see the **shape** of the vault (number of accounts, number of revisions, parent-child relationships, device IDs, timestamps) but not the **content** (no usernames, no URLs, no passwords, no notes). The threat model already acknowledges metadata leakage at the chain layer; aligning the local format with the chain format keeps the trust surface coherent.

**2. AEAD AAD binds revision context.** Each revision's `enc_payload` is sealed with AAD = `vault_id || account_id || parent_revision_id || schema_version`. This prevents any "transplant" attack where a revision blob from one account could be applied to another account by tampering with the SQL row. This matches the P1-3 `WrapContext` design pattern.

**3. Search strategy: in-memory decrypted cache.**
- On `unlock(password)`: decrypt all current account heads, build an in-memory `HashMap<account_id, AccountSnapshot>` plus an FTS-style word index over (display_name, username, url, notes_first_512_chars).
- On `search(query)`: hits the in-memory index. <50ms for 10k accounts.
- On `lock()`: drop the in-memory cache; underlying memory is `BoxedSecret`-wrapped + zeroized.
- For PoC: simple `HashMap` + linear-scan substring match. MVP-1 issue 1.3 will revisit FTS5 / tantivy / actual fuzzy matching.

**4. Atomic writes via SQLite WAL + `WITHOUT ROWID` discipline.**
- `PRAGMA journal_mode = WAL` for crash-resistant writes.
- `PRAGMA synchronous = FULL` for the durability we need (slightly slower than NORMAL but acceptable for a credential store).
- Vault-file backups (for MVP later) use SQLite's online backup API with `VACUUM INTO`.
- For BLOB updates: `INSERT OR REPLACE` is atomic at the row level.

**5. No `pub` on anything sensitive.** Following P1's discipline:
- `Vault` itself is the only public type that exposes credential operations.
- Internal types (raw row representations, decrypted snapshots) are `pub(crate)` or private.
- No `serde::Serialize` on any decrypted-secret-bearing type.
- All decrypted plaintext lives in `Zeroizing` / `BoxedSecret` wrappers; `Drop` zeroizes.

### Sub-issue breakdown (single PR, sequenced commits)

#### P2-1 — `pangolin-store` crate scaffold + SQLite schema

- Add SQLite dependency. Choose between `rusqlite` (mature, sync, well-known), `sqlx` (async, compile-time-checked), `sea-query`/`diesel` (heavier ORMs).
- **Recommended: `rusqlite` with `bundled` feature.** `bundled` ships SQLite as a vendored C library — no system dependency, no version drift across hosts, deterministic across Linux/Windows/macOS. The `bundled` build is what every other Rust crypto/storage project uses (1Password's web extension, signal-rust, etc). License: bundled SQLite is public domain; rusqlite is MIT.
- No async needed: P2 is local-only, blocks for ~1ms per query, async would add complexity without benefit.
- Schema migrations: hand-rolled `CREATE TABLE IF NOT EXISTS` statements run at vault open. `format_version` field on the `meta` table tracks schema changes; future versions `MUST` accept all old `format_version`s read-only and produce a stable error on unknown future versions (per master plan §18.7 schema-versioning policy stub).

#### P2-2 — Encrypted-blob layer + canonical CBOR

- `AccountSnapshot` struct: the in-memory representation of a fully-decrypted account at a point in time (display_name, username, password, url, notes, totp_secret, custom_fields).
- `AccountSnapshot` does NOT implement `Clone`, `Copy`, `Serialize` (the same discipline as `pangolin-crypto`'s secret types). `Debug` is redacted.
- Canonical CBOR encoding via the `ciborium` crate (RustCrypto adjacent, audited, deterministic — fields ordered, integer encoding canonical).
- The encoded CBOR bytes are sealed with `AeadKey::seal(...)` using the per-revision AAD described above.
- Result is the `enc_payload` BLOB.

#### P2-3 — `Vault` API surface

```rust
pub struct Vault { /* private */ }

impl Vault {
    /// Create a fresh vault file. Generates new vault_id, AuthorityKey,
    /// VdkKey, wraps VDK with the user's password-derived AEAD key.
    /// Writes the .pvf file to `path`. The vault is left in the *Locked*
    /// state — caller must `unlock` to add accounts.
    pub fn create(path: &Path, password: &SecretBytes) -> Result<Self>;

    /// Open an existing vault file (validates magic + format_version,
    /// reads meta but does NOT decrypt anything sensitive). Returns a
    /// Vault in the *Locked* state.
    pub fn open(path: &Path) -> Result<Self>;

    /// Unlock the vault. Derives the password-AEAD-key via Argon2id at
    /// the stored params, unwraps the WrappedVdk, decrypts all current
    /// account heads, builds the in-memory cache. Transitions to *Active*.
    /// Errors if password is wrong (returns AeadError::Tampered — same
    /// indistinguishable error as any other wrapping mismatch).
    pub fn unlock(&mut self, password: &SecretBytes) -> Result<()>;

    /// Lock the vault. Zeroizes the in-memory cache + VDK. Transitions
    /// back to *Locked*. Idempotent (calling on an already-locked vault
    /// is a no-op).
    pub fn lock(&mut self);

    /// Close the vault. Locks if needed, then closes the SQLite handle.
    pub fn close(self);

    /// Add a new account identity. Generates account_id + initial
    /// revision_id; seals the revision; writes to SQLite atomically.
    /// Requires *Active* state.
    pub fn add_account(&mut self, snapshot: AccountSnapshot) -> Result<AccountId>;

    /// Update an existing account. Builds a new revision referencing
    /// the current head as parent. Requires *Active*.
    pub fn update_account(&mut self, id: AccountId, new_snapshot: AccountSnapshot) -> Result<RevisionId>;

    /// Tombstone an account. Writes a special `{ deleted: true }`
    /// revision per master plan §3.5 PoC decision #5. Account remains
    /// queryable for history but is not returned by search.
    pub fn delete_account(&mut self, id: AccountId) -> Result<()>;

    /// Look up an account by id.
    pub fn get_account(&self, id: AccountId) -> Option<&AccountSnapshot>;

    /// Substring search across account display_name + username + url.
    pub fn search(&self, query: &str) -> Vec<AccountId>;

    /// List all accounts (for UI listing — not for export).
    pub fn list_accounts(&self) -> Vec<AccountId>;

    /// Vault-level metadata getter.
    pub fn vault_id(&self) -> [u8; 32];

    /// Current state — Locked / Active.
    pub fn state(&self) -> VaultState;

    // Revision history (for future P9 conflict resolution)
    pub fn revisions_for(&self, account_id: AccountId) -> Vec<RevisionMeta>;

    // The lineage primitive that P7 chain adapter will use to know
    // what's been published vs. needs publishing.
    pub fn unpublished_revisions(&self) -> Vec<RevisionId>;
    pub fn mark_published(&mut self, revision_id: RevisionId, anchor: ChainAnchor) -> Result<()>;
}
```

State machine:
```
                   ┌─────────────┐
                   │   Closed    │  (no SQLite handle)
                   └─────────────┘
                          │ Vault::open(path)
                          ▼
                   ┌─────────────┐
                   │   Locked    │  (handle open; no plaintext in memory)
                   └─────────────┘
                          │ unlock(password)  /  lock()
                          ▼
                   ┌─────────────┐
                   │   Active    │  (handle open; decrypted cache live;
                   └─────────────┘   accounts addable/searchable)
```

#### P2-4 — Corruption-safe writes + tests

- WAL mode on every open (`PRAGMA journal_mode = WAL` after open).
- `PRAGMA synchronous = FULL` for durability.
- Every write that touches multiple tables wraps in `BEGIN IMMEDIATE TRANSACTION ... COMMIT`. No multi-statement writes outside transactions.
- Crash-test: a test that creates a vault, opens it, starts a write, **kills the process mid-write** (via deliberate panic between statements), then re-opens the vault on a new SQLite handle and asserts no partial state. Done via `std::process::Command` spawning a child process that intentionally panics.
- File-system-level atomicity: when we eventually need to overwrite the entire vault file (e.g., compaction in MVP-2), use the standard write-temp-file-then-rename pattern. Not needed in P2 since SQLite WAL handles in-place mutation atomically.

## Success criteria

1. `pangolin-store` compiles, all tests pass, fmt + clippy + audit + deny stay clean.
2. **`Vault::create(path, password)` produces a `.pvf` file that begins with magic `PNGLNVT0` and has the format-version byte `0x00`.** Documented in a unit test.
3. **`Vault::create(path, password)` followed by `Vault::open(path)` followed by `unlock(password)` round-trips a freshly-added account byte-equal.** Property-based test, ≥1000 cases.
4. **Wrong password rejection:** `unlock(wrong_password)` returns `StoreError::AuthenticationFailed` (which maps to `AeadError::Tampered` underneath) and leaves the vault in *Locked* state.
5. **Plaintext-on-disk verification:** create vault, add an account whose password field contains the marker `"hunter2-marker-string-7e3a4b9c"`, lock the vault, close it, then read the raw bytes of the `.pvf` file and assert the marker string appears **zero times** in the file. Property test, ≥100 random markers.
6. **Revision lineage is correct:** add an account, edit it 5 times, walk the lineage from current head back to genesis revision, assert the chain is unbroken.
7. **Tombstone propagation:** delete an account, assert subsequent `get_account` returns `None`, but `revisions_for(id)` includes the tombstone revision.
8. **Search works:** add 100 accounts with varying display names, search for substrings, assert correct subset is returned.
9. **Lock zeroizes:** after `lock()`, the in-memory cache is dropped and `BoxedSecret`'s `ZeroizeOnDrop` discipline applies. Best-effort verification via the same `AtomicUsize` `DropCounter` harness used in `pangolin-crypto`.
10. **WAL mode is enforced:** after `open`, `PRAGMA journal_mode` returns `wal` (asserted in a test).
11. **Crash recovery:** test a crash mid-write (spawned child process panics between INSERT statements within a transaction), reopen the vault, assert vault is intact and the partial write was rolled back.
12. **Multi-instance safety:** opening the same vault file twice returns `StoreError::AlreadyOpen` from the second call (via SQLite's exclusive lock or our own file-lock).
13. **Forbidden-terms CI** doesn't fire. (No user-facing strings get added in this issue's diff.)
14. **`Cargo.lock` updated and committed.**
15. **No `unsafe` blocks** in this crate.
16. **No new `#[allow(...)]`** beyond the workspace-level allows from P1.

## Failure modes considered

| Failure mode | Mitigation |
|---|---|
| User loses password | Out of P2 scope. Recovery is MVP-3. P2 only validates that wrong-password errors are clean and indistinguishable. |
| Disk full during write | SQLite returns `SQLITE_FULL`; mapped to `StoreError::DiskFull`. Vault state remains consistent because writes are transactional. |
| Disk corrupted (bit-flips outside our control) | SQLite has page checksums; `PRAGMA integrity_check` on every open. If corrupted, return `StoreError::Corrupted` with file path; recovery is user's responsibility (PoC scope). |
| File deleted while open | SQLite holds OS-level file handles; on Linux/macOS the file's inode persists until close, on Windows the file is locked. Either way, our session continues; subsequent `open` fails. Tested. |
| Same vault opened twice | `flock()` style exclusive lock at file level (rusqlite's default). Second open returns clean error. |
| Vault file moved/renamed during session | SQLite tolerates this on most platforms. Not specifically tested in P2; documented as a known operational hazard. |
| Process killed mid-WAL | WAL replay on next open recovers consistent state. Test #11 verifies. |
| Argon2id parameters tampered with on disk | KDF params are inside the encrypted `meta` blob (so tampering breaks AEAD authentication on `unlock`). Wait — actually, KDF params are needed BEFORE we can derive the unwrap key, so they live in plaintext in the meta header. Tampering with them produces a derived key that fails to unwrap the VDK; same as wrong-password from the user's POV. Acceptable. |
| Schema downgrade attack (writing format_version=0xFF to fool an old client into ignoring the file) | We always check magic first, then format_version. format_version > current = clean error. Tested. |
| Side-channel: timing of unlock failure leaks whether password matched format vs. content | Argon2id always runs to completion before AEAD attempt; AEAD failure is constant-time per primitive. Document as best-effort; full constant-time auth is out of scope for PoC. |
| Memory: decrypted snapshots in cache live as long as session is unlocked | `BoxedSecret` discipline; lock() zeroizes; cache is dropped on Drop of Vault. |

## Threat enumeration (for `THREAT_MODEL.md` "Local encrypted store" row)

After P2 lands, append the following to `THREAT_MODEL.md` under "Local encrypted store" (which is currently `TBD (issue 0.2)`):

1. **Attacker reads disk image of locked vault.** Defense: every sensitive value lives inside an AEAD-sealed BLOB column. Decrypting any blob requires the AuthorityKey-derived wrap key, which requires the user password. The vault file's structural metadata (account count, revision parentage, timestamps) is visible — same as the on-chain log; this is a known design tradeoff.
2. **Attacker steals a `WrappedVdk` blob.** Defense: AEAD ciphertext is computationally infeasible to break absent the password.
3. **Attacker tampers with row contents to cross-account or cross-vault transplant.** Defense: AAD binds `vault_id || account_id || parent_revision_id || schema_version`. Tampered rows fail authentication.
4. **Attacker has memory dump after vault was unlocked.** Defense: `BoxedSecret` + `ZeroizeOnDrop` on every secret-bearing type. Best-effort; not a guarantee against attackers with kernel-level access (out of scope per top-level threat model).
5. **Concurrent corruption from two opens.** Defense: SQLite exclusive lock; `StoreError::AlreadyOpen` from the second open.
6. **Format-version downgrade attack.** Defense: explicit check; unknown future format-version returns clean error.
7. **`KDF` parameter tampering on disk.** Defense: tampered KDF params produce a derived key that fails the AEAD unwrap of `WrappedVdk`. From the user's POV, indistinguishable from wrong password.

## Rollback / abort plan

- If `rusqlite`'s bundled SQLite has a CVE: switch to `rusqlite` without `bundled` and document the system-SQLite dependency. `Cargo.lock` pinning still gives us reproducibility.
- If the in-memory cache approach turns out to be too slow for ≥100 accounts in PoC: pre-emptively switch to a write-through cache (decrypt-on-demand instead of decrypt-all-on-unlock). MVP-1 issue 1.3 will use FTS5 anyway.
- If `ciborium` produces non-canonical encodings under any input: switch to a hand-rolled fixed-field-order encoder. Document the change.
- If a fundamental design flaw is found mid-build (e.g., AAD encoding turns out to need additional context that breaks API surface): STOP, return to PLAN gate, revise this plan, get re-approval. **Do not silently drift.**

## Out of scope

- **Search via FTS5 / proper fuzzy matching.** Substring scan is sufficient for PoC. MVP-1 issue 1.3.
- **Multi-vault management.** This crate handles ONE vault per `Vault` instance. Multi-vault is a higher-level concern.
- **Backup/export to KDBX or other formats.** MVP-1 issues 1.9 (import) and 1.10 (export) handle this separately.
- **Vault compaction / vacuum.** SQLite's WAL grows over time; cleanup is MVP-1+ scope.
- **Encrypted SQLite (SQLCipher).** Per-blob encryption is the chosen design; full-database encryption is rejected to avoid the C dependency cost.
- **Network operations of any kind.** This crate has zero network code. P7 (chain adapter) is the chain integration.
- **Session policy enforcement.** P4 implements that on top of `Vault`.
- **Recovery / authority rotation.** MVP-3.
- **Full-text search relevance scoring.** Substring-match only.
- **Concurrent `Vault` access from multiple processes.** Excluded by single-writer invariant. Tests assert second `open` fails.
