# Issue P9: Conflicts & Resolve — `pangolin-cli resolve`

> **Status:** PLAN-gate. Per Kelvin's autonomy directive, dispatching the build directly upon §16-Plan-Gate approval.
> **Security-critical:** YES per §16.3. P9 is the user-facing release valve for the two related "vault is in an inconsistent state with chain" conditions P8 introduced (`is_forked` + `frozen_pending_resolve`); a flaw here either silently merges contradicting branches (violates Cardinal Principle 4: "never silent merge") or leaves the vault permanently locked (denies legitimate user access).

## Spec reference

- **Master plan §3.7 EPIC: Conflicts & Tombstones (P9–P10):**
  - `P9-1` Conflict detection raised to CLI when `pull` produces multiple heads.
  - `P9-2` `pangolin-cli resolve <account-id> --keep <revision-id>` — publishes new revision with parent = chosen head.
  - `P9-3` Convergence test: after both devices pull again, both see same single head.
- **Master plan §3.5:** Cardinal Principle 3 ("blockchain is a log, never an authority") + Cardinal Principle 4 ("never silent merge"). P9 is the locus where 4 stops being aspirational and becomes load-bearing: the chain has surfaced two heads, the user must decide which one ratifies forward.
- **Master plan §3.8 PoC exit criteria:** "two vaults converge on the same revision graph after publish + pull." P8 satisfied this for the no-conflict case; P9 satisfies it for the post-fork case.
- **DECISIONS.md:**
  - **D-005** — Base is the mainnet target; PoC runs against Base Sepolia at `0x8566D3de653ee55775783bD7918Fe91b66373896` (D-014).
  - **D-006** — No relay; each device is signer + gas payer. The PoC two-key deviation (P8 §A7) carries forward into P9 unchanged: resolve uses the same Foundry-keystore-for-gas + ephemeral-Ed25519-for-signing model as `publish`.
  - **D-007** — No persistent indexer; resolve's pre-publish view comes from `BaseSepoliaAdapter::pull_since`, the same primitive `publish_all` uses.
- **Cardinal principle 4:** "Never silent merge." `resolve` is the single path that takes a forked or frozen account from "user must decide" to "single head moving forward"; its on-chain action is explicit, signed, and user-attested via `--keep`.
- **Whitepaper §G3 (Conflict Resolution):** P9 is the surface this section describes — the user picks one of N heads, the picked head becomes the parent of a freshly-published merge revision, and convergence happens via every other device's next pull.

## What P0..P8 already shipped (the load-bearing surface for P9)

P9 introduces no new primitive. It composes existing pieces in a new way:

- **P3 / P8 — multi-head detection in `pangolin-store::Vault`:**
  - `Vault::account_heads(account_id) -> Vec<RevisionId>` — returns N≥1 heads. The SQL primitive is "every revision row that has no children for the same account." A merge revision lands as a child of one head; the chosen head ceases to be in `account_heads(...)` (it has a child); the OTHER heads still have no children → they remain in `account_heads(...)` until the user resolves them too OR a descendant is published with one of them as parent. **This is the core architectural fact P9 must reckon with — see A4.**
  - `Vault::is_forked(id) -> bool` — `account_heads(id).len() > 1`.
  - `Vault::all_forked_accounts() -> Vec<AccountId>` — every account currently in a multi-head state.
- **P8 — `frozen_pending_resolve` sentinel in `pangolin-store::Vault`:**
  - `Vault::ingest_chain_revision` lights `account_identities.frozen_pending_resolve = 1` when an event takes the genuine-foreign-INSERT path (none of the three idempotency arms hit). `is_account_frozen(id)` / `list_frozen_accounts()` / `frozen_set()` expose the state.
  - Read guards: `get_account` returns `None`; `list_accounts` / `search` filter out the row; `reveal_password` / `reveal_notes` / `reveal_totp_secret` / `export_payload` surface `StoreError::AccountFrozenPendingResolve { account_id }`.
  - Edit guards: `update_account` / `delete_account` / `mark_dirty` refuse with the same error.
- **P7 — `pangolin-chain` publish path:**
  - `ChainAdapter::publish(&SignedRevision) -> Result<ChainAnchor>` — same primitive `publish_all` calls.
  - `signing::build_signed_revision(...)` — produces an Ed25519-signed `SignedRevision`. The canonical hash binds `(vault_id, account_id, parent_revision, device_id, schema_version, enc_payload)`; **a merge revision shares vault_id + account_id with the chosen head, and its parent_revision IS the chosen head's revision_id, so the canonical hash differs from the chosen head's even when the payload bytes are byte-identical.**
- **P8 — `pangolin-cli sync` orchestrator:**
  - `publish_all<A: ChainAdapter>(...)` — drives the dirty list through the publish path. Resolve does NOT walk the dirty list; it publishes a single revision constructed in-line. We share the lower-level pieces (`build_signed_revision`, `ChainAdapter::publish`, `Vault::mark_published`) but do not reuse `publish_all` itself.
  - `pull_all<A: ChainAdapter>(...)` — chunked pull with fork + freeze surfacing. Resolve relies on `pull_all` happening BEFORE resolve (so the chain view is current) and AFTER resolve (so the local store ingests its own merge revision and the freeze flag clears via the natural ingest path).

P9 introduces the **first user-facing CLI for explicit conflict resolution**. The user reads `pangolin-cli status`, sees a forked-or-frozen account, and runs `pangolin-cli resolve --account-id <hex> --keep <revision-id-hex>` to commit to one branch.

## Goal

Ship `pangolin-cli resolve` as the production-shape PoC release valve for both fork and freeze states. Given a forked or frozen account and the user's choice of which revision to keep, build a merge revision (parent = chosen head, payload = a fresh AEAD seal of the chosen head's payload), sign with the device's ephemeral Ed25519 key, publish through the deployed `RevisionLogV0`, and clear the freeze sentinel on success. After resolve and a subsequent pull from every device sharing the vault, every device's `account_heads(id)` returns a single head equal to the merge revision (modulo the unchosen-heads-as-orphans wrinkle in A4 — addressed there).

## Architectural decisions (locked in this plan)

These decisions are committed to as part of plan-gate approval. Any deviation during BUILD requires returning to PLAN per §16.4.

### A1. Resolve UX — `--yes` for scripted use; interactive prompt for default

**Decision:** Default behavior on `pangolin-cli resolve --account-id <hex> --keep <revision-id-hex>` (no `--yes`) prints the planned action — "you are about to publish a merge revision under account X with parent = revision Y; the unchosen heads (Z1, Z2, …) will become orphans on this device until each is resolved or a descendant lands; press Y to proceed, anything else to abort" — and reads a single character from stdin via `rpassword::read_password` or a local `dialoguer` prompt. With `--yes`, skip the prompt. With `--dry-run`, print the same planned action then exit 0 without publishing.

**Rationale:** Default-interactive matches Cardinal Principle 4's "never silent merge" at the UX layer: even though the user typed `--keep <id>`, surfacing the consequences before the on-chain side-effect catches typos and unexpected orphan-head accumulation. `--yes` exists for scripted use (CI, MVP-1 host-app integration). `--dry-run` is cheap to implement and high-value for power users — the planned-action printer is the same code path either way.

**Alternative considered:** Always-interactive, no `--yes` flag (force a human in the loop). **Rejected** because it breaks scripted-test setups (the integration tests in §4 below need to drive resolve programmatically).

### A2. Merge revision payload — re-seal, do not copy ciphertext

**Decision:** The merge revision's `enc_payload` is a **fresh AEAD seal** of the chosen head's plaintext under the local VDK with a fresh nonce. It is NOT a byte-copy of the chosen head's ciphertext.

**Rationale (carefully — there's a real subtle threat to consider):**

The canonical hash binds `(vault_id, account_id, parent_revision, device_id, schema_version, enc_payload)`. The merge revision's `parent_revision` is the chosen head's `revision_id`, which differs from the chosen head's own `parent_revision`. So even if `enc_payload` were a byte-copy, the canonical hashes would differ. This means the hash-collision argument does NOT motivate the re-seal.

The actual motivations are:

1. **AEAD AAD discipline.** `pangolin-store`'s seal AAD binds `WRAP_AAD_DOMAIN_REV (8B "pgrev0\0\0") || vault_id (32B) || account_id (32B) || parent_revision_id (32B) || schema_version (1B)`. The chosen head's ciphertext was sealed with **the chosen head's** `parent_revision_id` in the AAD. The merge revision's `parent_revision_id` is the chosen head's `revision_id` (NOT the chosen head's parent). So a byte-copy of the ciphertext would have a stale AAD baked in; the AEAD open path would FAIL on the merge revision row when any future reader tried to decrypt it, because the open computes AAD from the merge row's own metadata. This is the load-bearing reason: **a ciphertext copy would produce an unopenable merge revision row.** Re-seal with the merge row's own AAD is mandatory for the merge revision to be readable downstream.

2. **AEAD nonce hygiene (defense-in-depth).** Even if AAD weren't an issue, re-using the chosen head's exact `(VDK, nonce)` pair with a different (account-binding) AAD is technically safe under AEAD's IND-CCA2 guarantees but invites a class of "did the developer think about nonce reuse?" audit findings. A fresh nonce generated by the same `OsRng`-backed nonce source the rest of `pangolin-store` uses is structurally cleaner and removes the need to reason about whether the AAD difference saves us.

3. **Forward consistency with MVP-1.** When MVP-1 switches to deterministic AEAD nonces (one of the items on the schema-versioning list), the merge path will need to compute its own nonce anyway. Coding the re-seal now means MVP-1 is a no-op upgrade rather than a subtle behavior change.

**Alternative considered:** Byte-copy the chosen head's ciphertext into the merge revision's `enc_payload`. **Rejected** for reason (1) above — the merge row would be unopenable. This is not a stylistic preference; it's a correctness requirement.

**Subtle threat acknowledged:** A user who runs `resolve` does NOT see the plaintext during the resolve operation by default (the merge is mechanical: read chosen head's plaintext, re-seal, publish). The plaintext is materialized in memory inside `Vault::read_payload_plaintext_for_resolve` and zeroized on drop. The `--dry-run` path does NOT read or display plaintext; it only prints metadata (revision_id, parent, frozen flag).

### A3. Atomicity — publish-then-clear is non-load-bearing for correctness

**Decision:** The resolve operation is "publish the merge revision, then on success run `Vault::clear_frozen(account_id, merge_revision_id)`." If the publish succeeds and clear_frozen fails (process killed, disk full), the user is in a known-recoverable state: the merge revision is on chain; on the next `pangolin-cli pull`, `Vault::ingest_chain_revision` will see the merge event and take idempotency arm #3 (content-merge with chain anchor stamping) IF the local row's `device_id` matches the merge event's `device_id`, which under the PoC two-key model means: the local pre-resolve row (which doesn't exist yet — resolve doesn't create a local pre-publish row, it goes straight to publish) — wait, this needs careful thought.

**Reconsidered:** Under the PoC two-key model, `pangolin-cli resolve` generates a fresh `DeviceKey` per run (same as `publish`). It does NOT pre-create a local pre-publish revision row before publishing — it builds the `SignedRevision` in memory and submits. So when the post-resolve pull runs, `ingest_chain_revision` sees the merge event for the FIRST time (no row matches idempotency arms #1, #2, or #3) and takes the genuine-foreign-INSERT path. That path **sets frozen_pending_resolve = 1** for the account (CRIT-1 sentinel). So a process killed between publish and clear_frozen leaves the vault in a state where the next pull will (a) ingest the merge revision, (b) RE-FREEZE the account, (c) require a SECOND `resolve` run to clear.

**This is a load-bearing correctness concern.** The fix: resolve must call `Vault::ingest_chain_revision` ITSELF immediately after the successful publish, BEFORE the next pull has a chance to run, AND it must call `clear_frozen` AFTER the ingest to advance the head pointer + clear the flag. The window between publish-success and ingest is small (microseconds) but if the process is killed inside it, recovery is: re-run resolve with the same `--keep` argument; the resolve flow detects the merge revision is already on chain (via the same A3-style pre-publish check `publish_all` uses), skips the publish, runs only the ingest + clear_frozen.

**Updated decision:** Resolve's commit step is THREE writes, not two:

1. `Vault::ingest_chain_revision(&merge_event)` — stores the merge revision locally with chain anchor populated. Under the genuine-foreign-INSERT path, this re-sets `frozen_pending_resolve = 1` on the account.
2. `Vault::clear_frozen(account_id, merge_revision_id)` — advances `head_revision_id` to the merge revision's id AND clears the freeze flag. One transaction.
3. (Implicit) the next `pangolin-cli pull` is a no-op for this revision because step 1 already ingested it (idempotency arm #1: exact-hash match).

A kill between (1) and (2) leaves `frozen=1` and the merge revision present locally. Recovery: re-run `resolve` with the same `--keep`; the pre-publish check sees the merge already on chain, skips the publish, calls `ingest_chain_revision` (idempotency arm #1: no-op), calls `clear_frozen` (clears the flag, advances head). Idempotent end-state.

**Rationale:** The natural ingest path under genuine-foreign-INSERT re-arms the freeze, so we cannot rely on "publish + ingest converges everything via the next pull." The explicit `clear_frozen` call after `ingest_chain_revision` is required for resolve to actually clear the state it was invoked to clear.

**Alternative considered:** Have `ingest_chain_revision` recognize a "this is my own merge publish" signal and skip the freeze sentinel for that one event. **Rejected** because it requires plumbing a side-channel from the publish path through the ingest path (e.g., a `mark_my_own_publish(revision_id)` call before publish), which is fragile and doubles the surface for bugs. The explicit clear_frozen is simpler and audit-reviewable.

### A4. The unchosen-heads orphan problem

**Decision:** Resolve keeps ONE head (the user-chosen one); the unchosen heads remain in `account_heads(...)` after resolve because they still have no children. They are **orphans**: locally readable as historical revisions, structurally present in the revision graph, but not on the canonical-path-from-genesis-to-current-head.

The next pull from any device that has the orphans will see them as accounts with multiple heads. That device's `pangolin-cli status` will report `is_forked = true` until the user resolves each orphan in turn (or a descendant of the orphan lands on chain — under the PoC, no such descendant will land because no device chose to keep that branch).

**Confirmed by reading `vault.rs` `account_heads` SQL:** the multi-head detector is "rows with no children for the same account," which is content-deterministic and does NOT have a "stale" mark. There is no automatic demotion of an orphan to "stale tip" — the user must resolve each branch explicitly OR the unchosen branch must be ignored forever. PoC ships with this behavior.

**Implication for the convergence test (P9-5):** The post-resolve `account_heads(account_id)` returns BOTH the merge revision AND any unchosen head that hasn't been resolved. For the simple two-head case, post-resolve `account_heads` returns 2 entries: the merge (chosen branch's descendant) and the unchosen orphan. After the user runs `resolve` AGAIN with `--keep <merge_revision_id>` (or runs it for the unchosen orphan as a no-op merge), the unchosen orphan ceases to be a head.

**For the integration test:** the `convergence_after_resolve` test resolves the first fork, observes that B's `account_heads` now contains 2 entries (the merge AND the orphan), then resolves the orphan with a second merge revision (parent = orphan-head, payload = the merge revision's plaintext, so the user is "abandoning the orphan branch by saying 'the merge content is what I want, but with the orphan as parent so the orphan is no longer head'"). After both resolves, `account_heads` returns 1 entry on both A and B. **This is the canonical PoC convergence pattern.**

**Rationale:** Cardinal Principle 4 says "never silent merge"; that includes "never silently demote an unchosen branch to stale." Every head transition is user-attested. The cost is that resolving an N-way fork takes N-1 explicit resolves. For the PoC's 2-way fork case (the only case the master plan §3.7 contemplates), this is one extra resolve.

**Alternative considered:** Add a `Vault::demote_orphan_heads(account_id, surviving_head_revision_id)` API that marks all OTHER heads as "stale" so they no longer appear in `account_heads`. **Rejected** for PoC: it requires a new column on `revisions` (e.g., `superseded_by`), and the semantic is "the user resolves all branches at once" which is not always what they want (they may want to merge the orphan's content into a future revision). MVP-1 may revisit if user testing shows the multi-resolve flow is awkward.

**Open question for Kelvin:** is the multi-resolve-N-times-for-N-way-fork behavior acceptable, or should P9 ship with the demote API? Recommend the multi-resolve behavior — simpler, audit-reviewable, doesn't introduce a new "stale" concept. See §8.

### A5. Tombstone-head resolve produces a tombstone merge revision

**Decision:** If the user's `--keep <revision-id>` references a tombstoned revision (a revision whose payload was a `{ "deleted": true, ... }` tombstone — P10 territory but the structural flag exists today as `revisions.is_tombstone`), the merge revision IS ALSO a tombstone. Specifically: read the chosen tombstone's plaintext (which is the tombstone payload), re-seal it under the merge row's AAD, set `is_tombstone=1` on the local merge row, publish the on-chain event with the tombstone payload bytes.

**Rationale:** Resolving to a tombstone = ratifying the deletion. The merge revision must structurally be a tombstone (so future reads of the account see "deleted") rather than a fresh non-tombstone revision (which would un-delete the account, contradicting the user's `--keep` choice). Under MVP-1's full tombstone semantics (P10), this gives the cleanest append-only history: `genesis → edit → tombstone → tombstone-merge` is structurally equivalent to `genesis → edit → tombstone` from the consumer's perspective, and the second tombstone is an explicit record that the user reaffirmed the deletion in a multi-head conflict.

**Implementation note:** P9 ships the tombstone-of-tombstone structural support (the `is_tombstone` flag round-trips through the merge build). Full tombstone semantics — what `get_account` returns for a tombstoned-then-merged account, whether `update_account` is allowed — is **P10's job**. P9's contribution is "if the chosen head is a tombstone, the merge revision is a tombstone."

**Alternative considered:** A tombstone-resolve produces a fresh non-tombstone revision (the user is "reviving" the account). **Rejected** as user-hostile and contrary to `--keep`'s plain English: keeping a tombstone means keeping the deletion.

### A6. Forbidden user-facing terms

**Decision:** The subcommand name `resolve` is fine. The flag `--keep <revision-id>` exposes "revision-id" which is a Pangolin term, not on the forbidden list. Help text MUST NOT use "blockchain", "wallet", "gas", "transaction", "fork" (in the chain sense), "merge" (in the git sense), "blockchain log", "smart contract".

**Permitted Pangolin terms in help text:** `revision`, `account`, `vault`, `head`, `conflict`, `frozen`, `device`, `payload`. These are the user-facing nouns of the data model; they appear throughout `chaincli` and `pangolin-cli` help text per the §3.5 cardinal-principle terminology rules.

**Rationale:** The cardinal principles list the forbidden terms explicitly because they invite the wrong mental model (gas = "this might cost me money," wallet = "this holds my crypto"). The user is editing a vault, resolving a conflict between two edits, picking which one to keep. No exposure of the chain layer leaks into the resolve command's surface text.

### A7. Concurrency — two devices running resolve simultaneously

**Decision (acknowledged behavior):** If devices A and B both run `pangolin-cli resolve` on the same forked account concurrently, both publishes will succeed (same race as P8 plan §A6 — the chain has no contention semantics for concurrent publishes on the same parent). The result is yet ANOTHER fork: parent = chosen_head, two children (A's merge and B's merge). The next pull from any device surfaces the new fork; the user resolves it again with a third resolve call.

**Rationale:** This is the same race as P8's concurrent-edit case. The chain is a log; it cannot enforce "only one merge revision per parent." The mitigation is the same as P8: the user re-resolves on next pull. P11 may add a "resolve must show the latest head before publishing" interactive guard (require the user to confirm "yes, parent X is still a head as of right now") that closes a small window of the race. P9 ships without it.

**Open question for Kelvin:** is this acceptable, or should P9 add the interactive freshness guard? Recommend ship-without; the race is rare in practice (concurrent resolve attempts on the same fork require both devices to be online and aware of the fork at the same instant) and the recovery (re-resolve) is straightforward. See §8.

### A8. Locking — resolve runs while vault is unlocked

**Decision:** Resolve requires the vault to be `Active`. The publish path needs to read the chosen revision's plaintext (re-seal under fresh nonce + correct AAD), which requires the VDK in the decrypted cache. The `read_payload_plaintext_for_resolve` accessor takes the same `&mut self` as other read paths and runs the same session-active check.

The `--vault-password` flag and the prompt-for-password fallback work the same way as `publish`. The user runs `pangolin-cli resolve` on an unlocked or about-to-be-unlocked vault. The CLI does NOT chain `unlock` and `resolve` — they are separate subcommands; the user calls `unlock` (when MVP-1 ships a host app) or supplies the password to resolve.

**Rationale:** The frozen-account read guards (`get_account` returns None on frozen) MUST NOT block resolve's own internal read of the chosen head's payload. That's why `read_payload_plaintext_for_resolve` is a new accessor that bypasses the frozen guard for this one call — the user's `--keep <id>` is the explicit proof-of-intent that they want to read this specific revision's plaintext to re-seal it. Other read paths (`get_account`, `reveal_password`, `export_payload`) keep the frozen guard, since they are not user-attested-by-`--keep`.

**Alternative considered:** Have resolve read the chosen revision's CIPHERTEXT only (no decrypt), and re-seal as a copy of those bytes. **Rejected** per A2: the AAD differs, so the merge row would be unopenable. Re-seal is mandatory; therefore plaintext must be in memory transiently; therefore the freeze guard must be bypassed for this one read.

```rust
// in crates/pangolin-store/src/conflict.rs (NEW module)
pub struct ConflictReport {
    pub account_id: AccountId,
    /// All current heads of the account. Length 1 with `frozen=true`
    /// means the chain landed an event that didn't fork the local
    /// graph but the local store still flagged a freeze (CRIT-1
    /// genuine-foreign-INSERT path on a fresh foreign account).
    /// Length > 1 means the account is forked.
    pub heads: Vec<RevisionId>,
    /// `account_identities.frozen_pending_resolve == 1` for this row.
    pub frozen: bool,
}

impl Vault {
    /// Snapshot every account currently in a conflict-needing-resolution
    /// state — fork or freeze or both. Sorted by `account_id` byte-order
    /// ASC.
    pub fn list_conflicts(&self) -> Result<Vec<ConflictReport>>;

    /// Clear the `frozen_pending_resolve` flag on `account_id` and
    /// advance `head_revision_id` to `chosen_revision_id`. Both writes
    /// run in one `BEGIN IMMEDIATE` transaction. Idempotent: a non-
    /// frozen account whose `head_revision_id` already equals
    /// `chosen_revision_id` is a no-op.
    pub fn clear_frozen(
        &mut self,
        account_id: AccountId,
        chosen_revision_id: RevisionId,
    ) -> Result<()>;

    /// Read the plaintext of an arbitrary revision (not necessarily
    /// the current head) belonging to `account_id`. Bypasses the
    /// `frozen_pending_resolve` read guard because the caller has
    /// supplied the explicit `revision_id` as proof-of-intent.
    /// **Used only by `pangolin-cli resolve`.** Returns the decrypted
    /// `AccountSnapshot` zeroized on drop.
    pub fn read_payload_plaintext_for_resolve(
        &mut self,
        account_id: AccountId,
        revision_id: RevisionId,
    ) -> Result<AccountSnapshot>;
}
```

## Threat model row (draft text for THREAT_MODEL.md)

The existing `pangolin-cli` row in `THREAT_MODEL.md` (lines 229+) covers publish + pull. P9 appends a resolve paragraph to the same row. Format below preserves the row's numbering style — the new threat indices are 12, 13, and 14 (continuing from the existing 1–11):

> 12. **Forged resolve (foreign device claiming to be the user's).**
>     Defense: the merge revision is signed by the device's Ed25519
>     `DeviceKey` via `signing::build_signed_revision`, same path as
>     `publish`. The canonical hash binds `parent_revision` (= the
>     chosen head's revision_id), `account_id`, `vault_id`,
>     `device_id`, and `enc_payload`. v0 contract does not verify on-
>     chain; v1 will (MVP-2 issue 2.1). Per Q6 defense-in-depth, the
>     receiving device's `pull_all` runs `VerifyingKey::from_bytes` on
>     the merge event's `device_id` before invoking
>     `Vault::ingest_chain_revision` — same gate that catches forged
>     publish events. The PoC two-key model (P8 §A7) carries forward
>     unchanged: resolve generates an ephemeral `DeviceKey` per run.
> 13. **Replay of an old resolve.** Defense: the canonical hash binds
>     `parent_revision`. A resolve replay against a moved-on head
>     (someone else has published a descendant in the meantime,
>     advancing the head past the chosen one) lands as another fork
>     rather than a duplicate, surfacing on the next pull as exactly
>     the concurrent-resolve race described in §A7. Re-publishing the
>     same merge revision with a stale parent is additionally guarded
>     by the resolve-flow's pre-publish check (same A3-style
>     `pull_since` lookup `publish_all` uses): if the merge revision
>     is already on chain, skip the publish, run only the local
>     `ingest_chain_revision` + `clear_frozen`.
> 14. **Frozen flag cleared without publish.** Defense: the
>     `clear_frozen` API takes `chosen_revision_id` as a parameter and
>     atomically advances `head_revision_id` to it; the resolve flow
>     ALWAYS calls `clear_frozen` only after a successful publish +
>     ingest of the merge revision. There is no API path that clears
>     the freeze flag without a corresponding revision row in the
>     local store — `clear_frozen` errors with
>     `StoreError::RevisionNotFound` if the supplied
>     `chosen_revision_id` does not exist as a `revisions` row. A
>     malicious local actor with vault-file access could `UPDATE
>     account_identities SET frozen_pending_resolve = 0` directly via
>     sqlite tooling, but that's the same as them tampering with any
>     other row — not a defense the application layer can mount.
> 15. **User keeps an attacker-controlled head (HIGH-1 from P8 audit).**
>     Defense (acknowledgement, UX-only): under the threat model
>     where a malicious RPC injects events with garbage `device_id`,
>     P8 fix CRIT-1 freezes the account so the user cannot read the
>     stale plaintext. If the user then runs `resolve --keep <id>`
>     where `<id>` references one of the attacker-injected events,
>     they have explicitly adopted attacker-controlled state. The
>     mitigation is UX: `pangolin-cli resolve` prints the metadata of
>     each candidate head (its `device_id`, its created_at, its
>     parent) so the user can spot an unfamiliar `device_id` (a
>     foreign device they don't recognize). Full defense requires v1
>     contract on-chain signature verification, which is MVP-2 issue
>     2.1; PoC ships with the UX surfacing as the only defense
>     against this class. Documented as a known UX-bound gap.
>
> *(Resolve does not introduce a new MED-1-class device_id-binding
> threat. The merge revision IS the user's own publish — the local
> ingest path immediately stamps it via `ingest_chain_revision`,
> recognizing it as own-publish via the same idempotency arms publish
> uses on its own round-trip. The freeze fires once on the post-pull
> ingest of a foreign device's merge; the receiving device clears it
> by running their own resolve against the merge revision as a head,
> i.e., a no-op merge — see §A4.)*

## Failure modes considered

| Failure mode | Mitigation |
|---|---|
| User runs `pangolin-cli resolve` while vault is `Locked` | CLI prompts for password (or uses `--vault-password`); on wrong password, surfaces `StoreError::AuthenticationFailed` verbatim and exits non-zero. Same as `publish`. |
| User passes `--keep <revision-id>` where `<revision-id>` is not a current head | Resolve flow refuses with a clear error: "the supplied revision is not a current head; resolve requires the kept revision to be one of `account_heads(...)`." Exit non-zero before any chain call. |
| User passes `--keep <revision-id>` with the wrong `--account-id` | The `read_payload_plaintext_for_resolve` accessor cross-checks `revisions.account_id == supplied account_id`; mismatch → `StoreError::AccountNotFound` (collapsing into the same error variant as "no such account" so an attacker cannot oracle a partial match). |
| User passes a `--keep` that decodes to a tombstone revision | Resolve produces a tombstone merge revision per §A5. The local row's `is_tombstone` is set; the on-chain payload is the tombstone payload bytes. P10 owns the consumer-side semantics. |
| Process killed between publish-success and `clear_frozen` | A3 — re-run `pangolin-cli resolve` with the same `--keep`; the pre-publish check sees the merge revision is already on chain, skips the publish, calls `ingest_chain_revision` (idempotency arm #1: no-op), calls `clear_frozen` (clears the flag). Verified by `resolve_idempotent_after_partial_failure_resumes_correctly`. |
| Two devices run `resolve` concurrently on the same fork | A7 — both publishes succeed; chain has another fork; user re-resolves on next pull. Acknowledged behavior. |
| User runs resolve on a `frozen=true heads.len()=1` account (the CRIT-1-on-fresh-foreign-account case) | Resolve treats the single head as the chosen branch by default if the user passes that head as `--keep`; if the user passes any other revision id, refuse. The merge publishes a fresh revision under the single existing head, clearing the freeze. Verified by a unit test in `resolve_publishes_merge_revision`. |
| User runs resolve on a `frozen=false heads.len()>1` account (pure local fork, no chain involvement yet) | Same flow as the chain-fork case. The merge revision is published normally; the freeze sentinel was never set so `clear_frozen` is a no-op for the freeze flag (it still advances `head_revision_id`). |
| User runs resolve on an account that is NOT in `list_conflicts()` (clean state) | Resolve refuses with "account is not in a conflict state; nothing to resolve." Exit non-zero before any chain call. Defensive — `--keep` only makes sense when there's an ambiguity to resolve. |
| RPC returns "transaction reverted" on the merge publish | Same as `publish` — `ChainError::Reverted { tx_hash }` surfaces; freeze flag remains set; user retries. |
| RPC returns the chain head is older than `last_pulled_block` (chain reorg) | Out of scope for P9 (and P7 / P8). Documented as a known limitation. MVP-2 may add reorg-aware resolve. |
| User runs resolve on a vault file that another `pangolin-cli` invocation is currently using | Vault file lock held by the first invocation; second hits the file-lock and fails-fast with "vault is in use." Inherited from `pangolin-store`'s lock discipline. |

## Rollback / abort plan

If BUILD reveals the design is fundamentally wrong (per §16.7):

- **A2 wrong (re-seal is structurally impossible — e.g., `read_payload_plaintext_for_resolve` collides with the freeze guard in a way that can't be reconciled):** Revert to a "resolve writes a sentinel-only merge revision with parent=chosen_head and a SPECIAL payload schema_version=resolve-marker that future readers know to skip" model. This requires a new schema version; document the deviation in DEVLOG and update the threat-model row. Likely needs a D-NNN entry locking the new schema-version.
- **A3 atomicity model is wrong (e.g., the ingest_chain_revision-then-clear_frozen sequence has a race the design didn't catch):** Revert to a publish-only resolve and rely on the next pull to ingest + a SECOND resolve to clear. Document as a known awkwardness; UX cost is high but design risk is low.
- **A4 orphan-handling is too painful (every audit reviewer flags it as a footgun):** Add the `Vault::demote_orphan_heads` API and ship the demote semantics. Net new SQL column on `revisions` (or a separate `superseded_revisions` table); requires §16.4 plan revision.
- **A8 read_payload_plaintext_for_resolve is too dangerous (audit deems it a backdoor around the freeze guard):** Replace with a "user re-supplies the password as a fresh proof, AND we read the plaintext under the freeze guard temporarily lifted for the duration of the read" model. More UX friction but more obviously safe.

If a more fundamental design problem surfaces (e.g., the convergence test reveals the merge-then-pull-then-resolve flow doesn't actually converge under the PoC two-key model because of a device_id mismatch we didn't anticipate), escalate to Kelvin per §16.7. Do not silently patch around.

## Open questions for Kelvin

1. **A4 orphan handling.** Is the multi-resolve-N-times-for-N-way-fork behavior acceptable, or should P9 ship with a `Vault::demote_orphan_heads` API that retires unchosen branches in one resolve call? Recommend the multi-resolve behavior — simpler, audit-reviewable, doesn't introduce a new "stale" concept; revisit in MVP-1 if user testing shows it's awkward.
2. **A7 concurrent-resolve race.** Should P9 add an interactive freshness guard ("verify chosen head is still a head before publishing")? Recommend ship-without; the race is rare and recovery is straightforward.
3. **Short revision-id prefixes.** Should `--keep` accept a 8-or-12-character prefix (git-style) or require the full 64-hex-char id? Recommend full-hex for PoC; short-prefix is an MVP-1 polish.
4. **Tombstone-of-tombstone vs. ratify-only.** §A5 says a resolve to a tombstone produces a tombstone merge revision (i.e., the structurally cleanest "tombstone-of-tombstone" append-only history). An alternative is "tombstone-resolve writes a sentinel that ratifies the deletion without creating another tombstone payload." Recommend tombstone-of-tombstone — clean append-only history, no new sentinel concept.
5. **Dry-run mode.** §A1 ships `--dry-run`. Confirm this is wanted at PoC scale; it's cheap to implement but adds one more flag to the help text. Recommend yes — high-value for power users diagnosing a fork.
6. **`read_payload_plaintext_for_resolve` as a documented backdoor around the freeze guard.** §A8 introduces this as the only path to read a frozen account's plaintext. Audit reviewers may want explicit confirmation that this is the intended design. Recommend documenting it loudly in the THREAT_MODEL row and in the API docstring; we accept the audit-noise cost in exchange for design clarity.
7. **Chain-side freshness check for `--keep`.** Should resolve re-pull from chain BEFORE publishing the merge revision (to ensure the chosen head is still a head as of right now)? Recommend yes — a cheap one-`pull_since` call before the publish, costing one round-trip but closing a small race window. Same chain view the A3 pre-publish check uses; just folded into resolve's flow.

## Security-critical?

**YES** per §16.3 (chain code; conflict resolution is the moment Cardinal Principle 4 stops being aspirational and becomes load-bearing). Audit at SIGNOFF will scrutinize:

- The A2 re-seal discipline. Does the merge revision's AAD bind the merge row's own `parent_revision_id` (= the chosen head's `revision_id`)? Or does a code path accidentally bind the chosen head's own `parent_revision_id`? Spot-check the `read_payload_plaintext_for_resolve` → re-seal handoff and confirm the AAD computation flows from the merge row's metadata, not the source row's.
- The A3 atomicity sequence. Does the resolve flow ALWAYS call `clear_frozen` after `ingest_chain_revision`, or is there a code path that skips the clear under some error condition? The freeze flag must NOT remain set after a successful resolve; if it does, the user has been told "resolve done" but their account is still frozen. Add a post-condition assertion in the resolve flow.
- The A4 orphan-handling consequence. Does the post-resolve `account_heads(...)` correctly include the merge revision PLUS any unchosen orphans? Confirm via the `resolve_against_three_heads_keeps_chosen_demotes_others_to_orphans` test and inspect the SQL: it should return ≥2 rows when there were ≥3 heads pre-resolve.
- The A8 `read_payload_plaintext_for_resolve` bypass. Does the accessor cross-check `revisions.account_id == supplied_account_id`? An attacker who somehow obtains a `revision_id` for a different vault's account must NOT be able to read its plaintext via this path. Add the cross-check explicitly and test it.
- The PoC two-key model carry-forward. Does resolve generate its `DeviceKey` the same way `publish` does? Does the merge event's `device_id` match what the receiving device's `pull_all` expects to see? Cross-check by reading both code paths and asserting the same `DeviceKey::generate()` semantics.
- The `convergence_after_resolve` integration test. Does it actually exercise the full A→publish→B→pull→B-frozen→A→resolve→A-publish→B→pull→B-converged flow without skipping any step? Walk through it manually and confirm every step is asserted (frozen state, ingest outcome, clear_frozen result, head count).
