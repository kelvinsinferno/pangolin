## Open questions for Kelvin (LOAD-BEARING — surface at TOP per §16)

### Q-a · Threshold value — hardcoded 100, configurable, or both?

Master plan §5 row 4.4 says "<100 unsynced revisions → slow-mode" verbatim. Three knob shapes:

**Option A: Hardcoded `pub const UNSYNCED_THRESHOLD: u64 = 100;`.** No override surface. Pro: simplest; spec-compliant verbatim; one path to test. Con: no test-clock-style escape hatch for hermetic tests (a test that wants to exercise "OfferFast" path with a small mock would need to mock chain_head + last_synced_block to span ≥100, which is fine but tedious).

**Option B: Const default + env var override (`PANGOLIN_SYNC_FAST_MODE_THRESHOLD`).** Mirrors 4.2 R-c (`PANGOLIN_INDEXER_IDLE_TIMEOUT_SECS`) + 3.4/3.5 env-var precedent. Pro: clamp-bound min/max defends against pathological values (e.g., min 10, max 100_000); test surface drives short thresholds. Con: marginal extra surface for an opinionated heuristic.

**Option C: Const default + env var override + per-vault stored override (additive column on `meta`).** Three layers. Pro: per-vault tuning if a heavy user wants always-fast on this vault. Con: triples the test surface for a heuristic; per-vault override is over-engineering for a UX nudge.

**Plan-gate recommends:** Option B — const default + env var override, with min/max clamp (e.g., 10..=100_000). Matches the 4.2 R-c idle-timeout precedent verbatim. The per-vault override (Option C) is unjustified for a heuristic; the user can always tune the global preference (Q-c) per vault.

**Kelvin's call.**

### Q-b · How is the unsynced count computed?

Master plan says "<100 unsynced **revisions**" — but a revision count is more expensive to compute than a block-distance proxy. Four shapes:

**Option A: Block-distance proxy: `chain_head - last_synced_block`.** One RPC call (`eth_blockNumber` via the existing `fetch_current_block_number`). Pro: fast; cheap; one RPC; matches what 4.1's `sync_from_chain` already needs to start anyway (it queries head before iterating chunks). Con: doesn't equal revision count if many blocks are empty for this vault; a vault that hasn't synced in 10_000 blocks but only has 3 revisions in that window would over-trigger the prompt. Master plan's "100 unsynced revisions" wording would be informally interpreted as "≥100 blocks of activity" rather than "≥100 events on chain for this vault".

**Option B: Single `eth_getLogs` query for revisions in `[last_synced_block+1, chain_head]` filtered by topic1 = vault_id; count the entries returned.** Accurate revision count. Con: pulls the full event data just to count; if the answer is "you have 5000 unsynced revisions, would you like fast-mode?", we've just done much of the fast-mode work to decide. Single RPC call but a potentially-fat response. The bounded chunk-size cap (`LOG_BLOCK_CHUNK = 9_000`) means a vault with a very stale checkpoint may even need multiple `eth_getLogs` requests just to count.

**Option C: Two-phase — block-distance proxy first; if it crosses a "maybe" zone (e.g., chain_head − last_synced > 1000), do a single `eth_getLogs` to refine.** Best of both. Con: more code; the "maybe" zone is itself a heuristic.

**Option D: Sample-based — fetch one `eth_getLogs` chunk (`LOG_BLOCK_CHUNK = 9_000` blocks worth) starting from `last_synced_block+1`; count revisions in that chunk; extrapolate linearly to the chain_head distance.** Avoids fat response; statistically reasonable. Con: extrapolation is noisy; over/under-trigger possible.

**Plan-gate recommends:** Option A (block-distance proxy). The 100-threshold itself is a UX heuristic; precision isn't load-bearing — both the user and the system are fine with "over-offer fast-mode" or "under-offer fast-mode" by a small margin. The "Spin up faster sync?" prompt has an Accept/Decline binary user response; the cost of a wrong offer is "user clicks Decline and falls through to slow-mode." Plan-gate also recommends documenting the framing shift in `master plan §5 row 4.4 + DECISIONS.md`: "**≥100 blocks since last sync** → offer fast-mode" rather than "≥100 unsynced revisions" — the implementation is honest about the proxy, and the user-facing prompt language ("you have a large gap since last sync — would you like a faster sync?") doesn't need the "100 revisions" claim.

**Kelvin's call.** (If Kelvin insists on Option B for accuracy, the implementation is one call to a new `pangolin_chain::chain_sync::count_unsynced_revisions(rpc_url, env, vault_id, from_block) -> u64` helper that wraps a single `eth_getLogs` + decodes only enough to count; ~30 LoC.)

### Q-c · User-preference storage — where + how?

The master plan row 4.4 says "User can disable the prompt and always use slow-mode." Three plausible storages:

**Option A: Additive nullable column on `meta` table — `sync_mode_preference TEXT`.** Per-vault; survives restart; survives backup/restore. Values: `NULL` (= "auto" = the default behavior); `"always_slow"` (= never offer fast-mode); optionally `"always_fast"` (= skip prompt, go straight to indexer if applicable). Migration: mirror 1.4's `migrate_session_idle_secs_column` exactly (additive `ALTER TABLE meta ADD COLUMN sync_mode_preference TEXT`; idempotent PRAGMA check). NO `format_version` bump (additive nullable column doctrine).

**Option B: Separate user-config file (per-machine, not per-vault).** A `~/.config/pangolin/preferences.json` or similar. Pro: cross-vault consistency; one preference. Con: NEW config-file surface; cross-platform path mess (Windows vs Linux vs macOS vs mobile); contradicts "each vault is independent" doctrine; lost on machine migration.

**Option C: In-memory only (no persistence).** User re-prompts every session. Defeats the "User can **disable** the prompt" master-plan wording.

**Plan-gate recommends:** Option A — vault-stored on `meta` table. Mirrors 1.4's `session_idle_secs` precedent exactly (additive nullable column; NULL = default; idempotent migration helper; sync read/write accessors in `meta.rs`). The vault-stored preference matches the "each vault is independent" doctrine (a user can have one vault on always-slow + another on auto) and survives backup/restore (the preference is part of the vault's user-customized state).

**Sub-question Q-c':** does the preference enum include `"always_fast"` (skip prompt, always spawn indexer)? Plan-gate recommends YES — three values covers the symmetric case; cheap test surface; matches the "user controls their own UX" doctrine. Master plan §5 row 4.4 only mentions "always use slow-mode" but the symmetric power-user case ("I always want the indexer; stop asking") is the natural completion.

**Kelvin's call.**

### Q-d · API surface — what does the selector return?

Three plausible shapes for the public entry:

**Option A: Pure read-only picker. `pub async fn select_sync_mode(vault: &Vault, rpc_url: &str, env: ChainEnv) -> Result<SyncMode, ChainError>` returning:**
```rust
pub enum SyncMode {
    Slow,
    OfferFast { unsynced_block_distance: u64 },
    AlwaysFast { unsynced_block_distance: u64 },  // if Q-c' = yes
}
```
The caller (host: CLI / FFI / Tauri shell) then either invokes `vault.sync_from_chain(...)` directly (Slow / OfferFast→decline) or — on OfferFast→accept or AlwaysFast — spawns the 4.2 `pangolin-indexer` and drains its output into `vault.ingest_pending_chain_revision`. Plan-gate strongly recommends this shape — matches the existing 3.3 `submit_signed_revision_v1` + 4.1 `sync_from_chain` pattern where the chain crate exposes primitives + the caller orchestrates.

**Option B: Opinionated convenience. `pub async fn auto_sync(vault: &mut Vault, rpc_url: &str, env: ChainEnv, prompt_fn: impl FnOnce() -> bool) -> Result<SyncReport, ChainError>` that picks + invokes internally; the `prompt_fn` callback surfaces the UX prompt to the caller.** Pro: caller doesn't see the decision. Con: callback shape forces a sync API onto an async path; the indexer-spawn logic lives inside this function (so it now depends on `pangolin-indexer`, expanding 4.4's dep surface).

**Option C: Both — `select_sync_mode` for hosts that render their own UI + `auto_sync` for callers who want the whole flow.** Doubled surface.

**Plan-gate recommends:** Option A — pure read-only picker. Crisp test surface (no callback mocking; no indexer-spawn-mocking). The caller orchestrates the prompt + the indexer spawn; 4.4 stays focused on picker logic. Matches the existing 4.1 R-e shape (the helper returns the decision; the caller acts).

**Sub-question Q-d':** module location. Plan-gate recommends NEW module `crates/pangolin-chain/src/sync_mode.rs` (logic is chain-side: it queries chain head + reads vault state); the public entry re-exports from `pangolin_chain::sync_mode::{select_sync_mode, SyncMode, SyncModePreference, UNSYNCED_THRESHOLD}`. The `&Vault` dep edge is fine — `pangolin-chain` already accepts `&Vault` / `&mut Vault` in `Vault::sync_from_chain`'s call graph (the `sync_from_chain` lives on `pangolin-store::Vault` but calls into `pangolin-chain`; the dep direction is `pangolin-store → pangolin-chain`, which is preserved). **Alternative:** put the selector on `pangolin-store::Vault` as `Vault::select_sync_mode(&self, rpc_url, env)` — the picker reads the vault preference + checkpoint + calls a chain helper for chain_head. This keeps the API discoverable on `Vault` (where callers already look). Builder picks based on which feels cleanest; plan-gate weakly prefers the `Vault` method shape.

## Adversarial framing — L-section (load-bearing risks)

> Light section — 4.4 is non-security-critical (the underlying sync paths carry the heavy crypto + on-disk discipline). The picker logic's only L-risks are:

### L-malicious-RPC-fakes-chain-head

**What goes wrong:** an attacker-controlled RPC returns `eth_blockNumber = small_value` (e.g., equal to `last_synced_block`), so the selector concludes "0 unsynced; pick Slow" when in reality the vault has 5000 unsynced revisions.

**Failure mode:** the user is over-served slow-mode (a slower-than-necessary sync). NO security failure (no forged revisions can land because the underlying sync paths — 4.1 + 4.2 — both run their own per-event verifier + chainid check + foreign-emitter rejection).

**Defense:** the selector is advisory; once the user invokes the chosen mode, the actual sync execution applies the load-bearing defenses (4.1 L3 chain-id check, L4 pinned-address check, L5 per-event signer recovery + signer-field cross-check). A spoofed selector decision picks the wrong UX path but doesn't make the actual sync less secure.

**Adversary leverage:** UX-degrade only (slow sync). Not a security concern.

### L-vault-state-staleness

**What goes wrong:** `last_synced_block_v1` could in principle be stale — e.g., a 4.1 sync that updated the checkpoint optimistically but failed mid-chunk (4.1's two-stage Pending→Finalized machinery already handles this case structurally, but the question is whether the selector's heuristic depends on a precise checkpoint).

**Failure mode:** the selector picks Slow when it should have picked OfferFast (or vice versa). UX-degrade only.

**Defense:** the selector uses `last_synced_block_v1` verbatim — 4.1's monotonic invariant (L12) guarantees the checkpoint never moves backward; 4.1's rollback machinery (`rollback_pending_revisions_in_range`) handles reorg-induced staleness at the slow-mode layer. The selector inherits whatever state 4.1 reports; if 4.1 says "I am at block X" and the chain is at X+50, the selector picks Slow (under threshold). If 4.1's reported X is stale by a small amount, the selector's decision is off by that small amount — well within the 100-threshold tolerance.

**Adversary leverage:** none.

### L-preference-flag-tamper

**What goes wrong:** a filesystem-tamperer flips `meta.sync_mode_preference` to `"always_slow"` (suppress fast-mode offers; deny the user the fast-sync UX) or to `"always_fast"` (force indexer spawn without per-session prompt — exposes the user to L-temp-file-leak from 4.2 + L-vault-id-disclosure from 4.2's RPC query, both of which are already-documented surfaces in 4.2/4.3 threat model).

**Failure mode:** UX-degrade or unwanted indexer spawn. NO new attack surface vs 4.2/4.3.

**Defense:** L2 — the user always retains the ability to flip the preference via `Vault::set_sync_mode_preference`. The preference is cleartext in `meta` by design (UX preference, not secret); filesystem-tamper resistance is out-of-scope per MVP-1/MVP-2 boundaries.

**Adversary leverage:** UX-degrade only. Same posture as `session_idle_secs` tamper would have.

```sql
ALTER TABLE meta ADD COLUMN sync_mode_preference TEXT;
```

Values: `NULL` (= `Auto`; the default for all existing vaults), `'always_slow'`, `'always_fast'` (if Q-c' = yes).

Migration helper `migrate_sync_mode_preference_column` mirrors `migrate_session_idle_secs_column` byte-for-byte in shape — PRAGMA `table_info(meta)` to check for existence; idempotent ALTER. Per the §18.7 additive-nullable-column doctrine: **NO `format_version` bump**. Legacy vaults open on new code without distinction; the column reads as NULL → `Auto`.
