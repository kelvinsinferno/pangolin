# Issue 3.4: Funder service — one-way ETH dispenser (off-chain HTTP)

> One-line scope: stand up the **first off-chain HTTP service in the Pangolin codebase**. A long-running daemon that (a) accepts a top-up request from a device, (b) verifies the user paid balance against D-018 `EntitlementRegistry` via a signed `Credit` attestation OR a chain-balance check (per Q-c), (c) signs and submits a `Redemption` attestation as the contract `REDEMPTION_AUTHORITY` to decrement the user on-chain balance, (d) sends ETH to the requesting device secp256k1 address from a funder-owned wallet, (e) returns a typed receipt the client crate can verify, all the while rate-limited per address, stateless beyond a small payment-ledger, and **strictly incapable of signing a revision or submitting a publish tx**. Per D-006: the funder never signs revisions, never submits transactions, never sees vault data, never holds custody of vault keys. 3.4 ALSO ships a fresh production-split-authority redeploy of the EntitlementRegistry (D-018 testnet stays as smoke-test surface; production gets D-019 — pending Q-d).
>
> **Status:** Plan-gate DRAFT 2026-05-15, awaiting Kelvin sign-off on Q-a..Q-g. Builder must not start until Kelvin lands resolutions in a "Resolved decisions" table that mirrors 2.1 / 2.2 / 2.3 / 3.1 / 3.2 / 3.3. L1..L12 + the L-section below are non-negotiables independent of how Q-a..Q-g resolve.
>
> **Security-critical: YES.** Master plan §16.3 enumerates "funder service" by name. Broader blast radius than 3.3: (a) redemption-attestation signer key handling bug = split-trust property collapses; (b) payment verification bug = treasury drain without payment; (c) rate-limiter bug = drain via valid repeated requests; (d) HTTP-surface bug (SSRF / admin endpoint) = full off-chain pwn; (e) response-signing bug = MITM tricks client into thinking ETH arrived. Same plan-gate cadence as 3.3 / 3.1 / 2.2.
>
> **Depends on:** 2.2 R-a..R-e (EIP-712 `Credit` + `Redemption` typehashes + strict-equality nonce + immutable authorities + `expiresAt`); 2.3 (D-018 deployed + verified at `0x08F8c394EB0c04ba0A4FBA1e64507b88F4b59D8d`; both authorities collapsed to pangolin-dev for smoke); 3.1 R-a..R-e (`secp256k1_signing.rs` typehash + domain primitives — funder reuses for `Redemption`); 3.2 (`derive_evm_wallet` / `EvmWallet` — but funder runs its OWN process-owned alloy `PrivateKeySigner`, NOT a `Vault`-derived wallet); 3.3 (EIP-1559 submission pattern from `chain_submit.rs` — funder reuses gas-cap / retry / receipt-mismatch defenses for `redeem` calls + ETH transfers); D-006 / D-008 / D-016. Current `main` tip `bd1e3ea`.

## Open questions for Kelvin (LOAD-BEARING — surface at TOP per §16)

### Q-a · HTTP framework

No HTTP server in the codebase. Four plausible choices:

**Option A: axum (tokio-native).** Tokio is a workspace dep (`=1.52.2`); axum 0.7.x slots on top. First-class tower-middleware for rate-limiting and timeouts; small surface; well-audited. Adds: `axum`, `tower`, `tower-http` (cors / compression / timeout / tracing layers). **Plan-gate recommends:** axum — best fit for the tokio + alloy stack.

**Option B: actix-web.** Mature, but its own runtime adapter shims tokio — runtime-mix surface. Ecosystem has moved away for new projects.

**Option C: hyper directly.** Smallest dep tree; forces hand-rolling routing + middleware + extractors. Trade: dep minimalism vs review-surface minimalism.

**Option D: tide / rocket / others.** Niche or stagnated. Not recommended.

**Kelvin call.**

### Q-b · Persistence shape

Master plan says "Stateless beyond payment ledger." Three shapes:

**Option A: Fully in-memory.** `HashMap<Address, RateLimitState>` + bounded LRU seen-set. **Pro:** zero on-disk state. **Con:** restart wipes rate-limiter (attacker triggers a restart, resets hourly counter); no audit trail.

**Option B: SQLite-backed payment ledger only.** Uses existing `rusqlite = "=0.39.0"` workspace pin (no new dep). **Pro:** durable audit trail; per-attestation replay guard survives restart; rate-limiter survives restart. **Con:** new operational surface.

**Option C: Hybrid — in-memory rate limiter + SQLite ledger.** Rate limiter in memory for performance, rebuilds on restart from ledger; ledger durable. **Plan-gate recommends:** Option C.

**Kelvin call.**

### Q-c · Payment verification protocol

Trust boundary between Kelvin billing system and the funder. Three models:

**Option A: Signed-`Credit`-attestation flow.** User pays processor off-band; processor signs a `Credit` attestation (EIP-712 per 2.2 R-a). User submits to funder along with device wallet address; funder verifies sig against `PAYMENT_AUTHORITY`, constructs + submits `redeem(...)`, on success transfers ETH. **Pro:** trust boundary cleanly enumerated; payment processor needn online at top-up time. **Con:** `expiresAt` bounds the window; the `userId → deviceAddress` mapping is the user responsibility — see L-userId-deviceAddress-binding for the defense (Q-g).

**Option B: Chain-balance check.** Funder reads `balance(userId)` from D-018 directly. **Con:** needs off-chain `userId → deviceAddress` mapping; either re-invents the Credit flow OR maintains a server-side mapping table. Does not decrement the on-chain balance unless funder ALSO submits `redeem` — back to Option A.

**Option C: Hybrid — signed credit for prod; chain-balance for testnet smoke.** Two code paths gated by `ChainEnv`.

**Plan-gate recommends:** Option A. Testnet can synthesize Credit attestations with pangolin-dev (already `PAYMENT_AUTHORITY` on D-018).

**Kelvin call.**

### Q-d · Production EntitlementRegistry redeploy strategy

D-018 has collapsed authorities. Three paths:

**Option A: Fresh redeploy with real split keys — D-019.** Two NEW Foundry keystores: `pangolin-payment-authority` + `pangolin-funder-service`. Deploy fresh `EntitlementRegistry` with both real authority addresses. Funder points at D-019; D-018 untouched.

**Option B: Keep D-018 for testnet only; defer production redeploy.** Funder tested against D-018 with collapsed authorities; production later (MVP-3+). **Con:** architectural property never validated against real key separation before mainnet.

**Option C: Hybrid — 3.4 ships split-key-capable code AND fresh TESTNET redeploy with real split keys (D-019). Mainnet deferred.**

**Plan-gate recommends:** Option C.

**Kelvin call.**

### Q-e · Rate-limiting strategy

On-chain `balance[userId]` bounds per-user financial loss; rate-limit is load-bearing for operational stability (gas burn per redeem; RPC quota). Four shapes:

**Option A: Fixed window per address.** Boundary effect: burst 2N across a boundary.

**Option B: Sliding window per address.** Eliminates boundary effect.

**Option C: Token bucket per address.** `tower::limit::RateLimitLayer` shape.

**Option D: Layered — per-address bucket AND global hourly cap.** **Plan-gate recommends:** Option D, testnet values: bucket burst 3 / refill 1-per-15-min; global cap 100/hour.

**Kelvin call** on shape + values.

### Q-f · Funder wallet management

Funder holds a secp256k1 key that signs Redemption attestations + EIP-1559 envelopes. Where?

**Option A: File-based Foundry keystore.** Same pattern as pangolin-dev. Passphrase at startup OR env var. Alloy supports via `alloy-signer-local` `keystore` feature. **Con:** rooted host = compromised.

**Option B: Env-var raw 32-byte key.** Known footgun. NOT recommended.

**Option C: HSM (KMS / dedicated).** Highest assurance; operational complexity. **Defer to mainnet.**

**Option D: Hybrid — file-based for testnet; HSM for mainnet (deferred). Signer behind `trait FunderSigner: Send + Sync` so future HSM plugs in.**

**Plan-gate recommends:** Option A for testnet (D testnet half). HSM scaffolded via trait but not implemented.

**Kelvin call.**

### Q-g · Should the Credit typehash bind the deviceAddress?

The 2.2 R-a typehash today: `Credit(bytes32 userId, uint256 amount, uint64 nonce, uint16 schemaVersion, uint64 expiresAt)` — no `deviceAddress`. An attacker with a leaked Credit attestation can submit it with their OWN device address as the ETH destination, redirecting the user paid credit.

**Option A: Add `address deviceAddress` to the Credit typehash.** Requires re-shaping 2.2 contract sketch + D-019 ships with new typehash. Permanent on-chain defense.

**Option B: Client-signed device-binding (no contract change).** Every funder request includes a separate signature from the user device wallet (3.1 signer) over `(userId, deviceAddress)`. **Cost:** extra sig per request + extra verify step. **Pro:** no contract change.

**Plan-gate recommends:** Option B for 3.4 (keep 2.2 typehash unchanged); Option A is the right longer-term answer but is a 2.2-revisit.

**Kelvin call.**

## Adversarial framing — L-section (load-bearing risks)

> Per §16, every plan-gate doc for a security-critical issue surfaces the load-bearing risks an adversary could leverage from each design choice. The funder is the broadest off-chain surface MVP-2 ships.

### L-funder-impersonation

**What goes wrong:** an attacker stands up a fake funder service at a URL similar to the real one (`funder.pangolin.example` vs `funder.pangolin.examp1e`). Users or the `pangolin-funder-client` crate are tricked into sending Credit attestations + payment details. Worst case: the fake service collects valid Credit attestations + submits them to the REAL funder later — but the on-chain `nonce` ratchet prevents that. Lesser case: the fake service denies users their ETH (DoS).

**Failure mode:** users cannot get gas top-ups; the legit funder is bypassed.

**Defense:** the funder service URL is pinned in `pangolin-funder-client` config (compile-time per release build OR runtime-configurable via signed config — `pangolin-funder-client` Q, not 3.4). The funder signs its OWN responses with its REDEMPTION_AUTHORITY secp256k1 key — a SEPARATE EIP-712 typed-data `FunderResponse` envelope over `{userId, deviceAddress, txHash, dispensedWei, timestamp}`. The client verifies that signature against the registry `REDEMPTION_AUTHORITY()` view at runtime.

**Adversary leverage:** limited to delay-or-deny — cannot reuse a Credit (nonce ratchet) and cannot forge a `FunderResponse` signature.

### L-credit-attestation-replay

**What goes wrong:** a leaked, stolen, or screen-scraped Credit attestation is submitted to the funder multiple times.

**Failure mode (mitigated):** the contract `nonce[userId]` mapping increments on every successful `redeem` (per 2.2 strict-equality — `redeem` ALSO bumps the nonce, not just `credit`); replayed Credit-then-Redeem lands as `ErrNonceTooLow` on second `redeem`. But: the funder still spent gas submitting the second `redeem` that reverted, AND may have sent ETH BEFORE confirming the receipt (depending on ordering — see L-payment-order).

**Defense:** (a) the funder tracks per-attestation seen-set in the payment ledger (Q-b) — a Credit-attestation hash already processed returns HTTP 409 without touching the chain; (b) order: redeem first, ETH-transfer second (see L-payment-order); (c) the rate-limiter (Q-e) bounds the attempt rate.

**Adversary leverage:** essentially none against the funder treasury; minor RPC capacity DoS (mitigated).

### L-funder-wallet-key-leak

**What goes wrong:** the funder REDEMPTION_AUTHORITY secp256k1 scalar leaks. Attacker can (i) sign arbitrary `Redemption` attestations against ANY userId — deflate every user balance to zero; (ii) sign ETH-transfer txs from the funder hot wallet — drain it; (iii) sign `FunderResponse` messages impersonating the funder.

**Failure mode (worst-case):** treasury drain + every user entitlement forced to zero. The contract has NO admin recovery — per 2.2 R-d, immutable authorities; fix is deploying a NEW registry + telling everyone (off-band). Days of downtime.

**Defense:**
- (a) **Key custody:** Q-f Option A on testnet; Option C (HSM) on mainnet (deferred).
- (b) **Hot-wallet balance ceiling (L5):** total compromise drains only bounded hot-wallet balance.
- (c) **Monitoring (§18.10):** dashboards for "ETH dispensed/hour" + "balance remaining"; anomalous drain fires an alert.
- (d) **Per-tx hard cap (mirror of 3.3 L6):** outbound ETH-transfer hard cap on `value` (e.g., 0.01 ETH per tx). A signed tx sending more than the cap is rejected by the funder signer wrapper BEFORE submission. Defense against a code bug + slows the drain if both happen at once.

**Adversary leverage:** treasury drain + per-user deflate are the worst losses. Recovery: deploy v2 registry (2.2 R-d) + re-key everyone. Days of downtime; bounded financial loss per L5.

### L-DOS-eth-drain (rate-limit bypass)

**What goes wrong:** a user with a legit paid balance repeatedly requests top-ups via valid Credit attestations within the rate-limit window. Each request causes the funder to (i) submit a `redeem` tx (gas burn for the funder signer), (ii) send a small ETH transfer (gas + value). The user is draining their OWN balance but each drain costs the funder operational capacity.

**Defense:**
- (a) Per-address rate-limit (Q-e).
- (b) Global rate-limit (Q-e).
- (c) Per-transfer cap (mirror of L-funder-wallet-key-leak (d)): 0.01 ETH per tx.
- (d) Per-credit-attestation-amount cap: Credit attestations with `amount > MAX_CREDIT_PER_ATTESTATION` rejected. Defense against a leaked PAYMENT_AUTHORITY key forging a 2^256-credit attestation.

**Adversary leverage:** the contract `balance` bounds per-user loss; the rate-limiter + caps bound funder operational loss per attacker.

### L-funder-service-MITM

**What goes wrong:** TLS termination is missing or misconfigured. Passive MITM observes Credit attestations + device addresses in transit (privacy leak); active MITM intercepts + injects fake responses (see L-funder-impersonation).

**Failure mode (passive):** the userId-to-deviceAddress mapping leaks to a network attacker — degrades 2.2 R-b opaque-bytes32 privacy posture.

**Defense:**
- (a) HTTPS-only. Two paths: (i) embedded TLS via `axum-server` + `rustls` (rustls is NOT on the crypto allowlist today — adding it requires Q-? deferral); (ii) behind a reverse proxy (nginx / caddy / Cloudflare) terminating TLS; funder serves plain HTTP on a localhost-bound socket. **Plan-gate recommends:** Option (ii) — operational standard for backend services; avoids putting TLS code in the Pangolin crypto tree.
- (b) HSTS + secure cookie attributes at the proxy layer (operator job, documented in 18.5 runbook).
- (c) `pangolin-funder-client` URL config rejects `http://` schemes (compile-time or runtime assertion).

**Adversary leverage:** mitigated to "operator has to put a TLS proxy in front" — documented in the runbook.

### L-payment-order (race between redeem and ETH-transfer)

**What goes wrong:** the funder flow is two sequential on-chain operations: (i) `redeem(...)` to decrement on-chain balance; (ii) ETH-transfer to device address. If (i) succeeds but (ii) fails (funder host crashes between; RPC timeout on (ii); hot wallet runs out of ETH between (i) and (ii)), the user is DEBITED on-chain but has NOT received gas. Paid for ETH they did not get.

**Defense:**
- (a) **State machine in payment ledger (Q-b):** every top-up recorded with explicit lifecycle: `received → pre_redeem → redeem_submitted → redeem_mined → eth_transfer_submitted → eth_transfer_mined → done` plus failure states. On restart, funder scans for `redeem_mined → !eth_transfer_*` rows and resumes the ETH transfer.
- (b) **Receipt-await discipline:** funder waits for `redeem` receipt BEFORE constructing ETH-transfer (3.3 R-e shape verbatim — 1-block conf). If receipt-await times out, row stays `redeem_submitted` and next restart-scan resumes (rather than retrying the redeem, which would double-debit).
- (c) **Idempotency in response:** response includes BOTH tx hashes (redeem + eth-transfer).
- (d) **Manual reconciliation path:** §18.5 runbook documents the operator process — a row stuck in `redeem_mined → eth_transfer_*` for >1h triggers an alert; operator manually completes.

**Alternative order (rejected):** ETH-first, redeem-second. **Pro:** user always gets gas. **Con:** the funder transferred ETH against a balance not decremented; if redeem then fails (unlikely under strict-equality but possible) the user got "free" ETH. Plan-gate recommends redeem-first — user-paid-for-nothing failure is recoverable via reconciliation; user-got-free-ETH failure is unbounded loss surface.

### L-replay-after-revert (mirror of 3.3 analog)

**What goes wrong:** the funder `redeem` tx mines but reverts on-chain (e.g., `ErrUnauthorizedSigner` from a Q-d misconfig; `ErrAttestationExpired` from clock skew; `ErrInsufficientBalance` from a multi-region race). The retry loop, if naive, re-submits and burns more gas.

**Defense:** mirror of 3.3 R-c retry taxonomy — contract reverts are FATAL, never retried. Retry only on (a) funder OWN-tx nonce-collision (independent of the contract userId nonce), and (b) RPC-transient. The funder `chain_submit`-shaped helper imports + uses 3.3 `ChainError` taxonomy verbatim.

**Adversary leverage:** only attacker-controllable revert path is `ErrAttestationExpired` (user delays after Credit). Funder rejects expired BEFORE submission with HTTP 400 — no gas burn. Other reverts are funder misconfig; mitigated at deploy.

### L-server-clock-skew

**What goes wrong:** funder host system clock drifts. Credit attestation `expiresAt` is compared against `block.timestamp` on chain (per 2.2 R-e) AND against `now()` on the funder side (defense-in-depth). If funder clock is faster than chain time, funder rejects valid attestations as expired. If slower, funder submits attestations chain rejects (gas burn + L-replay-after-revert).

**Defense:** (a) funder maintains a clock-skew tolerance window (e.g., ±60s) between system clock and most recent block timestamp; on cold start + every N redeems, fetch latest block timestamp + compute skew. (b) Skew exceeding tolerance logs OPERATOR-level alert (§18.10) and continues operating. (c) NTP / chrony at OS level is operator responsibility (runbook).

### L-userId-deviceAddress-binding

**What goes wrong:** funder flow receives `userId` (in the Credit, immutable) and `deviceAddress` (in the request body, mutable). An attacker submits a valid Credit under userId X with deviceAddress Y where Y is attacker-controlled. Funder dispenses ETH to Y; user X balance gets decremented. Attacker effectively redirects user X paid credit.

**Failure mode:** user X loses paid credit; attacker gets free ETH.

**How does the attacker get a valid Credit for X?** (a) X attestation leaked / stolen — nonce ratchet defeats double-spend but the FIRST use can go to any deviceAddress; (b) intercepted in transit (L-funder-service-MITM); (c) user voluntarily shared (Discord help post).

**Defense:** see Q-g. Plan-gate recommends Option B — client-signed device-binding proof. With this defense the attacker needs BOTH the leaked Credit AND a signature from the user device wallet — which requires compromising the device, at which point the attacker has bigger problems (vault unlock).

### L-secrets-in-logs

**What goes wrong:** the funder logs request bodies (full Credit attestations including secp256k1 signatures). Log files persist; aggregation services index them; backups carry them. Eventually a logfile leaks.

**Failure mode:** privacy leak (the mapping); modest replay window (if logs leak before the nonce has ratcheted past the attestation).

**Defense:** L12 verbatim — DEBUG-only with sampling; INFO never logs userId, deviceAddress, signature bytes. CI invariant: grep INFO-level macro args for the request body fields.

### L-cors-and-csrf

**What goes wrong:** funder HTTP surface allows cross-origin from arbitrary origins. A user browser at `evil.example` runs JS that triggers a request to the funder with the user cookies/auth. Funder dispenses ETH to attacker-controlled.

**Failure mode (mitigated):** the funder does not USE cookies or session auth — every request is self-contained (Credit + deviceAddress + device-binding sig). CSRF in the classical sense does not apply. BUT: a future maintainer adding cookie auth without thinking turns this into a real surface.

**Defense:** (a) CORS whitelist `Access-Control-Allow-Origin: <pangolin-client-app-only>` (not `*`); (b) `Access-Control-Allow-Credentials: false`; (c) funder NEVER sets cookies; (d) future maintainers adding auth must update this row. `tower_http::cors::CorsLayer` handles mechanics.

### L-funder-as-payment-correlator

**What goes wrong:** the funder LEARNS the `userId → deviceAddress` mapping (L-userId-deviceAddress-binding). Indefinite retention turns the funder into the per-user payment-history index the 2.2 R-b opaque-bytes32 model was supposed to prevent.

**Defense:** L12 + §18.5 "minimal logging, retention window, address-vs-payment correlation policy" — bounded retention (e.g., 30 days), no account linkage. 3.4 ships code that DEFAULTS safe; operational policy is the 18.5 job.

**Adversary leverage:** funder operator is the trust anchor; if Pangolin-the-org becomes a compelled-disclosure target, logs are subpoenaable. Defense is at the policy + retention layer.
