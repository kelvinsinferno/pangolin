## Decisions locked (post plan-gate)

| Q | Decision | Rationale |
|---|---|---|
| Q1 — MVP-1 capability set | **Just `Full`.** Stored as `INTEGER` (0 = `Full`); the enum/repr is designed to grow (read-only / limited / browser-extension-as-device are later-phase concepts). | MVP-1 has one device class that can do everything; nothing in MVP-1 distinguishes capabilities. Keep the schema column + enum so MVP-2/3 can add variants without a migration. |
| Q2 — `last_sync_at` semantics in a chain-less MVP-1 | **Dormant column.** `last_sync_at` is always `NULL`/`None` in MVP-1; MVP-2's chain sync populates it. Same doctrine as the existing `chain_anchor_*` columns on revisions — present for the schema, dormant until the feature that fills it lands. | MVP-1 has zero chain code → there is no "sync" in the MVP-2 sense. Repurposing it to "last opened/edited" would have to be un-done when real sync arrives. A documented-dormant column is cleaner. |
| Q3 — Device revocation / removal in MVP-1 scope? | **Out of scope. The trust list is add-only.** A device registers itself on first unlock; there is no MVP-1 path to *remove*/revoke a device. Keep P2's `revoked_at` column as the MVP-2/3 hook. | Revoking a device requires an authority-rotation mechanism (re-wrap the VDK under a new authority, atomically revoke old devices) — that *is* social recovery, which is MVP-3 (issue `6.5` Finalize + authority rotation). Add-only is the correct MVP-1 boundary. |
| Q4 — Does the `DeviceKey` sign anything in MVP-1? | **No. Generate + store the keypair only; no signing wired.** The `DeviceKey`'s on-chain revision-signing + gas-payer role is MVP-2 (issue `2.1` Revision Log v1 / `3.1` Signed-revision format / `3.2` Device wallet generation). | MVP-1 is zero-chain-code. Generating + persisting the keypair now (encrypted, in the `.pvf`) gives MVP-2 a hook; wiring signing now would be dead code with no verifier. |
| Q5 — Does `device_set_label` need a presence proof? | **No. Active (unlocked, non-expired) session only.** Renaming a device is a non-secret metadata edit, not a §5.4 reveal-class action. (The whitepaper's "approve devices" language refers to *enrolling/revoking* a device — both out of MVP-1 scope per Q3 — not renaming.) | The presence-escalation gate (1.4) is for revealing/exporting secrets and for the authority-mutating ops. A label rename mutates only a human-readable string; gating it would be friction without security value. |
| Q6 — Backfill / validate legacy `originating_device` values? | **No. Accept existing PoC `.pvf` values as-is; don't backfill, don't reject.** Going forward (post-1.5), `originating_device` is verifying-key-derived and register-on-unlock; pre-1.5 revisions keep their throwaway-random value. | The P2 `devices` table is a *dead stub* — `Vault::create`/`open` mint a per-handle random `device_id` that is never persisted, and nothing reads/writes the `devices` table — so every existing revision's `originating_device` is already a throwaway value. The trust list gates nothing in MVP-1, so a stale value is harmless. Backfilling would require guessing which (non-existent) device authored old revisions; rejecting would brick PoC vaults. Accept-as-is is the only sound choice. |
| Q7 — `device list` / `device label` CLI subcommand in 1.5 or deferred? | **Deferred to the standing CLI-V1-wiring follow-up.** 1.5 ships the device-identity model, the trust list, the schema/migration, and the FFI entries (`device_list` / `device_current` / `device_set_label`) — but no CLI subcommand. The CLI device subcommands land later alongside the `account` / `reveal` V1 wiring as one cohesive CLI-V1 issue. | Consistent with how 1.2/1.3/1.4 left their CLI surfaces (FFI-complete, CLI deferred). Keeps 1.5 tight; avoids a piecemeal CLI. |

## Goal

After this issue closes, every `.pvf` knows which device it is running
on. On the first successful unlock on a new vault file the vault
registers a device entry: it generates a fresh Ed25519 `DeviceKey`
(P1's type), derives a stable `device_id` from that key's public bytes,
inserts a `devices` row (`device_id`, the user-set label, `registered_at`,
the capability flags, the dormant `last_sync_at`, the device's public
verifying-key bytes), and stores the device's *secret* seed in an
encrypted-at-rest slot in the vault file. Subsequent unlocks of the same
`.pvf` re-load that device — they do **not** register a second device.
The trust list is the set of `devices` rows; in MVP-1 (zero chain code)
it is the local hook the MVP-2 on-chain authority registry will
canonicalise — it is populated, queryable, and stamped onto every
revision's `originating_device`, but it gates nothing destructive yet
(no revocation; the `frozen_pending_resolve` machinery is unrelated and
untouched). The `Vault` exposes `device_current()`, `device_list()`, and
`device_set_label(...)`; `pangolin-core` re-exports the new types;
`pangolin-ffi` gains `device_current` / `device_list` / `device_set_label`
entry points (a 1.1-surface additive amendment) returning a `DeviceInfo`
record (`DeviceId` + label + `registered_at` + `last_sync_at: Option` +
`capabilities` + `is_current` + `schema_version`). Existing PoC `.pvf`
vaults still open, unlock, search, and reveal; on first open under the
new code their `devices` table is migrated to the new shape (nullable
columns added with defaults) and a device entry is registered for the
opening device — legacy `originating_device` values already on their
revisions are accepted as-is (per Q6's answer). HIGH-1 (`cargo tree -p
pangolin-crypto | grep -ci serde == 0`), Q3 of 1.1 (`cargo tree -p
pangolin-core | grep -ci uniffi == 0`), Q3-bonus (`cargo tree -p
pangolin-store | grep -ci uniffi == 0`) all stay 0; `forbid(unsafe_code)`
survives on every crate except `pangolin-ffi`; zero chain code is
activated; the 1.3 `:memory:` FTS5 lifecycle and the 1.4 session state
machine are preserved untouched; AGPL-3.0-or-later SPDX header on every
new `.rs` file.

### 1. The device-identity model (`pangolin-store/src/device.rs` — new module)

A new `device` module in `pangolin-store` (re-exported through
`pangolin-store/src/lib.rs` and then `pangolin-core/src/lib.rs`, the same
re-export pattern 1.2/1.4 use — no physical relocation, per the standing
posture). It owns:

- `DeviceIdentity` (the in-memory view of a `devices` row): `device_id:
  DeviceId`, `label: String`, `registered_at: i64` (unix ms — matches
  `created_at`/`set_at_ms` elsewhere), `last_sync_at: Option<i64>`,
  `capabilities: DeviceCapabilities`, `public_key: VerifyingKey` (the
  device's Ed25519 verifying key bytes, non-secret — stored so MVP-2 can
  signature-verify this device's revisions without re-deriving), and a
  derived `is_current: bool` filled by the read path (true for the row
  matching the open handle's `device_id`).
- `DeviceCapabilities` — the capability-flags type. **Recommendation
  (Q1):** an enum with one variant in MVP-1 — `Full` — designed to grow
  (`ReadOnly`, etc., are later-phase concepts: browser-extension-as-a-
  limited-device, shared-vault read-only seats, etc.). Stored as a small
  integer column (`capabilities INTEGER NOT NULL DEFAULT 0`, `0 = Full`)
  so adding variants is a value addition, not a schema change. (If Kelvin
  wants a richer set now — e.g. `Full | ReadOnly` — the enum gains the
  variant; the storage is the same integer.)
- `device_id` derivation: **Recommendation** — `device_id = DeviceId(the
  32 bytes of the device key's Ed25519 verifying key)`. This is exactly
  what `revision.rs`'s `DeviceId` doc says it will become ("P3 will
  replace with the verifying-key bytes of the device's signing
  keypair"), it is stable across unlocks (re-derived from the stored
  seed), it is uniformly random (Ed25519 compressed public bytes), and
  it is the on-chain identity MVP-2 will register. (Alternative: a
  separate random UUID with the pubkey stored alongside — rejected: an
  extra column, an extra identifier to keep coherent, and it loses the
  "the device id IS the verifying key" property the revision-id doc
  already promised.)
- Label validation: same discipline as 1.2's `display_name` — non-empty
  after trim, ≤ 256 chars, NFC-normalised (reuse 1.2's validator
  helper); the default label on register-on-unlock is a generated
  placeholder (e.g. `"Device 1"` or `format!("Device {}", n)`) since the
  CLI has no UI to prompt for one (see Q7).

### 2. The trust list

The trust list **is** the `devices` table. In MVP-1:

- **What it does:** holds one row per device that has ever opened+unlocked
  this `.pvf` (register-on-unlock — see below); is the lookup target for
  `device_current` / `device_list`; supplies the `originating_device`
  stamped onto every new revision (the open handle's own `device_id`,
  which is now a real `devices`-row id rather than a per-session random).
- **What it does NOT do (MVP-1):** gate edits, reveals, publishes, or
  conflict resolution. Whether a revision's `originating_device` is "in
  the trust list" is **informational** in MVP-1 — the on-chain authority
  registry that would *enforce* "only enrolled devices may publish"
  lands MVP-2 (`Authority registry` row + `2.1 Signed-revision format`).
  1.5 leaves the data and the read API; it does not introduce a
  trust-check that would reject anything (that would risk bricking PoC
  vaults whose revisions carry orphan random `originating_device`s — see
  Q6).
- **Add path:** implicit, on first unlock on a new vault file. There is
  no explicit `device_register(label)` entry point in MVP-1 (the device
  is the machine running the binary; registration is "I unlocked this
  file here"). After unlock, if the stored device-key slot is empty, the
  vault generates a `DeviceKey`, derives `device_id`, inserts the
  `devices` row + writes the encrypted seed slot, all in one transaction.
  If the slot is populated, it loads the key, re-derives `device_id`, and
  sets the handle's `device_id` to it (so revisions this session stamp
  the real id).
- **Remove path (revocation):** **Recommendation (Q3):** out of scope
  for MVP-1. Device revocation is tied to the recovery flow (master plan
  `6.5 Finalize + authority rotation` → "old devices revoked
  atomically") and the on-chain authority registry — both MVP-2/MVP-3.
  MVP-1's trust list is **add-only**: register-on-unlock, never remove.
  The `revoked_at` column the P2 stub already has is kept (the hook for
  MVP-2/MVP-3); MVP-1 never writes it; `device_list` returns all rows
  (or filters `revoked_at IS NULL` — a no-op in MVP-1 since it is always
  NULL). (If Kelvin wants a *local-only* "forget this device" now —
  e.g. delete a stale row — say so; it would be a `device_forget(id)`
  entry point that errors on `id == current`. The plan-gate recommends
  against it: it has no real teeth without the on-chain registry and
  invites footguns — a forgotten device's old revisions still exist.)

### 3. `last_sync` semantics in a chain-less MVP-1

**Recommendation (Q2):** a **dormant column** — `last_sync_at INTEGER`,
nullable, always NULL in MVP-1, populated by MVP-2's chain-sync code
when this device last published-or-pulled through the contract. This is
the same doctrine the schema already uses for the `chain_anchor_*`
columns on `revisions` (nullable until P7 fills them) and the P7
`sync_state` table — the schema carries the shape; the field is dormant
until the chain code lands. The FFI `DeviceInfo` carries it as
`last_sync_at: Option<UnixTimestamp>` (always `None` in MVP-1; a host UI
renders "never synced" / hides the field). Rejected alternatives:
(b) repurpose it to "last opened" / "last edited" local activity — this
overloads the field's eventual meaning and a host UI would show a
misleading "synced 2 min ago" for a vault that has never touched a
chain; (c) leave it out of the schema entirely and add it in MVP-2 —
rejected because the master plan row explicitly lists "last sync" as a
1.5 deliverable, and adding the column now (dormant) is free and avoids
a later migration. The plan documents the dormancy in the schema DDL
comment + the `DeviceIdentity` doc + a test (see Test plan).

### 4. The `DeviceKey` generation + storage

- On first unlock on a new vault file (the register path above), generate
  `DeviceKey::generate()`. Derive `device_id` from its `verifying_key()`.
- **Where the secret lives:** an encrypted-at-rest slot in the `.pvf`.
  **Recommendation:** a new single-row `device_key` table
  (`id INTEGER PRIMARY KEY CHECK (id = 0)`, `enc_seed BLOB NOT NULL`,
  `enc_nonce BLOB NOT NULL`, `schema_version INTEGER NOT NULL`) holding
  the device-key seed **AEAD-sealed under the VDK** (XChaCha20-Poly1305,
  via the existing `blob`/`AeadKey` surface) with the device's
  `device_id` bound into the AAD (anti-transplant, same pattern as
  revision AAD). Sealing under the VDK is the right call — unlike the
  `pending_merges.device_secret` BLOB (which the P9 plan documents as an
  *ephemeral* merge-signing seed stored *un-AEAD-sealed* because
  at-rest exposure of the `.pvf` already compromises the VDK and the
  marginal exposure of an ephemeral key is bounded) — the *device key is
  long-lived* and is the MVP-2 on-chain signing identity / gas wallet,
  so it gets the AEAD layer that the `no_plaintext_on_disk` proptest
  already enforces for every other secret. (Alternative: an encrypted
  column on the `meta` row — rejected: `meta` is the
  pre-unlock-readable header (KDF params, salt, wrapped VDK); putting a
  VDK-sealed blob there is fine cryptographically but muddies the
  "everything in `meta` is non-secret structural metadata" doctrine. A
  dedicated table is cleaner and the schema-version slot per §18.7 is
  natural.) The seed crosses the `pangolin-crypto` → `pangolin-store`
  boundary via `DeviceKey::secret_seed_bytes() ->
  Zeroizing<[u8; 32]>` (the existing P9 accessor) on the way to the
  seal, and back via `DeviceKey::from_seed([u8; 32])` after the open —
  **the serialisation lives entirely in `pangolin-store`** (HIGH-1
  preserved; `pangolin-crypto` gains no serde and no new dep).
- **What the `DeviceKey` signs in MVP-1:** **Recommendation (Q4):**
  nothing. It is generated + stored as the hook for MVP-2's
  `2.1 Signed-revision format` (where it signs revisions for on-chain
  verification) and D-006's gas-payer role. MVP-1 has zero chain code;
  wiring local revision-blob signing now would be speculative work that
  MVP-2's signed-revision format issue will design properly (canonical
  body, signature placement, verification path). The plan-gate
  recommends: store it, do not wire signing. (If Kelvin wants a
  local integrity signature now — e.g. the device key signs each
  revision's `enc_payload` and the signature is stored in a new
  `revisions.local_sig` column — say so; the plan-gate considers it
  premature: MVP-1's at-rest integrity already comes from the AEAD tag,
  and a local signature with no verification consumer is dead weight.)

### 5. The FFI changes (`pangolin-ffi/src/device.rs` — new module)

A new `device` module in `pangolin-ffi` (added to `lib.rs`'s `pub mod`
list + re-exports), parallel to `identity` / `reveal`. Shapes:

- `DeviceCapabilities` — a `uniffi::Enum` mirroring the store enum
  (`Full` in MVP-1; grows later). Carries no `schema_version` (it is a
  closed enum, not a user-data record — same treatment as
  `SessionState`-adjacent enums).
- `DeviceInfo` — a `uniffi::Record`:
  `schema_version: u16` (set to `1`, per the 1.1/1.2/1.4 convention),
  `id: DeviceId` (the existing 1.1 record), `label: String`,
  `registered_at: UnixTimestamp`, `last_sync_at: Option<UnixTimestamp>`
  (always `None` in MVP-1 — Q2), `capabilities: DeviceCapabilities`,
  `is_current: bool`, `public_key: Vec<u8>` (the 32 Ed25519 verifying
  bytes — non-secret; lets a host render a fingerprint and is what
  MVP-2 will register on-chain).
- Entry points:
  - `device_current(handle: Arc<VaultHandle>) -> Result<DeviceInfo,
    FfiError>` — the device this `Vault` is running on. Errors
    `FfiError::Session` if no vault installed; `FfiError::Store` /
    `Validation` on a storage/crypto failure. (Does the device row exist
    on a *locked* vault? Yes — register-on-unlock means after the first
    unlock the row is persisted; a fresh `vault_open` on an existing
    vault can read `devices` without unlocking, so `device_current` works
    on a locked-but-previously-registered vault. On a brand-new vault
    that has only ever been `vault_create`d-and-`vault_open`ed but never
    `vault_unlock`ed, there is no device row yet → `device_current`
    returns a typed "no device registered; unlock first" error. Surface
    in the doc.)
  - `device_list(handle: Arc<VaultHandle>) -> Result<Vec<DeviceInfo>,
    FfiError>` — all `devices` rows (one in MVP-1). Same locked-vault
    behaviour as `device_current`.
  - `device_set_label(handle: Arc<VaultHandle>, id: DeviceId, label:
    String) -> Result<(), FfiError>` — rename a device. Validates the
    label (non-empty, ≤ 256, NFC); persists. **Recommendation (Q5):**
    requires an **unlocked vault only** — not a fresh presence proof.
    Renaming a device is a low-risk metadata edit (it touches no secret;
    it is not on the session spec §5.4 reveal-class list, which is
    `reveal password / export vault / modify recovery / approve devices /
    extend long sessions` — *device labels* are not "approve devices",
    which means *enrolling/revoking* a device, an MVP-2/3 operation). So:
    `check_session_freshness()` + `require_active()`, same as
    `account_update`'s display-name edit. (Note: 1.4 carries a
    `device_locked()` hook on `Vault` — unrelated; that is "the OS locked,
    expire the session", not "this device's row". Untouched.)
- These are an additive 1.1-surface amendment (the 1.1 freeze declared
  `DeviceId` but no `Device`/`DeviceInfo` shape, no `device_*`
  entries) — same posture as 1.2's Q1 and 1.4's Q4; nothing external
  binds the 1.1 surface yet. `docs/architecture/ffi-surface.md` is
  updated to add `DeviceInfo`, `DeviceCapabilities`, and the three
  `device_*` entries.
- **cabi:** the cbindgen-emitted C ABI in `cabi.rs` gets the
  corresponding `extern "C"` wrappers for the three entries (same
  pattern as the existing `account_*` / `reveal_*` cabi wrappers).

### 6. The schema / migration (`pangolin-store/src/schema.rs`)

- The `devices` table gains four columns (the P2 stub has `device_id,
  label, added_at, revoked_at`; 1.5 adds): `capabilities INTEGER NOT
  NULL DEFAULT 0` (0 = `Full`), `last_sync_at INTEGER` (nullable,
  dormant — Q2), `public_key BLOB` (the 32 Ed25519 verifying bytes;
  nullable for legacy rows but written for every row 1.5 creates),
  `schema_version INTEGER NOT NULL DEFAULT 1`. The `added_at` column is
  reused as `registered_at` (rename in the `DeviceIdentity` view; the SQL
  column name stays `added_at` to avoid a column rename — SQLite's
  `ALTER TABLE … RENAME COLUMN` works but a no-op rename is needless
  churn; document the mapping).
- A new `device_key` single-row table (see §4): `id INTEGER PRIMARY KEY
  CHECK (id = 0)`, `enc_seed BLOB NOT NULL`, `enc_nonce BLOB NOT NULL`,
  `schema_version INTEGER NOT NULL`.
- **Migration** — a `migrate_devices_columns` helper in `schema.rs`
  (same shape as `migrate_session_idle_secs_column` / `migrate_frozen_
  pending_resolve_column`): `PRAGMA table_info(devices)`, for each of the
  four new columns that is absent run `ALTER TABLE devices ADD COLUMN
  …`. Idempotent. The `device_key` table is `CREATE TABLE IF NOT
  EXISTS` in `SCHEMA_DDL` (picked up trivially on the fresh path) plus a
  belt-and-braces `migrate_device_key_table` helper (same pattern as
  `migrate_pending_merges_table`). **No `format_version` bump** — these
  are additive tables/columns, exactly the doctrine the existing
  migrations follow. Called from `apply_pragmas_and_schema` alongside
  the existing three migrations.
- **Register-on-unlock** lives in `Vault::unlock` (`pangolin-store/src/vault.rs`),
  after the VDK is unwrapped (the seal needs the VDK): if the
  `device_key` table is empty, generate + seal + insert (`devices` row +
  `device_key` row, one transaction) and set `self.device_id`; else
  load + decrypt + re-derive + set `self.device_id`. The per-handle
  random `device_id` minted in `Vault::create`/`open` (vault.rs 372/434)
  becomes a *pre-unlock placeholder* only — it is overwritten by
  `unlock`; document that no revision can be written before `unlock`
  (true: `account_add`/`account_update` call `require_active()`), so a
  revision never stamps the placeholder.
- **PoC vaults:** a P0–P11 `.pvf` opened under the new code gets the four
  new `devices` columns (with defaults) and the `device_key` table via
  the migrations; on its next `unlock` it registers a device entry. Its
  existing revisions keep their orphan random `originating_device`
  values — **accepted as-is** (Q6 recommendation): backfilling them to
  the new device's id would be a lie (they were authored by a different
  "device" — a prior session's random id) and the trust list gates
  nothing in MVP-1, so the orphan ids are harmless metadata. Document
  this in the migration comment + a test.
- **Schema-version hook for §18.7 (1.6's policy):** the new tables/columns
  carry `schema_version` slots; the DDL comments note that 1.6 locks the
  read-old-versions / reject-unknown-future-versions policy. 1.5 leaves
  the slot; it does not design the policy.

### 7. CLI (`apps/cli`) — optional, see Q7

**Recommendation (Q7):** add a thin `pangolin-cli device` subcommand
(`device list` + `device label <id> <new-label>`), mirroring the nested
`account <verb>` / `vault <verb>` shape (P11A/P11B precedent). It is a
small, well-bounded addition that makes the trust list user-visible (the
master plan lists "Devices" in the §7.1.6 desktop sidebar; the CLI is the
MVP-1 surface). The first `device list` after a vault create+unlock shows
the one registered device; `device label` renames it. If Kelvin would
rather keep 1.5 strictly to the core + FFI layers and let a later issue
add the CLI verb (the `device list` / `device label` UX has no real teeth
until there is more than one device — MVP-2's sync), say so; the plan-gate
leans toward shipping it now since it is the only way a human exercises
the feature in MVP-1.

## Success criteria

1. A freshly-`Vault::create`d-then-`Vault::open`ed-then-`Vault::unlock`ed
   `.pvf` has exactly one `devices` row, whose `device_id` equals the
   bytes of the registered `DeviceKey`'s verifying key, marked
   `is_current`.
2. `Vault::open` + `Vault::unlock` again on the same `.pvf` does **not**
   add a second `devices` row; the loaded `device_id` equals the one
   from step 1; `self.device_id` (the revision-stamping id) equals it.
3. `Vault::device_current()` returns the registered device; `device_list()`
   returns a one-element vector containing it with `is_current == true`.
4. `Vault::device_set_label(id, "Kelvin's MacBook")` validates (rejects
   empty / over-256 / produces NFC-normalised storage), persists, and the
   new label survives `Vault::close` + reopen + `device_current()`.
   `device_set_label` on a *locked* vault errors `SessionExpired` /
   `NotUnlocked` (Q5: no presence proof, but an active session is
   required).
5. The `DeviceKey` seed is stored AEAD-sealed under the VDK in the
   `device_key` table; it is **not** present in plaintext anywhere on
   disk (the `no_plaintext_on_disk` proptest is extended to include the
   device-key seed bytes among the markers it scans for in the raw
   `.pvf`); on a subsequent `unlock` the key is decrypted, re-derives the
   same `device_id`, and is a working `DeviceKey` (verifying key matches).
6. After register-on-unlock, every new revision (`account_add` /
   `account_update`) stamps `originating_device == device_current().device_id`
   (a real `devices`-row id, not a per-session random); `account_history`'s
   `RevisionMeta.device_id` and the FFI `PasswordHistoryEntry.originating_device`
   reflect it.
7. `DeviceCapabilities` round-trips through storage; the default for a
   register-on-unlock entry is `Full`; the stored integer for `Full` is
   `0`.
8. `last_sync_at` is `None` on a freshly-registered device and after any
   number of `account_add` / `account_update` / `lock` / `unlock` cycles
   (the dormant-column shape — Q2); a test asserts this with a comment
   that MVP-2's chain-sync code populates it.
9. An existing PoC `.pvf` (one written by P2..P11 with the stub `devices`
   table and revisions carrying orphan random `originating_device`
   values) still opens, unlocks (two-proof), searches, and reveals; its
   `devices` table is migrated to the new shape; a device entry is
   registered on first unlock; its legacy `originating_device` values are
   left as-is (Q6).
10. FFI: `device_current` / `device_list` / `device_set_label` work
    end-to-end through `VaultHandle` (using `VaultHandle::from_vault` with
    an unlocked vault, the 1.2/1.4 test pattern); locked-vault /
    no-handle error paths return `FfiError::Session`; `DeviceInfo` carries
    `schema_version == 1`, `last_sync_at == None`, `capabilities == Full`,
    `is_current == true` for the single device.
11. `cargo tree -p pangolin-crypto | grep -ci serde == 0`,
    `cargo tree -p pangolin-core | grep -ci uniffi == 0`,
    `cargo tree -p pangolin-store | grep -ci uniffi == 0` all hold;
    `forbid(unsafe_code)` is present on `pangolin-store`, `pangolin-core`,
    `pangolin-crypto`, `pangolin-totp`, `pangolin-kdbx` (not
    `pangolin-ffi`); no chain code is activated; every new `.rs` file
    carries the `// SPDX-License-Identifier: AGPL-3.0-or-later` header.
12. The 1.3 `:memory:` FTS5 search-index lifecycle (build on unlock, tear
    down on lock/expiry, sync on add/update/delete) and the 1.4 session
    state machine (`Locked`/`PendingAuthorization`/`Active`/`Expired`
    transitions, idle/absolute timers, reveal-class escalation, presence
    freshness, prompt timeout/dedup) are byte-for-byte unchanged — no
    1.5 change touches `search.rs` or the session region of `vault.rs`
    beyond the register-on-unlock insertion point in `unlock`.

## Failure modes considered

- **The `DeviceKey` seed leaks to disk in plaintext.** Mitigation: it is
  AEAD-sealed under the VDK in the `device_key` table (not stored
  un-sealed like the *ephemeral* `pending_merges.device_secret`, which
  the P9 plan justifies separately); the `no_plaintext_on_disk` proptest
  is extended to scan the raw `.pvf` for the seed bytes; the `DeviceKey`
  type already zeroizes on drop and redacts `Debug` (P1 invariants);
  the seed never crosses `pangolin-crypto`'s boundary except as a
  `Zeroizing<[u8;32]>` headed straight into the seal / out of the open,
  with all serialisation in `pangolin-store`.
- **A second device row is registered on every unlock** (the slot-check
  is wrong). Mitigation: the register branch is keyed on "is the
  `device_key` table empty?" — a single-row table with `CHECK (id = 0)`,
  so "empty" is unambiguous; criterion 2's test re-opens and re-unlocks
  and asserts the row count stays 1.
- **A revision is written before `unlock` and stamps the pre-unlock
  placeholder `device_id`.** Mitigation: `account_add`/`account_update`
  both call `require_active()` (a revision can only be written in the
  `Active` state, which is only reachable via `unlock`, which runs the
  register/load step *before* it transitions to `Active`). A test adds
  an account immediately after unlock and asserts the stamped id is the
  real one.
- **A PoC `.pvf` bricks** on the migration. Mitigation: the migration is
  `ALTER TABLE … ADD COLUMN` (additive, nullable / with defaults) +
  `CREATE TABLE IF NOT EXISTS`, the same pattern the four existing
  migrations use; no `format_version` bump; criterion 9's test opens a
  legacy fixture and asserts it opens/unlocks/searches/reveals;
  `apply_pragmas_and_schema` is idempotent (criterion 9b).
- **A trust-check introduced in 1.5 rejects a legitimate edit / reveal**
  (e.g. "this revision's `originating_device` isn't a known device").
  Mitigation: 1.5 introduces **no** such check — the trust list is
  informational in MVP-1; the enforcement point is MVP-2's on-chain
  authority registry. Criterion 9 explicitly asserts a PoC vault with
  orphan `originating_device` values still works.
- **HIGH-1 regression** — a serde path sneaks into `pangolin-crypto`.
  Mitigation: `deny.toml` bans `serde`/`serde_derive` from that crate's
  graph (the build fails if added); the device-key serialisation is
  entirely in `pangolin-store`; criterion 11's `cargo tree` greps are
  in CI.
- **`uniffi` leaks into `pangolin-core` / `pangolin-store`** via the new
  device types. Mitigation: the `uniffi::Record` / `uniffi::Enum`
  derives are on the **`pangolin-ffi`** `DeviceInfo` / `DeviceCapabilities`,
  not the `pangolin-store` `DeviceIdentity` / `DeviceCapabilities`; the
  bridge fn maps between them (same split 1.2's `AccountSnapshot` /
  `AccountIdentitySummary` uses); criterion 11's `cargo tree` greps.
- **`device_set_label` is treated as reveal-class and demands a presence
  proof**, breaking the "low-friction metadata edit" expectation — or,
  conversely, **a genuinely high-risk op (enrol/revoke a device) is
  added under a too-weak gate**. Mitigation: 1.5's only device write is
  *rename* (Q5 recommendation: active-session only, no presence proof —
  it is not on the §5.4 list); *enrol/revoke* (the §5.4 "approve
  devices" item) is explicitly out of scope (MVP-2/3, with the on-chain
  registry).
- **The `device_id`-from-verifying-key derivation collides with the
  per-session random ids on existing revisions** in a way that confuses
  `account_history`. Mitigation: it can't — the verifying-key bytes are
  a fresh Ed25519 public key (uniformly random; collision probability
  with any prior random 32-byte id is negligible); legacy revisions keep
  their old ids; no row is rewritten.
- **`Debug`-printing a `DeviceIdentity` / `DeviceInfo` leaks something.**
  Mitigation: every field is non-secret (`device_id`, label, timestamps,
  capability flags, the *public* verifying key) — there is no secret on
  these types (the secret is the seed in the `device_key` table, which
  has its own redacting wrapper via `DeviceKey`'s `Debug`). The `Debug`
  impls render bytes as hex (matching `DeviceId`'s existing `Debug`).

## Open questions for Kelvin

1. **MVP-1 capability set (Q1).** §F lists "capability flags". For MVP-1,
   what capabilities exist?
   - **(a) Just `Full`** (the default; can do everything), with the enum
     designed to grow (`ReadOnly`, browser-extension-limited, shared
     read-only seats, etc. are later-phase). — **Recommended.** MVP-1 is
     a single-device CLI vault; there is no UX for limited devices and no
     enforcement point for them yet. Keep the type minimal; grow it when
     MVP-2/4 needs it.
   - (b) `Full | ReadOnly` now. — More forward-looking but `ReadOnly` has
     no consumer in MVP-1 (nothing checks it), so it would be a dead
     flag.
2. **`last_sync` semantics in a chain-less MVP-1 (Q2).** The master plan
   row says "last sync", but MVP-1 has zero chain code.
   - **(a) Dormant column** — `last_sync_at` exists in the schema +
     `DeviceInfo`, always `None`/NULL in MVP-1, populated by MVP-2's
     chain-sync code (same doctrine as the `chain_anchor_*` columns on
     revisions). — **Recommended.** Keeps the field's eventual meaning
     honest; the master plan row is satisfied by the column existing;
     no later migration.
   - (b) Repurpose it to "last opened" / "last edited" local activity,
     renamed to "last synced" when chain sync arrives. — Overloads the
     field; a host UI would show a misleading "synced 2 min ago" for a
     vault that has never touched a chain.
3. **Device revocation / removal in MVP-1 scope? (Q3).**
   - **(a) Out of scope — register-on-unlock only; the trust list is
     add-only in MVP-1.** Device revocation is tied to the recovery flow
     (`6.5 Finalize + authority rotation` → "old devices revoked
     atomically") and the on-chain authority registry, both MVP-2/3.
     The P2-stub `revoked_at` column is kept as the hook; MVP-1 never
     writes it. — **Recommended.**
   - (b) Add a local-only `device_forget(id)` now (delete a stale row;
     errors on `id == current`). — The plan-gate recommends against it:
     it has no real teeth without the on-chain registry (a forgotten
     device's old revisions still exist) and invites footguns.
4. **Does the `DeviceKey` sign anything in MVP-1? (Q4).**
   - **(a) No — generate + store only; signing is MVP-2's
     `2.1 Signed-revision format` (revision signing for on-chain
     verification) + D-006's gas-payer role.** — **Recommended.** MVP-1
     has zero chain code; wiring local signing now is speculative work
     MVP-2 will design properly; MVP-1's at-rest integrity already comes
     from the AEAD tag.
   - (b) Have the device key sign each revision's `enc_payload` as a
     local integrity check (new `revisions.local_sig` column). — Premature:
     a signature with no verification consumer is dead weight.
5. **Does `device_set_label` need a presence proof? (Q5).**
   - **(a) No — an active (unlocked) session is enough.** Renaming a
     device is a low-risk metadata edit; it touches no secret; it is
     **not** on the session spec §5.4 reveal-class list (which is
     `reveal password / export vault / modify recovery / approve devices
     / extend long sessions` — "approve devices" means *enrol/revoke*, an
     MVP-2/3 op, not "rename"). Same gate as `account_update`'s
     display-name edit. — **Recommended.**
   - (b) Require a fresh presence proof, on the "anything touching the
     device/authority layer is escalated" principle. — Over-strict for a
     pure label edit; would add friction with no security gain.
6. **Backfill / validate legacy `originating_device` values? (Q6).** PoC
   `.pvf` vaults have revisions stamped with per-session random
   `originating_device` ids (the current per-handle random). When 1.5
   makes that field a real `devices`-row reference:
   - **(a) Accept legacy values as-is; do not backfill; do not
     reject.** The orphan ids are harmless metadata in MVP-1 (the trust
     list gates nothing); backfilling them to the new device's id would
     be a lie (they were authored under a different "device" — a prior
     session's random id). New revisions stamp the real id. — **Recommended.**
   - (b) Backfill all legacy `originating_device` values to the newly-
     registered device's id on first open. — Rewrites history with a
     fiction; also touches the append-only `revisions` table, which the
     codebase treats as never-`UPDATE`d except the `chain_anchor_*`
     columns.
7. **`device list` / `device label` CLI subcommand in 1.5, or deferred?
   (Q7).**
   - **(a) Ship a thin `pangolin-cli device` subcommand now** (`device
     list` + `device label <id> <label>`), mirroring the nested `account
     <verb>` / `vault <verb>` shape. It is the only way a human exercises
     the feature in MVP-1, and the master plan lists "Devices" in the
     §7.1.6 desktop sidebar. — **Recommended (mild).**
   - (b) Defer the CLI verb to a later issue; 1.5 ships the core + FFI
     layers only. — Defensible: `device list` / `device label` UX has no
     real teeth until there is more than one device (MVP-2's sync).

  (`Authority registry` row). MVP-1's trust list is purely local and
  informational.
- **Device-key signing of revisions** — the `DeviceKey` signing each
  revision for on-chain verification (canonical body, signature
  placement, verification path). MVP-2 issue `2.1 Signed-revision
  format`.
- **The `DeviceKey`-as-gas-wallet role (D-006)** — the device key paying
  gas for revision-publishing transactions. MVP-2's chain adapter +
  funder service.
- **Device enrolment** — adding a *new* device to a vault by
  authority-signed enrollment record (the multi-device pairing flow).
  MVP-2/3.
- **Device revocation via recovery** — `6.5 Finalize + authority rotation`
  rotating the authority and atomically revoking old devices. MVP-3
  (social recovery).
- **The funder-service device-wallet role** — the gas-abstraction relay
  that submits a device's signed revisions on chain. MVP-2.
- **Limited-capability devices in practice** — read-only devices,
  browser-extension-as-a-limited-device, shared read-only seats. The
  `DeviceCapabilities` enum is designed to grow; MVP-1 ships only `Full`.
- **`last_sync_at` population** — the column exists in MVP-1 but is
  dormant; MVP-2's chain-sync code writes it.
- **Multi-device conflict / merge UX around devices** — already covered
  by P9/1.6's conflict-resolution work; 1.5 only adds the device-identity
  data those flows reference.
- **§18.7 schema-versioning *policy* text** — 1.5 carries the
  `schema_version` slots on the new records/tables; the policy (read old
  versions, reject unknown future versions) is locked by 1.6.
