## Open questions for Kelvin (LOAD-BEARING — surface at TOP per §16)

### Q-a · PoC vault re-signing strategy — biggest decision in 3.1

Per 2.1 R-d (verbatim): *"Existing PoC `SignedRevision` blobs in `.pvf` files are NOT replayable on v1; they must be re-signed under the v1 domain. **That re-signing happens in MVP-2 issue 3.1** (the client-side signed-revision format issue)."* This is the explicit hand-off. Concrete options:

**Option A: Re-sign every locally-pending PoC `.pvf` revision under v1 at unlock time (lineage-preserving).** The first `Vault::unlock` after a 3.1 upgrade walks `revisions WHERE chain_tx_hash IS NULL`, recomputes each row's payload-keccak-binding canonical body, signs with the Active session's `EvmWallet` per the v1 EIP-712 typed-data shape, and **stores the new 65-byte sig in a new `revisions.signature_v1 BLOB` column** (additive migration). The Ed25519 `signature` column stays for read-back (no destructive write). The `device_id` column gets ALSO populated with the secp256k1 EVM address (left-padded `[u8; 32]`) so v1's `RevisionPublished` event-level `deviceId` round-trips on `pull_since`. Cost: O(unpublished-revisions) per first post-3.1 unlock; signatures themselves are microseconds. Preserves lineage: chain sees the same `parent_revision` graph (revision IDs are content-derived; the unsigned bytes were always v0/v1-domain-independent).

**Option B: Leave v0 PoC `.pvf` revisions orphaned; v1 starts at a fresh per-vault sequence.** No re-signing migration. Drop the `revisions.signature` column on the migration (or just stop reading it). Every vault that wants to "go on-chain under v1" publishes a fresh genesis revision under v1's self-bootstrap — the v1 contract sees a new `(vaultId)` with no prior registered devices, registers the device on first publish per R-b. **Downside:** the chain history is split; off-chain replay-from-chain for a vault that had v0 PoC activity sees v1 events starting mid-stream, not from genesis. **Upside:** the simplest possible 3.1 surface — no migration code, no new column, no audit risk in the re-sign loop.

**Option C: Hybrid — keep the v0 column, write a new v1 column on every NEW revision built after 3.1 ships, and leave previously-unpublished v0 revisions orphaned.** The v0 column becomes read-only legacy state; new revisions get both columns set during a transition window; later migration drops the v0 column. **This is the path 1.6 / 1.10 used for the §18.7 schema-version ladder** when payload formats evolved. Trade-off: extra column surface for a transition window; PoC users with un-published locally-pending revisions still lose lineage (same downside as Option B, narrower in scope).

**Recommendation (mild):** Option A if Kelvin values PoC-to-v1 lineage continuity (and PoC users exist who would be upset to see their work orphaned). Option B if Kelvin treats the PoC as a closed chapter and the simplest 3.1 surface wins. Option C if a transition window is desired without committing to a forever-dual-column shape. **Kelvin's call.** This decides whether 3.1 ships a migration helper at all + whether the `revisions` schema bumps.

### Q-b · v0 signing-path retention — does `signing.rs` keep `build_signed_revision` / `verify_signed_revision` / `SIGNED_REVISION_DOMAIN`?

The existing `pangolin-chain::signing` module is Ed25519-shaped (per P7 + the module docstring) and produces a 64-byte detached signature over the `pangolin-chain-signed-revision-v0` domain. Three plausible shapes after 3.1:

**Option A: Keep both modules side-by-side, no deletion.** `signing.rs` keeps `SIGNED_REVISION_DOMAIN = b"pangolin-chain-signed-revision-v0"` + `build_signed_revision` (Ed25519, 64-byte sig) for **read-back of legacy v0 `.pvf` records**. A new module `pangolin-chain::secp256k1_signing` per 2.1 R-a Implementation impact (3) lands with v1's EIP-712-encoded typed-data hash + `SIGNED_REVISION_DOMAIN_V1 = b"pangolin-chain-signed-revision-v1"` + `build_signed_revision_v1` (secp256k1, 65-byte sig). Production callers (`pangolin-store::publish_one` / `apps/cli::sync::publish`) switch to v1; `verify_signed_revision` keeps working on legacy rows. **Trade-off:** the surface area is wider; auditors must reason about two modules; the `SignedRevision` struct in `pangolin-chain::types` may need to gain an enum discriminant or a new sibling `SignedRevisionV1` type.

**Option B: Delete v0 entirely; one module only (v1).** `signing.rs` is rewritten in place: rename `SIGNED_REVISION_DOMAIN` from `-v0` → `-v1`, swap `Signature` (Ed25519 64-byte) for `[u8; 65]` (secp256k1 r/s/v), rename `SignedRevision.signature` field, delete `verify_signed_revision`'s Ed25519 path, replace `build_signed_revision` to use `EvmWallet::signer().sign_hash` over the EIP-712 digest. **Downside:** legacy v0 PoC `.pvf` rows become unreadable for signature-verification purposes (the bytes are still there; just nothing decodes them). Combined with Q-a Option B this is internally consistent ("v0 is gone"); combined with Q-a Option A this would be inconsistent (we re-sign legacy revisions but also delete the verifier that proves the re-signing was correct against the originals — though admittedly the re-sign code can verify under Ed25519 inline before discarding the v0 sig, no separate module needed).

**Option C: New module for v1; v0 module becomes read-only.** Keep `signing.rs` but mark `build_signed_revision` `#[deprecated]` and remove its callers; keep `verify_signed_revision` as a public read-back primitive. New `secp256k1_signing.rs` ships v1. Same as Option A but with explicit deprecation. Honest about which path is the active one.

**Recommendation (mild):** Option A (keep both) **if** Q-a = Option A or C (we need v0 verify for the re-sign loop or for read-back). Option B (delete v0) **if** Q-a = Option B (clean break, drop the burden). **Kelvin's call.**

### Q-c · Contract address sourcing — where does 3.1 read the deployed D-017 address from?

The existing `BaseSepoliaAdapter::new_with_device_key` (P7) takes a `deployment_path: &Path` and loads `contracts/deployments/base-sepolia.json` at construction time (workspace-relative; chained walk-up via `find_deployment_file`). The 2.3 deploy pipeline writes the canonical addresses there (D-014, D-015, D-017, D-018). For 3.1's NEW signing path the question is narrower: **the EIP-712 `verifyingContract` field binds the contract address into every signature** — so the signing primitive itself needs to know which address it is signing for, before any RPC happens. Three plausible models:

**Option A: Compile-time constant.** A const in `pangolin-chain::secp256k1_signing` that pins `0x179362Ad7fb7dA664312aEFDdaa53431eb748E42` + `chainId = 84_532`. Simplest. Locks the binary to Base Sepolia D-017. **Downside:** any future redeploy (a v1.1 contract; an L1 deploy) needs a rebuild + a new binary. **Upside:** auditor reads one line of source to verify the binding; no JSON parse path inside the security-critical signing primitive; no runtime way to redirect signing to a wrong address.

**Option B: Runtime config read from `contracts/deployments/<env>.json` at boot, cached for the process lifetime.** Same shape as the existing `BaseSepoliaAdapter` deployment loader (P7) — same `Deployment::load` function probably reused. The signing primitive takes the address as a parameter; callers thread it from the loaded deployment file. **Downside:** the signing primitive's correctness depends on a file-read that happens before signing; a misconfigured path or a tampered file silently redirects signing. **Mitigation:** the loader already does EIP-55 checksum + `chain_id` + `deployed_runtime_keccak256` cross-checks at adapter-construction (P7 MED-1 / MED-2); if 3.1 reuses that loader, signing happens *after* those checks pass, so a wrong address is rejected before any signature is built.

**Option C: Per-vault meta column.** Store the v1 contract address inside the `.pvf`'s `vault_meta` table at vault-creation time; signing reads from there. **Downside:** every vault carries a chain-binding state; users would need to "migrate to v1.1" by writing a new meta row. Adds vault-level state for what is effectively a workspace-level deploy concern. **Plan-gate recommends against** unless Kelvin has a concrete reason (e.g. per-vault chain selection — explicitly out of scope per master plan §0 cardinal principle 4 + the master-plan §5 MVP-2 architecture; one chain per build).

**Recommendation (mild):** Option B (reuse the existing `Deployment::load`-based runtime config; thread the address as a parameter into the new signing module's typed-data digest path). Inherits 2.1 / 2.3's existing deployment-file discipline. The signing-primitive purity argument for Option A is real but the operational cost of compile-time-bound chain addresses is higher (every contract redeploy is a forced binary cut). **Kelvin's call.**
### Q-d · Verifier path — does 3.1 also implement the *Rust* verifier (recover signer from a v1 signed revision)?

The v1 contract's `_recover` runs inside the EVM during `publishRevision`; that covers chain-side verification. But off-chain consumers (slow-mode chain sync in 4.1; ephemeral indexer in 4.2; the eventual `pangolin-cli sync pull` cross-check; the `MockChainAdapter` in `pangolin-chain::mock`) need a *Rust* `verify_signed_revision_v1(&signed) -> Result<Address, …>` that recomputes the EIP-712 digest + runs `k256::ecdsa::recover_from_prehash` + returns the recovered signer. Two options:

**Option A: Ship the Rust verifier in 3.1.** Parallels v0's `verify_signed_revision` per the existing module. Used by: (i) the `MockChainAdapter::publish` eager-verify path (P7 MED-4 — the mock verifies signatures so a regression in the builder is caught loudly in unit tests); (ii) the new typehash-cross-check test (test against the contract's `domainSeparator()` + `hashRevision()` view fns by either hard-coded constants or a forge-replayed artifact); (iii) 4.1 / 4.2 when they land. **Cost:** ~50 LoC of pure-Rust k256 (`recover_from_prehash` + the same EIP-712 digest function). Same domain-separated `SignatureInvalid` sentinel as v0 — no failure-cause leak.

**Option B: Defer the Rust verifier to issue 4.1 (slow-mode chain sync), which is the first downstream consumer that actually needs it.** 3.1 ships only the signer; the verifier follows when there is a chain-sync consumer to use it. **Trade-off:** the `MockChainAdapter::publish` eager-verify path becomes "we trust the new signer to be correct" rather than "we round-trip sign+recover in the same test". Reduces hermetic test coverage for 3.1. The `domainSeparator()` / `hashRevision()` cross-check tests can still happen (they do not need a Rust verifier — they need a way to call the contract's view fns, which is what `cast call` or an alloy `eth_call` does).

**Recommendation (mild):** Option A (ship in 3.1). The Rust verifier is ~50 LoC and unlocks the hermetic sign+recover round-trip test, which is the cheapest+strongest way to catch typehash/domain/v-byte drift. Option B is defensible but front-loads risk onto 4.1. **Kelvin's call.**

### Q-e · Test surface depth — match-the-contract (network) or match-our-spec (hermetic)?

The v1 contract's `_recover` is the ground truth. There are two structurally different ways to test that 3.1's builder produces signatures it accepts:

**Option A: Cross-check against the live deployed D-017 contract via `cast call` or alloy `eth_call`.** Build a `SignedRevision` in Rust; call the contract's `hashRevision(vaultId, ..., encPayload)` view fn over RPC; assert the returned digest matches the Rust-computed EIP-712 digest. Then call `publishRevision(...)` for a fresh vault (self-bootstrap) and assert success. **Strong:** catches every form of contract-Rust drift, including silent compiler-version digest divergences. **Weak:** requires an RPC URL + a funded keystore (D-006's `pangolin-dev` wallet); CI cannot run this hermetically; fragile against Base Sepolia outages; flaky in CI by definition. Same gate the existing `pangolin-chain` integration tests use (`--features integration-tests`, opt-in, skipped in default CI).

**Option B: Hermetic — sign in Rust, parse + recover in Rust via k256, assert byte equality on r/s/v + the typehash + the domain separator.** The Rust verifier from Q-d (Option A) is the round-trip. Cross-check the contract's typehash by pinning the literal byte string in a Rust const + asserting `keccak256(literal) == REVISION_TYPEHASH_CONSTANT`. Cross-check `EIP712_DOMAIN_TYPEHASH` similarly. Cross-check the runtime keccak of the contract bytecode (already in `base-sepolia.json` via P7 MED-2). **Strong:** hermetic; deterministic; runs on every CI invocation; catches typehash / domain / v-byte / canonical-s drift. **Weak:** does not cross-check against the actual deployed bytecode at runtime — a future contract redeploy whose source matches but whose compiler artifact diverges would not be caught by hermetic tests alone. (The P7 MED-2 runtime-keccak check IS the bytecode cross-check, but it lives in `BaseSepoliaAdapter`, not in the signing primitive's tests.)

**Option C: Both. Hermetic by default; gated network cross-check under `--features integration-tests` for manual smoke before merge.** Combines the two; pays the maintenance cost of both. The integration test exercises a real publish against D-017 for one self-bootstrap revision; the hermetic tests run on every CI invocation.

**Recommendation (mild):** Option C (both). Hermetic for everyday CI safety; network gated for manual smoke before merge. The integration test runs on Kelvin's box (the only place with the `pangolin-dev` keystore today); the rest of 3.1 still ships if Sepolia is down. Matches the existing `pangolin-chain` test gate pattern. **Kelvin's call.**

## Decisions locked (independent of Q-a..Q-e)

| # | Decision | Rationale |
|---|---|---|
| **L1** | The signed-revision signature is **65 bytes**, `r ‖ s ‖ v` (32 ‖ 32 ‖ 1), with **canonical-s** (`s ≤ secp256k1 curve order n / 2`) and **`v ∈ {27,28}`** (NOT EIP-155 chain-id-embedded `v`; the chain ID is bound into the EIP-712 domain separator). | `RevisionLogV1.sol::_recover` (`contracts/src/RevisionLogV1.sol:405-437`) verbatim: rejects `signature.length != 65`; rejects `v != 27 && v != 28`; rejects `s > 0x7FFFFFFFFFFFFFFFFFFFFFFFFFFFFFFF5D576E7357A4501DDFE92F46681B20A0` (secp256k1 n/2). Any drift from this format produces a signature that the contract's `ecrecover` will (a) reject outright (length / v-byte / canonical-s) → `ErrInvalidSignature`, OR (b) recover to address(0) → `ErrInvalidSignature`. `k256::ecdsa::SigningKey::sign_prehash_recoverable` produces a `RecoveryId` (0 or 1) + a low-s `Signature` by default — 3.1 wraps both with the `+ 27` offset on `v` to land in the contract's accepted range. |
| **L2** | The EIP-712 v4 domain separator binds **`name = "Pangolin RevisionLog"`**, **`version = "1"`**, **`chainId = block.chainid` (84_532 on Base Sepolia D-017)**, **`verifyingContract = address(this)` (= 0x179362Ad7fb7dA664312aEFDdaa53431eb748E42)**. The Rust client MUST construct the same domain bytewise — keccak256 over `abi.encode(EIP712Domain typehash, keccak(name), keccak(version), chainId, verifyingContract)`. | `RevisionLogV1.sol:202-212` constructor. ANY drift (typo in name, wrong chain id, wrong contract address) silently produces signatures that recover to a different signer; the contract's `_recover` returns a non-registered address (or address(0) if the sig encoding is also wrong); revert with `ErrInvalidSignature` or `ErrSignerNotRegistered`. The L2 enforcement test pins the literal `"Pangolin RevisionLog"` + `"1"` strings as Rust consts + computes the keccak digests at compile time (or in a const-fn-equivalent test) so a future contributor who mis-types either string fails the test immediately. |
| **L3** | The `Revision` struct typehash is exactly `keccak256("Revision(bytes32 vaultId,bytes32 accountId,bytes32 parentRevision,bytes32 deviceId,uint16 schemaVersion,bytes32 encPayloadHash)")`. The Rust client MUST pin this byte-for-byte (a const `REVISION_TYPEHASH_V1: [u8; 32]` derived at test time from the literal string) and the test MUST assert it matches what the contract's `domainSeparator()` would imply OR what a fresh `keccak256` over the literal string produces. | `RevisionLogV1.sol:132-134`. Drift is silent — the encoded-typed-data digest is computed from the typehash, so a mismatch produces a signature that recovers to a wrong signer (same failure mode as L2). The widening of `schemaVersion` from `uint8` (v0) → `uint16` (v1) is a load-bearing detail: the v0 Rust struct field is `u8`; v1's must be `u16`. Rust-side: serialize as big-endian 32-byte word (EIP-712 encodes `uint16` left-padded to a `bytes32`). |
| **L4** | The Rust signing-domain prefix bumps from `"pangolin-chain-signed-revision-v0"` to **`"pangolin-chain-signed-revision-v1"`** in (whatever module ends up holding v1's signing primitive — Q-b decides which). **Clean break per 2.1 R-d:** an attacker who steals a v0 Ed25519 signature CANNOT replay it as a v1 secp256k1 signature; the primitives differ AND the canonical-hash domain prefix differs. NOTE: the v1 contract itself does NOT verify a domain prefix — it verifies the EIP-712 typed-data digest. The Rust-side `-v1` domain string is for *off-chain* domain separation (e.g., the `pangolin-store` ingest-side replay check; the chain-sync indexer's cross-check). The contract-side replay defense is EIP-712's domain separator (L2). | 2.1 R-d (verbatim): *"The Rust signing-domain prefix bumps from `pangolin-chain-signed-revision-v0` to `pangolin-chain-signed-revision-v1` in `crates/pangolin-chain/src/signing.rs::SIGNED_REVISION_DOMAIN` — clean break with v0's bytes; an attacker who steals a v0 signature CANNOT replay it as a v1 signature (the digest differs in the domain prefix byte)."* The off-chain layer's domain separation matters for the case where a future v1.1 also uses EIP-712 but with a different struct shape — pinning the prefix in the Rust const documents the contract Rust-side too. |

Concretely: 3.1's `secp256k1_signing::build_signed_revision_v1(...)` takes a `&EvmWallet` parameter; the only place a caller obtains an `&EvmWallet` is via `Vault::evm_wallet() -> Result<&EvmWallet, StoreError>` (3.2 R-b), which calls `require_active()` and errors `StoreError::NotUnlocked` / `SessionExpired` on locked/expired vaults. ## Adversarial framing — L-section (load-bearing risks)

> Per §16, every plan-gate doc for a security-critical issue surfaces the load-bearing risks an adversary could leverage from each design choice. For each L1..L11 above the question is: *what could an attacker exploit if this risk slipped past the implementation?*

### L-typehash-drift

**What goes wrong:** the Rust client's `REVISION_TYPEHASH` is computed from a literal string that disagrees with `RevisionLogV1.sol:132-134`'s literal string by even one byte (e.g., a stray space; a re-ordered field; a `uint8` instead of `uint16` for `schemaVersion`; `encPayload` instead of `encPayloadHash`). The Rust client signs a digest derived from typehash-A; the contract's `_recover` computes a digest derived from typehash-B; `ecrecover` returns a *different* address (NOT address(0) — `ecrecover` always returns some address for a well-formed `r ‖ s ‖ v`); that address is not registered for the vault; revert with `ErrSignerNotRegistered`.

**Failure mode:** every `publishRevision` reverts; every user is broken; gas is burned on every reverting tx (per L6 of 2.1: revert paths still cost gas, by design — the spam filter). **No user can publish a single revision until the typehash is fixed + a new client build ships.**

**Defense (test gate):** pin the literal `"Revision(bytes32 vaultId,bytes32 accountId,bytes32 parentRevision,bytes32 deviceId,uint16 schemaVersion,bytes32 encPayloadHash)"` byte-string as a Rust `&'static [u8]` const; pin the resulting keccak digest as a `[u8; 32]` const computed at test time; have a test that asserts the keccak of the literal matches the const. **Cross-check against the contract via `cast call 0x179362... "domainSeparator()(bytes32)" --rpc-url …` + asserting the Rust-computed domain separator (which DERIVES from the type hashes) matches the contract's stored one.** Under Q-e Option A or C this is a live test; under Option B it is a pinned-constant test.

**Adversary leverage:** none directly — this is a self-inflicted bug. But: if the typehash drift is *introduced later* by a malicious / co-opted contributor (e.g., a PR that "cleans up" the typehash string and silently re-orders a field), the test catches it; without the test, every user's publishes start failing on the next release. The cost of the test is one Rust const + one assertion.

### L-canonical-s

**What goes wrong:** the Rust signer produces an ECDSA signature with `s > n/2` (a "high-s" signature). `k256::ecdsa::Signature::normalize_s` exists for a reason — secp256k1 has the property that `(r, s, v)` and `(r, n - s, v ^ 1)` are both valid signatures for the same `(digest, signer)`. EIP-2 / EIP-2098 / OpenZeppelin canonical-s discipline + the v1 contract's `_recover` (line 433) explicitly rejects `s > n/2` to defeat signature malleability.

**Failure mode (contract-side):** every reverting tx (same as L-typehash-drift but for a different reason). **Wider failure mode (off-chain):** even if the contract did not enforce canonical-s, an off-chain indexer that filters by signature bytes (e.g., for replay detection) would see two distinct-byte sigs for the same revision and treat them as different events.

**Defense:** k256's `SigningKey::sign_prehash_recoverable` returns a `Signature` + `RecoveryId`; the underlying `k256::ecdsa::Signature` should be normalised low-s by default in recent versions (k256 ≥ 0.13). 3.1 explicitly calls `.normalize_s()` (defensively, idempotent if already low-s) before serialising r/s/v. The test surface asserts on a fixed seed that the produced `s` is `≤ n/2` (one assert; one fixed seed; deterministic).

**Adversary leverage:** if canonical-s slips, the same revision payload could be re-broadcast with a flipped-s sig at any time — the contract rejects but the user thinks they are stuck. Combined with a network observer seeing the pending tx: a denial-of-service amplifier (every reverting tx burns the device wallet's gas). Mitigated to zero by the normalise + the test.

### L-v-byte

**What goes wrong:** k256's `RecoveryId` returns `0` or `1` (raw). EVM `ecrecover` expects `27` or `28`. Forget the `+ 27` offset → the contract's `_recover` rejects on `v != 27 && v != 28` (line 421).

**Failure mode:** `ErrInvalidSignature` on every tx. Cheap to fix if caught; expensive if not — same revert-every-tx failure mode as L-typehash-drift / L-canonical-s.

**Defense:** the test exercises `v` bytes explicitly. `v` SHOULD only ever be 27 or 28 (the legacy non-EIP-155 form for EIP-712 typed-data sigs). NOT 0/1 (raw), NOT 35/36/2*chainId+35 (EIP-155 forms used for legacy transactions but not for EIP-712 typed-data — the chain id is in the domain separator). Pin a fixed-seed test that asserts `signature[64] ∈ {27, 28}`.

**Adversary leverage:** same DoS amplifier as L-canonical-s. Mitigated to zero by the test.

### L-domain-binding

**What goes wrong:** signing against a wrong `chainId` (e.g., 1 instead of 84532), a wrong `verifyingContract` (e.g., a v0 contract address by accident), or a wrong `name` / `version` string. The resulting EIP-712 digest is well-formed; the `r ‖ s ‖ v` signature is well-formed; `ecrecover` recovers the correct *signer-of-the-wrong-digest* (a different address than the device's `EvmWallet`). The contract's `_recover` returns the wrong address; per R-b self-bootstrap, if the vault has no previously-registered devices, the **wrong address is silently registered as the legitimate device for that vault**. From that moment on, every future publish under the *correct* domain binding fails with `ErrSignerNotRegistered` — and there is NO recovery path in v1 (per 2.1 R-c: no on-chain revocation; recovery is MVP-3).

**Failure mode (worst-case):** **a domain-binding misconfig poisons a vault permanently on-chain** under self-bootstrap. The vault's `vaultId` is forever bound to a wrong device address; no future publish from the correct device can recover. The only escape is to start a fresh `vaultId` (= a different vault file) — i.e., abandon the on-chain history of the misconfigured vault.

**Defense (mandatory):** the contract-address sourcing (Q-c / L6) MUST be deterministic and auditor-traceable. The chain-id MUST be cross-checked against `eth_chainId` at adapter-construction (already done by P7 MED-1 / `check_chain_id`); ADD: the EIP-712 domain separator MUST be cross-checked against the contract's `domainSeparator()` view fn (cheap one-time `eth_call` at construction; one extra RPC; same cost as P7 MED-2's runtime-keccak check). The first time a `BaseSepoliaAdapter` (or its v1 sibling) is constructed against D-017, the construction asserts `rust_computed_domain_separator == contract_view_returned_domain_separator`; mismatch fails closed with `ChainError::DeploymentMismatch`. **This is the cheapest single defense in 3.1.**

**Adversary leverage:** the worst single bug the issue could ship. An attacker who manages to convince a user to publish their first revision against a wrong-domain configured client (via a tampered deployment file, a malicious build, etc.) permanently captures the vault's on-chain identity. The defense (the construction-time cross-check) reduces this to "the operator notices the construction error before publishing", which is the strongest defense Rust-level can provide.

### L-self-bootstrap-frontrun

**Distinct from but related to L-domain-binding.** v1's R-b self-bootstrap means the *first* publish for a fresh `vaultId` self-registers the signer. If an attacker observes a pending publish in the mempool (a fresh `vaultId`'s genesis revision):
- They cannot re-sign the same revision under their own key (they do not have the user's `EvmWallet` scalar).
- They CAN copy the user's bytes verbatim and submit them with a higher gas price (front-run); the *same* signature recovers to the *same* signer (the user's device address); the vault is self-bootstrapped to the *correct* signer regardless of `msg.sender`. The attacker pays gas for the user's tx; the user's tx then reverts as a duplicate-sequence on retry (no, actually — since v1's sequence is global, both txs would succeed at different sequence numbers, both emitting `RevisionPublished` for the same revision; the local merge dedup logic in `pangolin-store::ingest_event` (see `vault.rs:5605-5663`) handles the duplicate-on-chain-anchor case via the canonical-hash match).

**Failure mode:** **none** — same as v0's "anyone-can-publish" property. The signature is the gate, not `msg.sender`.

**Adversary leverage:** the only "win" is forcing the user to pay nothing while the attacker burns their own gas to register a vault for them. Harmless. Same posture 2.1 Threat #3 documents.

### L-mempool-leak-of-vault-binding

**Subtle:** publishing a signed revision puts the EIP-712 digest's input fields (`vaultId`, `accountId`, `parentRevision`, `deviceId`, `schemaVersion`, `encPayloadHash`) into the calldata of the pending tx, observable in the mempool. The `encPayload` is opaque ciphertext — not a leak. `vaultId`, `accountId`, `deviceId` are 32-byte opaque identifiers — not user-correlatable on their own. **BUT** the recovered signer (the device's secp256k1 address) is observable via `_recover(digest, signature)` once any third-party computes the same recovery — i.e., as soon as the tx is mined, the `RevisionPublished` event's `signer` field is publicly broadcast.

**Failure mode:** the device's EVM address is publicly correlatable with the `vaultId` and `accountId` it publishes for. This is **D-006's known mitigation** — the Privacy Phase-2 plan addresses this via per-account derived wallets or pseudonymous rotation. 3.1 does NOT change the posture; the same property holds for v0 (the device key bytes are in the v0 event's `deviceId` field already, just under a different primitive).

**Adversary leverage:** chain analysis correlates the device address across all vaults it publishes for. Out of scope per master plan §3.7 (privacy enhancements are MVP-3+) + DECISIONS.md D-006 mitigation note.

### L-payload-hash-prereduction

**Subtle:** the EIP-712 typed-data struct's `encPayloadHash` is the keccak digest of the raw `encPayload` bytes — NOT the raw bytes themselves. The struct binds `bytes32 encPayloadHash`, not `bytes encPayload`. The contract does this: `hashRevision(...)` calls `keccak256(encPayload)` internally (line 313). The Rust client MUST also reduce the payload via `keccak256` before EIP-712-encoding the digest.

**Failure mode:** if the Rust client forgets to pre-reduce (e.g., naively builds an EIP-712 struct with a `bytes` field instead of a `bytes32` field), the digest computed client-side differs from the digest the contract computes. Same revert-every-tx failure mode.

**Defense:** the EIP-712 struct definition is a Rust `alloy::sol!` block mirroring `Revision(bytes32 vaultId, bytes32 accountId, bytes32 parentRevision, bytes32 deviceId, uint16 schemaVersion, bytes32 encPayloadHash)`. The `bytes32 encPayloadHash` field type forces the Rust code path to pre-reduce; a type error catches the regression at compile time. **Same canonical-hash discipline `pangolin-chain::signing::canonical_hash` already follows for v0** (line 152: `let payload_keccak: [u8; 32] = keccak256(enc_payload).0;`). The reduction itself is invariant; what changes is the outer envelope (raw-keccak in v0; EIP-712 typed-data in v1).

**Adversary leverage:** none directly. Catches a regression that would otherwise look like L-typehash-drift.

### L-evmwallet-lifecycle

**What goes wrong:** the secp256k1 signing scalar is reachable outside an Active session — e.g., a `static` / `OnceCell<EvmWallet>` cache, a `Vault::sign_revision_bypass_lock(...)` shortcut, or an FFI entry point that returns a signing handle.

**Failure mode:** signing material in memory past the authorized session window. Defeats 1.5's "DeviceKey lives only in ActiveState" doctrine + 3.2's R-b. An attacker who steals a locked-but-process-running vault (e.g., via a memory dump) extracts the secp256k1 scalar long after the user's session expired.

**Defense:** L5 invariant — every signing entry point takes `&EvmWallet`, obtained only via `Vault::evm_wallet()` (3.2 R-b's accessor, which calls `require_active()`). The test exercises this: `evm_wallet_dropped_on_lock_idle_expiry_absolute_expiry` (already present in `pangolin-store::vault::tests:8477-8527`) covers the three teardown paths. 3.1 does NOT add a bypass; the test stays in place.

**Adversary leverage:** memory dump of an Active session is out-of-scope per `THREAT_MODEL.md` ("fully compromised operating systems / kernel-level malware"); but the dump of a *locked* session that still has the secret in memory IS a new failure mode that 3.1 has the opportunity to introduce (and must not).

### L-deployment-mismatch

**What goes wrong:** the deployment file at `contracts/deployments/base-sepolia.json` is tampered: the `RevisionLogV1.address` field is changed to point at an attacker-controlled contract that mimics the v1 ABI. The Rust client signs against the wrong `verifyingContract`. Same failure mode as L-domain-binding (worst-case: the attacker's contract self-bootstrap-registers the user's wallet for whatever fake-vault the attacker constructs).

**Defense:** the existing P7 MED-2 `check_runtime_keccak` cross-check (in `base_sepolia.rs:677`) reads the live bytecode at the recorded address + asserts its keccak matches the deployment file's `deployed_runtime_keccak256`. 3.1's new v1-targeted adapter (or v1-aware signing-builder construction path, depending on Q-c) MUST inherit this cross-check. **In addition:** the new construction path runs the contract's `domainSeparator()` view fn (one `eth_call`) and asserts `rust_computed == contract_returned`. Three independent cross-checks: chain-id, runtime-keccak, domain-separator. All must pass before the signer is usable.

**Adversary leverage:** mitigated to "the operator must notice the construction error". Strongest defense without an external-audit pin.

## Threat-model touch points

3.1 adds a new entry to `THREAT_MODEL.md`'s per-component table: **"Revision signing (secp256k1 v1; EIP-712 typed-data)"** — DOCUMENTED (issue 3.1). The per-surface enumeration:

1. **Adversary forges a signed revision under a stolen secp256k1 scalar.** Out of scope per the "Cryptographic failures: if XChaCha20-Poly1305 or Ed25519 is broken, Pangolin breaks. We assume modern primitives are sound." — extend to secp256k1 (2.1 R-a implicitly extended this; 3.1 makes it explicit on the client side).
2. **Adversary front-runs a self-bootstrap publish.** Documented L-self-bootstrap-frontrun: no leverage (the signature is the gate, not `msg.sender`).
3. **Adversary intercepts the EIP-712 digest's input fields in the mempool.** Documented L-mempool-leak-of-vault-binding: D-006's known mitigation; Privacy Phase-2 territory.
4. **Adversary correlates EVM address with `vaultId` across multiple publishes.** Same as #3; D-006's known mitigation.
5. **Adversary tampers with the deployment file to redirect signing to a wrong `verifyingContract`.** Mitigated by the P7 MED-2 runtime-keccak cross-check + the new `domainSeparator()` cross-check (L-deployment-mismatch).
6. **Adversary mounts a regression PR that drifts the Rust typehash from the contract typehash.** Mitigated by the L-typehash-drift test gate (pinned constants + assertion).
7. **(Q-a Option A only) Adversary tampers with the re-sign migration to swap a different `signature_v1` into a row before back-fill completes.** Out of scope (filesystem-level tamper of a locked vault is the same threat as v0; the AAD-bound payload + the v1 sig + the recovered-signer-must-match-registered-device gate triple-locks the on-chain semantics).

**Invariant additions: none new.** The existing invariants #1, #5, #11 cover 3.1's surface; #11 (§18.7 schema-version ladder) is exercised by the `schema_version: uint16` widening which 3.1 picks up from 2.1 L5 + R-d.
