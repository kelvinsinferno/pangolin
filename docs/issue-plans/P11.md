## What P0..P11A already shipped (load-bearing surface for P11)

P11 is **a documentation issue**. Every component the reproducer
exercises was built and tested in earlier issues. P11's job is to

### Existing CLI surface (from P8 + P9 + P11A)

A non-author dev now has a complete CLI for vault lifecycle:

- **`pangolin-cli status --vault-path <path>`** — read-only
  diagnostics: `dirty_count`, `account_count`, `last_pulled_block`,
  `last_published_block`, `head_count` per account. Does not make
  chain calls. No vault-unlock required if invoked without
  credentials. (P8.)
- **`pangolin-cli publish --vault-path <path> --account <keystore>`** —
  per-dirty-entry sign + submit + clear-dirty. A3 pre-publish guard
  catches duplicate publish on re-run; per-account error isolation.
  (P8.)
- **`pangolin-cli pull --vault-path <path>`** — incremental chunked
  pull from `last_pulled_block`. Every event signature-verified
  before persist (Q6 defense in depth). Forks reported via
  `PullReport.forks`; freeze sentinel set on foreign-event ingest
  under PoC two-key. (P8.)
- **`pangolin-cli resolve --vault-path <path> --account-id <hex>
  --keep <hex> --account <keystore>`** — clears the freeze sentinel
  for one account by re-sealing the chosen revision under merge AAD,
  publishing the merge revision, and ingesting it. `--dry-run`
  prints the planned action without side effects. (P9.)
- **`pangolin-cli account add | list | show | update | delete`** —
  full account-management surface. `add` creates entries with
  `--generate-password` / `--password-stdin` / interactive prompts;
  `list` enumerates active entries with `--include-frozen` /
  `--include-tombstoned` overrides; `show` reveals secret fields
  one-prompt-per-bundle under presence; `update` is always
  presence-gated under PoC; `delete --yes` writes a tombstone.
  (P11A.)

reproducible, copy-paste-able instructions that drive the existing

authoritative reproducer guide), `POC_README.md` (the non-author

on E2E-003/004/005/006, and a captured rehearsal transcript at
`docs/issue-plans/P11-rehearsal.md`. Closes the master-plan §3.9
PoC → MVP gate clause "`E2E_TESTS.md` reproduced by a non-author
developer from clean clone"; introduces no new cryptographic
primitive, no chain-side code, no library API, no test code, and no
new threat surface.

## Architectural decisions (locked in this plan)

These decisions are committed to as part of plan-gate approval. Any
deviation during BUILD requires returning to PLAN per §16.4.

### A1. Reproducer execution modes — MockChainAdapter as default; Base Sepolia live as opt-in

**Decision:** Each scenario in `docs/E2E_REPRODUCER.md` ships TWO
parallel execution modes: a **default mock mode** that points the
user at the pre-existing automated tests (`cargo test -p pangolin-cli
--test two_vault_roundtrip`, etc.), and an **opt-in live mode** that
walks the user through `pangolin-cli` commands against the deployed
RevisionLogV0 contract on Base Sepolia. The mock mode is the "did
the build work" smoke; the live mode is the "non-author dev
exercises the user-facing CLI end-to-end" rehearsal target.

**Rationale:**

1. **Mock-mode is non-credential, non-network, non-funded.** A
   non-author dev with a fresh clone and `cargo` installed can
   complete mock-mode in <2 minutes. The `MockChainAdapter` filters
   by `vault_id` and supports `set_disconnected(bool)` exactly so
   that all three scenarios can run unattended.
2. **Live-mode validates D-005 + D-014.** The `0x8566D3...896`
   contract is the canonical PoC deployment; running the
   reproducer against it proves the full client → contract →
   client loop works on real chain conditions (RPC rate limits,
   block-finality semantics, real keystore-password prompts). The
   PoC → MVP gate (§3.9) is more honestly satisfied if at least
   one rehearsal hits the live chain.
3. **Live-mode is opt-in to keep the cost-of-rehearsal low.**
   Funding a Base Sepolia wallet is a 5-minute side-trip the dev
   may not have appetite for; the reproducer's "Live mode
   (optional)" subsection makes the choice explicit.

**Alternative considered:** Mock-mode-only. **Rejected** because
the §3.9 gate language ("reproduced by a non-author developer from
clean clone") is most credibly satisfied if at least one rehearsal
exercises the production chain integration. Mock-mode-only would
leave the live-chain path untested by anyone who isn't the author.

**Alternative considered:** Live-mode-only. **Rejected** because
forcing a non-author dev to fund a wallet before they've even built
the binary is high friction; mock-mode-first lets the dev decide
whether to invest in the live walk-through.

### A2. Reproducer document format — single Markdown file with three scenario sections

**Decision:** All three scenarios live in a single Markdown file
(`docs/E2E_REPRODUCER.md`). Each scenario is a level-2 heading
(`## Scenario 1: …`), with level-3 subsections for prerequisites,
setup, mock-mode steps, live-mode steps, expected output, and
cleanup. The file is sequential reading; a non-author dev reads top
to bottom and runs the scenarios in order.

**Rationale:**

1. **One file, one mental model.** The three scenarios share most
   of their setup (clone, build, scratch dir, vault password); a
   single file lets the reader internalize once and reference
   thereafter.
2. **`E2E_TESTS.md` cross-references are simple.** Each
   `### Reproducibility` subsection is one line (`Non-author dev
   path: docs/E2E_REPRODUCER.md § Scenario 1`); a multi-file split
   (`docs/E2E_REPRODUCER/scenario-1.md` etc.) would require multi-
   line cross-refs.
3. **Search-ability.** A user `grep`-ing for "freeze sentinel" or
   "publish summary" finds matches in one file rather than three.

**Alternative considered:** Three separate files
(`docs/E2E_REPRODUCER/scenario-1.md` etc.) plus an index. **Rejected**
because the reproducer's audience is reading sequentially, not
reference-style; the index adds navigation friction without
information density gain.

**Alternative considered:** Inline each scenario's reproducer steps
into the matching `E2E_TESTS.md` entry. **Rejected** because
`E2E_TESTS.md` is the **author-facing** test ledger (entries are
written for the author + reviewer, with internal terms like
"freeze sentinel" / "canonical hash" / "AAD" used freely); the
reproducer is the **non-author-facing** runbook (must avoid
internal terms; uses "quarantined" / "the local copy and the chain
copy disagree" / "encryption tag" instead). Two audiences, two
documents — but cross-linked.

### A3. Output verification — literal expected-output snippets + cargo-test double-check

**Decision:** Every step in `docs/E2E_REPRODUCER.md` that produces
output ships an "Expected output" code block with the literal text
the user should see (or a representative subset, with `<…>`
placeholders for non-deterministic values like `account_id`,
`tx_hash`, `block_number`). For each scenario, the reproducer also
ends with a "Sanity check" step that runs the corresponding `cargo
test` invocation against the same `MockChainAdapter` setup; if the
expected literal output diverges from what the user saw,
`cargo test`'s pass/fail is the authoritative answer.

**Rationale:**

1. **Literal output is copy-paste-able assurance.** A user who
   sees `publish summary: 1 published, 0 failed` matches the
   reproducer's expected line and gains immediate confidence.
2. **`cargo test` is the structural truth.** The integration tests
   assert against vault state + chain state; a passing test means
   the underlying behavior is correct even if the literal stdout
   format changed (e.g., a logging tweak in `pangolin-cli` between
   two reproducer dispatches).
3. **Non-deterministic values are placeholdered.** `account_id`,
   `tx_hash`, `block_number`, `revision_id`, KDF salts, etc. are
   non-deterministic across runs. The expected-output blocks use
   the convention `<account-id-32-bytes-hex>` (lowercase angle
   brackets) so the user knows exactly which fields will differ.

**Alternative considered:** Snapshot-test the reproducer's expected
output via an `insta` test that runs the documented commands and
asserts the captured stdout matches the docs. **Rejected** —
introduces test code (out of P11 scope), creates a maintenance
burden if the CLI's output format evolves between PoC and MVP-1,
and conflates "the docs are correct today" (P11's job) with "the
docs stay correct" (P12-2's job).

### A4. PoC limitations honestly disclosed

**Decision:** `docs/E2E_REPRODUCER.md` includes a top-level
"**PoC limitations and surprises**" subsection that names every
known PoC quirk a non-author dev will hit if they probe edge cases.
The list is drafted in §A4-list below; the BUILD pass cross-checks
every item against the source code + the corresponding
`E2E_TESTS.md` entry's "Failure modes covered" subsection.

**A4-list (locked):**

- **P10 foreign-event "freeze on pull" under PoC two-key.** When
  vault A publishes and vault B pulls, the foreign device_id
  causes B's account to be FROZEN pending resolve (CRIT-1
  discipline). The user sees `pull summary: ... 1 frozen
  account(s)` and must run `pangolin-cli resolve` before B's view
  converges with A. This is the expected scenario-2 flow even
  though it surprises a user who expects "two-vault sync = silent
  convergence" from the master plan §3.7's "P11-1: A creates →
  publish → B pulls → identical state" wording. The reproducer's
  Scenario 1 (P11-1 happy path) is therefore reframed: "A creates
  → publish → B pulls → B sees the entry as FROZEN; user runs
  resolve → both converge." The P11-1 wording in the master plan
  predates P8's CRIT-1; the freeze-then-resolve is the actual PoC
  shape. (Master plan §3.7 may be patched in a future docs-only
  pass to reflect this; out of P11 scope.)
- **P9 multi-resolve under PoC two-key.** Scenario 2 (P11-2)
  conflict resolution requires BOTH A and B to run `pangolin-cli
  resolve` on their own freeze flags before full single-head
  convergence (per E2E-004's "manual path" step 4 narrative). The
  reproducer documents this as a known "two-resolve walk" and
  flags it as MVP-1's single-key model (D-006) closing the gap.
- **P11A presence-prompt phishing surface.** Scenario 1's
  `pangolin-cli account show --reveal-password` prompts for `'y'`;
  under PoC, the prompt is the only proof-of-presence available.
  Documented via a brief "what the prompt actually proves" callout
  + a pointer to `THREAT_MODEL.md` row #26 for the full discussion.
- **`canonicalize_vault_path` requires the vault file to exist**
  (P8 fix MED-3). The reproducer's setup steps create the vault
  file (`Vault::create` invocation via … wait, this is the
  reproducer pain point — see §A11 below).
- **`pangolin-cli account add` does NOT auto-create the vault file**
  (P11A open-question 1 → recommend NO answered NO at SIGNOFF). The
  reproducer's setup steps therefore include an explicit vault-
  creation step. **§A11 below: how the user creates the vault.**

**Rationale:**

1. **Honesty is the §3.9 gate's substance.** The PoC → MVP gate
   reads "reproduced by a non-author developer from clean clone"
   — if the reproducer hides surprises, the rehearsal will hit
   them and the gate is silently false. Surface them upfront.
2. **Each surprise is closed by a clearly-named MVP-N issue.** The
   reproducer's "limitations" list ends each item with the closing
   issue (e.g., "MVP-1 single-key model — D-006") so the reader
   sees "this is a PoC quirk, not a design failure."
3. **Sets non-author-dev expectations correctly.** A reader who
   understands the freeze-then-resolve flow before running it
   doesn't perceive the freeze as a bug.

**Alternative considered:** Hide the limitations and let the user
discover them through error messages. **Rejected** — recovery from
"the freeze sentinel surprised me" requires reading P8 + P9 docs;
surfacing the gotcha upfront is honest and reduces friction.

### A5. Live-chain mode safety — Foundry keystore + funded Base Sepolia wallet, with explicit warnings

**Decision:** The live-chain mode of each scenario in
`docs/E2E_REPRODUCER.md` requires:

- A Foundry keystore (`cast wallet import`) at
  `~/.foundry/keystores/<name>` (Linux/macOS) or
  `%USERPROFILE%\.foundry\keystores\<name>` (Windows).
- A Base Sepolia ETH balance on the keystore's address — funded
  via a public faucet (e.g., Coinbase Wallet faucet, Alchemy
  faucet, or any of the published Base Sepolia faucets) **out of
  band**. The reproducer points the user at one or two known
  faucet URLs but does not embed faucet API calls.
- An **explicit safety-block header** at the top of each "Live
  mode (optional)" subsection:

  ```
  > ⚠️ TESTNET ONLY. The keystore you use for this walkthrough must
  > NEVER hold mainnet ETH or be reused for any real-value
  > operation. Generate a fresh keystore (`cast wallet new`),
  > fund it from a Base Sepolia faucet, and discard it after
  > the rehearsal. Pangolin's PoC contract is on testnet only;
  > a future MVP-2 mainnet deployment will live at a different
  > address.
  ```

- A **funded-keystore prerequisite check** in the reproducer's
  prerequisites section: a `cast balance <address> --rpc-url
  https://sepolia.base.org` invocation that the user must run
  BEFORE the live walkthrough; expected output is a non-zero
  hex balance.

**Rationale:**

1. **Testnet-only-discipline.** A non-author dev is unlikely to
   already have a Base Sepolia keystore; they create one for the
   walkthrough. The "discard it after" framing avoids the failure
   mode where the dev later reuses the keystore for mainnet
   activity (and then learns Pangolin's keystore was disclosed
   in a public agent transcript).
2. **Faucet linkage is non-trivial.** Faucets rotate; embedding a
   live faucet URL in the docs creates a maintenance burden. Two
   well-known options ("Coinbase Wallet faucet at
   https://www.coinbase.com/faucets/base-ethereum-sepolia-faucet"
   and "Alchemy faucet at https://www.alchemy.com/faucets/
   base-sepolia") are mentioned with a "or any other Base Sepolia
   faucet — the requirement is that the keystore address has a
   non-zero ETH balance" disclaimer.
3. **The pre-flight `cast balance` check fails fast.** A user
   whose faucet drip hasn't landed yet sees the zero balance and
   waits, rather than running `pangolin-cli publish` and getting
   a confusing "insufficient funds" error mid-flow.

**Alternative considered:** Auto-fund from a project-controlled
keystore. **Rejected** — would require Pangolin to ship a
publicly-accessible funder service for the rehearsal, which is
out of scope for PoC + creates an ongoing operational liability.

**Alternative considered:** Skip live-chain mode entirely. **Rejected**
per §A1 above — the §3.9 gate is most honestly satisfied with at
least one live walkthrough.

### A8. Vault password example — domain-specific test sentinel, not "test123"

**Decision:** The reproducer's vault password example is the
literal string `pangolin-poc-test-vault-do-not-reuse`. Every
reference to this password in the docs is wrapped in a quoted
code block AND a "TEST password — do not reuse" annotation.

**Rationale:**

1. **The password value telegraphs intent.** A reader who copies
   `pangolin-poc-test-vault-do-not-reuse` cannot easily forget
   it's a test artifact; if the reader later discovers the same
   password in their shell history, the value's content reminds
   them.
2. **`test123` is the worst-case value.** It's short, memorable,
   reusable; a careless dev could plausibly use `test123` for a
   real vault.
3. **The keystore password in live-mode is separate.** That's the

**Alternative considered:** A randomly-generated value
(`pangolin-9f3a-poc-do-not-reuse`). **Rejected** — random values
are harder to remember across the multi-step walkthrough; the
fixed string trades off memorability against the very-narrow risk
that two users who both run the rehearsal could collide on test-
vault contents. (Test vaults are local; collisions are

### A9. Vault paths — project-local `tmp/` directory, with platform notes

**Decision:** The reproducer's example vault paths are
`./tmp/pangolin-poc-vault-A.pvf` and `./tmp/pangolin-poc-vault-B.pvf`

### A10. Single shell session per scenario, with explicit `cd`

**Decision:** Each scenario in the reproducer assumes the user
runs commands in **one shell session** that starts with `cd
<repo-root>`. Multi-window flows (e.g., scenario 2 needs to
simulate two devices on one machine; scenario 3 needs an offline
state) use **two separate shell windows** with explicit "open a
new terminal window now" markers. No backgrounded processes; no
`screen`/`tmux` plumbing; no shell-state persistence across
scenarios.

**Rationale:**

1. **One session per scenario keeps the mental model flat.** The
   user runs `cd <repo-root>` once and every subsequent `cargo` /
   `pangolin-cli` invocation is from the same cwd.
2. **Explicit second-window markers for two-vault scenarios.**
   Scenario 2 opens a second terminal with the marker
   `Open a second terminal window. Run `cd <repo-root>` in it.`;
   the user cannot misread it.
3. **No shell-state magic.** No environment variables persist
   across scenarios; if scenario 2 needs `BASE_SEPOLIA_RPC_URL`
   set, the reproducer says `export
   BASE_SEPOLIA_RPC_URL=https://sepolia.base.org` (or the
   PowerShell `$env:BASE_SEPOLIA_RPC_URL = ...` equivalent) at
   the top of the scenario.

**Alternative considered:** `direnv` / `.envrc` automation.
**Rejected** — adds a tool dependency; non-trivial to debug if it
silently fails to load.

### A13. Troubleshooting subsection — top 5 expected gotchas

**Decision:** `docs/E2E_REPRODUCER.md` ends with a "Troubleshooting"
section that names the top 5 expected gotchas + their fixes:

1. `cargo build` fails with "stable Rust 1.83+ required" → install
   via `rustup default stable` + verify `rustc --version`.
2. `pangolin-cli account add` exits with "could not canonicalize
   vault path" → the vault file doesn't exist yet; run the
   `cargo run --example create_vault` step from setup.
3. `pangolin-cli pull` exits with `ChainError::Rpc` in live mode
   → the RPC URL is wrong or the network is unreachable; verify
   `curl -s https://sepolia.base.org` returns a JSON-RPC envelope.
4. `pangolin-cli publish` says "publish summary: 0 published, 1
   failed" with an "insufficient funds" error → the keystore
   address has no Base Sepolia ETH; re-fund from a faucet.
5. The `cargo test --workspace --lib` smoke prints fewer than 384
   passing tests → the build is on a stale tree; `git status`
   should show no modifications and `git log -1` should show
   `422a65e` (or a later main-branch commit).

**Rationale:** The rehearsal agent's most likely escalation paths
are environmental (toolchain, network, funding); pre-emptively
naming the fixes shrinks the rehearsal-iteration loop.

**Alternative considered:** No troubleshooting section; let the
rehearsal surface gotchas. **Rejected** — guaranteed rehearsal
churn cost; the troubleshooting section is a small upfront
investment with high payoff.

## Threat model row

P11 likely needs **no new row** in `THREAT_MODEL.md`. Verification
walk:

- **The reproducer is documentation, not attack surface.** A
  malicious actor reading the reproducer learns the same
  information they would learn by reading the existing
  `pangolin-cli` `--help` output — PoC behavior, the deployed
  contract address (D-014; already public), the freeze-then-
  resolve flow (P9; already documented in `THREAT_MODEL.md`).
- **The example helper (`create_vault.rs`) is non-novel.** It
  calls `Vault::create` exactly as test fixtures do; the audit
  surface is unchanged.
- **Live-chain mode safety warnings** (§A5) are themselves the
  mitigation. A user who follows the docs uses a fresh testnet
  keystore and discards it; a user who reuses a mainnet keystore
  has ignored a clearly-printed warning. The "user mishandles
  keystore" risk is the inverse of P8's row #1 ("--keystore-
  password echoes in ps") — same threat shape, same mitigation
  language; P11's reproducer adds no NEW vector.

**If the rehearsal surfaces an unexpected user-facing risk** (e.g.,
the docs implicitly encourage a footgun), P11-5 appends a new
threat row at write-time per the §A14 numbering caveat from P11A.
Recommended starting index for any such row: 28 (THREAT_MODEL.md
currently ends at row 27 per the P11A-shipped cluster).

**Recommend NO new row** as the default position; flag at SIGNOFF
if rehearsal disagrees.
