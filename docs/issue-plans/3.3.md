## Open questions for Kelvin (LOAD-BEARING â€” surface at TOP per Â§16)

### Q-a Â· Nonce strategy

The device wallet's pending-tx nonce must be tracked across publish calls. Three plausible models:

**Option A: Fetch `eth_getTransactionCount(addr, "pending")` before every submit.** Simplest correctness model. Costs one extra RPC per publish. The "pending" tag returns the nonce the RPC would assign to the next inbound tx (counting both mined and mempool-resident txs from this address). Slow path under high throughput, but Pangolin's publish throughput is low (one revision per user action, not per second). **Recommendation context:** matches how the existing v0 adapter implicitly behaves today â€” alloy's `Provider::send` auto-fills the nonce per call.

**Option B: Local cache (start from chain query at first submit, increment locally; resync on any error).** Faster â€” saves one RPC per publish. **Trade-off:** adds a correctness surface. Concrete failure mode: another process (a second CLI invocation, a developer using `cast send` from the same wallet) submits a tx that bumps the chain-side pending nonce; the local cache is now stale; the next publish from this process collides. Mitigation: on every retry, force a refresh.

**Option C: Hybrid â€” cache for the hot path, force refresh on nonce-collision retry.** Combines Option B's speed with Option A's correctness on the failure path. Concrete shape: `NonceTracker` struct holds the last-known nonce; `publish_revision_v1` uses the cached value on first attempt; on `eth_sendRawTransaction` returning "nonce too low" / "replacement underpriced" / "already known", refresh via `eth_getTransactionCount(addr, "pending")` + retry once.

**Plan-gate recommends:** Option A for MVP-2 (simplicity wins; the extra RPC is negligible for Pangolin's actual write rate). Option B/C are future optimisations if throughput becomes a measurable bottleneck (it won't in MVP-2 â€” every publish is a human action).

### Q-b Â· Gas-price strategy

D-017 on Base Sepolia is cheap â€” the smoke-test publish at deploy time was `gas_used = 451_478 @ 0.006 gwei`, i.e. ~`0.0000027 ETH`. The question is which gas-fee primitive 3.3 uses and what its safety bound is:

**Option A: `eth_gasPrice` Ã— 1.2 multiplier (legacy gas).** Simplest. Alloy's `ProviderBuilder` auto-fills `gasPrice` if not set. **Downside:** Base Sepolia (post-Bedrock) is EIP-1559 native; using legacy `gasPrice` works but is operationally non-idiomatic. Multiplier `1.2` is a heuristic â€” common practice but not load-bearing.

**Option B: EIP-1559 (`maxFeePerGas` + `maxPriorityFeePerGas`) with caps.** Modern. Alloy's `eth_feeHistory` / `eth_maxPriorityFeePerGas` provide the inputs; the typical pattern is `maxPriorityFeePerGas = 1 gwei (cap)`; `maxFeePerGas = 2 Ã— baseFeePerGas + maxPriorityFeePerGas`. **Better behaviour during congestion** (the tx lands eventually rather than being stuck below `baseFee`). **Recommendation context:** matches how v0's `BaseSepoliaAdapter::publish` implicitly works (alloy `ProviderBuilder` defaults to EIP-1559 when the chain advertises it; Base Sepolia does).

**Option C: Manual cap (`max_fee_per_gas` hardcoded ceiling, e.g. 50 gwei).** Predictable cost â€” the user knows the upper bound. **Downside:** can fail to land on a busy chain (the tx sits in mempool until `baseFee` drops below the cap). Acceptable for testnet (where `baseFee` is consistently sub-gwei); risky for mainnet (where post-EIP-4844 base fees are usually well below 50 gwei but burst spikes happen).

**The load-bearing sub-question regardless of A/B/C: HARD CAP for L6 (DoS protection).** Pick a per-tx `max_fee_per_gas` ceiling above which 3.3 fails closed rather than spending arbitrary ETH. A malicious RPC reporting a huge `gasPrice` / `baseFeePerGas` should NOT result in the user spending more than the cap. Suggested cap: **50 gwei `max_fee_per_gas`** (well above Base Sepolia's typical sub-gwei baseFee; well below any mainnet spike that would burn a non-trivial fraction of a device wallet's balance). Below the cap â†’ submit; above â†’ fail with `ChainError::GasCapExceeded`.

**Plan-gate recommends:** Option B (EIP-1559) + a hard `max_fee_per_gas` cap of 50 gwei. The cap is THE load-bearing defense; the fee-primitive choice is secondary.

### Q-c Â· Retry policy + classification taxonomy

What's retriable, what's fatal? The publish-side error landscape is:

| Category | RPC surface | 3.3 disposition |
|---|---|---|
| **Nonce collision** | `eth_sendRawTransaction` returns "nonce too low" / "already known" / "replacement underpriced" | Refresh nonce, retry (max N=3); if it persists, fail with `ChainError::NonceUnresolvable` |
| **Insufficient funds** | `eth_sendRawTransaction` returns "insufficient funds for gas * price + value" | **Fatal** â€” never retry. Surface as `ChainError::InsufficientFunds` so the user knows to top up the device wallet. |
| **Contract revert** (`ErrInvalidSignature`, `ErrSignerNotRegistered`, `ErrUnsupportedSchemaVersion`) | tx mines, receipt status == 0 | **Fatal** â€” never retry. Retrying would just burn more gas. Decode the revert reason from the receipt's revert-data; surface as `ChainError::Reverted { reason, tx_hash }`. |
| **RPC timeout / transient 5xx** | alloy returns a transport error | Retry with exponential backoff (max N=3 attempts, 250ms / 1s / 4s) |
| **Chain-id mismatch** | construction-time `check_chain_id` already fires | Fatal at construction â€” no retry surface |
| **Pinned-address mismatch** | construction-time / per-publish `EXPECTED_DEPLOYED_ADDRESS_BASE_SEPOLIA` cross-check | Fatal â€” `ChainError::DeploymentAddressMismatch` |
| **Receipt-not-yet-available** | `get_receipt` returns None / pending | Poll with timeout (Q-e decides whether 3.3 blocks for receipt at all) |
| **Out-of-gas** | tx mines, receipt status == 0 with "out of gas" reason | **Fatal** under the gas-estimate strategy 3.3 uses â€” out-of-gas after an estimate means either the estimate undershot the actual cost (a 3.3 bug to fix), OR the contract state changed between estimate and mine (rare for D-017 since it has no admin / pause). Surface as `ChainError::Reverted { reason: OutOfGas }`. |

**Plan-gate recommends:** the above taxonomy verbatim. The two retriable categories (nonce + RPC transient) get bounded retries; everything else is fatal with a specific error variant the operator can act on. **Three retries with exponential backoff** is the proposed bound â€” common practice, low enough that a stuck pending tx doesn't burn five minutes of wall-clock.

### Q-d Â· API surface â€” sync, async, or both?

`pangolin-core` is sync today (`Vault::sign_revision_v1`, `Vault::evm_wallet` etc. are all sync). The chain RPC layer is naturally async (alloy uses tokio). The existing `apps/cli/src/main.rs` builds a tokio runtime at the top and `block_on`'s the async CLI body, and the existing `ChainAdapter::publish` (v0) is async. Three plausible shapes for 3.3's new entry point:

**Option A: Fully sync via `tokio::runtime::Runtime::new().block_on()` (blocking call from any caller).** Matches `pangolin-store::Vault::sign_revision_v1`'s sync shape; lets `Vault::publish_revision_v1(...)` be called from any context. **Trade-off:** every call pays the cost of spinning a runtime up (small but non-zero); deeply-nested async contexts can deadlock if the caller is already inside a tokio runtime (this is a known footgun â€” `tokio::runtime::Handle::block_on` panics if called from inside a runtime). The existing CLI's structure (one outer runtime + sync-from-async via `spawn_blocking`) sidesteps this; a sync `Vault::publish_revision_v1` would need to call `Handle::current().block_on(...)` if it's already inside one, or build a fresh runtime if not â€” fragile.

**Option B: Async-only (`async fn publish_revision_v1(...) -> Result<ChainAnchor>` on a new `pangolin_chain::ChainTransport` trait or directly on a v1 `BaseSepoliaAdapter`).** Cleanest. Requires every caller to be async-compatible. The CLI already is; future GUI host (Tauri / iOS / Android) FFI wrapper can call from a host-managed runtime. **Recommendation context:** matches v0's `ChainAdapter::publish` precedent. The friction is at the `Vault::publish_revision_v1` boundary â€” if `Vault` is sync, the broadcast either (a) doesn't live on `Vault` (it's a free function or trait method on the adapter, taking the wallet as a parameter), OR (b) `Vault::publish_revision_v1` exists but is itself async (breaks the "Vault is sync" doctrine).

**Option C: Both â€” low-level async fn + a sync wrapper helper for non-async callers.** Doubles the surface; halves the surprise. The async path is canonical; the sync wrapper is a thin `block_on` shim documented as "for non-async callers only".

**Plan-gate recommends:** Option B (async-only) + the publish entry point lives on `pangolin-chain` (not on `Vault`), taking `&EvmWallet + &SignedRevisionV1 + ChainEnv + rpc_url` as parameters. The sign step stays on `Vault` (`sign_revision_v1`); the broadcast step is a free async fn or trait method that the caller invokes after extracting the signed bytes. **This keeps `Vault` sync** (the 1.5 + 3.2 doctrine: the vault is a sync owner of session state; the network is somebody else's concern). The CLI's outer tokio runtime hosts the broadcast call.

**Kelvin's call.**

### Q-e Â· Receipt-handling â€” block until mined, return-on-broadcast, or both?

The async fn could:

**Option A: Block until mined.** Submit via `eth_sendRawTransaction`; await the receipt (alloy's `pending_tx.get_receipt()` polls until the tx is included); verify status == 1 + the `RevisionPublished` event was emitted; return a populated `ChainAnchor { tx_hash, block_number, log_index, sequence }`. Same shape v0's `BaseSepoliaAdapter::publish` uses today. **Wall-clock cost:** Base Sepolia's block time is ~2s; with 1-block confirmation the typical publish completes in ~3-5s end-to-end. Good for most user-facing flows ("publish + show 'on chain'").

**Option B: Return immediately after broadcast.** Submit; capture the tx hash; return without awaiting receipt. **Caller's responsibility** to poll for receipt later (a follow-up `await_receipt(tx_hash)` call). **Use case:** batch publishes where the caller wants to fire-and-forget a queue and then await the whole batch. Not load-bearing for MVP-2 â€” every Pangolin publish is a single user action.

**Option C: Both â€” a low-level fn that returns on broadcast + a high-level fn that blocks until mined.** Two entry points. The high-level wraps the low-level + polls.

**Plan-gate recommends:** Option A (block until mined) for the canonical entry point in 3.3. Option B's fire-and-forget is a future ergonomic; the current CLI flow ("publish") wants the "confirmed on chain" answer before returning. **Configurable confirmations:** the canonical entry point waits for **1 block confirmation** (the receipt is available); a future enhancement could let callers specify `n_confirmations` > 1 for additional reorg safety.

## Adversarial framing â€” L-section (load-bearing risks)

> Per Â§16, every plan-gate doc for a security-critical issue surfaces the load-bearing risks an adversary could leverage from each design choice.

### L-calldata-encoding-drift

**What goes wrong:** the Rust `sol!`-declared binding's selector or field encoding drifts from `RevisionLogV1.publishRevision`'s on-chain ABI. A wrong selector â†’ tx reverts at the dispatcher (no event; receipt status 0). A wrong field order or type â†’ fields are shifted into the wrong slots; the contract recovers a different `signer`; `ecrecover` returns address(0) under L-typehash-drift semantics from 3.1's analysis (same failure mode, different layer).

**Failure mode:** every publish reverts; every user is broken; gas burned per reverted tx. **No user can publish a single revision until the binding is fixed + a new client build ships.**

**Defense:** the `sol!` block mirrors the literal Solidity-shaped declaration byte-for-byte. The calldata-pinned test (see "Test plan") asserts the encoded bytes match a `cast calldata`-derived reference for a fixed-seed input. Same defense the v0 adapter uses in its `sol!` block (`base_sepolia.rs:60-89`).

**Adversary leverage:** none directly â€” this is a self-inflicted bug. But a malicious / co-opted contributor who "cleans up" the `sol!` block (reorders fields; widens a type) would trip the calldata test before merge. Cost of the test: one Rust const + one assertion.

### L-gas-griefing (load-bearing per L6)

**What goes wrong:** a malicious / co-opted RPC reports a huge `gasPrice` / `baseFeePerGas`. The Rust client, without a hard cap, computes `max_fee_per_gas = RPC-reported value Ã— some-multiplier` and submits a tx with that fee. The device wallet drains ETH at attacker-controlled rates.

**Failure mode (worst-case):** at 10_000 gwei Ã— 451_478 gas = ~4.5 ETH per publish on mainnet (post-MVP-2). On Base Sepolia: practically meaningless cost-wise (testnet ETH is free), but the wallet's faucet-supplied balance is exhausted faster than it can be refilled, denying publish capability.

**Defense:** L6 verbatim â€” hard `max_fee_per_gas` cap (proposed 50 gwei). Above-cap â†’ fail closed with `ChainError::GasCapExceeded { observed, cap }`. The cap is configurable per env (sepolia vs. mainnet) but compile-time-pinned per build; NO env-var override.

**Adversary leverage:** mitigated by the cap. The only "win" left for a malicious RPC is forcing the user's publish to fail to land (`baseFee > cap` means the tx sits in mempool forever) â€” same as Base Sepolia going offline; recoverable by switching to a different RPC URL.

### L-rpc-spoof

**What goes wrong:** a malicious RPC returns fake `eth_sendRawTransaction` success (a non-zero tx hash) but never broadcasts the tx. The Rust client polls for receipt; the receipt never comes; eventually a timeout fires. Or worse: a malicious RPC returns a fake receipt with `status == 1` for a tx that never existed.

**Failure mode (lower-impact):** the user thinks the revision landed; subsequent flows (chain sync in 4.1; offline replay) discover the discrepancy because the on-chain `RevisionPublished` event never appears in `eth_getLogs`. The local `revisions.chain_tx_hash` column points at a non-existent tx. Eventually the user notices and re-publishes.

**Failure mode (higher-impact):** if the malicious RPC also fakes `eth_getLogs` to include a fake `RevisionPublished` event matching the fake receipt, the client never detects the spoof until cross-checking against a different RPC.

**Defense:** poll for receipt with tx-hash; verify (a) `receipt.status == 1`; (b) the `RevisionPublished` event was emitted by the canonical contract address (NOT a different address â€” the v0 adapter's MED-4 defense at `base_sepolia.rs:494` already does this); (c) the event's `vaultId` / `accountId` / `signer` fields match what the Rust client submitted. Plus the construction-time `check_runtime_keccak` cross-check (P7 MED-2) that catches a tampered deployment file. **Three independent cross-checks: chain-id, runtime-keccak, post-receipt event-field verification.** A malicious RPC that fakes all three is indistinguishable from a legitimate RPC against a legitimate contract â€” out of scope (same "fully-compromised RPC" boundary 1.5 / 2.1 / 3.1 took).

**Adversary leverage:** limited to "delay the user's publish by one timeout cycle" or "cause the user to re-publish unnecessarily" â€” minor DoS. The user's funds are not at risk (the cap from L-gas-griefing protects ETH; the device wallet's secp256k1 scalar never leaves the process).

### L-nonce-collision-DoS

**Distinct adversary model:** nonces are per-address; only the user's own address is at risk. So "front-running" with another wallet's nonces doesn't apply. The DoS surface is: a stuck pending tx (e.g., one submitted under a too-low gas price during a fee spike) blocks the user's queue â€” every subsequent publish gets stuck behind it until the stuck tx mines or gets evicted.

**Failure mode:** the user's publishes silently queue up in the mempool; nothing lands; the user can't progress. Mitigation: per-tx timeout (Q-c's retry policy) + tx replacement (submit a higher-gas-price tx at the same nonce). **Tx replacement / cancel-tx is MVP-3 territory** â€” 3.3 fails closed with a clear error rather than implementing replacement.

**Defense:** Q-c's retry policy bounds the retry attempts; after N failed retries the user sees `ChainError::NonceUnresolvable` with the tx hash so they can manually replace via `cast` / a wallet UI.

**Adversary leverage:** essentially none â€” a stuck tx is the user's own doing (or a network condition). No attacker can cause this against the user's wallet directly without controlling the user's RPC (covered by L-rpc-spoof).

### L-replay-after-revert

**What goes wrong:** the tx mines but reverts on-chain (e.g., `ErrSignerNotRegistered` because someone else self-bootstrapped the vault first â€” a 2.1 L-self-bootstrap-frontrun analogue; or a stale schemaVersion). The retry loop, if naive, re-submits the same revision under a fresh nonce â€” and the same revert happens, burning more gas.

**Failure mode:** burned gas with no progress.

**Defense:** Q-c's classification â€” contract reverts (`ErrInvalidSignature`, `ErrSignerNotRegistered`, `ErrUnsupportedSchemaVersion`) are FATAL, never retried. The retry loop only retries (a) nonce-collision and (b) RPC-transient errors; it explicitly does NOT retry a `status == 0` receipt.

**Adversary leverage:** if an attacker can cause a revert (e.g., by self-bootstrapping a fresh `vaultId` with their own device â€” same threat as 2.1's L-self-bootstrap-frontrun), they can force the user's first publish to revert. After the revert the user knows the vault is poisoned (the wrong device is registered) and must abandon the vaultId. **Same posture 2.1's L-self-bootstrap-frontrun took.** Self-bootstrap defenses are 3.6 / MVP-3 territory.

### L-double-broadcast-on-retry (load-bearing per L12)

**What goes wrong:** the retry loop re-broadcasts a tx that, unbeknownst to the retry loop, already landed on chain. Two `RevisionPublished` events for the same revision at different sequence numbers. The downstream chain-sync (4.1) sees both; canonical-hash dedup matches; one is dropped client-side; but gas was burned twice.

**Failure mode:** gas waste; sequence-number inflation on chain (cosmetic; D-017's global counter is uint256 so no overflow risk).

**Defense:** L12 verbatim â€” the retry loop ALWAYS re-queries the receipt by tx-hash BEFORE re-broadcasting. A `status == 1` receipt short-circuits the retry. Pinned in `nonce_collision_does_not_double_broadcast` test.

**Adversary leverage:** none directly â€” this is a self-inflicted bug surface. Defense is mechanical.

### L-receipt-poll-timeout

**What goes wrong:** the `pending_tx.get_receipt()` poll times out (the tx hasn't been included within the wait window). Alloy's default polling is generous; on Base Sepolia (~2s blocks) a typical receipt arrives within ~3-5s. A network blip could push this past any naive timeout.

**Failure mode:** the user sees an error; the tx may still land later; the local state thinks "publish failed" while chain state thinks "publish succeeded".

**Defense:** the receipt poll has a generous timeout (proposed: 60s, ~30 Base Sepolia blocks); on timeout, surface as `ChainError::ReceiptTimeout { tx_hash }` so the user / next sync run can resolve via chain replay (4.1). The local `revisions` row records the tx hash with a "pending" status; the next chain-sync run reconciles. **The reconciliation logic is 4.1's job;** 3.3 just records the tx hash for it.

**Adversary leverage:** minor DoS by RPC withholding receipts. Mitigated to "delay" rather than "data loss" by the explicit pending-status record.

### L-evmwallet-tx-signing-lifecycle

**What goes wrong:** the `EvmWallet`'s signing scalar is used twice â€” once in `Vault::sign_revision_v1` (signs the EIP-712 digest in 3.1's path) and once in `publish_revision_v1` (signs the EIP-1559 tx envelope). Both calls must happen inside the active-session window. If the active session expires BETWEEN the sign step and the broadcast step, the broadcast must fail.

**Failure mode (avoided):** the tx-signing call uses the SAME `&EvmWallet` reference the caller obtained via `Vault::evm_wallet()`. If the session expires while holding that reference, the next call to `Vault::evm_wallet()` errors `SessionExpired` â€” but the reference itself remains valid until dropped (Rust borrow rules). **This is fine** because the reference lifetime is bounded by the borrow checker; the active session can't drop the wallet while a `&EvmWallet` is held. **But:** if the caller holds the reference too long (across a tokio await point that exceeds the session's idle-expiry window), the session has "expired" semantically while still being able to sign. **Mitigation:** the broadcast path takes the `&EvmWallet` reference immediately before the tx-build + send, NOT cached across awaits.

**Defense:** the broadcast helper takes `&EvmWallet` as a function argument (not a struct field on a long-lived adapter). The caller obtains the wallet, builds the signed revision, builds the tx, and broadcasts â€” all in one bounded scope. The session-active check is at the boundary (`Vault::evm_wallet()` errors on a locked vault); inside the helper, the borrow is short.

**Adversary leverage:** none â€” this is a lifecycle hygiene concern, not an attack surface.

### L-deployment-mismatch-broadcast (extends 3.1's L-deployment-mismatch)

**What goes wrong:** between 3.1's signing call and 3.3's broadcast call, the deployment file is tampered (`RevisionLogV1.address` swapped to a different contract). 3.1 signed under the original `verifyingContract`; 3.3 broadcasts to a different contract address. The tx's `to` field points at a contract that's NOT D-017; the contract's `_recover` (if the contract even has one) computes a different EIP-712 digest under its own domain separator â†’ recovers a different signer â†’ reverts. Or worse: the attacker's contract mimics the v1 ABI + happens to have a compatible domain separator + accepts the user's signature â†’ silently self-bootstraps the user's wallet for a fake vaultId on the attacker's contract.

**Failure mode:** worst-case the same vault-binding-misconfig 3.1's L-domain-binding describes â€” but with the attacker's contract on the receiving end rather than the user's intent.

**Defense:** 3.3 cross-checks the loaded address against `EXPECTED_DEPLOYED_ADDRESS_BASE_SEPOLIA` (3.1's pinned const) per L4 ABOVE broadcasting â€” same check 3.1 already does pre-signing. Both 3.1 and 3.3 hit the SAME pinned const, so a TOCTOU-style "tamper between sign and broadcast" cannot redirect the broadcast to a different address than the sign was bound to. Three independent cross-checks at construction: chain-id, runtime-keccak, pinned-address. All MUST pass before the broadcast can proceed.

**Adversary leverage:** mitigated to "operator notices the construction error before publishing" â€” same as 3.1's defense.

## Threat-model touch points

3.3 adds a new entry to `THREAT_MODEL.md`'s per-component table: **"Direct-submit chain transport (EIP-1559; eth_sendRawTransaction)"** â€” DOCUMENTED (issue 3.3). The per-surface enumeration:

1. **Adversary controls the user's RPC endpoint.** Mitigated by L-rpc-spoof's defense (three cross-checks at construction + post-receipt event-field verification). Residual: a fully spoofing RPC that fakes all three checks is indistinguishable from a legitimate one; out-of-scope per "fully compromised network path" boundary.
2. **Adversary observes the pending tx in the mempool.** Same as 3.1's L-mempool-leak-of-vault-binding â€” D-006's known mitigation; Privacy Phase-2 territory. The mempool leaks the `vaultId` / `accountId` / `deviceId` BEFORE the tx mines; correlation with the user's secp256k1 address is observable once the tx is mined.
3. **Adversary causes a stuck pending tx in the user's mempool queue.** L-nonce-collision-DoS: only the user's own address is at risk; defense is bounded retries + clear error surface to operator. Tx replacement / cancel is MVP-3 territory.
4. **Adversary causes the user to burn gas via repeated reverts.** L-replay-after-revert: mitigated by classifying contract reverts as fatal (no retry).
5. **Malicious RPC reports inflated gas-price/baseFee to drain the device wallet.** L-gas-griefing: mitigated by the hard `max_fee_per_gas` cap (L6).
6. **Tampered deployment file redirects broadcast to a wrong contract.** L-deployment-mismatch-broadcast: mitigated by the pinned-address cross-check (3.1's pre-existing defense extended into 3.3's pre-broadcast check).

**Invariant additions: none new.** Existing invariants #1, #5, #11 cover 3.3's surface.
