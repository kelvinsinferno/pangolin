# Issue P11B: pangolin-cli vault create subcommand

> **Status:** PLAN-gate. Per Kelvin's autonomy directive, dispatching the
> build directly upon §16-Plan-Gate approval.
> **Security-critical:** YES per §16.3. P11B exposes
> `Vault::create(path, password)` — the primitive that materialises a
> fresh `.pvf` on disk, derives an Authority key from a brand-new
> password under `Argon2id::RECOMMENDED`, generates the Vault Data Key
> (VDK), and wraps it. A flaw in this surface either ships a vault
> protected by an empty/weak/leaked password (Cardinal Principle 2 +
> threat-model row #1's "wrap key derived through Argon2id" assumption),
> overwrites a user's existing vault (data-loss event indistinguishable
> from a deliberate "I'm starting over" gesture), or leaks the brand-
> new vault password through the same `ps`/`/proc`/scrollback channels
> P11A spent its A3 architectural decision blocking. P11B introduces
> no new cryptographic primitive and no new chain-side code; the entire
> issue is a clap-derive surface plus a thin wrapper over an existing
> library API, but the audit surface is the credential-input
> discipline at first-time provisioning.

## Spec reference

- **Master plan §3.7 EPIC: PoC scope.** P2 shipped the library API
  (`Vault::create(path, password)`); every prior issue's tests
  construct vaults via that library call (or via fixture helpers
  like `crates/pangolin-store/src/test_support.rs::open_test_vault`).
  P8 shipped the CLI binary (`status`, `publish`, `pull`); P9 added
  `resolve`; P11A added `account add/list/show/update/delete`.
  The CLI is missing the verb that creates a fresh `.pvf` from
  scratch — P11B closes that gap. The master plan does not name
  P11B explicitly; like P11A it is a **scope-promotion of P11's
  "non-author dev reproduces all three scenarios from `E2E_TESTS.md`
  alone"** exit criterion. P11A's plan §8 Q1 explicitly answered
  NO to "auto-create vault on `account add` against a missing
  `.pvf`"; the deferred mechanism (a separate `pangolin-cli vault
  create --vault-path <path>` verb) is what P11B ships.
- **Master plan §3.5 forbidden user-facing terms** ("gas",
  "blockchain", "transaction", "decentralized storage", "hashes",
  "revisions"). P11B's clap help text MUST avoid these. The
  subcommand surface uses "vault" / "password" / "create" / "path"
  — all consistent with the password-manager domain, none on the
  forbidden list. Internal doc-comments inside `commands/vault.rs`
  may use Pangolin-internal terms (Argon2id, VDK, AEAD, KDF) freely;
  the audit gate is `--help` output and `eprintln!`/`println!`
  strings.
- **DECISIONS.md:**
  - **D-001** — Rust core + thin shells. P11B is a **thin-shell
    feature** inside the existing Rust core repo. The shell layer
    (clap + rpassword + path canonicalization + `Vault::create`
    invocation) is mechanical; the security layer is the existing
    library API (`Vault::create` derives the authority + wraps the
    VDK). P11B introduces zero new primitives.
  - **D-013** — repository at `C:\Users\kelvi\Projects\pangolin`.
    P11B introduces a new module
    `tools/pangolin-cli/src/commands/vault.rs` and (optionally) an
    integration test under `tools/pangolin-cli/tests/`.
- **Cardinal Principle 1 ("Rust core is the only security-critical
  component; extension asks, core decides").** P11B's binary is a
  **thin asker**. Every password byte transits through
  `Vault::create(path, password)`; the CLI does not derive, salt,
  wrap, or persist anything beyond reading the password from the
  user's input source and handing it (as `SecretBytes`) to the
  library. The library API is the security boundary.
- **Cardinal Principle 2 ("No plaintext on disk; no plaintext leaves
  the device").** P11B's password input MUST read from
  `rpassword::prompt_password` (terminal, no echo) or stdin
  (machine-readable, opt-in via `--password-stdin`). P11B does NOT
  ship a `--password <flag>` form (P11A A3 discipline carried
  forward).
- **Cardinal Principle 4 ("Append-only state, never silent merge").**
  Not directly load-bearing for `vault create` — there is no prior
  state to append to. The append-only invariants apply to revisions
  inside the freshly-created vault, not to the `.pvf` file itself.
  P11B's "refuse to overwrite an existing file" discipline (§A3
  below) is the moral equivalent: a `.pvf` is created exactly
  once at a given path; subsequent `create` calls on the same path
  surface `StoreError::Io(ErrorKind::AlreadyExists)` from the
  library and the CLI refuses to clobber.
- **Whitepaper §6 (account model) / §7 (revision lineage).** Not
  directly load-bearing — `vault create` produces an empty schema
  with zero accounts and zero revisions. The freshly-created vault
  satisfies every pre-condition for `account add` to fire (the
  meta row is in place; the VDK is wrapped under the authority;
  the schema tables exist).

## What P0..P11A already shipped (the load-bearing surface for P11B)

P11B introduces no new cryptographic primitive, no chain-side code,
no schema column. It composes existing pieces:

- **P2 — `Vault::create(path: &Path, password: &SecretBytes) ->
  Result<Vault>`** (vault.rs L211–L273). Refuses overwrite via
  `path.exists()` check that returns
  `StoreError::Io(ErrorKind::AlreadyExists)` with a message naming
  the path. Acquires the sidecar `.lock` via `acquire_lock` (whose
  `OpenOptions::create_new(true)` write open is the
  cross-process race-free guard). Opens a fresh SQLite WAL
  connection and applies the schema. Generates a 32-byte random
  `vault_id` via SQLite's `randomblob`. Generates a fresh
  `KdfSalt` (32 bytes from `OsRng`) and uses
  `KdfParams::RECOMMENDED` (256 MiB, t=3, p=1). Derives the
  authority seed via `kdf::derive_seed`. Constructs a fresh
  `AuthorityKey::from_seed`. Generates a fresh `VdkKey`. Wraps
  the VDK under the authority via `vdk.wrap(&authority,
  &wrap_ctx)`. Writes the `VaultMeta` row
  (`vault_id`, `created_at`, `kdf_params`, `kdf_salt`,
  `wrap_context`, `wrapped_ciphertext`, `wrapped_nonce`). Drops
  the in-memory authority + VDK so the next op exercises the
  unlock path. Returns `Vault` in `SessionState::Locked`. On any
  internal error, releases the lock AND removes the partially-
  created vault file via `std::fs::remove_file` (so the next
  `create` invocation can succeed).
- **P2 — `Vault::vault_id() -> [u8; VAULT_ID_LEN]`** (vault.rs
  L359–L361). 32-byte content-addressed identifier of the freshly-
  created vault. P11B's `--print-id` flag forwards this value
  hex-encoded to stdout.
- **P2 — `acquire_lock(vault_path) -> Result<File>`** (vault.rs
  L163–L191). Internal helper (not public). Opens
  `<vault_path>.lock` with `create_new(true)`; the second
  concurrent `create` observes the lock file and returns
  `StoreError::AlreadyOpen`. Combined with the `path.exists()`
  guard at the top of `Vault::create`, this gives two layers of
  defense against concurrent provisioning races (§A8).
- **P8 — `tools/pangolin-cli/src/cli.rs` clap-derive scaffold.**
  P11B extends `Cli::command` with one new variant (`Vault`)
  and adds a nested `VaultArgs` / `VaultCommand` enum with a
  single `Create(VaultCreateArgs)` variant initially. The same
  pattern P11A used for `Account(AccountArgs)`.
- **P8 — `tools/pangolin-cli/src/main.rs` dispatch.** P11B adds
  one new arm (`Command::Vault(sub) => commands::vault::run(...)`)
  to the existing `match args.command` block.
- **P11A — `tools/pangolin-cli/src/commands/account.rs::
  read_secret_first_line_from_stdin`** (account.rs L219–L243). The
  exact stdin-read primitive P11B reuses for `--password-stdin`
  (one-line, CRLF-aware, `SecretBytes`-zeroizing). P11B also
  reuses `reject_empty_password` (account.rs L277–L282) for the
  empty-password guard — see §A4 for the cross-module reuse
  decision.
- **P11A — `tools/pangolin-cli/src/commands/account.rs::
  prompt_password_with_confirmation`** (account.rs L445–L471). The
  exact double-prompt-with-bounded-retry primitive P11B reuses for
  the interactive password path. The bounded-retry budget
  (`PASSWORD_RETRY_BUDGET = 2`, three total typing attempts) is
  the same.
- **P11A A3 — Password-input discipline.** P11B inherits the three-
  path discipline (interactive prompt + confirmation; stdin opt-in;
  NO flag form) unchanged. See §A2 below.
- **P11A A16 — Forbidden-terms discipline.** P11B's `--help` text
  is parsed by the same `account_help_avoids_forbidden_user_facing_terms`
  test pattern (extended to walk `vault --help` and `vault create
  --help` — see §A14).
- **P8 fix MED-3 — `canonicalize_vault_path`** (vault_open.rs L31–
  L38). P11B does **NOT** call this on the `--path` argument
  because `canonicalize` requires the file to exist; the file we
  are about to create does not. P11B canonicalizes the parent
  directory instead — see §A5.

P11B introduces **one new module** (`tools/pangolin-cli/src/commands/vault.rs`),
**one new clap variant** (`Command::Vault(VaultArgs)`) on the
existing `Cli::command` enum, **one nested sub-subcommand enum**
(`VaultCommand::Create(VaultCreateArgs)`), and **zero new exports
in `pangolin-store`**, **zero new public types in
`pangolin-crypto`**, **zero new `Cargo.toml` dependencies**
(clap + rpassword + anyhow + hex are already present).

## Goal

Ship one new `pangolin-cli vault create --path <path>` subcommand
that exposes `Vault::create` at the user-facing CLI boundary,
preserving P11A's A3 password-input discipline (interactive prompt
+ confirmation OR `--password-stdin`; never `--password <flag>`).
Closes the structural gap "Pangolin's CLI cannot create a vault" so
P11's reproducer guide can drive a non-author developer through
`vault create` → `account add` → `publish` → `pull` without
bespoke fixture scaffolding (the `cargo run --example` /
`open_test_vault` escape hatches are no longer required for the
non-author dev's first interaction). Introduces no new
cryptographic primitive, no new chain-side code, no new vault-
schema column, no new public library API.

## Architectural decisions (locked in this plan)

These decisions are committed to as part of plan-gate approval. Any
deviation during BUILD requires returning to PLAN per §16.4.

### A1. Subcommand hierarchy — nested `vault create`, not flat `create-vault`

**Decision:** `create` lives under a nested `pangolin-cli vault
create` namespace, not as a flat top-level subcommand
(`pangolin-cli create-vault`). The existing flat verbs (`status`,
`publish`, `pull`, `resolve`) stay flat — same rationale as P11A
A1. The nested form mirrors P11A's `pangolin-cli account <verb>`
hierarchy.

**Rationale:**

1. **Convention alignment with P11A.** P11A's `account <verb>`
   hierarchy is the precedent; deviating for `vault` (flat
   `create-vault` vs. nested `vault create`) introduces avoidable
   asymmetry. Future MVP-1 verbs (`vault open`, `vault info`,
   `vault destroy`, `vault rotate-password`, `vault export`,
   `vault import`) slot into the same nested namespace cleanly.

2. **Top-level namespace conservation.** Flat `create-vault` would
   collide semantically with what should plausibly become a `create-account`
   verb; reserving the unqualified `create` overcommits the top-
   level surface. (`account add` already takes the
   account-creation slot under the nested noun namespace.)

3. **Help-text scalability.** `pangolin-cli vault --help` shows
   `create` (and any future `vault open` / `vault info`) in one
   place. Discoverability is local to the noun.

**Alternative considered:** Flat `pangolin-cli create
--vault-path <path>`. **Rejected** — see §1 + §2.

**Alternative considered:** Both forms via clap aliases
(`vault create` AND `create-vault` both work). **Rejected** —
every alias is documentation debt + a discoverability footgun.
Pick one.

### A2. Password input — interactive prompt + double-confirm; `--password-stdin` for CI; NO `--password <flag>`

**Decision:** Vault password input on `vault create` follows two
exclusive paths in strict precedence:

1. `--password-stdin` (read first line of stdin; trim trailing CR
   then LF; NO confirmation — CI/script use case; reject empty
   per §A4).
2. (default) Interactive prompt via `rpassword::prompt_password("Vault password: ")`
   followed by a confirmation prompt
   (`"Confirm vault password: "`); both reads return ordinary
   `String` buffers from `rpassword` (acknowledged PoC limitation
   — `rpassword`'s `String` is not `Zeroizing`-wrapped; see §A12);
   the comparison runs in **non-constant-time** because the two
   strings are user-supplied and equal (the timing only leaks
   "user typed the same string twice" not anything about a
   secret); on mismatch, re-prompt up to `PASSWORD_RETRY_BUDGET = 2`
   retries then abort with `bail!("password mismatch; aborting after 2 retries")`
   (consistent with `account add`'s behavior).

There is **NO** `--password <flag>` form. The flag form is a
process-listing leak (`ps aux` shows the password in argv) and
Pangolin refuses to ship that footgun even for "convenience." Same
discipline as P11A A3.

**Note:** Unlike `account add`, `vault create` does NOT ship a
`--generate-password` mode. The vault password is the user's
master credential; auto-generating it is a UX anti-pattern (the
user needs to remember + type this password every time they
unlock the vault). `account add`'s generate-password makes sense
because per-account passwords are stored inside the vault and
reveal-prompted on demand; the vault password is the user's
external-memory burden. Decision **NOT** to ship
`--generate-password` for vault creation; document explicitly
in §7 (out of scope).

**Rationale:**

1. **`ps`/`/proc` leak is the canonical flag-form bug.** P11A
   already locked this in for `account add`'s password input;
   `vault create`'s password is a higher-tier secret (one
   compromise unlocks every account in the vault). The flag
   form is a louder anti-pattern here than at `account add`.

2. **`--password-stdin` is the right CI shape.** A script can
   `echo "my-vault-pwd" | pangolin-cli vault create --path /tmp/v.pvf
   --password-stdin`; the password is in the script (under the
   user's own filesystem permissions) but not in argv or env.
   Stdin is redirected by the user, not parsed by clap.

3. **Confirmation re-prompt loop bounded at 2 retries (3 attempts
   total).** Same bound as `account add`; an unbounded retry is a
   footgun and three attempts is enough to catch caps-lock or
   keyboard-layout typos.

**Alternative considered:** `--password <flag>` with a deprecation
warning. **Rejected** — see P11A A3.

**Alternative considered:** Read password from a `--password-file
<path>` argument. **Rejected** — same reasoning as P11A A3
(stdin redirection is the standard mechanism with clearer
auditability).

### A3. Path-overwrite discipline — refuse-by-default; pre-flight + library-side double-check; no `--force`

**Decision:** `vault create` REFUSES to overwrite an existing file
at `--path`. Two layers of defense:

1. **Pre-flight check at the CLI boundary.** Before reading the
   password, `run_create` calls `path.exists()`; if true, exit
   with `"vault file already exists at <canonical-path>; refusing
   to overwrite"` BEFORE prompting. Saves the user a wasted
   password entry against a path that would fail anyway.
2. **Library-side check inside `Vault::create`.** Already shipped
   (vault.rs L214–L219). Returns
   `StoreError::Io(ErrorKind::AlreadyExists)`. The CLI maps this
   error variant to the same human-readable message as the
   pre-flight check.

P11B does **NOT** ship `--force` (overwrite the existing file).
The append-only spirit of Pangolin (Cardinal Principle 4) extends
to vault file creation: a `.pvf` is created exactly once at a
given path; deletion-then-recreation is a separate flow that the
user does explicitly via `rm <path>` then re-`vault create`.
A `--force` flag would let an attacker (or a script bug) silently
clobber the user's vault in a single command.

**Rationale:**

1. **Pre-flight saves a wasted password entry.** Without the
   pre-flight check, the user types a fresh password (twice if
   interactive), then sees "vault file already exists" — frustrating
   UX and a (minor) opportunity for the typed password to leak via
   keystroke buffering. Pre-flight surfaces the failure earlier.

2. **Library-side check is the race-safe fallback.** A TOCTOU
   race between the pre-flight `path.exists()` and the
   subsequent `Vault::create` call could let two concurrent
   `create` invocations both pass the pre-flight; the library's
   `path.exists()` + `acquire_lock`'s `create_new(true)` close
   the race (the second invocation hits `AlreadyOpen` from the
   lock acquisition or `AlreadyExists` from the file open).
   See §A8 for the full race analysis.

3. **No `--force` matches `git`'s discipline.** `git init` does
   not overwrite an existing `.git` directory; the user does
   `rm -rf .git && git init` if they want to start over. Same
   model.

**Alternative considered:** `--force` flag that overwrites
existing files. **Rejected** — see §3 above. MVP-1 may revisit if
user feedback complains, but under PoC the explicit-then-rerun
discipline is correct.

**Alternative considered:** Skip the pre-flight check; rely
solely on the library-side check. **Rejected** — see §1.

### A4. Cross-module reuse of `account.rs`'s password-prompt primitives

**Decision:** P11B reuses three helpers from `commands/account.rs`
verbatim:

- `read_secret_first_line_from_stdin` (account.rs L219–L243) — for
  `--password-stdin`.
- `reject_empty_password` (account.rs L277–L282) — for the empty-
  password guard.
- `prompt_password_with_confirmation` (account.rs L445–L471) — for
  the interactive prompt + confirmation path.

**Implementation choice:** make these helpers `pub(crate)` (or
`pub(super)` if a `commands/shared.rs` module is preferred) so
`commands/vault.rs` can call them. Document the cross-module
dependency in `commands/account.rs`'s module-level comment
(adding a one-line note: "These helpers are reused by
commands/vault.rs::run_create — keep their semantics stable.").

**Rationale:**

1. **DRY discipline.** Re-implementing the helpers in `vault.rs`
   would diverge the password-input semantics across two surfaces.
   The bounded-retry budget, empty-password guard, and stdin
   line semantics MUST be identical; reuse is the correct path.

2. **Test surface stays consolidated.** P11A's tests for the
   stdin-line semantics, the empty-password rejection, and the
   bounded-retry behavior already pin the helpers' contracts.
   P11B's tests assert that `vault create` calls these helpers
   on the right paths; the helpers' internals stay covered by
   P11A's tests.

3. **Audit-friendly.** A reviewer scanning P11B for password-
   input discipline immediately recognises the helpers from P11A
   (same names, same semantics) and does not need to re-audit
   the implementation.

**Alternative considered:** Move the helpers to a shared module
(`tools/pangolin-cli/src/commands/secret_input.rs` or
`tools/pangolin-cli/src/secret_input.rs`). **Acceptable but not
required for P11B** — the cross-module visibility change
(`pub(crate)` on the existing helpers) is the minimal viable
shape; a refactor to a dedicated module is a separate change.
**Recommend the minimal-visibility-change path for P11B**;
revisit at SIGNOFF if the audit reviewer flags the cross-module
import as un-clean.

**Alternative considered:** Duplicate the helpers verbatim in
`vault.rs`. **Rejected** — see §1.

### A5. Path canonicalization — canonicalize the parent dir; do NOT canonicalize the (non-existent) target

**Decision:** `--path <PathBuf>` is processed as follows:

1. Extract `parent = path.parent().ok_or_else(...)` and
   `file_name = path.file_name().ok_or_else(...)`. If `--path` is
   `/`, has no parent, or has no file-name component (e.g., ends
   in `/` on POSIX), surface a clear error
   (`"--path must name a vault file (got <path>)"`).
2. Canonicalize the parent: `parent.canonicalize().with_context(...)`.
   This surfaces the absolute path of the parent directory and
   resolves any `..` / symlinks; it ALSO errors out if the parent
   directory does not exist (`canonicalize` returns `NotFound` on
   a missing intermediate component). Surface the error as
   `"could not canonicalize parent directory of <path>: <error>"`
   and exit before prompting.
3. Re-join: `let canonical_path = canonical_parent.join(file_name);`.
   This is the path passed to `Vault::create` and printed in any
   user-facing message.

**Rationale:**

1. **`canonicalize` requires the file to exist** (Rust docs:
   "this function will return an error if `path` does not exist,
   or is not accessible"). The file we are creating does not
   exist; calling `canonicalize` on the target itself errors out
   with `NotFound`. P8's MED-3 fix (`canonicalize_vault_path`)
   applies only to existing-file paths; `vault create` needs a
   different shape.

2. **Canonicalizing the parent is the right shape.** The parent
   directory must exist (we're creating a file inside it); its
   canonicalization surfaces relative-path traversal (`./vault.pvf`
   → absolute) and symlink redirection (`./vault-link.pvf` where
   `vault-link.pvf` is a symlink to `/elsewhere/vault.pvf` is
   resolved). The file-name component itself does not need
   canonicalization.

3. **The pre-flight `path.exists()` check (§A3) uses the
   re-joined canonical path** so a user passing `./v.pvf` from
   one cwd vs. `/abs/path/v.pvf` against the same target hits
   the same overwrite-refuse outcome.

**Alternative considered:** Skip canonicalization entirely; pass
the raw `--path` to `Vault::create`. **Rejected** — leaves
relative-path / symlink redirection undisplayed in the success
message; user passing `./v.pvf` from a working directory other
than the one they think sees `created at ./v.pvf` and cannot
verify the absolute location without inspecting the cwd.

**Alternative considered:** `path.canonicalize()` on the target;
fall back to `parent.canonicalize() + file_name` only on
`NotFound`. **Acceptable but adds a branch.** The plain "always
canonicalize parent" path is simpler and equivalent under the
overwrite-refuse discipline (a target that does not exist hits
the parent-canonicalize path; a target that does exist is caught
by the pre-flight `path.exists()` check before any canonicalize
call).

**Subtle correctness concern acknowledged:** the parent-canonicalize
shape means a `--path` targeting a symlinked parent directory
shows the resolved path in success output; users running with
`/var/lib/pangolin -> /opt/pangolin/vaults` (symlink) and passing
`--path /var/lib/pangolin/v.pvf` see `created at /opt/pangolin/vaults/v.pvf`
in the success message. This is **deliberate** — the resolved
path is what was actually written; the user's symlink convention
is preserved in their next-invocation commands but the audit
trail names the canonical location.

### A6. KDF parameters — ship `KdfParams::RECOMMENDED` only; no `--kdf-params` selector

**Decision:** P11B's `Vault::create` call always uses
`KdfParams::RECOMMENDED` (256 MiB memory, t=3, p=1) — the same
value `Vault::create` already hard-codes (vault.rs L228). P11B
does NOT introduce a `--kdf-params <high|recommended|lite>`
selector at the CLI surface; if the library ever exposes an
override, MVP-1 picks up the CLI-surface plumbing.

**Rationale:**

1. **`Vault::create` does not accept a `KdfParams` argument.** The
   library API hard-codes `RECOMMENDED`. To introduce a CLI-side
   selector, P11B would have to also extend the library API
   (`Vault::create_with_params(path, password, params)`) — that
   is new public surface in `pangolin-store` requiring its own
   threat-model row + audit pass. Out of P11B's surgical scope.

2. **`RECOMMENDED` is the right PoC default.** 256 MiB / t=3 / p=1
   is the standard guidance from the Argon2 paper; matches what
   every PoC vault file already on disk uses. Shipping a
   `--kdf-params lite` mode at PoC introduces the footgun of
   "this vault is provisioned with weaker KDF params than
   `recommended` because the user clicked through a flag they
   didn't understand"; the tunable surface is MVP-1 polish, not
   PoC scope.

3. **Single-mode keeps audit-cost low.** Every vault produced by
   `pangolin-cli vault create` has identical KDF parameters at
   the meta row. A future `cargo audit-equivalent` script can
   assert this invariant trivially.

**Alternative considered:** `--kdf-params <preset>` selector (high
= 1 GiB / t=4 / p=1; recommended = 256 MiB / t=3 / p=1; lite =
64 MiB / t=2 / p=1 — for low-resource devices). **Rejected** for
PoC; flag at MVP-1 if low-resource-device support becomes a
priority. Also requires extending `Vault::create` first —
out of scope.

**Alternative considered:** Auto-detect the user's machine memory
and pick params accordingly. **Rejected** — the CLI runs in
diverse environments (CI containers with 512 MiB RAM; dev
laptops with 32 GiB); auto-detection introduces a non-deterministic
parameter choice that complicates the threat model
("what KDF strength does my vault use?") for a marginal UX gain.

### A7. `--print-id` flag — opt-in vault_id display; default off for minimum surprise

**Decision:** `--print-id` is an OPT-IN flag. Default behavior:
print only `"vault created at <canonical-path>"` to stdout. With
`--print-id`: also print `"vault_id: <64-char-lowercase-hex>"` on
a second stdout line.

**Rationale:**

1. **Default-quiet matches Unix convention.** `git init` prints
   `Initialized empty Git repository in <path>` only; no SHA, no
   tree hash. The minimal-output default makes scripted
   pipelines that grep for the path easy. The vault_id is
   useful for tracking but rarely needed in the default flow.

2. **JSON mode (`--json`) emits `vault_id` unconditionally** (per
   the table at the top of §3.7). The structured-output path
   already differentiates between "minimal human-friendly text"
   and "machine-readable bundle"; the human path's default is
   `--print-id` off.

3. **The vault_id is hex(content-addressed; fresh randomness from
   SQLite's `randomblob`)**; no security risk in printing it (it's
   a public identifier — the `vault_id` is part of every on-chain
   event and every revision's AAD). Default-off is purely a
   minimum-surprise UX choice, not a security choice.

**Alternative considered:** Default `--print-id` on, with
`--quiet` flag to suppress. **Rejected** — flips the polarity
from "minimal default" to "verbose default"; the
git-init / kubectl-create / openssl-genrsa convention is
minimal default, opt-in verbose.

**Alternative considered:** Always print the vault_id. **Rejected**
— see §1 (default-quiet is the convention).

### A8. Concurrent-create race — already covered by `Vault::create`'s lock + AlreadyExists

**Decision:** P11B's CLI does **NOT** add an additional lock on top
of `Vault::create`. The library's existing two-layer defense is
sufficient:

1. `path.exists()` check at the top of `Vault::create` (vault.rs
   L214) returns `StoreError::Io(AlreadyExists)` if the file is
   already there.
2. `acquire_lock(vault_path)` (vault.rs L221) opens
   `<vault_path>.lock` with `OpenOptions::create_new(true)`; the
   second concurrent invocation hits `StoreError::AlreadyOpen`.

A TOCTOU window exists between the CLI's pre-flight `path.exists()`
check and the library's check, but it is closed by the
`create_new(true)` write open inside `acquire_lock`. Two CLI
processes racing to create the same vault both see `false` from
their respective pre-flights; both call `Vault::create`; one
wins the `create_new(true)` lock acquisition, the other gets
`AlreadyOpen`; the loser's `create` returns the error and the
file is removed (via vault.rs L267–L271's "remove the partially-
created vault file" cleanup).

**Subtle correctness concern acknowledged:** the loser's cleanup
(`std::fs::remove_file(path)`) is best-effort — if the loser
created the SQLite WAL files but failed before the lock release,
the partial files may persist. **The next legitimate `create` on
the same path** observes `path.exists() == true` and refuses;
the user must `rm <path>` manually before retrying. This is
**acknowledged operational hazard, not a security failure**;
the inherited P2 threat-model row #5 ("Concurrent corruption
from two opens") covers this.

**Rationale:**

1. **Adding a CLI-side lock would be redundant and introduce
   its own failure modes.** Two locks are worse than one; the
   library's lock is the source of truth. P11B inherits the
   discipline unchanged.

2. **The pre-flight check is a UX optimization, not a race
   defense.** It saves a wasted password entry in the
   non-racing case; the actual race is closed by the library's
   `create_new(true)`.

**Alternative considered:** CLI-side `flock`-on-the-parent-dir
guard. **Rejected** — overengineering for PoC; the library's
discipline already covers this.

### A9. Vault password vs. account password — different credentials; clear `--help` text

**Decision:** The `--help` for `vault create --path <path>`
explicitly distinguishes the vault password from per-account
passwords. Help text:

```
Vault password — protects the vault as a whole. This is the
password the user types every time they run `pangolin-cli account
add` / `list` / `show` / etc., and is the master credential for
this vault. NOT the same as the per-account passwords stored
INSIDE the vault. Argon2id-derived (256 MiB memory, t=3, p=1).
```

**Rationale:**

1. **First-time-user confusion is a documented UX failure mode.**
   A user creating their first vault and being prompted for "the
   password" might enter their email password, the password they
   intend for the first credential entry, or a master-password
   they will not remember. Spelling out "this is the password
   for the vault as a whole; not for any account inside it" in
   the prompt + `--help` text is the load-bearing UX defense.

2. **The interactive prompt also includes this clarifying text.**
   `prompt_password_with_confirmation` is reused from `account.rs`,
   but `vault create` precedes the call with an `eprintln!`
   describing the role of the password being requested:
   `"Set a password for the new vault. Argon2id will derive a key
   from this password; the vault password protects every account
   inside it."`. (Implementation note: the helper takes a
   `prompt_label: &str` parameter or has a sister helper that
   takes a label; if `prompt_password_with_confirmation` is
   used unchanged, the contextual `eprintln!` precedes the
   `rpassword` call.)

3. **`--help` text avoids the §3.5 forbidden terms.** "Argon2id"
   is a developer term, not a user-facing forbidden term —
   `--help` may use it (it's in the same domain as "vault" and
   "password", not in the `gas`/`blockchain`/`transaction`
   namespace). The §3.5 audit gate is the user-visible help
   string; "Argon2id" appearing there is acceptable per A16's
   precedent (`account add --help` references "presence
   confirmation" without §3.5 violation).

**Alternative considered:** Drop the clarifying text, rely on the
prompt label `"Vault password: "` to convey the meaning.
**Rejected** — "Vault password" is ambiguous to first-time users
who do not yet know what a vault is. The `--help` paragraph is
the load-bearing documentation.

### A10. Error UX — clear, actionable messages for every failure path

**Decision:** `vault create` produces one of these user-facing
error messages, each surfaced to stderr with a non-zero exit code:

- **Existing path:** `"vault file already exists at <canonical-path>; refusing to overwrite"`. Exit 1.
- **Parent dir missing / inaccessible:** `"could not canonicalize parent directory of <raw-path>: <io-error>"`. Exit 1.
- **`--path` is `/` or has no file-name:** `"--path must name a vault file (got <raw-path>)"`. Exit 1.
- **Empty password from stdin:** `"password must not be empty"`. Exit 1.
- **Empty password after interactive prompt:** `"password must not be empty"`. Exit 1.
- **Password mismatch after retry budget exhausted:** `"password mismatch; aborting after 2 retries"`. Exit 1.
- **Library-side `Io` error (parent-dir not writable, disk full, etc.):** `"could not create vault at <canonical-path>: <io-error>"`. Exit 1.
- **Library-side `Sqlite` error (extremely rare; would indicate environmental corruption):** `"vault provisioning failed at <canonical-path>: <sqlite-error>"`. Exit 1.
- **Library-side crypto error (Argon2id derivation failure — should not happen in practice):** `"vault key derivation failed at <canonical-path>: <error>"`. Exit 1.

**Rationale:**

1. **Distinguishability avoids user confusion.** Each error
   message points at a specific fix: "rm the existing file" for
   AlreadyExists; "mkdir the parent" for parent-missing; "check
   disk space" for Io; "retype carefully" for password mismatch.

2. **The canonical path is in every error.** The user passing
   `./v.pvf` from a wrong cwd sees the absolute location in the
   error, matching P8 fix MED-3's discipline.

3. **Exit code 1 (not 2) for all failures.** Rust's `anyhow`
   default is exit-1 on `Result::Err` from `main`; clap-arg-
   parse failures (e.g., missing `--path`) are exit-2 from clap.
   The split is conventional.

**Alternative considered:** Surface the raw `StoreError::Display`
output verbatim. **Rejected** — `StoreError`'s display strings
are developer-facing ("AEAD authentication failed at offset
0x40"); the user-facing surface needs domain-translated
messages.

### A11. Lock-file persistence — `Vault::create`'s sidecar lock is released on `Vault::drop`

**Decision:** P11B does NOT explicitly call `Vault::close` after
`Vault::create` returns successfully. The library's `Drop` impl
on `Vault` releases the sidecar `.lock` file via
`release_lock(path)` (called from `Drop` per the library's
convention; verified at BUILD by reading `vault.rs::Drop`
implementation). The CLI's `vault create` lets the `Vault`
binding go out of scope at the end of `run_create`; the lock
file is cleaned up automatically.

**Rationale:**

1. **Symmetry with `commands/account.rs`'s pattern.** P11A's
   `run_add` calls `vault.close().context("Vault::close failed")?`
   explicitly (account.rs L184); the same explicit close is
   correct for `vault create`. **Reconsidered:** if `Vault::close`
   is the public way to close, use it for symmetry. The relevant
   audit observation is that `Vault::create` returns a Locked
   vault; calling `.close()` on a Locked vault is a no-op for
   session state (no active session to drop) but still releases
   the lock + closes the SQLite connection deterministically.
   **Decision:** call `vault.close()` explicitly after a
   successful `Vault::create` for parity with P11A's
   discipline.

2. **Explicit `close` makes the lock-release ordering visible
   in the source.** A reviewer reading `run_create` sees
   `Vault::create` → `vault.close()` → exit; no implicit
   destructor work to reason about.

**Alternative considered:** Implicit `Drop` via end-of-scope.
**Rejected** — works correctly but obscures the close ordering
relative to the success-print and exit. Explicit is better.

### A12. `rpassword`'s un-zeroized `String` — acknowledged PoC limitation, no test seam

**Decision:** P11B inherits the same PoC limitation P11A
documented at A13: `rpassword::prompt_password` returns an
ordinary `String` whose heap buffer may persist in memory after
the `String` is dropped (the underlying `Vec<u8>` is freed but
not zeroized). The plaintext password copy in `SecretBytes` is
the authoritative owner; the `rpassword`-returned `String` is a
transient that we cannot zeroize without changing the upstream
crate.

**Rationale:**

1. **Same exposure as P11A's password input.** P11A explicitly
   acknowledged this in `prompt_password_with_confirmation`'s
   doc-comment (account.rs L457–L463). P11B does not introduce
   a new exposure; it inherits the existing one.

2. **MVP-1 may pin a `Zeroizing<String>`-returning fork or wrapper.**
   Out of scope for P11B; flagged in §7 as a known PoC
   limitation.

3. **The `--password-stdin` path is exposure-free** (the bytes
   flow directly into `SecretBytes` from stdin without
   intermediate `String` materialization). Users who care about
   the exposure can use `--password-stdin <<<"..."` even for
   first-time vault creation.

**Alternative considered:** Wrap `rpassword` in a `PromptIo`
trait with a `RealPromptIo` and `MockPromptIo` (test seam);
implement zeroization in `RealPromptIo`. **Rejected** for the
audit-cost reason in P11A A13 + the upstream-crate-limitation
that wrapping does not actually zeroize the `String` returned
by `rpassword`.

### A13. Interaction with `account add`'s implicit "vault must exist" assumption

**Decision:** P11B does NOT change `account add`'s "the vault
must already exist" semantics. The user's flow is now:

```
pangolin-cli vault create --path <new.pvf>            # NEW
pangolin-cli account add --vault-path <new.pvf> ...   # P11A
```

`account add` continues to error out (via `open_and_unlock` →
`canonicalize_vault_path` → `NotFound`) if the vault does not
exist. P11A's plan §8 Q1 explicitly answered NO to "auto-create
on `account add`"; P11B preserves that decision.

**Rationale:**

1. **Separation of concerns.** Vault creation is a one-time
   operation with a different security shape (set the master
   password) than account addition (use the master password to
   authenticate, then store a per-account credential). Each
   verb has clear, separate semantics.

2. **The error message from `account add` against a missing path
   is friendly enough.** P8 fix MED-3's
   `canonicalize_vault_path` error includes the canonicalized
   path and a "file missing? broken symlink?" hint; the user
   reads this and runs `pangolin-cli vault create --path <path>`.
   No need to conflate the two operations.

3. **MVP-1 may add a `--init-vault` convenience flag on `account add`**
   if user feedback shows the two-step flow is friction. Under
   PoC + documented reproducer guide, the explicit two-step is
   the discoverable path.

**Alternative considered:** Auto-create on `account add` (P11A's
deferred Q1). **Already rejected by P11A** — see P11A plan §8
Q1.

### A14. Forbidden user-facing terms

**Decision:** No P11B user-facing copy introduces any §3.5 forbidden
term ("gas", "blockchain", "transaction", "decentralized storage",
"hashes", "revisions"). The one verb (`vault create`) and its
flag descriptions use "vault" / "password" / "create" / "path" —
all in the password-manager domain. Internal doc-comments inside
`commands/vault.rs` may use Pangolin-internal terms (Argon2id, KDF,
VDK, AEAD); the §3.5 gate is `--help` output and `eprintln`/`println`
strings.

**Verification:** the existing `account_help_avoids_forbidden_user_facing_terms`
test (cli.rs L1183–L1233) is extended (or duplicated as
`vault_help_avoids_forbidden_user_facing_terms`) to walk
`vault --help` and `vault create --help`. Implementation:
`cmd.find_subcommand_mut("vault")` → render help → grep for the
forbidden strings.

**Rationale:** §3.5 is a user-facing-string gate; `cargo doc` is
a developer surface and uses internal terminology freely. P11B
inherits the same discipline as P11A A16.

### A15. Threat-model row numbering

**Decision:** P11B appends a single new threat row (row 28) to

## Threat model row (draft text for THREAT_MODEL.md)

The existing `pangolin-cli` row in `THREAT_MODEL.md` covers
publish + pull (P8), resolve (P9), tombstones + offline (P10),

to the same row. The new threat index (placeholder `<N>`;
expected 28 per §A15):

> <N>. **Vault provisioning password leak (process listing,
>      shell history, scrollback) AND `.pvf` overwrite hazard.**
>      Defense (multi-fold): `pangolin-cli vault create` REFUSES
>      to ship a `--password <flag>` argument form. The only
>      paths to provide a vault password are interactive
>      terminal prompt (`rpassword::prompt_password`, no echo)
>      with confirmation re-prompt and bounded retry budget
>      (2 retries; 3 attempts total) or stdin
>      (`--password-stdin`, redirected by the user). The same
>      shape `pass init` and `bw create` use. The empty-
>      password guard (`reject_empty_password` from P11A's
>      MED-1 fix, reused per §A4) fires on both paths before
>      any library call. The vault-creation password is the
>      master credential for the new vault — its leak
>      compromises every account stored inside; the
>      input-discipline carries P11A's A3 weight forward to
>      this higher-tier secret. Path-traversal handling:
>      `--path` is processed via `parent.canonicalize() +
>      file_name` (per §A5; note `Path::canonicalize` requires
>      file existence, which the not-yet-created target lacks);
>      relative-path traversal and symlink redirection on the
>      parent surface as the canonical absolute path in the
>      success message and any error message, matching P8
>      fix MED-3's discipline. Overwrite refusal: a
>      pre-flight `path.exists()` check at the CLI boundary
>      (skips the password prompt if the target is already
>      occupied) AND the library's `Vault::create`'s
>      `path.exists()` + `acquire_lock`'s
>      `OpenOptions::create_new(true)` write open close the
>      TOCTOU race between the pre-flight and the library
>      call (per §A8); two concurrent `vault create` calls
>      against the same path produce one `Created` and one
>      `AlreadyExists`/`AlreadyOpen` cleanly, with the
>      loser's partial-file cleanup performed by
>      `Vault::create`'s existing
>      `std::fs::remove_file(path)` on-error path (vault.rs
>      L267–L271). Audit reviewers should scrutinize: does
>      the CLI's pre-flight surface the canonical absolute
>      path in the "vault file already exists" error (so a
>      user passing `./v.pvf` from a wrong cwd can spot the

>      `rpassword`-returns-unzeroized-`String` PoC limitation
>      (§A12) unchanged; `--password-stdin` is the
>      exposure-free path. KDF parameters are hard-coded
>      `KdfParams::RECOMMENDED` (256 MiB / t=3 / p=1) per

## Failure modes considered

| Failure mode | Mitigation |
|---|---|
| User runs `pangolin-cli vault create` with `--password <flag>` form | A2 — clap rejects (the flag does not exist); the user reads `--help` and switches to `--password-stdin` or interactive prompt. |
| User pipes the wrong password to `--password-stdin` | A2 — `vault create` requires only one password (no confirmation in the stdin path because confirmation defeats the scripted-pipe purpose); user retries by deleting the (now wrongly-passworded) vault and re-creating. **Note:** the wrongly-passworded vault is locked + unrecoverable from the user's POV; the rm-then-recreate flow is the only recovery. Document explicitly in §7 and in the `--help`. |
| User runs `vault create` against an existing path | A3 — pre-flight `path.exists()` check refuses before prompting; user reads "vault file already exists" + path; user runs `rm <path>` then re-runs `vault create`. |
| Two concurrent `vault create` invocations against the same fresh path | A8 — one wins via `acquire_lock`'s `create_new(true)`; the loser's partial vault file is cleaned up by `Vault::create`'s on-error path. User retries (loser sees `AlreadyOpen` or `AlreadyExists` cleanly). |
| User runs `vault create` with a `--path` whose parent dir does not exist | A5 + A10 — `parent.canonicalize()` returns `NotFound`; CLI surfaces `"could not canonicalize parent directory of <path>: ... NotFound"` with the original `--path` echoed. User runs `mkdir -p <parent>` then re-runs. |
| User runs `vault create --path /` or `vault create --path /tmp/` (trailing slash, no file_name) | A10 — `path.file_name()` returns `None`; CLI surfaces `"--path must name a vault file (got <path>)"`. User retries with a proper file path. |
| User runs `vault create` and types two different passwords at the prompt | A2 — bounded retry (`PASSWORD_RETRY_BUDGET = 2`) + abort. User re-runs. |
| User runs `vault create --print-id --json` | A7 — JSON mode emits `vault_id` unconditionally; `--print-id` is a no-op in JSON mode (documented in `--help` text). The flag's intent is for human-readable output. |
| User pipes empty stdin to `--password-stdin` | A2 + A4 — `reject_empty_password` fires before the library call; CLI surfaces `"password must not be empty"` and exits non-zero. |
| User runs `vault create --path /tmp/foo.pvf --password-stdin` and the stdin pipe contains a multi-line password (e.g., a heredoc) | A2 + A4 — `read_secret_first_line_from_stdin` reads up to the first LF; subsequent lines are ignored (left in stdin). The user gets a vault encrypted under the first-line password; if that does not match their intent, they re-run. Document the first-line semantics in `--help`. |
| User's disk is full / parent dir is read-only | A10 — library-side `StoreError::Io(_)` surfaces with the underlying io-error message; CLI displays "could not create vault at <path>: <io-error>". User fixes the disk / permissions. |
| User runs `vault create` against a path where the parent is canonicalize-resolvable but on a read-only filesystem | Same as above — library-side `Io` error on the SQLite `WAL` file write. |
| User runs `vault create` with `--path <path-on-network-filesystem>` (NFS / SMB) | Possible but un-tested under PoC. SQLite WAL mode warns about network filesystems; the resulting vault may have inconsistent file-locking semantics. **Out of scope acknowledgment** — the user is responsible for choosing a local-disk path; document in the `--help` ("recommend a local-disk path; network-filesystem paths are unsupported under PoC"). |
| User runs `vault create` with `--path /etc/pangolin.pvf` as a non-root user | A10 — library-side `Io(PermissionDenied)` surfaces; user re-runs with a writable path. |
| User runs `vault create` and the vault is created but the user immediately forgets the password | The vault is unrecoverable. PoC has no password-recovery flow (Recovery is MVP-1 issue 1.x). Document in the `--help`: "the vault password cannot be recovered; if forgotten, the vault is unreadable." A user reading this is expected to use a password manager (Pangolin itself, post-creation) or write the password down securely. **PoC limitation acknowledged.** |
| User runs `vault create` and the system suffers a power loss mid-create | The half-written `.pvf` may persist. The library's on-error path (`std::fs::remove_file(path)`) is best-effort; on power loss it does not run. The next `vault create` against the same path observes `path.exists() == true` and refuses; the user must `rm <path>` manually. Inherited P2 row #5 hazard. |

## Rollback / abort plan

If BUILD reveals the design is fundamentally wrong (per §16.7):

- **A3 pre-flight `path.exists()` introduces a TOCTOU bug** that
  the library's `acquire_lock` does not fully close (e.g., if
  the parent-dir-creation timing creates a window where the
  pre-flight passes but the library check passes too, leading
  to a partial-create that the on-error cleanup does not fully
  recover): **drop the pre-flight check; rely solely on the
  library's `path.exists()`.** Lose the "save a wasted password
  entry" UX optimization but preserve correctness. Documented
  as a rollback point.
- **A5 parent-canonicalize errors out for a parent dir that the
  user can write to but cannot resolve** (e.g., a Windows path
  with reparse points that `canonicalize` does not handle
  cleanly): **drop the parent-canonicalize step; pass the raw
  `--path` to `Vault::create` and accept that the success
  message may include a relative path.** Lose the absolute-path-
  in-error-message UX but preserve cross-platform correctness.
  Documented as a rollback point.
- **A4 cross-module reuse of `account.rs`'s helpers fails clippy
  visibility lints** (e.g., `pub(crate)` on the helpers exposes
  them to a downstream that should not see them): **move the
  helpers to a shared `tools/pangolin-cli/src/secret_input.rs`
  module** with `pub(crate)` visibility from there. Adds one
  new file to the diff but cleanly partitions the shared surface.
  Documented as a rollback point.
- **A7 `--print-id` flag conflicts with a future MVP-1 verb that
  needs `--id` for some other purpose**: **rename to
  `--print-vault-id`** (more verbose but unambiguous). Documented
  as a rollback point.
- **A11 explicit `vault.close()` introduces a `must_use` warning
  or a different error path than the library expects on a
  Locked vault**: **drop the explicit `close()`; rely on `Drop`.**
  Verify the lock-release happens cleanly via the `Vault::Drop`
  impl. Documented as a rollback point.

If a more fundamental design problem surfaces (e.g., the
`Vault::create` library API is itself the wrong shape — perhaps
because it should also unlock the vault and return an Active
session, so `vault create` chains naturally into the next
`account add`-style op without re-opening), escalate to Kelvin
per §16.7. Do not silently patch around. The current decision
(create-then-close, let the next op reopen + unlock) matches
P11A's pattern; deviating introduces a new "vault is created
but the process is exiting; should the next process reopen via
the same password?" question.

## Out of scope (explicit)

- **`vault open`** — MVP-1; opens an existing vault and prints
  its metadata (vault_id, created_at, KDF params, account count,
  last-pulled-block). P11B does not ship.
- **`vault info <path>`** — same as above; metadata-only display
  for an existing vault. MVP-1.
- **`vault lock`** — unlocked-vault → locked-vault transition is
  process-scoped; there is no persistent "unlocked vault" state
  to lock. The verb is a no-op under PoC. MVP-1 may revisit if
  a long-running daemon model lands.
- **`vault destroy <path>`** — deletes the vault file +
  associated WAL / journal files + lock file. MVP-1; requires
  the same confirmation discipline as `account delete` (literal
  `"yes"` per P11A A9) and additional friction (the user is
  destroying every account in the vault, not just one).
- **`vault rotate-password`** — change the master password.
  MVP-1; requires a re-derive-then-re-wrap flow that touches
  the meta row's `kdf_salt` + `wrapped_ciphertext` columns; not
  in PoC scope.
- **`vault export <path> <output-path>`** — produce an encrypted
  export of the vault payload. The library API
  (`Vault::export_payload`) is shipped (P3), but the CLI
  surface deferred to MVP-1 because the export-discipline (file
  shape, AAD discipline, recovery-vs-backup distinction) needs
  its own threat-model row.
- **`vault import <path>`** — import from a Pangolin export OR
  from KDBX / 1PUX / Bitwarden JSON. MVP-1 + parser-input
  surface concerns.
- **`--kdf-params <preset>` selector** — A6 above; MVP-1.
- **`--generate-password` for vault password** — A2 above; the
  vault password is the user's master credential and auto-
  generation is a UX anti-pattern.
- **`--force` flag to overwrite an existing vault** — A3 above;
  never. Documented as forbidden in the plan; refuse on
  principle.
- **Vault password recovery** — out of scope under PoC; MVP-1
  ships the Recovery flow per Whitepaper §10. A user who forgets
  the vault password loses every account in the vault; the
  `--help` text MUST surface this.
- **Auto-create vault on `account add`** — A13 above; preserved
  rejection from P11A's Q1.
- **Multi-tenant vaults** (one `.pvf` shared across multiple
  users) — out of scope architecturally; one vault = one user.
- **Cross-platform clipboard copy of the vault password after
  generation** — irrelevant given §A2's "no `--generate-password`"
  decision.
- **JSONLines vs. single-JSON output mode polish** — already
  shipped at the global flag level by P8; no change.
- **Vault password strength meter / zxcvbn integration** —
  MVP-1 polish; the only PoC discipline is the empty-password
  guard.

## Open questions for Kelvin

1. **Should the success message print the vault path absolute or
   relative?** Recommend absolute (canonicalized via
   `parent.canonicalize() + file_name`). Matches P8 fix MED-3's
   discipline; users running with `./v.pvf` from a wrong cwd
   see the actual file location. Lean toward absolute; flag at
   BUILD if the canonicalize-on-parent-only pattern breaks for
   some path edge case.

2. **Should `--print-id` be the default, with `--quiet` to
   suppress?** Recommend the opposite (default off; opt-in
   `--print-id`). Matches `git init`'s minimal default.
   The vault_id is rarely needed in the default flow; users
   piping the output to a logger will get the path-only line
   they expect.

3. **Is there a use case for `--password-cmd "<shell-command>"`
   (similar to GnuPG's `--pinentry-program`)?** Recommend NO for
   PoC. The use case (script that fetches the password from a
   secrets-manager like AWS Secrets Manager / HashiCorp Vault /
   1Password CLI) is real, but `--password-stdin` already
   covers it: `aws ssm get-parameter --name pangolin-pw |
   pangolin-cli vault create --password-stdin --path ...`.
   Adding `--password-cmd` introduces a new exec-then-read
   surface that needs its own audit pass; defer to MVP-1.

4. **Should `vault create` set the file's permissions to 0600
   (POSIX) on creation, in case the umask is too permissive?**
   Recommend YES if the library does not already do it.
   Verify at BUILD by inspecting `Vault::create`'s file-open
   path (likely uses SQLite's `open` which inherits umask). If
   the library does not enforce 0600, surface this as a
   library-side gap and either (a) fix it in
   `pangolin-store::vault::open_connection` (separate
   sub-issue) or (b) chmod the file from the CLI after
   `Vault::create` returns. **Likely BUILD will discover the
   library does not enforce 0600**; the CLI fix is a 3-line
   addition (`std::fs::set_permissions(canonical_path,
   PermissionsExt::from_mode(0o600))`) wrapped in a
   `cfg(unix)` block. Flag at BUILD.

5. **Should the `--help` text for `vault create` include a
   "vault password recovery" warning** (i.e., "if you forget
   this password, the vault is unrecoverable")? Recommend YES.
   PoC has no recovery flow; the user MUST be warned at vault-
   creation time. The warning lives in the long-form
   `about` / `long_about` doc-string on `VaultCreateArgs`.

6. **Should `vault create` accept a `--vault-name <str>` flag
   that is stored in the meta row** (e.g., for display in
   `vault info` later)? **Recommend NO for P11B** — the
   `VaultMeta` shape does not currently have a `name` column
   (verified by reading meta.rs); adding one is a schema bump
   out of P11B's scope. MVP-1 may revisit when `vault info`
   ships.

## Security-critical?

**YES** per §16.3 (touches credential-input paths at first-time
provisioning; the master credential for every account in the
new vault transits through this surface; ships new user-facing
surface that is the FIRST thing a non-author dev will type
when reproducing the system from zero). Audit at SIGNOFF will
scrutinize:

- The A2 password-input discipline. Does the CLI accept ANY
  flag-form for password input? Grep the `cli.rs` arg structs:
  no `--password <flag>` field; only `--password-stdin` (bool)
  and the implicit interactive-prompt fallback. Confirm via
  the `vault_help_avoids_forbidden_user_facing_terms` test PLUS
  a manual `cargo run -- vault create --help` inspection at
  SIGNOFF.

- The A3 overwrite-refuse discipline. Does `vault create`
  refuse to overwrite an existing file? Confirm via the
  `create_rejects_existing_path` test AND a manual test at
  SIGNOFF: create a vault at `/tmp/foo.pvf`, re-run
  `vault create --path /tmp/foo.pvf` and confirm the
  refuse-message + non-zero exit. Verify NO `--force` flag
  exists at the clap level.

- The A4 password-prompt-helper reuse. Does `commands/vault.rs`
  reuse the SAME helpers as `commands/account.rs`? Verify by
  reading the imports + checking that
  `read_secret_first_line_from_stdin`, `reject_empty_password`,
  and `prompt_password_with_confirmation` are the only password-
  input primitives in use; no fresh implementation. Audit-trace
  at SIGNOFF.

- The A5 parent-canonicalize discipline. Does `vault create`
  surface the canonical absolute path in success + error
  messages? Confirm via `create_canonicalizes_parent_dir` AND
  `create_rejects_path_in_nonexistent_parent` tests. Spot-check
  at SIGNOFF: run `vault create --path ./v.pvf` from `/tmp`
  and confirm the success message reads `created at /tmp/v.pvf`,
  not `created at ./v.pvf`.

- The A8 concurrent-create race. Does the CLI's pre-flight
  check + library's `create_new(true)` lock close the TOCTOU
  window? Audit-trace at SIGNOFF: read the
  `acquire_lock(vault_path)` body in vault.rs; confirm the
  `OpenOptions::create_new(true)` call is the race-defense
  primitive. Stress-test at SIGNOFF (optional): spawn two
  concurrent `vault create --path /tmp/race.pvf` invocations;
  observe one success + one clean error.

- The A11 explicit-close discipline. Does `run_create` call
  `vault.close()` after `Vault::create` returns? Read the
  `run_create` body; confirm the call is present and not
  conditionally-skipped on any branch.

- The A14 forbidden-terms discipline. Does `cargo run -- vault
  --help` and `cargo run -- vault create --help` output
  contain any of the §3.5 forbidden user-facing terms? Confirm
  via the `vault_help_avoids_forbidden_user_facing_terms` test
  (and manual inspection of every `eprintln!` / `println!` /
  `bail!` string in `commands/vault.rs`).

- The empty-password guard. Does `vault create` reject an empty
  password on BOTH the `--password-stdin` path AND the
  interactive-prompt path? Confirm via the
  `create_rejects_empty_password_via_stdin` test + the inherited
  `prompt_password_with_confirmation` behavior (which already
  bails on empty per `account.rs` L454–L456).

- The drop-zeroize discipline. Does the
  `password: SecretBytes` constructed inside `run_create` flow
  through to `Vault::create` and get zeroized on drop? Audit-
  trace each `SecretBytes::new` call and confirm no copy-by-
  value / no `.clone()` on the bytes / no intermediate `String`
  materialization beyond the `rpassword`-returns-`String`
  inherited limitation (§A12). Per-line review at SIGNOFF.

- The "vault is unrecoverable if password forgotten" warning in
  `--help`. Does the long-form `about` text include the
  no-recovery warning? Confirm at SIGNOFF: render
  `vault create --help` and visually verify the warning
  paragraph is present.

- The KDF parameters used. Does `Vault::create` derive under
  `KdfParams::RECOMMENDED` (256 MiB / t=3 / p=1) and is this

  introduces no new KDF surface.

- The vault_id printing discipline. Does `--print-id` emit a
  64-character lowercase-hex string and ONLY that (no extra
  whitespace, no ANSI escapes)? Confirm via the
  `create_with_print_id_outputs_hex` test (which asserts the
  output line matches `^vault_id: [0-9a-f]{64}$`).
