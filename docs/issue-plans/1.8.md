> **Status:** PLAN approved 2026-05-12. Decisions locked below; the
> §"Open questions for Kelvin" prose later in this file is the
> deliberation record but the table here is authoritative. **NOTE:** the
> §"Approach" / §"Open questions" bodies were written by the plan-gate
> recommending *exact-entropy-only* + a *curated* symbol set; Kelvin
> chose **both** the exact-entropy figure AND a zxcvbn-style strength
> estimator for typed passwords, plus the **full ASCII symbol set** — so
> §"Approach" §B (entropy) and the symbol-set discussion are superseded
> by the Decisions table (and by the builder prompt's detailed spec for
> the `zxcvbn` integration). Everything else in §"Approach" stands.
> **Security-relevant:** YES (qualified, narrow). This issue produces a
> *secret* (a password). The two security properties that matter: (a) it
> must draw entropy from a **CSPRNG** (the OS RNG via `pangolin_crypto::rng`),
> never `rand::thread_rng()` or a deterministic seed; (b) the character
> selection must be **unbiased** (uniform over the alphabet) — a biased
> index draw shrinks the effective keyspace, so it must use rejection
> sampling. Otherwise this is a small, pure-function issue: a `pwgen` module
> in `pangolin-core`, the `password_generate` FFI body, an additive
> `password_entropy_bits` helper, and (maybe) re-pointing the CLI's existing
> `--generate-password` flag at the FFI. Per §16 / `CONTRIBUTING.md`:
> plan-gate → builder → test → fix-pass-as-needed → audit → fix-pass-as-needed
> → merge, with Kelvin approving at plan and merge.
> **Depends on:** issue 1.1 (FFI surface frozen — `password_generate(policy:
> PasswordPolicy) -> Arc<SecretPassword>` declared in `pangolin-ffi/src/session.rs`
> with a `todo!()` body; the `PasswordPolicy` Record `{ schema_version: u16,
> length: u16, uppercase: bool, lowercase: bool, digits: bool, symbols: bool }`;
> `SecretPassword` is the zeroizing `uniffi::Object` wrapper around password
> bytes), issue 1.2 (`SecretBytes`, the `PASSWORD_MAX_BYTES = 4096`
> account-side cap, the `Validation { kind, message }` error variant on
> `pangolin_core::Error` / `FfiError`), the `pangolin-crypto` `rng` module
> (`fill_random(&mut [u8])`, `os_rng() -> OsRng`; `RngCore` kept crate-private
> per MEDIUM-11 — only `fill_random` is the public chokepoint). Current main
> tip `030207d`.

## Decisions locked (post plan-gate)

| Q | Decision | Rationale |
|---|---|---|
| Q1 — Symbol set | **Full ASCII printable punctuation** when `symbols: bool` is enabled — the 32 chars ``!"#$%&'()*+,-./:;<=>?@[\]^_`{|}~`` (with `exclude_ambiguous` on (default), `\|` is also dropped → 31 symbols). | Kelvin's call: maximum entropy from the symbol class. (Trade-off accepted: some password fields reject certain ASCII symbols — a "curated safe-symbols subset" option could be added later as a `PasswordPolicy` variant, but the default-on set is the full one.) |
| Q2 — Length floor / cap | **Floor 8 · default 16 · cap 128.** `PasswordPolicy::default().length == 16`. A `length < 8` or `> 128` policy → typed `Validation { kind: "password_policy" }`. Also: `length` must be ≥ the count of enabled character classes (so "at least one of each enabled class" is satisfiable) — `length < enabled_classes` → the same typed error. | "16+ chars" in the feature list / master-plan row characterises the *strong default*, not a hard minimum; sites with 12-char limits exist, so a low floor with a strong default is more usable than a hard floor at 16. The "≥ enabled-classes" constraint makes the at-least-one-of-each guarantee always satisfiable. |
| Q3 — Entropy-indicator scope | **BOTH.** (a) `password_entropy_bits(policy: &PasswordPolicy) -> f64` = `length × log2(alphabet_size)` — the *exact* entropy of a password produced by that policy (alphabet_size = (uppercase?26:0) + (lowercase?26:0) + (digits?10:0) + (symbols?31-or-32:0), minus the ambiguous chars removed if `exclude_ambiguous`); a pure function of the policy, computable without generating. Both helpers live in `pangolin-core::pwgen`; the `zxcvbn` dep goes in `pangolin-core`'s tree (it's a heuristic estimator, not a parser of attacker-structured data, so it doesn't need the separate-crate blast-containment treatment that `pangolin-totp`/`pangolin-kdbx` get for their parser deps; `pangolin-core` is the natural home — "the Rust core"). `cargo deny check` — `deny.toml` is a denylist + `wildcards = "deny"`, so `zxcvbn` + its transitives need no `deny.toml` entry as long as none is on the denylist; `=`-pin `zxcvbn` (and any of its transitives that aren't already pinned) + commit the updated `Cargo.lock`; `cargo audit` scans the new deps. | Kelvin's call: he wants both the generated-password entropy figure AND "this password is weak" feedback for arbitrary passwords (e.g. a KeePass-imported or manually-set password). zxcvbn is the de-facto standard, well-maintained, MIT/Apache, no `unsafe`; hand-rolling a comparable estimator is hundreds of lines and worse; a crude length+class heuristic that calls itself a "strength meter" gives false confidence. ~doubles the 1.8 effort vs exact-only — accepted. |
| Q4 — `PasswordPolicy` shape amendment | **Add `exclude_ambiguous: bool` (default `true`)** — excludes the visually-confusable chars `0 O 1 l I \|` (and similar — the exact set documented in `password-generator.md`) from whichever classes are enabled. Additive (the frozen `PasswordPolicy` Record gains one `bool` field + the build's `Default`; nothing external binds the FFI yet). | Excluding ambiguous chars is standard in good generators (no entropy cost worth worrying about — drops the alphabet by ~6 chars); on by default matches user expectation for a "good" generator. The frozen `PasswordPolicy` has a `schema_version` slot but no `Default` — `password_policy_default()` gives the shell the "strong defaults" without it having to know them. |
| Q5 — CLI `--generate-password` rewiring | **Rewire the existing `--generate-password` flag in `apps/cli` to call the `password_generate` FFI** (with `PasswordPolicy::default()`, plus the existing `--length` / class flags if any, mapped onto the policy). The CLI currently ships a *second, divergent* generator (a local 64-char power-of-two-alphabet, length-24 thing predating the FFI entry); 1.8 deletes that and routes through the one true generator. NO new subcommand. | Unifying the two generators is a code-quality fix (one implementation, one set of guarantees, one audit surface), not a feature-add; the `--generate-password` flag already exists on `account add`, so it's a few lines to point it at the FFI. (The CLI still uses the V0 `AccountSnapshot` write path, so a CLI-generated password is stored like any other PoC-CLI password — that's the standing CLI-V1 limitation, orthogonal to 1.8.) |
| Q6 — "Deterministic regeneration option (advanced)" | **Deferred to MVP-3+.** The Feature-list §1 "advanced" item (regenerate the same password deterministically from a master input + a site name — a stateless-password-manager-style feature) is explicitly *not* in MVP-1; the master-plan row 1.8 already distils it out. Out of scope. |
| Q-return — `password_generate` return shape | **`password_generate(policy: PasswordPolicy) -> Result<Arc<SecretPassword>, FfiError>`** (amends the 1.1-frozen `-> Arc<SecretPassword>`). ## Goal

After this issue closes: `pangolin-core` has a `pwgen` module — a pure
function `generate(&PwgenPolicy) -> Result<Zeroizing<String>, Error>` that
builds an alphabet from the enabled character classes, guarantees ≥1 char of
each enabled class, fills the remainder with **rejection-sampled, unbiased**
draws from the OS CSPRNG, and returns a zero-on-drop UTF-8 string; plus a
pure `entropy_bits(&PwgenPolicy) -> f64` = `length × log2(alphabet_size)`.
The 1.1-frozen `password_generate(policy: PasswordPolicy) -> Arc<SecretPassword>`
FFI entry is implemented (validates the policy, calls `pwgen::generate`, wraps
the bytes in `SecretPassword`); a new additive `password_entropy_bits(policy:
PasswordPolicy) -> f64` FFI entry exposes the entropy without generating; a
new additive `password_policy_default() -> PasswordPolicy` (or a `Default`
impl on the Record) gives callers the strong defaults. A statistical
distribution test proves the index draw is unbiased; per-class-guarantee /
length / alphabet / validation-error tests cover the rest. Documentation:
`docs/architecture/password-generator.md`, an `ffi-surface.md` update, an
`account-limits.md` row for the generator's length floor/cap, a DEVLOG entry.
Nothing else changes — the diff stays inside `pangolin-core` (new module),
`pangolin-ffi` (the `password_generate` body + two helper fns), `apps/cli`
(only if Q5 = rewire), and `docs/` + `DEVLOG.md`. No store, search, session,
device, lineage, or TOTP code is touched; existing PoC `.pvf`/`.pvf`-era
vaults and the e2e suite are trivially preserved (1.8 persists nothing).

## Approach

### 1. The generator — `crates/pangolin-core/src/pwgen.rs` (new module)

1. **The `pangolin-core`-side policy type.** `pangolin-core` must **not**
   carry `uniffi::` annotations (Q3 invariant: `cargo tree -p pangolin-core |
   grep -ci uniffi` stays 0). So the `pwgen` module defines a plain
   `pub struct PwgenPolicy { pub length: u16, pub uppercase: bool, pub
   lowercase: bool, pub digits: bool, pub symbols: bool, /* + exclude_ambiguous
   / symbols-mode if Q4 says so */ }` with `#[derive(Debug, Clone, Copy)]` and
   a `Default` impl (the "strong defaults"). The FFI `PasswordPolicy` Record
   in `pangolin-ffi` converts to/from this (drop the `schema_version` slot —
   it's a transport concern; the §18.7 policy is whole-record, and a policy
   record carries no durable secret, so the version slot is just passed
   through / ignored by the generator). Re-exported as `pangolin_core::pwgen`.
2. **`generate(policy: &PwgenPolicy) -> Result<Zeroizing<String>, Error>`:**
   - Validate (see §2 below). On failure → `Error::Validation { kind:
     "password_policy", message }`.
   - Build the alphabet: a `Vec<u8>` (or `&'static` slices concatenated) =
     (uppercase ? `A..=Z`) ++ (lowercase ? `a..=z`) ++ (digits ? `0..=9`) ++
     (symbols ? `<the chosen symbol set>`), minus the ambiguous chars if
     `exclude_ambiguous` (Q4). All bytes are ASCII so the result is valid
     UTF-8 by construction and `length` chars == `length` bytes.
   - **Guarantee ≥1 of each enabled class.** Recommended method: the
     *place-then-shuffle* construction — for each of the `k` enabled classes,
     draw one uniform char from that class's sub-alphabet and put it in slot
     `i` (`i = 0..k`); fill slots `k..length` with uniform draws from the full
     alphabet; then **Fisher-Yates shuffle** the whole `length`-byte buffer
     with unbiased index draws. This is provably uniform over *the set of
     strings that contain ≥1 of each enabled class* (a clean, documented
     distribution). Alternative: generate-uniformly-then-reject-if-a-class-is-
     missing (simpler code; for reasonable lengths the rejection rate is tiny;
     but the distribution is then "uniform over that same constrained set"
     reached by a different path — same end distribution). **Recommend
     place-then-shuffle** — it's O(length), never loops unboundedly, and the
     code is easy to audit. The builder picks one and the doc states which.
   - **Unbiased uniform index draws.** To pick a uniform index in `0..n` from
     CSPRNG bytes: draw a `u32` from `fill_random` (4 bytes), reject-and-redraw
     if it falls in the biased "tail" (`u32::MAX - (u32::MAX % n)`), else
     return `value % n`. (Equivalently a `u64`-based draw to make the
     rejection probability negligible.) This is the *only* security-load-
     bearing arithmetic in the issue; the doc and the test (§D) call it out
     explicitly. (Note: the CLI's existing `--generate-password` sidesteps
     this with a power-of-two alphabet + byte mask; the FFI generator's
     alphabet sizes are 62 / 94 / etc. — not powers of two — so rejection
     sampling is mandatory.)
   - Wrap the `Vec<u8>` → `String::from_utf8(...).expect(...)` (infallible by
     construction; or build the `String` directly) → `Zeroizing::new(s)`.
     The intermediate `Vec<u8>` is itself zeroized after the `String` is
     built (or built into the `String` in place to avoid a second buffer).
   - `#![forbid(unsafe_code)]` is already on `pangolin-core`; no unsafe
     needed. AGPL-3.0-or-later SPDX header on the new file.

### 2. Validation (the floor/cap — Q2)

- `policy.length` **floor**: recommend a *low* floor (`PWGEN_LENGTH_MIN`, e.g.
  **8**) — sites with a 12-char cap exist; refusing to generate a 10-char
  password because "16 is the recommendation" is hostile. The *recommended*
  length (16, the default) is documented but not enforced. (Hard alternative:
  floor at 16, matching the master-plan row literally — recommend against;
  see Q2.)
- `policy.length` **cap** (`PWGEN_LENGTH_MAX`, e.g. **128**) — a sane upper
  bound; well under the account-side `PASSWORD_MAX_BYTES = 4096` (so a
  generated password always satisfies `current_password` validation). Anything
  longer is almost certainly a caller bug.
- **At least one class enabled.** A policy with `!uppercase && !lowercase &&
  !digits && !symbols` → `Validation { kind: "password_policy" }` — can't
  generate from an empty alphabet.
- **`length ≥ number_of_enabled_classes`.** If 4 classes are enabled and
  `length == 3`, "≥1 of each" is unsatisfiable → `Validation { kind:
  "password_policy" }` (recommend a typed error over silently clamping or
  dropping the guarantee — the caller asked for something impossible).
- Errors map: `pangolin_core::Error::Validation { kind: "password_policy",
  message }` → `FfiError::Validation { kind: "password_policy", message }`
  (the `From` impl already passes `kind`/`message` through, per
  `pangolin-ffi/src/error.rs` L129–131).
- `PWGEN_LENGTH_MIN` / `PWGEN_LENGTH_MAX` get a row in
  `docs/architecture/account-limits.md`, explicitly noted as *the generator's
  own bounds*, distinct from `PASSWORD_MAX_BYTES` (which validates
  user-supplied passwords).

### 3. The defaults — `PwgenPolicy::default()` / the FFI mirror

- The "strong defaults": **length 16, all four classes on, symbols = the
  chosen default-on symbol set** (Q1). All four classes on + length 16 gives
  the ≥1-of-each guarantee headroom (16 ≥ 4). This matches the master-plan
  row's "16+ chars, mixed case + digits + symbols".
- The FFI side: 1.1's `PasswordPolicy` Record has **no `Default` impl** and
  `uniffi::Record` can't derive one usefully across the FFI boundary anyway.
  Add a `password_policy_default() -> PasswordPolicy` FFI fn (purely additive,
  same posture as 1.7's `parse_totp_secret`) that returns the strong-defaults
  Record (`schema_version` = the current FFI schema-version const). The Rust
  side also gets a `Default` impl on the Record for internal use. (Q4-adjacent:
  if the Record gains a `symbols` enum / `exclude_ambiguous` flag, the default
  sets those too.)

### 4. The entropy indicator — `entropy_bits` (§B)

- `pangolin_core::pwgen::entropy_bits(policy: &PwgenPolicy) -> f64` =
  `f64::from(policy.length) * (alphabet_size as f64).log2()`, where
  `alphabet_size` = `(uppercase ? 26 : 0) + (lowercase ? 26 : 0) + (digits ?
  10 : 0) + (symbols ? <symbol-set size> : 0)` (minus ambiguous-char count if
  `exclude_ambiguous`). Returns the bit-entropy of *a password generated by
  this policy*. (Caveat noted in the doc: the ≥1-of-each guarantee makes the
  *true* entropy a hair lower than `length × log2(alphabet)` — a fraction of a
  bit for `length ≫ k` — but `length × log2(alphabet)` is the standard
  reported convention; the doc states the tiny over-count. If Kelvin wants the
  precise figure instead, it's a closed-form inclusion-exclusion sum — flag
  it, but recommend the convention.) Empty alphabet → not reachable here
  because `generate` already rejects it; `entropy_bits` on an
  all-classes-off policy can return `0.0` or the validation error — recommend
  it mirror `generate`'s validation and the FFI wrapper return
  `FfiError::Validation` (so `password_entropy_bits` and `password_generate`
  agree on which policies are valid).
- **The "Visible entropy indicator" feature line is satisfied by this exact
  number** — for a *generated* password with a known policy, entropy is
  exact, not an estimate. A zxcvbn-style strength estimator for *arbitrary
  user-typed* passwords (the UI's "weak/medium/strong" meter on the
  password-edit field) is a *different, much bigger* feature (a large dep or
  hundreds of lines of dictionary/keyboard-walk/l33t-substitution heuristics)
  — recommend **out of scope for 1.8**, deferred to MVP-3+ (the UI). See Q3.

### 5. The FFI — `crates/pangolin-ffi/src/session.rs`

- Implement `password_generate(policy: PasswordPolicy) -> Arc<SecretPassword>`:
  convert the Record → `pwgen::PwgenPolicy`, call `pwgen::generate`, on `Ok`
  wrap the `Zeroizing<String>`'s bytes in `SecretPassword::new(bytes)` (the
  bytes move out of the `Zeroizing` into `SecretBuf`; no extra copy lingers).
  On `Err` — wait: the 1.1-frozen signature returns `Arc<SecretPassword>`,
  **not** `Result<Arc<SecretPassword>, FfiError>`. So a validation failure has
  nowhere to go. → **Surface as part of Q2/Q4:** either (a) the
  `password_generate` signature is amended to `Result<Arc<SecretPassword>,
  FfiError>` (an additive amendment to the *return shape* — allowed, nothing
  external binds yet — and the *right* call: a policy can be invalid, and a
  `todo!()`/panic on bad input is worse than a typed error), or (b)
  `password_generate` clamps/sanitises the policy to the nearest valid one and
  never fails (length clamped into `[MIN, MAX]`; if no class enabled, force
  all on). **Recommend (a)** — return `Result`; it's consistent with every
  other fallible FFI entry, and the validation errors are real. (If Kelvin
  prefers to keep the exact frozen `-> Arc<SecretPassword>` shape, then (b) is
  the fallback — clamp-and-never-fail — but document the clamping precisely.)
- Add `password_entropy_bits(policy: PasswordPolicy) -> Result<f64, FfiError>`
  (or `-> f64` if Q3 says don't validate) — purely additive.
- Add `password_policy_default() -> PasswordPolicy` — purely additive.
- If Q4 amends the `PasswordPolicy` Record (adds `exclude_ambiguous: bool`
  and/or a `symbols: SymbolsMode` enum `{ None, Curated, Full }` replacing the
  bare `symbols: bool`) — that's a 1.1-surface amendment (allowed; flagged).
  **Recommend the minimal amendment:** keep `symbols: bool` (Curated when
  true) and *add* `exclude_ambiguous: bool` (a real usability win — `0O1lI|`
  etc. mistyped from a screen are a genuine annoyance; the CLI's existing
  generator already does this). If Kelvin wants the full ASCII symbol set
  available, then the `symbols: SymbolsMode` enum is the cleaner shape — but
  it's a wider amendment. Default for `exclude_ambiguous`: **true** (matches
  the CLI prior art and good-generator convention).
- `ffi-surface.md` updated: `password_generate` row (implemented; note the
  return-shape decision), the new `password_entropy_bits` + `password_policy_default`
  rows, and the `PasswordPolicy` Record row if amended.

### 6. The CLI — `apps/cli/src/commands/account.rs` (only if Q5 = rewire)

- Today `--generate-password` calls a *local* `generate_password()` (64-char
  power-of-two alphabet, length 24, byte-mask sampling — see
  `account.rs` ~L72–95, ~L548). 1.8 *can* re-point it at the FFI:
  `account add --generate-password [--generated-length N] [--generated-symbols/--no-generated-symbols]
  …]` → builds a `PasswordPolicy`, calls `pangolin_ffi::password_generate`,
  prints the entropy (`password_entropy_bits`) to stderr alongside the
  password (which already goes to stderr via the MED-3-fixed path). **Recommend:**
  do the rewire in 1.8 — it's small (the flag, the prompt, the stderr emit
  all already exist; only the *source* of the bytes changes), it removes a
  divergent second generator (the local 64-char one), and it gives the
  generator real CLI exercise + an integration test. The counter-argument is
  consistency with 1.5/1.6/1.7's CLI-deferral pattern — but those deferred
  *new* subcommands; this is *re-pointing an existing flag at a now-real FFI*,
  which is the natural completion of the flag, not new surface. If Kelvin
  prefers the deferral pattern, leave the CLI alone and the local generator
  stays until CLI-V1 — surface it in the plan's Out-of-scope. **Note:** if the
  rewire happens, the `GENERATED_PASSWORD_LEN`/`GENERATED_PASSWORD_ALPHABET`
  consts are removed and the A8/A12 design notes in `account.rs` updated to
  point at `password-generator.md`.

## Success criteria

1. `password_generate` (and `pwgen::generate`) returns a password of **exactly
   `policy.length` characters**.
2. **Every character of a generated password is in the policy's alphabet** —
   no chars from a disabled class; if a curated symbol set is used, no symbol
   outside it; if `exclude_ambiguous`, none of the ambiguous chars.
3. **At-least-one-of-each-enabled-class:** with all four classes on, every
   generated password contains ≥1 uppercase, ≥1 lowercase, ≥1 digit, ≥1
   symbol; with only 2 classes on, ≥1 of each of those 2 and 0 of the other 2.
4. **Unbiased distribution** *(the security criterion)*: over a large sample
   (≥100k passwords) with a small known alphabet, every character's observed
   frequency is within the documented statistical tolerance of uniform
   (chi-squared p-value above threshold, or ±X% bound) — i.e. the index draw
   is uniform, no modulo bias.
5. **CSPRNG:** `pwgen::generate` draws entropy from `pangolin_crypto::rng`
   (the OS CSPRNG), not `rand::thread_rng()` and not a fixed seed — verified
   by code review (the only RNG call site) + criterion 4's distribution test.
6. `password_policy_default()` (and `PwgenPolicy::default()`) is the documented
   strong default (length 16, all four classes on, curated symbols, ambiguous
   excluded if Q4); generating from it yields a 16-char password with ≥1 of
   each of the four classes.
7. **Validation errors are typed** (`Validation { kind: "password_policy" }` /
   `FfiError::Validation { kind: "password_policy" }`): empty alphabet (all
   classes off) → error; `length < PWGEN_LENGTH_MIN` → error; `length >
   PWGEN_LENGTH_MAX` → error; `length < number_of_enabled_classes` → error.
   (Or, if Q2-fallback = clamp-don't-fail, the clamping is exactly as
   documented and a length-1/all-classes-off policy yields the documented
   sanitised result — pick one per Q2.)
8. **Entropy is correct:** `entropy_bits(default())` == `16.0 × log2(D)` where
   `D` is the default-policy alphabet size, within a small float tolerance;
   `entropy_bits` of a lowercase-only policy length L == `L × log2(26)`;
   lowercase+digits == `L × log2(36)`; all-four (no ambiguous-exclusion) ==
   `L × log2(94)` (or `L × log2(94 − ambiguous_count)` with exclusion); etc.
9. **Zeroization:** the returned `Zeroizing<String>` and any intermediate
   byte buffer are zeroed on drop; `password_generate`'s returned
   `Arc<SecretPassword>` is the existing zeroizing wrapper; the generated
   bytes are never written to disk (1.8 persists nothing).
10. **No regression:** `cargo tree -p pangolin-core | grep -ci uniffi` == 0;
    `cargo tree -p pangolin-store | grep -ci uniffi` == 0;
    `cargo tree -p pangolin-crypto | grep -ci serde` == 0;
    `forbid(unsafe_code)` retained on every crate except `pangolin-ffi`;
    `cargo build`, `cargo clippy --all-targets -- -D warnings`, `cargo fmt
    --check`, `cargo test --workspace`, `cargo audit`, the existing FFI
    integration tests, and the e2e suite (PoC vaults open + unlock +
    `account_history`) all pass; the `no_plaintext_on_disk` proptest still
    passes (1.8 doesn't touch the store; if Q5 = rewire, a generated password
    used as `current_password` goes through the same AEAD seal as any other
    password — proptest unaffected).
11. *(If Q5 = rewire)* `account add --generate-password` produces a password
    matching the policy and prints its entropy; an integration test covers it;
    the divergent local 64-char generator is removed.

## Failure modes considered

- **Biased index draw (`rng.next_u32() % alphabet_len` with non-power-of-two
  `alphabet_len`) → shrunken effective keyspace → weaker passwords.** *The*
  failure mode. Mitigation: rejection sampling against the alphabet size
  (reject draws in the modulo-biased tail), documented in
  `password-generator.md`, verified by test #4 (the chi-squared / ±5%
  distribution test).
- **Non-CSPRNG entropy source** (someone wires `rand::thread_rng()` or a
  fixed seed "for testability"). Mitigation: the generator calls *only*
  `pangolin_crypto::rng::fill_random` (the audited OS-CSPRNG chokepoint); the
  public API does **not** expose an RNG parameter (the deterministic-RNG
  surface in `pangolin-crypto` is crate-private per MEDIUM-11, so it can't be
  reached anyway); code-review checklist item; the distribution test stands in
  for "is this actually random".
- **Missing-class guarantee silently violated** (the swap/place logic has an
  off-by-one and a class ends up absent for some lengths). Mitigation: test #3
  over many samples and several class combinations; the place-then-shuffle
  construction makes the guarantee structural (the first `k` slots are seeded
  before the shuffle, so the multiset always contains ≥1 of each).
- **Place-then-shuffle made non-uniform by a biased shuffle.** Mitigation:
  Fisher-Yates with the *same* unbiased index draw as the fill step; test #4
  exercises the full pipeline (place + fill + shuffle) so a biased shuffle
  shows up as a non-uniform char distribution.
- **`length` too small for the enabled classes** (`length 3`, 4 classes) →
  infinite loop (reject-and-redraw never satisfied) or a panic (place step
  writes past the buffer). Mitigation: validate `length ≥ k_enabled` *before*
  generating → typed `Validation` error; the place-then-shuffle construction
  also can't loop (it's straight-line O(length)).
- **The 1.1-frozen `-> Arc<SecretPassword>` return shape can't carry a
  validation error** (a real conflict — see Q2/§5). Mitigation: amend the
  return shape to `Result<…, FfiError>` (recommended — additive, nothing
  binds yet) **or** clamp-and-never-fail (fallback, documented). Either way no
  `todo!()`/`panic!`/`unwrap` on caller-supplied input survives.
- **Entropy over-count** (reporting `length × log2(alphabet)` when the
  ≥1-of-each guarantee makes the true entropy slightly lower). Mitigation:
  documented in `password-generator.md` as the standard reported convention
  with the magnitude of the over-count noted; if Kelvin wants the precise
  figure, the closed form is available — flagged in Q3-adjacent prose.
- **Two divergent generators** (the CLI's local 64-char one vs the FFI one)
  drifting apart. Mitigation: Q5 = rewire removes the local one; if Q5 =
  defer, the divergence is documented as a known temporary state until CLI-V1.
- **Generated password printed to a terminal** (CLI) lands in scrollback /
  shell history. Mitigation: accepted limitation (same posture as 1.4's
  `reveal_*` notes); the CLI already routes the bytes via the MED-3-fixed
  stderr path, never a `--password <flag>`; documented in
  `password-generator.md`.

## Open questions for Kelvin

1. **Symbol set.** What symbols does the generator use by default?
   - **(a)** A *curated, default-on* set that avoids characters sites commonly
     reject — e.g. `!@#$%^&*-_=+` (12 chars) or the CLI's current `!@#$%^&*`
     (8). Maximises "the generated password actually works on the site you
     paste it into".
   - **(b)** The full ASCII printable-symbol range (` !"#$%&'()*+,-./:;<=>?@[\]^_\`{|}~` — 32 chars) — maximal entropy per char, but many password
     fields reject `< > " ' \` etc.
   - **(c)** Configurable — a curated set default-on, with a `symbols:
     SymbolsMode { None | Curated | Full }` enum on the policy so an advanced
     user can pick the full range. Wider `PasswordPolicy` amendment (see Q4).
   - **Recommendation: (a)** with the curated set `!@#$%^&*-_=+` (12 chars,
     log2 ≈ 3.58 — a 16-char all-classes password ≈ 16 × log2(74) ≈ 99 bits,
     well past "uncrackable"), keeping the frozen `symbols: bool` shape. If you
     want the full set available, go (c). Don't go (b) (usability cost > the
     ~1 bit/char entropy gain).
2. **Length floor / cap.** What are `PWGEN_LENGTH_MIN` / `PWGEN_LENGTH_MAX`?
   - **(a)** Low floor (`MIN = 8`), default 16, cap `MAX = 128`. Sites with
     12-char (or shorter) limits exist; the generator should be able to make a
     password that *fits* such a site. 16 is the *recommended* / default
     length, not the *minimum*.
   - **(b)** Hard floor at 16 (matching "16+ chars" in the master-plan row
     literally) — refuses to generate anything shorter; cap 128.
   - Also: **the 1.1-frozen `password_generate -> Arc<SecretPassword>` return
     shape can't carry a validation error.** Amend it to
     `Result<Arc<SecretPassword>, FfiError>` (additive return-shape
     amendment — recommended, consistent with every other fallible FFI entry,
     nothing external binds yet), or keep the frozen shape and have
     `password_generate` clamp the policy into the valid range and never fail?
   - **Recommendation: (a) — low floor 8, default 16, cap 128 — and amend the
     return shape to `Result<…, FfiError>`.** Don't be stricter than the real
     world requires; don't paper over invalid input with silent clamping.
3. **Entropy-indicator scope.** What does "Visible entropy indicator" mean for
   1.8?
   - **(a)** *Exact entropy of a generated password* only — `length ×
     log2(alphabet)`, exposed as `password_entropy_bits(policy)`. Easy, exact,
     fully covers "show the strength of the password I just generated".
   - **(b)** *Also* a zxcvbn-style strength estimator for *arbitrary,
     user-typed* passwords (the UI's weak/medium/strong meter on the
     password-edit field). Big: a large dep (`zxcvbn` crate ⇒ env-quirk #3
     `=`-pins + `Cargo.lock` commit + `cargo audit` scan) or hundreds of lines
     of dictionary / keyboard-walk / l33t-substitution heuristics. Would roughly
     triple the issue size and put a fuzzy heuristic in the codebase.
   - **Recommendation: (a) for 1.8.** The "entropy indicator" the feature line
     means — *for the generator* — is the exact figure. The typed-password
     strength meter is a separate UI feature; defer to MVP-3+. (If you want a
     *crude* typed-password estimate now — length × char-class-count, clearly
     labelled "estimate, not zxcvbn" — say so and I'll scope it in, but I'd
     advise against it: a bad strength meter is worse than none.)
4. **`PasswordPolicy` shape amendment.** The 1.1-frozen Record is `{
   schema_version, length, uppercase, lowercase, digits, symbols }`. Amend it?
   - **(a)** Add `exclude_ambiguous: bool` (default true) — excludes
     `0 O o 1 l I |` (and similar) from the alphabet. A real usability win;
     the CLI's current generator already does this; cheap.
   - **(b)** Also replace `symbols: bool` with `symbols: SymbolsMode { None |
     Curated | Full }` (needed only if Q1 = (c)).
   - **(c)** No amendment — the four bare class flags + `length` are adequate;
     ambiguous chars stay in.
   - **Recommendation: (a)** — add `exclude_ambiguous: bool` (default true);
     keep `symbols: bool`. Add (b)'s enum only if you pick Q1 = (c). All of
     these are additive 1.1-surface amendments (allowed — nothing external
     binds the Record yet).
5. **CLI `--generate-password` in 1.8.** The flag already exists but uses a
   *local* 64-char generator, not the FFI. Re-point it at
   `pangolin_ffi::password_generate` in 1.8 (small — only the byte *source*
   changes; the flag, prompt, and stderr-emit already exist; removes the
   divergent second generator; gives the FFI generator real CLI exercise + an
   integration test), or leave it as-is and defer the rewire to CLI-V1 (the
   1.5/1.6/1.7 deferral pattern — but those deferred *new* subcommands; this
   is completing an *existing* flag)?
   - **Recommendation: rewire it in 1.8.** The whole point of `password_generate`
     is to be the one generator; leaving a divergent local copy in the CLI is
     the worse outcome. It's a few lines.
6. **"Deterministic regeneration option (advanced)."** The Feature list §1
   block lists this (a deterministic password derived from master-secret +
   site + counter, à la LessPass — re-derivable without storage). It's marked
   "(advanced)" and the master-plan row 1.8 distills it *out* (the row says
   only "strong defaults, per-site overrides, entropy indicator"). Confirm
   it's **out of scope for 1.8 / deferred to MVP-3+** (it needs a master-secret
   derivation path and a UI to be useful, and it's an "advanced" toggle, not
   table-stakes)? **Recommendation: defer — confirm.** (If you want it in 1.8,
   that's a meaningfully bigger issue — say so and I'll re-scope.)

## Out of scope (MVP-3+ / later issues own these)

- **Zxcvbn-style strength estimator for arbitrary user-typed passwords** (the
  UI's weak/medium/strong meter on the password-edit field) — MVP-3+ (the UI).
  1.8's "entropy indicator" is the *exact* entropy of a *generated* password.
- **Deterministic / re-derivable password generation** ("Deterministic
  regeneration option (advanced)" in Feature list §1) — needs a master-secret
  derivation path; MVP-3+ (and contingent on Q6).
- **Stored per-account password policies** — in MVP-1, "per-site override"
  means *the caller passes whatever `PasswordPolicy` that site needs* on the
  `password_generate` call; there is *no* policy persisted on the
  `AccountIdentity`. Storing one would be another `payload_version` bump on the
  CBOR body — a UI/MVP-3+ concern, not 1.8.
- **Clipboard integration / "copy generated password" UX** — a shell/UI
  concern (the FFI hands back an opaque `Arc<SecretPassword>`; what the shell
  does with it — clipboard, autofill, print — is the shell's call). MVP-3+ for
  the GUI; in MVP-1's CLI it's the existing stderr emit.
- **A `password_generate` variant that also returns the entropy in one call**
  — kept as a *separate* `password_entropy_bits(policy)` fn so `password_generate`'s
  return shape stays minimal (the entropy is a pure function of the policy,
  computable without generating).
