# Issue P1: pangolin-crypto — primitives + key hierarchy types

> **Status:** PLAN-gate, awaiting Kelvin approval per §16.3.
> **Security-critical:** YES. Touches cryptography, key types, zeroization. No code lands until this plan is approved.

## Spec reference

- **Master plan §0 cardinal principle 1:** "Rust core is the only security-critical component … encryption, session decisions, revision lineage, conflict detection, social-recovery client logic, TOTP generation — all of it lives in core."
- **Master plan §0 cardinal principle 2:** "No plaintext leaves the device. Ever. For any reason."
- **Master plan §0 cardinal principle 6:** "Authority is layered … Recovery rotates *authority*; VDK is *re-wrapped*, never re-derived; guardians never see VDK."
- **Whitepaper §B.2 (MVP architecture):** "Vault Data Key (VDK) encrypts all vault contents; VDK is never stored plaintext; VDK is stored as Wrapped VDK, encrypted to current vault authority."
- **Whitepaper §F (Identity & Key Hierarchy):**
  > "Vault Authority — logical root of control. Authorizes device enrollment/revocation, revision publication, recovery cancellation. Can be rotated via social recovery.
  > Vault Data Key (VDK) — encrypts all vault contents. Stored only as Wrapped VDK. Re-wrapped when authority rotates. Never exposed to guardians.
  > Device Keys — generated per device. Authorized under vault authority. Can be revoked at any time."
- **Master plan §3.5 PoC decision #4:** "`chacha20poly1305` for AEAD, `argon2` for KDF, `ed25519-dalek` for signing, `secrecy` for in-memory zeroization."
- **Master plan §3.7 EPIC: Crypto (P1):** sub-issues P1-1 through P1-5.
- **THREAT_MODEL.md invariant 1:** "No plaintext writes to disk. All payloads encrypted client-side before any persistence."

## Goal

Ship the `pangolin-crypto` crate with:
1. A safe AEAD wrapper around XChaCha20-Poly1305 with RFC-aligned test vectors.
2. An Argon2id KDF wrapper with conservative, locked parameters.
3. An Ed25519 sign/verify wrapper.
4. A typed key hierarchy (`VdkKey`, `WrappedVdk`, `AuthorityKey`, `DeviceKey`) where every secret-bearing type implements `Zeroize` on drop and produces a redacted `Debug` impl.
5. Test vectors that demonstrate **wrap/unwrap** of the VDK under authority — proving that authority-rotation primitive works end-to-end before MVP-3 needs it.

After this issue, every subsequent crate that needs encryption or signing pulls from `pangolin-crypto`. **No other crate gets to instantiate raw cryptographic primitives.** That's the choke point that lets the audit surface stay small.

## Approach

**Hard rule before anything else:** This crate **does not invent crypto**. Every primitive is a thin, well-typed wrapper around a vetted, well-maintained Rust crate. If a primitive cannot be implemented as a wrapper, that's a discussion with Kelvin before any code lands.

### Sub-issue breakdown (single PR, sequenced commits)

#### P1-1 — AEAD wrapper (XChaCha20-Poly1305)

- Library: `chacha20poly1305` from RustCrypto (MIT/Apache-2.0; widely used; `Aead` trait conformance).
- Algorithm: **XChaCha20-Poly1305** (24-byte nonce, no nonce-reuse anxiety for our usage profile). Not regular ChaCha20-Poly1305 — we want the larger nonce space because revisions are produced on multiple devices.
- Public surface (sketch):
  ```rust
  pub struct AeadKey([u8; 32]);  // wraps Zeroizing<[u8; 32]>
  pub struct Nonce([u8; 24]);
  pub struct Ciphertext(Vec<u8>);  // includes the 16-byte auth tag

  impl AeadKey {
      pub fn generate() -> Self;
      pub fn from_bytes(b: [u8; 32]) -> Self;
      pub fn seal(&self, nonce: &Nonce, plaintext: &[u8], aad: &[u8]) -> Ciphertext;
      pub fn open(&self, nonce: &Nonce, ciphertext: &Ciphertext, aad: &[u8])
          -> Result<Vec<u8>, AeadError>;
  }

  impl Nonce {
      pub fn random() -> Self;  // 24 random bytes from OsRng
  }
  ```
- AAD usage: every revision encryption MUST bind `vault_id ‖ account_id ‖ parent_revision_id ‖ schema_version` as AAD so that a ciphertext can't be replayed in a different context.
- `AeadKey` and `Ciphertext::open` return paths must `Zeroize` any intermediate buffers.

#### P1-2 — KDF (Argon2id)

- Library: `argon2` from RustCrypto.
- Algorithm: **Argon2id** (hybrid; resists both side-channel and time-memory tradeoffs).
- **Locked parameters** (this PR establishes them; future bumps require a new issue + Kelvin approval):
  - Memory cost: **256 MiB** (262144 KiB)
  - Time cost: **3 iterations**
  - Parallelism: **1**
  - Output length: **32 bytes** (matches AeadKey size)
- Public surface:
  ```rust
  pub struct KdfSalt([u8; 16]);
  pub struct KdfParams { /* memory_kib, time_cost, parallelism */ }

  impl KdfParams {
      pub const RECOMMENDED: Self = /* the locked parameters above */;
      pub fn validate(&self) -> Result<(), KdfError>;  // rejects below-minimum configs
  }

  pub fn derive_key(password: &SecretBytes, salt: &KdfSalt, params: &KdfParams)
      -> Result<AeadKey, KdfError>;
  ```
- `validate()` enforces a **minimum** floor (e.g., ≥ 64 MiB / time ≥ 1) so future configurable params can't be silently weakened.
- Test vector: derive a known key from a known password+salt+params and assert byte-equal output. Cross-check against an independent Argon2id implementation (the RFC 9106 test vector).

#### P1-3 — Ed25519 sign/verify

- Library: `ed25519-dalek` v2 (the audited, hardened branch).
- Public surface:
  ```rust
  pub struct SigningKey(/* zeroizing wrapper */);
  pub struct VerifyingKey([u8; 32]);
  pub struct Signature([u8; 64]);

  impl SigningKey {
      pub fn generate() -> Self;
      pub fn verifying_key(&self) -> VerifyingKey;
      pub fn sign(&self, message: &[u8]) -> Signature;
  }

  impl VerifyingKey {
      pub fn verify(&self, message: &[u8], sig: &Signature) -> Result<(), SignatureError>;
  }
  ```
- `SigningKey` `Zeroize` on drop; `Debug` redacts.
- Verification uses **strict mode** (`ed25519_dalek::Verifier` with `verify_strict`) to reject non-canonical signatures.

#### P1-4 — Key hierarchy types

This is the heart of P1. Four newtypes that match Whitepaper §F. Each has explicit lifecycle semantics encoded in the type system.

```rust
/// Logical root of vault control. Used to authorize device enrollment/
/// revocation, revision publication, recovery cancellation. Rotated via
/// social recovery. NEVER stored on-chain.
pub struct AuthorityKey(SigningKey);

/// Per-device signing+gas keypair (per D-006: same key signs revisions
/// AND pays gas). Authorized under AuthorityKey. Can be revoked.
pub struct DeviceKey(SigningKey);

/// Encrypts all vault contents. Stored only in WrappedVdk form. Re-wrapped
/// when authority rotates. Never exposed to guardians, never written
/// plaintext.
pub struct VdkKey(AeadKey);

/// VDK encrypted under the current AuthorityKey's derived AEAD key.
pub struct WrappedVdk { ciphertext: Ciphertext, nonce: Nonce, /* metadata */ }
```

Operations:

```rust
impl VdkKey {
    pub fn generate() -> Self;
    pub fn wrap(&self, authority: &AuthorityKey) -> WrappedVdk;
}

impl WrappedVdk {
    pub fn unwrap_with(&self, authority: &AuthorityKey) -> Result<VdkKey, AeadError>;
    pub fn rewrap(self, old_authority: &AuthorityKey, new_authority: &AuthorityKey)
        -> Result<WrappedVdk, AeadError>;
}
```

Critical properties (verified by tests):
- `wrap` then `unwrap_with` round-trips byte-equal VDK.
- `rewrap` with `old=current_authority` and `new=fresh_authority` produces a `WrappedVdk` that the new authority can unwrap and the old one cannot.
- Wrapping does **not** alter the underlying `VdkKey` — re-wrapping is the recovery primitive (Whitepaper §F).
- All four types `Zeroize` on drop. `Debug` for any secret-bearing type returns `"<redacted>"`. `serde::Serialize` is **not** implemented for `VdkKey`, `AuthorityKey`, `DeviceKey` — only for `WrappedVdk` and `VerifyingKey` (the public halves).

Authority key derives an AEAD key for VDK wrapping via:
- `wrap_aead_key = HKDF-SHA512(authority.signing_key_bytes, info="pangolin-vdk-wrap-v0", L=32)` — using `hkdf` crate from RustCrypto.
- The hard separation between "authority signs messages" and "authority's wrap-AEAD-key encrypts VDK" is critical — the same bytes are used for two different keyed primitives, so HKDF with explicit `info` is mandatory.

#### P1-5 — Test vectors

A `tests/` directory in `pangolin-crypto/` with:

- **AEAD test vector** — RFC 8439 known-answer for ChaCha20-Poly1305 base, plus our XChaCha20 vector (the IETF-style construction), with explicit nonces, plaintexts, AADs, ciphertexts.
- **KDF test vector** — RFC 9106 Argon2id vector at our locked parameters.
- **Ed25519 test vector** — RFC 8032 known-answer pair (signing + verification + a deliberately-tampered signature that must fail).
- **VDK wrap/unwrap test** — generate VDK, wrap under authority A, unwrap with A, assert byte-equal. Tamper a byte in `WrappedVdk.ciphertext` — assert unwrap fails. Try to unwrap with authority B — assert fails.
- **Rewrap test** — generate VDK, wrap under A, rewrap with `old=A,new=B`, unwrap with B, assert byte-equal original VDK. Old authority A cannot unwrap the new wrapper.
- **Zeroize test** — using `zeroize` crate's `Zeroizing` wrapper plus a test that drops a `SigningKey` and inspects the underlying memory (best-effort; documented as "not a security claim, just a regression check").

## Success criteria

1. `chacha20poly1305`, `argon2`, `ed25519-dalek`, `hkdf`, `zeroize`, `secrecy`, `rand_core` are the only crypto-bearing dependencies. No other crypto-bearing crate may be added without revising this plan.
2. AEAD round-trip: seal then open returns byte-equal plaintext for arbitrary inputs (property test, ≥1000 cases).
3. AEAD tamper detection: flipping any single bit of ciphertext, nonce, or AAD causes `open` to return `AeadError::Tampered` (property test, ≥1000 cases).
4. AEAD RFC test vector: known-answer test passes against RFC 8439 (ChaCha20-Poly1305) and the XChaCha20 IETF construction's reference vector.
5. Argon2id RFC 9106 test vector passes with our locked parameters.
6. `KdfParams::validate()` rejects parameters below the minimum floor.
7. Ed25519 RFC 8032 known-answer test passes; tampered signatures fail with `SignatureError`; non-canonical signatures rejected by `verify_strict`.
8. VDK wrap/unwrap round-trip: byte-equal recovery.
9. VDK unwrap with **wrong** authority fails with `AeadError::Tampered`.
10. VDK rewrap: after `rewrap(old=A, new=B)`, B can unwrap, A cannot.
11. All secret-bearing types: `Drop` triggers `zeroize` (verified by `Drop` impl test); `Debug` returns redacted output (verified by snapshot test); `Serialize` is **not** implemented (verified at compile time by `static_assertions::assert_not_impl_all!`).
12. No `unsafe` blocks in this crate (workspace-level `unsafe_code = "deny"` covers this; success means CI stays green).
13. `cargo audit` shows zero advisories on the locked dependency set (CI step added to the workflow as part of this issue).

Adversarial tests:
- AEAD ciphertext truncated by 1 byte → must fail.
- AEAD with empty AAD when AAD is required → must fail.
- KDF with shorter-than-required salt → must fail.
- Ed25519 signature length-zero → must fail without panic.
- VDK rewrap with `old == new` (no-op) — must succeed and produce a *new* WrappedVdk (different nonce) that still unwraps to the original VDK.

## Failure modes considered

| Failure mode | Mitigation |
|---|---|
| Nonce reuse with the same AeadKey | XChaCha20's 192-bit nonce makes random nonces collision-resistant for our usage profile (≪ 2^96 messages per key). `Nonce::random()` is the only ergonomic constructor; deterministic-nonce constructors are *not* exposed. |
| `Drop` not running on panic-during-encryption | `zeroize` crate's `Zeroizing` wrapper handles drop-on-unwind. Tests use `std::panic::catch_unwind` to verify this in one targeted case. |
| Argon2 params silently weakened by future PR | `KdfParams::RECOMMENDED` is a `const`; minimum-floor check in `validate()`; PR review enforces. |
| HKDF info-string collision between `wrap` and other future uses of authority key | Info string is versioned: `"pangolin-vdk-wrap-v0"`. Any new derived key must use a distinct info. PR template requires the spec ref + reviewer to call out HKDF-info collisions. |
| Side-channel timing attack on `verify` | Underlying `ed25519-dalek` does constant-time ops. We rely on the upstream crate's audit. We add `cargo audit` to CI. |
| Constant-time comparison missed | Any equality on secret-bearing types uses `subtle::ConstantTimeEq`. `PartialEq` is **not** derived for `AeadKey`, `SigningKey`, `VdkKey`, `AuthorityKey`, `DeviceKey`. |
| Memory not actually zeroized due to compiler optimization | Use `zeroize` crate's `Zeroize` derive — it uses `core::ptr::write_volatile` to defeat optimization. Documented limitation: this is best-effort; OS-level swap and hibernation can defeat it; threat model already acknowledges full-OS-compromise is out of scope (THREAT_MODEL.md §"Out of scope"). |
| Dependency supply-chain compromise | `cargo audit` in CI; `cargo deny` (config in P1 PR) enforces a vetted list of allowed crypto crates and bans others; `Cargo.lock` is committed. |
| Unsafe code introduced via macro or proc-macro | Workspace-level `unsafe_code = "deny"`. Macro expansion that emits `unsafe` will fail the build. |

## Threat enumeration (for THREAT_MODEL.md update)

After P1 lands, append the following to THREAT_MODEL.md under "Local encrypted store" (the surface that consumes pangolin-crypto):

1. **Attacker reads disk image of locked vault.** Defense: every blob is XChaCha20-Poly1305 ciphertext with random nonce + AAD-bound context. AEAD key is a `WrappedVdk` decrypted only after KDF derives wrap key from password + per-device salt.
2. **Attacker steals the `WrappedVdk` blob.** Defense: AEAD ciphertext is computationally infeasible to break absent the authority key.
3. **Attacker has memory dump after vault closed.** Defense: `Zeroize` on drop. Best-effort; not a guarantee against attackers with kernel-level access (out of scope per threat model).
4. **Attacker tampers with on-chain ciphertext.** Defense: AAD binds revision metadata; tampered ciphertext fails AEAD authentication. (Note: this defends against tamper-then-replay; on-chain ciphertext immutability is enforced separately by the contract in P5-1.)
5. **Compromised authority key on one device used to read another device's encrypted local store.** Defense: device keys derive *different* AEAD keys (different HKDF info string). Authority key compromise lets an attacker decrypt the *vault state* (because authority unwraps VDK), but is contained per-device for non-vault-data (sessions, etc.). This is the trade-off the layered model accepts: authority compromise is catastrophic for the vault but does not propagate to other devices' session state.

## Rollback / abort plan

- If `chacha20poly1305` or `ed25519-dalek` is found to have a CVE during P1: stop, evaluate, swap to an alternate vetted crate (the `aes-gcm-siv` family is a backup AEAD; `ed25519-zebra` is a backup signer). Document the swap in DEVLOG and update this plan.
- If RFC test vectors don't match: this means either we're parameterizing the algorithm wrong or the crate has a regression. Stop, file a separate diagnostic issue, escalate to Kelvin. **Do not adjust the test vector to match the implementation.** That's the rule.
- If `Zeroize` cannot be made to work with `[u8; N]` arrays cleanly because of move semantics (a known footgun): wrap arrays in `zeroize::Zeroizing<Box<[u8; N]>>` and document the heap-allocation cost. Estimated cost: ~1-2 KiB per key, negligible.
- If the issue takes >2 days of agent time: stop, post status to DEVLOG, escalate to Kelvin for re-scoping.

## Out of scope

- **VDK lifecycle in MVP-1's session policy engine.** This issue produces the *types and primitives*; using them in a real Vault-open flow is MVP-1 issue 1.4.
- **On-chain signature verification** for the Revision Log contract. P5-1 ships v0 (just stores ciphertext bytes); MVP-2 issue 2.1 ships v1 (verifies signature). The Ed25519 wrappers here are the building blocks for both.
- **Device-key registration** (which device keys are authorized for a vault). That's MVP-2 issue 1.5 and the on-chain registry deferred to MVP-2.
- **Recovery shares / Shamir secret sharing.** Out of scope for P1 entirely. Recovery uses authority rotation + VDK rewrap, not key-share splitting. Master plan §6 / Whitepaper §F.
- **Hardware-backed key storage.** Platform secure storage (Keychain / Keystore / OS keystore) integration is per-platform shell work in MVP-1+ (issues 8.1.2, 8.2.2, 1.5).
- **TOTP** (RFC 6238) is its own crate slot or its own module — MVP-1 issue 1.7. Not in P1.
