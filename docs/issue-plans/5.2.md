## Open questions for Kelvin (LOAD-BEARING — surface at TOP per §16)

### Q-a · Where does the pull loop live — inside Vault (Vault owns the timer) or host-orchestrated (host owns the timer)?

The master plan says "On unlock + every 60s while session active." Three knob shapes for **who holds the timer**:

**Option A: Vault-owned timer.** `Vault::unlock` (or a new `Vault::start_pull_loop`) spawns a tokio background task that fires immediately + then every 60s; a `tokio::sync::oneshot` cancellation handle lives in `ActiveState`; every session-teardown path drops the cancel handle and the spawned task self-cancels on the next tick. Pro: encapsulated; works the moment the vault is unlocked without host wiring. Con: introduces tokio task spawn surface inside `pangolin-store` (today there are ZERO `tokio::spawn` calls in the crate — confirmed by Grep); the spawned task outlives the synchronous `unlock` return; cancellation semantics need careful documentation; `Vault` is currently `!Sync` and per the 4.4 implementation comment "the caller's runtime is single-threaded (host = CLI / Tauri main thread / mobile UI thread)" — a background tokio task would have to either hold `Arc<Mutex<Vault>>` (a refactor of every callsite) or be `!Send` and pin to the runtime's current thread (`tokio::task::spawn_local` + `LocalSet` — works for CLI but the host gets a `LocalSet` shape forced on it).

**Option B: Host-orchestrated.** Vault exposes `Vault::pull_once<A: ChainAdapter>(&mut self, adapter: &A) -> Result<PullReport, PullError>` (async; `&mut self`); the host owns the timer (tokio::time::interval or whatever scheduler the host already has) and calls `pull_once` every 60s + once at unlock. Pro: clean separation; matches 5.1's posture verbatim (5.1's `flush_publish_queue` is host-driven; host owns scheduling); no spawn surface inside `pangolin-store`; the `&mut self` borrow chain compile-time-prevents concurrent flush/edit/pull (matching the 5.1 + 1.x discipline); CLI/Tauri/mobile each implement the timer once for both pull + flush (so 5.4's "SyncOrchestrator" host-side wiring lands naturally). Con: every host has to remember to wire the scheduler.

**Option C: Both.** Vault provides `pull_once` primitive + an optional `Vault::start_pull_loop` convenience that spawns the timer internally for hosts that want it. Hosts that want fine-grained control use the primitive; CLI's simple case uses the convenience. Pro: ergonomics. Con: doubled API surface; Option A's tokio-spawn problem still has to be solved for the convenience path.

**Plan-gate recommends:** **Option B** — host-orchestrated. Matches 5.1 R-h verbatim. `pangolin-store` stays tokio-spawn-free. The 5.4 "always-on auto-sync" cycle will introduce a host-side `SyncOrchestrator` (CLI binary + Tauri shell + mobile UI each owning their event loop), and 5.4 will wire pull-loop + drain-on-teardown + sync indicator together. 5.2 ships only the primitive.

**Kelvin's call.** If Kelvin signs A, the doc has to absorb a tokio-task lifecycle inside `ActiveState` + cancellation discipline + the `!Sync` Vault shape gets a clarifying re-think; the test surface grows. If C, all of A's complications return AND a duplicate API surface.

### Q-b · Interval shape — hardcoded 60s, env-var override, or vault-stored?

Master plan §5 row 5.2 says "every 60s" verbatim. Three knob shapes:

**Option A: Hardcoded.** `pub const PULL_INTERVAL_SECS: u64 = 60;`. No override surface. Pro: simplest; spec-compliant verbatim; one path to test. Con: hermetic tests that want to exercise the cadence path need a clock mock or `tokio::time::pause` (achievable but tedious; the codebase already has `clock: Box<dyn Clock>` on Vault for hermetic clock-control).

**Option B: Const default + env-var override (`PANGOLIN_PULL_INTERVAL_SECS`, clamped 5..=3600).** Mirrors 4.2 R-c (`PANGOLIN_INDEXER_IDLE_TIMEOUT_SECS`) / 5.1 R-a (`PANGOLIN_BATCH_WINDOW_SECS` clamped 1..=300). Pro: consistent env-var pattern across §4/§5; clamp bounds defend against pathological values; test surface drives short intervals without clock mocking. Con: another env-var; clamp bounds need a defensive test.

**Option C: Const + env-var + per-vault stored override.** Three layers — additive nullable column on `meta`. Pro: per-vault tuning. Con: triples the test surface for a cadence knob; per-vault tuning is over-engineering for a 60-second loop that 5.4 will fold into a single host-level orchestrator anyway.

**Plan-gate recommends:** **Option B** — const + env-var clamped 5..=3600. Matches the established §4/§5 env-var pattern verbatim. Min 5s defends against `PANGOLIN_PULL_INTERVAL_SECS=0` flooding the RPC (L-pull-flood); max 3600s caps the longest the user could push the loop (still acceptable; 60s is the spec).

**Kelvin's call.**

### Q-c · Sync mode picker interaction — re-pick per pull cycle, or pick once at unlock?

4.4's `Vault::select_sync_mode` is cheap (read-only: one preference lookup + one `last_synced_block_v1` check; NO RPC call in the current implementation). Three shapes for how 5.2 invokes it:

**Option A: Re-pick per pull cycle.** Every 60s, call `select_sync_mode` first; act on the result. Pro: respects preference flips mid-session (user toggles AlwaysSlow → Auto via a host UI button between two pulls); cheap (single SQL read + a NULL check); the picker's "first-sync only" heuristic naturally degrades to `Slow` once any cycle has advanced the checkpoint past `None`, so even on a fresh vault the cadence settles into Slow after the first successful pull.

**Option B: Pick once at unlock; cache the decision in `ActiveState`.** Pro: marginally faster steady-state (no SQL read per cycle). Con: preference flips don't take effect until next unlock; one more `ActiveState` field; cache-invalidation discipline if 5.4 wires a preference-flip API.

**Option C: Re-pick on preference change only.** Requires a `Vault::set_sync_mode_preference` mutator (already exists from 4.4) to invalidate the cached decision in `ActiveState`. More code; the cache-invalidation surface is one more thing to test.

**Plan-gate recommends:** **Option A** — re-pick per cycle. The picker is functionally a single SQL read + a None check; the cost is negligible against a 60-second cadence + the eventual chain RPC call. Simpler; honest; no cache-invalidation surface. Note: under 4.4's first-sync-only heuristic, once the first cycle's `Slow` path advances the checkpoint, every subsequent re-pick returns `Slow` deterministically.

**Kelvin's call.**

### Q-d · Offline / network-failure backoff — flat retry at 60s, or exponential?

Three shapes for what the loop does when `pull_once` returns `Err(PullError::Chain(_))` because the RPC is unreachable / the network is down:

**Option A: Flat retry every 60s.** On failure, log + try again at the next regular interval. Pro: simplest; 60s is short enough that exponential backoff buys little before the user's next interaction; 5.4 will own the "Offline" indicator anyway. Con: a long offline period generates one failed pull per minute (no real cost; the failed pull is a single eth_blockNumber attempt that errors quickly).

**Option B: Exponential backoff.** Start at 60s; on consecutive failures, double the interval up to a cap (e.g., 600s). Reset to 60s on first success. Pro: lighter RPC pressure during outages. Con: more state in the host scheduler; the user has to wait longer for re-sync after the network returns (mitigated by a manual "pull now" hook).

**Option C: Linear + cap.** Start at 60s; on each failure add 60s; cap at 300s. Reset on success. Middle ground.

**Plan-gate recommends:** **Option A** — flat retry. Simpler; the cost of one failed-RPC ping per minute is negligible; offline detection is 5.4's job (5.4's indicator state machine reads the recent-pull failure state + transitions to "Offline" after N consecutive failures, where N is 5.4's call). 5.2 only ships the per-cycle primitive; the host scheduler decides whether to skip-tick or back off.

**Kelvin's call.** If Kelvin signs B or C, the host scheduler grows backoff state; `pull_once` itself stays unchanged.

### Q-e · Cancellation discipline — how does the pull loop notice teardown?

Three shapes for how the host's scheduler stops calling `pull_once` after `lock()` / idle-expire / 4h-absolute / `device_locked()`:

**Option A: Cancellation oneshot held in ActiveState.** A `tokio::sync::oneshot::Sender` (or `Notify` / `CancellationToken`) is stamped into `ActiveState` on unlock + dropped on every session-teardown path; the host scheduler holds the corresponding `Receiver` and uses `tokio::select!` per tick to check before calling `pull_once`. Pro: explicit signal; the host doesn't have to poll. Con: more wiring; pushes a tokio-specific primitive into `ActiveState` (mirroring the same complication Q-a Option A introduces); the cancellation handle has to be `Send` while `Vault` is `!Send` — needs an `Arc<Notify>` shape.

**Option B: Host scheduler exits when `pull_once` returns `PullError::NoActiveSession`.** Mirrors 5.1's `BatchFlushError::NoActiveSession` posture verbatim. Pro: simplest; no new tokio primitive; the host's scheduler just runs `match pull_once(...) { Err(PullError::NoActiveSession) => break, ... }` in its loop body. Con: scheduler depends on Vault to surface the teardown signal lazily (the host could call pull at most one tick after the lock — bounded by the interval, ~60s worst case; the call returns synchronously with `NoActiveSession`).

**Option C: Vault exposes `Vault::is_active() -> bool` accessor; scheduler polls before each tick.** Pro: explicit; no error-shape coupling. Con: TOCTOU between the `is_active` check and the `pull_once` call (the session could expire in between); the scheduler still has to handle the `NoActiveSession` error path anyway, so this just adds a redundant pre-check.

**Plan-gate recommends:** **Option B** — `PullError::NoActiveSession` posture. Mirrors 5.1's `BatchFlushError::NoActiveSession` verbatim. Scheduler tries to pull; on `NoActiveSession`, it exits the loop. No new tokio primitive; no new accessor. The one-tick worst-case latency between lock-time and scheduler-exit is bounded by the interval (60s) and is host-only — no engine-side leak.

**Kelvin's call.** If Kelvin signs A, `ActiveState` gains a cancellation channel + drop discipline + a `Send`/`!Sync` reconciliation; if C, a one-line accessor is added.

### Q-f · Test surface — hermetic only, hermetic + 1 live `#[ignore]`, or hermetic + property test?

Three shapes:

**Option A: Hermetic only.** Mock chain adapter (existing `MockChainAdapter` + a new `RpcFailingAdapter`) + mock clock (via `Vault::with_clock` precedent) + assert (a) pull cadence (call N times, checkpoint advances N times); (b) cancellation behavior (lock between ticks, next pull returns `NoActiveSession`); (c) retry behavior (transient RPC failure on tick 2, success on tick 3, checkpoint advances). ~12-14 hermetic tests.

**Option B: Hermetic + 1 live `#[ignore]` against D-017.** Same posture as 4.1 R-g / 4.2 R-f / 4.3 R-f / 5.1 R-g. One additional test that pulls a real chain segment against Base Sepolia D-017 + asserts the checkpoint advances + the revisions land. Provides env-quirk #14 contract-side-semantics defense.

**Option C: Hermetic + proptest.** Random sequences of (tick_outcome ∈ {success, transient_error, no_active_session}, teardown_at_tick) tuples; assert invariants (checkpoint monotonic; never advances past last successful tick's head; scheduler exits on NoActiveSession). Pro: catches state-machine regressions. Con: marginal value for a 3-state machine.

**Plan-gate recommends:** **Option B** — hermetic + 1 live `#[ignore]`. Matches §4.x / §5.1 precedent verbatim. Proptest is overkill for the simple state machine (tick-success / tick-retry / tick-stop); the live test defends env-quirk #14.

## Adversarial framing — L-section (load-bearing risks)

> Lighter section than 5.1 — 5.2 is dispatcher + cadence + cancellation; the heavy crypto + on-disk discipline lives in 4.1 / 4.4 underneath. The remaining surface:

### L-pull-during-flush-race

**What goes wrong:** 5.2's `pull_once` and 5.1's `flush_publish_queue` both call into chain primitives + both take `&mut self`. If the host scheduler accidentally invoked them concurrently on the same `Vault` handle, the rusqlite connection could see overlapping transactions OR the wallet borrow could deadlock.

**Mitigation:** structurally impossible. Both methods take `&mut self`; Rust's borrow checker rejects concurrent invocation at compile time. The host scheduler MUST sequence pull + flush through the same single-threaded executor (CLI/Tauri/mobile all hold the Vault on the main loop). Dedicated test `concurrent_pull_and_flush_is_compile_time_enforced` (doc-test that the negative case doesn't compile).

### L-pull-during-edit-race

**What goes wrong:** similar — pull and `account_update` both take `&mut self`. Sequential only.

**Mitigation:** same borrow-checker defense. Compile-time only.

### L-pull-after-lock-races

**What goes wrong:** the host scheduler ticks fire just after `lock()`; the first `pull_once` after lock returns. The flow: `lock()` clears `active`; the next `pull_once` does `if self.active.is_none() { return Err(PullError::NoActiveSession); }` BEFORE any RPC call.

**Mitigation:** the `NoActiveSession` early-return runs before any chain primitive is touched. No data corruption possible. Dedicated test `pull_once_on_locked_vault_returns_no_active_session_before_any_rpc`.

### L-pull-flood

**What goes wrong:** a malicious host or env-var `PANGOLIN_PULL_INTERVAL_SECS=0` (or =1) could flood the RPC + the engine with continuous pull cycles, draining the user's RPC quota + pinning the main loop.

**Mitigation:** env-var clamp (Q-b Option B 5..=3600 minimum). The `5` lower bound caps the rate at 12 pulls/min — well below any realistic RPC rate-limit. The host scheduler is also responsible for not calling `pull_once` faster than the interval; the engine itself never schedules. Dedicated test `env_var_pull_interval_clamps_to_min_5_max_3600`.

### L-checkpoint-corruption-during-pull

**What goes wrong:** inherited from 4.1's L12 (monotonic checkpoint). `sync_from_chain` updates `last_synced_block_v1` atomically with chunk ingest; a mid-chunk crash leaves the cursor at the last fully-ingested chunk_end. 5.2 calls `sync_from_chain` and inherits this defense.

**Mitigation:** 4.1 L12 carries; 5.2 adds zero new attack surface here. The 4.1 reorg-rollback machinery handles backwards-tip cases. Dedicated test `pull_once_under_simulated_rpc_failure_preserves_checkpoint` (mock adapter that errors mid-chunk; assert checkpoint unchanged).

### L-host-scheduler-leak

**What goes wrong:** if the host scheduler doesn't notice teardown (Q-e Option B), the tokio task surrounding the scheduler could outlive the session — calling `pull_once` after `lock()` returns `NoActiveSession`, but the loop body could keep running indefinitely if the host's match-arm doesn't break.

**Mitigation:** documentation contract — the host's pull-scheduler loop MUST exit on `PullError::NoActiveSession`. 5.2 ships an example loop in `docs/architecture/pull-loop.md` showing the canonical shape (`loop { tokio::time::sleep(...).await; match vault.pull_once(...).await { Err(PullError::NoActiveSession) => break, ... } }`). CLI test `cli_pull_scheduler_exits_on_lock` exercises this end-to-end. Engine-side, the `NoActiveSession` return is immediate (one branch check; no RPC); no actual leak.

### L-offer-fast-not-acted-on

**What goes wrong:** the picker returns `OfferFast`, the host prompts the user, the user is AFK, no decision is made; 60s later the next `pull_once` runs the picker again + returns `OfferFast` again; meanwhile the user's vault remains un-synced (slow-mode never ran).

**Mitigation:** **load-bearing UX question** — the host can either (a) auto-fall-through to `Slow` after N seconds of OfferFast-without-decision OR (b) keep prompting. Plan-gate recommends (a): the host scheduler treats a 2-tick-old unacknowledged OfferFast as an implicit decline and re-routes the third tick to `Slow`. This is HOST policy; 5.2 ships only the signal. Documented in `docs/architecture/pull-loop.md` §UX-contract.

### L-revision-replay-via-stale-RPC

**What goes wrong:** a malicious RPC serves the same `eth_getLogs` response twice; the pull loop would attempt to re-ingest events it already has.

**Mitigation:** 4.1's `ingest_pending_chain_revision` is idempotent (the row's natural key is `(account_id, revision_id)` derived from the canonical hash; duplicate inserts no-op). 5.2 inherits this. Dedicated test `pull_once_duplicate_event_response_is_idempotent`.

## Threat-model touch points

`THREAT_MODEL.md` gains a NEW "Pull loop (5.2)" row covering:

1. **Adversary floods RPC via env-var.** L-pull-flood. Mitigated by env-var clamp (5..=3600).
2. **Host scheduler leaks past session teardown.** L-host-scheduler-leak. Mitigated by `NoActiveSession` early-return + documented canonical host loop body.
3. **OfferFast prompt never acted on; user stuck without sync.** L-offer-fast-not-acted-on. Mitigated by documented host UX-contract (auto-fall-through to Slow after 2 ticks).
4. **Stale RPC replays events.** L-revision-replay-via-stale-RPC. Mitigated by 4.1's idempotent `ingest_pending_chain_revision` (natural key dedup via canonical hash).

**Invariant additions:** none new. Existing invariants from 4.1 / 4.4 / 5.1 cover 5.2's surface.
