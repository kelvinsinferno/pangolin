# Issue P8: Sync — `pangolin-cli` publish + pull, dirty-tracking

> **Status:** PLAN-gate. Per Kelvin's autonomy directive, dispatching the build directly upon §16-Plan-Gate approval.
> **Security-critical:** YES per §16.3. P8 is the first issue that wires the **vault** to the **chain** end-to-end; any flaw in the publish/pull plumbing is observable on Base Sepolia and bears directly on Cardinal Principle 3 ("blockchain is a log, never an authority").

## Spec reference

- **Master plan §3.7 EPIC: Sync (P8):**
  - `P8-1` `pangolin-cli publish` — for each dirty account, build revision, sign, submit, update local index.
  - `P8-2` `pangolin-cli pull` — incremental pull from `last_pulled_block`; non-conflicted heads applied automatically.
  - `P8-3` Dirty-entry tracking persisted across restarts.
- **Master plan §3.5:** "manual push/pull, no auto-sync." `pangolin-cli` is invoked by hand; no background daemon.
- **Master plan §3.8 PoC exit criteria:** `pangolin-cli` round-trips against the deployed `RevisionLogV0`; two vaults converge on the same revision graph after publish + pull.
- **DECISIONS.md:**
  - **D-005** — Base is the mainnet target; PoC runs against Base Sepolia at `0x8566D3de653ee55775783bD7918Fe91b66373896` (D-014).
  - **D-006** — No relay; each device is signer + gas payer.
  - **D-007** — No persistent indexer; PoC reads chain directly via `BaseSepoliaAdapter::pull_since`.
- **Cardinal principle 3:** "Blockchain is a log, never an authority." `pangolin-cli pull` must surface forks, never silently auto-resolve them.
- **Whitepaper §G2 (Automatic Sync):** P7 was the local *adapter*; P8 is the *orchestrator* that drives publish/pull cycles.

## What P0..P7 already shipped (the load-bearing surface for P8)

- **P2 / P4 — `pangolin-store::Vault`:** `Vault::create_revision(...)`, `account_heads(id) -> Vec<RevisionId>`, `is_forked(id) -> bool`, `revision_lineage(...)`, `tombstone(...)`, `unpublished_revisions() -> Vec<RevisionId>`, `mark_published(rev_id, ChainAnchor)`, `last_pulled_block() -> u64`, `advance_last_pulled_block(new_block)`. The `chain_anchor` columns on the `revisions` table are populated by `mark_published`; the `sync_state` single-row table holds `last_pulled_block`.
- **P7 — `pangolin-chain`:**

  - `BaseSepoliaAdapter::new_with_keystore(rpc_url, keystore_path, password)` — Foundry-keystore signer, gas-paying secp256k1 wallet.
  - `MockChainAdapter` — `cfg(feature = "test-utilities")`-gated in-memory impl; round-trip-equivalent to the production impl, suitable for unit + integration tests.

- **P6 — `chaincli`:** the **debug oracle**. P6 is intentionally decoupled from `pangolin-cli` — `chaincli` is a "truth-serum" tool, not a UX surface. P8 introduces the *first user-facing CLI* — `pangolin-cli` — that drives a real vault end-to-end through the chain.

## Goal

Ship `pangolin-cli` as the production-shape PoC user surface: open a vault, publish all dirty edits to chain, pull events since `last_pulled_block`, surface forks (without auto-resolving them), and report status. Persist a `dirty_accounts` index across restarts so a publish abort never loses track of which accounts still need to be pushed. After this issue, two devices sharing a vault file (copied between machines for the PoC; the real two-device scenario lands in P11) can edit independently, publish, pull, and converge — except where they conflict, which surfaces as a structured fork report for P9 to resolve.

## Architectural decisions (locked in this plan)

These decisions are committed to as part of plan-gate approval. Any deviation during BUILD requires returning to PLAN per §16.4.

### A1. Where does dirty-tracking live?

**Decision:** A new `dirty_accounts` table in `pangolin-store` (additive, no `format_version` bump; `CREATE TABLE IF NOT EXISTS` at vault open — same idempotent-create posture as P7's `sync_state` table).

**Schema:**

```sql
CREATE TABLE IF NOT EXISTS dirty_accounts (
    account_id   BLOB NOT NULL,        -- 32 bytes; AccountId::as_bytes()
    revision_id  BLOB NOT NULL,        -- 32 bytes; the revision this marker is for
    marked_at    INTEGER NOT NULL,     -- unix-ms when the marker was stamped
    PRIMARY KEY (account_id, revision_id)
);
CREATE INDEX IF NOT EXISTS dirty_accounts_marked_at_idx ON dirty_accounts (marked_at);
```

**Rationale:** P8-3 spec mandates "persisted across restarts," so an in-memory cache is rejected outright. Reusing the SQLite vault file means no new persistence layer to harden. The pair-key (`account_id`, `revision_id`) is justified in A2.

**Alternative considered:** Reusing the existing `unpublished_revisions()` query (filter `revisions WHERE chain_tx_hash IS NULL`) instead of introducing a new table. **Rejected** because (a) `unpublished_revisions` returns *every* unpublished revision in chronological order regardless of head/non-head status — P8 wants per-account "what's the latest unpublished thing for this account" semantics, which is computable but awkward through the existing query; (b) a separate marker table makes "publish was attempted but tx receipt never came back" recoverable without inferring it from the absence of `chain_tx_hash` (which could equally mean "edited offline, never tried to publish"); (c) cleaner audit trail for forensics — the `marked_at` timestamp is independent of `revisions.created_at`.

### A2. Dirty-marker key

**Decision:** Composite primary key `(account_id, revision_id)` — multiple in-flight markers per account are allowed.

**Rationale:** This is robust to the failure mode "publish was sent, tx mined, but the local DB write of `mark_published` + `clear_dirty` was killed before commit." On the next `publish` run we'd otherwise re-publish the *same* `(parent, payload)` — but the contract has no replay protection in v0 (P5-1 audit threat #1: "no on-chain authentication"), so a replay would succeed and create a duplicate revision in chain order with the same `(account_id, parent_revision)`, a *fork by the same device*. By keying on `revision_id` we let the marker track exactly which revision is unpublished; if a second edit happens before the first publish completes, both markers exist; if the first revision turns out to be on chain already (discoverable by querying `pull_since` for that block range or by `mark_published` itself succeeding), `clear_dirty(account_id, rev_id)` removes only that one marker.

**Alternative considered:** Key by `account_id` alone. **Rejected** because of the duplicate-publish risk above and because account-keyed markers cannot represent "two unpublished edits to the same account in sequence" — a real PoC scenario.

### A3. Atomicity of publish

**Decision:** The publish unit is `(account_id, revision_id)`. The on-chain step (`adapter.publish`) is atomic by EVM. The local-DB clearing step is **two writes in one SQLite transaction**: `mark_published` + `clear_dirty(account_id, rev_id)` wrapped in `BEGIN IMMEDIATE … COMMIT`. If the process is killed between the on-chain success and the COMMIT, the next `publish` run will:

1. See the marker still present.
2. Re-attempt `adapter.publish(signed)` — which would, on v0 contracts, double-publish.
3. Mitigation: **before re-attempting, call `adapter.pull_since(vault_id, last_pulled_block, None)` and check whether a revision with the locally-known `revision_id` already appears on chain**. If yes, treat as published and run only the local commit. If no, proceed with publish.

**This pre-publish check is encoded in `sync::publish_all` as the first step per dirty entry.** It costs one extra `eth_getLogs` call per stale marker, which is acceptable for a manually-invoked CLI.

**Why not rely on contract-side `nextSequence` invariant for replay rejection:** v0 RevisionLogV0 increments `nextSequence` unconditionally on each call; it does not reject duplicate `(parent, payload)` submissions. P5-1's audit confirms this (threat #2: "v0 has no on-chain authentication … any address can call `publishRevision` with arbitrary bytes"). v1 (MVP-2 issue 2.1) will add signature verification but still won't reject duplicates structurally — duplicates with the *same* `device_id` and *same* `(account_id, parent)` produce a fork the device wouldn't intend, but the contract has no model of "which fork is the device's own." So the **client** must avoid double-publish; the contract cannot.

**Alternative considered:** Skip the pre-publish check and accept the duplicate-publish risk as a known operator hazard ("if you kill the CLI mid-publish, run `chaincli list` to verify before retrying"). **Rejected** as too easy to forget; the PoC must demonstrate that `pangolin-cli publish` is safe to re-run after any failure.

### A4. Pull conflict surfacing

**Decision:** When `pull_all` encounters two children for the same parent revision (a fork), it:

1. Stores **both** child revisions in `pangolin-store` via `ingest_chain_revision`.
2. Marks the account as having multiple heads (this falls out automatically — `Vault::is_forked(id)` already returns true when `Vault::account_heads(id)` returns more than one).
3. Returns a `PullReport { applied: usize, forks: Vec<ForkSummary> }` where `ForkSummary { account_id, head_revision_ids: Vec<RevisionId> }`.
4. Prints the fork summary on stderr (one line per forked account, with the head IDs in hex).
5. Advances `last_pulled_block` to the chunk boundary regardless — pulling forks is not an error, it's the chain telling us what happened.

**Auto-resolution is explicitly NOT done.** P9 (`pangolin-cli resolve <account-id> --keep <revision-id>`) is the resolution UX; P8's responsibility is to surface forks reliably.

**Rationale:** Cardinal principle 3 — chain is a log, never an authority. The pull path stores what the chain says happened; the resolution policy is a user-facing decision, not a transport-layer one.

### A5. Pull range chunking — and the resolution of P7 audit MED-3

**Decision:** `pull_all` chunks the block range `[last_pulled_block + 1, current_head]` into `PULL_CHUNK_SIZE = 8_000`-block windows. After each chunk's events have been ingested, `Vault::advance_last_pulled_block(chunk_end)` is called *before* the next chunk's `pull_since` call. A chunk failure (RPC flake, decode error) returns from `pull_all` with `Err(...)`; partial progress through prior chunks is preserved on disk. Re-running `pangolin-cli pull` resumes from the new `last_pulled_block`.

**This is where P7's MED-3 actually matters.** P7's `pull_since` is *all-or-nothing* per call (P7 audit MED-3 deferred the partial-progress redesign to P8). P8 wraps that all-or-nothing primitive in a per-chunk loop where the *checkpoint* is the unit of progress. So the trait does not need to change; only the orchestrator does. The 8 000-block chunk size is below the 9 000-block window `BaseSepoliaAdapter` uses internally (which is in turn below the public RPC's 10 000-block cap), so each `pangolin-cli pull` chunk maps to exactly one `eth_getLogs` round-trip in the production adapter.

**Configurability:** `PULL_CHUNK_SIZE` is a `pub const` in `tools/pangolin-cli/src/sync.rs`. No CLI flag for the PoC; if it ever needs to be tuned, MVP-1 adds `--chunk-blocks` to the `pull` subcommand.

**Rationale:** Resolves MED-3 without a trait-shape change to `ChainAdapter`. Smaller chunks = better failure-resilience but more round-trips; 8 000 is a deliberate engineering pick that gives a >10x catchup-failure tolerance over a single all-or-nothing call for any vault that's been offline for more than a day's worth of blocks (Base Sepolia produces ~43 200 blocks/day at 2-second block time).

**Alternative considered:** Add a `pull_since_streaming` method to `ChainAdapter` and drive the per-chunk advancement at the trait level. **Rejected** because (a) the orchestrator can already achieve per-chunk semantics with the existing trait by making N calls; (b) adding a streaming method now would re-locking work that P8 hasn't yet validated; (c) MVP-2 issue 4.1 (ephemeral local indexer) is the right surface for streaming pulls of large backlogs — for the PoC, simple chunked reads suffice.

### A6. Network failure semantics — fail-fast, no built-in retries

**Decision:** Both `publish_all` and `pull_all` fail fast on the first transient error. No exponential-backoff retry loop inside the CLI. The user re-runs `pangolin-cli publish` or `pangolin-cli pull` manually.

**Rationale:** Master plan §3.5 explicitly mandates "manual push/pull, no auto-sync." Building retry loops drifts toward auto-sync. The idempotency guarantees from A3 + A5 mean a manual re-run is always safe. For `publish_all`, per-account error isolation (A3 enforced via per-account try/catch in `sync::publish_all`) means one account's RPC flake doesn't block the others within a single run — but if the RPC is fully down, the run reports all-failed and the user retries once the network is up.

**Alternative considered:** Add a single retry on transient errors (`alloy::transports::Transport`-classified retryable errors) per account. **Rejected** for PoC simplicity. MVP-1 may revisit if user testing shows the manual-re-run UX is too brittle.

### A7. Signing key sourcing — two-key model, matching P7

**Decision:** `pangolin-cli publish` uses the same two-key model that P7's `BaseSepoliaAdapter::new_with_keystore` was designed for:

- **Gas-paying secp256k1 wallet** — loaded from a Foundry keystore via `--account <name>` (or `--keystore-path <path>` as an alternative), password prompted at the terminal via `rpassword`. Same path as `chaincli publish`.
- **Revision-signing Ed25519 key** — derived from the vault's `DeviceKey` (P3); used by `signing::build_signed_revision` to produce the signature stamped on each `SignedRevision`.

The two keys are deliberately **separate**. P7 ships `evm::derive_evm_wallet(&DeviceKey)` for the alternative model where the gas wallet is *deterministically derived* from the device's Ed25519 seed (D-006 wording: "device's keypair is both the revision signer and the gas payer"). **P8 does not use that derivation.** Instead, P8 uses an externally-funded Foundry keystore for gas, matching `chaincli`'s pattern, and acknowledges in the threat-model row (§5 below) that the PoC ships with a two-key model that diverges from D-006's wording.

**Rationale:**

1. The PoC's funded-wallet strategy is "the dev funds the keystore"; using the derived EVM wallet would require funding the derived address — solvable but adds complexity to the PoC test setup with no security benefit.
2. The `evm::derive_evm_wallet` derivation is *available and tested* (P7 audit HIGH-1); switching to it is a one-line change in MVP-1 once the funder service (MVP-2 issue 3.4) lands.
3. v0 contract doesn't verify any signature, so the choice of revision-signing key is invisible on-chain at v0 — but the client always signs anyway as forward-prep for v1 (P7 §approach).

**Alternative considered:** Use `evm::derive_evm_wallet` to derive the gas wallet from the device key, satisfying D-006's exact wording. **Rejected for PoC** for the operational reason above; flagged for revisit in MVP-1.

This is the **one decision in P8 that adds friction with a locked decision (D-006).** The deviation is local to the PoC and reversible. If this is unacceptable, raise as Open Question §8.

### A8. Crate location and binary name

**Decision:** Crate lives at `tools/pangolin-cli/`, mirroring `tools/chaincli/`. Crate name: `pangolin-cli`. Binary name: `pangolin-cli`.

**Rationale:** The §16.8 repository layout shows `crates/pangolin-cli/` as the *PoC CLI; thin shell over core*. However, the §16.8 layout was drafted before `chaincli` was placed under `tools/` (P6); the convention "binaries that are tools live in `tools/`, libraries that other crates depend on live in `crates/`" has crystallized over P6 + P7. `pangolin-cli` is a binary with no library exports. It belongs in `tools/`.

**Note:** The §16.8 layout in the master plan references `crates/pangolin-cli/` — this plan deviates intentionally. Update the master plan layout table at SIGNOFF to record `tools/pangolin-cli/`. (No D-NNN required — it's a layout convention, not a security or architecture decision.)

**Alternative considered:** `crates/pangolin-cli/` per the master plan layout. **Rejected** for consistency with `tools/chaincli/`. If Kelvin prefers the master plan's wording, this flips trivially.

```rust
pub struct DirtyEntry {
    pub account_id: AccountId,
    pub revision_id: RevisionId,
    pub marked_at: u64, // unix-ms
}

impl Vault {
    /// Stamp `(account_id, revision_id)` into `dirty_accounts`.
    /// Idempotent — `INSERT OR IGNORE`. Auto-called from
    /// `Vault::create_revision`; callers don't usually need to invoke
    /// it directly.
    pub fn mark_dirty(&mut self, account_id: AccountId, revision_id: RevisionId) -> Result<()>;

    /// Remove the marker for `(account_id, revision_id)`. No-op if no
    /// marker exists.
    pub fn clear_dirty(&mut self, account_id: AccountId, revision_id: RevisionId) -> Result<()>;

    /// Snapshot the current dirty list, sorted by `marked_at` ASC.
    pub fn list_dirty(&self) -> Result<Vec<DirtyEntry>>;
}
```

```rust
// in crates/pangolin-store/src/vault.rs (additive)
impl Vault {
    /// Persist a chain-side revision into the local log.
    ///
    /// Distinct from `create_revision`: this is the *ingest* path used
    /// by `pangolin-cli pull`. The revision is stored with its chain
    /// anchor populated from the `RevisionEvent`. The dirty marker is
    /// NOT stamped (the revision is already on chain).
    ///
    /// Idempotent: if a revision with the same `revision_id` already
    /// exists locally, returns `Ok(IngestOutcome::AlreadyPresent)`
    /// without touching the row. Used by `pangolin-cli pull` for
    /// overlap chunks.
    pub fn ingest_chain_revision(
        &mut self,
        event: &pangolin_chain::RevisionEvent,
    ) -> Result<IngestOutcome>;
}

pub enum IngestOutcome {
    Inserted,
    AlreadyPresent,
}
```

`ingest_chain_revision` is the **one new pangolin-store API the chain side calls into.** It's keyed off `revision_id` (the canonical hash from `signing.rs`) so an event whose ID matches a locally-created revision (the device's own publish coming back as a pulled event) is recognized and treated as `AlreadyPresent`.

## Threat model row (draft text for THREAT_MODEL.md)

To be inserted between the existing `pangolin-chain` row (lines 85+) and the next row down. Same depth as the `pangolin-chain` row (8+ enumerated threats with defenses). Format:

> ### Pangolin sync orchestrator (`pangolin-cli`)
>
> > Source: `docs/issue-plans/P8.md` §"Threat model row" and the P8
> > build-gate. The `pangolin-cli` binary at `tools/pangolin-cli/` is
> > the user-facing PoC orchestrator that drives `pangolin-chain` to
> > publish dirty revisions and pull chain events into the local
> > vault. It does not introduce new cryptographic primitives; the
> > threats below concern orchestration, idempotency, and the
> > publish/pull state machine.
>
> 1. **Forged publish (foreign device claiming to be the user's).**
>    Defense: revisions are signed by the vault's `DeviceKey`
>    (Ed25519); `signing::build_signed_revision` binds the
>    canonical-hash digest to the device's Ed25519 verifying-key bytes
>    (`device_id`). v0 contract does **not** verify on-chain; v1 will
>    (MVP-2 issue 2.1). For PoC purposes, the canonical-hash discipline
>    means a foreign device's forged revision is detectable by *any*
>    receiving vault that runs `verify_signed_revision` after pulling
>    — a check P8 should add to the ingest path before persisting an
>    event. **TODO:** add `verify_signed_revision` to
>    `Vault::ingest_chain_revision` as a defense-in-depth measure even
>    though v0 contract has no signature semantics. Track as
>    audit follow-up at SIGNOFF.
> 2. **Replay of an old signed revision.** Defense: the canonical-hash
>    digest binds `parent_revision`. A revision with a stale parent
>    cannot apply to a moved-on head; ingestion only succeeds when the
>    parent is a known head. Re-publishing the same revision is
>    additionally guarded by the A3 pre-publish check
>    (`pull_since` + locally-known-revision-id lookup before any
>    re-publish attempt).
> 3. **Network partition during chunked pull.** Defense: A5 — the
>    chunked-pull design means a chunk failure preserves prior
>    chunks' progress on disk via `advance_last_pulled_block` per
>    chunk. Re-running `pangolin-cli pull` resumes from the new
>    `last_pulled_block`. Resolves P7 audit MED-3.
> 4. **Dirty-entry leak.** Defense: the `dirty_accounts` table stores
>    only `(account_id, revision_id, marked_at)`. `account_id` is
>    already an attacker-observable identifier on-chain (the
>    `RevisionPublished` event includes it as topic 2). `revision_id`
>    is the canonical-hash digest of the revision payload, which
>    becomes observable on chain anyway when the revision publishes.
>    `marked_at` is a unix-ms timestamp local to the device — the only
>    new piece of metadata, and it leaks only "when did this device
>    edit account X for the n-th time," visible only to an attacker
>    who has already compromised the local vault file (in which case
>    they have the AEAD-protected ciphertext, dwarfing the timing
>    leak).
> 5. **Replay protection across vaults.** Defense: `vault_id` is
>    bound into `signing::canonical_hash`, so a revision signed for
>    vault A cannot be replayed against vault B even by the same
>    device. The chain event includes `vault_id` as topic 1; pulled
>    events for vault B never include vault A's revisions because
>    `pull_since` filters server-side on `vault_id` topic equality.
> 6. **Pre-publish check race.** Defense: the A3 pre-publish check
>    runs `adapter.pull_since(vault_id, last_pulled_block, None)`
>    before any re-attempt. The race is: two devices publish the same
>    `(parent, payload)` simultaneously and both A3 checks succeed
>    (each sees the chain without the other's revision yet). Both
>    publishes succeed and create a fork. This is **expected
>    behavior** — concurrent edits by independent devices fork; A4
>    surfaces the fork on the next pull; P9 resolves it. The race is
>    not a defect.
> 7. **Misuse of `MockChainAdapter` in the binary.** Defense:
>    `pangolin-cli` does not enable `pangolin-chain`'s
>    `test-utilities` feature in its default `Cargo.toml`.
>    `MockChainAdapter` is consequently not constructible from the
>    production build. Tests compile a separate `cfg(test)` config
>    with the feature enabled. P7 audit's gating discipline is
>    inherited.
> 8. **Two-key model gas-wallet correlation.** Defense
>    (acknowledgement): the gas-paying secp256k1 wallet (Foundry
>    keystore) is **separate** from the device's Ed25519
>    revision-signing key. The gas wallet's address appears as the
>    `tx.from` on every publish; an observer who learns "address X
>    paid gas for these revisions" can correlate all publishes from
>    the same machine across all vaults that share the keystore. This
>    is the same observability surface as P7 audit threat #6 (EVM
>    address observability) and the same Phase-2 mitigation applies
>    (per-publish wallet rotation via funder service, MVP-2 issue
>    3.4). The PoC-specific divergence from D-006 (which mandates
>    *one* keypair as both signer and gas payer) is documented in
>    §A7 of the P8 plan; MVP-1 will switch to `evm::derive_evm_wallet`
>    to satisfy D-006's wording.

## Failure modes considered

| Failure mode | Mitigation |
|---|---|
| User runs `pangolin-cli publish` while vault is `Locked` | CLI prompts for password (or uses `--vault-password`); on wrong password, surfaces `StoreError::AuthenticationFailed` verbatim and exits non-zero. |
| User runs `pangolin-cli publish` with no Foundry keystore configured | Clap argument parsing requires either `--account` or `--keystore-path`; missing both → clap surface error before any vault unlock. |
| User runs `pangolin-cli pull` against a vault whose `vault_id` doesn't match anything on chain | `pull_since` returns zero events; `pangolin-cli pull` reports "0 applied, 0 forks, last_pulled_block advanced to <head>" — same as a quiet pull on a current vault. |
| Process killed between on-chain publish success and local clear-dirty commit | A3 pre-publish check on next run prevents double-publish. Verified by the `publish_idempotent_on_rerun_after_partial_failure` test. |
| RPC returns an event whose decoded `vault_id` differs from the requested filter | `BaseSepoliaAdapter` already drops these (P7 audit MEDIUM-4); `pangolin-cli` inherits the defense. |
| RPC returns an event whose `parent_revision` doesn't exist locally | Stored as an "orphan" — the chain has it, the local store has it, but `account_heads` for that account does not include it as a head until the parent is also pulled. (Pulls process events in chain order, so this only happens during partial-progress recovery; subsequent pulls of earlier blocks fill in the gap.) **Defense:** `ingest_chain_revision` does not fail on missing parent; it persists the event and the head-computation in `pangolin-store` (P3) handles partial-graph gracefully. |
| Wrong chain ID returned by RPC | `BaseSepoliaAdapter` already verifies at construction (P7 audit threat #2); `pangolin-cli` surfaces `ChainError::WrongChain` verbatim. |
| Chain reorg drops a published revision | Out of scope for P8 (and P7). Documented as a known limitation. MVP-2 may add reorg-aware pull logic. |
| `pangolin-cli pull` discovers the revision count to ingest is huge (e.g., 5000+ over a stale vault) | A5 chunked design means each chunk is bounded; pull is interruptible (Ctrl-C between chunks) without losing progress. The 5000-revision case is the motivating example for D-007's "ephemeral local indexer" — but that's MVP-2; for PoC, chunked direct reads are sufficient. |
| User edits the vault while `pangolin-cli publish` is mid-run | Vault file lock is held by `pangolin-cli`; the user's second invocation will hit the file-lock and fail-fast with "vault is in use." Inherited from `pangolin-store`'s lock discipline. |

## Rollback / abort plan
