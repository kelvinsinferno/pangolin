# Issue 3.6: Privacy mitigation — Phase-2 hooks (scaffolding only)

> One-line scope: ship the **trait / enum / accessor scaffolding** that Phase-2 Enhanced Privacy Mode (per-revision wallet rotation; CoinJoin pre-mixing of funder top-ups; optional fresh-address-per-vault) plugs into without forcing a refactor of 3.1 / 3.2 / 3.3 / 3.4 / 3.5. Whitepaper §8.3 is the spec reference; D-006 is the on-chain-observability mitigation. **NO production logic for any rotation / mixing / fresh-address behavior ships in 3.6.** The point of shipping 3.6 NOW (rather than deferring to MVP-3 / MVP-4) is **architectural locking**: the hook shapes get pinned while §3.x is fresh in everyone's head, so when Phase-2 lands in a later MVP the surfaces don't need to be re-shaped.
>
> **Status:** Plan-gate DRAFT 2026-05-15, awaiting Kelvin sign-off on Q-a..Q-d (top of doc). Builder must not start until Kelvin lands resolutions in a "Resolved decisions" table mirroring 3.1 / 3.2 / 3.3 / 3.4 / 3.5. L1..L7 are non-negotiables independent of how Q-a..Q-d resolve.
>
> **Security-critical: NO.** 3.6 ships ZERO production logic and ZERO observable behavior change against the 3.5 baseline. The blast radius of a 3.6 bug is "a Phase-2 implementer in MVP-3 has to adjust a trait signature" — not "a user loses gas" or "a signing primitive drifts." That said: the load-bearing property below (L1 / L4) is that the no-op default path MUST be byte-identical to the 3.5 behaviour; a regression there IS a behavioural drift on the hot path and would be caught in CI by a byte-equality test against pre-3.6 reference vectors.
>
> **Depends on:** 3.1 (the `build_signed_revision_v1(wallet, fields, enc_payload, env)` signature — already wallet-parameterised; the hook for per-revision rotation slots in here); 3.2 (`Vault::evm_wallet` + `Vault::evm_wallet_address`; the per-vault address hook would either return a different wallet or augment the accessor); 3.3 (publish path consumes the wallet from `Vault::evm_wallet`; the hook is whatever shape the wallet-selection chooses); 3.4 (`pangolin-funder-client::initiate_top_up`; the CoinJoin pre-mixing hook would intercept the funder-response handling); 3.5 (`balance_check.rs` `query_evm_balance(address)` — if Phase-2 ships per-vault addresses, balance aggregation across N derived wallets becomes a separate hook; aggregation logic itself is deferred); D-006 (the documented on-chain observability mitigation reference); Whitepaper §8.3 (Enhanced Privacy Mode spec, reproduced below). ## Open questions for Kelvin (LOAD-BEARING — surface at TOP)

### Q-a · Abstraction shape — enum, trait, or both?

Three plausible shapes for the scaffolding:

**Option A: `PrivacyMode` enum + `PrivacyConfig` struct only.** A user-visible enum `PrivacyMode { Default, EnhancedPrivacy }` + a config struct passed down to the signing / balance / funder paths; callsites `match` on it. Phase-2 implementations add new variants OR replace `match` arms with real logic.

```rust
pub enum PrivacyMode { Default, EnhancedPrivacy }
pub struct PrivacyConfig { pub mode: PrivacyMode, /* future fields */ }
```

**Pro:** Smallest surface; no new dyn-dispatch; easy to test. **Con:** every Phase-2 hook becomes a new `match` arm scattered across crates; encapsulation of Phase-2 logic in one place fights the grain.

**Option B: `PrivacyStrategy` trait only.** Methods like `derive_wallet_for_revision(&DeviceKey, revision_index: u32) -> Result<EvmWallet>`, `transform_funder_response(TopUpResponse) -> TopUpResponse`, `select_address_for_vault(&Vault) -> Address`. Two impls: `DefaultStrategy` (no-op; current behavior) + future `EnhancedPrivacyStrategy` (placeholder that returns an error). Callsites carry a `&dyn PrivacyStrategy` (or `Arc<dyn PrivacyStrategy>`).

**Pro:** Phase-2 logic encapsulates inside one impl; clean separation. **Con:** dyn-dispatch + Send/Sync bounds + ownership questions (where does the strategy live? `Vault` owns it? a per-call argument? lazy_static?).

**Option C: Both — enum for user-visible config + trait for hook points.** The enum is the host-facing knob; the trait is the internal implementation surface that callsites consume. A small adapter (`PrivacyConfig::into_strategy()`) maps the user-config enum to a `Box<dyn PrivacyStrategy>`. **Plan-gate recommends Option C** — best of both worlds at the cost of one extra type; matches how 3.4's `FunderSigner` trait + `FileKeystoreSigner` impl is shaped (user-facing config, trait-based impl).

**Option D: Feature-flag only (no runtime config).** `#[cfg(feature = "enhanced-privacy")]` gates the hook bodies. **Con:** loses runtime configurability; means rebuilding the binary per-user-preference; doesn't fit "optional per-revision rotation" or "per-device toggle" from Whitepaper §8.3 ("toggled per-device or per-update").

**Kelvin's call.**

### Q-b · Which of the three privacy modes get scaffolding in 3.6?

Master plan §5 row 3.6 enumerates three:

1. **Per-revision wallet rotation** — each revision uses a fresh-derived wallet from the device seed (via a counter / revision-index parameter to derive_evm_wallet); the hook is a new `derive_evm_wallet_at_index(&DeviceKey, index: u32) -> Result<EvmWallet>` + a callsite at the revision-signing path.

2. **CoinJoin pre-mixing of funder top-ups** — the funder client's `initiate_top_up` would intercept the funder's response to mix the ETH through a non-custodial CoinJoin tool BEFORE it lands at the device wallet; the hook would be a "transform_funder_response" callback on the strategy + an extra step in the top-up flow.

3. **Optional fresh-address-per-vault** — each vault has its own EVM wallet (not the shared device wallet); `Vault::evm_wallet()` would derive a per-vault wallet via a vault_id-keyed derivation.

**Option A: All three.** Maximum architectural locking; most scaffolding code.

**Option B: Just rotation + per-vault.** The CoinJoin hook depends on a CoinJoin tool integration (the off-chain mixer client); MVP-3 / MVP-4 territory. Defer the CoinJoin hook entirely.

**Option C: Just per-revision rotation.** The smallest viable scaffolding — the rotation hook is the one most tightly coupled to the existing 3.1 signing primitive; the other two are independently shapable later.

**Option D: All three, but the CoinJoin hook is a placeholder type (`pub trait CoinJoinClient: Send + Sync { ... }`) with no impl beyond a no-op stub.** **Plan-gate recommends Option D** — all three scaffolds, but with the CoinJoin hook reduced to a trait declaration + no-op impl; the trait's existence is the "architectural locking" deliverable; full CoinJoin client integration stays MVP-3 / MVP-4.

**Kelvin's call.**

### Q-c · Where does the scaffolding live — new crate or in-place?

The hooks are cross-cutting:

- Per-revision rotation → `pangolin-chain` (derivation) + `pangolin-store::Vault` (which wallet `sign_revision_v1` uses).
- CoinJoin pre-mixing → `pangolin-funder-client` (response transform).
- Per-vault address → `pangolin-store::Vault` + `pangolin-chain` (derivation).

**Option A: Consolidate into a new `pangolin-privacy` crate.** All trait/enum definitions live there; the other crates depend on it. **Pro:** clean single-source. **Con:** new workspace member; new dep edge; for ZERO production logic it's heavy.

**Option B: Distribute the scaffolding across the touched crates in-place.** `pangolin-chain::privacy` module for derivation hooks + `pangolin-funder-client::privacy` module for the CoinJoin transform + `pangolin-store::Vault` grows accessor variants. **Pro:** no new crate; matches the locality of each hook. **Con:** the three modules can drift in naming / shape.

**Option C: Hybrid — central enum + trait declarations in `pangolin-chain::privacy` (since `pangolin-chain` already declares wallet primitives and is a dep of everything chain-touching), impls in each consumer crate.** **Plan-gate recommends Option C** — central declarations live where wallet primitives already are; impls live next to the callsites they hook into. `pangolin-funder-client` already depends on `pangolin-chain`-shaped types implicitly via `Address`/`PrivateKeySigner`; an explicit dep on the `pangolin-chain` privacy module is consistent.

**Kelvin's call.**

### Q-d · Test surface for scaffolding-only code.

How do you test "this hook exists and is callable" without testing actual Phase-2 logic?

**Option A: Compile-time tests only.** Assert the trait is `Send + Sync`; assert `DefaultStrategy` impls the trait; assert the enum variants are pinned (name-stability test).

**Option B: Compile-time + bytewise-identity test against 3.5 baseline.** Run a representative 3.1 + 3.3 + 3.4 + 3.5 flow with `PrivacyMode::Default` and assert the produced signatures / calldata / tx hashes / balance-state outputs are byte-identical to a fixture captured pre-3.6. This catches the L1+L4 invariant (zero behavioural drift).

**Option C: Compile-time + bytewise-identity + an "enabled path returns error" test.** The `EnhancedPrivacy` variant has no impl yet; the path that instantiates it MUST fail loudly (per L7) with `ChainError::PrivacyModeNotImplemented { mode: EnhancedPrivacy }` or equivalent — fail-closed rather than silently degrade.

**Plan-gate recommends Option C.**

**Sub-question:** is the bytewise-identity fixture a captured-bytes file (golden test) or a re-derived value via a "build under 3.5-compatible flags" CI job? Plan-gate recommends captured-bytes file in `crates/pangolin-chain/tests/fixtures/3.6-baseline.json` (small surface; reviewable diff).

## Adversarial framing — L-section (light, per scope)

### L-3.6-accidentally-ships-partial-phase-2

**What goes wrong:** The builder over-interprets "scaffolding" and ships partial Phase-2 logic (e.g., implements `derive_evm_wallet_at_index` and wires it into the revision-signing path). The default-mode path now uses a derived wallet that differs from the 3.5 baseline; signed-revision bytes drift; chain submissions go to a different address than 3.5 published from; users who upgraded to 3.6 effectively migrated their on-chain identity without consent.

**Defense:** L1 + L4 verbatim. The bytewise-identity fixture test (Q-d Option C) is the mechanical lock. CI runs the fixture comparison every PR; a drift fails the build.

**Adversary leverage:** none directly (a builder isn't an adversary), but the failure mode is severe — users could lose linkability of their old on-chain history without noticing.

### L-trait-shape-drift-from-phase-2

**What goes wrong:** 3.6 ships a trait shape that doesn't actually fit what Phase-2 needs. When MVP-3 / MVP-4 lands the real CoinJoin client or rotation logic, the trait has to be re-shaped — defeating the architectural-locking purpose.

**Defense:** (a) the plan-gate doc explicitly references Whitepaper §8.3 + master plan §5 row 3.6 for the three modes; (b) the trait methods are designed against the THREE specific Phase-2 modes (not speculative future ones); (c) a `// TODO(Phase-2): adjust signature if real impl needs ...` comment lives next to each hook so the next builder has explicit license to renegotiate; (d) Kelvin's Q-a / Q-b sign-off pins which modes are scaffolded.

**Adversary leverage:** none; failure mode is "the 3.6 scaffolding was wasted effort and gets thrown away in MVP-3."

### L-enabled-path-silent-degrade

**What goes wrong:** The `EnhancedPrivacy` variant accidentally falls through to `Default` behaviour (e.g., a `match` arm with `_ => default_path()` catches it). User who opted in for Phase-2 privacy gets the observable default behaviour silently; they think they have privacy, they don't.

**Defense:** L7 verbatim. The `EnhancedPrivacy` variant errors with a typed `ChainError::PrivacyModeNotImplemented`. A pinning test asserts: `assert!(matches!(strategy_for(PrivacyMode::EnhancedPrivacy), Err(ChainError::PrivacyModeNotImplemented { .. })))`.

**Adversary leverage:** privacy-loss in the user model. Mitigated structurally.

### L-doc-drift-from-§8.3

**What goes wrong:** The 3.6 scaffolding gets shaped around what's easy to implement now rather than what Whitepaper §8.3 says. When Phase-2 lands, the spec changes were silently encoded in the wrong direction.

**Defense:** the plan-gate doc reproduces §8.3 verbatim (below in "Whitepaper §8.3 reference"). The R-rows record Kelvin's sign-off pinning which interpretation of §8.3 the scaffolding encodes.

## Whitepaper §8.3 reference (verbatim)

From `Pangolin Whitepaper.pdf` §8.3 "Enhanced Privacy Mode (Optional / High-Threat)":

> For users requiring maximal metadata protection, the system supports integration with non-custodial cryptocurrency mixers (e.g., CoinJoin protocols) when publishing on-chain updates or commitments. This obfuscates transaction origins, breaking traceable links between wallet addresses and update patterns without relying on centralized services.
>
> Key properties:
> - Mixers are invoked client-side as an optional step during synchronization.
> - Only non-custodial variants are recommended to avoid introducing trusted third parties.
> - This mode adds nominal fees and delays but enhances resistance to chain analysis surveillance.
> - It is compatible with both Object and Commit Modes, and can be toggled per-device or per-update.
>
> Users are advised to evaluate legal and operational risks based on their jurisdiction, as mixers may attract regulatory attention despite their privacy benefits.

**Notable gap between §8.3 and master plan §5 row 3.6:** §8.3 explicitly names ONLY CoinJoin mixing of on-chain updates / commitments. Master plan §5 row 3.6 expands this to THREE modes: per-revision wallet rotation, CoinJoin pre-mixing of funder top-ups, AND optional fresh-address-per-vault. The two later modes are NOT in §8.3 — they are master-plan-level extensions of the §8.3 posture. D-006 records the "per-revision wallet rotation" + "CoinJoin pre-mixing of funder top-ups" pair as the Phase-2 Enhanced Privacy Mode mitigations. **Builder MUST surface this gap with Kelvin before pinning trait shapes** — the scaffolding should reflect master plan §5 row 3.6 (three modes) since that is the authoritative source for what MVP-2 / MVP-3 / MVP-4 will ship, with §8.3 cited as the broader Whitepaper-level posture.
