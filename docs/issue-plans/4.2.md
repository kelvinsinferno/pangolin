### Q-a Â· Crate organization â€” single binary-and-library crate vs. multi-crate split

The `pangolin-indexer` crate exists today as a placeholder library at `crates/pangolin-indexer/{Cargo.toml, src/lib.rs}` (21 lines; just `pub fn name() -> &'static str { "pangolin-indexer" }`). 4.2 expands it. Three plausible shapes:

**Option A: Single `pangolin-indexer` crate â€” library + binary.** `lib.rs` for the indexing logic + lifecycle + IPC types; `src/bin/pangolin-indexer.rs` (or `[[bin]] name = "pangolin-indexer", path = "src/main.rs"`) for the standalone binary entry. Desktop spawns the binary; mobile uses the library directly in-process. Cargo-standard pattern. **Pro:** minimal scope; one Cargo.toml; one set of deps; one set of tests. **Con:** the binary depends transitively on the full library â€” including any tokio + alloy + tempfile transitive surface â€” which is fine for desktop (where you ship the binary anyway) but adds size to a mobile app that only wants the library. Plan-gate reads that cost as low (the library is what mobile imports either way).

**Option B: Two crates â€” `pangolin-indexer-lib` + `pangolin-indexer-bin`.** Separation of concerns: library is the substrate; binary is the desktop entry. **Pro:** binary's deps (clap for argv, the IPC server stub) don't leak into mobile builds. **Con:** double-bookkeeping for `Cargo.toml`; two sets of SPDX headers; an artificial cross-crate boundary that nothing structural justifies.

**Option C: Three crates â€” lib + bin + `pangolin-indexer-client`.** The "host-side API the desktop spawns + talks to" is split out as a separate crate so the host (CLI / FFI / Tauri shell / etc.) can pull just the client surface without the indexing logic. Mirrors the `services/funder/` + `pangolin-funder-client` split (where `services/funder/` is the binary that runs on a server + `pangolin-funder-client` is the thin client library callers use). **Pro:** clean substrate / endpoint separation; the host only sees the `Client` API + the IPC types, not the indexer guts. **Con:** three Cargo.tomls; deferral risk if a refactor later collapses them.

**Plan-gate recommends:** Option C â€” the funder pattern is the strongest precedent in tree for this exact shape (binary that runs in another process + client crate that talks to it). The indexer is functionally the same shape: a co-process the host orchestrates. Same architectural-locking benefit (host code can't accidentally pull the indexer's tokio runtime or alloy provider into its own dep tree).

### Q-b Â· Communication channel host â†” indexer process (desktop subprocess flow)

For the desktop subprocess flow, the host and the indexer need to exchange:
- **Host â†’ indexer:** start request (vault_id; rpc_url; chain env; from_block; until_block; opts mirroring `SyncOptions`); progress poll; shutdown / cancellation; idle-timeout reset (keep-alive).
- **Indexer â†’ host:** progress events (per-chunk progress; revisions indexed; rejected counts); fetch-results (the actual `VerifiedRevisionEvent` batches the host then ingests into the `Vault`); error events; final `SyncReport`-shaped completion.

Four plausible mechanisms:

**Option A: Stdio (line-delimited JSON over stdin/stdout).** Host spawns `pangolin-indexer` via `std::process::Command` (or `tokio::process::Command`); pipes stdin/stdout; line-delimited JSON in both directions. **Pro:** no new external crate dep (already have `serde_json`); no port-allocation concern; no firewall surface; cross-platform (Windows + Linux + macOS) trivial; the indexer process can't talk to anyone except its parent (zero network surface for IPC). **Con:** can't easily multiplex (one stream both ways); large binary blobs (the `enc_payload` field can be megabytes) get base64'd + transit through the parent's pipe buffer.

**Option B: Unix domain socket / Windows named pipe.** Indexer binds; host connects. **Pro:** bidirectional + multiplexed cleanly. **Con:** OS-specific (uds on Unix, named-pipe on Windows); new IPC crate dep (`tokio::net::UnixStream` is gated on Unix; cross-platform crates exist â€” `interprocess`, etc., would be a new external dep).

**Option C: HTTP over localhost (reuse axum from 3.4).** Indexer binds a port (host-allocated random ephemeral port; communicated via stdout once); host connects. **Pro:** reuses the axum + tower + reqwest dep set 3.4 already brought in; existing test infrastructure transfers. **Con:** materially over-engineered for a co-process; port-allocation race; firewall prompts on macOS / Windows; localhost binding is a real network surface (some hosts have policies that disallow localhost binding from unsigned binaries).

**Option D: gRPC / tonic.** Out of scope for MVP-2 (new heavyweight dep set).

**Plan-gate recommends:** Option A (stdio + line-delimited JSON) for MVP-2. Simplest; cross-platform free; zero network surface; large enc_payload concern is mitigated by the indexer's job NOT being to ship raw payloads back to the host â€” the temp DB *is* the payload buffer, and the host streams events through small per-event messages. **Builder note:** the line-delimited-JSON framing should be a typed enum of messages (not free-form JSON) â€” `IndexerMessage::{Start{...}, Progress{...}, Events{...}, Error{...}, Complete{SyncReport}, Shutdown}` mirroring 3.4's `FunderError`/`TopUpResponse` typed enum discipline.

**Kelvin's call.**

### Q-d Â· Temp DB security in 4.2 vs 4.3 boundary

D-007 implementation note: "Random-path encrypted temp DB; explicit zero-fill before unlink; cleanup on crash via OS-level temp-file conventions." Four properties:
1. **Random temp path** (an attacker who knows the indexer is running cannot guess the temp file path)
2. **Encryption at rest** (an attacker who reads the temp file off disk gets ciphertext, not chain events)
3. **Explicit zero-fill before unlink** (an attacker who reads raw disk blocks after the file is unlinked sees zeros, not residual ciphertext)
4. **Cleanup on crash** (the temp file does not persist past process death, even on SIGKILL / panic / OOM)

Where does 4.2 stop and 4.3 start? Three plausible boundaries:

**Option A: 4.2 ships properties 1 + 4; 4.3 ships properties 2 + 3.** Temp file via `tempfile::NamedTempFile` (random path; Drop-handler unlink on normal exit; OS-temp-dir conventions for crash cleanup). SQLite over that file. **No encryption yet (placeholder).** **No zero-fill (placeholder).** 4.3 adds the ephemeral encryption key (per-run random 256-bit key; never persisted; lives in process memory only); SQLite cipher integration (sqlcipher OR a custom AEAD layer over a raw-file backed by direct mmap â€” builder + Kelvin's 4.3 call); zero-fill before unlink.

**Option B: 4.2 ships properties 1 + 4 + ENCRYPTION HOOK (a `TempDbCipher` trait that 4.2 implements as a passthrough no-op stub; 4.3 swaps in the real AEAD impl).** Same disk shape as Option A but with the trait surface present so 4.3 is a pure-plug-in (no callsite changes). **Pro:** the architectural-locking property that 4.2 â†’ 4.3 transitions are additive, not invasive. **Con:** more code in 4.2; the trait surface is provisional (4.3 may want to change it).

**Option C: 4.2 ships only the lifecycle + indexing logic; the temp file is a plain unencrypted SQLite file in the OS temp dir; 4.3 ships ALL of properties 1â€“3 (path randomization + encryption + zero-fill) in one pass.** **Pro:** 4.2 stays smaller. **Con:** 4.3 has to rework 4.2's file-handling layer.

**Plan-gate recommends:** Option B â€” 4.2 ships properties 1 + 4 + the encryption hook (a `TempDbCipher` trait with a no-op `IdentityCipher` impl that 4.3 swaps for a real `Aead` impl). Architectural-locking the 4.2â†’4.3 transition. Same posture 3.6 took for the privacy hooks: scaffolding in 3.6 + fail-loudly stub + real impl in MVP-3 / MVP-4. The trait surface in 4.2 should be deliberately minimal (a single `encrypt_block(&self, plaintext, aad) -> ciphertext` + symmetric `decrypt_block` shape, or even just a `&dyn Write` / `&dyn Read` wrapper over the SQLite file handle) so 4.3 can re-shape it without churning the 4.2 callsites.

**Kelvin's call.**

## Adversarial framing â€” L-section (load-bearing risks)

> Per Â§16, every plan-gate doc for a security-critical surface enumerates the load-bearing risks an adversary could leverage from each design choice. 4.2's surface is the temp DB + the indexer process + the hostâ†”indexer IPC.

### L-temp-file-leak

**What goes wrong:** a crashed indexer (panic / SIGKILL / OOM / power-loss) leaves a temp file on disk containing chain-event data (encrypted payload bytes + vault_id + per-event metadata). An attacker with filesystem read access (post-compromise, forensic recovery, lost laptop) recovers the file.

**Failure mode:** the attacker learns the user's `vault_id` (the L2 filter pins everything to one vault); the per-event `encPayload` ciphertext (already protected by the vault's master key + MVP-1 AEAD); the per-event metadata (sequence, parent_revision, device_id_address, timestamps). The `encPayload` is opaque to the attacker without the vault password; the metadata is a per-revision correlation surface.

**Defense in 4.2:**
- (a) `tempfile::NamedTempFile` random-path (L1) â€” the file path is unguessable.
- (b) `NamedTempFile`'s Drop impl unlinks on normal process exit (L11).
- (c) On panic with `panic = unwind` (the workspace default), the Drop also fires during stack unwinding.
- (d) On `SIGKILL` / `panic = abort` / OOM / power-loss, the file persists until the OS-level temp-dir cleaning sweep (Linux `/tmp` clean-up; Windows `%TEMP%` clean-up; macOS `/var/folders/.../T/` clean-up; all triggered by the OS at boot or on a periodic schedule).

**Defense in 4.3 (deferred):**
- (e) Ephemeral per-run encryption key (256-bit; random; never persisted; lives only in process memory) â€” so even a recovered temp file is ciphertext.
- (f) Explicit zero-fill before unlink â€” so the unlinked file's disk blocks are zeroed.

**Adversary leverage in 4.2 (without 4.3):** in the abnormal-exit path (the only path where the temp file persists past Drop), the attacker recovers plaintext chain-event data including ciphertext + metadata. Mitigated to "ciphertext + metadata" because the `encPayload` is encrypted by the vault's MVP-1 AEAD independently. Worth surfacing explicitly: 4.2 alone does NOT close this surface â€” 4.3 does.

### L-vault-id-disclosure

**What goes wrong:** the indexer queries the RPC with `topic1 = vault_id`. A malicious or curious RPC operator learns the user's `vault_id` (the same correlation surface 4.1 already has â€” D-006's Phase-2 Enhanced Privacy Mode is the architectural mitigation). 4.2 inherits this property unchanged.

**Defense:** none new in 4.2. Phase-2 Enhanced Privacy Mode (3.6 scaffolding) is the documented Phase-2 mitigation. 4.2 documents the surface in the threat model but does not solve it.

**Adversary leverage:** same as 4.1's surface; no net change.

### L-stdio-injection (Q-b dependent)

**What goes wrong:** if Q-b resolves to Option A (stdio + line-delimited JSON), a malicious party with stdin access to the indexer process could inject crafted JSON messages.

**Defense:**
- (a) The indexer is spawned BY the host; the host owns the indexer's stdin (it's the writeable end of the parent's pipe). No third party has access to the indexer's stdin without first compromising the host.
- (b) If the host is compromised, the indexer is downstream of the compromise; closing this attack surface on the indexer is moot (the attacker already controls the vault keys).
- (c) Defense-in-depth: parse the JSON strictly into a typed enum (`IndexerMessage::{Start, ...}`); reject unknown variants; bound the line length (per-line read cap, e.g., 1 MiB); reject malformed JSON immediately.

**Adversary leverage:** mitigated to "host-level compromise â€” already game over."

### L-idle-timeout-DoS

**What goes wrong:** a hostile actor (a buggy host; a malicious host; a network actor with intermediate-pipe access) keeps poking the indexer's keep-alive endpoint to prevent the idle-timeout cleanup from firing. The indexer runs indefinitely; the temp file persists indefinitely; the network surface to the RPC stays open indefinitely.

**Defense:**
- (a) Absolute max-runtime ceiling regardless of activity (e.g., 1 hour). When the absolute ceiling fires, the indexer exits + cleans up regardless of whether the host is actively pinging. The idle-timeout is the soft cap; the max-runtime is the hard cap.
- (b) The host's keep-alive surface is the only attacker-controllable signal; a keep-alive bounded at "every < idle_timeout / 2" cadence is acceptable; anything faster is rate-limited (defense-in-depth â€” not a primary defense).

**Plan-gate recommends:** ship both. Default soft idle-timeout = 5 min (Q-c Option C); default hard max-runtime = 1 hour. Both env-var-overridable for tests + hosts.

**Adversary leverage:** bounded to "1 hour of indexer running per attacker session" â€” bounded resource cost.

### L-spurious-spawn

**What goes wrong:** malicious code on the user's machine spawns `pangolin-indexer` directly (bypassing the host) and points it at the user's `vault_id` + the chain RPC.

**Defense:** there is no auth model in 4.2 (any local process can run the indexer). But: spawning the indexer doesn't give access to anything secret â€” it just queries public RPC data filtered by `vault_id`. The temp DB belongs to the malicious process, not the host. The host's vault is untouched.

**Adversary leverage:** essentially none. The "attacker spawns their own indexer" is equivalent to "attacker queries the public RPC directly with `topic1 = vault_id`" â€” no additional capability gained. The threat is L-vault-id-disclosure restated; the defense is the same (Phase-2 Enhanced Privacy Mode).

### L-host-indexer-mismatch (Q-b + L12)

**What goes wrong:** host and indexer disagree on the IPC schema version (`IndexerMessage` variant set drifts between host build and indexer binary build â€” a stale binary in PATH, a partial upgrade).

**Defense:**
- (a) Every `IndexerMessage` carries a `protocol_version: u16` field; the indexer's `Start` response includes the indexer's supported protocol version. Host validates on first response; mismatch â†’ host kills indexer + surfaces a "version mismatch â€” reinstall" error.
- (b) Both sides reject unknown variants strictly (no permissive JSON parsing).

**Adversary leverage:** mitigated to a confused-deputy class bug, not a security surface. Worth surfacing in the threat model row.

### L-temp-dir-tampering

**What goes wrong:** an attacker pre-creates a symlink at the path `tempfile::NamedTempFile` is about to create, pointing at a sensitive location (e.g., `/etc/passwd` on Linux; `C:\Windows\System32\config\SAM` on Windows â€” though the indexer doesn't run as admin). The indexer writes to the symlinked location, clobbering the sensitive file.

**Defense:**
- (a) `tempfile::NamedTempFile` uses `O_CREAT | O_EXCL | O_NOFOLLOW` (or platform equivalent) â€” pre-existing files at the chosen path cause `EEXIST` and the call retries with a different random suffix. The `O_NOFOLLOW` arm rejects symlinks.
- (b) The temp dir is the OS-recommended user-specific temp dir (`%LOCALAPPDATA%\Temp` on Windows; `$TMPDIR` or `/tmp` on Linux/macOS) which is owned by the user; an attacker who can write symlinks there has the user's local creds.

**Adversary leverage:** mitigated to "user-local attacker who can already write to user's temp dir" â€” already game over.

- **4.3 security hardening of the temp DB.** Ephemeral encryption key; SQLite AEAD layer (sqlcipher OR custom); explicit zero-fill before unlink. 4.2 ships the `TempDbCipher` hook (Q-d Option B); 4.3 implements the real cipher.

- **Sync UI / progress reporting in the host.** The indexer emits `Progress` messages (Q-b); the host's UI plumbing is a future CLI / FFI / shell concern.
- **Multi-vault batch indexing.** 4.2 indexes one vault per indexer session. Spawning multiple indexers for multiple vaults is a host concern (4.4 + future).
- **Cross-chain support.** Per master plan Â§0 cardinal principle 4: one chain per build. The indexer hardcodes `ChainEnv::BaseSepolia` resolution through the shared `pangolin-chain` deployment loader.
- **Persistent caching of indexed events** (across indexer sessions). D-007 forbids it. The temp DB is **per-session**; subsequent invocations create a fresh temp DB. Caching across sessions defeats the auto-delete promise.
- **Multi-region indexer / load-balancing.** D-007 forbids it; there is no persistent service.
- **FFI / UniFFI exposure.** Deferred to a CLI-V1 / FFI-batch follow-up.
- **CLI subcommand `pangolin sync --fast`.** Deferred to CLI-V1 batch + 4.4.
- **WebSocket-mode indexer.** 4.1's WS-fallback is deferred (alloy WS feature not in tree); 4.2 reuses 4.1's HTTP-polling backend via `fetch_and_verify_chunk`. WS-mode indexer is an MVP-3 concern paired with 4.1's WS feature flip.
- **Per-event progress streaming to host.** The IPC channel emits per-chunk progress (not per-event); the host pulls completed events from the temp DB once the indexer signals chunk completion. Per-event streaming is an optimisation deferred to a future issue.
