## Open questions for Kelvin (LOAD-BEARING — surface at TOP per §16)

### Q-a · Storage strategy — does the OS keystore enter Pangolin's threat model in 3.2 at all?

The master plan §4 row 3.2 says verbatim: *"Each device generates an Ethereum keypair on first run. **Stored in platform secure storage (Keychain / Keystore / OS keystore)**. The same keypair is used for revision signing AND gas payment."* — written **before** 2.1's R-a locked Path B (deterministic Ed25519 → secp256k1 derivation). After R-a, the secp256k1 wallet is a *pure function* of the 1.5 sealed Ed25519 `DeviceKey` seed; it has no independent secret to store. This makes the master-plan row's "OS keystore" wording an *architectural choice*, not a foregone conclusion. Three plausible models:

**Option A: Vault-sealed-only (no OS keystore).** The secp256k1 wallet is **never persisted** — it is re-derived on demand from the 1.5 sealed Ed25519 `DeviceKey` whenever the vault is `Active`. The wallet's *public* 20-byte address is cached in the `.pvf` (`devices.evm_address` column, additive migration) for fast read-only access ("show me my gas wallet address") when the vault is locked-but-previously-unlocked. The secret never crosses out of `pangolin-chain` (the `PrivateKeySigner` is built inside `EvmWallet`; the secret-bearing scalar lives in `k256::SecretKey` whose `Drop` zeroizes; the wallet lives in `ActiveState` alongside the `DeviceKey` and dies on every session-teardown path the 1.5 doc enumerates). **Single source of secrecy: the VDK-sealed Ed25519 seed.** Adds zero new at-rest surface. **Recommended (mild).** Matches the architectural posture 1.5 set + the 2.1 R-a + the P7 EvmWallet design (the existing `BaseSepoliaAdapter::new_with_device_key` already does exactly this — 3.2 is "wire the same derivation into `Vault` so other crates don't need a `DeviceKey` reference to get the wallet").

**Option B: OS keystore as a redundant cache (`keyring` crate).** The Ed25519 seed remains VDK-sealed in the `.pvf` (load-bearing source of secrecy), AND the derived secp256k1 scalar is *also* stored in the OS keystore (Keychain on macOS / Credential Manager on Windows / Secret Service / libsecret on Linux). On `Vault::unlock`, the platform code preferentially reads the OS keystore (fast path); if absent it derives from the seed and back-fills. **Trade-offs:** (i) two at-rest surfaces means twice the attack surface for the same secret; (ii) the OS keystore secret is *not* AAD-bound to `vault_id` so a transplanted .pvf + a transplanted keystore entry compose into a "load this vault as if it were the original device" attack the AAD-bound .pvf alone defeats; (iii) Linux CI runners (env-quirk-adjacent — Secret Service not running headless) need a mock backend; (iv) `keyring` crate's per-platform behaviour drift becomes a Pangolin support burden; (v) splits the "single secrecy doctrine" 1.5 cleanly set. **Plan-gate recommends against** unless Kelvin has a concrete UX reason (e.g. "share the same EVM identity across vaults" — explicitly out of scope per master plan §0 cardinal principle 4: layered independent authorities).

**Option C: OS keystore as the ONLY storage (no vault-sealed derivation).** Cut the 1.5 Ed25519 ↔ secp256k1 derivation entirely; generate an *independent* secp256k1 wallet on first unlock, store it in the OS keystore only. **Rejected by 2.1 R-a's locked text** — the architecture is "Ed25519 device-key seed → HKDF → secp256k1 scalar … one device → one EVM address" verbatim. Option C is the "two independent keypairs per device" path 2.1 R-a's Implementation impact (3) closed when it locked Path B. Listed for completeness; the plan-gate considers it a re-litigation of 2.1 R-a.

**Pick one.** Default unless told otherwise: **Option A (vault-sealed-only).** Codifies the post-R-a reality + adds zero new at-rest surface + zero new crate dep + zero new threat-model row.

### Q-b · Key materialisation timing — at unlock-time, or lazily on first chain-write?

The 1.5 doc says the in-memory `DeviceKey` is materialised inside `ActiveState` on every `Vault::unlock` and dropped on every session-teardown path. The secp256k1 wallet derived from it could land in one of two places:

**Option A: Eager — materialise the `EvmWallet` inside `ActiveState` at the moment 1.5 materialises the `DeviceKey`.** Cost: one HKDF-SHA256 expand + one k256 `SecretKey::from_slice` per unlock (~hundreds of microseconds). Benefit: every consumer (`pangolin-chain::base_sepolia`, future indexer in 4.2, future direct-submit in 3.3) reads the wallet from `ActiveState` rather than re-deriving; one in-memory zeroize-on-drop lifecycle to reason about. **Recommended (mild).** Matches the 1.5 + 1.3 + 1.4 doctrine: every `Active`-only resource (decrypted account cache, `:memory:` FTS5 search index, the `DeviceKey` itself) is materialised once at unlock and dropped together on teardown.

**Option B: Lazy — materialise on demand at the first call site that needs it.** Saves the derivation cost for sessions that never publish. Adds a `Vault::evm_wallet() -> Result<&EvmWallet>` that derives on first call and caches in a `Cell<Option<...>>`. Trade-off: cache invalidation across re-unlocks. The savings are negligible (the derivation is microseconds; an unlock already pays milliseconds for Argon2id).

**Pick one.** Default unless told otherwise: **Option A (eager).** Matches the rest of `ActiveState`.

### Q-c · FFI exposure — address only, address + signing handle, or nothing?

The 1.5 FFI surface exposes `DeviceInfo { id, label, registered_at, last_sync_at, capabilities, is_current, public_key }` — no chain identity. After 2.1 R-a, the secp256k1 wallet address IS the on-chain identity. Three options:

**Option A: Add `evm_address: Vec<u8>` (20 bytes) to `DeviceInfo`; no signing handle crosses FFI.** Hosts can render "your gas wallet: 0x...". Signing stays inside the Rust core (every future chain-write — 3.3 direct-submit, 4.2 indexer reads, 3.4 funder client — happens inside `pangolin-core` and never asks FFI for a signature). Same posture as 1.5's `public_key` (verifying-key bytes crossed for fingerprint render; no signing handle crossed). **Recommended.** Matches the "FFI exposes display data; signing is a core-internal verb" doctrine 1.5 set.

**Option B: Add `evm_address` + a `device_sign_revision(...) -> Vec<u8>` entry point.** Premature — the canonical-revision-signing entry point is MVP-2 issue 3.1 (signed-revision client format). 3.2 surfacing a generic "sign these bytes with the device wallet" entry point would invite host code that bypasses 3.1's domain-separated EIP-712 discipline. Rejected by 2.1 R-a's posture ("the wallet signs revisions; the discipline lives in the Rust core's signing module, not in the host").

**Option C: Nothing crosses FFI in 3.2.** The wallet is purely internal until 3.3 needs it. Defensible (3.2 is a "wire the lifecycle" issue; user-visible address rendering can wait). Trade-off: no way for a host UI to display "your gas wallet" without a follow-up FFI amendment.

**Pick one.** Default unless told otherwise: **Option A (address only).**

### Q-d · CLI surface — `pangolin-cli wallet show` in 3.2, or deferred?

The 1.5 CLI-V1-wiring follow-up (decided at 1.5 Q7 = ship a `device` subcommand later as one cohesive batch) hasn't landed. A `pangolin-cli wallet show` could land alongside 3.2, OR be folded into the same standing CLI-V1-wiring follow-up batch.

**Option A: Ship `pangolin-cli wallet show` in 3.2.** Renders "device id: 0x…, EVM address: 0x…, on chain: <chain-id from deployment file>". Mirrors `pangolin-cli authority list`'s read-only inspection pattern.

**Option B: Defer to the CLI-V1-wiring follow-up batch.** Same posture as 1.4/1.5/1.7/1.9/1.10/1.11 took (FFI-complete, CLI deferred). **Recommended.** Keeps 3.2 tight; the CLI verb has no MVP-2 dependency that needs it before 3.3 / 4.2.

### Q-e · Test injection — mock keystore backend, or pure-Rust derivation only?

If Q-a resolves to Option B or C (OS keystore is in scope), the `keyring` crate's per-platform calls cannot run in CI (Linux runners have no Secret Service; Windows runners may not have Credential Manager service running for the GH Actions user). Three options:

**Option A: A trait abstraction (`KeystoreBackend`) with a real `OsKeyringBackend` for production + an in-memory `MockKeystoreBackend` for tests.** Mirrors the existing `ChainAdapter` / `MockChainAdapter` pattern (P7 invariant — production downstream consumers MUST NOT enable the `test-utilities` feature that exposes the mock). Cost: a new trait + two implementations + the feature-flag discipline P7 documents.

**Option B: A `cfg(test)` short-circuit that bypasses the OS keystore entirely.** Cheaper but harder to test the real-keystore path adversarially (the only assurance is "manual smoke on a real Mac / Windows / Linux box before merge"). Same posture `BaseSepoliaAdapter::new_with_keystore`'s tests take today (no automated `keyring` exercise).

**Option C: Pure-Rust derivation only (if Q-a = Option A).** No keystore = no mock backend = no new test infrastructure. **Recommended only if Q-a = Option A.**

**Pick one.** Default unless told otherwise: **Option C** (Q-a defaults to Option A, so the question evaporates).

## Adversarial framing — L-section (load-bearing risks)

> Per §16, every plan-gate doc for a security-critical issue surfaces the
> load-bearing risks an adversary could leverage from each design choice.

### L-key-material-location

**Where does the secp256k1 private scalar live at rest? In memory? In the OS keystore?**

- **At rest, under L1/L4:** nowhere except as a deterministic function of the VDK-sealed Ed25519 seed. The secp256k1 scalar bytes never touch disk. The .pvf carries only (i) the AEAD-sealed Ed25519 seed (1.5; AAD-bound to `vault_id || device_id`) and (ii) the *public* 20-byte EVM address (new in 3.2; non-secret per D-006's known mitigation).
- **In memory, during an `Active` session:** the scalar lives inside a `k256::ecdsa::SigningKey` wrapped by `alloy::signers::local::PrivateKeySigner`, wrapped by `pangolin_chain::evm::EvmWallet`. The `EvmWallet` struct holds the signer by value (not by reference); `k256::SecretKey`'s `Drop` zeroizes; the `PrivateKeySigner`'s `Drop` chains through; `EvmWallet`'s `Drop` is the trivial drop-the-field path. **Verification gate (success criterion):** add a `no_evm_secret_after_drop` proptest that allocates an `EvmWallet`, takes the raw scalar bytes via `signer().to_bytes()`, drops the wallet, and asserts (via a deliberately re-used `Vec<u8>` allocation path that the previous `signer().to_bytes()` returned) the bytes are zero or scrambled. **Limit:** Rust's `Drop` semantics do not guarantee in-place wipe of fields a compiler has elided into registers; the proptest's strength is "the field on the heap is zeroed", not "no bit of the scalar ever survives in a CPU register".
- **Adversary leverage:** an attacker who gains read access to the running process's memory during an `Active` session can extract the secp256k1 scalar — same threat as the 1.5 in-memory `DeviceKey` (1.5 already documents this as out-of-scope per `THREAT_MODEL.md` "Out of scope: fully compromised operating systems / Kernel-level malware"). 3.2 inherits that out-of-scope boundary; it does NOT widen it.

### L-keystore-availability

**What happens if the OS keystore is unavailable?** This risk **only exists if Q-a resolves to Option B or C**.

Per-platform failure modes:
- **macOS Keychain locked:** Keychain prompts the user to unlock; in a CLI / headless context this may fail synchronously with a "Keychain locked" error. `keyring` crate surfaces this as a `keyring::Error::PlatformFailure`. Fallback: re-derive from the 1.5 sealed seed (same as Option A).
- **Windows Credential Manager service not running:** rare on consumer Windows; possible on Server SKUs / minimal CI runner images. `keyring` surfaces this as `keyring::Error::PlatformFailure`. Fallback: same as above.
- **Linux Secret Service / libsecret not running:** common on headless servers, CI runners (env-quirk-adjacent), Docker containers, server distros without a desktop session. **This is the most likely real-world failure path.** `keyring` returns `keyring::Error::NoStorageAccess`. Fallback: same as above.

**Plan-gate recommendation if Q-a = Option B or C:** ALL `keyring` operations must be wrapped in a "fall back to vault-sealed derivation on error" path, so the OS keystore is a *cache*, never an *authority*. The vault-sealed Ed25519 seed is the load-bearing source of truth.

### L-key-derivation-determinism

**Is the secp256k1 keypair re-derivable on demand from the 1.5 sealed DeviceKey seed?**

**Yes, by construction.** The existing `evm.rs` module asserts this via three tests:
- `derive_is_deterministic` — same `DeviceKey` instance → same address (twice).
- `different_seeds_produce_different_addresses` — distinct devices → distinct addresses.
- `structural_property_distinct_seeds_distinct_signatures` — at the underlying `SigningKey` layer; pinned via `from_seed([0x11; 32])` vs `from_seed([0x22; 32])`.

**Implication for backup/recovery/multi-device:** Backup of the vault file alone is sufficient to recover the wallet (no separate keystore export needed). Multi-device sync of the same vault means *each device* derives its *own* secp256k1 wallet from its *own* 1.5 Ed25519 seed — i.e. **two devices for the same user have two different EVM addresses**. This matches the 2.1 R-a self-bootstrap model (each device's address is registered on first publish for that `vaultId`). It does NOT match a "shared on-chain identity across all of a user's devices" model — which is intentionally **out of scope** (would require a separate authority-rotation step every device pairing; MVP-3 territory).

### L-zeroize

- The Ed25519 seed: zeroized inside `DeviceKey::Drop` per 1.5 / P1.
- The HKDF intermediate buffer (`okm`): explicitly zeroized via `zeroize_array` inside `derive_evm_wallet`'s loop (already present in `evm.rs`).
- The secp256k1 scalar: zeroized via `k256::SecretKey`'s `Drop` (verified by RustCrypto's own zeroize discipline; documented in `evm.rs`).
- The `EvmWallet` struct itself: trivial drop; the only field is the `PrivateKeySigner` which transitively owns the `SecretKey`. **No `Clone` on `EvmWallet`** by design (the inner `PrivateKeySigner` derives `Clone`, but `EvmWallet` deliberately does not expose it). 3.2 inherits this discipline.

### L-cross-platform (conditional on Q-a)

If Q-a = Option B or C, per-platform behaviour drift:
- **Windows:** Credential Manager has a 2.5KB-per-credential limit. A 32-byte secp256k1 scalar fits trivially. Per-user storage; SYSTEM-account scenarios (e.g. running as a service) need a DPAPI fallback.
- **macOS:** Keychain entries are visible to other apps signed by the same developer ID unless explicitly scoped. ACL needs to be set tightly.
- **Linux:** Secret Service via libsecret is GNOME-default; KDE wallet has a separate path. `keyring` crate handles both; headless servers (no Secret Service running) need the mock backend per Q-e.

### L-relationship-to-1.5-DeviceKey

**The 1.5 Ed25519 DeviceKey continues to exist after 3.2 — is it still used?**

**Yes — and this is load-bearing.** Per 2.1 R-d (resolved decision): *"The Rust signing-domain prefix bumps from `pangolin-chain-signed-revision-v0` to `pangolin-chain-signed-revision-v1` in `crates/pangolin-chain/src/signing.rs::SIGNED_REVISION_DOMAIN`"* + R-a (Path B) means **the canonical hash is the same; the primitive switches from Ed25519 to secp256k1**. But the existing `pangolin-chain::signing::build_signed_revision` is Ed25519-based; it does not go away in 3.2. The 2.1 doc explicitly defers the v0→v1 client-side re-signing migration to **MVP-2 issue 3.1**: *"the actual v0→v1 client-side migration (re-signing PoC `.pvf` `SignedRevision` records under v1's primitive, if Q-e resolves to (b) or (c)) — MVP-2 issue 3.1"*. So:

- **3.2 wires the secp256k1 wallet into `Vault` lifecycle** (this issue).
- **3.1 builds the secp256k1 signing path** (`secp256k1_signing.rs` per 2.1 R-a's Implementation impact (3)) that consumes the wallet.
- **Until 3.1 lands**, the wallet is materialised but signs nothing (same posture as 1.5: generated + stored, signs nothing in MVP-1; signing landed in 2.1 R-a — 3.2 is the in-between "materialise the wallet" step).

The Ed25519 `DeviceKey` continues to be the *vault-local* device-identity key (it's the source of the `device_id` per 1.5; it remains the AEAD-AAD-bound identity for the sealed device-key blob; it remains what derives the wallet). The secp256k1 wallet is the *chain-side* identity. **One device key, two roles, one HKDF expansion between them.**

## Schema migration

- **`devices` table additive column:** `evm_address BLOB` (nullable; 20 bytes when present; NULL for legacy P2/1.5 rows that pre-date 3.2).

- **§18.7 schema-version slot:** the existing `devices.schema_version` slot (1.5) covers the new column; no new version bump.
- **Register-on-unlock backfill:** when `Vault::unlock` registers a new device (1.5 path), it now ALSO derives the EVM address and stamps it. WHERE device_id = ?` inside the same transaction). Idempotent on subsequent unlocks (the column is non-NULL; no write).

## Threat-model touch points

3.2 adds a new entry to `THREAT_MODEL.md`'s per-component table: **"Device EVM wallet (secp256k1; derived from 1.5 Ed25519 device key)"** — DOCUMENTED (issue 3.2). The per-surface enumeration (paraphrased; finalize during builder pass):
1. **Adversary steals the cached `devices.evm_address` from a stolen .pvf.** Non-secret; the address is on-chain-observable per D-006's known mitigation. No leverage.
2. **Adversary recovers the secp256k1 scalar from a leaked memory dump of an `Active` session.** Out of scope per existing "fully compromised OS / kernel-level malware" boundary; same posture as 1.5's in-memory `DeviceKey`. 3.2 does not widen this.
3. **Adversary tries to forge a revision under a stolen address.** Forging a signature without the scalar requires breaking secp256k1 ECDSA; out of scope per `THREAT_MODEL.md` "Cryptographic failures: if XChaCha20-Poly1305 or Ed25519 is broken, Pangolin breaks. We assume modern primitives are sound." — extend to secp256k1 (2.1 R-a already implicitly extended this).
4. **Adversary correlates EVM address with user identity.** D-006's known mitigation (Phase-2 Enhanced Privacy Mode); 3.2 does not change the posture.
5. **(Conditional on Q-a = Option B/C) Adversary transplants an OS-keystore entry between machines.** Mitigated by binding `vault_id` into a wrapper around the keystore secret OR by deriving from the always-AAD-bound 1.5 seed (which has `vault_id` in its AAD already). The plan-gate recommendation under Q-a Option A makes this risk N/A by construction.

If Q-a = Option B or C, the `THREAT_MODEL.md` entry MUST also include:
6. **OS keystore unavailable; user cannot sign revisions.** Mitigated by the fall-back-to-seed-derivation path per L-keystore-availability.

Invariant additions: none new. The existing invariants #1, #5, #11 cover 3.2's surface.

## Out of scope (explicit)

- **Multi-device EVM-identity sharing** — two devices for the same user have two different EVM addresses; this is intentional (matches 2.1 R-b self-bootstrap + Whitepaper §F per-device-keys model). MVP-3 social recovery may revisit ("recovery rotates authority; new device gets a new EVM wallet").
- **EVM-wallet recovery from a stolen .pvf alone** — the .pvf's AEAD-sealed seed requires the vault password to decrypt; without the password the wallet cannot be recovered. This is by design (Whitepaper §5 "VDK protects everything"). A separate "EVM wallet backup" feature is NOT in MVP-2 scope.
- **Hardware-attested key storage** (Secure Enclave / TPM / Android Keystore hardware-backed entries) — MVP-3/4 territory. 3.2's derivation is software-only.
- **Real on-chain revision signing** — that's MVP-2 issue 3.1 (signed-revision client format under v1 per 2.1 R-a / R-d).
- **Direct-submit chain transport** — MVP-2 issue 3.3.
- **Funder client / payment-driven top-up** — MVP-2 issues 3.4 / 3.5.
- **Privacy-mode wallet rotation** — MVP-2 issue 3.6 (scaffolding-only; full implementation later).
- **Cross-vault EVM-address sharing** — orthogonal to 3.2; if a user has two vaults on one device, each vault's device gets its own EVM address (one device-key-seed per vault → one EVM wallet per vault). Documented as the intended posture.
