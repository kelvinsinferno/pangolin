# Issue P11A: pangolin-cli account subcommands

> **Status:** PLAN-gate. Per Kelvin's autonomy directive, dispatching the
> build directly upon §16-Plan-Gate approval.
> **Security-critical:** YES per §16.3. P11A surfaces the
> credential-bearing primitives (`add_account`, `update_account`,
> `delete_account`, reveal/export) at the user-facing CLI boundary.
> Every previous PoC issue (P3 onwards) treated those primitives as
> library-only callable; P11A turns the binary into the place where a
> non-author developer types `pangolin-cli account add` and produces a
> credential row in their vault file. A flaw in this surface either
> leaks plaintext to disk (process listing, shell history, log files —
> Cardinal Principle 2 violation), accepts a credential the user
> didn't actually intend to write (UX-load-bearing for the discipline
> Cardinal Principle 4 builds on), or strips the presence-escalation
> gate that P4 spent its entire issue budget locking down. P11A
> introduces no new cryptographic primitive and no new chain-side
> code; the entire issue is a clap-derive surface plus a presence-
> proof orchestration shim, but the audit surface is the
> credential-input + secret-output discipline.

## Spec reference

- **Master plan §3.7 EPIC: PoC scope.** P3 shipped the library API
  (`Vault::add_account`, `Vault::update_account`,
  `Vault::delete_account`, `Vault::get_account`, `Vault::list_accounts`,
  `Vault::reveal_password`, `Vault::reveal_notes`,
  `Vault::reveal_totp_secret`, `Vault::export_payload`); P8 shipped the
  CLI binary (`status`, `publish`, `pull`); P9 added `resolve`. The CLI
  is missing the verbs that turn the library API into a user-facing
  surface — P11A closes that gap. The master plan does not name P11A
  explicitly; it is a **scope-promotion of P11's "non-author dev
  reproduces all three scenarios from `E2E_TESTS.md` alone"** exit
  criterion. Without `account add` / `account list` / `account show`
  / `account update` / `account delete`, the reproducer's manual
  steps for E2E-003/004/005 require the non-author dev to either
  (a) write Rust against the library API or (b) run pre-author-
  prepared fixture vault files. P11A removes both crutches.
- **Master plan §3.5 forbidden user-facing terms** ("gas",
  "blockchain", "transaction", "decentralized storage", "hashes",
  "revisions"). P11A's clap help text MUST avoid these. The
  subcommand surface uses "credential" / "account" / "password" /
  "TOTP" / "vault" / "delete" / "reveal" — all consistent with the
  password-manager domain, none on the forbidden list. Internal
  doc-comments inside `commands/account.rs` may use the Pangolin-
  internal terms (revision, tombstone, AEAD) freely; the audit gate
  is `--help` output and `eprintln!`/`println!` strings.
- **DECISIONS.md:**
  - **D-001** — Rust core + thin shells. P11A is a *thin-shell* feature
    inside the existing Rust core repo. The shell layer (clap +
    rpassword + presence-proof construction) is mechanical; the
    security layer is the existing library API. P11A introduces zero
    new primitives.
  - **D-006** — No relay; same-key signer + gas payer. P11A's
    `account add/update/delete` produce dirty markers that flow
    through the existing `publish_all` orchestrator; the chain-side
    publish discipline (P8 §A7's PoC two-key model) is unchanged.
  - **D-013** — repository at `C:\Users\kelvi\Projects\pangolin`. P11A
    introduces files under `tools/pangolin-cli/src/commands/account.rs`
    and (optionally) `tools/pangolin-cli/tests/account_cli.rs`.
- **Cardinal Principle 1 ("Rust core is the only security-critical
  component; extension asks, core decides").** P11A's binary is a
  **thin asker.** Every credential bytes -> vault transit goes through
  `Vault::add_account` / `Vault::update_account` / `Vault::delete_account`;
  the CLI does not validate, normalize, or transform the credential
  data beyond clap-arg-parsing. The library API is the security
  boundary.
- **Cardinal Principle 2 ("No plaintext on disk; no plaintext leaves
  the device").** P11A's password/notes/TOTP-secret input paths MUST
  read from `rpassword::prompt_password` (terminal, no echo) or stdin
  (machine-readable, opt-in via `--password-stdin`). P11A's reveal
  paths print plaintext to stdout (CLI consistency); shell history
  capture is the user's risk to manage, but the CLI itself does not
  write plaintext to any file or environment variable.
- **Cardinal Principle 4 ("Append-only state, never silent merge").**
  P11A's `account delete` writes a tombstone revision via
  `Vault::delete_account` — already P10-1's three-field tombstone
  payload (`{deleted, account_id, tombstoned_at_ms}`); P11A reuses
  the discipline unchanged. `account add` already refuses to
  resurrect a tombstoned id (P10-3's A4 anti-resurrection); the CLI
  forwards the resulting `StoreError::Internal` cleanly.
- **Whitepaper §6 (account model) / §7 (revision lineage).** Account
  identity is opaque + per-row; the linear `genesis → edit → edit →
  tombstone` model is structurally final. P11A's CLI surface
  preserves the model — `account add` always generates a fresh
  account_id; `account update` always writes a new revision linked
  to the current head; `account delete` always writes a tombstone.

## What P0..P10 already shipped (the load-bearing surface for P11A)

P11A introduces no new cryptographic primitive, no chain-side code,
no schema column. It composes existing pieces:

- **P3 — `Vault::add_account(snapshot)`** (vault.rs L762–L848). Takes
  a fully-built `AccountSnapshot`, derives a fresh `account_id` via
  `derive_fresh_account_id` (P10-3 anti-resurrection, ADD_ACCOUNT_RETRY_BUDGET=4),
  seals the snapshot via `seal_snapshot`, writes the genesis
  revision + dirty marker in one transaction, returns the new
  `AccountId`. P11A wraps this with clap arg parsing + interactive
  prompts; the `AccountSnapshot` is constructed inside
  `account add::run` from arg values + prompted bytes and dropped
  (zeroized) immediately on hand-off to the vault.
- **P3 / P8 — `Vault::update_account(id, snapshot)`** (vault.rs L893–L987).
  Refuses on frozen account (P8 CRIT-1 freeze guard) or tombstoned
  account; reads the head; seals a new revision linked to the head
  as parent; writes the new revision + advances the head pointer +
  marks dirty in one transaction. P11A wraps with clap + reads the
  current snapshot via `Vault::get_account` + applies the user's
  partial-field updates + builds the new `AccountSnapshot` + calls
  `update_account`. **The "reads current and applies updates"
  pattern requires a presence-gated read of the existing password /
  notes / TOTP secret IF the user did not respecify those fields**
  — that is the load-bearing UX/security observation in §A6 below.
- **P3 / P10 — `Vault::delete_account(id)`** (vault.rs L992–L1089).
  Refuses on frozen / already-tombstoned. Builds a P10-1 tombstone
  payload (`{deleted=true, account_id, tombstoned_at_ms}`) and seals
  it; writes the tombstone revision + flips `tombstoned = 1` +
  marks dirty in one transaction. P11A wraps with clap + a
  confirmation prompt. The library does the heavy lifting.
- **P3 / P8 — `Vault::get_account(id)`** (vault.rs L1116–L1128) +
  `Vault::list_accounts()` (vault.rs L1162–L1174). Both filter
  frozen + tombstoned accounts implicitly (P8 CRIT-1 / P3
  tombstone guards). P11A's `account list` calls `list_accounts`;
  `account show` calls `get_account`. Frozen + tombstoned accounts
  are invisible to these read paths by design — `account list`
  surfaces them via the existing `list_frozen_accounts` /
  `list_tombstoned_accounts` accessors which P11A reuses for the
  `--include-frozen` / `--include-tombstoned` flags (§A11).
- **P4 / P3 — `Vault::reveal_password(id, presence)` /
  `reveal_notes` / `reveal_totp_secret`** (vault.rs L2097–L2151).
  Each requires an active session AND a fresh
  `PressYPresenceProof`; consumes the proof via
  `presence.verify()`; clones the requested `SecretBytes` field out
  of the in-memory cache and returns. P11A's `account show
  --reveal-password` (and the two sibling flags) constructs a
  `PressYPresenceProof::confirmed()` after a `"presence required:
  type 'y' and press enter:"` interactive prompt, then calls the
  reveal API. The presence-prompt UX is the §A7 architectural
  decision below.
- **P3 / P9 — `Vault::export_payload(id, presence)`** (vault.rs L2310–L2359).
  Same proof discipline as reveal. P11A's `account show
  --export-sealed` (open question §8 — whether to ship; recommend
  defer to MVP-1 per "explicit out of scope" §7). Not shipped in
  P11A.
- **P8 — `tools/pangolin-cli/src/vault_open.rs` + `keystore.rs` +
  `cli.rs` + `commands/` modules.** P11A mirrors the existing
  patterns: `open_and_unlock` for the vault open + unlock; clap
  `#[derive(Parser, Subcommand)]` for the subcommand enum; a new
  `commands/account.rs` module with submodule per verb.
  `pangolin-cli account ...` does NOT need the keystore / chain
  adapter (no chain calls — these are local-only ops; the
  publish-on-chain happens later via `pangolin-cli publish`).
- **P8 fix MED-3 — `canonicalize_vault_path`.** P11A uses this
  unchanged so the absolute path appears in any error message.
- **P10-3 — `Vault::list_tombstoned_accounts()`** (vault.rs L1230–L1250).
  P11A's `account list --include-tombstoned` calls this directly.

P11A introduces **one new module** (`tools/pangolin-cli/src/commands/account.rs`,
with a small submodule per verb) and **one new clap variant**
(`Command::Account(AccountArgs)`) on the existing `Cli::command`
enum. Five new sub-verbs are exposed; zero new exports, zero new
public types in `pangolin-store`, zero new `Cargo.toml`
dependencies.

## Goal

Ship five new `pangolin-cli account ...` subcommands — `add`,
`list`, `show`, `update`, `delete` — that expose P3-era's library
account-management API at the user-facing CLI boundary, preserving
P4's presence-escalation discipline for credential reveals, P8's
freeze-guard discipline for frozen accounts, and P10's anti-
resurrection / tombstone-payload discipline for deletions. Closes
the structural gap "Pangolin is a password manager whose CLI cannot
manage passwords" so P11's reproducer guide can drive a non-author
developer through E2E-003/004/005 without bespoke fixture
scaffolding, and so MVP-1 picks up a complete CLI surface rather
than a half-finished one. Introduces no new cryptographic
primitive, no new chain-side code, no new vault-schema column, no
new public library API.

## Architectural decisions (locked in this plan)

These decisions are committed to as part of plan-gate approval. Any
deviation during BUILD requires returning to PLAN per §16.4.

### A1. Subcommand hierarchy — nested `account <verb>`, not flat `add-account`

**Decision:** The five verbs live under a nested `pangolin-cli account
<verb>` namespace, not as flat top-level subcommands
(`pangolin-cli add-account`, etc.). The existing flat verbs (`status`,
`publish`, `pull`, `resolve`) stay flat — they're vault-level / chain-
level orchestrators and don't share a noun.

**Rationale:**

1. **Convention alignment.** `git branch <verb>`, `gh repo <verb>`,
   `kubectl get <noun>` all use noun-then-verb hierarchy for noun-
   scoped operations. Pangolin's password-manager domain is noun-
   heavy ("the user works with accounts"); the nested form reads
   naturally.

2. **Help-text scalability.** `pangolin-cli account --help` shows
   the five verbs in one place. With flat naming
   (`add-account`/`list-accounts`/...), `pangolin-cli --help` mixes
   five account-shaped verbs with the four orchestrator verbs;
   discovery suffers.

3. **Future expansion.** MVP-1 may add `account export`,
   `account import`, `account totp <id>`, `account search <query>`,
   `account history <id>`. The nested namespace absorbs these
   additions cleanly. Flat form would force renames as the noun
   surface grows.

**Alternative considered:** Flat `pangolin-cli add` /
`pangolin-cli list` / etc. **Rejected** because `list` collides
semantically with what should plausibly become a `list-vaults` or
`list-devices` verb in MVP-1+; reserving the unqualified `list`
overcommits the top-level namespace.

**Alternative considered:** Both forms via clap aliases (`account add`
AND `add-account` work). **Rejected** — every alias is documentation
debt + a discoverability footgun. Pick one.

### A2. Single-file `commands/account.rs` vs. per-verb directory

**Decision:** Ship a single `commands/account.rs` with a `run`
dispatch + per-verb `run_<verb>` private helpers. Split into a
`commands/account/{add,list,show,update,delete}.rs` directory if
the single file grows past ~600 lines during BUILD. The split
threshold is a soft signal, not a contract.

**Rationale:** The existing `commands/{publish,pull,status,resolve}.rs`
files are each ~150–400 lines. P11A's five verbs at ~80–150 lines
each total ~500–700 lines; close to the natural file-split
threshold. Single-file is fine for the initial commit; a second
pass at SIGNOFF can split if comprehension suffers.

**Alternative considered:** Per-verb directory from the start.
**Rejected** because the per-verb logic is small and a directory
upfront is structural overhead for limited semantic gain.

### A3. Password input — interactive prompt + double-confirm; `--password-stdin` for CI; NO `--password <flag>`

**Decision:** Password input on `account add` follows three paths in
strict precedence:

1. `--generate-password` (auto-generate; print to stderr inside a
   "save this password now" warning block; never persist to a file).
2. `--password-stdin` (read first line of stdin; trim trailing newline
   only; NO confirmation — CI/script use case).
3. (default) Interactive prompt via `rpassword::prompt_password("Password: ")`
   followed by a confirmation prompt (`"Confirm password: "`); both
   reads return `Zeroizing<String>`-equivalent buffers; the
   comparison runs in **non-constant-time** because the two strings
   are user-supplied and equal (the timing only leaks "user typed
   the same string twice" not anything about a secret); on mismatch,
   re-prompt up to 2 retries then abort with `bail!("password mismatch")`.

There is **NO** `--password <flag>` form. The flag form is a process-
listing leak (`ps aux` shows the password in argv) and Pangolin
refuses to ship that footgun even for "convenience."

`account update`'s password change uses the same pattern (with the
explicit `--password-prompt` flag to disambiguate from the no-change
case — §3 row 4 above).

**Rationale:**

1. **`ps`/`/proc` leak is the canonical flag-form bug.** Every
   modern password-tool (`pass`, `1password-cli`, `bw`, `op`)
   refuses the flag form for this reason. Pangolin matches.

2. **`--password-stdin` is the right CI shape.** A script can
   `echo "pwd" | pangolin-cli account add --password-stdin
   --name=foo`; the password is in the script (under the user's
   own filesystem permissions) but not in argv or env. Stdin is
   redirected by the user, not parsed by clap.

3. **`--generate-password` is the safest default for non-CI users.**
   The user does not type a password they'll forget; the binary
   prints the generated password to stderr in a clearly-flagged
   block; the user copies it to their preferred password store
   (which, post-PoC, is Pangolin itself). §A12 details the generator.

4. **Confirmation re-prompt loop is bounded at 2 retries.** Three
   total typing attempts; otherwise the user is invited to re-run.
   Unbounded retry is a footgun — if the user can't type the same
   password twice in three attempts, something is wrong (caps lock?
   keyboard layout?) and an early exit is more helpful than 47
   prompts.

**Alternative considered:** `--password <flag>` with a deprecation
warning printed to stderr ("warning: password on argv is logged by
ps"). **Rejected** because it ships the footgun for the user who
ignores the warning; the warning is inconsistent enforcement.

**Alternative considered:** Read password from a `--password-file
<path>` argument. **Rejected** because the file then sits on disk
(in the user's CI filesystem); plain stdin redirection from a file
is the same outcome via the standard mechanism, with clearer
auditability of where the password came from.

### A4. TOTP input — same as password (interactive prompt; --stdin variant; no flag)

**Decision:** TOTP secret input on `account add` and `account update`
follows the password pattern: interactive prompt by default
(`rpassword::prompt_password("TOTP secret (base32): ")`),
`--totp-stdin` for CI. NO `--totp-secret <flag>`.

**Rationale:** The TOTP shared secret is structurally equivalent to
a password (a long-lived credential whose disclosure compromises
2FA). Same threat model, same input discipline. The fact that the
expected format is base32-encoded doesn't change the input path —
the library does base32 decoding inside `add_account` (or rejects
malformed input cleanly).

**Alternative considered:** Allow `--totp-secret <flag>` because
TOTP secrets are conventionally short (32 chars) and "less secret"
than passwords. **Rejected** — they are the same security tier per
P4's reveal hierarchy; differentiating the input path would require
a coherent rationale that doesn't exist.

### A5. Notes input — flag-form `--notes <str>` is permitted; `--notes-stdin` is preferred

**Decision:** `--notes <str>` flag form is acceptable on `account add`
and `account update`; `--notes-stdin` is the recommended path for
non-trivial notes (multi-line; or notes that contain shell-special
characters; or simply per habit). Document both in `--help`; the
default eprintln summary should prefer the stdin variant for users
running interactively.

**Rationale:**

1. **Notes are a lower-tier secret per P4's reveal hierarchy** —
   they are presence-gated for reveal (same as password / TOTP)
   but they are not load-bearing for account access. A user who
   accepts the shell-history risk of `--notes "auth_question:
   first_pet=Mr_Whiskers"` has an informed risk surface.

2. **Multi-line notes need stdin.** Shell argument quoting
   discipline for newlines is fragile; piping from a file or
   heredoc is robust.

3. **Consistency with the `--name` / `--username` / `--url` flags.**
   Those are non-secret + flag-form; the notes flag's existence
   matches the broader UX consistency. The presence-gated reveal
   discipline is on the **read** path (`account show
   --reveal-notes`), not the write path.

**Alternative considered:** Match passwords / TOTP exactly (no
`--notes <str>` flag). **Rejected** — see §1 + §3 above.

### A6. Partial-update presence escalation — three reveal calls per `update`

**Decision:** `account update` is **always** presence-escalation-required,
even if the user is only changing a non-secret field (e.g.,
`--name`). Implementation: at the top of `run_update`, prompt for
presence ONCE (§A7's prompt UX); construct three internal
`PressYPresenceProof::confirmed()` instances; call
`Vault::reveal_password(id, &p1)`, `Vault::reveal_notes(id, &p2)`,
`Vault::reveal_totp_secret(id, &p3)`; combine the revealed bytes
with the user's specified-field updates into a new
`AccountSnapshot`; call `Vault::update_account(id, snapshot)`.

**Rationale:**

1. **The library API requires it.** `AccountSnapshot::new` takes
   six required `SecretBytes` fields; there is no "preserve
   previous" sentinel value. To `update_account`, the CLI must
   build a complete snapshot, which means reading the existing
   secret-field bytes for any field the user did not respecify.

2. **The reveal calls are the only way to read those bytes** under
   P4's discipline. A future library API might add a "merge_partial"
   primitive that does the read internally without exposing
   plaintext to the CLI; under PoC, the CLI is on the read path.

3. **The three-reveal pattern is honest about presence-cost.** A
   user changing `--name` should know they are passing a presence
   gate, because the implementation reads (and surfaces, even
   transiently) every secret field of that account. Hiding this
   under a "no presence required if you only change a non-secret"
   path would be lying about the operational cost.

4. **MVP-1 fix.** A future library API (`Vault::update_partial(id,
   &PartialAccountSnapshot)` where the partial struct has
   `Option<SecretBytes>` for every field) lets the CLI skip the
   reveal calls for unspecified fields. P11A explicitly does NOT
   ship this — it would be new public surface in `pangolin-store`
   that needs a separate threat-model row + audit pass. Document
   as a known UX overhead in the `--help` text:
   `"update is presence-gated even for non-secret-field changes
   (PoC limitation; MVP-1 will offer a partial-update path)"`.

**Subtle correctness concern acknowledged:** the multi-reveal pattern
materializes the password / notes / TOTP plaintext in CLI-process
memory transiently (in stack-allocated `SecretBytes`). The
zeroize-on-drop discipline preserves the at-rest property — the
plaintext is gone after `update_account` returns. But a memory-
dump attacker who is in the address space at the moment of
`update_account`'s seal call sees the plaintext. This is the same
exposure as the existing reveal paths (`Vault::reveal_password`
returns plaintext to the caller, who decides what to do with it);
P11A inherits the threat surface unchanged.

**Alternative considered:** Refuse `update` of secret fields entirely
(force the user to use `delete` + `add` to change a password).
**Rejected** — the library API supports update-of-secret-fields
cleanly; refusing it at the CLI is an arbitrary restriction.

**Alternative considered:** Skip the reveal calls when the user
specifies every secret field on the command line (`--password`-stdin,
`--notes-stdin`, `--totp-stdin`). **Acceptable as a future
optimization but explicitly OUT OF SCOPE for P11A** because the
"every secret field is specified" branch adds a code path to audit
+ test, and the simplifying invariant ("update always presence-
escalates") is more valuable than the marginal UX gain. Documented
in §7 (out of scope).

### A7. Presence prompt UX — one user gesture, multiple internal proofs

**Decision:** When `account show --reveal-password` (or any combination
of `--reveal-{password,notes,totp-secret}`) or `account update` runs
under an Active session that needs a presence proof, the CLI
prompts ONCE: `"presence required: type 'y' and press enter:"`. On
correct input, the CLI constructs N fresh `PressYPresenceProof::
confirmed()` instances (one per reveal call needed) and threads
each into a separate library-API call. Each proof passes
`PressYPresenceProof::verify()`'s freshness window because each was
constructed within the same prompt's response.

**Rationale:**

1. **`PressYPresenceProof::verify()` is single-use.** Re-using one
   proof across two reveal calls fails the second call with
   `AuthError::PresenceAlreadyConsumed`. The CLI must construct
   one proof per call.

2. **`PressYPresenceProof::confirmed()` stamps `created_at = now`.**
   The freshness window (`PRESENCE_FRESHNESS`) is wall-clock time
   from construction; constructing N instances in a tight CLI
   loop after one user gesture leaves all N within the window.
   The single user gesture is the *intent* signal; the multiple
   internal proofs are *implementation*.

3. **MVP-1 hardware-tap parity.** A user tapping a YubiKey once
   produces one signed-attestation payload; an MVP-1 host would
   re-use that attestation across multiple library calls within
   the freshness window via a host-side cache. P11A's PoC pattern
   matches this shape — one user gesture, N internal proofs —
   even though under PoC the proofs are mostly a typecheck
   formality.

**Subtle concern acknowledged:** the multi-proof pattern under PoC
is structurally equivalent to "skip presence after one prompt"
because `PressYPresenceProof` is a stand-in. An auditor who reads
the CLI code may flag the loop as "presence is always satisfied
once you typed 'y' once." **The auditor's reading is correct under
PoC**; the discipline is type-system-honest (every reveal call
takes a `&dyn PresenceProof` and the proof is verified per call)
but the underlying proof type is a placeholder. MVP-1's switch to
hardware-attestation will expose any cases where the multi-proof
loop is too generous; under PoC the Q4-locked decision is "ship
the multi-proof loop; document the shortcut; revisit at MVP-1."

**Alternative considered:** Prompt once per reveal flag (3 prompts
for `show --reveal-password --reveal-notes --reveal-totp-secret`).
**Rejected** — friction without measurable security gain under
PoC's PressY proof. Each prompt would be the same gesture;
bundling them is the natural UX.

**Alternative considered:** Construct one proof and re-use a
`Cell<bool>`-cleared variant. **Rejected** — it requires either
forking `PressYPresenceProof` (new public type, audit cost) or
reaching into the proof's internal state (visibility violation).
The N-fresh-proofs path is type-clean.

### A8. Multi-value URL / username — defer to MVP-1

**Decision:** `account add` and `account update` ship single `--url
<url>` and `--username <user>` flags; multi-value (repeated flag,
clap `Vec<String>`) is deferred to MVP-1. The `AccountSnapshot`
struct has single `url` and `username` fields; multi-URL would
require either an internal CSV-encoding convention (fragile) or a
schema bump (out of P11A scope).

**Rationale:**

1. **`AccountSnapshot` shape doesn't support multi-value cleanly.**
   Splitting on commas / semicolons is a parsing convention that
   leaks into URL content (URLs can contain commas; semicolons
   appear in some service URLs). A clean implementation requires
   schema additions.

2. **The library API is the security boundary.** P11A is a thin
   shell; if multi-URL needs schema work, that's a separate
   library-side issue, not a CLI-side patch.

3. **MVP-1 issue 1.x can pick up multi-URL** (the design spec
   §5.4 specifies multi-URL via a separate `urls: Vec<Url>` field;
   schema + AAD discipline + migration story is non-trivial and
   warrants its own plan).

**Alternative considered:** Ship a CSV-split `--url
"a.com,b.com,c.com"` convention. **Rejected** — see §1.

### A9. Confirmation on delete — case-sensitive `"yes"`; `--yes` flag bypasses

**Decision:** The default `account delete` confirmation requires the
literal lowercase string `"yes"`. Anything else (including `"y"`,
`"YES"`, `"Yes"`, `""`, EOF) is interpreted as "abort." `--yes`
flag bypasses the prompt entirely (for scripted use).

**Rationale:**

1. **High-friction confirmation is the point.** `delete` is
   tombstoning; the chain-side write is irreversible (P10's
   tombstones are append-only). The friction "type 'yes' in
   full" is documented in `git push --force`, `rm -i`, and most
   credential-tooling — the user mistyping `"y"` and getting a
   "deleted!" outcome is exactly what we want to avoid.

2. **Case-sensitive avoids the "habit-formed-y" reflex.** Users
   who reflexively type `"y"` to every prompt have not paused to
   consider the action; the case-sensitive `"yes"` requires
   conscious typing.

3. **`--yes` is the audit-traceable opt-out.** In a script /
   CI / fixture-build context, the user explicitly opts out of
   the prompt; the flag's presence is visible in logs.

**Alternative considered:** Accept `"y"` / `"yes"` / `"YES"` (case-
insensitive). **Rejected** — see §2.

**Alternative considered:** Default `--yes` and require `--no` for
the prompted variant. **Rejected** — flips the polarity from "safe
default" to "footgun default."

### A10. `account show` / `update` / `delete` on frozen / tombstoned account — clear-error UX

**Decision:** When the user invokes `account show <id>` /
`account update <id> ...` / `account delete <id>` against an
account whose row is frozen or tombstoned, the CLI surfaces a
clear, actionable error message and exits non-zero. Specifically:

- **Frozen:** `"account <hex-id> is frozen pending resolve. Run
  `pangolin-cli resolve --account-id <hex-id> --keep <head>` to
  resolve, then retry."` (The `--keep <head>` hint cannot be
  filled in by the CLI without listing the heads; either the user
  inspects via `pangolin-cli status` or P11A's frozen error
  includes the head list — recommend the latter for UX.)
- **Tombstoned:** `"account <hex-id> has been deleted (tombstoned).
  Create a new account with a different identifier; resurrection
  is not supported under the PoC append-only model."`
- **Unknown:** `"no account with id <hex-id> in this vault."`

For `update` + `delete`, the underlying library calls
(`update_account` / `delete_account`) enforce these via
`StoreError::AccountFrozenPendingResolve` /
`StoreError::AccountTombstoned`; the CLI maps the error variant to
the user-facing string. For `show`, `Vault::get_account` returns
`None` for both frozen + tombstoned + unknown — the CLI
distinguishes them via `Vault::list_frozen_accounts()` /
`Vault::list_tombstoned_accounts()` membership checks before
deciding which message to print.

**Rationale:**

1. **Distinguishability avoids user confusion.** "Account not
   found" for a frozen account is misleading — the user knows
   they created the account. The error message must point at the
   correct fix (resolve vs. recreate vs. typo).

2. **`get_account`'s `None`-returning shape collapses three cases.**
   The CLI re-distinguishes via the public `list_frozen_accounts`
   / `list_tombstoned_accounts` accessors; the membership probes
   are O(N) but PoC scale (N < 10^4) makes this trivially fast.

**Alternative considered:** A new library-side `Vault::get_account_status(id)
-> AccountStatus` enum with `{Active, Frozen, Tombstoned, Unknown}`
variants. **Rejected** — three accessors at the CLI level work
fine; new public API is unwarranted for a UX-distinguishing
feature.

### A11. `account list` — flat enumeration with `--include-frozen` / `--include-tombstoned` flags

**Decision:** Default `account list` shows only Active accounts (matches
`Vault::list_accounts`'s default filter behavior). `--include-frozen`
adds frozen accounts to the output, suffixed with `[frozen]`.
`--include-tombstoned` adds tombstoned accounts, suffixed with
`[tombstoned]`. Both flags can be combined. Search / filter on
display_name / URL is OUT OF SCOPE for P11A — recommend MVP-1.

**Rationale:**

1. **`list_accounts` already filters by default.** P11A's CLI
   default mirrors the library default. The `--include-*` flags
   surface the structurally-existing-but-default-hidden rows for
   debugging / pre-resolve inspection.

2. **Suffix annotation prevents silent confusion.** Without the
   `[frozen]` / `[tombstoned]` suffix, a user running
   `--include-frozen --include-tombstoned` gets a flat list with
   no signal about which rows are which state. The suffix is the
   minimum honest UX.

3. **Search / filter is MVP-1 polish.** The library `Vault::search`
   is presence-gated (touches the cache); `account list` should
   stay metadata-only-ish + presence-light. A `--filter <pattern>`
   flag would either silently use `search` (presence-required,
   conflicts with the metadata-only intent) or duplicate the
   search logic at the CLI level (audit cost). Defer.

**Alternative considered:** Three separate verbs:
`account list-active`, `account list-frozen`, `account list-tombstoned`.
**Rejected** — verb proliferation; the flag form composes more
naturally.

### A12. `--generate-password` — 24-char alphanumeric+symbol via `pangolin-crypto::aead::Nonce::random`'s underlying RNG

**Decision:** `--generate-password` produces a 24-character password
drawn from a 70-char alphabet (uppercase + lowercase + digits + 8
symbols `!@#$%^&*`). The RNG seam is `getrandom`-via-`OsRng` (the
same primitive used by `pangolin-crypto::aead::Nonce::random` for
AEAD nonces); P11A does NOT introduce a new public API in
`pangolin-crypto` for this — it imports `getrandom::getrandom` and
implements the alphabet-mapping inline in
`commands/account.rs::generate_password()`. The generated password
is printed to stderr inside a clearly-flagged block:
```
=========================================================
GENERATED PASSWORD (save this now; will not be shown again):
<password>
=========================================================
```
After printing, the password bytes flow into the `AccountSnapshot::password`
field and through `vault.add_account`; the CLI process drops the
buffer (zeroized) on stack-frame exit.

**Rationale:**

1. **24 chars × log2(70) ≈ 147 bits of entropy.** Far above the
   64-bit-min-entropy floor every modern guidance specifies; well
   above the typical 80-bit "deeply uncrackable" line.

2. **`OsRng` is the right primitive.** Pangolin's existing
   crypto layer routes through `OsRng` (via `getrandom`) for every
   secret generation; matching here is consistent.

3. **70-char alphabet avoids ambiguous characters.** No `0`/`O`,
   no `1`/`l`/`I`, no `'`/`"`/`` ` ``/`\` (shell-quoting
   landmines). The 8 chosen symbols `!@#$%^&*` are the
   intersection of "shell-safe across bash/zsh/cmd/PowerShell"
   and "accepted by 95%+ of password-policy regexes."

4. **Stderr printing avoids stdout-capture leak.** A user running
   `pangolin-cli account add --generate-password --name=foo > log`
   captures only the account_id + summary line; the password
   stays on the terminal (stderr).

**Alternative considered:** Diceware (5-word passphrase from a
~7,776-word EFF list). **Acceptable but not P11A scope** — would
ship the wordlist as a 30 KB binary asset; recommend MVP-1.

**Alternative considered:** Hex-only (32 hex chars = 128 bits, no
symbols). **Rejected** — long passwords without symbols are
disproportionately rejected by password-policy regexes; symbols
are a good idea.

### A13. Test seam for `rpassword::prompt_password`

**Decision:** P11A does NOT introduce a test seam for
`rpassword::prompt_password`. The interactive-prompt code paths are
exercised end-to-end via integration tests that use the
`--password-stdin` / `--totp-stdin` paths instead. The interactive
paths are tested at the **logic level** (e.g., the
`confirm_password` comparison closure is unit-tested with
hand-supplied byte strings).

**Rationale:**

1. **`rpassword` is a thin wrapper around `tcgetattr` /
   `ReadConsoleInputW`.** Mocking it requires either a wrapper
   trait (new public surface) or a `cfg(test)`-gated alternate
   implementation (audit cost). The actual logic worth testing
   (the password-mismatch comparison; the bounded-retry loop;
   the eprintln of the prompt string) is testable in isolation
   without `rpassword` in the loop.

2. **The interactive path is a thin glue layer.** The `bail!` on
   mismatch + the loop bound are 5 lines of Rust; the test seam
   would weigh more than the code under test.

3. **MVP-1 may revisit** if the host-app integration needs
   programmatic prompt control. Under PoC + CLI-only model,
   `--password-stdin` covers every CI use case.

**Alternative considered:** Wrap `rpassword` in a `PromptIo` trait
with a `RealPromptIo` and `MockPromptIo`. **Rejected** for the
audit-cost reason in §1.

**Decision:** P11A extends E2E-003 ("pangolin-cli two-vault sync
round-trip") with a one-paragraph note that the same scenario is
now reachable from the CLI without library calls. Does NOT add a

**Rationale:** E2E_TESTS.md tracks **scenarios**, not surfaces. The

account-management round-trip." **Could be done** — recommend

### A16. Forbidden user-facing terms

**Decision:** No P11A user-facing copy introduces any §3.5 forbidden
term ("gas", "blockchain", "transaction", "decentralized storage",
"hashes", "revisions"). The five verbs and their flag descriptions
use "credential" / "account" / "password" / "TOTP" / "vault" /
"delete" / "reveal" / "presence" — all in the password-manager
domain. Internal doc-comments inside `commands/account.rs` may use
Pangolin-internal terms (revision, tombstone, AEAD); the §3.5 gate
is `--help` output and `eprintln`/`println` strings.

**Rationale:** §3.5 is a user-facing-string gate; `cargo doc` is a
developer surface and uses internal terminology freely. The
distinction is what the master plan intends.

`--help` are spot-checked at SIGNOFF; the

## Threat model row (draft text for THREAT_MODEL.md)

The existing `pangolin-cli` row in `THREAT_MODEL.md` covers publish
+ pull (P8), resolve (P9), tombstones + offline (P10). P11A appends
account-subcommand paragraphs to the same row. The new threat
indices (placeholder `<N>`..`<N+4>`; expected 23–27 per §A14):

> <N>. **Password disclosure via process listing
>      (`ps aux` / `/proc/<pid>/cmdline`).** Defense: P11A's
>      `account add` and `account update` REFUSE to ship a
>      `--password <flag>` argument form. The only paths to
>      provide a password are interactive terminal prompt
>      (`rpassword::prompt_password`, no echo) or stdin
>      (`--password-stdin`, redirected by the user). Same shape
>      as `pass`, `1password-cli`, `bw`, `op`. The
>      `--vault-password <flag>` and `--keystore-password
>      <flag>` arguments inherited from P8 retain the same
>      "echoes in ps; CI only" caveat in their `--help` text;
>      P11A does NOT ship a flag-form for credential
>      passwords. The TOTP secret follows the same discipline.
>      Notes accept the flag form (`--notes <str>`) per P4's
>      lower-tier reveal-class hierarchy and §A5's documented
>      trade-off; users running interactively are recommended to
>      use `--notes-stdin`. The `account show --reveal-password`
>      output prints to stdout; shell-history capture is the
>      user's risk to manage (no different from
>      `pass show <name>`).
> <N+1>. **Account-show plaintext leak via shell history /
>       terminal scrollback.** Defense (acknowledgement, UX-
>       bound): the `--reveal-password` / `--reveal-notes` /
>       `--reveal-totp-secret` flags require a presence proof
>       (P4 §5.4 high-risk-action escalation) — the user types
>       `'y'` at the `"presence required:"` prompt before any
>       reveal call fires. Once revealed, plaintext is on the
>       user's terminal scrollback; the CLI cannot retract it.
>       The presence prompt is the load-bearing mitigation; the
>       prompt's wording explicitly names the action ("reveal
>       password for account <id>"), so a user who didn't intend
>       a reveal can decline. Multi-flag invocations
>       (`--reveal-password --reveal-notes --reveal-totp-secret`)
>       prompt ONCE per §A7 and produce three internal
>       `PressYPresenceProof::confirmed()` instances against the
>       single user gesture; the same shape MVP-1's hardware
>       attestation will surface. Default `account show` (no
>       reveal flags) prints non-secret fields only and emits no
>       presence prompt. Audit reviewers should scrutinize:
>       does the JSON output `--json` mode omit (not null-
>       fill) the unrevealed secret fields? Verified by
>       `show_json_omits_password_field_when_not_revealed`.
> <N+2>. **Tombstone replay via `account delete`.** Defense:
>       same protection as P10's tombstone discipline — the
>       tombstone revision's canonical hash binds
>       `(vault_id, account_id, parent_revision, schema_version,
>       enc_payload)`; replay against a moved-on head produces a
>       fork rather than a duplicate (Cardinal Principle 4 holds
>       at the `delete` site too — same chain ordering as
>       publish/update). `Vault::delete_account` refuses on
>       already-tombstoned accounts with
>       `StoreError::AccountTombstoned`; the CLI surfaces "
>       already deleted" rather than silently re-tombstoning
>       (idempotency-by-clear-error). The append-only invariant
>       (P10 anti-resurrection: `Vault::add_account` refuses to
>       reuse a tombstoned `account_id`) extends to the CLI
>       boundary unchanged.
> <N+3>. **Reveal-confirmation phishing under PoC PressYPresenceProof.**
>       Defense (acknowledgement, PoC limitation): the PoC
>       `PressYPresenceProof` is a stand-in for MVP-1's hardware
>       attestation; under PoC, an attacker who has stolen the
>       user's session-active vault state (e.g., post-unlock
>       memory dump, or unattended unlocked terminal) can fire
>       any reveal call by typing `'y'` at the prompt. The
>       MVP-1 hardware-attestation switch closes this; under
>       PoC, the `'y'` keystroke is the only proof-of-presence
>       available. P11A inherits this limitation unchanged. The
>       `'y'` prompt's wording explicitly names the action
>       (account id + which secret) so an unattended terminal
>       attacker who automates the prompt response sees
>       distinguishable per-account audit lines. Documented as a
>       known PoC limitation; closed by MVP-1's hardware path.
> <N+4>. **`account update` of frozen account.** Defense:
>       `Vault::update_account` refuses with
>       `StoreError::AccountFrozenPendingResolve` (P8 CRIT-1
>       freeze guard). The CLI surfaces the error with a clear
>       hint ("run `pangolin-cli resolve --account-id <hex>
>       --keep <head>` first; the heads are: ..."). The user
>       cannot accidentally write a stale-plaintext-based update
>       to a frozen account (Cardinal Principle 4 protected at
>       the CLI boundary). Same discipline applies to `account
>       delete` of a frozen account (refuses with the same error
>       + hint). Resolve flow per P9 is unchanged.

## Failure modes considered

| Failure mode | Mitigation |
|---|---|
| User runs `pangolin-cli account add` with `--password <flag>` form | A3 — clap rejects (the flag does not exist); the user reads `--help` and switches to `--password-stdin` or interactive prompt. |
| User pipes the wrong password to `--password-stdin` | A3 — `account add` requires only one password (no confirmation in the stdin path because confirmation defeats the scripted-pipe purpose); user retries by deleting + re-creating. |
| User runs `pangolin-cli account update` for an unspecified-field-only edit; expects no presence prompt | A6 — explicit help-text documentation that update is always presence-gated under PoC; MVP-1's partial-update API will lift this. |
| User runs `pangolin-cli account show --reveal-password` against a tombstoned account | A10 — `Vault::get_account` returns `None` for tombstoned; CLI surfaces "account has been deleted" without prompting for presence (the prompt would be wasted user gesture). |
| User runs `pangolin-cli account show --reveal-password` against a frozen account | A10 — `Vault::get_account` returns `None` for frozen too; CLI surfaces "account is frozen pending resolve; run `pangolin-cli resolve` first." |
| User runs `pangolin-cli account delete` and types `"y"` at the prompt | A9 — refused; the literal lowercase `"yes"` is required. User retries with `"yes"`. |
| User runs `pangolin-cli account delete <typo-id>` | A10 + A9 — the lookup via `Vault::get_account` either returns `None` (clear error: "no account with id <hex>") or returns a snapshot whose display_name is shown in the confirmation prompt; the user spots the typo before typing `"yes"`. |
| User runs `account add` against a `.pvf` file that doesn't exist | Inherits P8 fix MED-3 — `canonicalize_vault_path` errors with "could not canonicalize vault path" + a "file missing? broken symlink?" hint. P11A does NOT auto-create the vault file (open question §8.1 — recommend NOT auto-creating; vault creation is a separate verb / responsibility). |
| User runs `account list --include-frozen --include-tombstoned` against a vault with no frozen / tombstoned accounts | Empty list; suffix annotations are absent; user sees the same output as the default. |
| User runs `account update` and the password reveal call fails (e.g., session expired mid-run) | The reveal call surfaces `StoreError::SessionExpired`; the CLI surfaces the error and aborts WITHOUT writing the new revision. The user re-runs (re-unlocks) with the same args. |
| User runs `account add` with `--generate-password` and pipes stdout to a logger | The generated password is printed to stderr (not stdout); only the `account_id` + summary line is on stdout. The logger captures the audit trail without the password. |
| User runs `account add` and then `account list` shows nothing | `Vault::list_accounts` requires an active session (the same vault open + unlock pattern); user must re-unlock. **Open question §8.2:** should the CLI keep the unlock alive across the two invocations (no — each invocation is a fresh process; vaults are session-scoped to a process)? |
| User runs `account delete` with `--yes` against a frozen account | A10 — `delete_account` refuses with the freeze error before the `--yes` short-circuit fires. The user is forced through resolve first. |
| User runs `account update --password-prompt` and types two different passwords | A3 — bounded retry (2 retries) + abort. User re-runs. |
| User runs `account show` against a vault that's never been unlocked | The vault is in `Locked` state; `Vault::get_account` returns `None`; the CLI's user-facing message must distinguish "vault is locked" from "account doesn't exist." Recommend: check `vault.is_session_active()` after `open_and_unlock`; if not active, surface "vault unlock failed" via the `open_and_unlock` error path. |
| User runs `account list` with a vault that has 10,000 accounts | `list_accounts` is O(N); 10K rows is fine for PoC. CLI prints all rows; if `--json`, the array is one entry per row. No pagination — open question §8.3 (recommend MVP-1 polish). |
| User runs `account add` with display_name containing newlines | `AccountSnapshot::new` accepts arbitrary bytes; the eprintln summary line may be confusing if the name has newlines. Recommend: clap-level `value_parser` rejects names with `\n` or `\r`; flag at BUILD if the rejection breaks legitimate use cases. |

## Rollback / abort plan

If BUILD reveals the design is fundamentally wrong (per §16.7):

- **A6 three-reveal pattern is too costly UX-wise** (e.g., the
  three presence prompts overlap awkwardly with rpassword's
  terminal-mode handling): split `update` into two paths — a
  "non-secret-only update" path that does NOT presence-escalate,
  and a "secret-field update" path that does. Requires a small
  library-side addition (`Vault::update_non_secret_fields(id,
  partial_snapshot)`) that takes only the non-secret fields. New
  D-NNN entry locks the architectural change. Documented as a
  rollback point.
- **A12 `--generate-password` ships entropy below 128 bits** (e.g.,
  the alphabet's chosen 70 chars are inadvertently restricted by
  shell-quoting in some tested shells): expand the alphabet to 88
  chars (drop the ambiguous-char filter); recompute entropy; ship
  with a documented "may surface as confusable in some terminal
  fonts" caveat.
- **A1 nested-namespace conflicts with future verb additions** that
  Kelvin discovers during BUILD (e.g., MVP-1 needs
  `pangolin-cli account` to be a chain-ish operation, conflicting
  with the local-only verbs P11A ships): add a clap alias-form
  fallback (`pangolin-cli credential <verb>` in addition to
  `account <verb>`); document the synonym; defer the rename to
  MVP-1.
- **A9 case-sensitive "yes" causes test flakiness on Windows
  CRLF-stripping** (`"yes\r\n"` → `"yes\r"` after a single trim):
  switch to `trim_end_matches(&['\r', '\n'][..])` explicitly;
  re-run tests; the `"yes"` discipline holds.
- **A11 frozen-included list rendering exceeds shell width** for
  many-row vaults: add automatic suffix-truncation or move to a
  table-format crate. MVP-1 polish; for P11A's PoC scale, OK.

If a more fundamental design problem surfaces (e.g., the partial-
update reveal-three pattern fundamentally breaks the
`PressYPresenceProof` discipline in a way that compromises P4's
single-use-flag invariant), escalate to Kelvin per §16.7. Do not
silently patch around.

## Open questions for Kelvin

1. **Auto-create vault on `account add` against a missing
   `.pvf`?** Recommend NO. Vault creation is a separate
   responsibility; conflating it with `account add` makes the verb's
   semantics ambiguous (was the vault freshly-created or did the
   user typo the path?). Recommend: a future `pangolin-cli vault
   create --vault-path <path>` verb (MVP-1) that creates the
   `.pvf` file with a fresh password; `account add` always
   requires an existing vault. Lean toward NO; flag at BUILD if
   the reproducer's manual steps suggest otherwise.

2. **Should `--reveal-password` print to stdout (current plan) or
   copy-to-clipboard?** Recommend stdout for CLI consistency under
   PoC (matches `pass show <name>`'s default; clipboard requires
   cross-platform deps that complicate the dependency graph).
   Clipboard is MVP-1 polish; recommend deferring.

3. **Should `account delete`'s confirmation prompt include the
   target's display_name to prevent typo-deletes?** Recommend YES
   (current plan). Without the display_name, a user who typoed
   the `<hex-id>` cannot easily catch the mistake before typing
   `"yes"`. The cost is one extra `Vault::get_account(id)` lookup,
   which is trivial.

4. **Should `account update` accept JSON via stdin (bulk edits)?**
   Recommend NO under PoC. Bulk edits are an MVP-1 polish item
   that introduces a stdin-parsing surface (JSON shape, error
   recovery, partial-success semantics) needing its own plan.

5. **Should `account add --generate-password` print the password
   to clipboard instead of stderr?** Recommend stderr (current
   plan). Cross-platform clipboard requires deps;
   stderr is universally available; the user can pipe stderr to
   a clipboard tool (`pangolin-cli account add --generate-password
   --name foo 2>&1 | xclip -selection clipboard`). MVP-1 polish.

6. **Should the `--include-frozen` and `--include-tombstoned`
   flags on `account list` be combined into a single
   `--include <state>` flag (`--include frozen,tombstoned`)?**
   Recommend separate flags (current plan). Compositional clarity;
   each flag toggles one structural axis. A combined flag adds
   parser complexity without UX gain.

7. **Should the `--reveal-*` flags on `account show` be combined
   into a single `--reveal-all` flag?** Recommend NO under P11A.
   Each flag is an explicit per-secret-field opt-in; bundling
   them into `--reveal-all` makes the command silent about which
   secrets it's about to expose. MVP-1 may revisit if user
   feedback complains; under PoC, explicit is better.

8. **Should `account delete --yes` require an additional
   `--force` flag to override the freeze-guard?** Recommend NO.
   The freeze-guard is a load-bearing safety property (P8 CRIT-1);
   a `--force` flag would be a footgun that bypasses Cardinal
   Principle 4. Resolve-then-delete is the right path under PoC.

9. **Should `account update --totp-clear` actually clear the
   field (set to empty `SecretBytes`) or remove the column
   entirely?** Recommend empty SecretBytes (matches the existing
   library convention; `AccountSnapshot::totp_secret` is `SecretBytes`
   that may be empty). Removing-the-column would be a schema bump
   out of P11A scope.

exposes presence-escalation discipline at the CLI boundary; ships

  flag-form for password input? Grep the `cli.rs` arg structs:
  no `--password <flag>` field, only `--password-stdin` (bool) and

  `show --reveal-password --reveal-notes --reveal-totp-secret` path
