## Decisions locked (post plan-gate)

| Q | Decision | Rationale |
|---|---|---|
| Q1 ‚Äî HMAC-SHA1 source | **`pangolin-totp` pulls its own `hmac` + `sha1` (+ `sha2` for the SHA-256/512 variants).** `hmac 0.12.x` is already in `Cargo.lock` (via `hkdf` in `pangolin-crypto`); `sha1` is the only genuinely-new transitive (plus `sha2` if not already present ‚Äî it is, via `pangolin-crypto`'s SHA-256/512 use). These deps live in `pangolin-totp`'s tree only ‚Äî they never reach `pangolin-core` or `pangolin-crypto` (verify the dep arrow: `pangolin-ffi ‚Üí pangolin-totp`; nothing points back). No `deny.toml` change: that file is a *denylist* (`ring`, `openssl`, `aes-gcm`, ‚Ä¶) + `wildcards = "deny"`, not an allowlist ‚Äî `sha1`/`hmac`/`sha2` aren't denied. The new deps just need `=`-version pins in `pangolin-totp/Cargo.toml` + a committed `Cargo.lock`; `cargo audit` scans them. | Blast-containment per master plan ¬ß16.8 footnote 2 ("Per-crate `forbid(unsafe_code)`, tight per-crate `deny.toml` scopes, blast-contained parser bugs") ‚Äî any RFC 6238 / HMAC-SHA1 bug stays in `pangolin-totp`; `pangolin-crypto`'s audited boundary (and its zero-serde HIGH-1 invariant) is untouched. Adding HMAC-SHA1 to `pangolin-crypto`'s public surface instead would widen that audited crate's API + put `sha1` in *its* tree ‚Äî more churn in the wrong place. |
| Q2 ‚Äî Configurable TOTP params | **Full configurable set: algorithm ‚àà {SHA1, SHA256, SHA512}, digits ‚àà {6, 7, 8}, period (seconds, default 30, validated > 0).** The params are stored *alongside* the secret: the `AccountIdentity` CBOR body extends to carry the TOTP entry as `{ secret_bytes, algorithm, digits, period }` (where `secret_bytes` empty == no TOTP). That is a new `payload_version` (V2) on the body ‚Äî 1.6's ¬ß18.7 machinery handles it: an older Pangolin reading a V2-payload account surfaces "requires upgrade" (per-account, not whole-vault); V0/V1 vaults still open (a V0/V1 `totp_secret: SecretBytes` blob is read as `{ secret_bytes: <those bytes>, algorithm: SHA1, digits: 6, period: 30 }` ‚Äî the RFC defaults ‚Äî on load). +3-4h vs defaults-only. | A password manager generating *wrong* TOTP codes (because it couldn't represent a SHA-256 / 8-digit / non-30s config and silently coerced to defaults) can lock a user out of a site ‚Äî worse than most bugs. With the full set every `otpauth://` URI and every KDBX-imported TOTP entry (1.9) Just Works on day one; no "not yet supported" refusals. The +3-4h is small in the MVP-1 context, and 1.6's `payload_version` mechanism already exists to absorb exactly this kind of body extension. |
| Q3 ‚Äî TOTP code generation access class | **Session-class.** `totp_generate(handle, id, at)` needs only an unlocked, non-expired vault ‚Äî NO presence proof (it does NOT take a presence argument; that matches the 1.1-frozen signature). The *raw seed* stays reveal-class (1.4's `reveal_totp_secret`, presence-gated). | The TOTP *code* is the ephemeral, user-facing artifact (it's what the user types into a site, refreshed every 30 s ‚Äî gating each one behind a presence prompt would be intolerable friction); the *seed* is the durable secret (revealing/exporting it is the ¬ß5.4 reveal-class operation). The spec's session model puts "copy TOTP code" in the same class as "open account" ‚Äî session-class. Generating a code does decrypt the seed transiently inside the vault, but the seed bytes never cross FFI (only the 6-8 digit code does); the transient plaintext seed in `pangolin-totp` is zeroized after use. |
| Q4 ‚Äî base32 decoding | **Hand-roll.** ~30 lines: RFC 4648 base32 (uppercase A-Z + 2-7, case-insensitive in practice, optional `=` padding tolerated/stripped), reject any other char with a typed error. No new dependency. | A 30-line decoder is trivially auditable, has zero supply-chain surface, and TOTP secrets are short ‚Äî there's no performance argument for a crate. Keeps `pangolin-totp`'s dep tree to just the HMAC/SHA primitives + `zeroize`. |
| Q5 ‚Äî `pangolin-cli account totp <id>` subcommand in 1.7? | **No new CLI subcommand** ‚Äî deferred to the standing CLI-V1-wiring follow-up alongside `account` / `reveal` / `device`. **But** 1.7 fixes a latent CLI bug: `--totp-stdin` / `prompt_totp_secret` currently stores the typed string's *raw bytes* verbatim even though the prompt says "base32" ‚Äî so a PoC CLI TOTP entry is garbage. 1.7 wires the new `pangolin-totp` parser there (accept a bare base32 secret *or* a full `otpauth://totp/‚Ä¶` URI; on a non-default-param URI, store the parsed params per Q2). | Consistent with 1.2-1.6's CLI-deferral; one cohesive CLI-V1 issue eventually. The base32-decode fix is in-scope because 1.7 is the issue that makes the stored bytes *mean* something (an RFC 6238 secret) ‚Äî leaving the CLI storing garbage would be a known-broken path. |

## Spec reference

- **Master plan ¬ß4 (Phase 1 ‚Äî MVP-1) row `1.7`** (verbatim):
  > "TOTP engine. RFC 6238, time-drift handling, generation purely local,
  > never sent over wire in plaintext. Seed is a high-risk asset (presence
  > escalation to reveal/export). | TOTP spec; Session spec ¬ß5.4"
- **Master plan ¬ß17 component matrix** (verbatim):
  > "| TOTP generation (RFC 6238) | core | 1.7 | TOTP spec |"
- **Master plan ¬ß16.8 footnote (separate-crate rationale)** ‚Äî `pangolin-totp/`
  is a standalone workspace crate, not a `pangolin-core` sub-module:
  per-crate `forbid(unsafe_code)`, tight per-crate `deny.toml` scopes,
  blast-contained parser bugs, and the HMAC-SHA1 dependency surface never
  reaching `pangolin-core` / `pangolin-crypto` (preserving HIGH-1 zero-serde).
- **RFC 6238 (TOTP) + RFC 4226 (HOTP)** ‚Äî the normative algorithm:
  - `T = floor((current_unix_time - T0) / X)`, `T0 = 0` (Unix epoch),
    `X = 30 s` (the standard time step / period).
  - `HOTP(K, C) = truncate(HMAC-H(K, C))` where `C` is the 8-byte big-endian
    counter; `H = SHA-1` by default (RFC 6238 also defines SHA-256 / SHA-512).
  - Dynamic truncation: `offset = HMAC[len-1] & 0x0F`;
    `binCode = (HMAC[offset] & 0x7F) << 24 | HMAC[offset+1] << 16 |
    HMAC[offset+2] << 8 | HMAC[offset+3]`; `code = binCode mod 10^digits`,
    left-zero-padded to `digits` (default 6; 7 / 8 also in use).
  - **RFC 6238 Appendix B test vectors** ‚Äî secret = ASCII `"12345678901234567890"`
    (SHA-1), `"12345678901234567890123456789012"` (SHA-256),
    `"1234567890123456789012345678901234567890123456789012345678901234"`
    (SHA-512); at `T ‚àà {59, 1111111109, 1111111111, 1234567890, 2000000000,
    20000000000}`; canonical 8-digit codes given. **The engine MUST reproduce
    these exactly.**
  - **RFC 4226 ¬ß4 R6** recommends a shared secret of at least 128 bits
    (16 bytes), 160 bits (20 bytes) RECOMMENDED. Real-world secrets are
    frequently shorter (80‚Äì160 bits); we do not reject below 16 bytes (see Q2 /
    failure modes) ‚Äî generation is well-defined for any non-empty key.
- **Pangolin TOTP spec** (`pangolin_main_text.txt` L3969‚Äì4110, "TOTP /
  TOTP Integration into Pangolin Session Authority") ‚Äî verbatim load-bearing
  fragments:
  > "In Pangolin, a TOTP secret is treated as a first-class credential secret
  > attached to an Account Identity (same as password history, URLs, notes).
  > TOTP generation occurs locally only."
  > "A) Low-risk (allowed with active session) ‚Ä¶ Copy TOTP code (with optional
  > clipboard auto-clear) ‚Ä¶ View the 6-digit code briefly in-context during
  > login ‚Ä¶ Requirement: Active session must exist (or session must be
  > started)."
  > "B) High-risk (presence escalation recommended) ‚Ä¶ Reveal the TOTP secret
  > (the seed / QR / URI) ‚Ä¶ Export TOTP secret ‚Ä¶ Display QR code ‚Ä¶ Rule:
  > Presence proof required for seed-level operations."
  > "Default Policy: TOTP codes follow normal session rules: session required;
  > no extra presence for each code. TOTP seeds require presence proof: always
  > require tap to show/export seed."
  > "7) Optional Phase 2 refinement ‚Ä¶ For high-risk accounts, allow per-account
  > policies: 'TOTP code requires presence every time' (paranoid mode) ‚Ä¶"
  - **Note:** the Pangolin TOTP spec describes only the *session/risk
    classification* of TOTP actions. It is **silent** on configurable
    parameters, on the import format (`otpauth://` URI / base32 / raw bytes),
    and on a minimum seed length. The "view the 6-digit code" phrasing implies
    the RFC-6238 defaults are the baseline; configurability is an
    interoperability nicety the spec neither mandates nor forbids ‚Üí Q2.

  the raw TOTP seed is reveal-class material; 1.4 already gates the *reveal*

## Goal

After this issue closes, `pangolin-totp` is a real RFC 6238 implementation: a
pure, deterministic `totp_at(secret, at, params) -> TotpCode` plus an
`otpauth://` URI / bare-base32 parser that turns a user- or KDBX-supplied
string into the raw seed bytes (+ params) stored in `AccountIdentity.totp_secret`.
The 1.1-frozen `totp_generate` FFI entry is wired: it looks up the account,
decrypts the seed via the unlocked vault, computes the current 30-second
window's code with the device-supplied timestamp, and returns the digits plus
the seconds remaining in the window (so the UI can render a countdown). The
engine reproduces the RFC 6238 Appendix B test vectors byte-for-byte. The raw
seed remains reveal-class (only `reveal_totp_secret` exposes it, presence-gated);
generating a code requires only an active session. No `unsafe`; `forbid(unsafe_code)`
retained on `pangolin-totp`. Existing PoC `.pvf` vaults still open / unlock /
`account_history`; if a PoC vault carries a `totp_secret`, `totp_generate`
either produces a code (the bytes are usable as an HMAC key ‚Äî always true) or
surfaces a clean typed error (e.g. empty seed ‚Üí "no TOTP for this account").

## Approach

### 1. The `pangolin-totp` RFC 6238 engine (`crates/pangolin-totp/src/lib.rs`, replacing the scaffold)

- Keep `#![cfg_attr(not(test), forbid(unsafe_code))]`. SPDX `AGPL-3.0-or-later`
  header on every new file.
- Core types:
  - `pub enum TotpAlgorithm { Sha1, Sha256, Sha512 }` ‚Äî `Sha1` is the
    interoperability default (Google Authenticator, the vast majority of sites).
  - `pub struct TotpParams { pub algorithm: TotpAlgorithm, pub digits: u8,
    pub period_seconds: u32 }` with `TotpParams::DEFAULT = { Sha1, 6, 30 }` and
    a `validate()` (digits ‚àà {6,7,8}; period ‚â• 1; ‚â§ some sane ceiling, e.g. 300).
    *(If Q2 = defaults-only, `TotpParams` collapses to a unit / is omitted and
    `totp_at` takes no params.)*
  - `pub struct TotpCode { pub code: String, pub seconds_remaining: u32,
    pub period_seconds: u32 }` ‚Äî the `pangolin-totp` value type (distinct from
    the FFI `pangolin_ffi::TotpCode` record). `code` is left-zero-padded digits.
    `Debug` is **redacting** (`TotpCode { code: "<redacted>", ‚Ä¶ }`) ‚Äî the code
    is the live second factor; treat it like a password in logs. Whether `code`
    is wrapped in a zeroizing buffer: recommend a redacting `Debug` + a `Drop`
    that zeroizes the `String`'s bytes (cheap; the code is short-lived (‚â§30 s)
    but is a credential ‚Äî defense in depth). Surface as part of Q3's discussion;
    not a separate question.
- Core function:
  ```rust
  pub fn totp_at(secret: &[u8], at_unix_secs: u64, params: &TotpParams) -> Result<TotpCode, TotpError>
  ```
  Steps: validate `params`; `counter = at_unix_secs / params.period_seconds`;
  `mac = HMAC-<algorithm>(key = secret, msg = counter.to_be_bytes())`; dynamic
  truncation per RFC 4226 ¬ß5.3; `code = (bin_code % 10u32.pow(digits)) ` zero-
  padded to `digits`; `seconds_remaining = params.period_seconds -
  (at_unix_secs % params.period_seconds)`. Intermediate `mac` bytes wrapped in
  `zeroize::Zeroizing` (the HMAC output is a function of the seed). Empty
  `secret` ‚Üí `TotpError::EmptySecret`.
- `pub enum TotpError` (`thiserror`): `EmptySecret`, `InvalidDigits(u8)`,
  `InvalidPeriod(u32)`, `MalformedOtpAuthUri { reason: String }`,
  `InvalidBase32 { reason: String }`, `UnsupportedAlgorithm(String)`,
  `SecretTooLong { len: usize, max: usize }` (mirrors 1.2's
  `TOTP_SECRET_MAX_BYTES`). All non-allocating-on-the-secret; `Debug`/`Display`
  never echo seed bytes.

### 2. `otpauth://` URI + bare-base32 secret parsing (`crates/pangolin-totp/src/parse.rs`)

- `pub struct ParsedTotpSecret { pub secret_bytes: zeroize::Zeroizing<Vec<u8>>,
  pub params: TotpParams, pub label: Option<String>, pub issuer: Option<String> }`
  ‚Äî `label`/`issuer` are advisory metadata (not stored by 1.7; the shell may
  use them to pre-fill `display_name`).
- `pub fn parse_otpauth_uri(input: &str) -> Result<ParsedTotpSecret, TotpError>`
  ‚Äî parse `otpauth://totp/Label?secret=BASE32&issuer=X&algorithm=SHA1&digits=6&period=30`
  by hand (no `url` crate dep in `pangolin-totp` ‚Äî a ~40-line scheme/path/query
  splitter; the `url` crate already lives in `pangolin-store`'s tree but not
  `pangolin-totp`'s, and `otpauth://` is a fixed-shape string). Required:
  scheme == `otpauth`, host == `totp` (reject `hotp` ‚Äî out of scope), a
  `secret=` query param. `algorithm` / `digits` / `period` optional ‚Üí default;
  unknown `algorithm` ‚Üí `UnsupportedAlgorithm`. Percent-decode the label.
- `pub fn decode_base32(input: &str) -> Result<Zeroizing<Vec<u8>>, TotpError>`
  ‚Äî RFC 4648 base32 (A‚ÄìZ, 2‚Äì7), case-insensitive, optional `=` padding
  tolerated (TOTP secrets are usually unpadded), spaces stripped (some sites
  print the secret in space-separated quads). Hand-rolled (~30 lines) per Q4;
  output zeroized.
- `pub fn parse_totp_secret(input: &str) -> Result<ParsedTotpSecret, TotpError>`
  ‚Äî the convenience front door: if `input` starts with `otpauth://` ‚Üí
  `parse_otpauth_uri`; otherwise treat the whole string as bare base32 ‚Üí
  `decode_base32` with `TotpParams::DEFAULT`. Enforce `secret_bytes.len() <=
  TOTP_SECRET_MAX_BYTES` (re-export the constant value, or accept it as a
  parameter to avoid a `pangolin-store` dep ‚Äî recommend a `pub const
  MAX_SECRET_BYTES: usize = 256;` in `pangolin-totp` with a doc-comment + a
  `static_assertions`-style note that it must equal
  `pangolin_store::limits::TOTP_SECRET_MAX_BYTES`; a unit test in the FFI or
  store crate cross-checks them).

### 3. HMAC dependency (the load-bearing choice ‚Äî Q1)

Two options; **recommend (b)**:

- **(a)** Add `hmac_sha1(key, msg) -> [u8; 20]` (and possibly `hmac_sha256` is
  already feasible via `sha2`) to `pangolin-crypto`'s public surface. Pros:
  one HMAC home; `pangolin-crypto` already uses `hmac` transitively (via
  `hkdf`). Cons: pulls a `sha1` crate into `pangolin-crypto`'s tree (currently
  `sha2` only) ‚Äî widens `pangolin-crypto`'s crypto-allowlist (its `lib.rs`
  enumerates allowed primitives + the workspace `deny.toml`); contradicts the
  master plan ¬ß16.8 "the HMAC-SHA1 dependency surface never reaches
  `pangolin-crypto`" rationale for the separate crate.
- **(b) ‚úÖ recommended** ‚Äî `pangolin-totp` pulls its own `hmac` + `sha1` (and
  `sha2` for the SHA-256/512 algorithm options, if Q2 = configurable) crates,
  kept entirely in `pangolin-totp`'s dep tree. This is exactly the
  blast-containment story ¬ß16.8 cites. `pangolin-crypto` stays untouched;
  HIGH-1 is preserved by construction (`pangolin-crypto` does not depend on
  `pangolin-totp`).
  - **`deny.toml` impact:** the `[bans]` table is closed-world only for the
    *deny list* (env-quirk #3 caveat) ‚Äî there is **no allowlist of permitted
    crates** in this `deny.toml`; `bans.deny` is a denylist plus
    `wildcards = "deny"`. Neither `hmac` nor `sha1` nor `sha2` is on the deny
    list, so **no `deny.toml` change is needed** as long as the new deps are
    exact-version-pinned (`wildcards = "deny"` requires `=X.Y.Z`). Add
    `hmac` / `sha1` (/ `sha2`) to `[workspace.dependencies]` with `=`-pins:
    `hmac = "=0.12.1"` (already in `Cargo.lock` via `hkdf` ‚Äî no new transitive
    crate, just a new direct edge), `sha1 = "=0.10.6"` (RustCrypto; **new** to
    the lock ‚Äî `cargo audit` will scan it; SHA-1 has no known practical
    HMAC-relevant attack ‚Äî the collision attacks do not affect HMAC), and (if
    needed) reuse the existing `sha2 = "=0.10.9"` workspace pin.
  - **`Cargo.lock` impact:** `cargo update -p pangolin-totp` (or a plain
    `cargo build`) regenerates the lock with the `sha1` entry; commit the lock.
  - **`cargo audit` impact:** re-run after the dep lands; no advisory against
    RustCrypto `sha1` 0.10.x as of this writing ‚Äî if one appears, the
    `--ignore` inlining pattern (env-quirk #3) is the escape hatch, but prefer
    a version bump.

### 4. The `totp_generate` FFI entry + the code-vs-seed access-class split (`crates/pangolin-ffi/src/totp.rs`)

- Implement the `todo!()` body:
  1. `let mut guard = handle.lock_vault(); let vault = guard.as_mut()?;` ‚Äî
     locked vault ‚Üí `FfiError::NotUnlocked` (the existing `as_mut()?` pattern,
     same as `reveal_*`).
  2. **Q3 ‚Äî access class.** Per the TOTP spec quote ("Copy TOTP code ‚Ä¶
     Requirement: Active session must exist; no extra presence for each code"),
     **recommend session-class**: `totp_generate` needs only an unlocked,
     non-expired session ‚Äî *no presence proof parameter* (matching the
     1.1-frozen signature, which has none). This is the right default: the code
     is the ephemeral, user-facing artifact; the durable seed is the
     reveal-class secret (`reveal_totp_secret`, 1.4-gated). The Phase-2
     "paranoid mode" (presence every time) is explicitly out of MVP-1 scope.
     *(If Q3 = reveal-class, the 1.1-frozen signature must grow a
     `presence: PresenceProof` parameter ‚Äî a surface amendment, allowed under
     the same posture as 1.2/1.4 amendments, but heavier and against the spec's
     stated default.)*
  3. Resolve the account; read the (decrypted) `totp_secret` bytes via a new
     crate-private `Vault` accessor ‚Äî the cleanest is a
     `Vault::totp_secret_bytes(&self, id) -> Result<Zeroizing<Vec<u8>>,
     StoreError>` that returns the head snapshot's `totp_secret` *without* a
     presence check (it does NOT cross the FFI as a secret ‚Äî it's consumed
     internally by `totp_at` and only the derived code escapes). Empty seed ‚Üí
     `StoreError::Validation { kind: "totp_secret", .. }` mapped to a typed
     "no TOTP configured for this account" `FfiError`.
  4. Read the stored params: if Q2 = defaults-only, `TotpParams::DEFAULT`; if
     Q2 = configurable, the params travel with the seed in storage (see ¬ß5).
  5. `let c = pangolin_totp::totp_at(&seed, at.into(), &params)?;` map
     `TotpError` ‚Üí `FfiError`.
  6. Return `TotpCode { schema_version: <current>, code: c.code,
     seconds_remaining: c.seconds_remaining.try_into().unwrap_or(u16::MAX) }`.
- Add `parse_totp_secret` to the FFI surface (additive amendment ‚Äî allowed):
  ```rust
  #[uniffi::export]
  pub fn parse_totp_secret(input: String) -> Result<ParsedTotpSecretFfi, FfiError>;
  // ParsedTotpSecretFfi { schema_version: u16, secret: Arc<TotpSecret>, algorithm: String, digits: u8, period_seconds: u32, label: Option<String>, issuer: Option<String> }
  ```
  The shell calls this on the user's pasted `otpauth://` URI / base32 string,
  then passes the resulting `Arc<TotpSecret>` to `account_update`
  (`AccountPatch.totp_secret = Some(Some(secret))`). This keeps `account_*`
  signatures **frozen** ‚Äî `pangolin-totp` owns all the format knowledge.
- The CLI (`apps/cli/src/commands/account.rs`): the interactive
  `prompt_totp_secret` already says "base32" ‚Äî 1.7 should route the typed
  string through `pangolin_totp::decode_base32` (or `parse_totp_secret`) before
  building the `SecretBytes`, fixing the latent "prompt says base32 but stores
  verbatim" gap. `--totp-stdin` likewise. This is a small, contained change in
  the CLI binary (not a new subcommand). Whether a *new* `pangolin-cli account
  totp <id>` (generate-and-print) subcommand lands ‚Üí Q5; **recommend no** ‚Äî
  defer to the CLI-V1 issue (consistent with 1.2/1.3/1.4/1.5/1.6 deferrals).

### 5. Storage of params (only if Q2 = configurable)

- If Q2 = **defaults-only ‚úÖ recommended-if-spec-allows**: nothing to store ‚Äî
  `TotpParams::DEFAULT` is implied; **no `payload_version` bump, no
  `pangolin-store` schema change**. `validate::totp_secret` stays length-only.
  The smallest, safest path; matches the spec (which only mentions "the 6-digit
  code") and is fully RFC-6238-compliant for the SHA-1/6/30 case the vast
  majority of users need.
- If Q2 = **configurable**: extend the `AccountIdentity` V1 CBOR body to carry
  `totp_algorithm: u8`, `totp_digits: u8`, `totp_period_secs: u32` alongside
  the `totp_secret` blob (absent ‚áí defaults, so old vaults read fine ‚Äî
  *backward*-compatible, no `payload_version` bump needed for the read path;
  but a *write* with non-default params produces a body an older Pangolin
  would parse with the defaults, silently using the wrong params ‚Üí arguably
  needs a `payload_version` bump per 1.6's ¬ß18.7 "reject unknown future
  version" so an old build refuses rather than computes wrong codes). This is a
  real format change ‚Üí routed through 1.6's machinery. **Recommend avoiding it
  in MVP-1** unless the spec is later amended to require configurability.

## Success criteria

1. `pangolin-totp::totp_at` reproduces the **RFC 6238 Appendix B test vectors
   exactly** for SHA-1 (and SHA-256 / SHA-512 if Q2 = configurable), at all six
   listed `T` values, 8-digit codes ‚Äî **the non-negotiable correctness test.**
2. `parse_otpauth_uri` parses a well-formed `otpauth://totp/...?secret=...&algorithm=...&digits=...&period=...`
   into the correct `{ secret_bytes, params, label, issuer }`; defaults applied
   when params absent; `hotp://`, missing `secret=`, unknown `algorithm`,
   non-base32 secret ‚Üí typed `TotpError` (no panic).
3. `decode_base32` round-trips RFC 4648 test vectors; case-insensitive; tolerates
   padding and embedded spaces; rejects characters outside `[A-Za-z2-7 =]`.
4. `totp_generate` end-to-end: set a seed on an account via `account_update`
   (passing bytes derived from `parse_totp_secret`), then `totp_generate(account,
   at=59)` returns the RFC test-vector code (using the RFC test secret);
   `seconds_remaining == 30 - (59 % 30) == 1`; an account with no seed ‚Üí
   typed "no TOTP" error; a locked vault ‚Üí `NotUnlocked`.
5. Window boundary: `totp_generate(at=29)` and `totp_generate(at=30)` differ;
   `totp_generate(at=0)` is window 0; `totp_generate(at=29)` and
   `totp_generate(at=0)` agree (same window).
6. `reveal_totp_secret` (1.4) still returns the raw seed under a fresh presence
   proof ‚Äî and now there is a meaningful seed to reveal (one set via the parsed
   path round-trips byte-identically through `reveal_totp_secret`).
7. Zeroization: `TotpCode` has a redacting `Debug` and zeroizes its `code` on
   drop; intermediate HMAC output and seed copies in `pangolin-totp` are
   `Zeroizing`. The `pangolin-store` `no_plaintext_on_disk` proptest still
   passes (it already scans all `SecretBytes` fields incl. `totp_secret`; extend
   the corpus to include a base32-decoded seed if it doesn't already).
8. `forbid(unsafe_code)` retained on `pangolin-totp`; `cargo geiger` / a `grep`
   for `unsafe` in `crates/pangolin-totp/src/` is empty.
9. `cargo tree -p pangolin-crypto | grep -ci serde == 0`;
   `cargo tree -p pangolin-core | grep -ci uniffi == 0`;
   `cargo tree -p pangolin-store | grep -ci uniffi == 0` (all unchanged ‚Äî
   `pangolin-totp`'s new deps are downstream of `pangolin-ffi`, not these).
10. `cargo deny check` passes (no `deny.toml` change; new deps `=`-pinned);
    `cargo audit` (with the existing `--ignore` set) reports no new advisory
    against `sha1` / `hmac`.
11. Existing PoC `.pvf` vaults open, unlock, `account_history`; if one carries a
    `totp_secret`, `totp_generate` produces a code or a clean typed error ‚Äî no
    panic, no migration failure.
12. `cargo test --workspace --all-targets` green in debug; no hard timing
    assertions added (env-quirk #11); the `pangolin-store` e2e suite still
    passes.

## Failure modes considered

- **Wrong RFC 6238 output** (off-by-one on the counter, big-vs-little-endian,
  truncation-offset bug, integer overflow in `10^digits` for 8 digits =
  100,000,000 < `u32::MAX` so OK but assert it). *Mitigation:* the Appendix B
  vectors are exact and exhaustive over the algorithm/time axes ‚Äî a single
  wrong byte fails the suite. SHA-256/512 vectors guard the algorithm dispatch.
- **Seed leaks via `Debug` / logs / panics.** `TotpCode` and `TotpError` have
  redacting `Debug`; the engine never `format!`s the seed; `totp_at` returns
  `Result`, never panics on a malformed seed (any non-empty bytes are a valid
  HMAC key). *Mitigation:* a `Debug`-output test; `forbid(unsafe_code)`.
- **`sha1` crate advisory / supply-chain.** `cargo audit` scans it; the
  `=`-pin + `Cargo.lock` commit make the version explicit; if an advisory lands,
  the `--ignore` escape hatch (env-quirk #3) or a version bump. SHA-1's
  collision weakness does not affect HMAC-SHA1 (no known practical attack).
- **HIGH-1 regression** ‚Äî accidentally making `pangolin-crypto` depend on
  `pangolin-totp` (e.g. if Q1 = option (a) and someone wires it backwards).
  *Mitigation:* the dep arrow is one-way by design; the `cargo tree` grep in
  CI catches it; Q1-recommendation (option (b)) keeps `pangolin-crypto`
  untouched entirely.
- **Clock skew on the device.** `totp_generate` takes `at` from the shell; if
  the device clock is wrong, the code is wrong ‚Äî that is inherent to TOTP and
  not Pangolin's to fix. The `seconds_remaining` field lets the UI warn "expires
  in 2 s ‚Äî wait for the next one". *Mitigation:* documented in `totp.md`; the
  master plan's "time-drift handling" for a *generator* (vs a validator) is
  exactly this countdown surfacing ‚Äî no T¬±1 window logic (that's for validators,
  which Pangolin is not).
- **PoC vault with opaque `totp_secret` bytes** (the PoC stored whatever was
  typed). *Mitigation:* `totp_at` works on any non-empty bytes; an empty PoC
  seed ‚Üí typed "no TOTP" error; no migration. Tested.
- **`account_update` with a `totp_secret` that wasn't parsed** (a caller passes
  raw garbage). *Mitigation:* 1.2's `validate::totp_secret` (length ‚â§ 256) still
  runs; the engine tolerates any bytes; the only consequence is wrong codes ‚Äî
  the *recommended* path is "shell calls `parse_totp_secret` first", documented.
- **base32 hand-roll bug** (alphabet typo, bit-packing error). *Mitigation:*
  RFC 4648 test vectors; a property test (encode-then-decode is identity for the
  subset we accept). If Q4 = use a crate, this risk moves to the crate's audit.

## Open questions for Kelvin

1. **HMAC-SHA1 source.** (a) Add `hmac_sha1` to `pangolin-crypto`'s public
   surface (one HMAC home; widens `pangolin-crypto`'s allowlist + pulls `sha1`
   into its tree); or **(b) ‚Äî recommended** ‚Äî `pangolin-totp` pulls its own
   `hmac` + `sha1` (+ `sha2` if Q2-configurable), blast-contained, exactly the
   master plan ¬ß16.8 rationale; `pangolin-crypto` untouched; HIGH-1 preserved by
   construction; **no `deny.toml` change** (no allowlist; new deps just need
   `=`-pins + a `Cargo.lock` commit). Recommend **(b)**.
2. **Configurable TOTP params.** (a) Support the full set ‚Äî `algorithm ‚àà
   {SHA1, SHA256, SHA512}`, `digits ‚àà {6,7,8}`, configurable `period` ‚Äî
   maximally compatible with real authenticators (Steam, some banks use 8
   digits / SHA-256 / non-30 periods), but the params need durable storage
   alongside `totp_secret` ‚Üí an `AccountIdentity` CBOR body extension (likely a
   `payload_version` bump via 1.6's ¬ß18.7 machinery so an old build refuses
   rather than silently uses defaults); or **(b) ‚Äî recommended-if-spec-permits**
   ‚Äî RFC-6238 defaults only (SHA-1 / 6 / 30): zero storage change, zero schema
   churn, fully RFC-compliant for the case ~95% of users need; the spec only
   mentions "the 6-digit code" and is silent on configurability. Recommend
   **(b)** for MVP-1; revisit if the spec is amended. *(If you want the
   middle ground: accept the full set on `parse_otpauth_uri` but **store only
   SHA-1/6/30**, rejecting non-default-param URIs with a "not supported in
   MVP-1" error ‚Äî keeps the parser future-proof without the storage change.)*
3. **Generating a TOTP code: access class.** (a) session-class ‚Äî needs only an
   unlocked, non-expired session, **no presence proof** (matches the 1.1-frozen
   `totp_generate` signature, which has no presence param, and the TOTP spec's
   stated default: "Copy TOTP code ‚Ä¶ Active session must exist; no extra
   presence for each code"); or (b) reveal-class ‚Äî fresh presence proof every
   time (matches the Phase-2 "paranoid mode", and would require growing the
   1.1-frozen signature with a `presence` parameter ‚Äî a surface amendment).
   Recommend **(a)** ‚Äî the code is the ephemeral user-facing artifact; the
   durable *seed* is the reveal-class secret and is already 1.4-gated via
   `reveal_totp_secret`. The Phase-2 per-account "paranoid mode" is out of
   MVP-1 scope.
4. **base32 decoding.** (a) Hand-roll a ~30-line RFC 4648 base32 decoder in
   `pangolin-totp` (no new dep); or (b) pull a tiny audited crate
   (`data-encoding = "=2.x"` ‚Äî pure Rust, no FFI, MIT/Apache, widely used).
   Recommend **(a)** ‚Äî base32 decode is trivial and well-specified; fewer deps
   = smaller audit surface = consistent with the "minimal deps" posture; the
   `otpauth://` parser is hand-rolled for the same reason. (If you'd rather not
   own a hand-rolled crypto-adjacent decoder, **(b)** is fine ‚Äî it's tiny.)
5. **`pangolin-cli account totp <id>` subcommand in 1.7?** (a) Yes ‚Äî a
   generate-and-print subcommand now; or **(b) ‚Äî recommended** ‚Äî no new
   subcommand; 1.7 only *fixes* the existing `--totp-stdin` / interactive
   prompt to base32-decode the input (closing the "prompt says base32, stores
   verbatim" gap), and the `account totp` generate command rides the CLI-V1
   issue alongside `account` / `reveal` / `device` (consistent with the
   standing CLI-V1 deferral that 1.2‚Äì1.6 followed). Recommend **(b)**.

## Out of scope (1.9 / MVP-2+ owns)

- **KDBX import of TOTP entries** ‚Äî 1.9 owns the `.kdbx` parser; it will *call*
  `pangolin-totp::parse_otpauth_uri` (KDBX entries carry TOTP as an `otpauth://`
  string or `TimeOtp-*` attributes), but the KDBX-side plumbing is 1.9's.
- **TOTP code *validation*** ‚Äî Pangolin *generates* codes for the user to type
  into sites; it never validates an incoming code (no T¬±1 drift-window check ‚Äî
  that's a server-side concern Pangolin isn't).
- **HOTP** (`otpauth://hotp/...`, counter-based) ‚Äî `parse_otpauth_uri` rejects
  it; HOTP support is not in any Pangolin spec.
- **Passkeys / WebAuthn** ‚Äî a "future auth method" per the feature list, not
  MVP-1.
- **QR-code rendering of the seed** ‚Äî the TOTP spec mentions "Display QR code"
  as a high-risk action, but rendering a QR image is a shell/UI concern, not
  `pangolin-totp`'s; the engine exposes the bytes/URI, the shell draws the QR.
- **Per-account "paranoid mode" (presence-per-code)** ‚Äî Phase-2 refinement;
  out of MVP-1 (see Q3).
- **Configurable-param storage** ‚Äî if Q2 = (b), this is deferred entirely;
  if Q2 = (a), the schema extension lands in 1.7 via 1.6's ¬ß18.7 machinery.
- **Clipboard auto-clear timeout for copied codes** ‚Äî a shell/UI concern, not
  the engine's.
