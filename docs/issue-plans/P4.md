# Issue P4: Session policy engine

> **Status:** PLAN-gate. **Security-critical** per §16.3 (touches session policy — explicitly listed). Per the autonomy directive, dispatching the build directly; security audit at SIGNOFF will catch any drift from the plan.
> **Sequencing:** Builds on P3's revision graph; both consume P2's `pangolin-store`.

## Spec reference

- **Master plan §0 cardinal principle 5:** "Session invariant is universal. Start = 2 proofs (presence + identity). Maintain = 1 proof. Expired = 2 proofs again. High-risk actions require explicit presence even mid-session."
- **Whitepaper §5 (Authority & Access Hierarchy):** the four-layer model — social recovery → ownership → session → operation.
- **Unified Session Authority spec** (`Pangolin.pdf` §4101–4419, full reading required):
  - §4 Session States: `Active`, `Expired`, `PendingAuthorization`
  - §5 Session Rules: start (2 proofs), active (1 proof), expiry (auto-revoke), high-risk (presence escalation)
  - §6 Hardware Integration: NFC primary, platform authenticator mandatory, USB optional
  - §7 Session Timing & Lifecycle: 15-min default idle, 4-hour absolute max, 30–60s presence freshness, ~60s prompt timeout
  - §8 Prompt & Interruption Behavior: never stack, auto-resume after success
- **Master plan §3.7 EPIC: Session Policy (P4):** P4-1..P4-6.

## What P2 already shipped

P2's `Vault` has a 2-state machine (`Locked ⇄ Active`) where:
- `unlock(password)` enters `Active` (single password proof; instantaneous; no expiry)
- `lock()` zeroizes the cache and returns to `Locked`
- All credential ops require `Active`

P4 adds the full session-policy spec on top of this minimal state machine.

## Goal

Implement the session policy engine per the Unified Session Authority spec — 2-proof start, 1-proof maintain, auto-expiry on idle + absolute max, presence escalation for high-risk operations, and the mid-action prompt/resume primitive.

For PoC the **proofs are stand-ins** (PIN-string for identity proof, "press-y" prompt for presence proof) but the trait-based abstraction is real and slots in real proofs in MVP-1 (NFC, platform passkey, biometric) without API changes.

After this issue:
- `Vault::unlock(presence: PresenceProof, identity: IdentityProof)` requires both proofs (not just a password).
- Session has a real `expires_at` timestamp; `Vault::is_session_active()` and `Vault::session_remaining()` expose state.
- Idle timer fires automatically on `add_account` / `update_account` / `delete_account` / `get_account` / `search` / etc.
- High-risk operations (`reveal_password`, `export_payload`, `change_recovery` placeholder) require an explicit fresh presence proof EVEN during an active session.
- Mid-action resume primitive: a closure-based API that wraps a write so an expired session triggers reauth and the write resumes after success.

## Approach

### Proof traits

```rust
pub trait PresenceProof {
    /// Verifies the presence proof. Returns Ok(()) on success.
    /// Implementations: PoC PIN dialog stand-in; MVP-1 NFC tap; etc.
    fn verify(&self) -> Result<(), AuthError>;
    /// How long this proof is "fresh." 30-60s for NFC; effectively
    /// instantaneous for PIN.
    fn freshness(&self) -> Duration;
}

pub trait IdentityProof {
    fn verify(&self) -> Result<(), AuthError>;
    fn freshness(&self) -> Duration;
}
```

PoC implementations:
- `PinIdentityProof(String)` — wraps a string; verification compares constant-time against a stored salted hash kept in `meta` table. (Adds a `pin_hash` column to meta or `pin_check.salted_hash` on a separate small row.)
- `PressYPresenceProof` — a literal "the test/CLI confirms y was pressed" trait impl. For tests it's a static `Confirmed` value; for CLI it prompts.

For PoC, both proofs contain the actual derived password material — `PinIdentityProof` IS the password (the user types it, we hash + verify). `PressYPresenceProof` is purely a confirmation step (no key material). The 2-proof requirement at unlock-time requires the user to (1) type the password AND (2) confirm the prompt.

### Session state machine

```rust
pub enum SessionState {
    /// Vault is locked; no plaintext in memory.
    Locked,
    /// Awaiting one or both proofs. Cache is zeroized; this is a UX
    /// state, not a security state.
    PendingAuthorization {
        why: AuthReason,
        original_action: Option<DeferredAction>,
    },
    /// Both proofs satisfied; cache live; expires at `expires_at`.
    Active {
        expires_at: SystemTime,
        last_proof_at: SystemTime,
    },
    /// Was Active; idle timer fired or absolute max hit. Cache zeroized
    /// at the moment of expiry. Next op needs full 2-proof unlock.
    Expired,
}
```

### Timing constants (per Session spec §7)

```rust
pub const IDLE_TIMEOUT_DEFAULT: Duration = Duration::from_secs(15 * 60); // 15 min
pub const ABSOLUTE_MAX_DEFAULT: Duration = Duration::from_secs(4 * 3600); // 4 hours
pub const PRESENCE_FRESHNESS: Duration = Duration::from_secs(60); // upper bound; spec says 30-60s
pub const PROMPT_TIMEOUT: Duration = Duration::from_secs(60);
```

These are the locked defaults. PoC does not implement user-configurable session duration (deferred to MVP-1 issue 1.4 per master plan).

### Unlock flow

```rust
impl Vault {
    pub fn unlock(
        &mut self,
        presence: &dyn PresenceProof,
        identity: &dyn IdentityProof,
    ) -> Result<()> {
        // Both proofs must verify successfully (order doesn't matter).
        presence.verify()?;
        identity.verify()?;
        // Identity proof IS the password derivation in PoC — extract it.
        let password = identity.password_bytes()?;
        // The existing pangolin-crypto unlock path takes over from here:
        let seed = kdf::derive_seed(&password, &self.meta.kdf_salt, &self.meta.kdf_params)?;
        let authority = AuthorityKey::from_seed(*seed);
        let wrapped = self.meta.wrapped_vdk();
        let vdk = wrapped.unwrap_with(&authority)?;
        let cache = build_decrypted_cache(&self.conn, &self.meta, vdk.aead_key())?;

        // Set session expiry.
        let now = SystemTime::now();
        let expires_at = now + IDLE_TIMEOUT_DEFAULT;
        self.session_state = SessionState::Active { expires_at, last_proof_at: now };
        self.active = Some(ActiveState { vdk, cache });
        Ok(())
    }
}
```

### Idle timer + absolute max

Every public credential op (`add_account`, `update_account`, `delete_account`, `get_account`, `search`, `list_accounts`, `revisions_for`, etc.) calls a `check_session_freshness()` helper at the top:

```rust
fn check_session_freshness(&mut self) -> Result<()> {
    match &self.session_state {
        SessionState::Active { expires_at, .. } => {
            if SystemTime::now() > *expires_at {
                self.lock();
                Err(StoreError::SessionExpired)
            } else {
                Ok(())
            }
        }
        SessionState::Locked => Err(StoreError::NotUnlocked),
        SessionState::PendingAuthorization { .. } => Err(StoreError::SessionPending),
        SessionState::Expired => Err(StoreError::SessionExpired),
    }
}

fn touch_session(&mut self) {
    if let SessionState::Active { last_proof_at, expires_at, .. } = &mut self.session_state {
        let now = SystemTime::now();
        *last_proof_at = now;
        *expires_at = now + IDLE_TIMEOUT_DEFAULT;
        // But never beyond the absolute max from session_started.
        // Track session_started_at; cap expires_at at session_started_at + ABSOLUTE_MAX_DEFAULT.
    }
}
```

`check_session_freshness` is called at the top of every credential op. `touch_session` updates the idle deadline at the bottom of each successful op.

### Presence escalation for high-risk ops

```rust
impl Vault {
    /// Reveal the plaintext password for an account. Requires a fresh
    /// presence proof EVEN if the session is active.
    pub fn reveal_password(
        &mut self,
        id: AccountId,
        presence: &dyn PresenceProof,
    ) -> Result<SecretBytes> {
        self.check_session_freshness()?;
        presence.verify()?;
        // ... return the password from the cache
    }

    /// Export an encrypted payload (for future key migration / backup).
    /// Requires fresh presence proof.
    pub fn export_payload(
        &mut self,
        id: AccountId,
        presence: &dyn PresenceProof,
    ) -> Result<Vec<u8>>;
}
```

The "fresh" check verifies the presence proof's `freshness()` window has not elapsed since its production. For PoC where presence is `PressYPresenceProof::Confirmed`, the freshness window is implicit (the proof was produced at the moment of construction, immediately consumed). For real NFC in MVP-1, the proof would carry a timestamp and the verify path would assert `now - proof.timestamp < freshness()`.

### Mid-action resume primitive

```rust
impl Vault {
    /// Run `op` against the vault. If the session is `Expired`, prompt
    /// for re-auth via the supplied callback, then resume the operation.
    /// Mirrors Session spec §8.5 (mid-action expiration semantics).
    pub fn with_session<F, T>(
        &mut self,
        op: F,
        reauth: impl FnOnce(&mut Self) -> Result<()>,
    ) -> Result<T>
    where
        F: FnOnce(&mut Self) -> Result<T>,
    {
        match self.check_session_freshness() {
            Ok(()) => op(self),
            Err(StoreError::SessionExpired) => {
                reauth(self)?;
                op(self)
            }
            Err(e) => Err(e),
        }
    }
}
```

The PoC tests use this to verify mid-action resume; real callers (MVP-4 Tauri shell, MVP-5 mobile shells) wrap this with their UI prompt code.

## Success criteria

1. Crate compiles, fmt + clippy + audit + deny stay clean.
2. **2-proof unlock requires both:** unlock with valid presence + invalid identity → `AuthenticationFailed`; unlock with invalid presence + valid identity → `AuthenticationFailed`; unlock with both valid → `Active`.
3. **Session expires after idle timeout:** unlock, advance clock past `IDLE_TIMEOUT_DEFAULT`, attempt operation → `SessionExpired`; vault auto-locks; cache is zeroized.
4. **Session expires at absolute max even with activity:** unlock, perform ops every minute for 4+ hours' worth of simulated time → after `ABSOLUTE_MAX_DEFAULT`, even an active session expires.
5. **Touch session extends idle deadline:** unlock, do op, advance clock 14 min, do op (succeeds), advance another 14 min, do op (succeeds — total 28 min from unlock).
6. **Reveal password requires fresh presence:** active session + `reveal_password` without presence → `PresenceProofRequired`. Active session + valid presence → returns the plaintext.
7. **Export payload requires fresh presence:** same shape as #6.
8. **Mid-action resume primitive:** `with_session(op, reauth)` where session is expired → reauth callback runs → op runs and returns its T.
9. **Session state observable:** `is_session_active()` and `session_remaining()` reflect the current state.
10. **Cache zeroized on expiry:** unlock + add account + advance clock past idle → next op surfaces `SessionExpired` AND the in-memory cache is gone (lock_zeroizes_cache test pattern from P2 still passes).

## Failure modes considered

| Failure mode | Mitigation |
|---|---|
| Clock skew (system time changes mid-session) | Use `SystemTime::now()` consistently; document that backwards clock changes can extend or expire sessions arbitrarily. PoC accepts this; MVP-1 may switch to `Instant` (monotonic). |
| Presence proof replay (attacker re-uses a prior `Confirmed`) | The PoC `PressYPresenceProof::Confirmed` is a single-use type; `verify()` consumes it. Real NFC in MVP-1 will carry a nonce + timestamp. |
| Concurrent unlock attempts | `&mut self` on `unlock` ensures Rust's borrow checker prevents concurrent calls. |
| Reveal password while session is expired | `check_session_freshness` runs first; returns `SessionExpired` before the presence proof is even validated. Caller is informed clearly. |
| `with_session` reauth callback returns `Err` | Op is NOT run; the original error is propagated. |

## Out of scope

- **Configurable session duration** (5 min / 15 / 30 / 60 / 240 / until-lock from session spec §7.2). MVP-1 issue 1.4. PoC uses the locked default 15 min idle / 4 hour max.
- **Hardware integration** (NFC, platform authenticator). MVP-1 issues 1.5, 8.x.
- **Phone-as-presence-bridge** (Session spec §6.3). Phase 2.
- **TOTP integration with session model.** MVP-1 issue 1.7.
- **Session-state persistence across vault closes.** Session is in-memory only by design (Session spec §2.3: "MUST NOT be transferable between devices, MUST NOT be stored on-chain" — by extension, also not on disk in PoC). Closing the vault always returns to `Locked`.
