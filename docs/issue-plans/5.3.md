# Issue 5.3: Conflict detection + resolution plumbing — FFI surfacing + pull-tick signal

> **One-line scope:** the explicit conflict-detection + UI-surfacing plumbing that the existing 1.6 + P8 + P9 machinery has built up but never exposed at the host-facing layer. Today: `Vault::list_conflicts` (P9-2), `Vault::account_status` (1.6 R-q), `Vault::resolve_fork` (1.6 R-c), `Vault::clear_frozen` (P9-1), `Vault::account_fork_branches` (1.6 R-x), `Vault::list_frozen_accounts` (P8), the `frozen_pending_resolve` column + auto-set on the genuine-foreign-INSERT path of `ingest_chain_revision`, and the `superseded_by` column + non-superseded-head detection ALL EXIST. What 5.3 ships: the FFI binding for `list_conflicts`, a per-pull-tick conflict-diff signal in `PullReport` (5.2) so the host scheduler reacts within one tick, and a thin `Vault::list_conflicts_since(prior_snapshot)` diff accessor for the 5.4 indicator. Master plan §5 row 5.3 verbatim: "Core reports conflicts; UI consumes via FFI; publish-resolution path produces canonical head" — every comma-separated clause is a question for Kelvin (see Q-a..Q-g top of doc); plan-gate's reading routes the load-bearing work to the FFI binding + the `PullReport` extension.

## Open questions for Kelvin (LOAD-BEARING — surface at TOP per §16)

### Q-a · Auto-freeze on ingest-time fork detection — confirm existing behavior is the spec

The 4.1 `ingest_chain_revision` path (called by `ingest_pending_chain_revision`) sets `frozen_pending_resolve=1` on the genuine-foreign-INSERT path (P8 CRIT-1) — this fires whenever a foreign event creates a new local revision row that did NOT match any idempotency arm.

**Option A: Confirm — the existing freeze is the spec.** 5.3 ships NO change to `ingest_chain_revision`. The freeze fires today; 5.3's job is to SURFACE it (FFI binding + `PullReport` extension). Pro: smallest 5.3 footprint; no risk of regression on a heavily-tested path. Con: a thoughtful audit may notice that "frozen on ANY foreign INSERT" is broader than "frozen on a NEW LEAF that extends an existing leaf" — the existing trigger freezes even when the foreign event is a brand-new account that the local store had never seen.

**Option B: Tighten — only freeze on TRUE fork.** 5.3 modifies `ingest_chain_revision` so the freeze fires only when the new row's parent is the same as an existing leaf's revision_id (= a true fork), not when the new row is a fresh-account first-revision. Pro: more precise UX. Con: regresses P9 §A4 + existing tests; large blast radius.

**Plan-gate recommends:** Option A. Confirm existing behavior. P9 already shipped + audited the broader trigger; tightening is a UX refinement deferred to MVP-3 user testing.

### Q-b · Self-publish loopback — confirm the existing idempotency arms catch own-device

When this device publishes revision R via 5.1 flush, the next 5.2 pull-tick ingests R from chain. `ingest_chain_revision`'s idempotency arms should detect "I already have a row with this canonical hash" and take a content-merge path (stamp the anchor; do NOT freeze).

**Option A: Confirm — 5.1's flush stamps the local row's anchor inline; 5.2's pull-tick sees idempotency arm #1 (exact-hash match) and is a no-op.** 5.3 ships NO change. Pro: smallest footprint. Con: assumes 5.1 flush actually does the inline-anchor-stamp (CONFIRM during BUILD).

**Option B: Add a defensive "this device published this; do not freeze" signal.** A `(account_id, revision_id) -> just_published_at` in-memory set on `ActiveState` populated by 5.1 flush + checked by 5.2 ingest. Pro: belt-and-suspenders. Con: state synchronization surface; in-memory set won't survive a process restart.

**Plan-gate recommends:** Option A, but ADD a hermetic regression test `pull_after_local_publish_does_not_self_freeze`. If the test fails, escalate to Option B.

### Q-c · PullReport extension — per-tick conflict-diff signal

5.2's `PullReport` carries `{ mode, sync_report, pulled_at_unix_ms }`. 5.3 needs to surface "this tick CREATED N new conflicts" so the 5.2 host scheduler can react within one tick.

**Option A: Extend `PullReport` with `newly_frozen_accounts: Vec<AccountId>` + `newly_forked_accounts: Vec<AccountId>` + `newly_resolved_accounts: Vec<AccountId>`.** Computed in `pull_once`: snapshot `list_frozen_accounts()` ∪ `all_forked_accounts()` BEFORE the dispatch, snapshot AGAIN after, return the set difference. Pro: single-call surface; host scheduler gets per-tick deltas; trivial to implement. Con: two extra `list_*` queries per tick (cheap — O(N-conflicted)).

**Option B: Add a separate `Vault::list_conflicts_since(prior_snapshot: ConflictSnapshot) -> ConflictDelta` accessor.** Host snapshots before + calls after. Pro: orthogonal to `PullReport`. Con: extra round-trip; host has to remember.

**Option C: Both.** Doubled surface.

**Plan-gate recommends:** Option A. The 5.2 host scheduler is the canonical consumer; baking the delta into `PullReport` is the obvious shape.

### Q-d · ConflictReport shape — enrich, or keep minimal?

Today `ConflictReport { account_id, heads: Vec<RevisionId>, frozen: bool }`. The host UI rendering a conflict-resolution screen needs per-branch metadata: device_id, observed_at_block, parent, schema_version, is_tombstone. The minimal report forces the host to make N extra round-trips.

**Option A: Enrich `ConflictReport`.** Replace `heads: Vec<RevisionId>` with `branches: Vec<ConflictBranchSummary { revision_id, parent, device_id, observed_at_block, schema_version, is_tombstone, on_canonical_chain }>` (mirror `FfiRevisionMeta` shape). Pro: single round-trip. Con: breaking change to existing `ConflictReport` callers (only `pangolin-cli resolve` today + tests).

**Option B: Keep minimal + add `Vault::describe_branch(account_id, revision_id) -> ConflictBranchSummary` accessor.** Pro: no breaking change. Con: N+1 round-trips.

**Option C: Both — keep `heads` + add ADDITIVE `branches_summary: Option<Vec<ConflictBranchSummary>>` via opt-in `list_conflicts_enriched()`.** Pro: no break AND single round-trip when wanted. Con: doubled accessor.

**Plan-gate recommends:** Option A. The existing callers are pangolin-cli resolve + tests; breaking change is bounded; FFI binding ships Option A's shape from day one.

### Q-e · FFI binding — `vault_list_conflicts` in 5.3, or defer to CLI-V1?

`account_status` + `account_fork_branches` + `account_resolve_fork` are already in `pangolin-ffi/src/revision.rs`. `list_conflicts` is NOT.

**Option A: Ship FFI in 5.3.** New `pub fn vault_list_conflicts(handle) -> Result<Vec<FfiConflictReport>, FfiError>`. Pro: closes the "UI consumes via FFI" clause of master plan §5 row 5.3 verbatim. Con: extends FFI surface; UniFFI bindings need regen.

**Option B: Defer FFI to CLI-V1 batch.** 5.3 ships Rust-side only. Pro: smaller 5.3. Con: master plan literally says "UI consumes via FFI" — deferring deflects the spec.

**Plan-gate recommends:** Option A. The master plan text is unambiguous on this one.

### Q-f · Auto-resolve heuristics — defer or ship?

P9 §"Out of scope" + Cardinal Principle 4 ("never silent merge") explicitly reject auto-resolve.

**Option A: Defer.** No auto-resolve in 5.3.
**Option B: Ship byte-identical auto-resolve.** Violates Cardinal Principle 4.
**Option C: Opt-in flag.**

**Plan-gate recommends:** Option A. Cardinal Principle 4 is bright-line.

## Adversarial framing — L-section (load-bearing risks)

### L-self-fork-on-publish

**What goes wrong:** this device publishes revision R via 5.1 flush; next 5.2 pull-tick ingests R; existing P8 CRIT-1 trigger fires + freezes the account; user sees a "conflict" badge on their own account.

**Mitigation:** Q-b Option A — 5.1 flush stamps anchor inline; 5.2 ingest sees idempotency arm #1 (exact-hash) + is a no-op. **Regression test mandatory:** `pull_after_local_publish_does_not_self_freeze`. If fails, escalate to Q-b Option B.

### L-conflict-surface-races-with-resolve

**What goes wrong:** 5.2 pull-tick's `PullReport.newly_frozen_accounts` includes account A; user races a `resolve_fork` call before the host scheduler updates.

**Mitigation:** structurally impossible to corrupt — both `resolve_fork` and `list_conflicts` take `&mut self` / `&self`; Rust's borrow checker sequences them.

### L-list-conflicts-perf-on-N-1000-frozen

**What goes wrong:** under malicious-RPC, N>1000 accounts frozen; `list_conflicts` becomes a UI hang.

**Mitigation:** existing impl is single-query union with HashSet dedup; indexed lookups. Perf check during BUILD: `list_conflicts_perf_with_1000_forked_accounts` <100ms on release.

### L-byte-flip-on-frozen-row-via-FFI

**What goes wrong:** tampered local DB row (byte-flipped `is_tombstone`) surfaces wrong metadata to host UI without authentication.

**Mitigation:** L2 — `list_conflicts` does NOT call any AEAD-open path; per-row metadata is best-effort + advisory. Load-bearing authentication is at AEAD-open call sites; host UI rendering is advisory.

### L-PullReport-delta-overcounts-on-existing-frozen

**What goes wrong:** Q-c Option A computes delta as `(after-set) - (before-set)`. An already-frozen carry-over from previous tick shouldn't appear in `newly_frozen_accounts`.

**Mitigation:** set-difference is correctly directional. Test `pull_tick_does_not_re_report_already_frozen_account` covers it.

### L-self-resolve-loopback

**What goes wrong:** user resolves account A; 5.1 flush publishes merge revision M; 5.2 pull ingests M; existing trigger RE-freezes A.

**Mitigation:** P9 §A3 race already addressed — `resolve_fork` stamps anchor inline + clears freeze atomically; 5.2 ingest sees idempotency arm #1 + is a no-op. Same regression test covers this.

### L-conflict-surface-leaks-frozen-payload

**What goes wrong:** Q-d Option A enriched `ConflictReport` carries metadata per branch; could it leak anything sensitive?

**Mitigation:** all fields are metadata-class (already exposed via `FfiRevisionMeta`). Freeze guard refuses PLAINTEXT reveal, not metadata. No leak.
