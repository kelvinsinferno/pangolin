# Issue 1.2: Account identity model — production

> **Status:** PLAN approved 2026-05-08. Q1-Q5 locked below.
> **Security-critical:** YES — owns the in-memory shape of every plaintext
> credential, the encrypted-blob payload format, and the per-update
> revision-creation contract that 1.6's lineage scheduler will ride on.
> **Depends on:** issue 1.1 (FFI surface frozen on `fd8d8f7`).

## Decisions locked (post plan-gate)

| Q | Decision | Rationale |
|---|---|---|
| Q1 — FFI shape contradiction | **Amend 1.1 FFI shapes inside 1.2.** Widen `AccountDraft`/`AccountPatch`/`AccountSnapshot` in `pangolin-ffi/src/identity.rs` to multi-username (`Vec<String>`), multi-URL (`Vec<String>`, validated via `url` crate), `tags: Vec<String>`, `notes: Option<String>`, `totp_secret: Option<TotpSecret>` slot, `password_history: Vec<PasswordHistoryEntry>`. Land in same merge as 1.2. | No external bindings consume the 1.1 shapes yet; the amendment is safe and avoids a separate §16 cycle. Master plan §1.2 row mandates these fields. |
| Q2 — Type placement | **Continue 1.1's Q2 pattern.** Rewrite production `AccountIdentity` in `pangolin-store::account`; `pangolin-core` re-exports. Physical relocation to `pangolin-core::identity` deferred to 1.4's session rewrite. | Smallest 1.2 diff; sets up cleanly for 1.4 batch-relocation. |
| Q3 — URL scheme policy | **Any scheme via the `url` crate.** Accept any RFC 3986 URL. | Needed for KDBX imports + app URIs (mailto:, app://, ssh://, etc.). |
| Q4 — Schema-version mismatch handling | **Accept-and-record; defer reject policy to 1.6.** Read the `schema_version` field on stored payloads, record it on the snapshot, but never reject for being newer/older. | Forward-compat policy lives in §18.7 / 1.6; 1.2 is data-model-only. |
| Q5 — Password history cap + dedup | **Neither.** Append every change; never prune; never deduplicate consecutive identical passwords. | Spec doesn't ask for either; storage cost is negligible; full audit trail preserved. |

## Spec reference

## Goal

After this issue closes:

1. The production `AccountIdentity` model is the single in-memory
   plaintext-bearing shape for an account. It carries display name,
   tags, notes, **multiple** URLs, **multiple** usernames, a
   **password history** (each entry timestamped with the originating
   device id), and a **TOTP secret slot** (the slot exists; the actual
   RFC-6238 generator lands in 1.7).
2. The five FFI entry points `account_add` / `account_update` /
   `account_search` / `account_get` / `account_history` have working
   bodies that route through `pangolin_core::Vault` (re-exported from
   `pangolin-store` per Q2 of 1.1) and produce on-disk encrypted
   revisions. `todo!()` panics are gone.
3. Every update produces a new revision row pointed at the previous
   head — the contract that 1.6's lineage scheduler relies on. Password
   rotations append to the in-payload password history; the previous
   password value is preserved (encrypted) inside the new revision's
   payload, not erased.
4. The CBOR encoding of the encrypted blob is widened to the new shape
   in a forward-compatible way: a `payload_version: u8` discriminator
   distinguishes the legacy P2-era 6-field flat snapshot from the new
   1.2-era extended snapshot, and decode handles both. The 1.6
   schema-versioning policy will lock the read-only-old / reject-future
   semantics; 1.2 commits to the *slot* with both versions decodable.
5. Validation rejects malformed drafts at the public-API boundary with
   `pangolin_core::Error::Validation { kind, message }` mappings to
   `FfiError::Validation` per the §18.8 taxonomy.
6. **HIGH-1** (`cargo tree -p pangolin-crypto | grep -ci serde == 0`)
   and **Q3** (`cargo tree -p pangolin-core | grep -ci uniffi == 0`)
   remain at zero. `forbid(unsafe_code)` survives on every crate
   except `pangolin-ffi`.

## Approach

### A. Type model

The production `AccountIdentity` lives at
`pangolin-store::account::AccountIdentity` (per **Q2 below**, recommend
**continuing** the 1.1 re-export pattern; the physical relocation to
`pangolin-core::identity::*` is still planned for 1.4 alongside the
session rewrite).

```rust
// crates/pangolin-store/src/account.rs (rewrite)
//
// PoC AccountSnapshot (6 flat SecretBytes fields) is renamed to
// AccountIdentityV0 internally and kept reachable for legacy-payload
// decode only. Public type is the new V1 shape below.
//
// `AccountIdentity` is the in-memory plaintext shape. Production
// invariants (carried over verbatim from PoC AccountSnapshot):
//   - no Clone / Copy / PartialEq derives
//   - ZeroizeOnDrop transitively via SecretBytes fields
//   - manual Debug impl that redacts every secret field
//   - secret-bearing fields are `pub(crate)`; external readers route
//     through Vault::reveal_* (presence-gated per spec §5.4)
pub struct AccountIdentity {
    /// Non-secret display name (e.g., "GitHub – Main"). Encrypted at
    /// rest but field is `pub` because spec §5.4 does not list it as
    /// reveal-class.
    pub display_name: SecretBytes,

    /// Tags (e.g., `["work", "shared"]`). Non-secret per spec §5.4;
    /// public field. Tags are case-preserving on input but matched
    /// case-insensitively in 1.3's FTS index. NFC-normalised on write.
    pub tags: Vec<SecretBytes>,

    /// Notes — recovery-class secret per the existing PoC discipline.
    /// Crate-private; reach via Vault::reveal_notes.
    pub(crate) notes: SecretBytes,

    /// Associated URLs (multiple per identity per Whitepaper §6).
    /// Validated as non-secret host strings by `validate_url` below;
    /// public field.
    pub urls: Vec<SecretBytes>,

    /// Usernames / emails (multiple per identity per Whitepaper §6).
    /// Public field — the PoC `username` field was already public.
    pub usernames: Vec<SecretBytes>,

    /// Password history. The HEAD entry (index 0) is the current
    /// password; older entries are previous values, retained for
    /// forensics + the "I just rotated; let me copy the old one"
    /// convenience flow. Crate-private; reach via Vault::reveal_password
    /// (which surfaces head-only) or the new presence-gated
    /// Vault::reveal_password_history (returns the full history Vec).
    pub(crate) password_history: Vec<PasswordEntry>,

    /// TOTP secret slot. Empty `SecretBytes` when not configured. The
    /// 1.7 RFC-6238 generator consumes this; 1.2 only stores + reveals.
    /// Crate-private; reach via Vault::reveal_totp_secret.
    pub(crate) totp_secret: SecretBytes,
}

/// One historical password value. Always at least one entry (the
/// genesis password) for any non-tombstoned identity.
pub struct PasswordEntry {
    /// The password bytes. Crate-private; reachable only through the
    /// `password_history` field accessor on AccountIdentity which is
    /// itself crate-private.
    pub(crate) password: SecretBytes,
    /// Wall-clock unix-ms timestamp at which this password was set.
    pub set_at_ms: i64,
    /// 32-byte authoring device id. Same DeviceId type as
    /// `pangolin_store::DeviceId`. P2's stub form is acceptable for
    /// 1.2; 1.5 promotes it to a real verifying key.
    pub originating_device: DeviceId,
}
```

`AccountId` keeps its current PoC shape: 32 random bytes from
`SQLite`'s `randomblob(32)`, generated client-side at create time, with
an `account_id` namespace bound into the AEAD AAD on every revision
(transplant-resistance). **Recommend not switching to UUIDv7/ULID** —
the 32-byte opaque blob already round-trips through the SQL primary
key, the on-chain `RevisionPublished` event topic, and the FFI surface
unchanged.

### B. CBOR payload format (widened)

The current PoC payload is a fixed-6-entry CBOR map keyed by
`display_name`, `username`, `password`, `url`, `notes`, `totp_secret`.
1.2 widens to:

| Key | CBOR type | Cardinality |
|---|---|---|
| `payload_version` | positive integer | required |
| `display_name` | text (UTF-8) | required |
| `tags` | array of text | required, possibly empty |
| `usernames` | array of text | required, ≥ 1 |
| `urls` | array of text | required, possibly empty |
| `notes` | text | required (empty allowed) |
| `password_history` | array of `[bytes, i64, bytes32]` | required, ≥ 1 |
| `totp_secret` | bytes | required (empty allowed) |

Keys in canonical alphabetical order, fixed map length (8). The
`payload_version` discriminator allows readers to:

- read **0** — legacy P2 6-field flat snapshot, hydrated into the
  new `AccountIdentity` shape with `usernames = vec![username]`,
  `urls = vec![url]` (filtered if empty), `tags = vec![]`,
  `password_history = vec![PasswordEntry { password, set_at_ms:
  account_identities.created_at, originating_device:
  revisions.device_id }]`. **No data loss; legacy vaults open
  cleanly.**
- read **1** — production 1.2 shape.
- reject **≥ 2** until the 1.6 schema-versioning policy is locked
  (1.6 may flip "reject" to "read-only-old + migrate-on-write").

Per-blob nonces stay (carry-over from PoC). AAD shape unchanged
(`vault_id || account_id || parent_revision_id || schema_version`);
the `schema_version` field in the AAD continues to track
`meta.schema_version` — that's distinct from the new
`payload_version` inside the encrypted body, by design.

### C. Persistence and crate placement

**Recommendation: continue the 1.1 re-export pattern (Q2-deferred).**

- Production code lives at `crates/pangolin-store/src/account.rs`
  (rewrite of the existing file) and `crates/pangolin-store/src/blob.rs`
  (extended encoder/decoder).
- `pangolin-core` re-exports `AccountIdentity`, `PasswordEntry` from
  `pangolin-store`. The existing `pangolin_core::AccountSnapshot`
  re-export is renamed to `pangolin_core::AccountIdentity` (the FFI
  side keeps `AccountSnapshot` as the read-back shape — see §D).
- `crates/pangolin-core/src/identity/mod.rs` stays a doc-only
  scaffolding module per the 1.1 layout. 1.4 physically relocates
  these types when the session rewrite lands.

This minimises 1.2's diff (no cross-crate file moves); the alternative
of relocating to `pangolin-core::identity::*` now is recorded as Q2
below for Kelvin to weigh.

The SQL schema does NOT change — `account_identities` and `revisions`
already carry the right columns; the new shape is entirely inside the
encrypted `revisions.enc_payload` blob. **No migration required for
existing PoC vaults**: legacy payloads decode via `payload_version=0`
path; on the next `update_account`, the rewritten revision's payload
goes out as `payload_version=1`.

### D. Public API + FFI mapping

The locked-in-1.1 FFI signatures route to new `Vault` methods. The
PoC's existing `add_account` / `update_account` / `get_account` /
`search` / `reveal_*` methods continue to exist for internal callers
(they use the V0 `AccountSnapshot` shape) but are wrapped:

| FFI entry point | New `Vault` method | Notes |
|---|---|---|
| `account_add(VaultHandle, AccountDraft) -> AccountId` | `Vault::account_add(&mut self, draft: AccountIdentityDraft) -> Result<AccountId>` | Consumes draft, validates, builds initial AccountIdentity (single-entry `password_history`), seals + persists genesis revision. |
| `account_update(VaultHandle, AccountId, AccountPatch) -> RevisionId` | `Vault::account_update(&mut self, id, patch: AccountIdentityPatch) -> Result<RevisionId>` | Loads current head identity, applies patch, **password change appends a new PasswordEntry** to the head of `password_history` rather than overwriting. Persists new revision with parent = current head. |
| `account_search(VaultHandle, &str) -> Vec<AccountSnapshot>` | `Vault::account_search(&self, query: &str) -> Result<Vec<AccountIdentitySummary>>` | 1.2 ships a simple full-table scan over decrypted `display_name` / `tags` / `urls` (host-only). FTS5 hookup is **1.3's** scope — the in-memory scan is the documented placeholder. |
| `account_get(VaultHandle, AccountId) -> AccountSnapshot` | `Vault::account_get(&self, id) -> Result<AccountIdentitySummary>` | Returns a presence-NOT-required summary: display_name, tags, urls, usernames, head_revision_id, password_history_count, has_totp. **Does not include the password value or notes** — those still route through presence-gated `Vault::reveal_password` / `reveal_notes`. |
| `account_history(VaultHandle, AccountId) -> Vec<RevisionMeta>` | `Vault::account_history(&self, id) -> Result<Vec<RevisionMeta>>` | Wrapper around the existing `Vault::revisions_for(id)`. |

The FFI shapes need extending — see Q1 below. Every record continues
to carry `schema_version: u16` per 1.1's slot policy.

`AccountDraft` (FFI input) extends to:

```rust
#[derive(Debug, Clone, uniffi::Record)]
pub struct AccountDraft {
    pub schema_version: u16,
    pub display_name: String,
    pub tags: Vec<String>,
    pub usernames: Vec<String>,         // was: username: String
    pub urls: Vec<String>,              // was: url: Option<String>
    pub notes: Option<String>,
    pub password: Vec<u8>,              // initial password
    pub totp_secret: Option<Vec<u8>>,   // optional at create; empty == no TOTP
}
```

`AccountPatch` (FFI input) similarly extends — `Option<Vec<String>>`
fields for tags/urls/usernames (`None` = leave unchanged; `Some(_)` =
replace), `Option<Vec<u8>>` for `new_password` (`Some` triggers a
history append), `Option<Option<Vec<u8>>>` for `totp_secret` (outer
`None` = leave unchanged; `Some(None)` = clear; `Some(Some(bytes))` =
set/replace).

`AccountSnapshot` (FFI read-back) becomes the **non-presence-gated
summary** view — display_name, tags, urls, usernames, head revision
id, password_history_count, has_totp. The actual password / notes /
totp_secret bytes do **not** appear on `AccountSnapshot`; reveals
require dedicated presence-gated FFI entry points which are 1.4's
scope (the existing PoC `Vault::reveal_password` etc. become reachable
via 1.4's session FFI). 1.2's account_get returns this summary;
account_search returns `Vec<AccountSnapshot>` of these summaries.

### E. Validation

Validators run at the public-API boundary (FFI input → core call
site), surfacing `pangolin_core::Error::Validation { kind, message }`
which maps to `FfiError::Validation` via the existing 1.1 `From` impl.
All `kind` strings are stable category labels (no plaintext).

| Field | Rule | `kind` on failure |
|---|---|---|
| `display_name` | NFC-normalise; non-empty after trim; ≤ 256 chars; no control chars except `\t` | `display_name` |
| `tags` | NFC-normalise each; trim; reject empty after trim; reject control chars; dedupe (case-insensitive); ≤ 32 tags; each ≤ 64 chars | `tags` |
| `usernames` | NFC-normalise each; non-empty after trim; ≤ 16 entries; each ≤ 320 chars (RFC-5321 max email length); no control chars | `usernames` |
| `urls` | parse via `url::Url::parse`; accept any scheme (Q3 below); ≤ 32 entries; each serialised form ≤ 2048 chars | `url` |
| `notes` | accept any UTF-8; ≤ 65 536 chars | `notes` |
| `password` | non-empty `Vec<u8>`; ≤ 4096 bytes; **no UTF-8 check** (passwords are arbitrary bytes per the PoC discipline) | `password` |
| `totp_secret` | accept any `Vec<u8>`; ≤ 256 bytes; further RFC-6238 validation defers to 1.7 | `totp_secret` |

The `url` crate (`=2.5.x`) is the proposed parser — pure Rust,
permissive license, **no serde unless feature-flagged** (we depend
without `serde` feature). Under HIGH-1 verification, `url` lives only
in `pangolin-store` (downstream of `pangolin-crypto`), so the HIGH-1
counter cannot be affected.

### F. Schema-versioning hooks

Every FFI record listed in `docs/architecture/ffi-surface.md` already
carries `schema_version: u16`. 1.2 populates it at construction sites:

- `AccountId.schema_version = 1` (was `0` placeholder in 1.1).
- `AccountDraft.schema_version = 1` (caller-supplied; reject if not 1
  per a 1.6-deferred policy — 1.2 records the field but does not yet
  reject; documents the deferred-rejection in Open Question Q4).
- `AccountSnapshot.schema_version = 1` returned-by-construction.
- `RevisionMeta.schema_version = 1` (kept consistent with 1.1's slot).

Per 1.1's "policy slot" doctrine, 1.2 commits only to the **slot
having a value**; the read-only-old / reject-future / migrate-on-write
semantics are still 1.6's commit. The CBOR `payload_version` byte
inside the encrypted blob is a separate concern (D above) — it
distinguishes legacy-PoC blobs from 1.2-blobs at decode time, and 1.6
will tie it together with the FFI `schema_version`.

### G. Revision-creation contract for 1.6

Every `account_update` call appends a new revision whose
`parent_revision_id` is the current head. The new revision's
`enc_payload` carries the **complete** updated AccountIdentity (NOT a
diff); that's what makes the lineage walk correct under fork detection
in P3 and what 1.6's graph-consumer relies on. 1.2 does not implement
fork merging — that's already in P3 as a detection-only primitive.

Tombstones (delete account) are out of scope for 1.2 — the PoC's
tombstone path stays as-is and continues to use the V0 payload shape
(`{ "deleted": true, ... }`); the V1 payload-version discriminator
does not apply to tombstones.

## Success criteria

1. `cargo build --workspace --all-targets` and
   `cargo clippy --workspace --all-targets -- -D warnings` both clean
   on 3-OS CI under rustc 1.94.0.
2. `cargo test --workspace` passes; new tests listed in §"Test plan"
   below all green.
3. `cargo tree -p pangolin-crypto | grep -ci serde` = 0 (HIGH-1).
4. `cargo tree -p pangolin-core  | grep -ci uniffi` = 0 (Q3 of 1.1).
5. `account_add` round-trip: a draft with 3 usernames, 2 urls, 4 tags,
   a TOTP slot, and a password produces an `AccountId`; `account_get`
   on that id returns a summary listing all 3 usernames + 2 urls + 4
   tags + `password_history_count == 1` + `has_totp == true`.
6. `account_update` with a new password produces a new revision and
   bumps `password_history_count` to 2; `account_history` lists 2
   revisions in lineage order.
7. `account_search("github")` over a vault with 3 accounts (one named
   "GitHub – Main", one with tag "github", one with url
   `https://github.com/...`) returns all 3 summaries. Search is
   case-insensitive.
8. **PoC-vault open path:** an existing PoC `.pvf` file (V0 payload
   shape) opens cleanly under the 1.2 build; `account_get` on each
   pre-existing account returns a hydrated 1-entry-history summary.
9. After one `account_update` on a hydrated PoC account, the next
   revision's payload decodes as `payload_version = 1`; the row before
   it remains decodable as `payload_version = 0`. Lineage walk crosses
   the version boundary cleanly.
10. Validation: `account_add` with an empty `display_name` returns
    `FfiError::Validation { kind: "display_name", ... }`; with an
    unparseable URL returns `kind: "url"`; with 17 usernames returns
    `kind: "usernames"`. (One assertion per `kind` listed in §E.)
11. **Forbidden-terms CI** still passes (no new banned strings
    introduced; "revisions" allowlist already covers
    `pangolin-store` / `pangolin-ffi`).
12. **`forbid(unsafe_code)`** retained on every crate except
    `pangolin-ffi`; `pangolin-ffi`'s `deny(unsafe_op_in_unsafe_fn)`
    survives.

## Failure modes considered

- **PoC vault legacy decode regression.** A hydrated V0 → V1 path
  silently dropping a non-empty `notes` field would corrupt user data.
  Mitigation: a `legacy_v0_payload_decodes` unit test that asserts
  every PoC field round-trips into the V1 shape, and a fixture-vault
  integration test using a real pre-1.2 `.pvf` snapshot committed
  under `crates/pangolin-store/tests/fixtures/`.
- **Password-history unbounded growth.** A high-rotation account on
  long-running systems could grow the encrypted blob past
  practical limits. **Recommend NOT capping the history in 1.2** —
  Whitepaper §6 says "multiple passwords over time" without a cap,
  and prematurely truncating loses forensic value. But the
  per-revision blob payload size enforces a soft cap at the SQLite
  layer. Surfaced as Open Question Q5 below.
- **TOTP slot misuse.** A caller passing in a non-base32, non-bytes
  TOTP secret. Mitigation: 1.2 stores opaque bytes; 1.7's RFC-6238
  generator does the decode. 1.2 only checks the byte-length cap.
- **Search leaking secrets.** A fuzzy search that scans usernames or
  passwords would defeat the §1.3 indexing discipline. Mitigation:
  1.2's full-table-scan placeholder scans `display_name`, `tags`,
  and `urls` only — never `usernames`, `notes`, `password`, or
  `totp_secret`. The scan helper has a comment-asserted whitelist
  AND a `compile_fail` doctest mirroring the P4 H-1 pattern.
- **Multi-URL / multi-username order non-determinism.** Two devices
  appending the same username concurrently shouldn't produce two
  spurious "different" revisions. Mitigation: order is preserved as
  insertion order; deduplication is left to the user. Spec §6 doesn't
  mandate canonical ordering. Surfaced as Open Question Q5.
- **`url` crate dep tree drift.** The `url` crate's transitive
  closure (idna, unicode-bidi, percent-encoding) must not introduce
  serde. Mitigation: `cargo tree -p pangolin-store --no-default-features`
  check during the build agent's pass; pin `url = "=2.5.x"` in
  `[workspace.dependencies]`. Verified to be serde-clean as of url
  2.5.4.
- **Linux umask quirk #1.** New tests that write fixture vaults must
  reuse the existing `UmaskGuard::Drop` pattern; the legacy fixture
  path is read-only so this only affects the V0→V1 upgrade test.
- **PowerShell stdout encoding (quirks #5/#6).** Test assertions on
  CLI output (none planned for 1.2 — but if any `apps/cli` smoke
  tests get added, they must use `cargo run --bin pangolin --
  account add ...` and capture via Rust, not PS-piped.

## Open questions for Kelvin

### Q1 — FFI shape: amend 1.1 surface, or stretch the existing 1-singular fields?

**Background.** 1.1 froze `AccountDraft`, `AccountPatch`, `AccountSnapshot`
with single-valued `username`, `url`, `password`, no `tags`, no
`totp_secret`. The §1.2 master plan row mandates "multi-URL,
multi-username, password history with timestamps + originating-device
IDs, TOTP secret slot." **The frozen FFI surface and the §1.2 spec
contradict each other.**

**Options:**
1. **(Recommended) Treat 1.1's identity records as having scaffolding-
   shape only and amend in the same merge commit as 1.2.** Per the
   ffi-surface drift discipline ("After-MVP-1 changes are additive
   only — never field/variant removals, never argument-type changes"),
   field additions are still legal. But this changes
   `username: String → usernames: Vec<String>` (a *type* change), so
   strictly requires a 1.1 amendment commit landed before any external
   binding consumer relies on the surface. Status: no external
   consumers exist yet (1.5 / iOS / Android shells haven't started).
   **Cost:** small. **Cleanliness:** high.
2. **Keep the singular fields and represent multi-* via the
   draft-level conventions** (e.g., `username: String` carries
   semicolon-separated values; `url: Option<String>` stays
   first-of-many). **Cost:** every shell has to parse the convention.
   **Cleanliness:** terrible. Almost certainly a "future amendment
   that breaks existing UI flows."
3. **Defer multi-* / TOTP to a later 1.2.x.** The §1.2 master plan row
   becomes a partial-coverage issue. Not recommended.

**Recommendation:** Option 1. The 1.2 plan above assumes Option 1.

### Q2 — Crate placement: rewrite-in-place or relocate to `pangolin-core::identity` now?

**Background.** 1.1's Q2 deferred the `AccountSnapshot` /
`Vault::*` relocation from `pangolin-store` → `pangolin-core` to 1.4
(session rewrite). 1.2 either continues that pattern or relocates the
identity types now, ahead of session.

**Options:**
1. **(Recommended) Rewrite in place: `AccountIdentity` lives in
   `pangolin-store::account`; `pangolin-core` re-exports.** The diff is
   contained to `pangolin-store` + a small re-export change in
   `pangolin-core`. 1.4's session-rewrite still owns the physical move
   for *all* of identity/session/revision/sync together, in one
   coherent migration.
2. **Relocate now: identity types move into
   `pangolin-core::identity::*`; revision + session stay in
   `pangolin-store`.** Cleaner end-state but introduces a half-finished
   layout (identity here, revision there) that 1.4 still has to clean
   up. Higher 1.2 churn for no immediate gain.

**Recommendation:** Option 1. 1.4's session rewrite is the natural
place to relocate the whole bundle.

### Q3 — URL validation: any-scheme or http(s)-only?

**Background.** Whitepaper §6 says "URLs and applications are
associated attributes" — applications include things like
`com.android.app:org.example.app` or `app://settings`, which don't
parse cleanly as http(s). KDBX (issue 1.9) emits non-http URLs too
(e.g., `ssh://...`).

**Options:**
1. **(Recommended) Accept any URL the `url` crate parses.** The
   parser rejects truly malformed strings; the user's identity model
   accepts arbitrary schemes.
2. **http/https only.** Cleaner UX for the 90% case; rejects
   legitimate KDBX imports.
3. **http/https default + explicit "I really meant it" override.**
   Over-engineered for 1.2.

**Recommendation:** Option 1. The 1.3 search index treats hostnames
as "the host part of the parsed URL when the scheme is http(s); the
raw URL string otherwise" so non-http URLs are still searchable by
substring.

### Q4 — Schema-version policy: reject or accept-and-record?

**Background.** Per 1.1, `schema_version: u16` is a slot; the policy
locks in 1.6. But 1.2 has to do *something* with mismatched versions
right now.

**Options:**
1. **(Recommended) Accept and record only.** Decode every record
   regardless of `schema_version`; persist whatever was supplied.
   1.6's policy commit will retroactively classify which versions
   should have been rejected. Documented as a 1.6-deferred TODO.
2. **Reject anything but `1`.** Tightest discipline but pre-empts
   1.6's policy authority.

**Recommendation:** Option 1.

### Q5 — Password history: cap, normalise order, both, neither?

**Background.** Whitepaper §6 says "multiple passwords over time"
without a numeric cap. Two related decisions:

(a) Bounded length? (The encrypted blob will grow with every
rotation; SQLite's BLOB is fine up to ~2 GB, but per-blob nonces
suggest we want each revision to stay under, say, 64 KB for
sync-friendliness.)

(b) De-duplicate consecutive identical entries? (User accidentally
re-saves the same password twice — append, or no-op?)

**Options:**
1. **(Recommended) No cap; no dedup.** Spec doesn't ask for either.
   Forensic value preserved; user sees what they did.
2. **Cap at 64 entries; no dedup.** Defensive limit; 64 is a sane
   number for any human's account-rotation history.
3. **No cap; dedup consecutive.** Removes accidental double-saves.

**Recommendation:** Option 1. If an issue surfaces later we can add a
cap as an additive policy in 1.6.

## Out of scope (for explicitness)

- **FTS5 search index** (`pangolin-store::search` mature impl) —
  that's **1.3**'s scope. 1.2 ships an in-memory full-table scan as
  the documented placeholder. The scan whitelist (display_name, tags,
  urls only) is the contract 1.3 must preserve.
- **Session policy enforcement** — `Vault::reveal_password` /
  `reveal_notes` / `reveal_totp_secret` already exist (PoC P4) and
  ride on the existing presence-gated path; their FFI exposure is
  **1.4**'s scope. 1.2 does NOT expose reveal entry points at the
  FFI surface.
- **Revision-lineage scheduler / publish queue** — that's **1.6**.
  1.2 only commits to the per-update revision-creation contract.
- **TOTP code generation** (RFC 6238) — that's **1.7**. 1.2 stores
  the slot and lets reveal route through the existing
  `Vault::reveal_totp_secret`.
- **KDBX import** — that's **1.9**.
- **Encrypted export** — that's **1.10**.
- **UI / shell wiring** — `apps/cli` will gain `account add` / `list`
  / `update` subcommands as part of the 1.2 build (it's the only way
  to smoke-test end-to-end), but the *shell-side* design is
  trivially-thin and out of plan-gate scope.
- **Account deletion / tombstoning** — PoC's tombstone path stays
  unchanged; 1.2 does not extend the V1 payload to tombstones.
- **Schema-version rejection policy** — **1.6**.
- **Type relocation to `pangolin-core::identity::*`** — **1.4**.
- **Chain integration** — **MVP-2**.
