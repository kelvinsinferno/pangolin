## Open questions for Kelvin (LOAD-BEARING — surface at TOP per §16)

### Q-a · Source of the ephemeral encryption key — what is the "device secret" the master plan §5 phrase refers to?

Master plan §5 row 4.3: "encrypted with ephemeral key derived from device secret." D-007 verbatim is silent on the derivation source. Five plausible shapes:

**Option A: The full `DeviceKey` (Ed25519 seed bytes).** The indexer crate gains a path-dep on `pangolin-crypto` (currently absent — 4.2 L7 says pangolin-indexer does NOT depend on pangolin-store/core/crypto). The host hands the indexer a `&DeviceKey` at session construction; the indexer derives the temp-DB AEAD key via HKDF-SHA256 over the Ed25519 seed bytes with a versioned info string (`"pangolin-indexer-tempdb-key-v0"`). **Pro:** Simplest derivation; deterministic per device. **Con:** widens pangolin-indexer's dep surface to include `pangolin-crypto` (the crate that holds `DeviceKey`); the indexer process now has direct access to the device's Ed25519 signing material — a large secret-material expansion of its blast radius. Per the L7 invariant (4.2): the indexer crate is structurally minimal; Option A widens it.

**Option B: A purpose-derived sub-key produced by `pangolin-chain` and handed to the indexer as a `Zeroizing<[u8; 32]>`.** A new helper in `pangolin-chain::evm` (or a new `pangolin-chain::indexer_key` module) — `derive_indexer_tempdb_seed(device: &DeviceKey, run_nonce: [u8; 16]) -> Zeroizing<[u8; 32]>` — uses the same HKDF-from-Ed25519-signature pattern `derive_evm_wallet` already uses (with a different domain-separator string + a per-run nonce mixed into the info). The host calls this helper, then passes the resulting opaque 32 bytes (no `DeviceKey` reference) into the new `IndexerSession::new(config, cipher: Arc<dyn TempDbCipher>)` constructor — replacing the cipher with `AeadCipher::from_seed(seed)`. The indexer crate stays free of any `DeviceKey` / `pangolin-crypto` direct dep; it only sees the derived 32-byte seed. **Pro:** mirrors the existing `derive_evm_wallet` separation ("the chain crate owns the derivation; consumers see only the derived material"); cleanest dep surface; matches master plan §5 wording without expanding indexer's secret-material reach. **Con:** more code (new helper + the per-run nonce wiring through the IPC + session constructor).

**Option C: The `EvmWallet`'s secp256k1 private bytes.** Reach through `EvmWallet::signer()` and use the secp256k1 secret bytes as IKM. **Con:** EvmWallet's whole point per `pangolin-chain::evm.rs` is "device's gas wallet" — feeding it through HKDF for *any* purpose other than chain signing/gas crosses the cryptographic separation property the existing comments lock in. Rejected on cryptographic-discipline grounds.

**Option D: Purely random per-run 32-byte key (NOT derived from device secret).** Generate via `OsRng::fill_bytes(&mut [0u8; 32])` at session startup; never derive from anything. **Pro:** simplest; zero new dep; no key-recovery property to defend. **Con:** contradicts master plan §5 row 4.3 verbatim ("derived from device secret"). May be desirable for non-recoverability (if the temp DB is ephemeral anyway, what does "derived from device secret" buy?), but that's a spec-revision question, not an implementation choice. **Sub-question:** if Option D is chosen, surface a DECISIONS.md amendment to D-007 explicitly relaxing the "derived from device secret" language.

**Option E: `Zeroizing<[u8; 32]>` derived from a per-run nonce mixed with `OsRng` AND a static install-id (e.g., from `pangolin-store::Vault::install_id` if such exists; deferred to a Q if it doesn't).** Hybrid; gets the spec language ("device-derived" loosely interpreted as "this install's ID") without requiring DeviceKey access. **Con:** install_id may not exist as a separate primitive in MVP-2; would need to verify.

**Plan-gate recommends:** Option B — purpose-derived sub-key in `pangolin-chain` honoring the master plan §5 wording while keeping `pangolin-indexer`'s dep surface and secret-material reach minimal. Mirrors the existing `derive_evm_wallet` separation pattern. The per-run nonce ensures L-key-derivation-collision resistance.

**Kelvin's call.**

### Q-b · SQLite encryption layer — how does the AeadCipher actually wrap the temp DB?

The 4.2 `TempDbCipher` trait shape (`encrypt_page(&[u8]) -> Vec<u8>` + `decrypt_page(&[u8]) -> Vec<u8>`) was scaffolded around an as-yet-unspecified call-pattern. For a real AEAD impl, three architectural shapes:

**Option A: sqlcipher (well-known SQLite extension).** Replace `rusqlite` with `rusqlite` + sqlcipher feature; configure the temp DB connection with `PRAGMA key='<derived-key-hex>'`. The page encryption + per-page nonce + tag handling all happen inside sqlcipher's C-level page handler. **Pro:** battle-tested; one PRAGMA call. **Con:** new build dep (sqlcipher needs the libsqlcipher native library — adds platform-specific build complexity, especially on Windows + iOS); doesn't reuse `pangolin-crypto` (so the workspace ends up with two AEAD primitives — XChaCha20-Poly1305 in pangolin-crypto AND AES-256-CBC in sqlcipher). Violates 4.2 L6 ("no new external crate dep"). Rejected on dep-discipline grounds.

**Option B: Custom page-level via the existing `TempDbCipher` trait, wrapping each SQL-write blob in an XChaCha20-Poly1305 seal.** The session's `persist_chunk` path (today writes plaintext BLOBs via rusqlite) gets a wrap layer: each `enc_payload` (and possibly `signer`, `block_hash`, `tx_hash` — the metadata correlation surface) is sealed via `cipher.encrypt_page` before INSERT; the `handle_pull` read path unseals via `cipher.decrypt_page`. **Sub-decision:** which columns get sealed? Plan-gate recommends ALL BLOB columns + ALL metadata that is per-vault correlatable (vault_id, account_id, parent_revision, device_id, signer, block_hash, tx_hash, enc_payload). **Sub-decision:** the trait surface needs nonce wiring — current `encrypt_page(&[u8]) -> Vec<u8>` has no nonce parameter. Two options for nonce discipline:
  - **B1: Random 24-byte nonce per call**, prepended to the returned Vec (decrypt splits nonce + ciphertext from the head). XChaCha20-Poly1305's 192-bit nonce makes random-collision negligibly unlikely.
  - **B2: Internal monotonic counter** managed by AeadCipher's interior mutability (`AtomicU64`), combined with a 16-byte random session-id prefix; nonce = `session_prefix || counter_be_bytes`. Strictly unique per call.
  - Plan-gate recommends B1 (random nonces) — simpler; no shared state between encrypt+decrypt; matches the existing `Nonce::random()` discipline in `pangolin-crypto`.

**Option C: Full-file blob encryption.** Encrypt the entire SQLite DB file as an opaque blob; SQLite operates on a temp decrypted buffer in memory. **Con:** defeats SQLite's random-access (every read decrypts the entire file); memory footprint balloons; rejected on perf grounds.

**Option D: Page-level via SQLite's native VFS shim.** Implement a custom VFS in Rust that wraps each 4 KiB page write with AEAD seal + each page read with AEAD open. **Pro:** transparent to the SQL layer. **Con:** rusqlite doesn't expose the VFS extension surface in safe Rust; would require unsafe code (violates L7); rejected.

**Plan-gate recommends:** Option B + B1 (custom column-level via TempDbCipher trait + random per-call nonces). Reuses `pangolin-crypto` exclusively (zero new deps); per-call random nonces eliminate state-management complexity; the trait surface needs a minor extension to accommodate nonce-prepending semantics (the cipher returns `nonce || ciphertext || tag` as a single `Vec<u8>` — the trait shape stays compatible with 4.2's signature).

**Kelvin's call.**

**Sub-question (Q-b'):** does the trait signature need to change (`encrypt_page` now returns a longer buffer; `decrypt_page` now expects nonce+ciphertext+tag concatenated)? **Plan-gate's view:** NO — the trait signature stays `Vec<u8>` in/out; the impl handles framing internally. 4.2's `NoOpCipher` round-trip property still holds (encrypt → decrypt = identity on the full buffer). New tests verify the round-trip + tampered-tag rejection.

### Q-c · Zero-fill strategy before unlink

D-007 verbatim: "explicit zero-fill of file before unlink." Four plausible disciplines:

**Option A: Single-pass zero-fill + fsync + unlink.** On `IndexerSession` Drop (and explicit shutdown): seek to start of NamedTempFile; write zeros up to file length; fsync; let NamedTempFile's Drop unlink. **Pro:** simplest; defeats casual forensic recovery (the unlinked file's blocks contain zeros until OS GC). **Con:** SSDs with wear-leveling may write the zeros to a different physical block than the original ciphertext (the original ciphertext's physical block persists until TRIM/garbage-collection); modern filesystems (APFS, NTFS, ext4) may also defer the actual write.

**Option B: Random-fill + zero-fill + fsync + unlink.** Two passes — first random data (defeats some forensic-recovery patterns that look for "zeroed = recently deleted" markers), then zeros. **Pro:** marginal forensic-recovery improvement. **Con:** doubles the write cost; still doesn't defeat SSD wear-leveling.

**Option C: Multi-pass DoD 5220.22-M style.** Three+ passes (zeros, ones, random). **Con:** overkill for modern SSDs (where wear-leveling makes overwrite unreliable anyway); excessive write cost; rejected.

**Option D: TRIM / discard hint via OS-level API.** On Linux: `posix_fadvise(fd, 0, 0, POSIX_FADV_NOREUSE | POSIX_FADV_DONTNEED)`. On Windows: `DeviceIoControl` with `FSCTL_FILE_LEVEL_TRIM`. On macOS: `fcntl(fd, F_NOCACHE, 1)`. Plus the zero-fill from Option A. **Pro:** best chance of actually freeing the underlying SSD blocks. **Con:** OS-specific; many platforms ignore the hint; some require root.

**Plan-gate recommends:** Option A (single-pass zero-fill + fsync + unlink) — simplest; documented limit ("best-effort against forensic recovery; assumes attacker doesn't have raw-SSD access; for raw-disk-access scenarios the chain data is non-secret anyway — only the vault_id correlation is sensitive"). The zero-fill is mostly belt-and-suspenders since 4.3's encryption already means the unlinked blocks contain ciphertext (not plaintext). The combined defense is: on-disk = ciphertext (the encryption); after unlink with zero-fill = zeros (the zero-fill); SSD-physical-block residue = ciphertext-residue = still safe (the encryption key is gone with the process). **The zero-fill is therefore primarily a defense for the per-row metadata IF Q-b chooses NOT to encrypt every BLOB column** — surface this dependency to Kelvin.

**Kelvin's call.**

**Sub-question (Q-c'):** does zero-fill happen on panic-unwind too? Plan-gate recommends YES — the zero-fill discipline must run in the Drop impl of a wrapper around NamedTempFile (so it fires on stack-unwinding panics). For `panic = abort` mobile profiles, OS-level temp-dir GC remains the only defense (same posture 4.2 L11 documents).

### Q-d · Key lifecycle in memory — wrapper type

The ephemeral 32-byte key lives in the IndexerSession's address space for the session's lifetime (≤ 1 hour per L-idle-timeout-DoS ceiling from 4.2). Three wrapper shapes:

**Option A: `Zeroizing<[u8; 32]>` from the `zeroize` crate.** Stack-allocated; zeroed on Drop via `core::ptr::write_volatile`. Already used by `pangolin-crypto::kdf::derive_seed` and `pangolin-chain::evm::EvmWallet`. **Pro:** simple; widely used; matches existing patterns. **Con:** the `[u8; 32]` lives on the stack (not heap), so move-semantics may leave stale unzeroed copies in stale stack frames (the `pangolin-crypto::secret::BoxedSecret` exists exactly to defend this — see MEDIUM-8 in the audit notes).

**Option B: `pangolin-crypto::secret::BoxedSecret<32>`.** Heap-allocated; stable address; zeroed on Drop. Matches the discipline `AeadKey` already uses. **Pro:** stronger move-safety; matches the most-secret-material-conscious pattern in the workspace. **Con:** requires pangolin-indexer to depend on pangolin-crypto (which Option A in Q-a explicitly rejects); BUT — if Q-a Option B (purpose-derived sub-key) lands, the seed is opaque 32 bytes from pangolin-chain and can be wrapped in either Zeroizing or BoxedSecret without pulling pangolin-crypto into pangolin-indexer (BoxedSecret is in pangolin-crypto, so Option B here DOES require the dep).

**Option C: `pangolin_crypto::aead::AeadKey` directly.** The cipher impl holds an `AeadKey` (which already wraps `BoxedSecret<32>` and zeros on Drop). The cipher's seal/open paths use it. **Pro:** matches pangolin-crypto's discipline exactly; no wrapper code in pangolin-indexer. **Con:** requires pangolin-indexer to depend on pangolin-crypto (same as Option B).

**Option D: Plain `[u8; 32]`.** Rejected — bad discipline; key bytes survive in stack/heap until GC.

**Plan-gate recommends:** Option C (`AeadKey` directly inside the cipher impl) — IF pangolin-indexer adds a path-dep on pangolin-crypto. Best discipline. **The dep-edge question is the actual decision** — adding pangolin-crypto as a pangolin-indexer dep crosses the 4.2 L7 invariant ("downstream of pangolin-chain; does NOT depend on pangolin-store or pangolin-crypto"). Two paths:
  - **Path 1:** Q-a Option B + Q-d Option C — pangolin-indexer adds pangolin-crypto dep (for AeadKey + BoxedSecret); the seed comes from pangolin-chain (via Q-a B); the cipher uses AeadKey internally. Cleanest crypto discipline; widens dep surface by one crate.
  - **Path 2:** Q-a Option B + Q-d Option A — pangolin-indexer stays free of pangolin-crypto; uses bare `Zeroizing<[u8; 32]>` from the `zeroize` crate (which is a transitive dep already via pangolin-chain) and instantiates the XChaCha20-Poly1305 primitive directly via the `chacha20poly1305` crate (same as pangolin-crypto::aead does internally). **Con:** duplicates the AEAD primitive wiring outside pangolin-crypto — violates the "all crypto goes through pangolin-crypto" discipline.
  - **Path 3:** Q-a Option B + Q-d Option C BUT pangolin-crypto exposes a *minimal* surface (just `AeadKey` + the seal/open methods needed for page encryption) — pangolin-indexer pulls pangolin-crypto but only for the AEAD primitive (doesn't reach DeviceKey or other secret-bearing types). **Plan-gate recommends Path 3** — best balance.

**Kelvin's call.**

## Adversarial framing — L-section (load-bearing risks)

> Per §16, every plan-gate doc for a security-critical surface enumerates the load-bearing risks an adversary could leverage from each design choice. 4.3's surface is the temp-DB-on-disk encryption + the in-memory ephemeral key + the zero-fill discipline + the key derivation.

### L-key-leak-via-memory-dump

**What goes wrong:** an attacker with memory-dump access (post-compromise; coredump capture; debugger attach) reads the ephemeral 32-byte key out of the IndexerSession's address space.

**Failure mode:** the attacker can decrypt the temp DB's contents (per-event metadata correlation surface). Note: the per-event `enc_payload` is independently AEAD-protected by the vault VDK in MVP-1; the metadata (vault_id, signer addresses, block hashes, tx hashes) is what's exposed.

**Defense:**
- (a) Short-lived process: 5-min idle timeout + 1-hour absolute-runtime ceiling (4.2 R-c).
- (b) Wrapper-type discipline (Q-d): `Zeroizing<[u8; 32]>` or `BoxedSecret<32>` zeros memory on Drop.
- (c) The session never logs the key; `Debug` impls redact (mirrors `pangolin-crypto::aead::AeadKey`'s `<redacted>` debug shape).

**Adversary leverage:** bounded by the process lifetime + the wrapper-type discipline. Memory-dump access is already a host-level compromise; the indexer is downstream.

### L-tampered-ciphertext

**What goes wrong:** a malicious party with disk-access (between two indexer reads of the same temp DB row) tampers a temp DB page's ciphertext bytes.

**Failure mode:** without AEAD authentication, the indexer would silently consume tampered plaintext. With AEAD authentication, decrypt fails with `AeadError::Tampered`.

**Defense:**
- (a) AEAD tag verification at decrypt time (XChaCha20-Poly1305's Poly1305 tag is checked before the plaintext is exposed).
- (b) Tampered page → typed `AeadError::Tampered` → indexer surfaces a typed `IndexerError::CipherTamper` (Q-e') → indexer aborts.
- (c) AAD per page binds (vault_id, page_id, schema_version) so a row's ciphertext can't be replayed into a different row.

**Adversary leverage:** mitigated to "indexer aborts; user sees typed error; nothing silently corrupted." The vault data itself is unaffected (the temp DB is per-session; the indexer's job is to feed events into the vault, not to be the vault).

### L-nonce-reuse-catastrophic

**What goes wrong:** if two `encrypt_page` calls use the same nonce + same key, XChaCha20-Poly1305 leaks both plaintexts (CRITICAL — known cryptographic property of all stream-cipher AEADs).

**Defense:**
- (a) The nonce-discipline choice in Q-b MUST guarantee uniqueness. Random 24-byte nonces (XChaCha20-Poly1305's 192-bit nonce) make collision probability ~ 2^-96 per call (negligible for ≤ 2^32 calls). Counter-based nonces (B2) use AtomicU64 to guarantee strict monotonicity; combined with a random session prefix to ensure cross-session distinctness.
- (b) Property test: 10_000 calls, assert all returned nonces are pairwise-distinct.

**Adversary leverage:** none if nonce discipline holds; catastrophic if it breaks. This is THE load-bearing crypto property.

### L-forensic-recovery-via-deleted-file

**What goes wrong:** an attacker with raw-disk access (post-compromise; lost laptop; physical seizure) reads the unlinked temp file's underlying disk blocks before the OS / SSD overwrites them.

**Defense:**
- (a) Zero-fill before unlink (L4) — the on-disk blocks contain zeros after unlink (best-effort against casual forensic recovery).
- (b) The data was encrypted at rest in the first place (L1 + L3) — even if the zero-fill failed, the recovered blocks contain ciphertext, not plaintext. The ephemeral key is gone with the process (L3).
- (c) Documented limit: SSDs with wear-leveling may write the zeros to a different physical block than the original ciphertext; the original ciphertext's physical block persists until TRIM/garbage collection. This is a best-effort defense; for raw-SSD-access scenarios the chain data is non-secret anyway (only the vault_id correlation is sensitive — and that's what the encryption protects).

**Adversary leverage:** mitigated to "ciphertext residue on SSD physical blocks until SSD GC; ephemeral key is gone." The combined defense (encryption + zero-fill + ephemeral key) makes recovery essentially impossible.

### L-key-derivation-collision

**What goes wrong:** two indexer runs derive the same key (because the per-run randomness in the derivation is missing or weak). A captured ciphertext from run-1 decrypts under run-2's key, leaking metadata across runs.

**Defense:**
- (a) Per-run nonce in the key derivation (Q-a Option B's `run_nonce: [u8; 16]` parameter — 128 bits of entropy from `OsRng`).
- (b) HKDF-SHA256 expansion ensures the derived bytes are pseudorandom function output of the (seed, nonce) pair.
- (c) Even if two runs accidentally collide on the run_nonce (probability ~ 2^-64 per pair after 2^32 sessions — astronomically unlikely), the plaintext-bytes-vs-ciphertext relationship is per-vault-correlatable but not cross-decrypt vulnerable (each AEAD seal binds AAD that includes vault_id + schema_version).

**Adversary leverage:** none if run_nonce is fresh OsRng each session.

### L-cipher-not-wired-into-sql-path

**What goes wrong:** a builder regression leaves the cipher held in `IndexerSession` but never invoked on the `persist_chunk` write path or `handle_pull` read path. The temp DB is unencrypted-on-disk despite the cipher trait being present (this is exactly 4.2's current state — the cipher is held but never used beyond the constructor probe).

**Defense:**
- (a) Mandatory test: read raw bytes from the temp DB file (bypass SQLite, use `std::fs::read`); assert NO plaintext column value (e.g., the bound vault_id) appears in the file. Pre-4.3 this test FAILS (4.2 writes plaintext); post-4.3 it MUST PASS.
- (b) CI invariant grep: the production `persist_chunk` and `handle_pull` callsites MUST reach `cipher.encrypt_page` / `cipher.decrypt_page`.

**Adversary leverage:** the entire 4.3 deliverable is the wiring; this risk is the load-bearing implementation correctness check.

### L-indexer-grows-pangolin-crypto-secret-material-reach

**What goes wrong:** if Q-a resolves to Option A (full DeviceKey access) or Q-d resolves to Option C (pangolin-crypto dep), the pangolin-indexer crate now imports types that hold secret material. A future bug in the indexer (e.g., a serde derive that gets accidentally added) could leak the DeviceKey — an attack surface that didn't exist before 4.3.

**Defense:**
- (a) Q-a Option B (the recommended resolution) keeps the indexer seeing only opaque 32-byte derived bytes — no `DeviceKey` reference reaches pangolin-indexer.
- (b) `assert_not_impl_any!(AeadCipher: Clone, Copy, serde::Serialize)` at the type level (mirrors the discipline pangolin-crypto already uses for its key types).

**Adversary leverage:** structural; mitigated by Q-a Option B.

## Builder notes (advisory; not load-bearing)

1. **The `pangolin-indexer` crate's `cipher.rs` already exists** as a 4.2-shipped scaffolding file (127 lines). 4.3 ADDS `AeadCipher` to it; does NOT delete `NoOpCipher` (that stays for `#[cfg(test)]` per L10). Builder must NOT re-create the file.

2. **D-007 verbatim** (locked): "Random-path encrypted temp DB; explicit zero-fill before unlink; cleanup on crash via OS-level temp-file conventions." 4.2 shipped the random-path + cleanup-on-crash properties; 4.3 ships the encryption + zero-fill properties. Master plan §5 row 4.3 adds the "encrypted with ephemeral key derived from device secret" specification — the source of the derivation is Q-a's domain.

3. **The 4.2 `cipher.rs` trait surface (`encrypt_page(&[u8]) -> Vec<u8>`) is intentionally minimal.** A real AeadCipher needs a per-call nonce — Q-b's recommended B1 (random nonce prepended to the returned Vec) keeps the trait signature unchanged while satisfying the AEAD discipline. Builder must implement the framing inside `AeadCipher`'s methods, not by changing the trait.

4. **The 4.2 `IndexerSession::new` constructor probe** (`let probe = cipher.decrypt_page(&cipher.encrypt_page(b"4.2-cipher-probe"));`) ALREADY assumes encrypt+decrypt are inverses on the same buffer — this works for AeadCipher with the recommended Q-b B1 framing. Builder MUST verify this probe still passes with `AeadCipher` (it should — random nonce + seal + return `nonce || ct` + later split + open returns the original plaintext bytes).

5. **The 4.2 `persist_chunk` and `handle_pull` paths write/read plaintext BLOBs today.** 4.3's load-bearing wiring is to insert `cipher.encrypt_page(...)` / `cipher.decrypt_page(...)` calls at every BLOB column. Builder must NOT skip a single column — the L-cipher-not-wired-into-sql-path test reads the raw file and grep-checks for plaintext.

6. **The `ZeroFillTempFile` newtype is the load-bearing L4 + L11 piece.** Drop impl: seek to 0, write zeros up to file length, fsync, then drop the inner NamedTempFile. Builder must verify the Rust drop order — fields are dropped in declaration order; the `ZeroFillTempFile` must own the NamedTempFile and Drop in the right sequence.

7. **The ephemeral key is per-session.** Each `IndexerSession::new` call generates a fresh `run_nonce` and derives a fresh key. The host code that spawns the indexer (apps/cli or the desktop subprocess wrapper) is the place where the derivation happens; the indexer crate just receives the opaque 32 bytes (per Q-a Option B).

8. **Q-a Option B's `derive_indexer_tempdb_seed` helper must NOT live in the indexer crate.** It belongs in `pangolin-chain` (per L7's dep-direction discipline) so the indexer never imports `DeviceKey`. The helper takes `&DeviceKey` + `[u8; 16]` and returns `Zeroizing<[u8; 32]>`; the host calls the helper, then passes the seed to `AeadCipher::new_arc(seed)`.

9. **Per L10**, the `NoOpCipher` MUST stay accessible from tests (the `noop_round_trips` etc. tests in 4.2's cipher.rs still need to pass). Builder gates the production callsite (`bin/pangolin-indexer.rs`) to use `AeadCipher::new_arc(...)` exclusively; the test surface keeps using `NoOpCipher::new_arc()`.

10. **The L-cipher-not-wired-into-sql-path raw-disk test** is the load-bearing implementation-correctness check. It catches "builder forgot to wrap a column" regressions. Builder runs it locally before requesting merge.
