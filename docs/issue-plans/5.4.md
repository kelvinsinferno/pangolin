## Open questions for Kelvin (LOAD-BEARING тАФ surface at TOP per ┬з16)

### Q-a ┬╖ Where does the SyncOrchestrator live?

**Option A: Engine helper module (`pangolin-store::sync_orchestrator`).** Ship `SyncOrchestrator { vault: Arc<Mutex<Vault>>, balance_monitor: Arc<BalanceMonitor>, ... }` + a `tick()` method that handles one cycle of `(pull тЖТ conflict-delta тЖТ flush-if-window-elapsed тЖТ state-machine-transition)`. Host owns the `tokio::interval` + calls `tick()` per interval. Pro: one consolidated implementation across CLI / Tauri / mobile. **Con тАФ architectural blocker:** introduces `Arc<Mutex<Vault>>` wrapping which BREAKS the !Sync Vault discipline (P4 audit M-3: inner `rusqlite::Connection` holds a `RefCell`; `dyn Clock` is also !Sync). Every `&mut Vault` callsite would need to be reworked. Plan-gate confirms this is structurally hard, not stylistic.

**Option B: New `pangolin-orchestrator` crate.** Same as (A) but in its own crate. Same !Sync blocker as (A) + new crate.

**Option C: Pure host concept.** `pangolin-store` ships only:
1. A `SyncStatus` enum (Q-b)
2. A pure `compute_next_status(prev_status, inputs: SyncStatusInputs) -> SyncStatus` transition function
3. A bundling accessor `Vault::sync_status_inputs() -> Result<SyncStatusInputs>` that returns a snapshot for the pure transition function
4. (Optional Q-e) A new `Vault::lock_with_drain(adapter, device_key)` async method that runs `flush_publish_queue` BEFORE dropping `active`, closing the 5.1 L1 deviation

The host owns ALL the wiring. Pro: ZERO new tokio surface in engine; matches 5.2 R-a verbatim; smallest engine footprint; no !Sync reconciliation. Con: every host has to wire it.

**Option D: BalanceMonitor pattern (mimic 3.5).** Engine ships `SyncStatusMonitor` with its own background task. **Con тАФ architectural blocker:** BalanceMonitor avoids the Vault reference issue by polling stateless RPC. `SyncStatusMonitor` would need to invoke `flush` + `pull` (both `&mut Vault`) тАФ same `Arc<Mutex<Vault>>` blocker as (A). Plan-gate rejects.

**Plan-gate recommends: Option C** (pure transition function + state-machine enum + bundling accessor + optional `lock_with_drain` in engine; host owns the loop). Matches 5.2 R-a verbatim. The "every host has to wire it" concern is mitigated by shipping a canonical host loop body example in `docs/architecture/sync-orchestrator.md`. If Kelvin signs (A) or (D), the !Sync reconciliation is its own substantial cycle of work тАФ arguably out of scope for an MVP-2 tail-end issue.

**Kelvin's call.**

### Q-b ┬╖ SyncStatus enum shape тАФ how many variants and what data?

Master plan ┬з5 row 5.4 verbatim reads "`Synced / Syncing... / Offline`" тАФ only 3 variants. The 5.1-5.3 plans have written in "Conflicts pending" / "Action required" / "BlockedOnBalance" as if 5.4 covers them. Honest question: does the indicator chip carry the conflict / balance state too, or are those orthogonal pills?

**Option A: Minimal 3-variant (spec verbatim).** `Synced` / `Syncing` / `Offline`. Pro: smallest; spec-compliant. Con: doesn't distinguish "Synced with N conflicts pending" from "Synced + clean"; "blocked on balance" must surface via separate channel (already does via PublishQueueState).

**Option B: 5-variant.** `Synced` / `Syncing { mode: SyncMode }` / `Offline { consecutive_failures: u32 }` / `ConflictsPending { count: u32 }` / `ActionRequired { reason: String }`. Pro: covers the common UX cases. Con: more variants to test.

**Option C: 6-variant.** Above + `BlockedOnBalance { needed_wei: u128, available_wei: u128 }`. Pro: surfaces 5.1's everything-or-nothing balance refusal at the indicator layer. Con: another variant; needs ┬з8.1.5 vocabulary check (L5).

**Option D: Composite тАФ 4 orthogonal dimensions.** `SyncStatus { connectivity, queue, conflicts, balance }`. Pro: precise. Con: 4├Ч state space; host has to render N pills, not one chip.

**Plan-gate recommends: Option C** тАФ 6-variant single enum. The indicator is rendered as a SINGLE chip; one variant at a time. The composite shape is over-engineered for an indicator. The 6-variant superset captures the dominant UX cases (the spec's 3 + the three load-bearing exception states). **Vocabulary discipline (L5):** variant names follow ┬з8.1.5 vocabulary verbatim тАФ NEVER pricing-copy ("LowBalance" / "OutOfGas" / "Upgrade").

**Kelvin's call.** If Kelvin signs A (spec verbatim), the engine ships the minimal 3 and the host renders conflict / balance pills as separate UI surfaces.

### Q-c ┬╖ Offline detection тАФ how many consecutive pull failures before `Offline`?

**Option A: 1 failure тАФ immediate.** Pro: most responsive. Con: a 1-tick RPC hiccup flaps the chip at 60s cadence.

**Option B: 3 failures тАФ ~3 min at 60s cadence.** Pro: tolerates one-off RPC blips; surfaces real outages within ~3 min.

**Option C: 5 failures тАФ ~5 min.** More conservative.

**Option D: Configurable via env-var.** Per-host tuning; another env-var; test surface grows.

**Plan-gate recommends: Option B** тАФ 3 consecutive failures. The counter resets on the FIRST successful pull (any `Ok(_)` from `pull_once`, including signal-only OfferFast / AlwaysFast cycles тАФ they're not RPC failures even though they don't actually read the chain).

**Kelvin's call.**

### Q-d ┬╖ Tick coordination тАФ pull-then-flush, flush-then-pull, or interleaved?

**Option A: Pull-then-flush.** Each tick: pull тЖТ flush-if-dirty тЖТ compute status. Pro: ingest before publish.

**Option B: Flush-then-pull.** Each tick: flush тЖТ pull тЖТ compute status. Pro: minimizes write-side delay.

**Option C: Interleaved (two host timers).** Pull when 60s pull-interval elapses; flush when 30s flush-window elapses; state-machine update on either event. Pro: each primitive's natural cadence; matches per-issue plan-gate decisions verbatim (5.1 R-a 30s / 5.2 R-b 60s). Con: canonical host loop is two `tokio::select!` arms.

**Plan-gate recommends: Option C** тАФ interleaved. The 5.1 30s and 5.2 60s cadences are different by design; forcing them to align discards the per-issue decisions.

**Kelvin's call.**

### Q-e ┬╖ Pre-lock drain тАФ does 5.4 ship the host orchestration that calls `flush_publish_queue` before `lock`?

5.1 L1 declared "queue ALWAYS drains on every session-teardown path" but the BUILD-pass deviation was: the engine's `lock()` is sync; can't await flush. The host owns the drain. 5.4 can close this.

**Option A: Engine ships `Vault::lock_with_drain(adapter, device_key) -> Result<(), BatchFlushError>`.** NEW async method: runs `flush_publish_queue(force=true)` BEFORE dropping `active`; on flush failure (network, balance) it STILL proceeds to drop `active` (best-effort; matches 5.1 L1 doctrine "teardown wins"). Pro: closes the 5.1 L1 deviation properly.

**Option B: Leave it to the host.** CLI / Tauri / mobile each call `flush ... ; lock();` themselves. Documented in `sync-orchestrator.md`. Pro: zero engine change. Con: every host has to remember; behavior diverges across hosts.

**Option C: Defer to a separate issue.** Pro: smallest 5.4. Con: 5.1 L1 deviation stays open.

**Plan-gate recommends: Option A** тАФ closes the 5.1 L1 deviation properly.

**Kelvin's call.**

## Adversarial framing тАФ L-section (load-bearing risks)

### L-orchestrator-races-with-edits

**What goes wrong:** orchestrator holds `&mut Vault` during a tick; user concurrent edit attempts. Borrow checker catches. Inherited from 5.1/5.2 family.

### L-shutdown-during-flush-race

**What goes wrong:** Q-e Option A `lock_with_drain` calls flush; concurrent `lock()` races. Sequential via `&mut self`.

### L-offline-flapping

**What goes wrong:** borderline RPC toggles status. Mitigated by Q-c Option B 3-consecutive-failures threshold. Test `offline_threshold_requires_three_consecutive_failures`.

### L-status-leaks-balance-detail

**What goes wrong:** Q-b Option C `BlockedOnBalance { needed_wei, available_wei }` carries u128 wei values. **Acceptable** тАФ same info already exposed via 5.1 `BatchFlushError::BalanceInsufficientForBatch`. L5 vocabulary discipline ensures variant NAME doesn't leak pricing copy.

### L-orchestrator-leaks-past-lock

**What goes wrong:** host's orchestration loop outlives session. Same posture as 5.2 L-host-scheduler-leak: documented canonical host loop body; engine ensures `NoActiveSession` returns immediately on locked vault. Dedicated test `orchestrator_tick_on_locked_vault_transitions_to_locked_terminal`.

### L-conflict-pill-flashes-on-self-publish

**What goes wrong:** user publishes locally; round-trip pull-tick could (in a buggy implementation) transition `Synced тЖТ ConflictsPending тЖТ Synced`.

**Mitigation:** 5.3 R-b regression test guarantees `newly_frozen_accounts.is_empty()` on the round-trip tick тАФ `compute_next_status` never sees a conflict-delta event on the self-publish path. Dedicated test `self_publish_round_trip_does_not_flash_conflicts_pending`.

### L-balance-state-stale-vs-flush-error

**What goes wrong:** `BalanceMonitor` is on its own 30s cadence; orchestrator's tick may read a stale `Sufficient` while fresh `flush` returned `BalanceInsufficientForBatch`. Transition function gets contradictory inputs.

**Mitigation:** transition function PREFERS the most recent direct signal тАФ a `BatchFlushReport` error variant overrides the BalanceMonitor's cache. Encoded explicitly: `if last_flush_error == BalanceInsufficientForBatch { return BlockedOnBalance(...) }` BEFORE consulting `GasBalanceState`. Dedicated test `balance_state_stale_overridden_by_fresh_flush_error`.
