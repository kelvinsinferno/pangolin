# Issue 2.1: Revision Log v1 contract (signature-verifying append-only log)

> One-line scope: a **new** Solidity contract `RevisionLogV1.sol` deployed at a fresh Base Sepolia address that augments v0's append-only `publishRevision` with **on-chain signature verification** + a **"signer must be a registered device key for `vaultId`" check** + an **event-level `schema_version`** discriminator. v0 (D-014 at `0x8566D3de653ee55775783bD7918Fe91b66373896`; D-015 at `0x74f28794c180bb1BEB698b294F69554D0ACCA9c4`) stays append-only-immutable wherever it is and is NOT touched by this issue. This is the first issue of MVP-2 (master plan §5) and the load-bearing prerequisite for every subsequent chain-side MVP-2 issue (2.2 Entitlement Registry; 2.3 deploy pipeline; 3.1 signed-revision client format; 3.2 device-wallet generation; 4.1 slow-mode chain sync; 4.2 ephemeral indexer).
>
> **Status:** Plan-gate **signed off by Kelvin on 2026-05-13.** All five Q-a..Q-e load-bearing questions resolved (see "Resolved decisions" below). Locked decisions L1..L13 + resolved R-a..R-e together are the non-negotiables. If anything in the "Approach" body drifts from a row in either table, the row wins.
>
> **Security-critical: YES.** Master plan §16.3 lists "contract code" by name. 2.1 (a) ships the first contract that performs on-chain cryptographic verification (it now has a wrong/right answer that an attacker can exploit; v0 had none); (b) introduces the on-chain device-key registry — a brand-new authorisation surface; (c) locks the signature-primitive choice that will propagate into `pangolin-chain::signing` (the canonical-hash construction transfers from v0 per P7 audit HIGH-2; the primitive may not — Path A vs Path B is the load-bearing question Q-a/Q-e); (d) the bytecode is ultimately mainnet-bound (deploys are a separate issue 2.3, but the bytecode this issue ships IS what eventually goes to mainnet). Same plan-gate cadence as 1.4 / 1.9 / 1.10 / 1.11 / P5-1: Kelvin's sign-off on the open questions, then builder → test → adversarial audit → fix-pass as needed → merge with Kelvin approving at the merge boundary. **External audit before MVP-3 mainnet deployment (D-011) is the eventual gate — but 2.1's adversarial pass is the in-house audit it has to clear first.**
>
> **Depends on:** the deployed v0 contract (D-014 + D-015, P5-1 / P5-4 / D-015 redeploy proof — establishes the append-only / no-admin / no-upgrade posture v1 inherits); the v0 invariant test discipline (`invariant_sequenceMonotonic`, `invariant_eventCountEqualsSequence`, `invariant_noStorageMutationBesidesSequence`, `invariant_onlyRevisionPublishedEventEmitted` — v1 amends and extends these); `pangolin-chain::signing` canonical-hash construction (P7; the keccak256-of-fixed-width-fields layout that v1 verifies on-chain — what survives Path A and Path B is the digest layout, not the primitive); `pangolin-chain::evm::derive_evm_wallet` (P7; the Ed25519→secp256k1 derivation that means "one device key" ≡ "one EVM address" — pre-staged so Path B is mechanically possible); master plan §18.7 schema-versioning ladder (event `schema_version` is a chain-event instance of the same policy); the foundry pin discipline (env-quirk #4 — forge v1.0.0, forge-std v1.16.1, solc 0.8.24, evm_version "shanghai"). ## Goal

A self-contained on-chain log that authenticates publishers and tags events with a schema version, while inheriting every v0 hard guarantee (no admin, no upgrade, no `selfdestruct`, no `delegatecall`, no `receive`, non-`payable`, monotonic sequence). Concretely, 2.1 ships:

1. `contracts/src/RevisionLogV1.sol` — the new contract per L1/L2/L3/L4/L5/L6/L7/L9/L10/L11.
2. `contracts/test/RevisionLogV1.t.sol` — unit tests covering every happy path + every error path (bad signature, unregistered signer, replay, malformed `schema_version`, ETH-rejection on all call paths, no-admin selector probe, etc.).
3. `contracts/test/RevisionLogV1.invariant.t.sol` — Foundry invariant tests at ≥10 000 runs × 32-call depth (per `foundry.toml` `[invariant]` defaults; same as v0). The v0 invariants (`sequenceMonotonic`, `eventCountEqualsSequence`, `noStorageMutationBesidesSequence`, `onlyRevisionPublishedEventEmitted`) are amended for the registry-mapping case + the new revert paths.
4. `contracts/script/DeployRevisionLogV1.s.sol` — Foundry deploy script, callable by the same `pangolin-dev` keystore that handled D-014/D-015 (D-006).
5. `contracts/abi/RevisionLogV1.json` — committed ABI; CI ABI-drift check is extended to cover both v0 and v1 (same `diff -u` pattern as the v0 job today).
6. `slither contracts/src/RevisionLogV1.sol --fail-high` reports zero high-severity findings.
7. CI updates: an additional `forge script --tc DeployRevisionLogV1` dry-run step + ABI-drift step + slither step. No change to the v0 jobs.
8. New `THREAT_MODEL.md` entry "Revision Log v1 contract" (replacing the `TBD` placeholder on L57; the v0 row stays unchanged at L65–80).
9. New `docs/architecture/chain-revision-log-v1.md` (or amended `chain-revision-log.md` if a v0 doc exists; verify during build) cataloguing the v1 event schema + the canonical-hash layout + the device-registry semantics + the §18.7 chain-event-schema reject rules + the Q-e/Path-A-vs-Path-B decision and why.

**Non-goals (per L13):** no changes to `crates/pangolin-chain`; no changes to the v0 contract; no production Base Sepolia deploy of v1 (that's 2.3); no mainnet deploy of anything (D-011 / MVP-3); no recovery / revocation surfaces (MVP-3 issue 2.2); no entitlement-registry coupling (issue 2.2 in MVP-2 — separate contract, separate plan-gate); no client-side `pangolin-chain` adapter dual-read (v0 + v1) — that's the integration issue.

## Approach

### 1. The new contract — sketch (pseudocode; final shape rides on Q-a/Q-b/Q-d/Q-e)

```solidity
// SPDX-License-Identifier: AGPL-3.0-or-later
pragma solidity 0.8.24;  // exact-pinned, per L3 + env-quirk #4

/// @title RevisionLogV1
/// @notice Append-only log of encrypted vault revisions, with on-chain
///         signature verification + a per-vault registered-device gate.
///         Same cardinal rules as v0 (no admin, no upgrade, no pause,
///         non-payable, no fallback / receive, no selfdestruct, no
///         delegatecall). The contract has more state than v0 but the
///         "the chain is a log, never an authority" framing is preserved:
///         every check is structural / cryptographic, none is policy.
contract RevisionLogV1 {
    // --- Events ---
    // NOTE topic-0 differs from v0's RevisionPublished — a v0-only
    // client filtering on topic-0 will not pick up v1 events (this is
    // CORRECT behaviour per L11; mixing the two streams would be a bug).
    event RevisionPublished(
        bytes32 indexed vaultId,
        bytes32 indexed accountId,
        bytes32 indexed parentRevision,
        bytes32 deviceId,        // semantics depend on Q-e (Path A: Ed25519 pubkey bytes; Path B: secp256k1 addr left-padded)
        uint16  schemaVersion,   // widened from v0's uint8 per L5
        uint256 sequence,
        bytes   encPayload
    );

    // Future-prep: a *minimal* event emitted on first registration of a
    // (vaultId, signer) pair, so off-chain indexers can quickly enumerate
    // the device set without scanning every RevisionPublished. SHAPE PINNED
    // ONLY IF Q-b RESOLVES TO MODEL (b) — i.e., an explicit registerDevice
    // function exists. Under model (a) self-bootstrap there is no separate
    // event (RevisionPublished's first event for a (vault, signer) IS the
    // registration). DO NOT FINALISE UNTIL Q-b ANSWERED.
    // event DeviceRegistered(bytes32 indexed vaultId, address indexed signer);

    // --- Errors (custom; Solidity 0.8.4+; gas-cheaper than string reverts) ---
    error ErrInvalidSignature();
    error ErrSignerNotRegistered(bytes32 vaultId, address signer);  // signer type rides on Q-e
    error ErrUnsupportedSchemaVersion(uint16 got, uint16 max);
    // Q-b model (b): error ErrCallerNotAuthorisedToRegister(...)
    // Q-b model (c): error ErrRegistrationCallerIsNotAuthority(...)

    // --- Storage ---
    uint256 public nextSequence;  // slot 0 (same as v0; mutated only on successful publish)
    // Slot 1 (sentinel): MAX_KNOWN_SCHEMA_VERSION as a `uint16 public immutable`
    //   so a future v1.1 client can read it from chain. Actual value pinned in Q-d.
    uint16 public immutable MAX_KNOWN_SCHEMA_VERSION;

    // Slot 2+: the per-vault registered-device set. Slot SHAPE rides on Q-e.
    //   - Q-e Path A: mapping(bytes32 vaultId => mapping(bytes32 pubKey => bool)) isRegisteredDevice;
    //   - Q-e Path B: mapping(bytes32 vaultId => mapping(address signer => bool)) isRegisteredDevice;
    mapping(bytes32 => mapping(/*signerKeyType*/ bytes32 => bool)) public isRegisteredDevice;

    constructor(uint16 maxKnownSchemaVersion_) {
        MAX_KNOWN_SCHEMA_VERSION = maxKnownSchemaVersion_;
    }

    // --- External API ---
    function publishRevision(
        bytes32 vaultId,
        bytes32 accountId,
        bytes32 parentRevision,
        bytes32 deviceId,
        uint16  schemaVersion,
        bytes calldata encPayload,
        bytes calldata signature   // 64 B (Q-e Path A) or 65 B (Q-e Path B)
    ) external returns (uint256 sequence) {
        if (schemaVersion > MAX_KNOWN_SCHEMA_VERSION) {
            revert ErrUnsupportedSchemaVersion(schemaVersion, MAX_KNOWN_SCHEMA_VERSION);
        }
        // Recompute the canonical hash exactly as pangolin-chain::signing builds it.
        bytes32 digest = _canonicalHash(vaultId, accountId, parentRevision, deviceId, schemaVersion, encPayload);
        // Verify the signature under the embedded deviceId. Branch rides on Q-e:
        //   Path A: import an ed25519-solidity verifier; call .verify(digest, signature, deviceId).
        //   Path B: address signer = ecrecover(digest, v, r, s); require(signer == address(uint160(uint256(deviceId))));
        bool ok = _verify(digest, signature, deviceId);
        if (!ok) revert ErrInvalidSignature();
        // Q-b model (a) — self-bootstrap:
        //   if (!isRegisteredDevice[vaultId][deviceId]) {
        //       isRegisteredDevice[vaultId][deviceId] = true;
        //   }
        // Q-b model (b) — explicit register: require isRegisteredDevice already true.
        //   if (!isRegisteredDevice[vaultId][deviceId]) revert ErrSignerNotRegistered(vaultId, deviceId);
        // Q-b model (c) — authority-gated: same as (b) here; registration is the
        //   only place that differs.
        sequence = nextSequence;
        unchecked { nextSequence = sequence + 1; }
        emit RevisionPublished(vaultId, accountId, parentRevision, deviceId, schemaVersion, sequence, encPayload);
    }

    // Internal canonical-hash. MUST match pangolin-chain::signing::canonical_hash bit-for-bit.
    // The domain prefix needs to be re-decided in Q-d (v0 of `pangolin-chain::signing` uses
    // the literal "pangolin-chain-signed-revision-v0"; an on-chain v1 verifier may want a
    // distinct prefix like "pangolin-chain-signed-revision-v1" to defeat cross-version replay).
    function _canonicalHash(...) internal pure returns (bytes32) { ... }

    function _verify(bytes32 digest, bytes calldata signature, bytes32 deviceId) internal view returns (bool) { ... }

    // No receive() / fallback() — same as v0. Non-payable means ETH sends revert.
}
```

### 2. Signature verification flow

The contract's `publishRevision` recomputes a canonical hash from the call arguments byte-for-byte the way `crates/pangolin-chain/src/signing.rs::canonical_hash` does it, then verifies a signature over that digest under the embedded `deviceId`. Implementing this on-chain is where Path A and Path B diverge:

- **Path A (Solidity Ed25519 verifier).** Vendor a `forbid(unsafe_code)`-equivalent (Solidity has no unsafe) Ed25519 verifier (e.g., `ed25519-solidity` or an audited fork). Cost ≈ 500k gas per verify; ~$0.01–0.02/verify on Base. Lets the existing `signing.rs` API surface (Ed25519 over the canonical hash, embedded `device_id` = Ed25519 verifying-key bytes) transfer **unchanged**. **The vendoring choice is itself a sub-decision** — pinning a specific commit hash + Apache-2.0/MIT-compat license + an in-tree audit pass. Resolves with Q-e.
- **Path B (secp256k1 via `ecrecover`).** The 3 000-gas precompile. Forces `deviceId` semantics to mean "the secp256k1 EVM address left-padded into a `bytes32`" (or a separate `bytes32` mapping if we want the on-chain identity to be cryptographically distinct from the gas-paying address). The current `signing.rs` is Path-A-shaped (Ed25519 sign + verify); under Path B, MVP-2 issue 3.1 (the signed-revision client format) lands a `secp256k1_signing.rs` companion that signs the same canonical-hash digest under the device's secp256k1 key. **Re-signing existing PoC-era `SignedRevision` records is NOT in 2.1's scope** — they live in PoC `.pvf` files and never made it to chain anyway; MVP-2 3.x decides whether to migrate them or treat them as drop-on-MVP-1-close artifacts.

Whichever path Kelvin picks at Q-e, **the canonical hash itself is identical**. The hash domain prefix may be bumped from `-v0` to `-v1` (sub-decision under Q-d) to defeat any cross-contract replay if v0 ever gets a hypothetical client-side verifier later.

### 3. Device-registration flow

See Q-b for the load-bearing choice. To make the trade-offs concrete:

- **Model (a) — self-bootstrap.** The first `publishRevision` for any `(vaultId, signer)` pair flips `isRegisteredDevice[vaultId][signer] = true` *as a side-effect* of the publish. Subsequent publishes for the same `vaultId` from *other* signers will revert with `ErrSignerNotRegistered`. The bootstrap is trivial (one tx — the first revision). Drawback: there's no on-chain *expression of intent* that a vault is meant to allow signer X; a signer who somehow obtains a Pangolin device key for a vault with no prior on-chain history could "claim" that vault. The mitigation: Pangolin's client-side flow guarantees that the first revision of a vault is published by the device that *created* the vault locally (the per-device `DeviceKey` from MVP-1 issue 1.5; AEAD-sealed under the VDK; never leaves the device); an adversary who hasn't already compromised that device cannot self-bootstrap a vault they don't control. So model (a) is **sound under the assumption that "owns vault" ≡ "physically controls a device whose key is sealed inside the vault"**, which is the same assumption every other Pangolin invariant rests on.
- **Model (b) — explicit `registerDevice(vaultId, newSigner, attestation)`.** A separate function. Requires the caller to be an already-registered signer for `vaultId` (which then has the same chicken-and-egg). The most plausible bootstrap is an `attestation` blob — e.g., a signature by the new device over a "I am the new device for vault X, this is my public key" message — which the *first* device verifies + publishes. But who registers the *very first* device? Either (b.1) a separate `bootstrapVault(vaultId, firstSigner)` callable once per `vaultId` by anyone (collapses to model (a)'s threat profile) or (b.2) ties to the entitlement registry (the same wallet that bought the entitlement is the one that bootstraps — couples 2.1 to 2.2's contract surface, which makes 2.1's plan a dependency rather than a standalone). Model (b) is cleanly extensible to recovery (the recovery contract MVP-3 2.2 can call `registerDevice` for a new device after authority rotation) — but recovery doesn't exist yet, so the wiring is theoretical at this point.
- **Model (c) — authority-rotation.** A `vaultAuthority[vaultId]` slot stores the current Whitepaper §F vault-authority key; that key is the only thing that can register / revoke devices. Bootstrap: first-vault-create publishes the authority. Revocation: the authority signs an "unregister X" message. Recovery: the recovery contract rotates `vaultAuthority[vaultId]` to a new key. **This is the Whitepaper §F design fully realised on-chain** but it's a much larger surface for 2.1, and the recovery contract that's supposed to *use* the rotation primitive doesn't ship until MVP-3.

The plan recommends **model (a) self-bootstrap as the default unless Kelvin overrides**, on the grounds that (i) it's the smallest surface; (ii) the threat profile rests on assumptions Pangolin already makes elsewhere; (iii) model (b) and (c) re-introduce admin-like surfaces (the question "who can call `registerDevice`?" has the same shape as "who can call `pause()`?" — an answer Pangolin explicitly avoids); (iv) recovery-aware revocation lands cleanly on top of (a) in MVP-3 (the recovery contract publishes a "vault X's signer set is now {Y, Z}" event that off-chain clients honour; the v1 contract itself stays append-additive). But this is Kelvin's call.

### 4. Threat-model interactions

| Invariant | v1's posture |
|---|---|
| **#1 No plaintext writes to disk** | Unchanged — v1 sees only ciphertext, identical to v0. The chain is still a log of encrypted blobs. |
| **#2 Relay/funder/indexer cannot forge revisions** | **Strengthened.** v0 had zero on-chain authentication (any address could call `publishRevision` with arbitrary bytes; client-side AEAD AAD discipline was the only defence). v1 adds on-chain signature verification + per-vault device-key gate — *the relay/funder/indexer literally cannot publish a revision*. The single biggest security upgrade vs v0. |
| **#3 Indexer cannot decrypt** | Unchanged — `encPayload` is still opaque ciphertext from the contract's POV. |
| **#5 Session authority never stored on-chain** | Preserved — the `isRegisteredDevice` mapping stores device-key identifiers (Ed25519 pubkey OR secp256k1 address — neither is a "session authority"; both are long-lived device-identity material that already lives in the local vault per MVP-1 1.5). |
| **#11 §18.7 ladder** | **Extended.** Chain-event `schema_version` joins the ladder. A v1.0 client reading a v1.1 event with `schemaVersion > MAX_KNOWN_SCHEMA_VERSION_CLIENT_SIDE` rejects per #11 (the contract also rejects on write — L5/L6 — so the asymmetry is "contract rejects unknown-future writes; client rejects unknown-future reads"). Documented in §18.7's ladder catalogue. |
| **#12 AEAD AAD covers every disambiguating byte** | Preserved. The contract verifies a signature over a canonical *hash* of metadata + ciphertext; it does not re-verify the AEAD AAD. The AAD discipline stays in `pangolin-store::seal_account` upstream; v1 is downstream of that boundary. |

**New per-surface threats** (for the `THREAT_MODEL.md` v1 row):

1. **Adversary captures a signed revision in transit + replays it.** Defense: signed revision binds `parent_revision`; a replayed revision against a moved-on head fails ingestion at the client per existing 1.6 lineage discipline. **The contract itself does NOT prevent replay at the chain level** — re-publishing the identical bytes succeeds and assigns a new sequence number. This is by design: the contract is a log, not an authority. Documented residual; matches v0 posture.
2. **Adversary publishes a valid signature from a registered device for vault X with an `encPayload` that wasn't actually authored by that device.** Defense: the canonical hash binds `keccak256(encPayload)`, so the device's signature only covers a digest derived from the specific ciphertext bytes. An attacker who substitutes different ciphertext invalidates the signature. **What this does NOT defend against:** a *compromised* device key publishing forged-but-validly-signed revisions — that is the residual that recovery (MVP-3) addresses via authority rotation. Documented.
3. **Front-running a self-bootstrap (Q-b model (a)).** A pending publish for a fresh `vaultId` becomes visible in the mempool; an attacker copies the bytes into their own tx with a higher gas price. **Defense: the signature in the captured bytes was made by the original device's key, which the attacker doesn't have. Re-broadcasting the same signature with a different `msg.sender` does not change `recoverSigner(digest, signature)`. The attacker has nothing to forge.** The only "win" the front-runner gets is paying gas for the user's tx, which is harmless. Documented.
4. **A v1.0 client refusing to read a v1.1 event because `schemaVersion > MAX_KNOWN`.** Per §18.7 invariant #11 this is correct behaviour, not a bug. The user sees "this revision requires a newer Pangolin." Same posture as the per-account `payload_version > MAX` reject. Documented.
5. **Bytecode size + Solidity-Ed25519 vendoring cost (Path A only).** Vendoring an Ed25519 verifier expands the contract's bytecode size dramatically (~10-20 KiB depending on the implementation chosen — vs v0's 443 bytes). Audit risk increases linearly with vendored code. Mitigations: pin an audited commit; `slither --fail-high` over the *whole* contract including the vendored verifier; reproduce the bytecode bit-exactly across two independent builds (env-quirk reproducibility). **Documented as the load-bearing engineering risk of Path A.**
6. **Permanent contract corruption (storage attack) — registry mapping edition.** Defense: the only writes to `isRegisteredDevice` are inside `publishRevision`'s success path; no admin / no admin-keyed setter / no zero-out / no reset. Probed by the amended `invariant_noStorageMutationBesidesSequenceAndRegistry` test — every non-whitelisted slot remains zero, AND every whitelisted-mapping-slot was written by a legitimate fuzzed publish (cross-checked against the handler's record of registrations).
7. **Anvil / Base Sepolia behaviour drift on `ecrecover` (Path B only).** Defense: `ecrecover` is a standardised EVM precompile at `0x01`; behaviour is identical across Anvil / Base Sepolia / Base mainnet / Ethereum L1 at every fork ≥ Frontier. No path B-specific drift risk.

## Open questions for Kelvin

These are the questions that need single-sentence answers before the builder runs. Each one **changes the implementation shape**, not just the wording.

### Q-a · Signature hash format

**EIP-712 typed-data hash** (interoperable with standard wallets / hardware signers / WalletConnect / standard tooling — anything that can already EIP-712-sign can sign a Pangolin revision; comes with a free domain separator with `name`/`version`/`chainId`/`verifyingContract`) **vs domain-separated `keccak256` of a canonical encoding** (smaller / cheaper / bespoke — what `crates/pangolin-chain/src/signing.rs` already builds today, with the `"pangolin-chain-signed-revision-v0"` literal prefix and the 161-byte body).

The Whitepaper §D1 does NOT pin this. The current `signing.rs` is bespoke (and rationalised on the grounds that "the same digest is computed on-chain and off-chain"; an EIP-712 hash would also have that property). The tradeoff: EIP-712 lets the device wallet show a human-readable "Pangolin: publish revision for vault 0x…" prompt in any wallet UI (relevant if Path B / secp256k1 happens — device wallets are secp256k1, hardware wallets speak EIP-712 natively, would also matter for browser-extension-mediated publishes in MVP-4+); bespoke gives ~150 fewer gas per verify + simpler audit (one Solidity function vs the EIP-712 typehash machinery) + matches what's already built in `signing.rs`.

**Recommendation if no override: bespoke (= what `signing.rs` already does)** — because (a) Pangolin devices don't use standard wallets (per D-006 the device's own keypair signs + pays gas; there is no MetaMask in the loop); (b) `signing.rs` is already merged; (c) EIP-712 buys interop Pangolin doesn't need. But Kelvin may want EIP-712 anyway for the MVP-4 browser-extension story.

**Kelvin's call:** EIP-712 typed-data hash, or bespoke domain-separated keccak256?

### Q-b · Device-registration model

Pick one of the three models documented under "Approach §3" above:

- **(a) Self-bootstrap** — first publish for a fresh `(vaultId, signer)` self-registers; thereafter `ErrSignerNotRegistered`. Smallest surface. No on-chain "authority" concept.
- **(b) Explicit `registerDevice(...)`** — separate function; caller must be an already-registered signer; bootstrap is a separate (b.1) `bootstrapVault(vaultId, firstSigner)` callable once per vault OR (b.2) entitlement-registry-gated.
- **(c) Authority-rotation** — Whitepaper §F realised on-chain; `vaultAuthority[vaultId]` slot; recovery rotates the authority. Larger surface; depends on recovery substrate that doesn't ship until MVP-3.

**Recommendation if no override: (a) self-bootstrap.** Smallest surface; threat profile matches existing assumptions; cleanly extensible by (c)-style recovery later. But (a) cannot express "this device is revoked" without recovery; if Kelvin wants any in-v1 revocation, (a) is wrong.

**Kelvin's call:** (a), (b.1), (b.2), or (c)? — and if (b.2), is 2.1 willing to depend on 2.2 (Entitlement Registry) being plan-locked before 2.1 builds?

### Q-c · Revocation in v1, or deferred to MVP-3?

Confirm: **revocation is deferred to MVP-3 (recovery)** and v1's storage layout leaves room (e.g., the mapping value could later widen from `bool` to a `uint8` status enum without a v2 redeploy if Kelvin wants per-device revoke later; mapping slots are address-hashed not slot-indexed so adding a status field is forward-compatible without disturbing slot 0/1/2).

### Q-d · Event `schema_version` starting value

v0's events have no `schema_version` field. v1 introduces one. Two valid choices:

- **`1`** — v1 events start at schema-version 1 explicitly; treat v0 as "absence of the field" (no mapping from v0's events to a v1 schema-version value, because they're different event signatures with different topic-0s anyway).
- **`0`** — treat the v1.0 event format as "version 0 of the v1 contract's event schema." Future v1.1 extensions go to 1, 2, etc.

The two are interchangeable for the contract logic (`MAX_KNOWN_SCHEMA_VERSION` is set in the constructor regardless). The choice affects how off-chain code (the `pangolin-chain` adapter, `pangolin-indexer`) discriminates events. **Recommendation: `1`**, on the grounds that "the v0 contract emitted events implicitly at v0; the v1 contract emits at v1; bumping monotonically across contract generations gives us one global axis." Documentation problem: explicit-vs-absent vs implicit-vs-bump-counter. Kelvin's call.

Sub-decision riding on this: **the canonical-hash domain prefix string**. Currently `pangolin-chain::signing::SIGNED_REVISION_DOMAIN = "pangolin-chain-signed-revision-v0"`. Options for v1's on-chain verifier: (i) keep `-v0` (every PoC-era signed revision verifies under v1 too — *desired* if Kelvin wants v0 PoC revisions to be MVP-2-usable; *undesired* if Kelvin wants cross-version replay defeated by the prefix); (ii) bump to `-v1` (every existing signed revision in `pangolin-store` becomes un-republishable on v1 without re-signing; matches the event `schema_version=1` answer). **Recommendation: bump to `-v1`** for clean separation, with the understanding that the PoC-era `SignedRevision` records in `.pvf` files will need re-signing on MVP-1→MVP-2 vault migration (MVP-2 issue 3.1's scope). But this is the kind of decision that wants Kelvin's explicit yes.

**Kelvin's call:** event `schema_version` starts at 1 or 0; canonical-hash domain bumps to `-v1` or stays `-v0`.

### Q-e · Chain-key parity with the existing device_id (THE big one)

The Rust `pangolin-store::device_key.verifying_key` (Ed25519) is the in-vault device identifier per MVP-1 1.5. The on-chain v1 signer (ECDSA over secp256k1 because Solidity) under Path B is a *different* key. The P7 module already implements `derive_evm_wallet`: Ed25519 → HKDF → secp256k1 wallet, deterministic, **one device → one EVM address**.

Three answers:

- **(a) Same key, two formats** — Path A. v1 verifies Ed25519 on-chain (~500k gas / verify; viable only on L2). `device_id` = Ed25519 verifying-key bytes (32 B), exactly what `signing.rs` already builds. **Pro:** the existing `signing.rs` API surface transfers unchanged; the existing P7 audit HIGH-1 cryptographic argument carries through; one device-key concept, two on-chain incarnations (Ed25519 for revision-signing, derived secp256k1 for gas-paying). **Con:** Path A requires vendoring a Solidity Ed25519 verifier (~10-20 KiB of audited code; non-trivial supply-chain surface); L1 mainnet is not viable (~$25-50/verify); locks Pangolin to L2 mainnet (Base is the locked-in chain per D-005 so this is fine *for now* but loses optionality).
- **(b) Derived from one master seed** — Path B with `derive_evm_wallet` producing the on-chain identity. The Ed25519 device key stays the in-vault identity; the derived secp256k1 wallet is BOTH the on-chain signer AND the gas payer. `device_id` (on-chain) = the secp256k1 EVM address left-padded into `bytes32` (or stored as `address`). **Pro:** uses Ethereum's native `ecrecover` (3 000 gas, ~150× cheaper); usable on L1 mainnet; matches D-006 ("each device's keypair is both the revision signer (verified by contract) and the gas payer") *literally* — same key signs and pays. **Con:** requires a new `crates/pangolin-chain/src/secp256k1_signing.rs` (or a `Signer` trait abstraction); the on-chain `device_id` is *not* the same bytes as the in-vault `device_id` (the in-vault `device_id` stays Ed25519; the on-chain one is the derived secp256k1 address); MVP-2 issue 3.1 has to specify the mapping. The P7 HIGH-1 cryptographic argument *survives* (HKDF is one-way; leaking the secp256k1 doesn't leak the Ed25519 seed) but the cross-key correlation is on-chain and observable (anyone watching the chain learns "all revisions from secp256k1-address-X share an Ed25519-device-key" — same posture as the existing `evm.rs` privacy tradeoff documented in `THREAT_MODEL.md` `pangolin-chain` row #6).
- **(c) Totally independent** — the on-chain identity is a fresh secp256k1 key generated separately on first run (NOT derived from the Ed25519 seed); a registration step maps Ed25519 ↔ secp256k1. **Pro:** maximum key separation; a leaked on-chain key reveals nothing about the in-vault key. **Con:** doubles the device-key management surface; requires a second key in secure storage; complicates 3.2 device-wallet generation (now generates *two* keys); contradicts D-006's "same keypair signs + pays gas" framing.

**Recommendation: (b) Path B + derive_evm_wallet** — because (i) it matches D-006 literally; (ii) the P7 derivation already exists, tested, audited; (iii) `ecrecover` is the cheapest verifier; (iv) preserves L1 optionality even if Base stays the locked mainnet; (v) the privacy tradeoff is already in scope (#6 in the `pangolin-chain` threat row). **(a) is the only choice that lets v0-era `SignedRevision` records re-publish on v1 unchanged** — if Kelvin values "MVP-1 vaults can publish their PoC revisions on v1 without re-signing," (a) wins. (c) is unattractive on every axis.

This is the single biggest decision-shape question — affects 3.1 (signed-revision format) and 3.2 (device-wallet generation) directly, and decides whether `crates/pangolin-chain` gets a new module or stays as-is.

**Kelvin's call:** (a), (b), or (c)?
