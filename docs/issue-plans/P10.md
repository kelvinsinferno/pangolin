# Issue P10: Tombstones & Offline Mode

> **Status:** PLAN-gate. Per Kelvin's autonomy directive, dispatching the build directly upon §16-Plan-Gate approval.
> **Security-critical:** YES per §16.3. P10 closes the last open structural gap in the PoC's revision graph (the `is_tombstone_i64 = 0` hardcode in `Vault::ingest_chain_revision` flagged by the P8 audit as the CRIT-1 origin) and pins the offline-edit-then-online-sync invariant — both of which are load-bearing for Cardinal Principle 4 ("never silent merge"). A flaw in tombstone-bit propagation either silently un-deletes a tombstoned account (the chain says the account was deleted but the local store displays it as live) or silently drops an account the user expected to keep (the local row gets a tombstone bit that wasn't user-attested). A flaw in the offline-edit invariant violates Cardinal Principle 1's spirit ("core decides") by tying the user's ability to make local edits to network reachability.

## Spec reference

- **Master plan §3.7 EPIC: Conflicts & Tombstones (P9–P10):**
  - `P10-1` Tombstone payload: `{ "deleted": true, "account_id": ..., "tombstoned_at": ... }` encrypted normally.
  - `P10-2` Offline test: lock vault, kill network, edit, unlock, work, reconnect, sync. No data loss.
- **Master plan §3.5 Cardinal Principles:**
  - Principle 1 — Rust core is the only security-critical component; **edits MUST succeed locally without network**.
  - Principle 2 — No plaintext leaves the device. Tombstone-bit decoding happens in-memory at ingest, plaintext wiped on drop, encrypted ciphertext is the only at-rest form.
  - Principle 3 — Blockchain is a log, never an authority. Offline edits queue locally and replay deterministically; a chain outage is operationally identical to chain unreachability.
  - Principle 4 — Append-only state, never silent merge. Tombstone is an explicit append; resurrection is an explicit add of a new `account_id`. The bit propagated at ingest is content-derived (the AEAD-decoded plaintext), never a chain-side signal.
- **Master plan §3.8 PoC exit criteria:**
  - "Two vaults converge on the same revision graph after publish + pull." — P8 (no-conflict) + P9 (post-fork) closed the live-snapshot path. P10 closes the same invariant for tombstone heads in the own-publish round-trip case.
  - "All edits queue locally when offline; reconnect publishes deterministically." — P10 is the explicit pin of this exit-criterion line.
- **DECISIONS.md:**
  - **D-005** — Base is the mainnet target; PoC runs against Base Sepolia at `0x8566D3de653ee55775783bD7918Fe91b66373896` (D-014). Offline mode is implemented via the `MockChainAdapter`'s disconnect toggle for the automated test; the production flow is "user edits while their device is offline; `pangolin-cli publish` errors out cleanly with a `ChainError::Rpc` that the operator can read."
  - **D-006** — No relay; each device is signer + gas payer. The PoC two-key deviation (P8 §A7, P9 §A7) carries forward into P10 unchanged. P10 introduces no new chain-side primitive; tombstone events use the same `RevisionLogV0::publishRevision` path as live revisions.
  - **D-007** — No persistent indexer; offline-mode reconnect uses the same `pull_since` + `publish_all` primitives the rest of the PoC uses.
- **Cardinal Principle 4 ("never silent merge"):** P10 closes the bit-propagation half of this. P9 closed the conflict-resolution half. Together they are the load-bearing guarantee that Pangolin never silently turns a deletion into an undeletion (or the reverse) at any boundary in the system.
- **Whitepaper §6 (account model):** the tombstone is an append-only revision whose payload structurally signals "deleted," not a row removal. P10 makes the on-chain → local round-trip of that signal correct under PoC two-key.
- **Whitepaper §7 (revision lineage):** `genesis → edit → tombstone` is structurally final for the account lifecycle; resurrection is a new account_id. P10 enforces the no-resurrection rule in `Vault::add_account`.

## What P0..P9 already shipped (the load-bearing surface for P10)

P10 introduces no new cryptographic primitive and almost no new chain-side code. It composes existing pieces in two specific ways:

- **P3 — `revisions.is_tombstone INTEGER` schema column** (`crates/pangolin-store/src/schema.rs`). Already in every vault file. The plumbing is in place; what's missing is an ingest path that sets the column to `1` when the chain-event payload AEAD-decodes to a tombstone marker.
- **P3 — `Vault::delete_account`** (vault.rs L920–L1015). Already writes a tombstone revision with `is_tombstone = 1` and seals the marker via `blob::seal_tombstone`. The current marker payload is `{ "deleted": true }` (a single-entry CBOR map), NOT the master plan §3.7 P10-1 shape `{ "deleted": true, "account_id": ..., "tombstoned_at": ... }`. P10 widens the payload schema and updates `delete_account` to use it.
- **P3 — `blob::seal_tombstone`, `blob::open_payload`, `blob::DecodedPayload::Tombstone`** (blob.rs L168–L199). The seal + open primitives already exist. P10 widens the `encode_tombstone_cbor` / `decode_payload` round-trip to handle the three-field payload.
- **P8 — `Vault::ingest_chain_revision`** (vault.rs L2810–L3085). The ingest path that hardcodes `is_tombstone_i64 = 0` (vault.rs L2988). The CRIT-1 audit finding identified this as the structural origin of the freeze sentinel ("we can't tell if the chain event is a tombstone, so we freeze and let the user resolve"). P10 fixes the underlying gap: when the AEAD plaintext IS decryptable (own-publish round-trip case), inspect it for the tombstone marker and set the bit accordingly. Foreign-ingested events still freeze (placeholder zero nonce → AEAD open fails → can't tell → keep `is_tombstone = 0` → freeze sentinel still fires from the existing P8 code path).
- **P8 — `frozen_pending_resolve` sentinel.** Foreign-ingested events trigger the freeze regardless of tombstone-bit — the freeze is the user-facing safety net. P10 does not change this; P10 makes the bit correct for the case where the AEAD CAN open (own-publish + cross-vault under MVP-1 single-key model).
- **P8 — `pangolin-cli sync` orchestrator** (sync.rs). `publish_all` walks the dirty list; `pull_all` ingests events. Tombstone revisions ride through these primitives identically to live revisions — they are signed-revision rows that happen to have `is_tombstone = 1`. The dirty marker mechanism queues them under network failure; on reconnect, the same `publish_all` flushes them. **No new code in `sync.rs`** for the offline-mode path; P10's contribution is a test that validates the invariant against a deliberate disconnect.
- **P9 — `Vault::build_merge_payload_for_resolve`** (vault.rs L1505–L1595) — already has a tombstone branch that re-seals via `seal_tombstone` if the chosen revision is a tombstone (P9 §A5). P10 reaffirms this — the branch is correct AS-WRITTEN; the only adjustment is that `seal_tombstone` will now seal the wider payload schema, which the merge-revision-build automatically inherits.
- **P9 — `Vault::read_payload_plaintext_for_resolve`** (vault.rs L1360–L1469). Already returns `StoreError::Cbor` on tombstone-revision input ("resolve flow must use seal_tombstone for tombstone heads"). P10 carries this forward unchanged; the resolve flow's tombstone path is via `build_merge_payload_for_resolve`, which detects `is_tombstone` from the revisions row, NOT via the plaintext reader.

P10 introduces **two surfaces**: (a) the `TombstonePayload` struct + ciborium-ll round-trip helpers + the widened `seal_tombstone` / `open_payload` round-trip, and (b) the `offline_edit_then_online_publish` integration test + E2E-005 documentation entry.

## Goal

Ship the structurally-correct tombstone-bit-propagation path through `Vault::ingest_chain_revision` (own-publish round-trip case under PoC two-key; foreign-ingest case under MVP-1 single-key) AND pin the offline-edit-then-online-sync invariant with a deliberate-disconnect integration test. Together these close the last two §3.8 PoC exit criteria — own-publish tombstone round-trip is now as cleanly testable as own-publish live-revision round-trip; offline-edit queueing is now a covered scenario rather than an implicit assumption.

## Architectural decisions (locked in this plan)

These decisions are committed to as part of plan-gate approval. Any deviation during BUILD requires returning to PLAN per §16.4.

### A1. Tombstone payload encoding — deterministic CBOR, three fields, fixed key order

**Decision:** The widened tombstone payload is a deterministic three-entry CBOR map with text keys in **alphabetical order** (`account_id` then `deleted` then `tombstoned_at_ms`). `account_id` is a CBOR Bytes value of length `ACCOUNT_ID_LEN` (32). `deleted` is the CBOR `true` simple value. `tombstoned_at_ms` is a CBOR positive integer (`Unsigned`). The legacy P3-era single-entry `{ "deleted": true }` payload **continues to decode** (as `TombstonePayload { deleted: true, account_id: [0; 32], tombstoned_at_ms: 0 }`) so existing vault files keep opening cleanly.

**Rationale:**

1. **Forward-compat with MVP-1.** A schema-bumped tombstone with additional fields (e.g., `device_id` for forensic attribution — open question §8.2) lands as a four-entry map; alphabetical key order means future fields slot in deterministically. The `decode_payload` dispatch on `entries == 1 || entries == 3` is explicit; `entries == 2` and `entries >= 4` reject. MVP-1's potential schema-bump becomes an additive entry-count arm.

2. **Cryptographic AAD discipline.** The AEAD AAD already binds `(vault_id, account_id, parent_revision, schema_version)`. Adding `account_id` and `tombstoned_at_ms` to the *plaintext* doesn't re-bind anything cryptographically — the AAD path is unchanged. The plaintext fields are a defense-in-depth layer for deserialization (the `account_id` field MUST equal `event.account_id` at decode time; mismatch is rejected inside `detect_tombstone_bit_at_ingest` via a constant-time `subtle::ConstantTimeEq::ct_eq` comparison and silently collapses to `is_tombstone = 0` — the same bucket as AEAD failure / CBOR failure / locked vault, preserving the non-oracle property). This catches a class of errors where the AAD passes (e.g., an attacker who knows the AAD components but not the AEAD key has no leverage; an attacker who has the AEAD key has already won regardless of payload shape) but the plaintext was malformed at seal time. **Audit reviewers should scrutinize: is the cross-check of `payload.account_id == aad.account_id` actually meaningful, or is it cargo-culted?** Recommendation: ship the cross-check because (a) it's free, (b) it documents the intended invariant, (c) future MVP-1 forensic-attribution schemas may add a `device_id` field that DOES need plaintext-level cross-checks against AAD-bound metadata. **Implementation note (P10 fix-pass M-1+M-2):** the cross-check lives in `detect_tombstone_bit_at_ingest` — NOT in `decode_payload` — so the decoder remains oracle-free at the type level (decode succeeds for any well-formed payload; the ingest layer chooses whether to act on the decoded value). The decoder error variant for a wrong-length account_id field is still `StoreError::Cbor` per the field's structural shape requirement.

3. **The `tombstoned_at_ms` field is non-load-bearing for security but load-bearing for forensics.** A user reading the tombstone history of an account wants to know when it was tombstoned. The local row's `last_modified_at` is one source; the in-payload timestamp is a second, tamper-evident source (covered by AEAD). For PoC, both are sealed at the same `current_unix_ms()` value at `delete_account` time. MVP-1's KDBX import / cross-device sync may diverge them (e.g., if a user deletes on device A at T1 and the chain event is ingested on device B at T2, the local row's `last_modified_at` reflects T2 but the payload's `tombstoned_at_ms` reflects T1). Documenting the field now lets MVP-1 use it without a schema bump.

**Alternative considered:** Bare-bytes payload (`[0xFF; 32]` + account_id + timestamp, fixed 73-byte structure). **Rejected** because it loses CBOR's self-describing audit trail — a future schema reader inspecting a leaked vault file with `cbor-debug` can decode the live-snapshot map *and* the tombstone map with the same tooling. Bare bytes would require a separate decoder.

**Alternative considered:** Stick with the legacy `{ "deleted": true }` single-entry shape; ignore the master plan's three-field schema. **Rejected** because the master plan §3.7 P10-1 is explicit on the shape and the compatibility cost of widening now (one round of seal/open + decoder-arm coverage) is dwarfed by the cost of widening later (every vault file in production carries the legacy shape forever; any future tombstone-attribution feature is gated on a schema bump). PoC pays the cost up-front.

### A2. Tombstone-bit set inline in `ingest_chain_revision` (opportunistic decryption)

**Decision:** The tombstone-bit is detected and set INSIDE `Vault::ingest_chain_revision`'s genuine-foreign-INSERT branch by attempting to AEAD-open the `event.enc_payload` under the local VDK with the placeholder zero nonce. Three outcomes (per §2 row 2 above): (a) decrypts to tombstone → set bit + flip `account_identities.tombstoned = 1`; (b) decrypts to live → bit stays 0 (existing behavior); (c) decrypts FAILS → bit stays 0; freeze sentinel still fires (existing P8 behavior).

**Rationale:**

1. **Locality.** The freeze sentinel pathway already touches the row in the same transaction. Adding the bit-detection inline means one transaction for both writes; no separate scan pass; no possibility of bit-set-but-freeze-clear (or freeze-set-but-bit-clear) intermediate states. The `BEGIN IMMEDIATE … COMMIT` envelope from P8 is preserved.

2. **PoC two-key reality.** Under PoC, the foreign-ingest case has a placeholder zero nonce — the AEAD will fail to open. So tombstone-bit-detection works ONLY for the own-publish round-trip case (where the chain event is the local device's own previous publish; the local row already has the real nonce; the new ingest hits idempotency arm #2 or #3 and never enters the genuine-foreign-INSERT branch in the first place). Wait — let me reconsider. The own-publish case lands in idempotency arm #2 (`(account_id, chain_tx_hash, block, log)` after `mark_published`) and never enters this branch. Under what case DOES this branch fire WITH a decryptable payload?

3. **Reconsidered scope.** The genuine-foreign-INSERT branch fires when none of the three idempotency arms hit. Under PoC two-key, this is "vault A creates account X, publishes; vault B copies vault A's file and pulls" (the cross-vault case — same VDK because B copied A's file). In that case, vault B's VDK CAN decrypt vault A's payload (same key). But vault B doesn't know vault A's nonce — the ingest path stores a placeholder zero nonce. So `open_payload(B's_VDK, placeholder_zero_nonce, A's_ciphertext, AAD)` will fail because the nonce is wrong, NOT because the key is wrong. Tombstone-bit-detection therefore fails in the PoC cross-vault case too. **The opportunistic decode succeeds only in the case where the chain event's enc_payload happens to be sealed under a placeholder-zero-nonce-knowable-by-AEAD-failure-of-the-wrong-key — which is impossible under correct seal practice.** So under PoC, tombstone-bit-detection in `ingest_chain_revision`'s genuine-foreign-INSERT branch **always sets bit = 0**. The fix is structurally correct but functionally a no-op under the PoC two-key model.

4. **Why ship it anyway.** (a) The hardcode `is_tombstone_i64 = 0` is currently a §3.5 audit-flagged comment ("Per Q's: P10 owns tombstone semantics"). Removing the comment + replacing it with the opportunistic-decode logic is the structurally honest fix; the comment is a TODO pointing at this exact issue. (b) MVP-1 ships nonce-on-chain (the `RevisionPublished` event will carry the nonce as part of the schema-version-bumped event ABI), at which point this branch CAN decrypt foreign events and the bit-set becomes functional without a code change. (c) The own-tombstone round-trip case CAN be exercised in a test: a fresh vault, no idempotency arms hit on a synthetic `RevisionEvent` whose `enc_payload` is the vault's own seal of a tombstone payload under a known nonce that the test passes through `event.enc_nonce`-equivalent metadata. Wait, the chain event has no nonce field. So the test for `ingest_own_tombstone_sets_tombstone_bit_on_ownpub_round_trip` must construct the round-trip via the regular publish + pull primitives (the local row has the real nonce; the chain event has the ciphertext; ingest's idempotency arm #2 stamps the chain anchor on the local row WITHOUT going through the genuine-foreign-INSERT branch). **So the test pins the bit-already-set-on-local-row case** (the `delete_account` write set `is_tombstone = 1` directly; ingest just stamps the chain anchor; no opportunistic-decode happens). That's a structurally honest test of the round-trip but it does NOT exercise the new opportunistic-decode logic.

5. **The opportunistic-decode logic IS exercised by a synthetic test** (`ingest_foreign_event_with_unreadable_payload_leaves_tombstone_clear_and_freezes`) that constructs a `RevisionEvent` foreignly (e.g., from a different `MockChainAdapter` whose vault_id matches but whose payload was sealed under a different VDK), calls `ingest_chain_revision`, and asserts `is_tombstone = 0` (decode failed) AND `frozen_pending_resolve = 1` (freeze fired). The "would-have-set-the-bit-if-the-decode-succeeded" branch is exercised by `ingest_synthetic_decryptable_tombstone_event_sets_bit` (the synthetic ciphertext is sealed under THIS vault's VDK with the placeholder zero nonce that ingest will use for the open attempt — a deliberately-constructed scenario that bypasses how chain events actually look in production but DOES exercise the new logic). This second test is the load-bearing one for P10-2 acceptance.

**Alternative considered:** Defer tombstone-bit-detection entirely until MVP-1's nonce-on-chain ships; leave the hardcode as-is. **Rejected** because the master plan §3.7 P10-1 explicitly carves out tombstone-payload as a PoC deliverable; "the bit is hardcoded to 0 forever" is not consistent with that.

**Alternative considered:** Move tombstone-bit-detection to a separate `Vault::scan_for_tombstones()` pass invoked by the CLI's `pull` subcommand after `pull_all`. **Rejected** because (a) it doubles the transaction surface (two scans = two BEGIN IMMEDIATEs = two crash windows), (b) it's not idempotent without extra bookkeeping (a re-scan would re-decrypt every row), (c) the inline approach is one branch in an existing function with one new test case.

### A3. Decryption-failure handling — silent fallback to bit=0 + freeze

**Decision:** When the opportunistic AEAD-open inside `ingest_chain_revision` fails, the path silently falls back to `is_tombstone = 0` and lets the existing freeze sentinel fire. **No third state.** No "potentially-tombstoned, can't read" intermediate flag.

**Rationale:** A third state would propagate through every read path (`list_accounts`, `get_account`, `reveal_password`, `export_payload`, the resolve path, the search index, etc.) and require a UX surface ("this account might be deleted; we can't tell"). The freeze sentinel already provides exactly this UX ("this account needs your attention; resolve it to find out what happened"). Adding a third state increases the audit surface for marginal user-visible benefit; freeze is the single mid-ground state in the system, and centralizing the "uncertainty about an account's state" UX through it keeps the audit story tight.

**Subtle correctness concern acknowledged:** If a future codepath decides that "freeze + bit=0 means the account is alive but contested," and another codepath decides "freeze + bit=1 means the account is deleted but contested," then the absence of "freeze + bit=?" leaves a hole. P10's stance: under PoC two-key, the bit on a frozen row is **structurally meaningless** (we couldn't decode the payload), so the bit's value is a side-effect of "did the local store happen to have the right nonce for this row" rather than a chain-attested signal. The resolve flow (P9 §A5) reads `revisions.is_tombstone` for tombstone-merge dispatch — that's why the bit must be correct for the **resolved** row, not for the **frozen** row. The frozen row is replaced (in the head pointer) by the merge revision, and the merge revision's bit IS user-attested via the resolve flow's `--keep` choice (resolving to a tombstone produces a tombstone merge per P9 §A5). So the bit-on-frozen-row is dead code from the user's perspective; correctness only matters for the post-resolve-merge-row case, which is local-write-controlled.

**Alternative considered:** Add a `Vault::ingest_chain_revision` outcome variant `IngestOutcome::InsertedFrozenUndecodable` that signals to the caller "we couldn't decrypt; freeze sentinel set." The caller (in this case `pull_all`) would log/surface this as a stronger UX signal than the current `IngestOutcome::Inserted`. **Rejected** because the freeze sentinel is already surfaced in `PullReport.frozen_count`; adding a parallel "undecodable" count is duplicative.

### A4. Resurrection of a tombstoned account — forbidden

**Decision:** `Vault::add_account` refuses with `StoreError::AccountTombstoned` if the (randomly-derived) `account_id` already exists in `account_identities` AND that row's `tombstoned = 1`. Under PoC the random-32-via-sqlite-derived `account_id` makes this collision cryptographically negligible (the 256-bit account_id space dwarfs any plausible vault size), so the guard is **defense-in-depth + spec compliance** with the P3 invariant ("tombstones are append-only; resurrection is a new account_id"). MVP-1 may revisit if a user-feature for "I deleted this and want to bring it back" emerges; the path under PoC is "create a new account with the same plaintext fields; the new account has a new `account_id`."

**Rationale:**
1. **Append-only.** Cardinal Principle 4 says append-only state with no silent merges. A tombstone followed by a fresh `add_account` with the same `account_id` IS a silent merge (the audit trail has "deleted at T1; created at T2" as the same logical entity, contradicting the linear-history model).
2. **Cross-device confusion.** Under PoC two-key, vault A tombstones account X; the chain event lands; vault B ingests; the freeze sentinel fires. If vault A then re-creates account X with the same `account_id`, vault A publishes a fresh genesis revision for X — vault B sees TWO genesis revisions for the same account_id (the original genesis from P3 onboarding, and the new genesis from the resurrection), which `account_heads` reports as a multi-head fork plus a tombstone in the lineage. The user-facing surface is "we have an account that's both deleted and recently created," which is incoherent. Forbidding resurrection at `add_account` time prevents the publish; the user is forced to use a new `account_id` (= a new `add_account` call generates a fresh random 32-byte id), which lands as a new account in vault B's store with no lineage tie.
3. **Implementation is one extra SQL query.** `Vault::add_account` already calls `random_32_via_sqlite(&self.conn)` to derive `account_id`. We add a SELECT for `account_identities.tombstoned WHERE account_id = ?1`; if `Some(1)` returned, error out and (defensively) re-derive. Under random 32-byte derivation, the post-error retry is essentially guaranteed to find a fresh id. **For audit-cleanliness, the retry loop is bounded at 4 attempts**; if all 4 random ids land on tombstoned rows (probability ≈ 4 × N / 2^256 where N is the tombstone count, vanishingly small), surface `StoreError::Internal("account_id derivation collision after 4 attempts")` rather than loop indefinitely.

**Alternative considered:** Allow resurrection (pure user-facing convenience). **Rejected** per (1) above.

**Alternative considered:** Forbid resurrection only at the CLI layer (a `pangolin-cli add` precondition) and let the library API allow it. **Rejected** because the library API is the security boundary; the CLI is a thin asker (Cardinal Principle 1).

### A5. Tombstone-merge resolve confirmed by P9 — no new code in P10

**Decision:** P9's `Vault::build_merge_payload_for_resolve` (vault.rs L1505–L1595) already correctly produces a tombstone merge revision when `revisions.is_tombstone == 1` for the chosen revision. P10 reaffirms this — the branch is correct AS-WRITTEN; the only adjustment is that `seal_tombstone` will now seal the wider P10-1 payload schema, which `build_merge_payload_for_resolve` automatically inherits via the function call to `seal_tombstone(active.vdk.aead_key(), &merge_aad, &payload)`. P10's commit-point: **add a unit test that pins the merge-of-tombstone correctness post-P10-1** to ensure the wider payload doesn't break the P9 resolve flow.

**Rationale:** P9 §A5 already locked the architectural decision ("resolving to a tombstone ratifies the deletion; the merge revision is a tombstone"). P10's job is to widen the payload, not to redesign the resolve flow. The P10 widening of `seal_tombstone`'s signature (now takes `&TombstonePayload`) means `build_merge_payload_for_resolve`'s tombstone branch needs a one-line adjustment: it must construct a `TombstonePayload` from the merge revision's metadata (`account_id` from `event.account_id`, `tombstoned_at_ms` from `current_unix_ms()`, `deleted: true`). The adjustment is mechanical; no architectural decision needed.

**Subtle question:** should the merge revision's `tombstoned_at_ms` be the original tombstone's timestamp (preserving forensic trail) or the merge revision's own seal time (matching the merge revision's `created_at`)? Recommendation: **the merge revision's own seal time**, because the merge revision is a fresh chain event published at merge time; the `tombstoned_at_ms` field is the timestamp of the *seal*, not the *concept*. The original tombstone's timestamp is recoverable from the chain history (the chain event for the original tombstone has its own block timestamp). MVP-1 may revisit if forensic-attribution use cases need otherwise.

**Alternative considered:** P10 ships P9's tombstone-merge branch as-is with the legacy single-entry payload (don't widen `seal_tombstone`). **Rejected** because then P9's tombstone-merge produces the legacy payload while P10's `delete_account` produces the wide payload — same vault file has two tombstone shapes for the same logical operation. The widening is one signature change in `seal_tombstone`, applied uniformly to both call sites.

### A6. Offline-mode test mechanics — `MockChainAdapter` disconnect toggle

**Decision:** Add a `set_disconnected(bool)` toggle to `MockChainAdapter` (test-utilities-feature-gated; the production binary cannot construct a "disconnected" mock). When `true`, every adapter method returns `ChainError::Rpc("simulated offline".into())` synchronously (no internal state acquisition, no I/O). The integration test toggles between connected and disconnected modes to simulate the offline-edit scenario.

**Rationale:**
1. **Minimal new code.** The `MockChainAdapter` already mediates every adapter call; adding an `AtomicBool` and a `if disconnected { return Err(...); }` guard at each method's entry is ~10 lines.
2. **Deterministic.** A fault-injector that wraps `BaseSepoliaAdapter` would need to either disable the network at the OS level (test environment-specific) or insert mock layers underneath alloy (alloy-specific complexity). The mock toggle is in-process, deterministic, and runs in CI without network.
3. **Production-safety.** The `set_disconnected` method is `#[cfg(any(test, feature = "test-utilities"))]`-gated, mirroring the existing `MockChainAdapter` exposure (which is itself test-utilities-gated). The production binary cannot construct a disconnected mock; an audit reviewer who finds `set_disconnected` in the binary's symbol table is looking at a test build.

**Alternative considered:** A separate `OfflineChainAdapter` type that always returns `ChainError::Rpc`. **Rejected** because it doubles the trait-implementing code surface for marginal gain over a one-line toggle.

**Alternative considered:** Disable network at the OS level (e.g., spawn the test in a network namespace; iptables-block the test process; use `tc` to drop packets). **Rejected** because it's not portable across CI runners (Linux-specific; brittle on macOS / Windows CI).

### A7. Offline `pangolin-cli publish` UX — error cleanly with chain-error message

**Decision:** When `pangolin-cli publish` runs with an unreachable chain (production: real network outage; test: `MockChainAdapter` disconnected), `BaseSepoliaAdapter::with_signer` errors during the `eth_chainId` precheck with `ChainError::Rpc(...)`; the CLI surfaces this verbatim and exits non-zero. The dirty markers remain queued (P8 invariant); the user retries on reconnect. If the user runs `publish` while there are no queued dirty entries AND the chain is unreachable, the CLI **still errors** with the same `ChainError::Rpc` (the chain-connectivity precheck happens before the dirty-list walk). This matches the principle "publish requires connectivity; offline-edit doesn't."

**Rationale:**
1. **Cardinal Principle 3.** The chain is a log; publishing TO the chain requires the chain. There's no fallback ("queue this for later" is exactly what the dirty list already does — the `publish` invocation itself is a flush attempt).
2. **No silent success.** A user runs `publish` expecting their edits to land on chain. Returning success with "0 published, 0 failed, chain unreachable" is misleading; the user might believe their edits are durable when they're still local-only. Erroring out makes the connectivity gap explicit.
3. **Distinguishability from chain-side errors.** Open question §8.3: should `ChainError::Rpc` distinguish "no connectivity" (DNS failure / timeout / refused connection) from "chain returned 5xx" / "chain returned malformed JSON" / "chain returned valid response but unexpected"? Recommendation: the `ChainError::Rpc` variant currently wraps an alloy `TransportError` whose `Display` discriminates these cases. The CLI surfaces the wrapped string verbatim; the user can read "DNS resolution failed" vs "503 Service Unavailable" themselves. P10 doesn't add new variants; it just confirms the current behavior is the intended UX.

**Alternative considered:** `pangolin-cli publish --offline` flag that exits 0 silently when chain is unreachable. **Rejected** because the dirty markers stay queued without the `--offline` flag too; `--offline` is a no-op semantic that adds a footgun ("the user passed `--offline` thinking it would queue something special; nothing different happened").

### A8. `pangolin-cli status` includes tombstone count

**Decision:** Append a `tombstoned_count: <n>` line to the `status` output, alongside the existing `frozen_count`, `dirty_count`, `account_count` lines. The query is `SELECT COUNT(*) FROM account_identities WHERE tombstoned = 1`. Polish item; small.

**Rationale:** The user can see "I have 5 accounts; 3 frozen, 1 tombstoned" at a glance. Without the line, tombstoned accounts are invisible from the CLI surface (they're filtered out of `list_accounts`). The visibility supports the §3.5 audit principle "every state in the system is observable to the user."

**Alternative considered:** Defer the line to a future P11 polish issue. **Rejected** because it's a trivial change (one extra SQL query), it's already touching `commands/status.rs` for nothing else in P10, and the audit value of "the user knows they have tombstoned accounts" is high.

### A9. Forbidden user-facing terms

**Decision:** No user-facing copy from P10 introduces new forbidden terms. The status-output line `tombstoned_count` uses the Pangolin-internal term "tombstoned" — which is not on the §3.5 forbidden list. Help text for `pangolin-cli status` and `pangolin-cli publish` is unchanged. The `StoreError::AccountTombstoned` variant is internal; user-facing surfacing of it (via the resolve flow's error display) uses the message "this account has been deleted" rather than "this revision is a tombstone" — same as the existing P3 path.

**Rationale:** "tombstone" is a Pangolin-internal data-model term per §3.5. The user-facing surface uses "deleted." The CLI's status output shows internal terms because the CLI is a developer-debugging surface, not a user surface (the user surface is MVP-1's host app).

### `pangolin-store` additions

// in crates/pangolin-store/src/blob.rs (widened)

/// Decoded tombstone payload. Master plan §3.7 P10-1 specifies
/// `{ "deleted": true, "account_id": <id>, "tombstoned_at_ms": <ts> }`.
/// Legacy P3-era single-entry `{ "deleted": true }` payloads decode to
/// `TombstonePayload { deleted: true, account_id: [0; 32],
/// tombstoned_at_ms: 0 }` (forward-compat with existing vault files;
/// the all-zeros account_id documents the legacy origin).
#[derive(Debug, Clone, PartialEq, Eq)]
pub struct TombstonePayload {
    deleted: bool,
    account_id: [u8; ACCOUNT_ID_LEN],
    tombstoned_at_ms: u64,
}

impl TombstonePayload {
    /// Construct a fresh tombstone payload for a delete operation.
    pub fn new(account_id: AccountId, tombstoned_at_ms: u64) -> Self;

    /// Construct a legacy-shape payload (single `{ "deleted": true }`).
    /// Used only by the decode path for forward-compat; new tombstones
    /// MUST go through `Self::new`.
    pub(crate) fn legacy() -> Self;

    pub fn is_deleted(&self) -> bool;
    pub fn account_id(&self) -> &[u8; ACCOUNT_ID_LEN];
    pub fn tombstoned_at_ms(&self) -> u64;
}

/// Widened — was `pub enum DecodedPayload { Live(AccountSnapshot),
/// Tombstone }`; now carries the parsed payload so the caller (in
/// particular `Vault::ingest_chain_revision`'s tombstone-bit-detection
/// branch) can introspect.
pub enum DecodedPayload {
    Live(AccountSnapshot),
    Tombstone(TombstonePayload),
}

/// Widened signature — previously took no payload (it sealed the fixed
/// `{ "deleted": true }` byte sequence). Now takes the payload so the
/// caller controls `account_id` and `tombstoned_at_ms`.
pub fn seal_tombstone(
    vdk_aead: &AeadKey,
    aad: &[u8; REV_AAD_LEN],
    payload: &TombstonePayload,
) -> Result<(Ciphertext, Nonce)>;

> 14. **Forged tombstone (foreign device claiming to delete the user's
>     account).**
>     Defense: tombstone revisions are signed by the device's Ed25519
>     `DeviceKey` via `signing::build_signed_revision`, same path as
>     publish + resolve. The canonical hash binds `parent_revision`,
>     `account_id`, `vault_id`, `device_id`, `schema_version`, and
>     `enc_payload`. v0 contract does not verify on-chain; v1 will
>     (MVP-2 issue 2.1). Per Q6 defense-in-depth, every chain event
>     including tombstones passes the `VerifyingKey::from_bytes` check
>     in `pull_all` before reaching `ingest_chain_revision`. Under the
>     PoC two-key model (P8 §A7) carries forward unchanged — the
>     ephemeral signing `DeviceKey` per run discipline applies to
>     tombstone publishes too. The plaintext-level cross-check (`payload
>     .account_id == event.account_id`) added in P10-1 is defense-in-
>     depth: an attacker who has somehow constructed a valid AEAD seal
>     under the user's VDK but with a wrong-account_id payload is
>     rejected inside `detect_tombstone_bit_at_ingest` by a constant-
>     time `subtle::ConstantTimeEq::ct_eq` comparison; mismatch silently
>     returns `is_tombstone = 0` (same bucket as AEAD failure / CBOR
>     failure / locked vault — non-oracle property strengthened). The
>     freeze sentinel still fires for the row's INSERT, so the
>     user-facing safety property is unaffected.
> 15. **Tombstone-bit non-propagation under PoC two-key foreign-ingest
>     (P8 audit CRIT-1 origin).** Defense (acknowledged PoC limitation):
>     under PoC two-key, the chain event carries no AEAD nonce, so
>     `ingest_chain_revision` stores a placeholder zero nonce and
>     cannot decrypt foreign events. The opportunistic-decode logic in
>     P10-2 silently falls back to `is_tombstone = 0` for the affected
>     row, and the existing P8 freeze sentinel fires. The user-facing
>     consequence is "the foreign tombstone is not auto-applied; the
>     account is frozen until the user resolves." The user resolves
>     by reading the chain event's metadata via `pangolin-cli status`
>     (which lists frozen accounts including tombstones), running
>     `pangolin-cli resolve --keep <chosen-revision-id>` against the
>     tombstone revision id (the resolve flow's tombstone branch
>     produces a tombstone merge per P9 §A5), and the post-resolve
>     state has `is_tombstone = 1` correctly set on the merge revision.
>     Closed by MVP-1's nonce-on-chain (the `RevisionPublished` event
>     ABI gains a nonce field; foreign events become decryptable; the
>     opportunistic-decode logic becomes functional without code change).
>     Documented as a known PoC limitation.
> 16. **Resurrection of a tombstoned `account_id`.** Defense:
>     `Vault::add_account` refuses with `StoreError::AccountTombstoned`
>     if the (randomly-derived) `account_id` collides with a row whose
>     `tombstoned = 1`. Under PoC the random-32-via-sqlite-derived
>     `account_id` makes this collision cryptographically negligible,
>     so the guard is defense-in-depth + spec compliance with the
>     append-only invariant (Cardinal Principle 4). MVP-1 may revisit
>     for a deliberate "undelete" feature; under PoC, undelete = create
>     a new account with a fresh `account_id`. The guard's retry loop
>     is bounded at 4 attempts; failure-after-4 surfaces
>     `StoreError::Internal`, NOT a silent skip.
> 17. **Offline edit replay (a queued dirty marker for an edit made on
>     device A is published from a different device B).** Defense:
>     dirty markers are local-only — they live in the `dirty_accounts`
>     table inside the `.pvf` file; another device cannot read them
>     without the `.pvf` file. Under the PoC two-key model, the same
>     `.pvf` file copied to device B (the cross-vault case) shares the
>     dirty list with device A; device B running `publish_all` would
>     publish A's queued entries under B's ephemeral signing
>     `DeviceKey`. **This is the same threat as P8 row #5
>     (cross-vault replay protection).** The `vault_id` binding in
>     `signing::canonical_hash` ensures the published revisions are
>     cryptographically tied to the shared vault; the `device_id`
>     binding identifies B's device as the publisher (which is
>     correct — B is the one who broadcast the transaction). Recovery:
>     either device pulling sees both the original dirty entries and
>     B's ephemeral-signing-key publish; the freeze sentinel fires on
>     A's next pull (since B's `device_id` != A's `device_id`); A
>     resolves. **MVP-1's switch to derived wallet (D-006) closes
>     this — both devices have the same `device_id`; cross-device
>     publish is structurally indistinguishable from same-device
>     publish.**
> 18. **Offline-mode test bypassing the freeze sentinel.** Defense
>     (acknowledgement, test-only): the `MockChainAdapter`
>     `set_disconnected` toggle is `#[cfg(any(test, feature =
>     "test-utilities"))]`-gated; the production binary cannot
>     construct a disconnected mock. An audit reviewer who finds
>     `set_disconnected` in the binary's symbol table is looking at a
>     test build. The toggle does not flush adapter state on
>     disconnect/reconnect; events published before disconnect remain
>     queryable after reconnect. The test verifies the offline-edit
>     invariant: edits queue locally; `publish_all` errors cleanly
>     while disconnected; reconnect-then-publish flushes the queue
>     successfully; **the freeze sentinel does NOT fire during the
>     offline session** (no chain ingest happened — `pull_all` was
>     never called against a disconnected adapter that returned events;
>     `ingest_chain_revision` was never invoked). The cross-check
>     in `offline_session_does_not_set_freeze_sentinel` pins this
>     explicitly.

## Failure modes considered

| Failure mode | Mitigation |
|---|---|
| User runs `pangolin-cli publish` while offline | A7 — `BaseSepoliaAdapter::with_signer`'s `eth_chainId` precheck fails with `ChainError::Rpc`; CLI surfaces verbatim and exits non-zero. Dirty markers preserved. |
| User runs `pangolin-cli publish` while offline with empty dirty list | A7 — same `ChainError::Rpc` from precheck; CLI does NOT exit-0-noop. The connectivity-required precondition is consistent across all `publish` invocations. |
| User runs `pangolin-cli pull` while offline | Same as publish: `BaseSepoliaAdapter::pull_since` errors with `ChainError::Rpc`; CLI surfaces and exits non-zero. No partial pull state (the chunked-pull discipline preserves prior chunks if some succeed; if NONE succeed because the chain is unreachable, `last_pulled_block` is not advanced). |
| Foreign tombstone arrives via cross-vault `.pvf` copy under PoC two-key | Threat #15 — opportunistic-decode silently falls back to bit=0; freeze sentinel fires (existing P8 behavior); user resolves via P9. |
| Own tombstone round-trip — vault A deletes, publishes, then re-pulls its own publish | Idempotency arm #2 (chain-anchor stamp) hits; the genuine-foreign-INSERT branch is bypassed; the local row already has `is_tombstone = 1` from the original `delete_account` write. The ingest is a no-op for the bit. Verified by `own_tombstone_round_trip_via_chain`. |
| User attempts to add an account whose random-derived id collides with a tombstoned row | A4 — `add_account` refuses; the random-32 derivation retries up to 4 times; failure-after-4 surfaces `StoreError::Internal`. |
| User attempts to update or reveal a tombstoned account | Existing P3 + P8 read guards: `update_account` returns `StoreError::AccountTombstoned`; `reveal_password` / `reveal_notes` / `reveal_totp_secret` / `export_payload` return `StoreError::AccountTombstoned`. P10-3 reaffirms via integration tests. |
| Offline session ends with the user power-cycling the device (vault closes uncleanly) | The `delete_account` / `update_account` / `add_account` writes are each transactional (`BEGIN IMMEDIATE … COMMIT`). The dirty markers are persisted as part of the same transaction. Power-cycle leaves either the pre-edit state or the post-edit state, never an intermediate. On reopen + reconnect, `publish_all` walks the persisted dirty list. |
| User resolves a tombstoned-head account during the offline session | Resolve requires `pangolin-cli resolve` which requires the chain (the resolve flow publishes a merge revision). Offline → `ChainError::Rpc` → resolve aborts cleanly → freeze flag remains set → user retries on reconnect. P9 §A3 guarantees idempotency on retry. |
| RPC returns "transaction reverted" on a tombstone publish | Same as P8 publish — `ChainError::Reverted { tx_hash }` surfaces; dirty marker preserved; user retries. The reverted tx didn't change chain state, so re-publish doesn't double-tombstone. |
| Tombstone payload's `account_id` field mismatches the AAD-bound `account_id` | A1 — `detect_tombstone_bit_at_ingest` performs a constant-time `subtle::ConstantTimeEq::ct_eq` comparison after a successful decode; mismatch silently returns `is_tombstone = 0` (same bucket as AEAD failure / CBOR failure / locked vault — non-oracle). The freeze sentinel still fires. The decoder itself does NOT perform the cross-check (it stays type-pure); the rejection lives in the ingest layer. |
| Vault file's tombstone payload is the legacy P3-era single-entry shape | A1 — `decode_payload` arity-1 arm decodes to `TombstonePayload::legacy()` (`account_id = [0; 32]`, `tombstoned_at_ms = 0`). Ingest sets `is_tombstone = 1` correctly. |

## Rollback / abort plan

If BUILD reveals the design is fundamentally wrong (per §16.7):

- **A2 opportunistic-decode is unreachable in practice (e.g., the synthetic-decryptable-tombstone test cannot be constructed without violating production invariants):** Drop P10-2 entirely; leave the `is_tombstone_i64 = 0` hardcode but update the comment to "PoC: foreign events freeze; bit-set deferred to MVP-1's nonce-on-chain." P10 still ships the widened payload (P10-1) and the offline-mode test (P10-4); the audit-flagged comment is updated rather than the code. Document the reduction in DEVLOG.
- **A4 anti-resurrection guard breaks an existing P3 test (e.g., a test exercises a deterministic account_id that collides with a tombstoned row):** Add an `expect-collision` test variant; the production guard still fires, but the test infrastructure uses fresh seeds. Likely a 1-line test fix.
- **A6 `MockChainAdapter` toggle introduces flaky behavior under concurrent test runs:** Switch from `AtomicBool` to a `RwLock<bool>`; serialize the toggle. Marginal cost.
- **The widened `TombstonePayload` struct introduces a serde dependency (HIGH-1 invariant violation):** The plan uses `ciborium-ll` directly (no serde). Audit at SIGNOFF: confirm `cargo tree -p pangolin-store | grep -ci serde` count is unchanged.

If a more fundamental design problem surfaces (e.g., the master plan §3.7 P10-1 payload schema is structurally incompatible with the existing `seal_tombstone` API in a way that requires a schema-version bump on revisions), escalate to Kelvin per §16.7. Do not silently patch around.

## Out of scope (explicit)

- **Foreign-event tombstone propagation under PoC two-key.** Closed by MVP-1's nonce-on-chain. Deferred. Documented as Threat #15.
- **Resurrection of tombstoned accounts.** MVP-1 if the user-feature emerges. Under PoC, resurrection = new account with fresh `account_id`. A4.
- **Auto-recovery from chain disconnection.** Never. Offline operation is a feature; there's no "the system tried and gave up" UX. The user explicitly retries.
- **Tombstone deletion from chain (cryptographic erasure / right-to-be-forgotten).** Chain is append-only by design. Tombstones are markers, not deletes. Documented as a known limitation.
- **Browser extension wiring.** MVP-2 / MVP-4.
- **Mobile shell.** MVP-3 / MVP-5.
- **E2E recorded screencast for the offline scenario.** P11 / P12.
- **Funder service.** MVP-2 issue 3.4. PoC funds the keystore manually.
- **Per-publish wallet rotation.** MVP-2 issue 3.4. PoC reuses the same Foundry keystore for every publish (acknowledged observability surface, P8 row #8).
- **Multi-tombstone batching (delete N accounts in one chain event).** P11 polish item; P10 ships single-account-per-tombstone-event.
- **Tombstone-attribution forensic schema** (adding a `device_id` field to the tombstone payload). Open question §8.2; recommend deferring to MVP-1 if forensic-attribution use cases emerge.
- **Selective publish** (`pangolin-cli publish --account-id <id>` to flush only one account's dirty entries from the queue). MVP-1 polish item.
- **Retry-exhaustion deterministic test for `derive_fresh_account_id`** (P10 fix-pass M-3, deferred per auditor's recommendation). The retry-loop's failure path requires a test-only RNG seam to drive `random_32_via_sqlite` through 4 successive collisions; the existing happy-path coverage plus the `~4×N/2^256` probability bound (where `N` is the tombstoned-row count) is sound for PoC. MVP-1 may revisit if a deterministic seam lands.

## Open questions for Kelvin

1. **Tombstone payload schema — three fields or four?** §A1 ships the master plan §3.7 P10-1 three-field shape (`deleted`, `account_id`, `tombstoned_at_ms`). Should P10 also include a `device_id` field for forensic attribution? Recommend three-field for v0; add `device_id` only if MVP-1 needs it. Adding now is cheap (one more CBOR entry); deferring is also cheap (a future schema bump adds the entry without breaking existing readers because the decoder dispatches on entry count). Lean toward deferring.

2. **`tombstoned_at_ms` value in the merge-of-tombstone case.** §A5 — should the merge revision's payload carry the original tombstone's timestamp (forensic preservation) or the merge revision's own seal time (matches `created_at`)? Recommend merge revision's own seal time (the field is the *seal* timestamp, not the *concept*). The original tombstone's timestamp is recoverable from the chain history via the original tombstone event's block timestamp.

3. **`pangolin-cli publish` connectivity-error message specificity.** §A7 — should the CLI distinguish "no chain connectivity" (DNS / timeout / refused) from "chain returned 5xx" / "chain returned malformed" in the user-visible error message? Recommend yes — the user can act on connectivity (retry on a different network) but not on a transient 5xx (just retry). Current behavior is "surface the alloy `TransportError`'s `Display` verbatim," which already discriminates these via the wrapped error's content. Confirm: should P10 add a structured-error CLI-side category mapping? Recommend NO — the alloy error string is already discriminating enough; an extra mapping layer is friction without payoff.

4. **Offline-mode test placement.** §6 — should the offline-mode test be in `tools/pangolin-cli/tests/offline_mode.rs` (integration; current plan) or as a sibling unit-style file under `crates/pangolin-store/tests/`? Recommend integration because the test exercises the CLI's `publish_all` orchestrator, which is the seam the user invokes; per-vault unit tests don't give the CLI-shape coverage the offline scenario warrants.

5. **`MockChainAdapter::set_disconnected` toggle persistence.** §A6 — should the toggle reset on `MockChainAdapter::clone()`? Recommend NO — the `Clone` impl shares the `Arc<Mutex<MockChainState>>`; sharing the disconnect-toggle alongside is consistent (one mock, one state, one connection-state). A test that wants two mocks with different disconnect states constructs two separate `MockChainAdapter::new()` instances.

6. **`tombstoned_count` line in `pangolin-cli status`.** §A8 — should the line be omitted when count is 0 (matching the existing `frozen_count` discipline)? Recommend yes — matches the no-news-is-good-news UX of the rest of the status output.

7. **A4 anti-resurrection bounded-retry budget.** §A4 — 4 attempts before surfacing `StoreError::Internal`. Is 4 the right number? Probability bound: under N tombstoned rows, the per-attempt collision probability is N / 2^256; 4 attempts gives 4 × N / 2^256 worst-case error probability. For N = 2^48 (256 trillion tombstones), this is 4 × 2^48 / 2^256 ≈ 2^-206. A single attempt is also safe under any reasonable N. The retry budget is over-engineered defense-in-depth. Recommend 4 — small enough to not paper over a genuine bug (e.g., a broken RNG), large enough to absorb a one-in-2^256 stutter.

## Security-critical?

**YES** per §16.3 (chain code; tombstone semantics are the load-bearing primitive that turns Cardinal Principle 4 into a closed loop). Audit at SIGNOFF will scrutinize:

- The A1 widened tombstone payload schema. Does the encoder emit deterministic CBOR with fixed three-entry alphabetical key order? Spot-check by feeding the same payload through the encoder twice and asserting byte-equal output. Confirm the legacy single-entry decoder arm is reachable AND cleanly rejects payloads that aren't either arity 1 or arity 3.
- The A2 opportunistic-decode discipline. Does the AEAD-open inside `ingest_chain_revision` use the **placeholder zero nonce** (because that's what the row gets persisted with) and not some other nonce source? Audit-trace the nonce flow: `ingest_chain_revision` constructs the placeholder zero nonce; the open call uses the SAME nonce that gets stored on the row; if the open succeeds, the bit is set; subsequent reads of the same row use the same placeholder nonce and either succeed (consistent) or fail (the row is unreadable but the bit is correct). The non-oracle property is "the only observable side-effect of decode-success-vs-decode-failure is the `is_tombstone` bit on the inserted row + (if the bit is 1) the `account_identities.tombstoned = 1` flag — there is NO error variant difference, NO timing difference (the AEAD open is constant-time), NO subsequent-read-path difference."
- The A3 silent-fallback discipline. Does the decode-failure path ALWAYS land in the freeze-sentinel arm, or is there a code path that decode-fails and skips the freeze? Walk through every error-return inside the genuine-foreign-INSERT branch; confirm the freeze-sentinel UPDATE/INSERT runs unconditionally regardless of decode outcome.
- The A4 anti-resurrection guard. Does `add_account` actually call the new SELECT-tombstoned probe? Add a regression test that constructs a vault with a tombstoned row, monkey-patches the random_32_via_sqlite to return that row's account_id on the first call, asserts the retry on the second call generates a fresh id. The retry budget is 4; assert the error after 4 attempts has the `StoreError::Internal` variant.
- The A5 P9 tombstone-merge correctness post-P10-1 widening. Does `build_merge_payload_for_resolve`'s tombstone branch construct a `TombstonePayload` with `account_id = event.account_id` (NOT some other id) and `tombstoned_at_ms = current_unix_ms()` (NOT the original tombstone's timestamp)? The §A5 recommendation is the merge revision's own seal time; confirm this is wired correctly.
- The A6 mock-disconnect toggle. Is `set_disconnected` actually `#[cfg(any(test, feature = "test-utilities"))]`-gated? Inspect the binary symbol table after release build: `set_disconnected` should not appear. Inspect the test build symbol table: it should appear.
- The A7 `pangolin-cli publish` precheck order. Does the chain-connectivity precheck (`with_signer`'s `eth_chainId` call) fire BEFORE the dirty-list walk? - The A4 cross-checks at the AAD layer vs the plaintext layer. The tombstone payload's `account_id` field is plaintext (post-AEAD-open); it MUST cross-check against `event.account_id` (which is AAD-bound). Trace: `event.account_id` flows into `build_aad` (AAD binding); `decode_payload`'s tombstone arm extracts `payload.account_id` (plaintext); the cross-check lives inside `detect_tombstone_bit_at_ingest` (NOT inside `decode_payload`, so the decoder stays type-pure) and uses `subtle::ConstantTimeEq::ct_eq` over the 32-byte arrays. Mismatch silently collapses to `is_tombstone = 0` — same bucket as AEAD failure / CBOR failure / locked vault, no error variant escapes (non-oracle property strengthened). Confirm: the only new chain-side operation in P10 is the tombstone payload schema, not the publish path.
