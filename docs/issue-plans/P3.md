# Issue P3: Revision graph + fork detection

> **Status:** PLAN-gate. Per Kelvin's "approve only at merge" directive, no plan-gate review pause; plan exists for §16 audit-trail compliance and to scope the build.
> **Security-critical:** No (no new crypto; fork detection is a data-structure/algorithm primitive). Peer review at SIGNOFF is sufficient.

## Spec reference

- **Master plan §3.7 EPIC: Identity & Revision Model:** P3-1..P3-5.
- **Whitepaper §7 (Multi-Device Synchronization & Conflicts):** "Updates are immutable. Each update references a previous state. If two updates occur concurrently, a conflict is detected. The system never silently merges secrets — conflicts are surfaced to the user."
- **Master plan cardinal principle 4:** "Append-only state. Every credential change is a new immutable revision referencing a parent. Forks surface to the user. **Never silent merge.**"

## What P2 already shipped (so this issue doesn't duplicate)

P2's `pangolin-store` already provides:
- `AccountSnapshot` (display, username, password, url, notes, totp_secret) — covers P3-1 minimal scope
- Revision row schema with `parent_revision_id`, `device_id`, `schema_version`, `enc_payload`, `enc_nonce` — covers P3-2
- Deterministic CBOR via `ciborium-ll` — covers P3-3
- `Vault::revisions_for(account_id)` linear walk — partial P3-4
- A presumption of single-head per account: `account_identities.head_revision_id` is one column

What's still missing for the full P3 scope:
- **Fork detection** when two revisions share the same `parent_revision_id` (multiple "heads" exist)
- **Per-account head computation** that returns `Vec<RevisionId>` rather than assuming singleton
- A type that exposes the per-account revision graph for downstream consumers (P9 conflict resolution, P7 chain adapter sync logic, the eventual MVP-1 UI conflict dialog)
- Tests pinning the detection across multiple pathological lineage shapes

## Goal

Add a `RevisionGraph` type to `pangolin-store` that, given an `AccountId`, builds the full parent→child graph of all revisions for that account, computes the set of current heads (revisions with no children), and surfaces fork state cleanly. Expose enough on `Vault` that P9 can later build a resolution UI on top.

After this issue:
- `Vault::revision_graph(account_id) -> RevisionGraph` returns the full graph.
- `Vault::account_heads(account_id) -> Vec<RevisionId>` returns the head set; length > 1 means the account is forked.
- `Vault::is_forked(account_id) -> bool` is the cheap boolean predicate.
- `Vault::all_forked_accounts() -> Vec<AccountId>` lists accounts that need user attention.
- The schema gracefully accommodates multi-head state without breaking the existing single-head fast path. The `account_identities.head_revision_id` column becomes "the most recently chosen canonical head" rather than "the only head"; multi-head detection happens by querying `revisions` for orphans (revisions with no child rows referencing them as parent).

## Approach

### `RevisionGraph` data type

```rust
pub struct RevisionGraph {
    /// Topologically-ordered list of all revisions in this graph.
    revisions: Vec<RevisionMeta>,
    /// child → parent index (the parent_revision_id of each row).
    parents: HashMap<RevisionId, RevisionId>,
    /// parent → set of children. Order within a children-set is insertion
    /// order; deterministic ordering for tests + UI is the caller's job.
    children: HashMap<RevisionId, Vec<RevisionId>>,
    /// The set of revisions that have NO children — these are heads.
    heads: Vec<RevisionId>,
    /// Genesis revision (parent_revision_id == 0x00...00). At most one
    /// per well-formed account; multiple genesis would indicate either
    /// corruption or attacker-injected revisions.
    genesis: Option<RevisionId>,
}

impl RevisionGraph {
    pub fn revisions(&self) -> &[RevisionMeta];
    pub fn heads(&self) -> &[RevisionId];
    pub fn is_forked(&self) -> bool;
    pub fn parent_of(&self, id: &RevisionId) -> Option<&RevisionId>;
    pub fn children_of(&self, id: &RevisionId) -> &[RevisionId];
    pub fn genesis(&self) -> Option<&RevisionId>;
    /// Walks ancestry from `id` to genesis. Returns oldest→newest.
    pub fn ancestors(&self, id: &RevisionId) -> Vec<RevisionId>;
    /// Common-ancestor of two heads (the fork point). Returns the
    /// `RevisionId` where the two branches diverged. Returns None if
    /// the two revisions don't share any common ancestor (corruption).
    pub fn common_ancestor(&self, a: &RevisionId, b: &RevisionId) -> Option<RevisionId>;
}
```

### `Vault` API additions

```rust
impl Vault {
    /// Build the full revision graph for `account_id`. Reads from SQL,
    /// constructs the index, returns. Does not require Active state
    /// (graph is metadata, not plaintext payload — cheap to compute).
    pub fn revision_graph(&self, id: AccountId) -> Result<RevisionGraph>;

    /// Quick check: does this account have multiple heads?
    pub fn is_forked(&self, id: AccountId) -> Result<bool>;

    /// Heads of an account. Length == 1 → no fork. Length > 1 → fork.
    /// Length == 0 → account doesn't exist (returns Err::AccountNotFound).
    pub fn account_heads(&self, id: AccountId) -> Result<Vec<RevisionId>>;

    /// All accounts in the vault that currently have multiple heads.
    /// The "needs attention" set for a P9 conflict resolution UI.
    pub fn all_forked_accounts(&self) -> Result<Vec<AccountId>>;
}
```

### Schema implications

`account_identities.head_revision_id` stays as the column-of-record for the **canonical head** (the one we surface to users via `get_account`). For non-canonical heads, the discovery is at query time:

```sql
-- Heads = revisions for an account_id whose revision_id appears in NO other row's parent_revision_id.
SELECT r.revision_id FROM revisions r
WHERE r.account_id = ?
  AND NOT EXISTS (
    SELECT 1 FROM revisions r2
    WHERE r2.parent_revision_id = r.revision_id
  )
```

This is O(n) per account scan. PoC scale (≤10k revisions per account) makes this trivial. For larger scale, an index on `(account_id, parent_revision_id)` makes it faster.

### Fork-creation primitive (test-only)

To exercise fork detection meaningfully, the tests need a way to manufacture a fork. Production `add_account`/`update_account` always advance the canonical head linearly; they don't produce forks. Forks happen when **two devices independently advance from the same parent**, then sync. Simulating that without going through P7 (chain adapter) means writing two child revisions with the same `parent_revision_id` directly via SQL in a test helper.

Test approach:
1. `Vault::create` + `unlock` + `add_account` → genesis revision R0
2. `update_account` → R1 (parent = R0)
3. `update_account` → R2 (parent = R1)
4. **Test helper**: directly INSERT a third revision R2' (parent = R1, encrypted with same VDK + correct AAD) — simulating "another device's update from R1"
5. `Vault::revision_graph(account_id)` should report two heads: R2 and R2'
6. `Vault::common_ancestor(R2, R2')` should return R1
7. `Vault::all_forked_accounts()` should return `[account_id]`

The test helper is gated behind `#[cfg(test)]` and uses the existing crypto primitives — no new crypto introduced.

## Success criteria

1. Crate compiles, fmt + clippy + audit + deny stay clean.
2. **Linear lineage round-trip:** create vault, add account, edit 5 times → `revision_graph(id).heads()` returns 1 element (the latest); `is_forked` returns false. Verified by unit test.
3. **Two-way fork detection:** synthesize a fork (R2 and R2' sharing parent R1) → `revision_graph(id).heads()` returns 2 elements (set equality); `is_forked` returns true; `all_forked_accounts()` includes the account; `common_ancestor(R2, R2')` returns R1. Verified by integration test.
4. **Three-way fork detection:** three children of the same parent → `heads()` returns 3 elements. Verified.
5. **Genesis detection:** `genesis()` returns the revision whose `parent_revision_id` is the all-zeros sentinel. Verified.
6. **Ancestor walk correctness:** `ancestors(head)` returns the chain head→genesis in the right order. Verified.
7. **Multi-genesis corruption case:** if (somehow — should never happen in production) two revisions both have `parent_revision_id = 0x00...00`, the graph reports both as heads (since both have no parent in the graph) AND the integration test asserts `genesis()` returns one of them with a documented tie-break (e.g., earliest `created_at`). Documented as "should never happen with well-formed data; if it does, it's an attacker injection."
8. **Performance:** `revision_graph` over 1,000 revisions completes in <100ms on a clean build. (Sanity bound; not a security claim.)
9. **No plaintext leak:** `RevisionGraph` exposes only `revision_id`, `parent_revision_id`, `device_id`, `schema_version`, `created_at`, `is_tombstone`, `chain_anchor` fields — never `enc_payload`. The plaintext-on-disk verifier from P2 still passes (sanity check; no regression).

## Failure modes considered

| Failure mode | Mitigation |
|---|---|
| Graph cycle (revision points back at an ancestor) | Detect during graph build; return `StoreError::Corrupted("revision lineage contains a cycle")`. |
| Dangling parent (revision references a parent_revision_id that doesn't exist in the table) | Treat as "this revision is a NEW genesis" and flag as `genesis_extra` in the graph. Document as corruption signal. |
| Tombstoned revision in the head set | Allowed; tombstones are heads. The UI layer (P9) decides what to do. |
| Account with zero revisions | `revision_graph` returns an empty graph; `heads()` returns empty Vec; `is_forked` returns false. |

## Out of scope

- **Conflict resolution UI / publish-resolution-revision flow** — that's P9 (or MVP-1 issue 5.3 production version).
- **Auto-merge of forks** — explicitly not done; cardinal principle 4 forbids silent merge.
- **Chain-side fork detection** — P7 chain adapter; this issue only handles local store.
- **AccountIdentity model expansion** (multi-URL, multi-username, password history with timestamps + originating device IDs) — that's MVP-1 issue 1.2 production version. P2's flat 6-field model is PoC-sufficient.

## Security-critical?

**No.** No new crypto, no contract changes, no recovery. Pure data-structure work over already-encrypted-and-authenticated revision metadata. Peer-review at SIGNOFF is sufficient. The fork detection logic itself is a defensive primitive (it surfaces silent attacker-injected revisions to the user, per cardinal principle 4) but the implementation does not handle secrets directly.
