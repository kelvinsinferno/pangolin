## Spec reference

  > "Session policy engine — production. All rules from session spec §7:

  > "A session is a time-bounded, local authorization state. Properties:
  > MUST be local-only; MUST expire automatically; MUST NOT grant

  > be stored on-chain."

  > "3.1 Presence Proof — Confirms physical presence. Requirements: MUST

  > "4.1 Active — session valid; access allowed. 4.2 Expired — session

## Goal

After this issue closes the session-policy engine is production-grade and
spec-complete against session spec §2.3, §5–§8. Concretely: a session
starts only on a successful 2-proof unlock (presence + identity); is
maintained by a single proof / activity touch with a configurable idle
timeout (5/15/30/60/240 min or "until device lock", default 15 min,
stored in the vault `meta` with a 15-min default for vaults that predate
1.4); always expires at the absolute-max ceiling (default 4 h) regardless
of activity; surfaces a typed error on idle/absolute expiry and zeroizes
the in-memory cache + tears down the `:memory:` FTS5 index at that
moment; re-prompts (via `with_session`) and resumes the original
operation when a session is found expired mid-action; gates every
high-risk reveal-class operation (`reveal_notes`, `reveal_password_history`,
`reveal_totp_secret`) behind an explicit fresh presence proof whose
freshness window is 30–60 s (60 s constant); dedups presence prompts
within that window (one proof satisfies concurrent reveals); and exposes
a typed prompt-timeout (~60 s) failure mode. The `pangolin-ffi`
`vault_unlock` / `vault_lock` / `vault_close` / `session_status` /
`session_extend` bodies are wired against this engine (no more
`todo!()`); presence-gated `reveal_notes` / `reveal_password_history` /
`reveal_totp_secret` FFI entry points are added (a 1.1-surface
amendment); the FFI `AccountSnapshot` no longer carries plaintext-bearing
secret handles directly (Q5). Per Q1's answer, the session/proof types
either move into `pangolin-core/src/session/` (recommended: partial) or
stay re-exported from `pangolin-store`. The 1.3 `:memory:` FTS5
search-index lifecycle (build on unlock, tear down on lock/expiry, sync
on add/update/delete) is preserved unchanged. Existing PoC `.pvf` vaults
still open, unlock (two-proof), search, and reveal. HIGH-1
(`cargo tree -p pangolin-crypto | grep -ci serde == 0`), Q3 of 1.1
(`cargo tree -p pangolin-core | grep -ci uniffi == 0`), and Q3-bonus
(`cargo tree -p pangolin-store | grep -ci uniffi == 0`) all stay 0;
`forbid(unsafe_code)` survives on every crate except `pangolin-ffi`;
zero chain code is activated; AGPL-3.0-or-later SPDX headers on every new
`.rs` file.

### 1. The session state machine — production (session spec §4, §5, §7)

The four-state machine already exists (`SessionState::Locked /
PendingAuthorization / Active{expires_at, last_proof_at,
session_started_at} / Expired`) with the correct transitions. 1.4
changes:

1. **Configurable idle duration.** Add a `session_idle_secs: Option<i64>`
   column to the `meta` table (single-row) — `NULL`/absent ⇒ default
   15 min for pre-1.4 vaults; otherwise one of the spec's
   `{300, 900, 1800, 3600, 14400}` second values, or a sentinel for
   "until device lock" (`-1`, meaning: no idle timeout, only the
   absolute-max ceiling applies — and on a real device-lock signal the
   session expires immediately; CLI has no device-lock signal so
   "until device lock" behaves like "until process exit"). `meta.rs` +
   `schema.rs` gain the column (idempotent `ALTER TABLE … ADD COLUMN`
   like 1.3's `fts_meta` precedent). A `Vault::set_session_idle(choice:
   SessionDuration) -> Result<()>` setter persists it (this is itself a
   high-risk action per §5.4's "extend long sessions" — gate it behind a
   fresh presence proof when the new value is *longer* than the current;
   shortening is always allowed). `unlock` reads the column and uses it
   for the first `expires_at`; `next_idle_deadline` takes the configured
   idle duration as a parameter (currently hard-codes
   `IDLE_TIMEOUT_DEFAULT`). `IDLE_TIMEOUT_DEFAULT` (15 min),
   `ABSOLUTE_MAX_DEFAULT` (4 h) stay as the defaults.
2. **Absolute-max ceiling** — unchanged. `next_idle_deadline` already
   caps `expires_at` at `session_started_at + ABSOLUTE_MAX_DEFAULT`;
   the absolute max is *not* configurable per the spec (§7.4 — fixed
   default 4 h; the spec lists 4 h as a *session*-length option but the
   absolute ceiling is the max of that list, so 4 h is both the longest
   configurable session and the absolute ceiling — i.e. picking "4 hours"
   means idle == absolute and the session can't be extended).
3. **Device-lock expiry hook.** Add a `Vault::device_locked()` method
   that, if the session is `Active`, drops the cache + index and
   transitions to `Expired` (same path as idle expiry). It is the
   storage-layer hook for MVP-3/4 desktop/mobile to call on the OS
   lock-screen event. **For CLI it is unused** (CLI has no OS-lock
   signal); the hook exists so the state machine "leaves the door open"
   per the task. `vault_lock` FFI already covers the explicit case;
   `device_locked` is the implicit-on-OS-event case.
4. **`Expired` semantics** — unchanged. `check_session_freshness`
   detects expiry, drops `ActiveState` (zeroizing the cache + freeing
   the `:memory:` index), flips to `Expired`, returns
   `StoreError::SessionExpired`. Next op needs the full 2-proof unlock.
5. **`PendingAuthorization`** — already a UX state the vault treats as
   "not Active". `with_session` is the primitive that enters/exits it
   conceptually (the actual proof gathering is the host UI's job). 1.4
   keeps this; no host-UI prompt-state machine lands in `pangolin-store`.
6. Files: `crates/pangolin-store/src/session.rs` (parameterise
   `next_idle_deadline`; add `SessionDuration` enum + its `meta`
   serialisation; keep all proof + clock types), `meta.rs` + `schema.rs`
   (the new column), `vault.rs` (`unlock` reads the configured idle;
   `set_session_idle`; `device_locked`).

### 2. Two-proof start; single-proof maintain (session spec §2.3, §5.1, §5.2)

Already implemented and tested in the PoC:
`unlock(&dyn PresenceProof, &dyn IdentityProof)` verifies *both* (a
missing/empty/replayed presence ⇒ `AuthenticationFailed`; an empty PIN ⇒
`AuthenticationFailed`; a wrong PIN ⇒ `AuthenticationFailed` after the
full Argon2id round-trip — MEDIUM-1 indistinguishability). A failed
re-`unlock` while `Active` does NOT lock the vault
(`second_unlock_with_wrong_password_does_not_lock_vault`). `touch_session`
extends the idle deadline (capped at absolute-max) on every successful
credential op; metadata-only ops use `maybe_expire_active_session` (the
soft variant). 1.4 keeps all of this; the only change is `unlock`'s
first `expires_at` now derives from the configured idle (step 1) rather
than the hard-coded 15 min. The `PressYPresenceProof` single-use /
freshness discipline and the `PinIdentityProof` zeroize-on-drop / redacted
`Debug` discipline stay. **The proof traits stay `&self`-shaped and
`dyn`-compatible so MVP-3/4's NFC / passkey / biometric impls slot in
without API churn** (this is the P4-2 promise; the traits already exist).
Q2 asks Kelvin to confirm the CLI-tier proof forms.

### 3. Presence freshness, prompt timeout, prompt dedup, mid-action resume (session spec §5.4, §7.6, §7.7, §8.5, §8.6)

1. **Presence freshness window (§7.6 — 30–60 s).** The PoC's
   `PresenceProof::freshness()` returns `PRESENCE_FRESHNESS` (60 s), and
   `PressYPresenceProof::verify` rejects a proof older than 60 s
   (`AuthError::NotFresh`). 1.4 keeps 60 s as *the* constant (Q3),
   documents it as the upper bound the spec permits, and **adds an
   engine-side last-presence timestamp**: `ActiveState` gains
   `last_presence_at: Option<SystemTime>`. On every successful
   presence-gated op, `last_presence_at = clock.now()`. A presence-gated
   op first checks `now - last_presence_at <= PRESENCE_FRESHNESS` — if
   true, the op proceeds *without consuming a new proof* (the prompt-dedup
   case, see below). If false (or `None`), the supplied proof must
   `verify()` (which itself enforces the 60 s window on the *proof's*
   construction time) and then sets `last_presence_at`. This is the
   single source of truth for "is presence fresh right now".
2. **Prompt timeout (§7.7 — ~60 s).** Add `StoreError::PromptTimedOut`
   and `pangolin_core::Error::Session(...)` mapping. The *host UI* runs
   the actual 60 s wall-clock timer on a prompt; the storage layer's
   contribution is (a) the `PROMPT_TIMEOUT` constant (already exists,
   60 s) so callers don't pick a different value, and (b) a
   `Vault::reveal_*_with_timeout`-style API is *not* needed — instead,
   when a presence proof is supplied that has *already aged past its
   freshness window* (because the host UI prompted, the user took >60 s,
   the proof's `created_at` is now stale), `verify()` returns
   `AuthError::NotFresh` which 1.4 maps to `PromptTimedOut` rather than
   `AuthenticationFailed` when it comes from a reveal path (a stale
   proof at a reveal site == "the prompt timed out"). The op fails
   loudly with a typed error — never silently (§8.2). The CLI's
   `--reveal-*` flow surfaces this as "presence prompt timed out;
   re-run the command".
3. **Prompt deduplication (§8.6).** Implemented by the
   `last_presence_at` timestamp from step 1: two reveal ops within 60 s
   of the first successful presence both see "presence fresh" and neither
   re-prompts. The CLI already has the "one-prompt-N-proof" pattern at
   the `apps/cli` layer (one `PressYPresenceProof::confirmed()` reused
   across N `--reveal-*` flags via the engine's dedup); 1.4 makes the
   *engine* the authority for dedup so non-CLI callers get it for free.
   (Note: the PoC's `PressYPresenceProof` is single-use — a second
   `verify()` on the *same* proof returns `PresenceAlreadyConsumed`.
   With the `last_presence_at` dedup, a second reveal within 60 s never
   calls `verify()` on the proof at all, so the single-use discipline
   and the dedup coexist: dedup is "don't re-prompt"; single-use is
   "don't replay a captured proof".)
4. **Mid-action resume (§8.5).** `with_session(op, reauth)` already does
   this: `Ok(())` ⇒ run `op`; `Err(SessionExpired)` ⇒ run `reauth`,
   re-check freshness (L-3: a reauth that lies must not let `op` run),
   run `op`; `Err(other)` ⇒ propagate. The PoC's
   `with_session_resumes_op_after_reauth` /
   `with_session_revalidates_after_reauth_returns_ok` tests cover it.
   1.4 keeps it; the only generalisation is that the same primitive now
   also covers a presence-gated `op` (the `reauth` callback can re-supply
   a fresh presence proof, not just re-unlock). The known limitation
   stands: a session that expires *mid-op* (e.g. during a 1.5 s Argon2id
   run inside `op`) is not retried — `op` returns whatever it returned;
   MVP-3+ may add a transactional retry wrapper. Documented in
   §"Failure modes considered".
5. Files: `crates/pangolin-store/src/vault.rs` (`ActiveState.last_presence_at`;
   `reveal_secret_field` freshness/dedup logic; the
   `NotFresh`→`PromptTimedOut` mapping at reveal sites), `session.rs`
   (no change beyond doc), `error.rs` (`PromptTimedOut` variant),
   `pangolin-core/src/error.rs` (map `PromptTimedOut` → `Session(...)`).

### 4. The reveal-class entry points — re-introducing presence-gated reveal (session spec §5.4)

Reveal-class assets per §5.4 (and the Phase-2 note): full password
history, notes, raw TOTP seed, recovery material. 1.4 surfaces the V1
model's full data for the first three (recovery material is MVP-3 scope):

1. **`Vault::reveal_notes(id, &dyn PresenceProof) -> Result<SecretBytes>`**
   — already exists (PoC P4-fix-pass-H-1), returns the decrypted notes.
   1.4 keeps it; it now runs the step-3 freshness/dedup logic.
2. **`Vault::reveal_password_history(id, &dyn PresenceProof) ->
   Result<Vec<PasswordHistorySummaryEntry>>`** — **new.** Returns the
   *full* history (every `PasswordEntry`: plaintext bytes + `set_at_ms`
   + `originating_device`), not just the head. The existing
   `reveal_password` (head-only) stays as a convenience. Presence-gated.
3. **`Vault::reveal_totp_secret(id, &dyn PresenceProof) ->
   Result<SecretBytes>`** — already exists; returns the raw TOTP seed
   bytes (empty `SecretBytes` if none configured). 1.7's RFC-6238
   generator consumes the seed internally without a reveal; the *reveal*
   of the raw seed is the high-risk action this gates. 1.4 keeps it.
4. **All three** route through a shared `reveal_secret_field`-style
   helper: `check_session_freshness` (does not consume the proof on
   failure) → `refuse_if_frozen` (P8 CRIT-1, does not consume the proof)
   → presence freshness check / dedup / `verify()` (step 3) → read from
   the in-memory cache → `touch_session`. Errors: `SessionExpired` /
   `NotUnlocked` (session not active, cache zeroized as a side-effect),
   `PromptTimedOut` (stale presence proof at a reveal site),
   `AuthenticationFailed` (presence proof failed to verify for a
   non-staleness reason — replayed, generic), `AccountNotFound`,
   `AccountFrozenPendingResolve`.
5. **FFI:** add `reveal_notes` / `reveal_password_history` /
   `reveal_totp_secret` entry points to `pangolin-ffi/src/identity.rs`
   (or a new `pangolin-ffi/src/reveal.rs` — recommend the latter for
   clarity). Each takes `Arc<VaultHandle>`, the `AccountId`, and a
   `PresenceProof` record (the 1.1-frozen `{schema_version, bytes}`
   shape — the bridge converts it to a `PressYPresenceProof` for the CLI
   tier; the `bytes` field is currently unused for `PressY` but is the
   slot the MVP-3/4 hardware impls populate). Returns: `reveal_notes` →
   a `RevealedSecret` Object (Zeroizing wrapper, byte-length accessor
   only — same discipline as `SecretPassword`); `reveal_password_history`
   → `Vec<PasswordHistoryEntry>` (the existing FFI shape, but now reached
   *only* through this presence-gated path); `reveal_totp_secret` → a
   `RevealedSecret` Object (or reuse `TotpSecret`). **This is a
   1.1-surface amendment** — see Q4. Bodies live in
   `pangolin-ffi/src/reveal.rs`; the bridge conversions in
   `identity_bridge.rs`.

### 5. The relocation: `pangolin-store` → `pangolin-core` (1.1 Q2 / 1.2 Q2 / §16.8)

**This is THE scope question — see Q1.** What depends on `Vault` /
`AccountIdentity` / the session types living in `pangolin-store`:

- `pangolin-ffi` — `session.rs`, `identity.rs`, `identity_bridge.rs`,
  `cabi.rs` import `pangolin_core::Vault` / `pangolin_core::AccountId` /
  `pangolin_core::AccountIdentitySummary` etc. (via the re-export). If
  the *re-export* stays (it must, regardless of where the types live),
  `pangolin-ffi` is unaffected by a relocation.
- `apps/cli` — imports `pangolin_store::session::PressYPresenceProof` and
  `pangolin_store::Vault` / `pangolin_store::reveal_*` **directly**
  (`apps/cli/src/commands/account.rs` line 47: `use
  pangolin_store::session::PressYPresenceProof;`). A full relocation must
  either keep those re-exported from `pangolin-store` *and*
  `pangolin-core`, or update ~6 CLI import sites. Cleanest: have
  `pangolin-store` re-export the moved types back from `pangolin-core`
  (the reverse of today's arrow) so `apps/cli`'s
  `pangolin_store::session::PressYPresenceProof` path keeps resolving.
  But that creates a dep cycle if `pangolin-core` then needs `Vault`
  (which lives in `pangolin-store`)… which is exactly why **option 2
  (partial)** is cleanest: `session.rs` (proof types, `Clock`,
  `SessionState`, timing constants — *no* `Vault`) has *no* dependency
  on `Vault` or `AccountIdentity`, so it can move to
  `pangolin-core/src/session/` with `pangolin-store` taking a
  `pangolin-core` dep and re-exporting `pangolin_core::session::*` from
  `pangolin_store::session::*`. `Vault` and `AccountIdentity` stay in
  `pangolin-store` (a later issue — 1.5 device identity or 1.6 revision
  lineage — moves them, or they stay).
- `pangolin-store/src/vault.rs`, `account.rs`, `search.rs` — all
  reference the session types via `crate::session::*`; under option 2
  they'd reference `pangolin_core::session::*` (or keep `crate::session`
  as a re-export module). Under option 1 (full) the `Vault` move also
  drags `search.rs`'s `SearchIndex` lifecycle, the `meta` / `schema` /
  `blob` modules, and the entire `revisions` machinery into
  `pangolin-core` — a very large diff that churns 1.3's just-merged
  search work and 1.2's account model.

**Estimated diff size:**
- *Option 1 — full relocation.* ~3000–4000 LOC moved + every import site
  in `pangolin-ffi` / `apps/cli` / tests rewired. `pangolin-store`
  becomes a thin re-export shell or is deleted. Highest churn; touches
  1.3's `search.rs` and 1.2's `account.rs` / `blob.rs`. **Not
  recommended for 1.4** — it would make 1.4 unreviewable and risk
  regressing freshly-merged work.
- *Option 2 — partial (session-only).* ~750 LOC moved (`session.rs` →
  `pangolin-core/src/session/mod.rs`), `pangolin-core/Cargo.toml` keeps
  its `pangolin-store` dep (or the arrow flips — see below),
  `pangolin-store/Cargo.toml` gains a `pangolin-core` dep,
  `pangolin-store/src/session.rs` becomes a 5-line re-export, `vault.rs`
  / `account.rs` / `search.rs` import paths change (mechanical). **The
  dep-arrow subtlety:** today `pangolin-core → pangolin-store`. If
  `session.rs` moves to `pangolin-core` and `pangolin-store/vault.rs`
  uses it, then `pangolin-store → pangolin-core` — combined with the
  existing `pangolin-core → pangolin-store` re-export that's a **cycle**.
  Resolution: `pangolin-core` stops re-exporting `Vault` etc. and
  instead `pangolin-store` becomes the upstream crate that
  `pangolin-core` depends on for *everything except* `session` (which
  `pangolin-core` now owns). i.e. `pangolin-core` re-exports
  `pangolin_store::{Vault, AccountId, …}` AND owns `session` natively;
  `pangolin-store` depends on `pangolin-core` *only* for `session` — but
  that's still a cycle. **Cleanest:** move `session.rs` to a fresh tiny
  leaf crate? No — that's over-engineering. **The genuinely clean
  option-2:** move `session.rs` to `pangolin-core`, and break the cycle
  by having `pangolin-core` *not* depend on `pangolin-store` at all for
  the session types' sake (it still re-exports `Vault` from
  `pangolin-store`, so `pangolin-core → pangolin-store` survives), and
  `pangolin-store` *not* depend on `pangolin-core` — instead
  `pangolin-store/src/session.rs` stays the canonical home and
  `pangolin-core/src/session/mod.rs` re-exports it (status quo). **Which
  means option 2 as "physically move session.rs" is blocked by the dep
  cycle unless we also move `Vault`.** The honest conclusion: **the
  relocation is all-or-nothing** — you can't move `session` without
  either moving `Vault` (option 1) or accepting it stays in
  `pangolin-store` (option 3). The middle ground that *does* work:
  **option 2′ — leave the types in `pangolin-store` but make
  `pangolin-core/src/session/mod.rs` and `pangolin-core/src/identity/mod.rs`
  the *documented* canonical import paths** (they already re-export), so
  §16.8's "session lives in `pangolin-core/src/session`" is satisfied
  *namespace-wise* without the physical move and without the cycle.
- *Option 3 — no relocation.* Keep everything in `pangolin-store`, keep
  the re-export pattern. Smallest 1.4 diff; `pangolin-core/src/session/`
  stays a re-export shell. 1.5/1.6 inherit the question.

**Plan recommendation: option 2′** (= keep the physical layout, treat the
`pangolin-core` re-export modules as canonical, document the deferral to
"the issue that moves `Vault`" — likely never, or MVP-2 when chain code
forces a `pangolin-core` reorg). Rationale: the dep-cycle analysis shows
"move session but not Vault" is not actually possible without a crate
split; "move everything" is a 4kLOC diff that churns 1.2 + 1.3; and the
namespace freeze that §16.8 cares about is already satisfied by the
re-exports. **But this is Kelvin's call** — if he wants the full
relocation, 1.4 is the place the master plan named for it, and the
estimated +1 week of build time is the cost.

### 6. FFI surface changes

1. **`vault_create` / `vault_open`** — bodies. (1.1 said "1.3"; 1.3
   didn't wire them — it focused on the search index. 1.4 wires them
   since `vault_unlock`/`lock` need a real handle.) `vault_create(path,
   password)` → `Vault::create`; `vault_open(path)` → `Vault::open`,
   storing the `Vault` in the `VaultHandle`'s `Mutex<Option<Vault>>` (the
   1.2 slot). Returns the `VaultHandle` Object / `()`.
2. **`vault_unlock(handle, password, presence)`** — verify both proofs:
   wrap `password` in a `PinIdentityProof`, `presence.bytes` (ignored for
   the CLI tier) → `PressYPresenceProof::confirmed()`, call
   `Vault::unlock`. Returns a `SessionInfo` populated from
   `Vault::session_state` / `session_remaining` (see #4 below).
3. **`vault_lock(handle)`** → `Vault::lock`. **`vault_close(handle)`** →
   `Vault::close` (consumes the inner `Vault`, leaves the handle empty).
4. **`session_status(handle) -> SessionInfo`** — read-only. **The
   1.1-frozen `SessionInfo` is `{schema_version, last_refresh_unix:
   i64, is_active: bool}`** — that's enough to *report* but not enough
   for a host UI to render a countdown or show the configured duration.
   1.4 adds (additive — allowed post-lock): `idle_deadline_unix: i64`,
   `absolute_deadline_unix: i64`, `configured_idle_secs: i64`,
   `last_presence_fresh_until_unix: i64`. Flag as an additive amendment
   in `ffi-surface.md` (same posture as 1.2's record widening). The
   bridge maps `Vault::session_state` → these fields;
   `last_refresh_unix` ← `last_proof_at`.
5. **`session_extend(handle) -> SessionInfo`** — this is the "extend long
   sessions" high-risk action (§5.4). 1.1 froze it as
   `session_extend(handle)` — *no presence proof argument*. That's a
   problem: §5.4 says extending requires presence. **Resolution:** 1.4
   amends the signature to `session_extend(handle, presence:
   PresenceProof) -> Result<SessionInfo>` (additive *argument*, which is
   technically a wire-shape change — but, like 1.2's Q1, nothing
   external binds yet, so it's safe). Body: verify the presence proof,
   then `touch_session` (which re-extends the idle deadline, still capped
   at absolute-max). Flag in `ffi-surface.md`.
6. **`reveal_notes` / `reveal_password_history` / `reveal_totp_secret`**
   — new (see §4.5 above). New file `pangolin-ffi/src/reveal.rs`.
7. **`account_search` signature** — **frozen, do not touch.** The
   internal `Vault::account_search(&mut self, &str) ->
   Result<Vec<AccountIdentitySummary>>` stays. If `AccountIdentitySummary`
   moves crates (it won't under option 2′), the bridge's import path
   changes but the wire shape (`Vec<AccountSnapshot>`) does not.
8. **`account_get` / `AccountSnapshot` secret fields** — see Q5: 1.4
   strips `current_password` / `password_history` / `totp_secret` off the
   FFI `AccountSnapshot` and replaces them with non-secret metadata
   (`password_history_count: u32`, `has_totp: bool`) — matching what 1.2
   *intended* (1.2's plan §D said `account_get` "does not include the
   password value or notes"; the implemented FFI shape over-shared). The
   plaintext now flows *only* through the presence-gated `reveal_*`
   entries. This is a *removal* from a frozen shape — strictly a §16
   escalation, which is why Q5 surfaces it for Kelvin; the alternative
   (leave the over-sharing) defeats §5.4.
9. **`error.rs`** — `pangolin_core::Error` gains no new *variant*
   (`Session(String)` already covers `PromptTimedOut` via the `From`
   mapping); `pangolin-ffi/src/error.rs`'s `FfiError::Session` is the
   FFI-side category. The `From<StoreError>` mapping in
   `pangolin-core/src/error.rs` extends to map `StoreError::PromptTimedOut`
   → `Session(...)`.
10. Files: `pangolin-ffi/src/{session.rs, identity.rs, identity_bridge.rs,
    reveal.rs (new), lib.rs, error.rs}`, `pangolin-core/src/error.rs`,
    `docs/architecture/ffi-surface.md`.

### 7. Config / schema versioning

The configurable idle duration lives in `meta.session_idle_secs`
(`NULL`/absent ⇒ 15 min for pre-1.4 vaults). No `format_version` bump —
the column is additive and absence is a valid (default) state, exactly
1.3's `fts_meta` doctrine. The §18.7 reject/migrate policy (1.6) will own
"if a future schema version constrains the duration set, validate it"; 1.4
only adds the slot + the validator (`SessionDuration::from_secs` rejects
out-of-set values with `Validation { kind: "session_duration", … }`).
Leave the hook; don't design the policy.

   30 min), the deadline reflects it.

   4 h+ ⇒ after `ABSOLUTE_MAX_DEFAULT`, even an active session expires
   regardless of touches; `next_idle_deadline` never returns past
   `session_started_at + ABSOLUTE_MAX_DEFAULT`.

   a successful op; `touch_extends_idle_deadline` /
   `session_remaining_decreases_with_time` pass with the configured idle.

    successful presence succeeds *without* a new proof verify (dedup); a
    `reveal_*` after 60 s requires a fresh proof; two `reveal_*` calls
    within 60 s of the first ⇒ the proof verifies once.
11. **Prompt timeout:** a `reveal_*` supplied a presence proof whose
    `created_at` is >60 s old ⇒ `PromptTimedOut` (typed, not silent,
    distinct from `AuthenticationFailed`).
12. **Mid-action resume:** `with_session(op, reauth)` where the session
    is expired ⇒ `reauth` runs (re-unlock or re-supply presence) ⇒ `op`
    runs and returns its `T`; if `reauth` returns `Err`, `op` does not
    run; if `reauth` returns `Ok` but the vault is not actually `Active`,
    the post-reauth freshness re-check surfaces `SessionExpired`

13. **Reveal-class:** `reveal_notes` / `reveal_password_history` /

## Failure modes considered

- **Clock skew (system time jumps backward mid-session).** `SystemClock`
  reads `SystemTime::now()` each call (no caching); a backward jump can
  spuriously extend or expire a session. PoC accepts this (documented in
  `session.rs`); 1.4 keeps the posture. *Mitigation hook:* MVP-3/4 may
  switch to `Instant` (monotonic) — leave the `Clock` trait as the seam.
- **Presence-proof replay.** `PressYPresenceProof::verify` is single-use
  (interior `Cell<bool>`); a second verify returns
  `PresenceAlreadyConsumed` → `AuthenticationFailed`. The new
  `last_presence_at` dedup never *replays* a proof — it skips the verify
  entirely when presence is already fresh. MVP-3/4 hardware proofs carry
  a nonce + timestamp; the trait shape supports it.
- **Session expires *mid-op* (e.g. during a 1.5 s Argon2id inside `op`).**
  Not retried — `op` returns whatever it returned; `with_session` only
  re-prompts on expiry detected *at or before* the op starts. Documented;
  MVP-3+ transactional-retry wrapper is out of scope.
- **`reauth` callback lies (returns `Ok(())` without re-`Active`).** The
  L-3 post-reauth `check_session_freshness()` re-check catches it ⇒
  `SessionExpired`; `op` never runs against an invalid session.
- **Reveal while expired.** `check_session_freshness` runs first, returns
  `SessionExpired` *before* the presence proof is consumed — the caller
  can re-auth and retry with the same proof envelope. Same for the
  `refuse_if_frozen` check (P8 CRIT-1).
- **Prompt timed out vs. wrong proof — distinguishability.** 1.4
  *intentionally* distinguishes `PromptTimedOut` (stale proof at a reveal
  site) from `AuthenticationFailed` (proof failed for another reason).
  This is *not* an oracle: a timed-out prompt reveals nothing about any
  secret; it's a UX signal ("re-run the command"). The
  `AuthenticationFailed` collapse for *content-class* failures (wrong
  PIN etc.) is preserved.
- **Configurable-idle downgrade attack on disk.** An attacker who edits
  `meta.session_idle_secs` to `14400` (4 h) on a stolen `.pvf` lengthens
  the *next* session — but they need the password to unlock anyway, and
  the absolute-max ceiling (4 h, not configurable) bounds it. Setting it
  to an out-of-set huge value is rejected by `SessionDuration::from_secs`
  at unlock time (treated as "use default 15 min" + a warning, *not* an
  unlock failure — a corrupt-but-decryptable meta field shouldn't brick
  the vault). Documented.
- **The dep-cycle trap if option 2 (partial relocation) is mis-applied.**
  Moving `session.rs` to `pangolin-core` while `pangolin-store/vault.rs`
  still uses it creates `pangolin-store → pangolin-core` *and*
  `pangolin-core → pangolin-store` (the `Vault` re-export) — a cycle that
  won't compile. The build agent MUST NOT attempt the partial move; only
  option 1 (full) or option 2′/3 (no physical move) are buildable. This
  is the single biggest scope landmine — Q1 must be answered before the
  builder starts.
- **`apps/cli` import breakage on a full relocation.** `apps/cli` imports
  `pangolin_store::session::PressYPresenceProof` and
  `pangolin_store::Vault` directly (~6 sites). A full relocation must
  rewire them (or `pangolin-store` re-exports the moved types back — but
  that's the cycle again). Grep `apps/cli` for `pangolin_store::` before
  the move.
- **FFI shape removal (`AccountSnapshot` secret fields).** Strictly a §16
  escalation (removing from a frozen shape). Surfaced as Q5 for Kelvin's
  explicit sign-off; the alternative (leave the over-sharing) violates
  §5.4's "high-risk reveal requires presence even mid-session".
- **CI Linux umask quirk #1.** The session rewrite touches
  `Vault::create`/`open`'s `UmaskGuard` region; leave it intact. Any new
  test that writes a fixture `.pvf` reuses the existing
  `UmaskGuard::Drop` / `nix::umask(0o022)` pattern.
- **HIGH-1 / Q3 dep arrows after a reorg.** `pangolin-crypto` has no
  upward deps so it can't gain serde/uniffi regardless. After any
  relocation, re-run `cargo tree -p pangolin-crypto | grep -ci serde`
  (must be 0) and `cargo tree -p pangolin-core | grep -ci uniffi` /
  `-p pangolin-store | grep -ci uniffi` (must be 0). `pangolin-core` /
  `pangolin-store` types must carry **no** `uniffi::` annotations — those
  live only in `pangolin-ffi`'s wrapper types; the bridge converts.

## Open questions for Kelvin

### Q1 — The relocation: full / partial / none?

**Background.** 1.1 Q2 and 1.2 Q2 both deferred the `Vault` /
`AccountIdentity` / session-type relocation from `pangolin-store` →
`pangolin-core` to "1.4's session rewrite". §16.8 (amended by 1.1) names
`pangolin-core/src/session/` and `pangolin-core/src/identity/` as the
canonical homes. **But the dep-arrow analysis (Approach §5) shows
"move session but not Vault" is not buildable** — `session.rs` has no
dep on `Vault`, but moving it to `pangolin-core` while
`pangolin-store/vault.rs` uses it creates a cycle with the existing
`pangolin-core → pangolin-store` re-export. So the real choice is binary:

- **Option 1 — full relocation.** Move `Vault`, `AccountIdentity`, the
  session/proof types, `RevisionGraph`, `search.rs`'s `SearchIndex`,
  `meta`/`schema`/`blob`, all of it into `pangolin-core`. `pangolin-store`
  becomes a thin re-export shell (or is deleted; `apps/cli` switches to
  `pangolin_core::*`). **~3000–4000 LOC moved + every import in
  `pangolin-ffi`/`apps/cli`/tests rewired; churns 1.2's `account.rs`/
  `blob.rs` and 1.3's `search.rs`.** Estimated +5–8 days on top of the
  base 1.4 work. It's the §16.8-correct end state.
- **Option 2′ — keep the physical layout; treat the `pangolin-core`
  re-export modules as canonical.** `pangolin-core/src/session/mod.rs`
  and `identity/mod.rs` already re-export the real types; document them
  as *the* import path; defer the physical move to "whenever a future
  reorg (likely MVP-2's chain work) forces it, or never". Dep arrows
  unchanged; ~0 LOC of relocation; §16.8's *namespace* intent satisfied.
- **Option 3 — no relocation, no doc change.** Status quo. 1.5/1.6
  inherit the question explicitly.

**Recommendation: Option 2′.** The dep cycle makes the "clean partial
move" impossible; the full move is a 4kLOC diff that would make 1.4
unreviewable and risk regressing freshly-merged 1.2/1.3 work; the
namespace freeze §16.8 cares about is already in place. If Kelvin wants
the §16.8-correct physical layout *now*, option 1 is the named place for
it — but it should arguably be its own issue (1.4a) rather than ride 1.4.
**1.5 (device identity) and 1.6 (revision lineage) both touch these
types — if 1.4 doesn't move them, those issues face the same question;
note that 1.6's revision-lineage rewrite is the *other* natural batch
point.**

### Q2 — Production CLI presence + identity proof forms?

**Background.** MVP-1 is CLI-only; the real presence proof
(biometric / device-unlock / NFC) lands in MVP-3 (mobile) / MVP-4
(desktop). The PoC uses `PinIdentityProof` (the password, structurally
validated then run through KDF+AEAD) and `PressYPresenceProof` (a
single-use terminal-confirmation token with a 60 s freshness window).
The trait abstraction is real and `dyn`-compatible (P4-2's "trait-based
so real proofs slot in MVP-1").

**Options:** (1) **(Recommended) Keep `PinIdentityProof` +
`PressYPresenceProof` as the "CLI tier" impls, document them as such, no
API change.** They already carry the security disciplines (single-use
replay resistance, zeroize, redacted `Debug`); the traits are the seam.
(2) Add a `TerminalPresenceProof` rename + a `PlatformAuthenticatorStub`
identity proof now. Over-engineering — no platform-authenticator code to
back it. (3) Drop the presence proof to a no-op for CLI ("confirm at the
terminal" is theatre). Rejected — it violates the 2-proof invariant and
the high-risk-reveal gate; the terminal confirmation *is* the CLI's
presence proof.

**Recommendation: Option 1.** The plan assumes it.

### Q3 — Presence-freshness window value (the spec says "30–60 s")?

**Background.** §7.6 says presence proofs are valid for "30–60 seconds".
The PoC adopts the upper bound: `PRESENCE_FRESHNESS = 60s`. The dedup
window (Approach §3) and the `PressYPresenceProof::verify` staleness
check both key off this.

**Options:** (1) **(Recommended) Keep 60 s** — the spec's permitted
upper bound; smoothest UX; matches the PoC. (2) 45 s (the midpoint).
(3) 30 s (the lower bound — strictest; more re-prompts). (4) Make it a
field on `PresenceProof::freshness()` per impl (already is) *and*
user-configurable in `meta` — over-engineering for 1.4; the spec doesn't
ask for it.

**Recommendation: Option 1 — 60 s, documented as the upper bound.**

### Q4 — Are the `reveal_*` FFI entry points a 1.1-surface amendment?

**Background.** 1.1 froze the FFI surface: `vault_*`, `session_status`,
`session_extend(handle)` *with no presence arg*, `account_*` — but **no
`reveal_*` entry points**. 1.4 needs to add `reveal_notes` /
`reveal_password_history` / `reveal_totp_secret` (per the task's §C and
1.2's audit C-1 deferral) *and* amend `session_extend` to take a
presence proof (§5.4 says extending requires presence). Nothing external
binds the 1.1 surface yet (no shells started). Per the ffi-surface drift
discipline, additions are legal; an *argument-type change* (`session_extend`)
strictly needs a 1.1-amendment commit landed first — same posture as
1.2's Q1 (which widened `AccountDraft`'s field types under a 1.1
amendment riding 1.2's merge).

**Options:** (1) **(Recommended) Amend in 1.4's merge** — add the
`reveal_*` entries + the `session_extend` presence arg + the
`SessionInfo` widening + the `AccountSnapshot` narrowing (Q5), all in the
1.4 commit, with `ffi-surface.md` updated and a "1.1 amendment" note.
(2) Add only the additive `reveal_*` entries; leave `session_extend`
without presence (violates §5.4). (3) Defer the FFI reveal surface to a
later issue (leaves 1.2's C-1 hole open longer). Rejected.

**Recommendation: Option 1.** The plan assumes it.

### Q5 — Wire `ingest_chain_revision`'s FTS5 resync? And presence-gate the existing FFI `AccountSnapshot` secret fields?

**Background — two coupled sub-questions.** (a) 1.3 left
`ingest_chain_revision` not resyncing the `:memory:` FTS5 index ("revisit
in 1.4"). `ingest_chain_revision` is MVP-1-dormant chain code (the CLI
test path can reach it, but it's an MVP-2 concern). (b) 1.2's FFI
`AccountSnapshot` carries `current_password` + the *full* `password_history`
+ `totp_secret` (as `Arc<SecretPassword>` / `Arc<TotpSecret>`) through
the *non-presence-gated* `account_get` / `account_search` path — which
**conflicts with §5.4's "high-risk reveal requires presence even
mid-session"** and with 1.2's own plan §D ("`account_get` … does not
include the password value or notes"). The implemented FFI shape
over-shared relative to the plan.

**Options:**
- (a) **(Recommended) Leave the FTS5 resync to MVP-2.** When
  `ingest_chain_revision` writes a revision, mark the FTS5 row stale (or
  just rely on the next `unlock` rebuild — the index is `:memory:` and
  rebuilt every unlock anyway). MVP-2's chain-ingest issue owns the
  proper resync. Alternative: wire it now (small — ~30 LOC mirroring the
  `account_update` sync site) — defensible but it's polishing dormant
  code.
- (b) **(Recommended) Strip the secret-bearing fields off the FFI
  `AccountSnapshot`** (`current_password` / `password_history` /
  `totp_secret` → replaced by `password_history_count: u32` +
  `has_totp: bool`) and surface the plaintext *only* through the new
  presence-gated `reveal_*` entries. This is what 1.2's plan intended;
  it's a *removal* from a frozen shape (§16 escalation — hence this
  question), but the alternative (leave the over-sharing) means
  `account_get` is a presence-free password-reveal, which §5.4 forbids.
  Alternative: keep the fields but document them as "metadata-only
  handles you must pass back into a reveal" — but the bytes are *right
  there* on the snapshot; that's not a real gate.

**Recommendation: (a) defer the FTS5 resync; (b) strip the secret fields
and route plaintext through `reveal_*`.** Both flagged because (a) is a
"do we polish dormant code" call and (b) is a knowing removal from a
frozen FFI shape that needs an on-the-record sign-off.
