# Issue 1.11: Capture-authority registration primitive

> One-line scope: a local-only **registry of which component owns credential capture per context** (browser / desktop / mobile_os) — Browser-Ext spec §2.3's "Capture Authority Rule" (one component per context) made into a Core-engine API + a small persisted table — so a future browser extension (MVP-2), desktop shell (MVP-4), or mobile-OS autofill extension (MVP-5) can register itself / yield to a more authoritative one. This is the **last functional issue of MVP-1** (the threat-model doc 0.2 is the only other open MVP-1 item).
>
> **Status:** Plan-gate **signed off** by Kelvin on 2026-05-13. All 6 open questions resolved (see "Resolved decisions" below). The "Decisions locked" table is the single source of truth; if anything in the "Approach" body drifts from a row in the table, the row wins.
>
> **Security-critical: YES.** Master plan §16.3 lists "capture authority" by name. 1.11 (a) defines a security *invariant* — "only one component may prompt for a browser credential event" (Threat Model invariant #8, Browser-Ext spec §2.3); (b) lands a new FFI surface body (`capture_authority_register`) and finalises two 1.1-frozen Records (`CaptureAuthority`, `CaptureContext`); (c) persists a new vault-level table whose contents inform every future capture-flow message. Same plan-gate cadence as 1.4 / 1.9 / 1.10.
>
> **Depends on:** 1.1 (the FFI placeholder), 1.2 (Vault + NFC discipline), 1.4 (session-class semantics), 1.5 (the additive `CREATE TABLE IF NOT EXISTS` template + per-row `schema_version`), 1.6 (§18.7 schema-versioning policy), 1.10 (`ArchiveSnapshot` additive growth). ## Goal

A persisted, queriable, exclusive-per-context registry of capture authorities, exposed through a session-class FFI surface and (per Q-d) a `pangolin-cli` query path. Concretely, MVP-1 ships:

1. A new SQL table `capture_authorities` (L2). Plaintext (this is non-secret metadata — version strings + public identifiers — same posture as the `devices` table's `label`).
2. A typed Rust API on `Vault`: `register(authority, context, replace_existing) -> Result<RegistrationOutcome>`, `query(context) -> Result<Option<...>>`, `list() -> Result<Vec<...>>`, `clear(context) -> Result<bool>` (test/MVP-2 helper; not on the FFI surface in 1.11).
3. The FFI body for `capture_authority_register`, plus `capture_authority_query` / `capture_authority_list` (L9 / L12).
4. Finalised `CaptureAuthority` / `CaptureContext` Records + `CaptureAuthorityKind` / `CaptureContextKind` enums + `CaptureAuthorityEntry` Record (L5).
5. Round-trip through 1.10's `ArchiveSnapshot` per L10.
6. (Per Q-d) `pangolin-cli authority list` read-only subcommand.
7. New `docs/architecture/capture-authority.md`; updates to `ffi-surface.md`, `schema-versioning.md`, `THREAT_MODEL.md`, `DEVLOG.md`.

**Non-goals (per L1):** per-credential anti-phishing origin binding (MVP-2); on-chain authority/entitlement registry (MVP-2); JSON-RPC server / native-messaging host (MVP-4); the actual autofill matching engine (MVP-2+); browser-extension code (MVP-2 / MVP-4).

## Approach

### 1. The new `capture_authorities` table

```sql
-- MVP-1 issue 1.11: Browser-Ext spec §2.3 / API contract §16. Vault-level
-- metadata; additive CREATE TABLE IF NOT EXISTS (no format_version bump).
-- Legacy vaults pick it up on next open via apply_pragmas_and_schema.
-- All fields non-secret: component identifier + version string + context
-- discriminator. Same posture as the devices table's label / last_sync_at.
CREATE TABLE IF NOT EXISTS capture_authorities (
    context_kind        INTEGER NOT NULL,  -- 0=Desktop, 1=BrowserExtension, 2=MobileOsAutofill
    platform_hint       TEXT    NOT NULL DEFAULT '',
    component_id        TEXT    NOT NULL,
    component_version   TEXT    NOT NULL,
    registered_at       INTEGER NOT NULL,  -- advances on every replace
    schema_version      INTEGER NOT NULL,  -- §18.7 per-row ladder
    PRIMARY KEY (context_kind, platform_hint)
);
```

### 2. `pangolin-store::capture_authority` (new submodule)

Public API on `Vault`:
- `register(authority, context, replace_existing) -> Result<RegistrationOutcome>` — preamble `check_session_freshness()`; validate; `IMMEDIATE` transaction: lookup → INSERT (no existing row) or REPLACE (only if `replace_existing` AND payload differs) or reject (`Validation { kind: "capture_authority_exclusivity" }`); `touch_session()`; return `RegistrationOutcome::{Created, Replaced { prior }, NoOp { existing }}`.
- `query(context) -> Result<Option<CaptureAuthorityEntry>>` — single-row.
- `list() -> Result<Vec<CaptureAuthorityEntry>>` — table scan sorted by `(context_kind, platform_hint)`.
- `clear(context) -> Result<bool>` — DELETE for tests + MVP-2 "extension uninstalled"; not on FFI.
- All require an unlocked vault (`NotUnlocked` if locked); none presence-gated.

### 3. The FFI module — `crates/pangolin-ffi/src/capture_authority.rs` (new)

1.11 also moves `CaptureAuthority` / `CaptureContext` / `capture_authority_register` OUT of `kdbx.rs` (where the 1.1 scaffold parked them) into this new module; `lib.rs` re-exports update accordingly.

```rust
#[derive(Debug, Clone, Copy, uniffi::Enum)]
pub enum CaptureAuthorityKind { Desktop, BrowserExtension, MobileOsAutofill }

#[derive(Debug, Clone, Copy, uniffi::Enum)]
pub enum CaptureContextKind { Desktop, Browser, MobileOs }

#[derive(Debug, Clone, uniffi::Record)]
pub struct CaptureAuthority {
    pub schema_version: u16,
    pub kind: CaptureAuthorityKind,
    pub component_id: String,
    pub component_version: String,
}

#[derive(Debug, Clone, uniffi::Record)]
pub struct CaptureContext {
    pub schema_version: u16,
    pub kind: CaptureContextKind,
    pub platform_hint: Option<String>,
}

#[derive(Debug, Clone, uniffi::Record)]
pub struct CaptureAuthorityEntry {
    pub schema_version: u16,
    pub context: CaptureContext,
    pub authority: CaptureAuthority,
    pub registered_at: UnixTimestamp,
}
```

Entry points: `capture_authority_register(h, authority, context, replace_existing)`, `capture_authority_query(h, context)`, `capture_authority_list(h)`. Error mapping extends `error_taxonomy.rs` with new `StoreError::CaptureAuthorityExclusivity { context }` → `FfiError::Validation { kind: "capture_authority_exclusivity" }` (message names the context kind only — no info leak on the existing component_id); `StoreError::CaptureAuthorityValidation { reason }` → `FfiError::Validation { kind: "capture_authority" }`.

### 4. CLI — `pangolin-cli authority list` (per Q-d)

```
pangolin-cli authority list --vault-path <pvf> [--json]
```

Two-proof unlock (same shell as `vault export`); prints one line per registered authority (sorted by context_kind, platform_hint); `--json` → JSON-Lines on stdout. ~30 lines. Defer `register` / `clear` CLI to MVP-2 when an extension exists.

### 5. 1.10 archive round-trip

Add an optional `capture_authorities: Vec<CapturedCaptureAuthority>` field to `ArchiveSnapshot` at the end of the array (additive). A decoder that doesn't see the field treats it as empty. `restore_to_new_vault` ignores it per Q-f (new vault starts with an empty registry).

## Resolved decisions

Plan-gate sign-off by Kelvin on 2026-05-13. Each ruling preserved with its reasoning.

### R-a — Spec interpretation

**Confirmed L1.** "Capture authority" = the *component owning capture for a context* per Browser-Ext spec §2.3 / Threat-Model invariant #8 / API contract §16. Master plan §4 row 1.11's wording is unambiguous; the other readings (per-credential anti-phishing origin binding, on-chain authority/entitlement registry, Revision-Log capture authority) are explicitly different concepts that land in MVP-2.

### R-b — Identifier shape

**Locked: closed `uniffi::Enum` for kind + allowlisted lowercased ASCII `platform_hint` + NFC-bounded `component_id`/`component_version` (= L5/L7).** Structural exclusivity invariant (Threat-Model #8 becomes a type-system invariant, not a string convention); defeats Unicode-homoglyph impersonation (`chr​ome` with a zero-width-space). MVP-2 wanting a 4th kind = a §18.7 minor bump (additive enum variant, accepted).

### R-c — Auth tier: HYBRID

**Locked: hybrid — session-class for the first register, presence-required (reveal-class) for replace (= L6).** Plain session-class is enough until an extension actually exists to be impersonated; the moment MVP-2 ships the browser extension and the registry gates real capture routing, a stale-session attacker silently replacing the honest registration becomes a real attack path. The hybrid is precisely tuned to that threat shape — first register is benign metadata setup (no trust to overthrow); replace is the security-relevant moment. Adopting it now means zero migration in MVP-2, the threat model is correct from day one, no "1.11 was too loose, we tightened it in 2.x" record. Implementation cost is small — the 1.4 `ensure_presence_fresh` machinery already exists; threading it through the `Replaced` branch is a few extra lines.

(The plan doc's initial recommendation was plain session-class with a "revisit at MVP-2" note. Kelvin's plan-gate question — "best sets up the planned future" — flipped the default to the hybrid that MVP-2 would have demanded anyway.)

### R-d — CLI surface

**Locked: read-only `pangolin-cli authority list` only.** Forensic / debugging / audit visibility right now (~30 LoC + a small integration test); `list` stays useful across every future MVP (it's what the MVP-4 desktop-GUI settings panel will show too). `register` / `clear` CLI subcommands defer to MVP-2 — by then the browser extension's native-messaging host is the *real* consumer of those flows; CLI `register`/`clear` written now become vestigial. Don't build code that's orphaned in one cycle.

### R-e — Replace posture

**Locked: reject-on-conflict by default; opt-in `replace_existing: bool` flag; `registered_at` advances on every replace as a tiny audit trail.** Structural exclusivity invariant — a buggy or hostile caller can't double-register silently. The `registered_at` timestamp gives a forensic trail in MVP-1 without duplicating substrate MVP-2 will rebuild on-chain (the signed Revision Log + entitlement registry is the *real* home for a chronological audit log; building a full local `capture_authority_log` table now means writing infrastructure MVP-2 deprecates).

### R-f — Restore from archive

**Locked: always start fresh — new vault has 0 registered authorities.** Mirrors 1.10's `let _ = &snapshot.devices` "ignored on restore" decision. The destination is a new environment (new device, possibly new OS); the source's registration is stale; the user re-registers helpers on the new device (when they're also re-installing extensions anyway). The archive *does* carry the registry (additive optional CBOR field) so a future MVP-3+ "advanced restore" flow could opt to re-register; 1.11 keeps it simple.
