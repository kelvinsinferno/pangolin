# Issue P0-1: Cargo workspace + rustfmt/clippy + GH Actions CI

## Spec reference
Master plan §3.7 EPIC: Foundation (P0): "Cargo workspace, rustfmt/clippy, GH Actions CI, MIT-or-Apache-2 license placeholder."
Master plan §16.8 repository layout for crate names and structure.

## Goal
Establish a buildable Cargo workspace with all seven Pangolin crates listed in §16.8, configured with rustfmt + clippy, and a GitHub Actions CI workflow that runs on every PR. After this issue, every subsequent Rust issue (P1 series, P2 series, etc.) can immediately add code to its crate without further scaffolding work, and CI catches formatting/clippy regressions automatically.

## Approach
1. Top-level `Cargo.toml` declares a workspace with members: `pangolin-core`, `pangolin-crypto`, `pangolin-store`, `pangolin-chain`, `pangolin-indexer`, `pangolin-funder-client`, `pangolin-cli`. Resolver = "2". Workspace-level `package.rust-version`, `package.edition = "2021"`, `package.license = "Apache-2.0"`.
2. Each crate gets a minimal `Cargo.toml` and a placeholder `src/lib.rs` (or `src/main.rs` for `pangolin-cli`) that compiles cleanly with no warnings under `clippy::all` and `clippy::pedantic` (we'll allow some pedantic lints repo-wide via workspace lints).
3. Workspace `[workspace.lints]` enables `clippy::all`, `clippy::pedantic`, `clippy::nursery` with a small list of explicit allows (e.g., `module_name_repetitions`).
4. `rustfmt.toml` at workspace root: `edition = "2021"`, `max_width = 100`, `imports_granularity = "Crate"`, `group_imports = "StdExternalCrate"`.
5. `.github/workflows/ci.yml`: jobs for fmt, clippy, test on Ubuntu + Windows + macOS. Use stable Rust toolchain pinned via `rust-toolchain.toml` (use a recent stable, e.g., 1.83 or whatever the current stable is at runtime).
6. `rust-toolchain.toml` pins the toolchain channel and components (rustfmt, clippy).

## Public surface
Workspace structure (no public Rust API yet — every crate exposes a single `pub fn name() -> &'static str` placeholder so CI can verify the build chain end-to-end):

```
crates/
├── pangolin-core/         lib.rs         pub fn name() -> &'static str { "pangolin-core" }
├── pangolin-crypto/       lib.rs         "pangolin-crypto"
├── pangolin-store/        lib.rs         "pangolin-store"
├── pangolin-chain/        lib.rs         "pangolin-chain"
├── pangolin-indexer/      lib.rs         "pangolin-indexer"
├── pangolin-funder-client/lib.rs         "pangolin-funder-client"
└── pangolin-cli/          main.rs        prints "pangolin v0.0.0"
```

CI workflow surface:
- `fmt` job: `cargo fmt --all --check`
- `clippy` job: `cargo clippy --workspace --all-targets -- -D warnings`
- `test` job: `cargo test --workspace --all-targets`

## Success criteria
1. `cargo build --workspace` succeeds on a clean clone (Linux, Windows, macOS).
2. `cargo fmt --all --check` succeeds.
3. `cargo clippy --workspace --all-targets -- -D warnings` succeeds with zero warnings.
4. `cargo test --workspace` succeeds (no real tests yet — the placeholder `name()` functions exercise build).
5. `cargo run -p pangolin-cli` prints `pangolin v0.0.0` and exits 0.
6. CI workflow file is valid (passes `actionlint` if available; at minimum, GitHub Actions accepts it).
7. The seven crates are listed in `Cargo.toml` workspace members and each has a `Cargo.toml` with name, version 0.0.0, edition 2021, Apache-2.0 license.

## Test plan

## Failure modes considered
- **Toolchain drift across OSes** → mitigated by `rust-toolchain.toml` pinning the channel; CI uses `dtolnay/rust-toolchain@stable` action with channel from the file.
- **Clippy nursery lints flake on toolchain bumps** → workspace lints allow specific noisy lints; pedantic + nursery enabled but with documented allows.
- **Windows path / line-ending issues** → `.gitattributes` will normalize line endings (will add as part of this issue).
- **`cargo fmt --check` failing on Windows due to CRLF** → handled by `.gitattributes` + rustfmt config.

## Rollback / abort plan
If after 2 hours of work CI is still red on one OS, drop that OS from the matrix and file a follow-up issue. Linux + macOS CI is sufficient for P0; Windows CI can be added separately.
If the workspace lints level (pedantic + nursery) produces unmanageable noise, drop to `clippy::all` only and file a follow-up to tighten later.

## Out of scope
- Real implementation in any crate (placeholder `name()` only — that's P1 and beyond).
- Deny lists for specific lints — only the workspace-wide allowlist for known-noisy lints.
- Code coverage tooling (deferred to issue 18.9 — Test Infrastructure).
- Foundry/Solidity CI — separate workflow added in P5-1.
