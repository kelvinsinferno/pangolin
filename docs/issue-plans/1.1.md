### B. FFI scaffolding shape (Q3 = separate crate)

1. **Generation strategy:** UniFFI **proc-macro mode** (the modern path), not
   UDL. Annotate types in `pangolin-ffi` (which re-exports / wraps types from
   `pangolin-core`) with `#[uniffi::export]` /
   `#[derive(uniffi::Record/Enum/Object/Error)]`. UniFFI emits Swift + Kotlin
   bindings at build time via `uniffi-bindgen`.
2. **Tauri C-ABI:** generated with `cbindgen` from `pangolin-ffi` against a
   thin `extern "C"` surface that hand-mirrors a subset of the UniFFI surface.
   Tauri's Rust shell can call the regular `pangolin-core` API directly when
   it's in-process; the C-ABI is for situations where the desktop shell
   wants the same calling convention as mobile (and for native-messaging
   bridges from the browser extension in MVP-4).
3. **Build pipeline:**
   - `cargo build -p pangolin-ffi` produces a `staticlib`/`cdylib`.
   - `cargo run -p pangolin-ffi --bin uniffi-bindgen -- generate ...` (or the
     equivalent build script) emits `pangolin.swift` and `pangolin.kt` into
     `target/ffi-bindings/{swift,kotlin}/`.
   - `cargo run -p pangolin-ffi --bin cbindgen-build` emits `pangolin.h` into
     `target/ffi-bindings/c/`.
   - CI step `ffi-bindings` runs all three on Linux + Windows + macOS and
     uploads the artefacts so MVP-5 (iOS/Android) and MVP-4 (Tauri) consume
     a known-good set without re-running bindgen locally.
4. **Crate layout for `pangolin-ffi`** (created this issue, body grows over
   1.2-1.11):
   ```text
   crates/pangolin-ffi/
   ├── Cargo.toml          # depends on pangolin-core, pangolin-totp, pangolin-kdbx, uniffi
   ├── build.rs            # uniffi::generate_scaffolding for crate-internal types
   ├── src/
   │   ├── lib.rs          # crate-level uniffi setup; re-exports
   │   ├── identity.rs     # FFI shapes for AccountIdentity (1.2)
   │   ├── session.rs      # FFI shapes for Session state machine (1.4)
   │   ├── revision.rs     # FFI shapes for revision lineage (1.6)
   │   ├── totp.rs         # FFI shape for TOTP code generation (1.7)
   │   ├── kdbx.rs         # FFI shape for KDBX import flow (1.9)
   │   ├── error.rs        # FfiError mapping from pangolin_core::Error
   │   └── cabi.rs         # extern "C" surface for Tauri/cbindgen
   └── tests/
       └── roundtrip.rs    # smoke: every FFI type survives bindgen → reparse
   ```

### E. Lint, formatting, MSRV (Q5 = pin to 1.94.0)

- **MSRV:** `rust-toolchain.toml` set to `channel = "1.94.0"`. Workspace
  `rust-version = "1.94"`. CI was last green on this stable; deliberate bumps
  via PR going forward (matches the foundry-toolchain pinning approach).
- **`unsafe_code`:** retained as `forbid` on every crate **except**

  workspace `disallowed_methods` clippy lint that bans `serde_*` types

## Public surface

The frozen FFI surface for MVP-1 (every type either materialises as a
UniFFI-annotated declaration in `pangolin-ffi/src/<module>.rs` or as a thin
`extern "C"` shim in `pangolin-ffi/src/cabi.rs`):

```rust
// Vault lifecycle (impl arrives in 1.3 / 1.4)
pub fn vault_create(path: &str, password: &SecretPassword) -> Result<(), FfiError>;
pub fn vault_open(path: &str)                              -> Result<VaultHandle, FfiError>;
pub fn vault_unlock(h: &VaultHandle, p: &SecretPassword,
                    presence: PresenceProof)                -> Result<SessionInfo, FfiError>;
pub fn vault_lock(h: &VaultHandle)                          -> Result<(), FfiError>;
pub fn vault_close(h: VaultHandle)                          -> Result<(), FfiError>;

// Identity (1.2)
pub fn account_add(h: &VaultHandle, draft: AccountDraft)    -> Result<AccountId, FfiError>;
pub fn account_update(h: &VaultHandle, id: AccountId, patch: AccountPatch)
                                                            -> Result<RevisionId, FfiError>;
pub fn account_search(h: &VaultHandle, query: &str)         -> Result<Vec<AccountSnapshot>, FfiError>;
pub fn account_get(h: &VaultHandle, id: AccountId)          -> Result<AccountSnapshot, FfiError>;
pub fn account_history(h: &VaultHandle, id: AccountId)      -> Result<Vec<RevisionMeta>, FfiError>;

// Session (1.4)
pub fn session_status(h: &VaultHandle)                      -> SessionInfo;
pub fn session_extend(h: &VaultHandle)                      -> Result<SessionInfo, FfiError>;

// TOTP (1.7) — backed by pangolin-totp
pub fn totp_generate(h: &VaultHandle, id: AccountId, at: UnixTimestamp)
                                                            -> Result<TotpCode, FfiError>;

// Password generator (1.8)
pub fn password_generate(policy: PasswordPolicy)            -> SecretPassword;

// KDBX import (1.9) — backed by pangolin-kdbx
pub fn kdbx_import(h: &VaultHandle, path: &str, kdbx_password: &SecretPassword)
                                                            -> Result<KdbxImportReport, FfiError>;

// Encrypted export (1.10)
pub fn vault_export_encrypted(h: &VaultHandle, dest: &str)  -> Result<(), FfiError>;
pub fn vault_export_plaintext(h: &VaultHandle, dest: &str,
                              second_confirmation: PlaintextExportConfirmation)
                                                            -> Result<(), FfiError>;

// Capture authority (1.11)
pub fn capture_authority_register(h: &VaultHandle,
                                  authority: CaptureAuthority,
                                  context: CaptureContext)  -> Result<(), FfiError>;
```

Records / enums that cross the boundary (carry `schema_version: u16` where
they hold user data):

`SecretPassword` (Zeroizing wrapper; never logged), `PresenceProof`,
`SessionInfo`, `VaultHandle` (UniFFI Object), `AccountId`, `AccountDraft`,
`AccountPatch`, `AccountSnapshot`, `RevisionId`, `RevisionMeta`, `TotpCode`,
`UnixTimestamp`, `PasswordPolicy`, `KdbxImportReport`,
`PlaintextExportConfirmation`, `CaptureAuthority`, `CaptureContext`.

`FfiError` mirrors §18.8's taxonomy: variants `Crypto`, `Store`, `Session`,
`Sync`, `Chain`, `Recovery`, `Validation`, `Internal`. **No `Debug` or
`Display` impl ever leaks plaintext** — `FfiError::message()` returns a
UI-safe string per Design Spec §15. `pangolin_core::Error → FfiError` is a
total `From` mapping verified by an exhaustive-match test in
`pangolin-ffi/tests/error_taxonomy.rs`.

Types that **do not cross FFI:** `pangolin_crypto::*` (raw keys, AEAD nonces,
ed25519 secret bytes — all stay internal), the SQLite connection handle,
`pangolin-store` internal `RevisionGraph`, all `pangolin-chain` types
(dormant for MVP-1).

## Failure modes considered

- **UniFFI proc-macro mode pulls `serde` into `pangolin-ffi`'s tree, which
  pulls it transitively into `pangolin-core`.** Mitigation: dep arrow goes
  ffi → core, never the reverse. Verify with `cargo tree -p pangolin-core`.
- **UniFFI's generated extern blocks violate `forbid(unsafe_code)`.**
  Mitigation: `pangolin-ffi` is the only crate without that forbid; it uses
  `deny(unsafe_op_in_unsafe_fn)` instead, and the generated module is
  segregated under `mod scaffolding` with a `#[allow(unsafe_code)]` at the
  module boundary documented in `lib.rs`.
- **Tauri's C-ABI surface drifts from the UniFFI surface.** Mitigation:
  `tests/roundtrip.rs` walks every FFI declaration and asserts the C-ABI
  surface either mirrors it or is explicitly marked "UniFFI-only" with a
  reason.
- **MSRV pin to 1.94 breaks crypto-bearing pinned crates.** Mitigation:
  rerun `cargo deny check` and `cargo audit --ignore <known>` after the
  toolchain pin; if any pinned RustCrypto crate fails to build, hold the
  pin and consult Kelvin.
- **PoC `pangolin-store` already exposes `Vault`/session types that 1.4
  expects to live in `pangolin-core`.** Mitigation: out-of-scope for 1.1
  (Q2 deferred); the FFI surface is wired against re-exports under
  `pangolin_core::*` (which can re-export from `pangolin-store` until 1.4
  relocates them). 1.4's plan absorbs the move.
- **CI Linux umask 0o077 (quirk #1) breaks any FFI-bindings test that
  writes to disk.** Mitigation: bindings emit to `target/ffi-bindings/`
  which inherits cargo's normal mode; reuse the `UmaskGuard::Drop`
  pattern from PoC tests.
- **Forbidden-terms CI step (issue 18.12) flags `revisions` in the FFI
  surface docs.** Mitigation: the allowlist already covers
  `pangolin-chain` internals; extend it to cover `pangolin-ffi` and the
  `docs/architecture/ffi-surface.md` reference.
- **`apps/cli/` move breaks downstream PoC scripts that hard-code
  `tools/pangolin-cli/` paths.** Mitigation: grep the repo for that path
  during the build pass; update `release-windows.ps1`, any `gh` workflow
  references, and DEVLOG entries with the new path.
- **`cargo install --path apps/cli` and `cargo run -p <binary-name>`
  expectations.** Mitigation: keep the binary's `name` field in
  `Cargo.toml` stable (likely `pangolin` — verify with PoC's current
  binary name) so external scripts/users don't break on the directory
  rename.
