## R-e' detail · ARCH-1 host-derives-and-sends pattern

**Problem:** the pangolin-indexer binary's current `main()` generates `[0u8; 32] + OsRng::fill_bytes` as its session AEAD key — Deferral #2. The in-process API path uses `derive_indexer_key(device_key, run_nonce)`. Both paths must converge on `derive_indexer_key`-derived material.

**Two architectures considered:**

- **ARCH-1 (LOCKED): host derives, sends derived key over stdio.** Host already holds `device_key` for chain ops. Host generates a fresh `run_nonce` per binary launch, calls `derive_indexer_key(device_key, run_nonce)`, sends the resulting 32-byte derived key + `run_nonce` to the binary via the stdio handshake. Binary stores key as session-ephemeral in `IndexerSession` state (zeroized on Drop, mirroring 4.3 R-c). Host zeroizes its local copy after handshake completes.
- **ARCH-2 (REJECTED): host sends raw material, binary derives.** Binary would need access to `device_key` (loaded from keystore on binary side). This expands binary's secret-handling surface and forces keystore-access logic into the binary. Rejected.

**Security analysis:** ARCH-1 keeps `device_key` resident ONLY in the host process. Binary process compromise exposes only the ephemeral derived key (limited to one indexer session's worth of temp-DB plaintext recovery). ARCH-2 would expose `device_key` if binary process compromised.

**IPC protocol change:** the binary's stdio handshake currently consumes `(chain_rpc_url, contract_address, ...)`. NEW message added at handshake start: `IndexerHandshake { derived_key: [u8; 32], run_nonce: [u8; 16], ... }`. Encoded with the same framing the binary already uses (length-prefixed CBOR per existing convention). Sent over stdin BEFORE the chain-RPC config. Binary zeroizes the stdin buffer holding the key after deserialization.

**Test additions:**
- `host_derives_and_sends_derived_key_over_stdio` — hermetic end-to-end with mock binary.
- `binary_zeroizes_handshake_buffer_after_consume` — R-e ARCH-1 zeroization: assert the `IndexerHandshake.derived_key` array equals `[0u8; HANDSHAKE_KEY_LEN]` after explicit `zeroize()` (and after `Drop` runs idempotently). This post-Drop field-equality check is the strongest portable Rust check for a `Zeroize`-on-`Drop` stack array — heap-scan / memscan tests are platform-fragile (page reuse, allocator reuse, ASAN/MSAN interactions) and not worth the false-positive surface.
- `binary_refuses_session_start_without_handshake` — typed error.
- `binary_random_key_path_removed` — sweep grep on binary source for `OsRng::fill_bytes` on key material.

## Open questions for Kelvin (RESOLVED — historical context)

All Q's below are resolved in the table above. Preserved for audit trail.

### Q-a · AAD canonical encoding shape

The 4.3 plan-gate suggested binding `(vault_id, page_id, schema_version)`. Three plausible canonical encodings:

**Option A: Fixed-width concatenation.** `vault_id (32 B) ‖ page_id (8 B big-endian u64) ‖ schema_version (2 B big-endian u16)` = 42 bytes. Deterministic; no allocations beyond a stack `[u8; 42]`; no new dep. Pro: simplest; smallest surface; trivially round-trippable for the AAD-byte-pin test. Con: field-order changes are silent (no self-description).

**Option B: CBOR-encoded map.** `{"vault_id": ..., "page_id": ..., "schema_version": ...}`. Self-describing; CBOR already in tree (1.7 TOTP). Con: heavier; allocation per call.

**Option C: serde_json bytes.** Self-describing; already in tree. Con: heaviest; non-canonical.

**Plan-gate recommends Option A** — fixed-width concatenation. The AAD is per-call ephemeral input not on the wire; the byte-pin test locks the layout.

**Kelvin's call.**

### Q-a' · `page_id` source — chicken-and-egg

Page_id must be known at SEAL time (BEFORE the INSERT) so the AAD can be bound. The temp DB uses `rowid INTEGER PRIMARY KEY AUTOINCREMENT` which isn't known until AFTER insert via `last_insert_rowid()`. Three options:

**Option α: Use `rowid`** — requires INSERT-bare-plaintext, then UPDATE-with-wrapped-blobs after reading rowid. Two SQL round-trips per row. Rejected for cost.

**Option β: Use `(account_id, sequence)`** — known at INSERT time. But `account_id` is itself a BLOB that gets wrapped (per Q-b Option A), so it'd need to be UNWRAPPED in the SELECT projection to assemble AAD before decrypt. Workable but couples Q-a' to Q-b.

**Option δ: Session-local `page_seq` AtomicU64 counter** + new UNWRAPPED INTEGER column. The IndexerSession holds an `AtomicU64` that increments on every insert; the column stores the counter value for AAD reconstruction on the SELECT path. Single INSERT pass; clean AAD; new column is additive to the ephemeral temp DB.

**Plan-gate recommends Option δ** — single-pass INSERT; clean AAD; the `page_seq` column is just a session-monotonic counter. Schema change is additive to the ephemeral temp DB only (no .pvf change; no migration story per Q-c Option A).

**Kelvin's call.**

### Q-b · BLOB columns to wrap — exhaustive list?

The `cached_revisions` schema has these BLOB columns: `vault_id` (32), `account_id` (32), `parent_revision` (32), `device_id` (32), `enc_payload` (var), `signer` (20), `block_hash` (32), `tx_hash` (32). Three options:

**Option A: All 8 BLOB columns wrapped.** Maximum defense-in-depth; sweep-grep tests stay mechanically tight.

**Option B: Only the "fat" payload columns** (`enc_payload`, `signer`) + correlatable metadata (`vault_id`, `account_id`, `parent_revision`, `device_id`) — leave the public chain-data BLOBs (`block_hash`, `tx_hash`) unwrapped since they're publicly observable on-chain anyway.

**Option C: Skip `vault_id`** — it's already in the AAD per Q-a, so encrypting it adds redundancy. **Plan-gate rejects** — plaintext `vault_id` column → trivial sweep-test grep failure.

**Plan-gate recommends Option A** — defense-in-depth; the L-cipher-not-wired-into-sql-path raw-disk sweep test stays mechanically tight. Cost is ~40 bytes (24 nonce + 16 tag) framing overhead per column per row — ~320 bytes/row, negligible for a 5-min-idle session.

**Sub-question Q-b':** index on `block_number` — does Option A break it? `block_number` is INTEGER, NOT a BLOB; no index involves a wrapped column. Index survives.

**Kelvin's call.**

### Q-c · Migration of the wire format?

The temp DB is EPHEMERAL — created on session start, destroyed via Drop + zero-fill + unlink. No migration story.

**Option A: No migration; new wire format ships with new code.** Wire-format-additive confirmed: `AeadKey::seal` output is `(ciphertext ‖ tag)` regardless of AAD; AAD is per-call argument mixed into Poly1305 but NOT serialized. On-disk frame stays `nonce(24) ‖ ciphertext ‖ tag(16)`.

**Option B: Bump a `TempDbSchemaVersion` field** in `indexer_meta` for paranoia.

**Plan-gate recommends Option A** — ephemeral DB; no migration; wire format unchanged.

**Kelvin's call.**

### Q-d · Test surface depth

**Option A: Hermetic only** — per-column round-trip; AAD tamper detection; cross-page swap rejection; cross-session rejection; raw-disk-no-plaintext sweep.

**Option B: Hermetic + 1 live `#[ignore]`** — `live_per_column_wrap_against_d017` runs a full indexer session against D-017, dumps the raw temp DB file, asserts no plaintext recoverable from any wrapped column.

**Option C: Hermetic + proptest** over `(page_count, page_sizes, AAD perturbations)`.

**Plan-gate recommends Option B** — mirrors §5.x posture; live test catches "wrapping works hermetically but a real chain payload contains a byte pattern that survives" regressions hermetic synthetic plaintexts miss.

**Kelvin's call.**

### Q-e · Defer Deferral #2 (binary uses fresh random key) to its own cycle?

The `pangolin-indexer` binary generates a fresh `[0u8; 32] + fill_random` key on startup instead of receiving `derive_indexer_key`-derived material via stdio handshake.

**Option A: Defer.** This cycle only addresses Deferrals #1 + #3 (per-column wrapping + AAD). Deferral #2 ships in a separate "indexer-derived-key-via-IPC" cycle which changes the binary's IPC protocol (non-trivial: needs `device_key` material + `run_nonce` via stdio handshake; host-authenticated-to-binary question).

**Option B: Include.** Wire device_key + run_nonce through stdio in this cycle; tighter security; bigger blast radius for review.

**Plan-gate recommends Option A** — keep this cycle focused. ~3-5h already for per-column wrapping + AAD; binary key-derivation is its own architecture question.

**Kelvin's call.**

### Q-f · Plaintext fall-back for backwards compat?

**Option A: No plaintext fall-back.** Encrypted-only on the SQL path. `NoOpCipher` stays `#[cfg(test)]` for tests; production always uses `AeadCipher`. A bare-plaintext BLOB read on the SELECT path fails `decrypt_page` with `CipherError::FramingTooShort` or `CipherError::TagMismatch`.

**Option B: Magic-byte sniffer** that distinguishes wrapped from bare BLOBs.

**Plan-gate recommends Option A** — ephemeral DB; no backwards-compat story; trait + cipher signal cleanly with typed `CipherError`. Option B introduces parser ambiguity for zero benefit.

**Kelvin's call.**

## Adversarial framing — L-section (closes a known gap)

### L-cross-page-cut-and-paste

**What goes wrong:** attacker with mid-run disk access swaps two pages' ciphertext blocks between rows. Without AAD binding, `decrypt_page` succeeds + returns wrong plaintext for swapped page; `handle_pull` surfaces foreign-row data.

**Defense:** AAD binding (`page_id` in AAD per Q-a/Q-a') — decrypt on swapped page fails with `CipherError::TagMismatch` because the AAD attached at SEAL time differs from the AAD computed at OPEN time. `IndexerError::TempDbIo` surfaces typed error.

**Test:** `cross_page_swap_within_session_detected`.

### L-cross-session-replay

**What goes wrong:** attacker captures a page from session A's temp DB before unlink; injects into session B's running temp DB.

**Defense:** AAD binding (`vault_id` in AAD) — different vault_id → different AAD → TagMismatch. Plus ephemeral key is per-session per 4.3 L3, so key mismatch already defeats this — AAD binding adds belt-and-suspenders at the column boundary.

**Test:** `cross_session_replay_detected`.

### L-future-schema-version-poison

**What goes wrong:** future schema version's pages presented to older code that doesn't know how to interpret them.

**Defense:** AAD binding (`schema_version` in AAD) — mixing versions surfaces TagMismatch. Defense against accidental version-confusion bugs (partial-upgrade builder regressions).

**Test:** `decrypt_with_mismatched_schema_version_aad_fails_with_tag_mismatch`.

### L-key-derivation-not-pinned-in-binary (Deferral #2; OUT OF SCOPE per Q-e)

**What goes wrong:** binary uses fresh random per-startup key; in-process API uses derived key.

**Defense in this cycle:** none. Per Q-e Option A this is deferred to a dedicated cycle that changes the binary's IPC protocol.

**Caveat documented:** 4.3 audit explicitly accepted this gap (OPTION-AUDIT-A) — binary's fresh-random-key path is cryptographically equivalent for ephemeral runs.

### L-runtime-memory-dump

**What goes wrong:** attacker with mid-run memory access reads AEAD key from process memory.

**Defense:** NOT closed by this cycle. Memory protection is host-level (mobile platform sandboxing; Tauri runtime isolation) not pangolin-indexer's defense. Runtime-memory-access caveat documented in 4.3 L-temp-file-leak STAYS. This cycle closes only the on-disk-mid-run case.
