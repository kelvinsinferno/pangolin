## Decisions locked (post plan-gate)

| Q | Decision | Rationale |
|---|---|---|
| Q1 — Deterministic canonical-head tiebreak rule | **Largest `revision_id` only — clock-free.** When an account's revision graph has multiple leaves, the canonical head is the leaf with the lexicographically-largest `revision_id` (byte-order). NO `created_at` component in the head election. Document `revision_id` byte-order as the device-independent total order. *(Diverges from the plan-gate's in-table recommendation — Kelvin chose the clock-free variant.)* | Zero clock dependency — `created_at` is device-stamped and not trustworthy across devices; a buggy/malicious device could stamp a future timestamp to hijack the head election. `revision_id` byte-order is not usefully attacker-steerable. Standard CRDT "highest hash wins"; cleanest to prove all devices / chain-replicas agree (MVP-2 sync soundness inherits this rule). The "winning" branch isn't necessarily the most-recent edit — but that doesn't matter: a forked account is read-only-at-canonical-head pending explicit user resolution, and the resolution revision (newest, so highest `revision_id` by construction) becomes the unambiguous sole leaf afterward. Where the rule actually fires (a fork), determinism >> recency. |
| Q2 — Forked-account behaviour in MVP-1 | **Readable, non-blocking.** A forked account remains readable at its deterministic canonical head; reveals/edits are NOT blocked by fork-state alone. `resolve_fork` needs only an active (unlocked, non-expired) session — it's not §5.4 reveal-class (it reparents the graph, doesn't expose secrets). The existing P10 `frozen_pending_resolve` flag (set only by the dormant `ingest_chain_revision` path) is the *separate*, stricter freeze. Resolution is always explicit user choice — no auto-resolve, ever. | Matches §G3 ("always explicit user choice, never silent merge") + the existing PoC semantics; minimises behavioural churn; the canonical head gives a coherent read-only view while the user decides which branch to keep. |
| Q3 — Future-schema-version granularity (`payload_version` > MAX_KNOWN, or revision-row `schema_version` > MAX_KNOWN) | **That account requires upgrade; rest of vault fine.** The file-level `format_version` continues to gate whole-vault openability (P2 `UnsupportedFormatVersion`). An individual revision with a future schema version makes *that account* surface a "requires-newer-Pangolin" status — metadata-only reads where possible, but reveals/edits/head-decryption on that account blocked with a typed error — without bricking the rest of the vault. **Never silently skip the future-versioned revision.** Master plan §18.7's "skip applying that revision until updated" wording is the MVP-2 chain-replay framing and is unsound for the local store (a head with a future version IS the account's current state; silently skipping shows stale data with no signal); §18.7 should be amended/clarified to mark that wording chain-context-only — the 1.6 docs note this. | A future-versioned head is real state — surfacing "needs upgrade" is the only honest behaviour; whole-vault-refusal is heavy-handed for a per-account problem; silent-skip is a correctness bug. |
| Q4 — §18.7 implementation scope | **Add the revision-schema-version reject check + write `docs/architecture/schema-versioning.md`.** Do NOT refactor the existing per-surface checks (file `format_version`, `fts_schema_version`, `devices.schema_version`, `device_key.schema_version`, the `session_idle_secs` validator) into one module — they work; a refactor is churn for no functional gain. The new doc retroactively catalogues every versioned surface + its MAX_KNOWN value + the policy each follows going forward (read-old / reject-unknown-future / the granularity per surface). | Smallest change that locks the policy; the doc is the durable artifact 1.7/1.9/etc. follow. |
| Q5 — Losing branch on resolve | **Keep.** The losing branch's revision rows stay in the `revisions` table (audit/recovery; they're just off the head chain). Only the `pending_merges` stash row for the resolved fork is cleaned up — exactly as P9's `prune_orphan_pending_merges` already does. | Append-only is a cardinal principle; pruning revision history would violate it. |
| Q6 — `pangolin-cli resolve` subcommand in 1.6? | **No new CLI subcommand.** Consistent with the standing CLI-V1-wiring deferral that 1.2/1.3/1.4/1.5 followed. No `pangolin-cli resolve` exists today (P9 shipped the store-layer resolve API + `list_conflicts`, not a CLI subcommand). 1.6 ships the production lineage + the §18.7 policy + the FFI `fork_*`/`resolve_*` entries; the CLI `resolve` subcommand rides the CLI-V1 issue alongside `account`/`reveal`/`device`. | Keeps 1.6 tight; one cohesive CLI-V1 issue eventually. |

## Goal

After this issue closes: the `pangolin-store` revision-lineage layer is
production-grade. Every account has a **deterministic canonical head** —
given any account's revision graph, every device (now, and MVP-2's chain
replicas later) computes the same canonical head by a documented total
order that has no clock-iteration-order dependency on the load-bearing
component. Fork detection is unchanged in semantics (an account is forked
iff its graph has >1 leaf) and stays cheap (the existing SQL `NOT EXISTS`
multi-head detector). Conflict resolution is the existing P9 store API,
promoted: `Vault::resolve_fork(account_id, keep_revision_id)` validates
that `keep_revision_id` is a current head of the (forked) graph, writes a
new revision parented at that leaf — un-forking the graph — and clears
any `frozen_pending_resolve` flag; it never auto-resolves and never
prunes the losing branch. The §18.7 schema-versioning policy is **locked
and documented** (`docs/architecture/schema-versioning.md`): every
persisted record carries a schema-version field; on read, a version ≤
this build's MAX_KNOWN parses (migrating old→current as 1.2 already does
for V0→V1); a version > MAX_KNOWN is **rejected with a clean typed
error**, and the *granularity* is well-defined (file-level → whole vault
unopenable; revision-level → that account "requires upgrade", rest of the
vault fine). The FFI gains additive `account_is_forked` /
`account_fork_branches` / `account_resolve_fork` / `account_status`
entries (the last surfacing "requires upgrade"). All existing PoC `.pvf`
vaults still open, unlock, `account_history`, and resolve. The 1.3
`:memory:` FTS5 index, the 1.4 session machine, and the 1.5
device-registration path are untouched.

## Approach

> **⚠ Read the "Decisions locked" table at the top of this file first — it
> supersedes the §Approach and §"Open questions for Kelvin" prose below
> where they differ.** Two points were written by the plan-gate before
> Kelvin's answers / before the as-built work and are now superseded:
> **Q1 locked the clock-free rule** — the canonical head is the head with
> the **largest `revision_id`** (a 256-bit CSPRNG value; ties are
> cryptographically impossible), with **no `created_at` / timestamp
> tiebreak**; the Q1-body text that floated a "newest-timestamp-primary"
> rule does not reflect what shipped. And **`resolve_fork` un-forks the
> account via the `revisions.superseded_by` metadata column**, not by the
> merge revision "becoming the sole leaf" on its own: a single-parent
> merge revision parented at the chosen leaf does **not** orphan the
> unchosen leaves; `resolve_fork_commit` writes `superseded_by = <merge
> id>` onto every losing-branch head so the `NOT EXISTS` head detector
> stops counting them (append-only is preserved — the losing rows are
> kept, this is a pointer column like the chain-anchor columns). See
> `docs/architecture/revision-lineage.md` for the as-built description.
>
> **Audit L1 (fix-pass 2):** the §18.7 "reject a future
> `revisions.schema_version`" check (`row_schema_version >
> REVISION_SCHEMA_VERSION_MAX`) runs **after** the AEAD open, not before
> — that byte is bound into the AAD, so a bare on-disk byte-flip of it
> surfaces `AuthenticationFailed` (tampering); a *legitimately*
> future-versioned revision (sealed by a future build with that byte in
> its AAD) still surfaces `UnsupportedRevisionSchemaVersion` because that
> build's open succeeds and the post-open check fires.

Numbered steps. Crates: `pangolin-store` (the bulk), `pangolin-ffi`
(additive entries), `pangolin-core` (re-exports only — no relocation,
per the standing Q1-of-1.4 doctrine), `docs/`.

### 1. Deterministic canonical-head computation (`pangolin-store`)

1.1. Add `RevisionGraph::canonical_head(&self) -> Option<&RevisionId>`
to `crates/pangolin-store/src/revision.rs`. Definition: of the `heads`
set (already computed at `build` time), the canonical head is the
**maximum under the documented total order** (see Q1 for the exact
order). `None` only when the graph is empty. For a linear chain this
is exactly the single head (no change to existing behaviour). For a
forked graph it is the rule-winner.

1.2. The total order (recommended; Q1 finalises): compare two head
revisions by `(created_at, revision_id_bytes)` and take the **maximum**
— i.e. "newest `created_at`; ties broken by lexicographically-largest
`revision_id` byte string". `revision_id` is 32 bytes of CSPRNG output
(P2 scope; the MVP-2-deferred content-hash switch only *strengthens*
this), so the byte-order tiebreak is a device-independent total order on
any pair of heads. Property: (i) deterministic — `revision_id` is the
load-bearing component and is fixed per revision; the `created_at`
component is best-effort and only orders heads that *don't* tie on it,
which in the genuine concurrent-edit case they generally won't, but the
fallback is always available; (ii) total — every pair of distinct heads
has distinct `revision_id` bytes; (iii) stable — adding an unrelated
revision to a different sub-tree doesn't change which head wins.
*Alternative if Kelvin wants zero clock dependency at all:* drop the
`created_at` component entirely — canonical head = lexicographically-max
`revision_id`. This is the standard CRDT "highest hash wins" tiebreak;
it's even simpler and provably clock-independent. (The cost: the chosen
head isn't necessarily the most-recently-authored one — but with random
revision-ids "most recent" is already not meaningful across devices.)

1.3. Wire `canonical_head` into the consumers:
- `Vault` adds `fn canonical_head(&self, id: AccountId) -> Result<RevisionId>`
  that builds the graph and returns `canonical_head` (errors
  `AccountNotFound` / `Corrupted` like `account_heads`).
- The `account_identities.head_revision_id` column stays "the most
  recently chosen canonical head" — but 1.6 makes the read paths that
  *currently trust the column* fall back to / cross-check against
  `canonical_head` when the account is forked. Concretely: the cache
  hydration in `unlock` (`build_decrypted_cache` over "every live head")
  and the FTS5 rebuild (1.3, indexes each account's head revision) and
  `account_search`'s per-hit head re-read should index/cache the
  *canonical* head of a forked account, not an arbitrary one. **Lowest-
  churn implementation:** in the unlock path, for each account, if
  `account_heads(id).len() > 1` use `canonical_head(id)`; else use
  `head_revision_id` as today. This keeps the linear fast path
  untouched.
- `account_search` already re-reads `head_revision_id` from SQL per hit
  (line ~3330 in `vault.rs`); change that to "if forked, canonical_head;
  else head_revision_id". The 1.3 search lifecycle is otherwise
  preserved.

1.4. Document at the type level on `RevisionGraph` that `canonical_head`
is the production head-selection rule and the per-account
`head_revision_id` column is a *cache* of it that the resolve flow
advances; the SQL `NOT EXISTS` query is the authoritative head-*set*
detector.

### 2. Fork detection (`pangolin-store`) — promote, don't rewrite

2.1. Fork detection is already production-shaped: `account_heads`'s SQL
`NOT EXISTS` subquery (scoped by `account_id`, per P3 audit M-1) is the
cheap detector; `is_forked` is `account_heads(id).len() > 1`;
`all_forked_accounts` is the "needs attention" set. 1.6 keeps all of
this. No `forked` flag column is added — the SQL query is already O(rows
for the account) with the `idx_revisions_parent` index, and adding a
maintained flag means another invariant to keep correct on every
insert/ingest/resolve for no measured benefit. (If a future profiling
pass shows this is hot, a flag is a clean additive migration — note it
as a deferred option, not 1.6 work.)

2.2. `frozen_pending_resolve` (P8) is **distinct from** fork-state and
stays distinct: it's a stricter "this account was mutated under your
handle's nose by the dormant `ingest_chain_revision` path; refuse reads
and edits until resolve runs" flag. A forked-but-not-frozen account is
readable at its canonical head (recommended — see Q2). 1.6 documents the
two states' relationship in `docs/architecture/revision-lineage.md`.

### 3. Conflict resolution → canonical head (`pangolin-store`)

3.1. Add `Vault::resolve_fork(&mut self, account_id: AccountId,
keep_revision_id: RevisionId) -> Result<RevisionId>`. Semantics:
- Validate the account exists; validate `keep_revision_id` is a row of
  *this* account (cross-check `account_id AND revision_id`, same
  defense-in-depth as the P9 `clear_frozen` / `read_payload_plaintext_for_resolve`
  paths — never let a cross-account `revision_id` become an oracle).
- Validate `keep_revision_id` is a **current head** of the account's
  revision graph (the `NOT EXISTS` predicate). If not → typed error
  (`StoreError::NotAHead { account_id, chosen, current_heads }` — the
  existing variant fits exactly).
- Validate the account is actually forked (`account_heads(id).len() > 1`).
  If it's linear → the recommended behaviour is a typed error
  (`StoreError::Validation { kind: "not-forked", message: ... }`) rather
  than a silent no-op, so the FFI/CLI surfaces "nothing to resolve"
  loudly. (Open to Kelvin in Q-flavour but low-stakes; flagging.)
- Read the chosen leaf's plaintext via the existing
  `read_payload_plaintext_for_resolve` (the "user named the specific
  revision; trust the read for that one revision for the duration of one
  in-memory operation" argument from P9 carries over verbatim).
- Build the merge revision's ciphertext via `merge_payload_for_resolve`
  (re-seal under a fresh nonce + the merge revision's own AAD —
  `parent_revision_id = keep_revision_id`, `schema_version` inherited
  from the chosen leaf). Tombstone leaves re-seal via `seal_tombstone`,
  as P9 already handles.
- INSERT the new revision row (parent = `keep_revision_id`, device_id =
  the local registered device per 1.5), advance
  `account_identities.head_revision_id` to the new id, clear
  `frozen_pending_resolve` to 0, and write the `dirty_accounts` marker
  (the merge revision is unpublished — MVP-2's publish path consumes
  it). All in one SQLite transaction (same posture as `update_account`).
- Prune the `pending_merges` stash row for `(account_id, *)` — actually,
  call the existing `prune_orphan_pending_merges(account_id)` *after*
  the head advance so any stash row whose `target_head_id` is no longer
  a head (which now includes the just-resolved one) is cleaned up.
- The **losing branch's revision rows are NOT deleted** (Q5 rec).
- Return the new revision id.

3.2. Relationship to P9's `clear_frozen`: `clear_frozen` is the *lower-
level* "the merge revision already exists (it was published & re-ingested
in the MVP-2 flow); just clear the flag + advance the head" primitive.
`resolve_fork` is the *MVP-1, no-chain* primitive that *creates* the
merge revision locally and then does the same flag-clear + head-advance.
Both stay; `resolve_fork` is what the FFI/CLI calls in MVP-1. Document
the split.

3.3. `__test_synthesize_sibling_revision` (the only way to *create* a
real fork in MVP-1, since there's no chain sync and the chain-ingest path
is dormant) stays as a `cfg(any(test, feature = "test-utilities"))`
helper. It's the test instrument for every fork/resolve test. **Surface
in the plan + DEVLOG:** in MVP-1 a fork can only arise from (a) the test
helper, (b) importing an already-forked `.pvf` (no import path exists in
MVP-1 either), or (c) the dormant `ingest_chain_revision` path
(MVP-2-only). So the *machinery* is fully production-grade and fully
tested via the helper + the resolve API is real and FFI-exposed, but **no
real multi-device fork can occur until MVP-2's chain sync lands** —
that's the honest scope, and it matches how 1.5 scoped the dormant
`last_sync_at` / unsigned `DeviceKey`.

### 4. §18.7 versioned-schema policy + forward-compat parsing

4.1. **The revision-schema-version reject.** Two version fields gate a
revision blob: the `revisions.schema_version` row column (a `u8` on disk,
also a byte in the AAD) and the `payload_version` in the V1 CBOR body
(`u8`; V0=arity-6-no-field, V1=arity-8-with-`payload_version`=1). Add a
crate constant `pub const REVISION_SCHEMA_VERSION_MAX: u8 = 1;` (the
current max — V1 / `payload_version` 1) in `pangolin-store/src/revision.rs`
or `blob.rs`. On read of a revision (`read_identity_at`, the unlock cache
hydration, `account_search`'s head read, the resolve-flow reads):
- If `revisions.schema_version` (the row column) `> REVISION_SCHEMA_VERSION_MAX`
  → return a new typed error
  `StoreError::UnsupportedRevisionSchemaVersion { account_id, revision_id, found: u32, supported: u32 }`
  (parallel to the existing `UnsupportedFormatVersion(u32,u32)` for the
  file level). Recommend a *new* variant rather than reusing
  `UnsupportedFormatVersion` so callers can distinguish "the whole vault
  is from the future" from "one revision/account is".
- In `blob.rs::decode_v1_live_inline`, replace the "Q4 accept-and-record
  … 1.6 owns the reject policy" stanza: if the parsed `payload_version
  > REVISION_SCHEMA_VERSION_MAX` (i.e. > 1), return the same
  `UnsupportedRevisionSchemaVersion` (the blob layer doesn't know the
  `account_id` / `revision_id`; it returns a `Cbor`-adjacent typed
  error and the `vault.rs` caller decorates it with the ids — or
  `blob.rs` returns a lighter `StoreError` and the caller maps it).
  Practical: have `decode_v1_live_inline` keep returning a `StoreError`,
  add a `StoreError::UnsupportedRevisionSchemaVersion` with the ids
  defaulted/zero at the blob layer and let `read_identity_at` re-wrap
  with the real ids — or simpler, have the blob layer return a small
  marker and `vault.rs` constructs the full error. Builder picks the
  cleaner of the two; either way the *behaviour* is "clean typed reject,
  no crash, no silent skip".
- A future map-arity (9+) currently hits `StoreError::Cbor("unexpected
  map arity for identity payload: N")` — that's *almost* fine
  (it's a clean typed error, not a crash) but it's mis-categorised. 1.6
  maps arity > 8 to `UnsupportedRevisionSchemaVersion` too (a higher
  arity is by construction a future payload shape). Arity ∈ {1,3} (V0
  tombstone shapes) and {6} (V0 live) and {8} (V1 live) keep their
  current handling.

4.2. **Granularity (Q3 rec — that-account-requires-upgrade).** When
`read_identity_at` (or the unlock cache hydration) hits
`UnsupportedRevisionSchemaVersion` for a revision that *would be* an
account's canonical head: the rest of the vault must still open and
function. Implementation: `unlock`'s cache build catches this per-account
and marks the account as "requires upgrade" (a transient in-RAM set on
the `Active` state, NOT a new persisted column — the on-disk truth is
"there's a revision with version > our max"; we don't need to write
anything). Reads of *metadata* for that account (`account_history`,
`list_accounts`, the FTS5 index entry from the *last parseable* head if
one exists, `is_forked`) keep working; `get_account` / `reveal_*` /
`update_account` on that account return `UnsupportedRevisionSchemaVersion`
/ a "requires upgrade" status. The file-level `format_version` check
(P2, `Vault::open` rejects `> MAX`) is unchanged — that's still the
whole-vault gate.

4.3. **No refactor of the existing hooks (Q4 rec).** Leave the file
`format_version` check, the `fts_schema_version` meta value (1.3,
currently always 1), the `devices.schema_version` (1.5, default 1), the
`device_key.schema_version` (1.5), the `pending_merges.schema_version`,
and the 1.4 `SessionDuration::from_meta_secs` / `try_from_meta_secs`
validator (rejects unknown `session_idle_secs` values) exactly as they
are. 1.6 only *adds* the revision-schema-version reject and *documents*
the whole family.

4.4. **`schema_version` stays `u8` on disk (recommended).** It's a byte
in the AEAD AAD — widening it to `u16` is a format change (the AAD shape
must not change). `u8` gives 255 schema versions; the FFI continues to
widen at the wire boundary (`RevisionMeta.schema_version: u16` in
`pangolin-ffi/src/revision.rs` — `From<u8>` is lossless). Document the
convention in the policy doc.

## Success criteria

1. `RevisionGraph::canonical_head` exists; for a linear chain it equals
   the single head; for an N-way fork it equals the documented-rule
   winner; for an empty graph it's `None`.
2. `Vault::canonical_head(account)` returns that id; the same id on a
   reopened vault (the rule has no run-to-run dependency).
3. The rule is clock-independent on its load-bearing component: building
   the graph from rows whose `created_at` are all-equal still picks a
   deterministic head (the `revision_id` byte-order tiebreak).
4. Adding an unrelated revision (different sub-tree, or a new linear
   child of a *non-canonical* head) does not change `canonical_head`.
   *(With the recommended "newest `created_at`" rule, extending the
   *canonical* head's chain naturally moves the head forward — that's
   correct, not a violation. The criterion is about *unrelated*
   additions.)*
5. After `unlock`, a forked account's in-RAM cache entry and its FTS5
   index entry reflect the *canonical* head (search for the canonical
   head's display name finds it).
6. `resolve_fork(account, keep_leaf)` on a forked account: writes a new
   revision parented at `keep_leaf`; `canonical_head` is now that new
   revision; `is_forked` is now false; `frozen_pending_resolve` (if it
   was set) is cleared; the `dirty_accounts` marker is written; the
   losing branch's revision rows still exist in `revisions`; the
   `pending_merges` stash row for the resolved target (if any) is gone.
7. `resolve_fork` with a `keep_revision_id` that exists for the account
   but isn't a current head → `NotAHead`; with a `revision_id` from a
   *different* account → collapsed into the same `NotAHead` / not-found
   error (no oracle); on a non-forked account → `Validation { kind:
   "not-forked", .. }` (typed, not a panic, not a silent no-op).
8. A revision row with `schema_version` > `REVISION_SCHEMA_VERSION_MAX`
   (synthesised in a test): reading that revision →
   `UnsupportedRevisionSchemaVersion`; the rest of the vault still opens
   and other accounts work; the affected account's `account_status`
   reports `requires_upgrade = true` and `get_account` / `reveal_*` on
   it return `UnsupportedRevisionSchemaVersion`; `account_history` /
   `is_forked` on it still work (metadata-only).
9. A V1 CBOR body with `payload_version` = 2 (synthesised) → same
   `UnsupportedRevisionSchemaVersion` on read; a map-arity-9 body → same.
   A V0 (arity-6) body and a V1 (arity-8, `payload_version` 1) body
   parse exactly as today (no regression to the 1.2 migration).
10. The file-level `format_version` check is unchanged
    (`UnsupportedFormatVersion` still gates whole-vault open).
11. Every existing PoC `.pvf` test fixture (and the e2e migration
    fixtures from 1.2/1.5) still opens, unlocks, `account_history`s, and
    — for any with a P9-shaped pending merge — `resolve_fork`s. All

## Failure modes considered

- **The head rule isn't actually total** (two heads with identical
  `revision_id` bytes) → impossible: `revision_id` is the `revisions`
  table PRIMARY KEY; duplicates surface as `Corrupted` at
  `RevisionGraph::build`. Mitigation: documented + the dup check stays.
- **A forked account's cache/index points at a non-canonical head**
  (the bug 1.6 fixes) → mitigation: unlock + `account_search` use
  `canonical_head` when `account_heads().len() > 1`; tests 5.
- **`resolve_fork` advances the head but a concurrent `ingest_chain_revision`
  changes the head set in between** → mitigation: the head-membership
  check + the head advance happen inside one SQLite transaction (same
  pattern P9's `clear_frozen` already uses); the `NotAHead` variant
  fires if the chosen leaf was demoted.
- **`resolve_fork` re-seals the chosen leaf's plaintext in memory** —
  bounded exposure (the user explicitly named that revision; in-memory
  for the duration of one op; the P9 argument). Mitigation: re-use the
  exact P9 `read_payload_plaintext_for_resolve` / `merge_payload_for_resolve`
  path; `Zeroizing` plaintext as today.
- **A future-version revision blob is attacker-supplied** (someone edits
  the `.pvf` to set `schema_version=255` on a head row to brick an
  account or trigger a panic) → mitigation: the reject is a *typed
  error*, not a panic; the account surfaces "requires upgrade"; the rest
  of the vault is untouched; the AEAD AAD includes `schema_version` so a
  bare byte-flip without re-sealing fails authentication first
  (collapses to `AuthenticationFailed`) — only a *legitimately re-sealed*
  future blob reaches the version check, which is exactly the "newer
  Pangolin wrote this" case.
- **The `payload_version` reject in `blob.rs` doesn't know the ids** →
  mitigation: blob layer returns a marker `StoreError`; `read_identity_at`
  re-wraps with `account_id` / `revision_id` (builder's choice of the
  two clean wirings in step 4.1).
- **`account_history` building the graph per call is O(rows)** — same
  cost as `revision_graph`; acceptable (history is a UI-triggered op).
  Mitigation: none needed; note as a deferred optimisation if profiling
  ever flags it.
- **§18.7 "skip applying that revision" vs MVP-1 "freeze the account"**
  — the master plan's wording is the MVP-2 chain-replay framing; for the
  local store there's no "the revision is optional, skip it" — a head
  with a future version *is* the account's state. Mitigation:
  documented explicitly in the policy doc; surfaced as Q3 for Kelvin.
- **Existing PoC `.pvf` with a P9-shaped pending merge** — `resolve_fork`
  must work on it. Mitigation: e2e fixture test; the `pending_merges`
  table migration (P9) already runs on legacy open.
- **clippy `too_many_lines` on `resolve_fork` / the enriched
  `account_history`** — mitigation: extract helpers (the codebase
  already does this — `compute_genesis_extra`, the P3 audit
  extractions); keep each fn under the floor.

## Open questions for Kelvin

**Q1 — The deterministic canonical-head tiebreak rule.** P3 never picked
a single canonical head from a forked set; it returns the *head set*
ordered by `(created_at ASC, revision_id byte-order ASC)`. 1.6 must pick
*one*. Options:
- **(a) Newest `created_at`, then lexicographically-largest `revision_id`**
  (recommended default). Total (revision-ids are the PK). Stable.
  Mostly clock-independent — `created_at` only orders heads that don't
  tie on it, and the `revision_id` tiebreak is always the
  device-independent fallback. Matches the intuition "the latest edit
  wins" when clocks are roughly sane. **Risk:** if a device has a badly
  skewed clock, its head could win/lose "unfairly" — but since the user
  *explicitly* resolves forks, an unfair auto-pick is only the
  pre-resolution display state, not a committed choice.
- **(b) Lexicographically-largest `revision_id` only** (zero clock
  dependency). The standard CRDT "highest hash wins". Simpler;
  provably deterministic. **Cost:** the displayed head of an unresolved
  fork is arbitrary-but-stable rather than "most recent".
- **(c) Longest chain, then (a) or (b) as tiebreak.** More "intuitive"
  in some races but adds a depth computation and isn't obviously better
  — a long chain of trivial edits shouldn't beat a short chain with the
  edit you want. Not recommended.

My recommendation: **(a)**, with the doc explicitly noting `revision_id`
byte-order is the load-bearing total order and `created_at` is
best-effort. If Kelvin wants the cleanest provable story for MVP-2's
chain-replica-agreement, **(b)** is the safer pick. **This is the call
that needs to be right** because MVP-2's chain sync inherits it.

**Q2 — Forked-account behaviour in MVP-1.** Today: a forked-but-not-frozen
account is *readable* (the cache uses *some* head — 1.6 makes it the
canonical one); only `frozen_pending_resolve` (set by the dormant
chain-ingest path) blocks reads/edits. Options: **(a) keep that** —
forked accounts readable at canonical head, reveals/edits allowed,
resolution is purely the user's prerogative (recommended; matches
Whitepaper §7 "conflicts are surfaced, not blocking"; least churn);
**(b)** block edits (but not reads) on a forked account until resolved
("you can look but a new edit would deepen the fork"); **(c)** block both
(treat fork like the `frozen` state). Recommendation: **(a)**. Flagging
because §16.3 wants the access-control posture explicit. *(Sub-question:
should `account_resolve_fork` require a fresh presence proof? Rec: no —
it ratifies a branch the user authored; reveals nothing. Flag.)*

**Q3 — Future-schema-version granularity.** When a revision blob has
`payload_version` / `schema_version` > our MAX, what's "frozen"? Options:
**(a) the whole vault** (refuse to open — heavy-handed; conflates with
the file-`format_version` gate that already does this for *file*-level
changes); **(b) just that account** — "requires upgrade", metadata-only
reads still work, reveals/edits blocked, rest of the vault fine
(recommended); **(c) just that revision** — skip it, treat the
last-parseable head as the account state (this is the master plan's
§18.7 "skip the revision" wording — **but it's the MVP-2 chain-replay
framing and is unsound locally**: a head with a future version *is* the
account's current state; "skipping" it shows stale data with no signal).
Recommendation: **(b)**. The §18.7 wording vs (b) is the one place the
master plan and the recommended implementation diverge — calling it out
explicitly: the plan's "skip" clause should be read as "skip *for the
purpose of chain replay ordering*", not "silently show stale local
state".

**Q4 — Refactor the version hooks into one `schema_versioning` module,
or add + document?** Options: **(a)** add only the revision-schema-version
reject + write the policy doc cataloguing all existing hooks (recommended
— the existing per-surface checks work; refactoring is churn); **(b)**
build a `schema_versioning` module that all surfaces route through (more
"cohesive" but touches the file-format, FTS, devices, device-key,
session-config code for no functional gain — and re-touching the AEAD-AAD-
adjacent file-format check carries audit risk). Recommendation: **(a)**.

**Q5 — Losing branch on resolve: keep or prune?** Options: **(a) keep**
the losing branch's revision rows in `revisions` (off the head chain,
for audit/recovery) — recommended; append-only is the cardinal
principle; only the `pending_merges` stash row gets cleaned up (P9's
`prune_orphan_pending_merges` already does exactly this); **(b) prune**
the losing branch on resolve (smaller `.pvf`, but violates append-only
and forecloses "wait, I picked wrong"). Recommendation: **(a)**.

  graph 1.6 builds is *what* gets anchored, but 1.6 wires zero chain code
  (`ingest_chain_revision` stays dormant; it's the MVP-2 ingest entry).
- **Real multi-device fork creation via sync** — in MVP-1 a fork can only
  arise via the `__test_synthesize_sibling_revision` test helper or the
  dormant ingest path. Real concurrent multi-device edits → forks land
  with MVP-2's chain sync.
- **The on-chain Revision Log v1's signature verification** — `2.1` /
  `3.1` (verify a signed revision from a registered device). 1.6's
  revisions are still unsigned locally (1.5's `DeviceKey` exists but
  signs nothing yet).
- **Content-deterministic `revision_id`** (keccak256 of the canonical
  revision body, so racing devices produce the same id) — noted in
  `revision.rs` as a future switch; not 1.6. (The recommended head rule
  works the same with random ids; the content-hash switch only
  *strengthens* the byte-order tiebreak's "highest hash wins" story.)
- **A maintained `forked` flag column on `account_identities`** — a clean
  additive optimisation if profiling ever shows the `NOT EXISTS` query is
  hot; not 1.6.
- **The `pangolin-cli resolve` subcommand** — rides the CLI-V1 issue
  alongside `account` / `reveal` / `device`.
- **Authority rotation / device revocation interplay with the lineage**
  — MVP-3 (`6.5`).
