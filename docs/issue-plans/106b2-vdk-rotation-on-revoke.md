# Issue #106b-2 — VDK-rotation-on-revoke FLOW (the cryptographic device-kill) — plan-gate LOCKED

**Status: LOCKED — Kelvin sign-off 2026-05-21 (see §0a). Q-a..Q-h resolved; ONE stage.** The second slice of #106b. Composes the MERGED #106b-1 pairing/per-device-wrap primitive + the MERGED #104a/#104b guardian escrow into the **forward-secrecy device-kill flow**: when a device is removed (#106a `removeDevice`), the surviving devices mint a fresh VDK epoch, re-wrap it to themselves (and the password anchor), re-point the guardian recovery escrow at the new VDK, and advance the shared per-vault epoch — so the removed device is permanently locked out of all POST-revoke data. This issue introduces **no new crypto primitive** — every byte is composition over already-audited surfaces — but the *composition* (re-creating the VDK + re-pointing the escrow) is net-new and catastrophic-if-wrong, so it gets #104a-grade in-house adversarial-audit rigor. Mirrors the §16 plan-gate format of `104b-recovery-orchestration.md` / `105-recovery-core-hardening.md`.

## 0. One-paragraph summary

#106b-1 built the catastrophic-if-wrong *primitive* (seal the VDK to a new device's X25519 pairing key; per-device-wrap the VDK at rest). #106b-2 is the **flow** that composes it into a *cryptographic device-kill*: on a `removeDevice` (#106a), mint a fresh VDK for a new **VDK epoch**, re-wrap that new VDK to every SURVIVING device (via #106b-1 `wrap_vdk_for_device` + the pairing seal) and under the password anchor — never to the removed device — re-point the guardian recovery escrow at the new VDK (Q-d, THE crux), advance the ONE shared per-vault epoch, persist atomically (the #105a single-transaction lesson), and optionally re-encrypt the LOCAL vault store to the new VDK. Because on-chain `encPayload` history is IMMUTABLE, "rotate the VDK" can NOT mean re-encrypting on-chain history; instead it mints a NEW VDK for new entries while OLD on-chain entries stay under the OLD VDK — the client therefore keeps a **chain of epoch-keyed VDKs** and decrypts each entry under its epoch's VDK. The removed device keeps its synced PRE-rotation data forever (immutable + old-VDK; unretractable), and is permanently shut out of POST-rotation data. It introduces **no new crypto** — composition, an epoch-chain store model, atomic persistence, and the escrow re-point only.

## 1. Scope

**#106b-2 builds:**

1. **The VDK-epoch model + the epoch-keyed VDK chain (§3.1 — the load-bearing one).** The persisted shape that lets a client decrypt every entry under the VDK of the epoch it was written in: a chain of `(epoch, VdkKey)` where each VDK is wrapped per-device (#106b-1 `DeviceWrappedVdk`), under the password anchor (`keys.rs` `WrappedVdk`), and under the guardian escrow (`WrappedVdkRecovery`). The CURRENT epoch's VDK encrypts new writes; old epochs' VDKs are retained read-only to decrypt pre-rotation entries. (`pangolin-store` persistence; the crypto crate provides only the keys.)
2. **`rotate_vdk_for_survivors` (the rotation orchestration, §3.3).** A `pangolin-core` pure driver (peer of `recover_vdk_from_shares`) that, given the surviving device set + the current escrow state + the bumped epoch, produces: a fresh `VdkKey` (epoch n+1), the per-device wraps / pairing seals to each survivor, the password-anchor wrap input, and the re-pointed escrow (Q-d). PURE — no chain, no `uniffi`, no serde-on-secrets; every secret op delegates to a #106b-1 / #104a fn.
3. **The atomic rotation-commit (`pangolin-store`, §3.3).** A single `Vault` method (working name `Vault::commit_vdk_rotation`) that persists ALL rotation writes (new epoch's wraps + the re-pointed escrow + the advanced epoch + the new current-VDK pointer) in ONE `unchecked_transaction()`, mirroring `commit_recovery_rekey` (#105a). A crash mid-rotation must not strand a half-rotated vault.
4. **The Q-d escrow re-point** (§3.2, THE crux): on rotation, make the guardian recovery escrow point at the NEW VDK so a future guardian recovery restores the live key, not the dead old one. Recommend a mechanism (a/b/c below).
5. **The coupled regression gates (§8):** the forward-secrecy hermetic test (removed device can't open the new VDK; survivors can; escrow re-point correct; epoch chain decrypts old+new), the crash-injection atomicity test (#105a class), and ideally a coupled anvil test tying `removeDevice` → rotation.

**Deferred / out of scope:**
- **The `removeDevice` ON-CHAIN broadcast + `DeviceRemoved` observation/orchestration glue** (the watcher that turns the event into a rotation trigger) — #106c orchestration. #106b-2 provides the off-chain crypto reaction + the pure driver + the atomic store commit; the host drives the chain step (mirrors #104b/#105 keeping the broadcasts in the host).
- **The FFI/uniffi surface** for rotation — a later issue (the #105b-class follow-up), like recovery's FFI. `pangolin-core` stays pure.
- **Pairing UX / trust bootstrap** (QR / short-code) — #106e.
- **The read-side set-membership honor rule** — #106d.
- **Mainnet** — Base-Sepolia-only until D-011.

## 3. The design problems (designed; decisions surfaced in §5)

### 3.1 The VDK-epoch model — resolving "rotate the VDK" against IMMUTABLE on-chain history (LOAD-BEARING)

**The tension.** #106a's `RevisionLogV2.publishRevision` emits `RevisionPublished(... bytes encPayload ...)`, an APPEND-ONLY log entry; the contract has "No upgrades / no admin / never touches the VDK" cardinal rules and there is **no slot a VDK could live in and no way to re-encrypt a past entry**. Locally, the `revisions` table likewise stores each entry's `enc_payload` AEAD-sealed under the VDK (per-column AEAD bound to `(vault_id, page_id, schema_version)`, per `4.3-per-column-aead.md`), and revision history is append-only (P9 — losing-branch rows stay). So **"rotate the VDK" can NOT mean "re-encrypt all history under a new VDK"** — the on-chain bytes are frozen forever.

**The resolution — a chain of epoch-keyed VDKs.** Rotation mints a NEW `VdkKey` for a NEW **VDK epoch**; from that point, NEW writes encrypt under the new-epoch VDK; OLD on-chain (and old local) entries stay under their original (old-epoch) VDK. Concretely:

- Every revision is **already** stamped with `schemaVersion` and is signed by a device whose membership window is known; #106b-2 adds an explicit **`vdk_epoch` tag** the client uses to select the decrypting VDK. The recommended carrier is a small field inside the AEAD-protected `encPayload` envelope (NOT a new on-chain field — the contract is immutable and we must not need a v3) OR the existing `schemaVersion`/local `revisions` columns, resolved as **Q-a**. The client keeps a persisted **epoch→VDK chain**: `epoch 0 → VDK_0, epoch 1 → VDK_1, …`, each `VdkKey` wrapped three ways (per-device #106b-1 `DeviceWrappedVdk`; password `WrappedVdk`; escrow `WrappedVdkRecovery` — only the CURRENT epoch's VDK needs the live escrow, see §3.2/Q-c).
- **Sync picks the right VDK per entry:** decrypt entry `e` under `chain[e.vdk_epoch]`. An entry whose epoch is missing from the local chain is undecryptable on this device (correct for a removed device; a benign "not yet synced the wrap" state for a survivor, recoverable by pulling the wrap).
- **What "revoke a device" MEANS (the sentence for the spec/audit/UX):** the removed device retains access to data it had already synced at revoke time — it physically holds VDK epochs ≤ revoke-time and the on-chain ciphertext is immutable, so it can decrypt all PRE-rotation entries FOREVER (unretractable; no scheme can un-give plaintext already decrypted). Forward secrecy applies ONLY to POST-rotation entries: the new-epoch VDK is never sealed/wrapped to it, so it can never open anything written after the revoke.

**Local re-encrypt (cheap, optional — Q-b).** The on-chain log is multi-epoch forever, but the LOCAL store CAN be opportunistically re-encrypted to the current VDK (the local `revisions` rows are ours to rewrite — append-only is a *graph* discipline on revision IDs, not a ciphertext-immutability one; the row's `enc_payload` is just an at-rest blob). This collapses the local epoch chain to a single live VDK for the common case (smaller key chain, simpler unlock), while the on-chain history stays multi-epoch and the client still keeps OLD VDKs around to decrypt anything it re-pulls from chain that predates a local re-encrypt. **Recommend: keep the epoch chain as the source of truth (always correct); local re-encrypt is a later optimization (#106c), NOT required for #106b-2.** For a password manager (small payloads, infrequent revokes) the chain is cheap. (Surfaced as **Q-b**.)

**Why this is the load-bearing model:** it is the ONLY shape that simultaneously honors (1) immutable on-chain history, (2) forward secrecy for new data, and (3) continued decryptability of old data on surviving devices. It is under-specified everywhere upstream — #106b §3.1 says "re-encrypt the vault payload under the new VDK" as if it were a one-shot bulk op; this §3.1 corrects that to the epoch-chain reality and scopes the bulk re-encrypt as a local-only optimization.

### 3.2 Q-d — the guardian-escrow re-point (THE crux)

When rotation mints VDK epoch n+1, the existing `WrappedVdkRecovery` (sealing the OLD VDK under the OLD RWK, shares held by guardians) still points at the **dead old VDK**. A future guardian recovery would `reconstruct_rwk` → `unwrap_vdk_under_rwk` → restore VDK_n (the WRONG, dead key), silently stranding the user on a vault they can no longer decrypt new data with. The escrow MUST be re-pointed at VDK_{n+1}. Three mechanisms:

- **(a) Re-split a fresh RWK'** — exactly #104b's recovery re-split: `RecoveryWrapKey::generate()` → `wrap_vdk_under_rwk(new_vdk, &rwk', ctx)` → `split_rwk` → `seal_share` to all M guardians, bumped epoch (literally call `onboard_guardian_escrow` against the new VDK). **Forward-secure + reuses an already-audited path verbatim**, BUT every device-revoke pushes fresh shares to all M guardians — i.e. guardians must RE-RECEIVE their share on EVERY revoke. For a vault where you revoke devices often (selling a phone, rotating a work laptop) this is repeated "your guardians got new pieces" churn (the delivery is passive/non-interactive — no guardian *ceremony* — but the shares DO change and must propagate, #106e transport).
- **(b) Re-wrap the new VDK under the EXISTING RWK** — keep the same RWK + same guardian shares; only regenerate `WrappedVdkRecovery` = `wrap_vdk_under_rwk(new_vdk, &existing_rwk, ctx)`. **Zero guardian churn** (their shares still reconstruct the SAME RWK, which now unwraps the NEW VDK). BUT the device performing the revoke does NOT hold the RWK — it is threshold-split among guardians and only ever reconstructed at recovery from ≥ t shares. So (b) needs the RWK available locally at revoke time, which undermines the threshold-only custody model (the whole point is no single device holds the RWK).
- **(c) Device-held re-wrap capability** — the device retains the RWK (or a dedicated "re-wrap capability" = the RWK, or a separate `WrappedVdkRecovery`-mint key) wrapped under its OWN device key (biometric-gated, like the #106b-1 `DeviceWrappedVdk`). At revoke, it unwraps the RWK locally and does mechanism (b) WITHOUT bothering guardians AND without the RWK ever in the clear at rest. **Zero guardian churn + no threshold violation at rest.** Tradeoff: the RWK now exists in a single-device-recoverable form (a device whose biometric/enclave is compromised yields the RWK → that one device can re-point the escrow / recover the VDK without a quorum), versus the pure threshold-only model where the RWK exists nowhere except transiently during a recovery quorum.

**Recommendation: (a) for v1 (mandatory re-split), revisit (c) as a v2 UX optimization.** Rationale:
- (a) reuses the #104b re-split discipline the project already designed, audited, and accepted (the recovery-model memory's "all M shares are points on ONE polynomial — you cannot retire a subset" makes a *partial* re-split impossible anyway; if you re-split at all, it is all-M). It introduces ZERO new custody surface: the RWK still lives nowhere at rest, exactly the property the threshold model guarantees today.
- (b) is rejected as stated — it cannot run without the RWK, which the device doesn't have.
- (c) is the genuinely better UX (no guardian churn on revoke) but it WEAKENS the custody model: it puts the RWK (or its equivalent) on a single device behind only a biometric, so a compromised-but-not-yet-revoked device can reconstruct/re-point alone. That is a real security regression versus today, and it is exactly the kind of net-new custody decision that should NOT be smuggled into a flow issue. **Surface (c) explicitly to Kelvin as the lever:** *guardian-UX cost (re-distribute shares on every revoke) vs RWK-custody cost (RWK lives on a single biometric-gated device).* If Kelvin values revoke-without-churn enough to accept the single-device RWK risk, (c); otherwise (a).

**The escrow-re-point is MANDATORY regardless of mechanism (L-invariant L8 below, inherits #104b L6).** A rotation that skips it is a silent catastrophe — TESTED (a deliberately-skipped re-point must make a simulated guardian recovery restore VDK_n and turn a hermetic gate RED). (Surfaced as **Q-d**; mechanism (a) recommended.)

### 3.3 The rotation orchestration shape + where it lives + atomicity

Driven off a `removeDevice` (#106a), the surviving-device reaction is:

1. Observe `DeviceRemoved` (host/#106c) → resolve the surviving authorized set + the current escrow state + the next epoch (`current_epoch.next()`, the shared counter, §4).
2. `pangolin-core::rotate_vdk_for_survivors(...)` (pure): mint `VdkKey::generate()` (epoch n+1) → for each SURVIVOR, produce the new-epoch wrap (`wrap_vdk_for_device` for an already-paired survivor, and/or `seal_vdk_to_device` for the pairing path) → produce the password-anchor wrap input → produce the re-pointed escrow (Q-d mechanism (a): call `onboard_guardian_escrow(new_vdk, …, epoch.next())`). Never wrap/seal to the removed device. Output a `RotationArtifacts` (peer of `RecoveryArtifacts`): the new VDK + the survivor wraps + the re-split escrow + the new epoch.
3. `pangolin-store::Vault::commit_vdk_rotation(...)` (atomic): in ONE `unchecked_transaction()` write — (a) the new-epoch entry into the VDK-chain table (per-device wrap + password wrap), (b) the re-pointed escrow via `write_recovery_escrow_tx` (reuse #105a), (c) advance the persisted shared epoch counter, (d) update the "current VDK epoch" pointer; commit ONCE; update in-memory `ActiveState` ONLY after commit. A crash before commit rolls ALL back — the vault stays on the OLD epoch (still fully functional, the removed device still in the set on-chain until the off-chain reaction re-runs; the rotation is safely retryable, #105a L8 class).
4. (Optional, #106c) local re-encrypt to the new VDK (§3.1/Q-b).

**Where it lives:**
- `crates/pangolin-core/src/recovery/` (or a new `rotation` sibling) — `rotate_vdk_for_survivors` + `RotationArtifacts`. Pure, mirrors `orchestration.rs`.
- `crates/pangolin-crypto/src/pairing.rs` — REUSED as-is for the survivor wraps/seals; **expected ZERO change** (the primitive already does per-device-wrap + pairing-seal with epoch binding).
- `crates/pangolin-store/src/vault.rs` — `Vault::commit_vdk_rotation` (atomic), mirroring `commit_recovery_rekey`; a new VDK-chain table in `schema.rs` (additive, the `recovery_escrow`/`meta` idiom).
- `crates/pangolin-store/src/recovery_escrow.rs` — `write_recovery_escrow_tx` REUSED for the escrow re-point (no change needed; it already `INSERT OR REPLACE`s + DELETEs prior guardian rows, so the old generation can't linger).

**Atomicity concern (the #105a lesson, explicit):** the new daily/per-device wraps, the re-pointed escrow, and the epoch advance MUST commit together or not at all. A crash that lands the new escrow but not the new per-device wrap (or vice versa) could strand a vault where the escrow points at a VDK no surviving device can produce, or where survivors hold a new VDK the escrow can't recover — both catastrophic. The single-transaction wrapper is the structural defense; the crash-injection gate (L11) proves it.

### 3.4 Interaction with recovery (#104b) + the shared epoch (§4)

Two operations touch the VDK/authority and MUST NOT desync the shared per-vault epoch:

- **Recovery (#104b)** RE-WRAPS the SAME VDK to a new authority + re-splits a fresh RWK; the VDK is preserved bit-for-bit; epoch advances (recovery re-split).
- **Device-revoke (#106b-2)** RE-CREATES the VDK for survivors + re-points the escrow; epoch advances (rotation).

**One shared monotonic per-vault epoch (Q-f, #106b §0a LOCKED).** Both advance the SAME counter (`RecoveryEpoch`, the existing `u64` in `recovery_escrow` + its 16-byte `to_escrow_bytes` form). Reconciliation rules:
- **The epoch is the single source of truth for "key-state generation."** Recovery and rotation both call `current_epoch.next()` and persist the bumped value in the SAME atomic transaction as their wraps/escrow. Because each is atomic and reads-then-bumps the persisted counter, they cannot double-count or skip (a crash rolls back the bump with everything else).
- **The VDK-chain entry is keyed by epoch.** Recovery's re-wrap of the SAME VDK does NOT add a new chain entry (same VDK, new password authority) — it bumps the epoch ONLY for the escrow re-split's domain separation, and the current VDK pointer is unchanged. Rotation ADDS a new chain entry (new VDK) at the new epoch and moves the current pointer. So the chain entry is created iff the VDK actually changed — **Q-e** (confirm: recovery bumps the epoch without a new VDK-chain entry; rotation bumps the epoch AND adds one). This keeps "epoch" = "key-state generation" while "VDK-chain length" = "number of distinct VDKs," and the two are intentionally not equal.
- **No on-chain coupling.** Neither operation needs a new on-chain field; `removeDevice` is the trigger, the contract never carries the epoch or the VDK (RevisionLogV2 cardinal "never touches the VDK / no slot"). The shared epoch is purely off-chain client state.

## 4. L1..Ln invariants (proposed — mirror 104a/104b/105/106b style)

- **L1 (forward secrecy — CATASTROPHIC-IF-WRONG, LOAD-BEARING).** After `rotate_vdk_for_survivors` + `commit_vdk_rotation`, a removed device CANNOT derive/open the post-rotation VDK: the new-epoch VDK is never sealed (`seal_vdk_to_device`) or wrapped (`wrap_vdk_for_device`) to it, and the epoch bump rejects any stale envelope. TESTED: a removed device (its OLD device key + OLD wraps) cannot open the new `DeviceWrappedVdk`/`SealedVdkForDevice` and cannot decrypt a payload encrypted under VDK_{n+1}. A broken rotation that re-wraps to the removed device MUST turn the gate RED.
- **L2 (escrow always points at the CURRENT VDK after rotation — LOAD-BEARING; §3.2).** After rotation the recovery escrow recovers VDK_{n+1}, NOT the dead VDK_n. TESTED: a simulated guardian recovery (`reconstruct_rwk` → `unwrap_vdk_under_rwk`) yields `ct_eq` VDK_{n+1}; skipping the re-point makes it yield VDK_n (the catastrophe the test guards). Inherits #104b L6 / the "can't retire a subset of one polynomial's shares" discipline.
- **L3 (old entries still decryptable via the epoch chain).** Surviving devices retain every OLD-epoch VDK and decrypt pre-rotation entries (on-chain immutable + local) under `chain[entry.vdk_epoch]`. TESTED: post-rotation, a survivor opens both an old-epoch entry (under VDK_n) and a new-epoch entry (under VDK_{n+1}).
- **L4 (atomic rotation persistence — the #105a lesson).** The new-epoch wraps (per-device + password), the re-pointed escrow, the advanced epoch, and the current-VDK pointer commit in ONE sqlite transaction or not at all. No reachable on-disk state has a new escrow generation beside stale per-device wraps (or vice versa). Asserted by the crash-injection gate (L11). Inherits #105a L2.
- **L5 (single monotonic shared epoch; no desync with recovery; §4).** One per-vault `RecoveryEpoch` advances on onboarding, recovery re-split, AND device-revoke rotation; bound into every seal header + escrow domain. Recovery and rotation read-then-bump it inside their own atomic transaction so they cannot double-count or skip. (Q-f LOCKED upstream; Q-e confirms the chain-entry-vs-epoch-bump distinction.)
- **L6 (reuse audited #106b-1 / #104a primitives; NO novel primitive).** The survivor handoff is the #106b-1 `seal_vdk_to_device`/`wrap_vdk_for_device` (XSalsa20-Poly1305 sealed box + XChaCha20-Poly1305 AEAD, already audited); the escrow re-point is the #104a/#104b `onboard_guardian_escrow` path verbatim; the new VDK is `VdkKey::generate()` (the ONE legitimate re-create, gated to revoke, distinct from recovery — L9). ZERO new HKDF info strings, ZERO new sealed-box, ZERO new deps (`cargo tree -p pangolin-crypto | grep -ci serde == 0` preserved).
- **L7 (zero-serde / secret-type discipline).** `RotationArtifacts` and any epoch-chain key type carry the #104a discipline: no serde on the secret path, `!Clone`/`!Copy` on secrets, zeroizing, redacted `Debug`, `ct_eq`, fixed-layout byte encodings. The pure driver carries only public context (epoch, vault_id, survivor device_ids, X25519 pubkeys), never key material across the pure boundary.
- **L8 (device-revoke re-points the guardian escrow — LOAD-BEARING; §3.2).** A rotation MUST re-point the escrow at the new VDK (mechanism (a): fresh RWK' re-split to all M; or (c) if Kelvin accepts the custody tradeoff). The escrow can never point at a dead VDK. Inherits #104b L6.
- **L9 (recovery vs revoke do not conflict).** Recovery re-wraps the SAME VDK (no re-create, no new chain entry); device-revoke re-creates the VDK (new chain entry). The whitepaper "VDK is re-wrapped, not re-created" is the RECOVERY invariant (#106b §3.2, ratified); the spec addendum records device-revoke as the distinct operation that legitimately re-creates the VDK. `VdkKey::generate` is on the revoke path ONLY, never the recovery path.
- **L10 (`forbid(unsafe_code)` except FFI; AGPL SPDX; testnet-only until D-011).** Every new file. `pangolin-core`/`pangolin-crypto`/`pangolin-store` stay zero-`unsafe` (only the documented uniffi/ffi surface may carry it). The rotation flow stays Base-Sepolia-only until D-011.
- **L11 (hermetic adversarial test posture = CI gate; §8).** The forward-secrecy gate (removed device can't open the new VDK), the escrow-re-point gate (recovery restores the new VDK), the epoch-chain gate (old+new both decrypt), and the crash-injection atomicity gate are CI gates; a deliberately-broken binding (wrap to the removed device / skip the re-point / split the commit into two transactions / mis-tag an entry's epoch) MUST turn it RED (env-quirk #14 class). Ideally a coupled anvil test ties `removeDevice` → the rotation.
- **L12 (§16 ledger).** `git merge --no-ff`; DECISIONS.md Q-resolution entries; DEVLOG + THREAT_MODEL rows (device-kill / VDK rotation, post-audit); spec-addendum row recording the epoch-chain model + the §3.2 re-point mechanism choice; explicit Kelvin approval at the merge boundary (§16.3).

## 5. Open decisions for Kelvin (Q-a … Q-h) — recommendation + plain-English stakes

### Q-a · How is each entry's VDK epoch tagged so the client picks the right decrypting VDK?
- *Term:* every saved password entry was encrypted under whatever VDK was current when you saved it; after a revoke there's a new VDK, so each entry needs a little label saying "I'm from VDK generation N."
- *Recommendation:* tag it **inside the AEAD-protected `encPayload` envelope** (a 1–2 byte epoch field the client writes + reads), NOT as a new on-chain field — the contract (`RevisionLogV2`) is immutable and we must never need a v3 just to add a tag. Mirror the tag in the local `revisions` table for fast selection. (Confirm whether the existing `schemaVersion` byte can double as / coexist with the epoch tag, or it gets its own field.)
- *Plain-English stakes:* this label is what lets your surviving phone still read old passwords (under the old key) AND new ones (under the new key) after you've revoked a device. Putting it inside the already-encrypted blob means we don't have to change the on-chain contract. **Stakes: confirm we tag epoch in the payload envelope, not on-chain.**

### Q-b · Does the LOCAL store get re-encrypted to the new VDK on rotation, or do we keep the full epoch chain locally?
- *Recommendation:* **keep the epoch chain as the source of truth (always correct); make local re-encrypt a later optimization (#106c), NOT part of #106b-2.** On-chain history is multi-epoch forever (immutable); locally we *could* rewrite our own rows to the newest VDK to shrink the key chain, but that's a perf nicety, not a correctness need.
- *Plain-English stakes:* on-chain, your old encrypted entries can never be re-encrypted — they're frozen. Your own device's local copy could be rewritten to use just the newest key (fewer keys to juggle), but that's an optimization we can add later; for now your device just keeps the small list of keys and picks the right one per entry. For a password manager that list is tiny. **Stakes: confirm we ship the always-correct epoch chain now and defer local re-encrypt.**

### Q-c · Does every epoch's VDK stay in the guardian escrow, or only the current one?
- *Recommendation:* **only the CURRENT epoch's VDK is in the live guardian escrow.** Old-epoch VDKs are retained locally (per-device + password wrapped) for read-only decryption of old entries, but a guardian recovery only ever needs to restore the CURRENT VDK (which, plus the locally-retained old VDKs once you're back in, gives you everything). A lost-everything recovery restores the current VDK; the old VDKs ride along inside the (current-VDK-encrypted) local backup / re-pull.
- *Plain-English stakes:* guardians only ever help you get back your *current* key, not a museum of every key you've ever had — simpler and smaller. The catch (flag): in a true lost-everything recovery you get the current VDK back, and your old entries become readable again only once your restored vault re-obtains the old VDKs (from your encrypted backup or a re-pull). **Stakes: confirm guardians escrow only the current VDK; old-VDK retention is a local-backup concern.**

### Q-d · THE CRUX — how does a device-revoke re-point the guardian escrow at the new VDK? (a) re-split fresh RWK', (b) re-wrap under existing RWK, (c) device-held RWK re-wrap.
- *Recommendation:* **(a) re-split a fresh RWK' to all M guardians, MANDATORY** (reuses #104b's audited recovery re-split; introduces no new RWK-custody surface). **(b) is rejected** (the revoking device doesn't hold the RWK — it's threshold-split). **(c) device-held re-wrap** is the better UX (no guardian churn on revoke) but WEAKENS custody (the RWK lives on a single biometric-gated device, so a compromised device can re-point/recover alone) — offer it as the explicit lever.
- *Plain-English stakes (the real trade Kelvin is choosing):*
  - **(a) guardian-UX cost:** every time you revoke a device, your guardians' recovery pieces silently change and must re-sync to them. Revoke devices a lot (selling phones, work laptops) and that's repeated "your guardians got new pieces" churn — annoying but security-clean (no one device ever holds the master recovery key).
  - **(c) RWK-custody cost:** revoke without ever bothering guardians, because the device keeps the master recovery key behind its own fingerprint lock. Slick, BUT a thief who unlocks one of your devices now holds the master recovery key — they could re-point or recover your vault alone, without needing your guardians. That's strictly weaker than today's "no single device holds it."
  - **Recommend (a)** unless Kelvin specifically wants revoke-without-guardian-churn enough to accept a single-device master-key. **Stakes: this is the decision with a real cost either way — confirm (a), or consciously choose (c).**

### Q-e · Recovery bumps the epoch WITHOUT a new VDK-chain entry; rotation bumps the epoch AND adds one — confirm the distinction.
- *Recommendation:* **YES.** The shared epoch counts "key-state generations" (advances on recovery AND rotation); the VDK chain counts "distinct VDKs" (grows on rotation only — recovery keeps the same VDK). So after a recovery, epoch went up but the VDK chain didn't; after a revoke, both did.
- *Plain-English stakes:* recovery gives you the SAME vault key back (your old entries still open with it); revoking a device makes a NEW key for new entries. Both tick the version counter, but only a revoke adds a new key to your key list. **Stakes: confirm the version-counter and the key-list are intentionally not the same length.**

### Q-f · One shared per-vault epoch for both recovery re-split AND rotation? (already LOCKED upstream — confirm it holds here)
- *Recommendation:* **YES, ONE shared monotonic per-vault epoch** (#106b §0a Q-f LOCKED; reuse `RecoveryEpoch`). Both operations read-then-bump it inside their own atomic transaction, so they can't desync or double-count.
- *Plain-English stakes:* there's one "version number" for your vault's key state; recovery and revoke both tick it up, and anything stamped with an old number is handled correctly by the per-entry epoch tag. One clean clock. **Stakes: confirm the single shared clock carries over to rotation.**

### Q-g · Where does the rotation orchestration live + how does the host trigger it off `removeDevice`?
- *Recommendation:* the **pure driver** (`rotate_vdk_for_survivors`) lives in `pangolin-core`; the **atomic commit** (`commit_vdk_rotation`) lives in `pangolin-store/vault.rs` (mirroring `commit_recovery_rekey`); the **`removeDevice` broadcast + `DeviceRemoved` observation** that *trigger* the reaction live in the host/#106c (engine stays pure, host owns the chain choreography — exactly like recovery). #106b-2 ships the crypto reaction + the atomic store primitive; #106c wires the watcher.
- *Plain-English stakes:* the engine is a toolbox ("here's how to mint a new key and re-key the survivors atomically"); the app decides WHEN to call it (when it sees a device was removed on-chain). Same split as recovery. **Stakes: confirm #106b-2 builds the reaction + the atomic store step, and the on-chain trigger/watcher is #106c.**

### Q-h · Split #106b-2 into a VDK-epoch primitive (#106b-2a) + the orchestration (#106b-2b)?
- *Recommendation:* **NO — keep it as one stage** (optionally 2 PRs on one branch). Unlike #106b-1, the epoch-chain key type is a thin record over already-audited `VdkKey`/`WrappedVdk`/`DeviceWrappedVdk` — no new primitive, no new HKDF info, no new sealed-box — so it doesn't warrant its own audit boundary. The forward-secrecy + escrow-re-point properties can only be asserted with the orchestration present (the coupled test IS the value), same logic as #104b.
- *Plain-English stakes:* the risky new crypto ("hand/kill a key") was already built and audited as #106b-1; what's left is wiring, which is best audited as one connected whole rather than split into a half-wired piece with no real test. **Stakes: confirm one stage.**

## 6. Test posture (#104a-grade hermetic adversarial)

- **Forward secrecy (L1, the centerpiece):** mint VDK_n → wrap/seal to devices {A, B, C} → revoke C → `rotate_vdk_for_survivors` mints VDK_{n+1} wrapped to {A, B} only → assert A and B open VDK_{n+1} (their `DeviceWrappedVdk`/`SealedVdkForDevice`); assert C (its OLD device key + OLD wrap) CANNOT open the new wrap/seal; assert a payload encrypted under VDK_{n+1} does NOT decrypt under VDK_n. A rotation that re-wraps to C MUST fail.
- **Escrow re-point (L2/L8):** after rotation, a simulated guardian recovery (`reconstruct_rwk` from ≥ t opened NEW shares → `unwrap_vdk_under_rwk`) yields `ct_eq` VDK_{n+1}; the OLD shares reconstruct the OLD (dead) RWK and unwrap-fail against the new wrapper (the #104b `old_shares_cannot_recover_post_recovery_vault` pattern, retargeted to revoke); skipping the re-point makes recovery yield VDK_n (the guarded catastrophe).
- **Epoch chain (L3):** post-rotation, a survivor opens an old-epoch entry under VDK_n AND a new-epoch entry under VDK_{n+1}; an entry mis-tagged to the wrong epoch fails to decrypt (the per-entry epoch-tag selection is correct).
- **Atomic rotation persistence (L4/L11, the #105a class):** drive `commit_vdk_rotation`, force a rollback between the wrap-write and the escrow-write (or before the epoch-bump), re-open the vault, assert the OLD epoch is still the live generation (old wraps + old escrow intact, NO new-epoch chain row, epoch counter unchanged) and the rotation is safely retryable; positive path: commit → new epoch live, removed device dead for new data, escrow recovers the new VDK.
- **Recovery × revoke composition (L5/L9):** a recovery (same VDK, epoch+1, no new chain entry) followed by a revoke (new VDK, epoch+1, new chain entry) keeps the shared epoch monotonic and the chain correct; the escrow tracks whichever VDK is current.
- **Proptest (≥1024 cases):** random vault_id / survivor-set / device_id / epoch / VDK round-trip the survivor wraps + the escrow re-point byte-equal; the removed device is never in the wrap set.
- **Domain-separation grep assertion:** REUSED from #106b-1 (`domain_strings_are_versioned_and_distinct`) — #106b-2 adds NO new info strings; the assertion confirms that (a regression that adds an unversioned one fails).
- **`#[ignore]`'d coupled anvil test (ideal):** deploy RevisionLogV2 → bootstrap + addDevice×N → `removeDevice`(C) → off-chain rotation → assert C's signer dropped from the on-chain SET AND C can't open the new VDK AND the escrow recovers the new VDK. Same posture as #104b's coupled E2E / the live `#[ignore]` track.
- **Debug-redaction snapshots; `cargo tree` serde-count == 0; `forbid(unsafe_code)` compile check.**

## 9. Where the existing crypto / store / contracts do NOT cleanly support rotation (GAP FLAGS)

1. **(§3.1 — THE tension) Immutable on-chain history vs "rotate the VDK".** `RevisionLogV2`'s `RevisionPublished.encPayload` is APPEND-ONLY/IMMUTABLE and the contract has no VDK slot — so "rotate the VDK" CANNOT re-encrypt history. The merged code has NO epoch-keyed VDK chain and NO per-entry epoch tag; `#106b §3.1` over-simplified rotation as a one-shot "re-encrypt the vault payload." #106b-2 must build the epoch-chain model + the per-entry tag (Q-a) — this is the load-bearing new shape and is under-specified everywhere upstream.
2. **(§3.2 / Q-d — RWK custody) The revoking device does not hold the RWK.** The RWK is threshold-split among guardians and reconstructed ONLY at recovery (≥ t shares); there is NO local RWK at rest (by design). So the "cheap" re-point (mechanism (b)) is impossible without a custody change (mechanism (c)), and the clean re-point (mechanism (a)) re-involves all M guardians on every revoke. The merged escrow surface supports (a) verbatim (`onboard_guardian_escrow`) but offers no path for (b)/(c) without net-new key-custody — the genuine open decision.
3. **(VDK-chain persistence) No epoch-keyed VDK store exists.** Today `meta` holds exactly ONE `wrapped_vdk` (the single current VDK under the password authority) and `ActiveState` holds ONE unwrapped `VdkKey` whose `aead_key()` is THE column-AEAD key. There is no notion of multiple retained VDKs. #106b-2 must add an additive `vdk_chain` table + epoch-aware unlock (load all retained VDKs; decrypt each entry under its epoch's VDK). Additive, low-risk, but net-new store surface — flag for the in-house re-audit (it touches the at-rest model the 4.3 per-column-AEAD audit signed off on).
4. **(`VdkKey::generate` on a re-key path) The recovery invariant says the VDK is never re-created.** `keys.rs` notes `VdkKey::generate` is "never on a re-key path" (the recovery invariant). #106b-2 introduces the ONE legitimate re-create-the-VDK operation, gated strictly to device-revoke and explicitly distinct from recovery (L9). The spec addendum + D-011 package MUST record this so an auditor doesn't read it as a violation of the "re-wrapped not re-created" invariant (which is the RECOVERY invariant — #106b §3.2, already ratified).

No other gaps — the #106b-1 pairing/wrap primitive, the #104a/#104b escrow + re-split, and the #105a atomic-commit pattern otherwise compose cleanly for the rotation flow.
