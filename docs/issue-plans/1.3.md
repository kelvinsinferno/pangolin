> **Status:** PLAN approved 2026-05-11. Q1-Q5 locked below.

## Goal

After this issue closes, `Vault::account_search` is backed by a
persistent SQLite **FTS5** virtual table over the *non-secret*
searchable projection of every live account identity — `display_name`,
each `tag`, and the `host_str()`-extracted hostname of each `urls`
entry — never `usernames`, `notes`, `password_history`, or
`totp_secret` (the whitelist is structural: the FTS5 columns simply do
not contain those fields). The index is kept in sync from
`Vault::account_add` / `account_update` / `delete_account` (and the V0
`add_account` / `update_account` / `delete_account` shims that 1.2 left
in place), is rebuilt from the encrypted blobs on first open of a vault
that predates 1.3 (V0-format and 1.2-V1-format vaults alike), and is
rebuilt rather than erroring if it is found corrupt/missing. Search of
a 10k-account vault returns in well under 50 ms. WAL mode and the
existing transactional discipline carry over unchanged. No new crate is
added (`rusqlite`'s `bundled` SQLite already ships FTS5); HIGH-1
(`pangolin-crypto` zero-serde) and Q3 (`pangolin-core`/`pangolin-store`
zero-uniffi) are unaffected. Existing PoC `.pvf` vaults still open;
`forbid(unsafe_code)` survives on every crate.

## Approach

> **⚠ Read this first — the "Decisions locked" table at the top of this
> file supersedes the prose below where they differ.** This §Approach
> was written by the plan-gate before Kelvin's Q2 answer; it still
> describes a *persistent* `account_search_src` + `account_search_fts`
> pair living inside the `.pvf` SQLite file. **Kelvin locked Q2 to the
> `:memory:` rebuild-on-unlock design instead:** the FTS5 virtual table
> and its plaintext source projection live in a `rusqlite::Connection::
> open_in_memory()` connection that is built from the decrypted blobs at
> unlock time and torn down on `lock()`/`close()` — **nothing
> plaintext-searchable is ever written to the `.pvf` file**, so the
> `no_plaintext_on_disk` proptest is unchanged and the persistent-table
> schema, triggers, and on-disk-corruption-recovery paths described
> below do **not** apply (the index is RAM-only and trivially rebuilt
> from the intact blob table on every unlock). The column set
> (`display_name`, `tags`, `hostnames`), the trigram tokenizer, the
> hostname-extraction rule, the bm25+recency ranking, the cap-200, and
> the whitelist enforcement all carry over as written; only the
> *storage location* changed (disk → `:memory:`). The implementation
> lives in `crates/pangolin-store/src/search.rs`; see
> `docs/architecture/search.md` for the as-built description.

### 1. SQLite schema + the FTS5 table

**Current schema** (`schema.rs`, unchanged): `meta` (single-row;
magic, format_version, vault_id, KDF params, salt, wrapped VDK),
`account_identities` (`account_id` PK, `created_at`, `last_modified_at`,
`tombstoned`, `head_revision_id`, `frozen_pending_resolve`),
`revisions` (`revision_id` PK, `account_id`, `parent_revision_id`,
`device_id`, `schema_version`, `created_at`, `enc_payload` BLOB,
`enc_nonce` BLOB, `is_tombstone`, chain-anchor columns), `devices`,
`sync_state`, `dirty_accounts`, `pending_merges`. **All secret content
lives only inside `revisions.enc_payload`** (AEAD ciphertext); every
SQL column is non-secret structural metadata. **Blobs are stored
*inside* SQLite as BLOB columns — there is no separate per-blob file on
disk.** ("Atomic blob rename" in the master plan row therefore means
the SQLite WAL/atomic-row-replace discipline P2-4 already implements,
not a temp-file+`rename()` dance — see §3.)

**New in 1.3** — an external-content FTS5 virtual table plus a small
shadow table holding the (necessarily plaintext) projection it indexes:

- `account_search_src` — the plaintext non-secret projection, one row
  per *searchable token-group* per account, `rowid`-linked to
  `account_identities` via a stable integer surrogate key (FTS5 needs an
  integer rowid; `account_id` is a 32-byte BLOB). Columns:
  `rowid INTEGER PRIMARY KEY` (the surrogate), `account_id BLOB NOT
  NULL`, `display_name TEXT`, `tags TEXT` (space-joined canonical tag
  list — already lowercased+deduped by 1.2's validator), `hostnames
  TEXT` (space-joined `Url::parse(u).host_str()` of each `urls` entry;
  for schemes with no host — `mailto:`, `app://settings`, etc. — fall
  back to the raw serialised URL string so non-http URLs stay
  searchable, matching 1.2's Q3 note). `UNIQUE(account_id)`.
- `account_search_fts` — `CREATE VIRTUAL TABLE ... USING fts5(
  display_name, tags, hostnames, content='account_search_src',
  content_rowid='rowid', tokenize='<TOKENIZER — see Open Q1>')`.
  External-content mode keeps the FTS5 index from duplicating the
  source columns; the source row is the single place the plaintext
  projection lives in the DB file.
- A `meta`-level marker row for the FTS schema version (see §7) — a new
  single-row `fts_meta(id INTEGER PRIMARY KEY CHECK(id=0),
  fts_schema_version INTEGER NOT NULL DEFAULT 1)` table, idempotent
  `CREATE TABLE IF NOT EXISTS` like `sync_state`/`dirty_accounts`. No
  `format_version` bump (1.6 owns whether a future tokenizer change
  forces a rebuild; 1.3 only stamps the slot).

**Why a shadow table rather than triggers on `account_identities`:**
the searchable projection is derived from the *decrypted blob*, which
SQLite cannot see. The plaintext projection has to be computed in Rust
at `account_add`/`account_update` time anyway, so the FTS5 sync is a
plain `INSERT OR REPLACE` into `account_search_src` + the matching FTS5
`INSERT`/`'delete'` command, inside the same transaction that writes
the revision (§5). No SQL triggers; sync is explicit and lives next to
the blob write so it cannot drift.

**FTS5 availability:** `rusqlite = { version = "=0.39.0",
default-features = false, features = ["bundled"] }` (workspace
`Cargo.toml`) — the `bundled` amalgamation is compiled with
`SQLITE_ENABLE_FTS5`, so no Cargo change and no new crate. Confirm at
build time with a `SELECT fts5(?1)` smoke or a `CREATE VIRTUAL TABLE
USING fts5` probe in a test; if (unexpectedly) absent, add
`features = ["bundled", "modern_sqlite"]` — still no new crate.

### 2. The secret/non-secret boundary — the load-bearing question

`AccountIdentity` (1.2) wraps `display_name`, `tags`, `urls` in
`SecretBytes` "encrypted at rest" — but FTS5 needs them in plaintext to
index. Whitepaper §C1's properties are *"Fast search"* **and** *"No
plaintext secrets at rest."* The master plan §4 row 1.3 resolves the
tension explicitly: **"FTS5 on non-secret fields only — name, tags,
hostnames; never on usernames/passwords."** That sentence is the design
authority — it classifies `display_name`, `tags`, and URL-derived
`hostnames` as **non-secret** (the leak a stolen `.pvf` already exposes:
account count, revision parentage, timestamps, device ids — adding
service names/tags/hostnames is the same metadata-leakage class P2's
threat-model row #1 already acknowledges), while `usernames` and
`passwords` (and by 1.2's §5.4 classification `notes` and `totp_secret`)
stay encrypted-only and out of the index.

**Recommendation (carried by this plan, surfaced as Open Q2):**
**persistent plaintext FTS5 + plaintext shadow projection** (master-plan
option 1) — the `account_search_src`/`account_search_fts` design above.
Keep the `SecretBytes` wrapper on `display_name`/`tags`/`urls` *inside*
`AccountIdentity` (don't reclassify the model fields — minimal 1.3 diff,
and the wrapper still buys zeroize-on-drop for the in-memory copy and
keeps the blob payload encrypted-at-rest), but accept that the FTS5
projection is a plaintext copy of those three derived fields in the DB
file. This is what §4-row-1.3 implies and what makes "fuzzy" achievable.
The residual choice (tokenizer — `trigram` for substring/fuzzy vs.
`unicode61` for prefix; sync mechanism; whether the 10k/<50ms gate is a
hard test assertion or a `cargo bench` target) is Open Q1/Q3/Q4.

**Why not the alternatives** (documented so the audit doesn't re-litigate):
- *Option 2 — encrypted FTS5 over HMAC'd tokens* — exact-token match
  only; defeats "fuzzy matching". Rejected.
- *Option 3 — `:memory:` FTS5 rebuilt on every unlock* — satisfies a
  literal "no plaintext on disk" reading and is a fallback if Kelvin
  rejects the plaintext-projection leak (see Open Q2). Cost: the
  rebuild reads + AEAD-decrypts every head blob at unlock, which the
  PoC already does anyway (`unlock` decrypts all heads into the
  in-memory cache), so building a `:memory:` FTS5 from those is a few ms
  for 10k accounts on top. It's strictly more secure (nothing extra on
  disk) at the cost of unlock latency and code complexity (a second
  SQLite connection). **If Kelvin picks option 3, the rest of this plan
  carries over with `account_search_src`/`_fts` living in a `:memory:`
  attached DB rebuilt in `unlock`/dropped in `lock`, and §F's
  `no_plaintext_on_disk` extension becomes a no-op (nothing changes on
  disk).**
- *Option 4 — salted-but-not-secret persistent form* — no real security
  gain over option 1 (the salt is in the same file) and loses fuzziness.
  Rejected.

### 3. Corruption safety

- **WAL mode:** already set — `schema.rs::apply_pragmas_and_schema`
  runs `PRAGMA journal_mode = WAL` then `PRAGMA synchronous = FULL` on
  every open; `assert_wal_mode` verifies it; `assert_integrity` runs
  `PRAGMA integrity_check` on open. **1.3 keeps all of this unchanged.**
  The new FTS5 + shadow tables are ordinary SQLite objects covered by
  the same WAL.
- **"Atomic blob rename":** blobs are BLOB *columns* inside the single
  `.pvf` SQLite file, not separate files — so WAL covers their
  atomicity and no temp-file+`rename()` dance exists or is needed (P2-4
  noted this; documented here for the audit). The FTS5 sync write
  (§5) goes inside the same `unchecked_transaction()` that writes the
  revision row, so an interrupted update cannot leave the FTS5 index
  desynced from the blob table — either both land or neither.
- **Crash-recovery test:** the existing `crash_during_write_recovers_via_wal`
  (in `crates/pangolin-store/tests/e2e.rs`) stays. 1.3 adds: (a) a
  variant that kills the process between the `revisions` INSERT and the
  `account_search_src`/FTS5 INSERT *within* the transaction — on
  reopen, the transaction is rolled back, FTS5 still matches the blob
  table; (b) a "FTS5 index deleted/corrupt" test — `DROP TABLE
  account_search_fts; account_search_src` on a closed vault, reopen, and
  assert `Vault::open` rebuilds the index from the blobs rather than
  erroring (the open path detects the missing table via
  `sqlite_master` and runs the rebuild — same posture as the
  `migrate_*` helpers in `schema.rs`).
- **umask quirk #1:** the FTS5 shadow tables are in-DB so no new file
  is created in the persistent-design path. If Open Q2 lands on the
  `:memory:` design, the attached DB is `:memory:` (no file) — still
  no umask exposure. Either way, no `UmaskGuard` change is needed; any
  *test* that writes a fixture `.pvf` reuses the existing
  `UmaskGuard`/`nix::umask(0o022)` pattern (quirk #1).

### 4. Performance — 10k accounts, <50 ms

- A new `cargo bench` (`crates/pangolin-store/benches/search_10k.rs`,
  Criterion or a hand-rolled `Instant`-timed harness — see Open Q4)
  that: generates a 10k-account vault (display names like
  `"Service {i}"`, a couple of tags, one `https://host{i}.example`
  URL each), unlocks it, and times `Vault::account_search("service")`,
  `account_search("host42")`, `account_search("<a rare tag>")`. With an
  FTS5 index this is an indexed lookup + a bounded result decode, ~µs–
  low-ms for the index hit; the dominant cost is AEAD-decrypting the
  head blobs for the *matched* accounts to build the
  `AccountIdentitySummary` results (so the API should `LIMIT` results —
  see §5). Documented target: **median < 25 ms, p99 < 40 ms** for any
  single-term query over 10k accounts, leaving headroom under the §4
  exit-criterion's 50 ms.
- One *test* (not just a bench) asserts a coarse upper bound (e.g.
  `< 40 ms` for a single-term search over a 2k-account vault, scaled
  down so CI flakiness doesn't fail the build) — or, if Open Q4 lands
  on "bench-only", the test is replaced by a `#[ignore]`-by-default
  10k-account `cargo test --release -- --ignored search_10k` that the
  CI `bench` lane runs explicitly. The bench is authoritative; the
  test is a smoke.
- If Open Q2 lands on the `:memory:` rebuild-on-unlock design, the
  bench also times the unlock-time rebuild for 10k accounts and
  documents it (expected: dominated by the AEAD decrypts the PoC
  `unlock` already does, so the FTS5 build adds only a few ms).

### 5. The `account_search` API contract + sync mechanism

- **Signature:** `Vault::account_search(&mut self, query: &str) ->
  Result<Vec<AccountIdentitySummary>>` — **unchanged from 1.2**, so the
  frozen `account_search` FFI entry point (`pangolin-ffi/src/identity.rs`)
  keeps calling it as-is. `&mut self` because the PoC pattern uses it
  for session-freshness bookkeeping (`check_session_freshness` /
  `touch_session`); keep that.
- **Backing:** FTS5 `MATCH` over `account_search_fts`, joined back to
  `account_identities` (skip `tombstoned = 1` and
  `frozen_pending_resolve = 1` rows — preserve 1.2's `frozen_set`
  filtering), then for each surviving `account_id` read the head
  revision, AEAD-decrypt it via `read_identity_at`, and build the
  `AccountIdentitySummary` exactly as 1.2 does (`identity_to_summary`).
- **Ranking:** FTS5's `bm25()` relevance ranking, `ORDER BY
  bm25(account_search_fts)` — best-match-first. (Recency as a
  tiebreaker — `account_identities.last_modified_at DESC` — is a small
  add; surface as part of Open Q3.)
- **Result limit:** `LIMIT 200` (the §16.8-class "render a list"
  ceiling) — enough for any human's search, bounds the blob-decrypt
  cost. Document as a constant `ACCOUNT_SEARCH_RESULT_CAP = 200`.
- **Empty-query semantics:** `query.trim().is_empty()` ⇒ return all
  live accounts (ordered by `last_modified_at DESC`, capped at the same
  limit) — matches 1.2's placeholder behaviour and the search-as-you-
  type "show everything before I type" UX. (Alternative: error on
  empty query — rejected; the placeholder already returns-all and the
  FFI shape is frozen around `Vec<…>`.)
- **Query parsing:** sanitise the user string into an FTS5 query — split
  on whitespace, strip FTS5 syntax metacharacters (`"` `*` `(` `)` `:`
  `^` `-` etc.) or quote each token, then AND/OR the tokens (default
  AND for "all words" / OR for "any word" — surface as Open Q3) and, if
  the tokenizer is `unicode61`, append `*` for prefix matching. With
  `trigram` the tokens are matched as substrings natively. **Never pass
  raw user input to FTS5 unescaped** (a stray `"` is a syntax error,
  not a security hole, but it's a bad UX bug — and FTS5 has no
  injection-into-other-tables vector since `MATCH` only reads the FTS5
  table).
- **Sync — three call sites, all inside the existing transaction:**
  - `Vault::account_add` (and the V0 `add_account` shim): after the
    `revisions` INSERT, compute the projection from the just-built
    `AccountIdentity` (or, for the V0 shim, from the `AccountSnapshot`'s
    `display_name`/`url` — no tags in V0), `INSERT INTO
    account_search_src(...) VALUES(...)` + `INSERT INTO
    account_search_fts(rowid, display_name, tags, hostnames)
    VALUES(...)`.
  - `Vault::account_update` (and the V0 `update_account` shim): the
    head identity is already decrypted (`read_identity_at`); after the
    new `revisions` INSERT, recompute the projection and `INSERT OR
    REPLACE` the `account_search_src` row + the FTS5 `'delete'`-then-
    `INSERT` dance (external-content FTS5 deletes need the *old* column
    values; use the `account_search_src` row read just before the
    replace, or use FTS5's `'delete-all'`+rebuild for that one rowid).
  - `Vault::delete_account` (the tombstone path, both V0 and any future
    V1 tombstone): `DELETE FROM account_search_src WHERE account_id=?`
    + FTS5 `'delete'`. (Tombstoned accounts must not appear in search —
    1.2's `account_search` already filters `tombstoned = 0`; removing
    the FTS5 row makes the whitelist structural for deleted accounts
    too.)
  - `Vault::ingest_chain_revision` is MVP-1-out-of-scope chain code but
    it exists in the PoC; if it's reachable in a CLI test path, the
    plan adds the same projection-resync after its INSERT (or, simpler,
    marks the FTS5 row stale and lets the next `open` rebuild — surface
    as a minor Open Q if the builder finds the call path live).
- **NFC:** the projection is built from the *already-NFC-normalised*
  `AccountIdentity` fields (1.2's validator did this), so the FTS5
  index sees the NFC form. The `unicode61` tokenizer does its own case-
  folding + diacritic-stripping; `trigram` is raw codepoints, so for
  `trigram` we lowercase the projection strings before indexing (and
  lowercase the query) to keep case-insensitivity. Document whichever
  the tokenizer choice (Open Q1) implies.

## Failure modes considered

- **`bundled` SQLite lacks FTS5.** Mitigation: the
  `fts5_available_on_bundled_sqlite` test fails loudly at the start of
  the build pass; fix is `features = ["bundled", "modern_sqlite"]` in
  the workspace `rusqlite` dep — no new crate, no API change. (rusqlite
  `0.39.0`'s `bundled` does ship FTS5; this is belt-and-suspenders.)
- **FTS5 index desyncs from the blob table** (sync write outside the
  transaction, or a bug in the `'delete'`-then-`INSERT` external-content
  dance). Mitigation: every sync write is inside the same
  `unchecked_transaction()` as the revision INSERT (atomic); a
  `dropped_fts_index_rebuilds_on_open` + `interrupted_update_does_not_desync_fts`
  pair tests both directions; the audit pass reviews the three call
  sites for transaction boundaries.
- **Search leaks `usernames`/`passwords`/`notes`** (someone adds a
  column to the FTS5 table later). Mitigation: the whitelist is
  *structural* — those fields are never written to `account_search_src`
  or `account_search_fts`; `search_never_matches_username_password_notes`
  is a regression test; the projection builder in `search.rs` has a
  module-doc comment forbidding additions (mirrors `identity_matches_search`'s
  existing comment and the P4 H-1 `compile_fail` doctest pattern).
- **Plaintext-projection-on-disk vs. "no plaintext at rest"** — the
  central tension. Mitigation: master-plan §4 row 1.3 classifies
  name/tags/hostnames as *non-secret*; the plan documents that
  classification at the table-creation site and updates the
  `no_plaintext_on_disk` proptest's rationale comment; if Kelvin
  rejects even that leak (Open Q2), the `:memory:` design is the
  drop-in alternative with no on-disk change.
- **10k/<50ms not met** (e.g. the per-result blob-decrypt dominates and
  a popular query matches thousands of accounts). Mitigation: the
  `ACCOUNT_SEARCH_RESULT_CAP = 200` `LIMIT` bounds the decrypt cost;
  `ORDER BY bm25(...)` returns the best matches first within the cap;
  the bench measures the worst realistic case (a common term). If it
  still blows the budget, fall back to returning *ids only* from
  `account_search` and let the caller `account_get` lazily — but that's
  an FFI shape change (`Vec<AccountSnapshot>` is frozen), so it would be
  a §16 escalation, not a silent drift.
- **Unicode tokenizer surprises** (`unicode61`'s default
  diacritic-folding vs. `trigram`'s raw-codepoint matching produce
  different "fuzzy" behaviour; CJK/emoji edge cases). Mitigation: the
  tokenizer choice is Open Q1 (recommended `trigram` for true
  substring/fuzzy, falling back to `unicode61` if `trigram`'s 3-char
  minimum is too coarse for short queries); `search_nfc_equivalence` +
  `search_is_case_insensitive` pin the agreed behaviour; the
  `fts_schema_version` slot (§7) lets 1.6 change the tokenizer later
  with a rebuild.
- **CI Linux umask quirk #1.** No new file is created by the store in
  either design (FTS5 shadow tables are in-DB; the `:memory:` fallback
  is in-RAM). Tests that write fixture `.pvf` files reuse the existing
  `UmaskGuard` / `nix::umask(0o022)` pattern (quirk #1, precedent commit
  `0b81edf`).
- **PowerShell stdout/UTF-8 quirks #5/#6.** No new CLI smoke tests
  planned for 1.3. The `no_plaintext_on_disk` proptest already writes
  raw bytes via `stdout().write_all` in some paths — unchanged; the
  release-script preflight already redirects to a file (quirk #6).
- **`ingest_chain_revision` (chain code, MVP-1-out-of-scope but extant
  in the PoC) writes a revision without resyncing FTS5.** Mitigation:
  if the build agent finds that call path reachable from a CLI test,
  add the projection-resync after its INSERT, or mark the row stale for
  the next `open` rebuild — flagged as a minor open item for the builder.

## Open questions for Kelvin

### Q1 — FTS5 tokenizer: `trigram`, `unicode61`, or `unicode61` + prefix queries?

**Background.** §4 row 1.3 says "fast **fuzzy** search". FTS5 ships:
`unicode61` (word tokenizer, Unicode-aware, case-folds + strips
diacritics by default) — gives whole-word + (with a trailing `*` on the
query) prefix matching, but not mid-word substring; `trigram` — indexes
overlapping 3-character grams, gives true substring / "contains"
matching (the closest thing FTS5 has to "fuzzy"), at the cost of a
larger index and a 3-char minimum useful query length; `porter`/`ascii`
— stemming/ASCII-only, weaker for our needs. (FTS5 has no built-in
edit-distance/Levenshtein.)

**Options:** (1) **`trigram`** — true substring match, "github" finds
"my-github-acct", "ithu" finds it too; index ~3× the projection size
(negligible for 10k short strings); short queries (<3 chars) fall back
to a `LIKE` scan over `account_search_src`. (2) **`unicode61` + prefix
(`token*`)** — smaller index, prefix-from-word-start matching only
("git" finds "github" but "hub" doesn't find "github"); cheaper, less
"fuzzy". (3) **`unicode61` + `trigram` on a second indexed copy** —
both behaviours; double the index. (4) Defer: ship `unicode61`+prefix
now, add `trigram` in a later additive issue (the `fts_schema_version`
slot supports the migration).

**Recommendation: option 1 (`trigram`).** It's what "fuzzy matching"
(the feature-list wording) and "search that doesn't suck" actually mean
to users; the index-size cost is trivial at our scale; the <3-char
`LIKE`-fallback is a few lines. Surfaced because it's a UX-defining
choice and changing it later forces a rebuild.

### Q2 — The secret/non-secret boundary: persistent plaintext FTS5 projection, or `:memory:` rebuild-on-unlock?

**Background — THE load-bearing question.** Whitepaper §C1's properties
are "Fast search" **and** "No plaintext secrets at rest." Master-plan §4
row 1.3 settles *which fields* are searchable — "**FTS5 on non-secret
fields only — name, tags, hostnames; never on usernames/passwords**" —
i.e. it classifies `display_name`, `tags`, and URL `hostnames` as
*non-secret* (the same metadata-leakage class a stolen `.pvf` already
exposes: account count, revision graph, timestamps, device ids — P2
threat-model row #1). But 1.2 wraps `display_name`/`tags`/`urls` in
`SecretBytes` ("encrypted at rest") — so a *persistent* FTS5 index
necessarily puts a plaintext copy of those three derived fields in the
`.pvf` file, which a literal reading of §C1's "no plaintext at rest"
forbids. **The §4-row-1.3 wording resolves it in favour of "those three
are non-secret; a plaintext index is the intended design" — but it's
worth Kelvin confirming, because it's a knowing relaxation of the
cardinal-principle-2 phrasing and it interacts with the existing
`no_plaintext_on_disk` proptest (see Q5).**

**Options:** (1) **(Recommended) Persistent plaintext FTS5 +
`account_search_src` shadow table**, as designed above; keep
`SecretBytes` on the model fields (for in-memory zeroize + encrypted
blob payload), accept the plaintext projection in the DB file, document
the §4-row-1.3 classification at the table-creation site, and update
the `no_plaintext_on_disk` proptest to exclude the three indexed
fields with that rationale. (2) **`:memory:` FTS5 rebuilt on every
`unlock`, dropped on `lock`** — nothing extra on disk; satisfies the
strict "no plaintext at rest" reading; cost is unlock-time rebuild
(dominated by the AEAD decrypts `unlock` already does — a few extra ms
for 10k) + the complexity of a second SQLite connection; `account_search`
queries that connection. (3) **Reclassify `display_name`/`tags`/`urls`
as non-secret in `AccountIdentity` itself** (drop the `SecretBytes`
wrapper on those three) — cleanest end-state if §C1 truly means "those
aren't secrets", but it's a 1.2-model change (touches `account.rs`,
`blob.rs` CBOR, `ct_eq`, the FFI summary) — bigger diff, and arguably
1.4/1.6's call, not 1.3's.

**Recommendation: option 1**, with option 2 as the documented fallback
if Kelvin wants the stricter posture. Option 3 is out of 1.3's scope
regardless.

### Q3 — `account_search` ranking, multi-term semantics, result cap?

**Background.** FTS5 gives `bm25()` relevance ranking for free. Open
sub-choices: (a) `ORDER BY bm25()` alone, or `bm25()` with
`last_modified_at DESC` as a recency tiebreaker? (b) Multi-word query
"github work" — AND (both terms, narrower) or OR (any term, broader)?
(c) Result cap — 200 (proposed), 100, 50, or unbounded?

**Options / recommendation:** (a) **`bm25()` then `last_modified_at
DESC` tiebreaker** — relevance first, recency for ties; cheap. (b)
**Default AND** ("all words") — matches user expectation for search-
boxes; an explicit OR mode is a later additive feature. (c) **Cap at
200** — generous for any human; bounds the per-result blob-decrypt cost;
exposed as `ACCOUNT_SEARCH_RESULT_CAP`. Empty query ⇒ all live accounts,
`last_modified_at DESC`, same cap. None of these touch the frozen FFI
shape.

### Q4 — The 10k/<50ms gate: hard `cargo test` assertion or `cargo bench` target?

**Background.** §4's exit criterion is "search 10k accounts in <50ms".
A hard `#[test]` assertion on wall-clock is CI-flaky (a loaded GitHub
runner can blow 50 ms on anything). Options: (1) **`cargo bench`
(Criterion) with a documented target** (median < 25 ms / p99 < 40 ms)
that the CI `bench` lane runs and *reports* (and optionally fails on a
generous regression threshold), plus an `#[ignore]`-by-default
`--release` smoke test (`< 40 ms` over 10k) the same lane runs
explicitly. (2) **A hard `#[test]` assertion** at a scaled-down,
generous bound (e.g. `< 40 ms` over a 2k-account vault) in the normal
test run. (3) Both — the scaled-down hard test in CI + the full 10k
Criterion bench for tracking.

**Recommendation: option 1** (bench-authoritative + `#[ignore]` 10k
release smoke). It honours the exit criterion (the bench *is* the
"search 10k accounts in <50ms" check) without making CI flap on every
loaded runner. Surface because the test-vs-bench choice affects what
`cargo test --workspace` does on every PR.

### Q5 — Updating the `no_plaintext_on_disk` proptest: narrow it, or leave it (with the `:memory:` design)?

**Background.** The existing `no_plaintext_on_disk` proptest creates a
vault, writes accounts with marker strings, locks, and asserts the
markers appear **zero times** in the raw `.pvf` bytes. With the
persistent-plaintext-projection design (Q2 option 1), `display_name` /
canonical `tags` / `hostnames` markers *will* now appear in the file
(by design — they're in `account_search_src` / the FTS5 index). So the
test must either (a) be narrowed — exclude those three fields from the
"must not appear" set, with a `// per master plan §4 row 1.3:
name/tags/hostnames are non-secret metadata, intentionally in the FTS5
index` comment, while still asserting `usernames` / `passwords` /
`notes` / `totp_secret` markers are absent — or (b) stay unchanged if
Q2 lands on the `:memory:` design (nothing extra hits disk).

**Recommendation: (a) narrow it**, tied to Q2 option 1. This is the
*only* place the plaintext-projection relaxation is observable in the
test suite, so the rationale comment there is load-bearing
documentation. Flagged explicitly because "we deliberately weakened a
cardinal-principle-2 test" is exactly the kind of change that needs an
on-the-record sign-off, not a silent diff.
