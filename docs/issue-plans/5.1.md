## Open questions for Kelvin (LOAD-BEARING — surface at TOP per §16)

### Q-a · Window duration — hardcoded 30s, env-var override, or vault-stored?

Master plan §5 row 5.1 says "30s window" verbatim. Three knob shapes:

**Option A: Hardcoded `pub const BATCH_WINDOW_SECS: u64 = 30;`.** No override surface. Pro: simplest; spec-compliant verbatim; one path to test. Con: tests that want to exercise the window-elapsed path have to wait 30s wall-clock (or mock the clock — which the codebase already does via the `clock: Box<dyn Clock>` field on `Vault`, so the cost is small).

**Option B: Const default + env-var override (`PANGOLIN_BATCH_WINDOW_SECS`, clamped 1..=300).** Mirrors 4.2 R-c (`PANGOLIN_INDEXER_IDLE_TIMEOUT_SECS`) + 3.4/3.5 env-var precedent. Pro: test surface drives short windows. Con: another env-var; clamp bounds need to be defended in a test.

**Option C: Const default + env-var + per-vault stored override (`meta.batch_window_secs`).** Three layers. Per-vault tuning if a power user wants a longer window. Pro: maximum flexibility. Con: triples the test surface for a UX heuristic; per-vault override is over-engineering for a coalescing window.

**Plan-gate recommends:** Option B (const + env-var clamped). Matches the 4.2 R-c / 4.4 R-a precedent verbatim. Per-vault (Option C) is unjustified — 5.4's "always-on auto-flush" knob will live elsewhere.

**Kelvin's call.**

### Q-b · Drain triggers — which subset?

**Minimum set (mandatory per L1):**
1. **Window elapsed** — 30s has passed since the FIRST dirty marker stamped in the current window
2. **Manual flush** — host invokes `Vault::flush_publish_queue()` directly (host UI button / CLI subcommand)
3. **Session teardown — `lock()`**
4. **Session teardown — idle-expire**
5. **Session teardown — 4h-absolute**
6. **Session teardown — `device_locked()`**

**Optional caps (Q-b proper):**

7. **Count cap (e.g., 100 dirty markers)** — forces flush regardless of window. Defends against unbounded queue growth if a script auto-creates 10k accounts.
8. **Byte cap (e.g., 1 MB total `enc_payload` across all queued revisions)** — chain-tx-size + memory pressure defense.
9. **App-shutdown** — host signals shutdown intent (CLI exits, Tauri window closes). Plan-gate weakly recommends NO — `lock()` already covers graceful shutdown; ungraceful shutdown (kill -9) drops the queue same as before 5.1 because the dirty markers are SQLite-persisted (P8-2) and survive crash unaltered.

**Plan-gate recommends:** triggers 1-6 (mandatory) + 7 (count cap = 100) + 8 (byte cap = 1 MB total). Skip 9 (app-shutdown) — the dirty markers are already SQLite-persisted; no in-memory queue state to lose.

**Kelvin's call.**

### Q-c · Coalescing scope — per-account (master plan verbatim) vs cross-account

**LOAD-BEARING question. Cross-account batching is INCOMPATIBLE with the deployed `RevisionLogV1` contract** — `RevisionPublished` carries ONE `bytes32 accountId` per event. There is no way to fit N accounts into one chain event without re-deploying the contract (out of scope; D-017 is locked).

Master plan §5 row 5.1 verbatim reads: "Coalesce **multiple edits to same account** within 30s window into a single revision." The honest interpretation:

**Option A (master-plan verbatim): Per-account coalescing.** If the user edits account X's password 5 times in 20s, the local revision graph still records all 5 revisions (1.6 append discipline preserved), but only the latest revision per account is broadcast — the dirty markers for revisions #1-#4 are pruned at flush time. N accounts edited in the window = N chain txs (one per account, all submitted in the same flush pass). Pro: spec-compliant; no contract change; smallest implementation; preserves per-edit local lineage for 5.3 conflict detection. Con: gas cost is N × per-tx-cost, not 1.

**Option B (cross-account composite — REJECTED): Multi-account batched composite payload.** Requires either (a) a contract redeploy with a multi-account event (MVP-3 territory) OR (b) `Multicall3`-style wrapper (not in D-017). Plan-gate REJECTS this option — D-017 is locked.

**Option C: Same as A, but with a single signed-revision payload that batches multiple operations FOR THE SAME ACCOUNT.** E.g., V3 schema_version where `enc_payload` is CBOR `Array<EditOp>` instead of a single `AccountIdentity`. Saves nothing for the common case (one user one device editing one account 5x in 20s = one final `AccountIdentity` head, not 5 ops to replay). Intermediate revisions are already coalesced naturally by Option A (drop dirty markers; the latest head represents the cumulative state).

**Plan-gate recommends:** Option A — per-account coalescing, no V3 payload-version bump, no contract change. This matches master-plan-verbatim, matches the deployed contract, and the implementation is small (one SQL pruning pass before flush).

**Kelvin's call.** If Kelvin signs A, the L-section collapses substantially — the queue holds DIRTY MARKERS (not signed txs), and signing happens at flush time (matches existing P8-3 architecture).

### Q-d · Queue persistence — in-memory window state vs new persisted table

Per Q-c Option A, the queue IS the dirty_accounts table — already persisted by P8-2. What 5.1 ADDS:

**Option A: No new persistence. Window state is in-memory only.** The `ActiveState` gains a `window_started_at: Option<Instant>` field that ticks from the first edit in a fresh window. On lock/idle-expire/teardown the window state evaporates with the rest of `ActiveState`; the dirty_accounts rows persist unaltered. Next unlock starts a fresh window IF there are dirty markers. The host can call `flush_publish_queue` at any time to force-drain.

**Option B: Persist the window-started-at to a new `meta.batch_window_started_at` column.** Survives lock/crash. Pro: continuity. Con: trivial UX value (the user just unlocked; they're about to edit more; the new window starts naturally).

**Option C: Persist the FULL queue state to a new `publish_queue` table (including pre-signed revisions, status flags).** Plan-gate recommends NO — this is duplicate state vs `dirty_accounts`. The signed-revision-on-disk model assumes signing happens at edit time; in this codebase signing happens at flush time, so the queue holds nothing that requires AEAD sealing.

**Plan-gate recommends:** Option A (in-memory window state, on-disk dirty markers preserved). Smallest change; correctly leverages the existing P8-2 machinery.

**Sub-question Q-d':** on `lock()`, should we attempt a flush BEFORE dropping ActiveState, or just leave the dirty markers for the next unlock? Plan-gate recommends "attempt flush before teardown; failures (network, balance) leave dirty markers intact; teardown completes regardless of flush success" — matches L1.

**Kelvin's call.**

### Q-e · Pre-flush balance gate behavior

3.5 ships `pre_publish_balance_gate` inside `publish_revision_v1_with_config`. When balance is below the `MIN_BUFFER_REVISIONS × estimate_next_publish_cost` floor, 3.3 returns `ChainError::PrePublishBalanceInsufficient`. The question: where in 5.1's flush flow does the gate fire?

**Option A: Block flush at the queue layer.** Before any per-revision chain submit, `flush_publish_queue` calls `compute_balance_state` (3.5); if NOT `Sufficient`, return `BatchFlushError::BalanceInsufficient` immediately without attempting any chain submits. Dirty markers stay; next attempt re-checks. Pro: fail-fast; no half-batched state.

**Option B: Let 3.3's pre_publish_balance_gate fire per-revision.** Each `publish_one` call inside the flush loop runs its own gate; first failure surfaces, remaining dirty markers stay. Pro: respects 3.3's existing posture. Con: in a multi-account batch, account #1 might publish (decrementing the wallet balance) and account #2 might fail — partial batch.

**Option C: Combine — top-of-flush single balance check + early bail.** Compute total estimated cost = N × `estimate_next_publish_cost`; if balance < total, bail before any submit. Per-revision gate still runs as defense-in-depth. Pro: prevents partial batches.

**Plan-gate recommends:** Option C. Matches 3.5's "Sufficient" semantic at the batch level; defends partial-batch surfacing as `BatchFlushError::BalanceInsufficientForBatch { needed, available, queued_count }` distinct from 3.3's per-tx `PrePublishBalanceInsufficient`.

**Kelvin's call.**

### Q-f · Blocked-queue append semantics

When the queue is blocked on balance (Q-e Option C surfaces `BalanceInsufficientForBatch`), the user might keep editing.

**Option A: Append normally; coalesce on retry.** Master-plan "30s window" semantically just gets longer — the next flush attempt runs the same per-account coalescing across all dirty markers. Matches Q-c Option A naturally.

**Option B: Refuse new edits with `StoreError::QueueBlocked`.** User must resolve the balance situation first. Pro: clear UX. Con: paradoxical — the local edit IS local, why refuse it?

**Option C: Open a new queue; first one ships first on retry.** Doubled state.

**Plan-gate recommends:** Option A. The 30s window doctrine is timer-based; balance-block extends the effective window until retry. No new state needed.

**Kelvin's call.**

## Adversarial framing — L-section (load-bearing risks)

> Lighter section than the initial framing suggested, because Q-c Option A (per-account coalescing without payload changes) eliminates several risks the initial framing introduced. The remaining surface:

### L-tombstone-coalesced-away

**What goes wrong:** user edits account X (revision A), then deletes account X (revision B, tombstone) within the 30s window. The coalescing rule "keep the latest revision per account" SHOULD keep revision B and drop revision A's marker. A bug here that picks A over B loses the delete intent locally — the chain never receives the tombstone, and on the next pull the account "reappears" (P10 reconciliation will eventually re-tombstone if another device deleted, but if THIS device was the deleter, the deletion silently fails).

**Mitigation:** explicit ordering rule "latest `marked_at` wins; ties broken by `is_tombstone DESC`" — tombstones beat non-tombstones at identical timestamps. Dedicated test `tombstone_always_wins_over_prior_update` + `coalescing_preserves_delete_intent_under_clock_skew`.

### L-flush-during-lock-race

**What goes wrong:** user clicks "lock" while a flush is in-flight on the chain (RPC has the request; receipt not yet returned). `lock()` synchronously drops `ActiveState`, including `evm_wallet`. The in-flight flush's async future still holds a borrow of the wallet — Rust's borrow checker prevents the synchronous drop while the borrow is live, so the lock blocks until the flush returns. OR — if 5.1 implements drain-on-lock as "fire-and-forget", the future could outlive the session and re-enter when the session is gone.

**Mitigation:** flush takes `&mut Vault` (or `&Vault` + interior mutability via a flush-only mutex); the session-teardown path AWAITS the in-flight flush before dropping `ActiveState`. Dedicated test `concurrent_lock_during_flush_completes_cleanly`. Alternative: if drain-on-lock is best-effort, the in-flight flush continues but its results are stored only in the dirty-marker layer (already SQLite-persisted; survives teardown).

### L-window-DoS

**What goes wrong:** a malicious host-layer wrapper sets `PANGOLIN_BATCH_WINDOW_SECS=u64::MAX` (or doesn't call `flush_publish_queue` ever), making the user think their edits landed when they're sitting locally as dirty markers.

**Mitigation:** env-var clamp (Q-a Option B: 1..=300) plus mandatory drain on every session-teardown path (L1). The host cannot suppress the lock-time drain without crashing the engine. Dedicated test `env_var_window_clamps_to_max_300_secs`.

### L-balance-blocked-grows-unbounded

**What goes wrong:** per Q-f Option A, new edits append to a blocked queue. If balance is never restored, dirty markers grow forever (one per edit; the local `revisions` table also grows; no DoS at the chain layer because nothing is sent).

**Mitigation:** the count cap (Q-b Option 7 = 100) + byte cap (Option 8 = 1 MB) clamp queue size. When either cap fires while balance is still blocked, `flush_publish_queue` returns `BatchFlushError::BalanceInsufficientForBatch` immediately and the LOCAL edits still succeed (the user's vault is fine; only chain submission is blocked).

### L-malicious-RPC-fakes-receipt

**What goes wrong:** a flush submits revision R; the RPC returns a fake receipt for a wrong tx hash. 3.3's `publish_revision_v1` already has the load-bearing `tx_hash` cross-check; 5.1 inherits this defense without modification.

**Mitigation:** 3.3 L-receipt-mismatch defense; 5.1 adds no new attack surface here.

### L-coalescing-skips-foreign-edit

**What goes wrong:** account X's local lineage has a chain-frozen revision from device D2 (P8 fix CRIT-1 `refuse_if_frozen` already prevents local edits on a frozen account). 5.1's coalescing must respect this — a frozen account should NOT have any dirty markers at all (because edits are refused at the `account_update` layer before they can stamp a dirty marker).

**Mitigation:** inherited from P8 CRIT-1; verified by dedicated test `coalesce_skips_frozen_accounts_trivially_because_no_dirty_markers_exist`.

### L-clock-skew-coalesce-wrong-order

**What goes wrong:** `dirty_accounts.marked_at` uses `current_unix_ms()` (host-clock). On a host with backwards-jumping clock (NTP correction), a later revision could have an earlier `marked_at` than an earlier revision. Coalescing-by-marked_at could then pick the wrong "head."

**Mitigation:** coalescing rule resolves ties + ordering via the `revisions.parent_revision_id` chain — the head per account is `account_identities.head_revision_id` (which is updated atomically inside `account_update` / `delete_account`), NOT `MAX(marked_at)`. So the head selection is clock-independent. Dedicated test `coalesce_under_backwards_clock_skew_picks_head_from_account_identities_table`.

## Threat-model touch points

`THREAT_MODEL.md` gains a NEW "Publish queue + batching (5.1)" row covering:

1. **Adversary forces tombstone to be coalesced away.** L-tombstone-coalesced-away. Mitigated by L10.
2. **Adversary races lock against in-flight flush.** L-flush-during-lock-race. Mitigated by await-before-drop.
3. **Adversary DoSes window via env-var.** L-window-DoS. Mitigated by env-var clamp + mandatory teardown drain.
4. **Adversary keeps user balance below threshold to grow local dirty-marker queue.** L-balance-blocked-grows-unbounded. Mitigated by count + byte caps.
5. **Backwards-jumping clock corrupts coalescing order.** L-clock-skew-coalesce-wrong-order. Mitigated by reading head from `account_identities.head_revision_id` (atomic), not `MAX(marked_at)`.

**Invariant additions:** none new.
