# Issue 1.10: Encrypted export

> One-line scope: a Pangolin-native, self-contained **encrypted vault
> archive** (`vault_export_encrypted`) — a portable, AEAD-sealed snapshot
> of the vault you can move between devices / keep off-site — plus the
> spec-mandated, double-confirmed, delayed, loud **plaintext export**
> branch (`vault_export_plaintext`); both presence-gated (Session spec
> §5.4), both shipped with a `pangolin-cli vault export` subcommand.
>
> **Status:** PLAN drafted 2026-05-12; Kelvin's plan-gate sign-off
> received 2026-05-12 — all open questions resolved (see "Resolved
> decisions" below). The "Decisions locked" table is now the **single
> source of truth**; if anything in the "Approach" body drifts from it,
> the table wins.
>
> **Security-critical: YES (strong).** 1.10 (a) materialises *every*
> secret in the vault — every head password, the full password history,
> notes, TOTP seeds — into a single portable artifact; (b) introduces a
> **new at-rest crypto format** (the archive: magic + format-version +
> Argon2id params + salt + nonce + AEAD-sealed CBOR payload); (c) adds a
> new **presence-gated FFI operation** that emits plaintext-bearing data
> across the boundary; and (d) ships a CLI path that writes that artifact
> to disk — including, by spec, an *intentional cleartext* form. That is
> the §16.3 profile (same footing as 1.4 and 1.9): Kelvin's approval on
> the plan (**received 2026-05-12**), then plan-gate → builder → test →
> fix-pass-as-needed → adversarial audit → fix-pass-as-needed → merge,
> Kelvin approving at plan and merge. Adversarial focus: the archive
> header parser (truncated / lying-length headers, hostile Argon2 params
> → OOM/DoS — same clamps lesson as 1.9; tampered ciphertext → AEAD auth
> failure, no oracle), the AEAD-AAD binding (the plaintext header MUST be
> the AAD so a header tweak fails the open), the
> wrong-passphrase-vs-tampered-archive no-oracle collapse (both → one
> error), the plaintext-export confirmation/delay/copy gate, the
> `no_plaintext_on_disk`-style guarantee that the *encrypted* archive
> file leaks nothing (extend the scan to the exported file), the
> secret-material discipline (the serialized snapshot only ever in
> `Zeroizing`, sealed before any disk write; the restore path zeroizes
> the decoded snapshot after writing the new vault), and the
> presence-proof wiring on both FFI entries.
>
> **Depends on:** issue 1.1 (the FFI surface froze
> `vault_export_encrypted(h: &VaultHandle, dest: &str) -> Result<(),
> FfiError>` and `vault_export_plaintext(h: &VaultHandle, dest: &str,
> second_confirmation: PlaintextExportConfirmation) -> Result<(),
> FfiError>` with `todo!()` bodies, plus the `PlaintextExportConfirmation
> { schema_version: u16, token: Vec<u8> }` Record and the `UnixTimestamp`
> alias, all in `pangolin-ffi/src/session.rs`); issue 1.2 (the production
> `AccountIdentity` model + the V1/V2 CBOR body + the `validate::*` field
> rules — the export payload serialises this); issue 1.4 (the
> session/presence-gate model — `PressYPresenceProof` / `PinIdentityProof`,
> `ensure_presence_fresh`, the `reveal_*` family, `RevealedSecret` as a
> zeroizing `uniffi::Object`, and `Vault::export_payload` which already
> routes a per-account sealed-blob export through `check_session_freshness`
> + `ensure_presence_fresh` + `touch_session`); issue 1.6 (the
> revision-lineage / `.pvf` schema — `revisions` table, `account_history`,
> `RevisionMeta`); issue 1.7 (`TotpParams` persisted on the account);
> issue 1.8 (the zxcvbn `strength()` API — D3 runs the export passphrase
> through it for a weak-backup-password warning). Current main tip
> `da28a51`.

## Approach

### 1. The archive format (`docs/architecture/encrypted-export.md` is canonical)

Byte layout of a Pangolin encrypted archive (extension `.pvea` — D7):

```
[magic: b"PANGOLIN-VEA\0" or similar, fixed length]
[format_version: u16, big-endian]                 ; = 1
[kdf_algo_id: u8]                                 ; = 1 (Argon2id) — future-proof slot
[kdf_memory_kib: u32][kdf_time_cost: u32][kdf_parallelism: u32]
[salt: 16 bytes]                                  ; KdfSalt::random()
[nonce: 24 bytes]                                 ; Nonce::random()
[ct_len: u64, big-endian]
[ciphertext: ct_len bytes]                        ; XChaCha20-Poly1305(payload, AAD = the header bytes above, canonically encoded)
```

The header (everything before the ciphertext) — a canonical
length-prefixed encoding — is the AEAD AAD, so any tamper with magic /
version / KDF params / salt / nonce / ct_len makes `open` fail
authentication. `ct_len` is also length-checked against the file size
with a generous ceiling (refuse > ~256 MiB → `TooLarge`, to bound a
hostile file). The plaintext payload is a CBOR document (`ciborium-ll`):

```
{
  "schema_version": u16,                  ; the export-payload schema
  "exported_at": i64 (unix seconds),      ; D6 — non-secret provenance, inside the ciphertext
  "source_device_id": [u8; 32],           ; D6
  "vault_id": [u8; 32],                   ; D6
  "accounts": [
    { ...the AccountIdentity fields — display_name, tags, usernames, urls,
      notes, password (bytes), totp_secret (bytes), totp_params,
      created_at, head_revision_id, ...
      + the full revision history (D1 — the historical password *bytes*):
        [{revision_id, parent, timestamp, device_id, body}, ...] },
    ...                                   ; whole vault by default; or just
                                          ; the --accounts selection (same format)
  ],
  "devices": [ ...the trust list — device IDs + labels (D1)... ],
  "meta": { idle_timeout, schema_version_markers, ... }   ; the vault meta settings (D1)
}
```

Snapshot scope is **D1**: a full move-to-new-device backup — current head
of every account + per-account password history (the historical password
*bytes*, with their change timestamps + originating-device IDs) + the
device trust list + the `meta` settings + `vault_id` (+ the D6 provenance
fingerprint). A `--accounts <selection>` flag (threaded through the export
call + the CLI) narrows that to the chosen accounts without changing the
archive format — useful for sharing a handful of logins.

**Format-versioning:** `format_version` u16 (container) + the payload
`schema_version` u16 are independent — same two-axis approach as the
vault's `format_version` vs the CBOR-body `schema_version`. Document the
"unknown future version → typed `UnsupportedFormatVersion`, never a
silent partial read" rule (mirror `docs/architecture/schema-versioning.md`).

### 2. `pangolin-store::export` (the new submodule)

- `header.rs` — `ExportHeader { magic, format_version, kdf_algo_id,
  kdf_params: KdfParams, salt: KdfSalt, nonce: Nonce, ct_len: u64 }`;
  `write_header(&mut Vec<u8>)`; `parse_header(&[u8]) -> Result<(ExportHeader,
  rest: &[u8])>` with strict bounds checks (truncated buffer, lying
  `ct_len`, magic mismatch, unknown `format_version`, unknown
  `kdf_algo_id`, Argon2 params outside a sane clamp →
  `ExportError::UnsupportedKdf` — reuse `pangolin_crypto::kdf`'s
  `KdfParams::validate` floor + an upper clamp ≈ memory ≤ 1 GiB / t ≤ 64).
- `payload.rs` — the CBOR snapshot codec: `encode_snapshot(&VaultSnapshot)
  -> Zeroizing<Vec<u8>>` (`ciborium-ll`), `decode_snapshot(&[u8]) ->
  Result<VaultSnapshot>`. `VaultSnapshot` is the in-memory shape gathered
  from the open vault.
- `mod.rs` — `Vault::export_encrypted(&mut self, passphrase: &SecretBytes,
  accounts: AccountSelection /* All | Subset(Vec<AccountId>) — D1 */,
  presence: &dyn PresenceProof) -> Result<Vec<u8>>`:
  1. `self.check_session_freshness()?;`
  2. `self.ensure_presence_fresh(presence)?;` (`touch_session()` at the
     end) — copy `export_payload`'s exact pre-amble.
  3. Gather `VaultSnapshot` from `self` (iterate `list_accounts` — filter
     to the `accounts` selection per D1 — read each head's sealed body +
     decrypt under the VDK, gather the full per-account history with the
     historical password bytes, the `devices` rows, the `meta` rows, plus
     `vault_id` / `source_device_id` / `exported_at` per D6) — everything
     in `Zeroizing`.
  4. `let plain = encode_snapshot(&snapshot);` (`Zeroizing<Vec<u8>>`).
  5. (Already done at the CLI/FFI layer per D3: run `passphrase` through
     1.8's zxcvbn `strength()` and emit a "weak backup password" warning
     — not a hard gate.) `let salt = KdfSalt::random(); let params =
     KdfParams::RECOMMENDED; let seed = kdf::derive_seed(passphrase,
     &salt, &params)?; let key = AeadKey::from_bytes(seed.into());`
     (match `Vault::create`'s VDK-wrapping adapter).
  6. `let nonce = Nonce::random();` build `header` with `ct_len` filled
     after seal; compute the canonical AAD bytes from the header;
     `let ct = key.seal(&nonce, &plain, &aad)?;` set `ct_len = ct.len()`;
     write `header || ct`; return it. `plain`, `seed`, `key` drop-zeroed.
  - `Vault::export_plaintext(&mut self, accounts: AccountSelection,
    presence, confirmation: PlaintextExportConfirmation) ->
    Result<Vec<u8>>`: same presence pre-amble; verify the confirmation
    token (single-use; D4 — the CLI/UI owns the 30 s delay + the second
    y/N prompt + the warning copy per master plan §4 row 1.10, the FFI
    just requires a non-empty structurally-valid `token` so a UI can't
    skip the gesture); gather the same `VaultSnapshot` (subject to the
    `accounts` selection); serialise as *plaintext* — a human-readable
    JSON-ish form (the `.pvtxt` form, D7) — with a prominent in-file
    "THIS FILE CONTAINS YOUR PASSWORDS IN CLEARTEXT" banner; return the
    bytes. No KDF, no AEAD, no zxcvbn check — that's the point, and why
    it's dangerous.
  - `Vault::decode_archive(bytes: &[u8], passphrase: &SecretBytes) ->
    Result<VaultSnapshot>` — parse the header, derive the key, AEAD-open
    with the header as AAD (wrong passphrase *and* tampered archive both →
    `ExportError::AuthFailed`, no oracle), CBOR-decode the snapshot. Used
    by the round-trip test, the `restore` path, and the FFI restore entry
    (D2).
  - `Vault::create_from_archive(snapshot: VaultSnapshot, dest: &Path,
    new_vault_password: &SecretBytes) -> Result<Vault>` — write a
    **brand-new `.pvf`** populated from the decoded snapshot (D2 —
    fresh-vault restore only; *no* merge into an existing vault). Zeroizes
    the decoded snapshot after the new vault is written (L10).
- `error.rs` — `ExportError` (`thiserror`): `Io`, `NotPangolinArchive`,
  `UnsupportedFormatVersion`, `UnsupportedKdf`, `Truncated`, `BadLength`,
  `AuthFailed` (AEAD-open failure — covers *both* a tampered archive AND a
  wrong passphrase, **no oracle** — one variant, like 1.9's
  `WrongCredentials`), `Cbor`, `PlaintextExportNotConfirmed`, `TooLarge`.
  Maps into `StoreError` → `pangolin_core::Error` → `FfiError` (a
  `Validation`-class arm with an `export_*` `kind` label); the exhaustive
  `error_taxonomy.rs` test covers the new arm. No variant ever echoes a
  passphrase / secret bytes.

> **Re-import (D2):** 1.10 ships the `decode_archive` decryptor (header
> parse → Argon2id → AEAD-open → CBOR-decode) AND a `restore` path that
> creates a brand-new `.pvf` from a decoded archive (`create_from_archive`
> + the `vault restore` CLI subcommand + the FFI restore entry). It does
> **not** implement merging/reconciling an archive into an *existing*
> vault — that's a sync/conflict-resolution problem deferred to MVP-2
> (Revision Log v1 with signature verification — the proper provenance
> substrate; doing it pre-MVP-2 would be the insecure-first version,
> grafting foreign revisions in on "it decrypted" alone), fallback home
> MVP-3 (recovery + guardians).

### 8. Failure modes considered

- **The archive header parser mishandles a hostile file (truncation,
  lying lengths, unknown version/algo) → panic / over-read / OOM.**
  Mitigation: `forbid(unsafe_code)` (no UB even on a hostile file); strict
  bounds checks + a `ct_len` ceiling + an Argon2-param clamp *before* any
  allocation/derive (test #4); typed errors not panics (test #3); the AEAD
  AAD covers the header so a tamper fails auth, not a mis-parse.
- **Wrong-passphrase / tampered-archive oracle.** Mitigation: both
  collapse to one `AuthFailed` variant (test #2); the Argon2id derive runs
  before the failure is known, so timing is dominated by the
  (intentionally slow) KDF — no useful side channel — the vault file's own
  posture.
- **The serialised plaintext snapshot lands on disk un-sealed (for the
  *encrypted* path), or the restore path leaves the decoded snapshot
  un-zeroed.** Mitigation: the snapshot exists only transiently in
  `Zeroizing` inside `export_encrypted`, sealed before anything is
  written; the on-disk archive is ciphertext + non-secret header; the
  `restore` path zeroizes the decoded `VaultSnapshot` after the new
  `.pvf` is written; test #5 scans the archive bytes for plaintext
  markers.
- **The plaintext-export branch is too easy to trigger / a UI skips the
  gesture.** Mitigation: presence proof (L2) + a structurally-valid
  single-use `PlaintextExportConfirmation.token` enforced FFI-side + the
  CLI's mandatory y/N + 30 s delay + warning copy (§16 / Design Spec §11)
  + the in-file banner; the FFI returns `PlaintextExportNotConfirmed` for
  a missing/invalid token (test #6).
- **A secret leaks into an error string / the CLI summary / a log line.**
  Mitigation: `ExportError` variants carry only category labels; the CLI
  prints counts + a path; a test asserts no error/Display ever echoes the
  passphrase or secret bytes.
- **HIGH-1 / uniffi-isolation regression** — the new CBOR codec
  accidentally pulling serde into `pangolin-crypto`, or `pangolin-store`
  growing a uniffi dep. Mitigation: the codec lives in `pangolin-store`
  (already uses `ciborium-ll`, already downstream of nothing uniffi); the
  `cargo tree` CI greps catch any drift.
- **The exported file is world-readable.** Mitigation: create it with the
  same umask-respecting restrictive-permission helper the vault file uses
  (env-quirk #1); a test asserts the mode (with a pinned umask).
- **Hostile file size on decode (a 10 GiB "archive").** Mitigation: the
  `ct_len` ceiling (~256 MiB) → `TooLarge` before reading; stream-read
  with a bounded buffer if practical.
- **PoC `.pvf` vault export.** Mitigation: the snapshot gather uses the
  same `list_accounts` / sealed-blob-read paths that already work on PoC
  vaults; tested in #11.

## Resolved decisions

> Kelvin's plan-gate rulings on the genuine judgment calls, received
> 2026-05-12. Each decision is also a row in the binding "Decisions
> locked" table (D1–D7) — this section keeps the rationale on the record.
> There are no remaining open questions.

### R-a — "Encrypted export" = a Pangolin-native portable encrypted archive; the plaintext branch's guards live in the CLI/UI; ship both (was Q-a → L1 / D4)

"Encrypted export" was confirmed to mean **a Pangolin-native,
self-contained encrypted archive** — *not* a `.kdbx`/CSV/1Password/
Bitwarden writer (1.9 explicitly deferred format-export; the spec text
doesn't ask for it). Interop-export to other managers (the mirror of
1.9's import) is a separate later issue; 1.10's job per the spec is the
*vault backup / move-to-a-new-device* artifact. For the plaintext branch
(§4 row 1.10: "guarded behind double-confirmation + 30 s delay + warning
copy", Design Spec §11): **the CLI/UI owns the 30 s delay + the typed
confirmation + the second y/N + the warning copy; the FFI just requires a
structurally-valid, single-use `PlaintextExportConfirmation.token`** — the
delay/copy are UX concerns, the FFI is the wrong layer for a
`sleep(30 s)`. And **both** the encrypted *and* the plaintext branch ship
in 1.10 (the 1.1 surface declared both `vault_export_encrypted` +
`vault_export_plaintext`; shipping both is the natural reading).

### R-b — Snapshot scope = full move-to-new-device backup + `--accounts` subset filter (was Q-b → D1)

The encrypted-archive payload is a **full backup**: the current head of
every account + per-account password **history** (the historical password
*bytes*, with their change timestamps + originating-device IDs — so the
destination can reconstruct the full lineage; the archive is encrypted
anyway) + the device trust list (device IDs + labels) + the vault `meta`
settings (session idle timeout, schema-version markers, etc.) + `vault_id`.
**Plus** a `--accounts <selection>` option threaded through the export
call + the CLI so the user can export only selected accounts instead of
the whole vault — *same archive format either way*, the subset flag just
narrows what's included. Rationale: a full backup is the useful artifact
and the cleanest input to the restore path; the subset capability serves
"share a handful of logins".

### R-c — Re-import = decode + create-fresh-vault-from-archive; merge-into-an-existing-vault DEFERRED to MVP-2 (fallback MVP-3) (was Q-c → D2)

1.10 ships the **archive decoder** (header parse → Argon2id → AEAD-open →
CBOR-decode of the snapshot) AND a **`restore` path that creates a
brand-new `.pvf`** from a decoded archive. It does **not** implement
merging/reconciling an archive into an *existing* vault — that's a
sync/conflict-resolution problem that belongs with **MVP-2** ("on-chain
sync goes production; Revision Log v1 with signature verification" — the
proper reconciliation + cryptographic-provenance substrate; 1.6 only
built the *local* fork/resolve primitives), with **MVP-3** ("recovery +
guardians") as the alternative home. Rationale to keep on the record:
doing archive-merge in MVP-1 would (a) duplicate MVP-2's reconciliation
engine and create two divergent paths; (b) be the *insecure-first*
version — pre-MVP-2 there's no signed Revision Log, so it would graft
foreign revisions into a live vault on "it decrypted under the passphrase"
alone, with no provenance check — exactly what the threat model wants
signature-gated. So 1.10 = encrypted backup/portability artifact +
fresh-vault restore; merge is its own future issue.

### R-d — Archive key = fresh user-supplied passphrase + Argon2id (`RECOMMENDED`, random salt in the header), independent of the vault master password, with a zxcvbn warning (was Q-d → D3)

The export prompts for a **NEW passphrase** (not the vault password);
Argon2id (the same `KdfParams::RECOMMENDED` the vault uses) over
(passphrase, random 16-byte salt — only `fill_random` for the salt) → the
archive key; salt + KDF params live in the archive's plaintext header
(bound as AEAD AAD). The export passphrase is run through 1.8's zxcvbn
`strength()` and a "this backup password is weak" warning is surfaced if
it's weak — **not a hard gate**, but it's the only thing protecting a
portable copy of every secret, so a nudge is warranted; the
plaintext-export branch skips this (no passphrase). Rationale: **key
separation** — a leaked archive passphrase never compromises the vault and
a leaked vault password never compromises archives; you can escrow/hand
off the archive with a distinct secret (fits MVP-3 guardian recovery,
MVP-4 GUI backup flow); it's the standard pattern (KeePass exports,
`age`). Deriving from the VDK would couple the archive to the vault
password (a leaked vault password decrypts every archive) — **rejected**.
A raw 256-bit key the user must store — operationally unacceptable for a
consumer password manager — **rejected**.

### R-e — File extension / name = `.pvea` (encrypted) / `.pvtxt` (plaintext) (was Q-e → D7)

The encrypted archive is **`.pvea`** ("Pangolin Vault Encrypted Archive",
parallels `.pvf`); the plaintext export gets an unmistakably different
name — **`.pvtxt`** (a `.pvf`-plaintext dump). A bikeshed; locked. The
CLI suggests/documents these but doesn't enforce the extension on the
user-supplied out-path.

### R-f — CLI subcommands ship in 1.10 (was Q-f → D4)

`pangolin-cli vault export <out> --vault-path <…> [--accounts <…>]
[--plaintext]` **and** `pangolin-cli vault restore <archive> --out
<new.pvf>` both ship in 1.10. Rationale: 1.9 set the precedent shipping
`pangolin-cli import`; the MVP-1 exit criterion names "encrypted-export"
as a user flow and the CLI is MVP-1's only shell, so it must be demoable
there; `export` is already named in the `vault` noun's doc-comment.

### R-g — Both export FFI ops are reveal-class / presence-gated (was Q-g → D5)

**Both** `vault_export_encrypted` and `vault_export_plaintext` are
reveal-class — routed through 1.4's session-freshness + presence-freshness
+ touch-session machinery, exactly like the existing `reveal_*` /
`Vault::export_payload`. Not session-class — they materialise *every*
secret. Rationale: Session spec §5.4 lists "export vault" (not "export
plaintext") as a high-risk action requiring presence even mid-session, and
bulk-materialising every sealed secret into a portable artifact is exactly
the risk §5.4 is about.

### R-h — Provenance fingerprint goes inside the encrypted payload (was Q-h → D6)

The archive payload includes `source_device_id` (32 bytes) + `exported_at`
(unix-seconds timestamp) + `vault_id` (32 bytes) — non-secret provenance
metadata, *inside the ciphertext* (not the plaintext header, so a finder
can't tell which device/vault it came from). Rationale: costs nothing
(a few dozen bytes); helps the restore path and any future merge.

## Out of scope (later issues own these)

- **Writing `.kdbx`** (export to KeePass) — deferred from 1.9; still
  deferred; not on any Pangolin spec.
- **CSV / 1Password / Bitwarden / LastPass / Chrome export** — the mirror
  of 1.9's per-format importers; later issues, if ever.
- **Cloud / remote backup** (auto-upload the archive somewhere) — way out
  of MVP-1 (local-first, zero network).
- **Full merge-of-an-archive-into-an-existing-vault** with conflict /
  lineage reconciliation (D2) — deferred to **MVP-2** (Revision Log v1
  with signature verification — the proper reconciliation +
  cryptographic-provenance substrate), fallback home **MVP-3** (recovery
  + guardians). 1.10 ships only decode + `restore`-to-a-fresh-vault.
- **The social-recovery / guardians flow** (initiate / approve / finalize /
  VDK rewrap) — MVP-3 (master plan §6); the encrypted archive may *inform*
  it (a guardian-shareable artifact) but 1.10 doesn't build it.
- **GUI export wizard** (pick-which-accounts, preview, drag-out) — MVP-3+
  (the desktop shell); 1.10 ships the FFI + a minimal CLI.
- **Hardware-bound export keys** (seal the archive to a YubiKey / passkey
  rather than a passphrase) — MVP-3/4 hardware subsystem, if ever.
