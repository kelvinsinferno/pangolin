# Issue P7: pangolin-chain — Rust chain adapter

> **Status:** PLAN-gate. Per Kelvin's autonomy directive, dispatching the build directly.
> **Security-critical:** YES per §16.3 (chain code). Sequenced after P6 to reuse the alloy dep set.

## Spec reference

- **Master plan §3.7 EPIC: Chain Adapter (P7):**
  - P7-1 `pangolin-chain::ChainAdapter` trait — `publish` / `pull_since(block)` / `get_revision(id)`
  - P7-2 `BaseSepoliaAdapter` impl using `alloy`
  - P7-3 Local index in SQLite mirrors on-chain revisions; `last_pulled_block` checkpoint
  - P7-4 Wallet management — PoC uses a local file with funded keypair; production model deferred to MVP-2
  - P7-5 Signed revision builder in core; contract verifies (P5-2 wired up — deferred to MVP-2 RevisionLog v1; v0 doesn't verify on-chain, but the client signs anyway as forward-prep)
- **Cardinal principle 3:** "Blockchain is a log, never an authority. Compromising the chain or any service cannot reveal or modify plaintext."
- **Whitepaper §G2 (Automatic Sync):** "Edit → revision → signed locally → submitted via relay → chain stores encrypted blob → indexer observes → other devices pull and apply." P7 is the LOCAL side of this loop (sign + submit + pull); the actual sync orchestration is P8.
- **DECISIONS.md D-005 + D-006:** Base Sepolia testnet; device key = wallet (one keypair signs revisions AND pays gas).

## What P5-1 + P5-4 + P6 already shipped

- **P5-1:** `RevisionLogV0.sol` with `publishRevision(vaultId, accountId, parentRevision, deviceId, schemaVersion, encPayload) returns (sequence)` and `RevisionPublished(...)` event. **v0 does NOT verify signatures** — that's planned for MVP-2 issue 2.1.
- **P5-4:** Deployed at `0x8566D3de653ee55775783bD7918Fe91b66373896` on Base Sepolia.
- **P6:** `chaincli` CLI proves alloy + Foundry-keystore + Base Sepolia round-trip works end-to-end at the binary-tool layer.

P7 takes the same chain integration and wraps it in a **library-quality `ChainAdapter` trait** that the rest of the Rust core can consume.

## Goal

Ship `pangolin-chain` as a Rust library with:
1. **`ChainAdapter` trait** — async, sync-friendly surface for publish/pull/get operations against the deployed RevisionLogV0.
2. **`BaseSepoliaAdapter` impl** — alloy-backed, talks to the real deployed contract.
3. **`MockChainAdapter` impl** — for `pangolin-store` and `pangolin-cli` tests; in-memory chain state.
4. **`pangolin-store` integration** — the `chain_anchor` field already on `revisions` rows (P2) gets populated when a revision is published; `unpublished_revisions()` (P2) returns the input set; `mark_published(revision_id, ChainAnchor)` (P2) commits the publish.
5. **`SignedRevision` builder** — the client signs every revision with its device key (P1's `DeviceKey`) over a domain-separated hash. v0 contract ignores the signature; v1 (MVP-2) will verify it. Building the discipline into the client now means MVP-2 doesn't need a client-side migration.

After this issue:
- A high-level test can `unlock vault → add account → adapter.publish(unpublished_revisions[0]) → adapter.pull_since(0) → assert the revision is present on chain → assert mark_published was called → assert the revision_id appears in chain_anchor in the SQL row.`
- The chain-side roundtrip is fully covered by Rust tests against `MockChainAdapter`. A `--features integration-tests` gate runs the same scenario against `BaseSepoliaAdapter` for manual smoke.

## Approach

### Core trait

```rust
#[async_trait]
pub trait ChainAdapter: Send + Sync {
    /// Publish a signed revision. Returns the on-chain anchor (tx hash + block + log index + sequence).
    async fn publish(&self, signed: &SignedRevision) -> Result<ChainAnchor, ChainError>;

    /// Stream events for `vault_id` since (and excluding) `from_block`.
    /// Yields `RevisionEvent`s in chain order. Bounded by `until_block` (None = current head).
    async fn pull_since(
        &self,
        vault_id: &VaultId,
        from_block: u64,
        until_block: Option<u64>,
    ) -> Result<Vec<RevisionEvent>, ChainError>;

    /// Look up a single event by tx hash + log index. Used for replays and disputes.
    async fn get_revision(&self, location: &EventLocation) -> Result<Option<RevisionEvent>, ChainError>;

    /// Current chain head block. Used for sync-checkpoint advancement.
    async fn current_block(&self) -> Result<u64, ChainError>;
}
```

### `BaseSepoliaAdapter`

```rust
pub struct BaseSepoliaAdapter {
    provider: Arc<dyn Provider>,  // alloy provider (HTTP-backed)
    contract: Arc<RevisionLogV0::RevisionLogV0Instance>,
    signer: Option<LocalSigner<...>>,  // None if read-only; Some for publish
}

impl BaseSepoliaAdapter {
    pub fn new_read_only(rpc_url: &str) -> Result<Self, ChainError>;
    pub fn new_with_keystore(rpc_url: &str, keystore_path: &Path, password: &SecretBytes) -> Result<Self, ChainError>;
    pub fn new_with_device_key(rpc_url: &str, device: &DeviceKey) -> Result<Self, ChainError>;
}
```

The third constructor is the key one for `pangolin-store` integration: a `DeviceKey` (from P1) becomes the gas-paying signer. **D-006**: the same Ed25519 keypair signs the revision payload AND pays gas. But — wait — Ethereum doesn't natively support Ed25519 signatures for transactions; it's secp256k1. The "device key" for chain submission is structurally a **separate secp256k1 wallet**, even though Pangolin's revision *signature* (the one v1 contract will eventually verify) is Ed25519.

This is a real wrinkle. Options:
- **Option A (clean):** D-006's "same key" intent is satisfied by deriving the secp256k1 wallet deterministically from the Ed25519 device-key seed (e.g., `SHA-256(device_seed || "pangolin-evm-wallet-v0")` truncated/scaled to a secp256k1 private scalar). One Pangolin device key produces one EVM address. v1 contract can verify Ed25519 over the original device key, while gas comes from the derived EVM key.
- **Option B (compromise):** Use a separate EVM wallet for gas (Foundry keystore as in P6), sign revisions with the Pangolin device's Ed25519 key. Two keys per device. Doesn't match D-006's wording but is operationally simpler.

**Decision:** Option A — derivation. Document the derivation in `pangolin-chain::evm` module, add a test that the same device-key seed always produces the same EVM address, and update `THREAT_MODEL.md` Phase-2 privacy notes to acknowledge the EVM address is observable on-chain (matching the existing P2 row).

### Integration with `pangolin-store`

Adds a method to `Vault`:

```rust
impl Vault {
    /// Publish all unpublished revisions through `adapter`. After each
    /// successful publish, calls `mark_published` to anchor the row.
    /// Errors stop the loop; partial progress persists.
    pub async fn sync_publish<A: ChainAdapter>(&mut self, adapter: &A) -> Result<usize>;

    /// Pull events from the chain via `adapter` since the local
    /// `last_pulled_block` checkpoint, apply non-conflicting heads, and
    /// advance the checkpoint. Conflicts surface via the existing
    /// `is_forked` machinery (P3) — they are NOT auto-resolved.
    pub async fn sync_pull<A: ChainAdapter>(&mut self, adapter: &A) -> Result<SyncStats>;
}
```

Wait — P7 should NOT introduce auto-sync to `Vault`. The orchestration (when to publish, when to pull, conflict surfacing) is **P8**'s job. P7 stays at the **adapter-and-types layer**: it builds `SignedRevision` from raw fields, talks to the chain, and exposes `mark_published` plumbing. The `sync_publish` / `sync_pull` methods on Vault are **P8**.

For P7, the `pangolin-store` integration is just:
- A new column or table for `last_pulled_block` checkpoint (or extend existing meta).
- `pangolin-store` re-exports `ChainAnchor` from `pangolin-chain` (already-existing P2 type — verify the shape matches).

Actually reviewing P2's `ChainAnchor`: it's at `crates/pangolin-store/src/revision.rs` — fields `tx_hash: Option<Vec<u8>>`, `block_number: Option<i64>`, `log_index: Option<i64>`. Shape matches. P7 ships its own `ChainAnchor` in `pangolin-chain::types` with fixed-size byte arrays + `u64`s; the conversion happens at the `mark_published` callsite (P2's takes `Option<Vec<u8>>` from SQL; P7's takes `[u8; 32]` for type safety).

To avoid duplication: `pangolin-store` should **import `pangolin-chain::ChainAnchor`** as the canonical type, and convert at the SQL boundary. That's a small refactor — touches `revision.rs` and `vault.rs::mark_published`. Worth doing to keep one source of truth.

### `last_pulled_block` checkpoint

Add a new row to the `meta` table (or a new single-row `sync_state` table — cleaner): `last_pulled_block: u64` defaulting to `deployment.deploy_block - 1` so the first sync includes the deploy block.

Schema migration: P2's `format_version` is currently 0; P7 either bumps to 1 with a small migration (add the column / table) or stays at 0 with the column added in a "this is a no-op for P2 vaults" sense. For PoC, **schema version stays at 0** but the new table is created via `CREATE TABLE IF NOT EXISTS`; existing P2 vaults pick it up on next `Vault::open` with default value. Document the choice in `pangolin-store/src/schema.rs`.

### Failure handling

- **RPC down:** propagate `ChainError::Rpc(...)`. Not a security failure; surface clearly.
- **Wrong contract address:** `BaseSepoliaAdapter::new_*` reads the canonical `contracts/deployments/base-sepolia.json` and verifies `chain_id`; any mismatch returns `ChainError::WrongChain`.
- **Tx revert (e.g., out of gas):** propagate verbatim; caller handles.
- **Reorg:** out of scope for P7 (P8's concern). For now, every successful `publish` is treated as anchored even if the block could theoretically reorg out. Document this in `THREAT_MODEL.md` follow-up.

7. **`SignedRevision` includes Ed25519 signature** over a domain-separated hash; v0 contract ignores but builder produces it. Test verifies signature verifies under the device's public key.
8. **`last_pulled_block` checkpoint** persists across `Vault::close` + `Vault::open` cycle.
9. **No `unsafe`** in `pangolin-chain`.
10. **No new vault-file format-version bump** (schema migration is additive `CREATE TABLE IF NOT EXISTS`).
11. **`MockChainAdapter` gated behind `test-utilities` feature** so production downstream consumers cannot accidentally substitute it.
12. **Cardinal principle 3 preserved:** the chain adapter is a transport; it never becomes an authority over local state. `pull_since` returns events; the *application* of those events to local state is P8.

## Failure modes considered

| Failure mode | Mitigation |
|---|---|
| Network failure mid-publish | Tx may or may not have mined; caller treats `Err(ChainError::Rpc)` as "publish status unknown — retry pull, check anchor before re-publishing." Document this in adapter docs. |
| Reorg drops a published revision | P7 doesn't handle; P8 does. For now, document as known limitation. |
| Adversary-controlled RPC returns garbage | Adapter parses event topics + data via alloy's typed ABI; structurally-bad responses produce `ChainError::Decode`. No silent acceptance. |
| Adversary writes a revision with valid `vaultId` but wrong contents | v0 contract has no signature check — this IS possible and is documented in P5-1 audit threat #2. Local `pangolin-store` rejects on AAD authentication. |
| Wallet password compromise | Same as P5-4: keystore is encrypted-at-rest; loss of password = loss of access. PoC accepts; MVP-2 may add HSM. |
| Tests leak the keystore password | Tests use a known-fixed-password keystore in `tests/fixtures/` for integration tests; never the user's real keystore. |

## Security-critical?

**YES** per §16.3 (chain code). Audit at SIGNOFF will scrutinize:
- Ed25519 → secp256k1 derivation (no side-channels; deterministic; cryptographically sound)
- `SignedRevision` canonical hash (byte-stable across runs and devices; domain-separated; resists length-extension attacks)
- Wallet handling (no plaintext PK; same Foundry-keystore + DeviceKey-derivation patterns as P5-4 / P6)
- `MockChainAdapter` cannot be substituted for `BaseSepoliaAdapter` in production (cfg-gated)
- `ChainError` taxonomy doesn't leak secret material in `Display`/`Debug`
- The single `ChainAnchor` source-of-truth doesn't accidentally re-introduce P2's loose-types path
