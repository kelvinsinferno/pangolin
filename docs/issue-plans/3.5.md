# Issue 3.5: Top-up flow + low-balance UX (entitlement-state language)

> One-line scope: ship the **client-side balance-state machine** that observes the device wallet's on-chain ETH balance, estimates the next-revision cost, surfaces the Apple-HIG-shaped state (`Sufficient` / `RequiresActiveAccount` / `TopUpInFlight`) to the FFI host, and — per Q-e resolution — either provides a manual `initiate_top_up` API the host calls or auto-invokes the 3.4 funder when the state trips. The state-label vocabulary follows §8.1.5 ("entitlement-state model"): NEVER "out of gas", "low balance", or any pricing copy; the user-facing surface is "requires active account" (or whatever the host renders from the typed state). 3.5 is the FIRST issue that wires 3.4's `pangolin-funder-client` into a CLI / host flow; it ALSO ships the canonical next-revision-cost estimator used for the pre-publish balance gate. Honours 3.1 / 3.2 / 3.3 / 3.4 verbatim.

## Open questions for Kelvin (LOAD-BEARING — surface at TOP per §16)

### Q-a · Where does the balance-state tracker live?

**Option A: `pangolin-chain` (closest to balance reads).** The chain crate already owns `MAX_FEE_PER_GAS_CAP_WEI`, the alloy provider construction, and the per-tx gas-estimate logic inside `publish_revision_v1`. A new `balance_state.rs` module would expose `query_balance(rpc_url, address) -> Result<U256>` + `estimate_next_publish_cost(...) -> Result<U256>` + `compute_balance_state(balance, estimate) -> BalanceState`.

**Option B: `pangolin-store::Vault` (user-facing state owner).** `Vault::gas_balance_state(rpc_url, env).await` is a new async method (breaks the "Vault is sync" doctrine 1.5 + 3.2 + 3.3 R-d).

**Option C: New crate `pangolin-balance`.** Single-purpose; depends on `pangolin-chain` + `pangolin-funder-client`. Cross-crate test setup tax.

**Option D: Hybrid — free fn in `pangolin-chain` + sync accessor on `Vault` for the address.** `Vault` grows a SYNC `evm_wallet_address() -> Result<Address>` accessor (reads cached `devices.evm_address`; locked-vault-safe); balance/estimate logic stays in `pangolin-chain` as free async fns; the caller (CLI / FFI host) orchestrates. **Plan-gate recommends:** Option D — keeps `Vault` sync; locality of balance logic stays in `pangolin-chain`; the cross-crate flow is identical to how 3.3's `publish_revision_v1` is invoked today.

**Kelvin's call.**

### Q-b · When is the balance checked?

**Option A: Synchronous on every `publish_revision_v1` call.** Adds ~one RPC round-trip per publish; freshest reading.

**Option B: Background poll (every N seconds while session active).** Lower latency on publish; freshness gap.

**Option C: Lazy — only on publish failure.** Zero proactive RPC traffic but worst UX.

**Option D: Both eager + lazy.** Best UX coverage; more code; background-poll lifecycle on FFI host requires callback/stream.

**Plan-gate recommends:** Option A for MVP-2 — synchronous-on-publish + a separate explicit `Vault::gas_balance_state_now()` (or chain-crate free fn per Q-a) the host calls when it wants to render state outside the publish path. Defer background polling to MVP-3.

**Kelvin's call.**

### Q-c · What's the "next revision cost" computation?

**Option A: Static heuristic — `1.5 × (last_observed_gas × current_max_fee_per_gas)`.** Bootstrap problem: no `last_observed_gas` until first publish.

**Option B: Dynamic — `eth_estimateGas` against a payload.** Most accurate; requires synthesizing an estimate-only payload (fragile).

**Option C: Conservative fixed — `MAX_REVISION_GAS_CEILING × MAX_FEE_PER_GAS_CAP_WEI` = ~0.03 ETH.** Simplest; pessimistic — users with 0.025 ETH get `RequiresActiveAccount` even though they could publish ~10 revisions.

**Option D: Hybrid — dynamic when possible, conservative fallback.** Use `eth_feeHistory` to read current `baseFeePerGas`, multiply by `EXPECTED_REVISION_GAS = 500_000`, apply 1.5x safety; fall back to Option C's ceiling on RPC failure. **Plan-gate recommends:** Option D.

**Sub-question:** safety-margin buffer = `MIN_BUFFER_REVISIONS = 3` (trip to `RequiresActiveAccount` when balance < `3 × estimate`).

**Kelvin's call.**

### Q-d · How does the state surface across FFI?

**Option A: New typed enum returned from a NEW FFI method** — `pub fn gas_balance_state(handle: Arc<VaultHandle>) -> Result<GasBalanceState, FfiError>`.

**Option B: Field on existing `DeviceInfo`.** Every `device_current()` call now does network I/O — breaks locked-vault contract.

**Option C: Event stream / observer.** Complex FFI plumbing; not how existing surface is shaped.

**Option D: Hybrid — state accessor + companion `next_revision_estimate_wei` field on `DeviceInfo`.**

**Plan-gate recommends:** Option A alone — keep `DeviceInfo` shape stable; the state accessor is the load-bearing surface.

**Proposed enum:**
```rust
pub enum GasBalanceState {
    Sufficient { balance_wei: u128, estimate_wei: u128 },
    RequiresActiveAccount { balance_wei: u128, estimate_wei: u128 },
    TopUpInFlight { initiated_at: UnixTimestamp },
    Unknown { reason: String }, // RPC failure, locked vault, etc.
}
```
Variant names follow §8.1.5: NO "InsufficientFunds", NO "LowBalance", NO "OutOfGas".

**Kelvin's call.**

### Q-e · Top-up trigger model.

**Option A: Manual via `Vault::initiate_top_up()` → returns `TopUpAttemptHandle`.** Host plumbs a Credit attestation into the call. Matches Apple HIG.

**Option B: Automatic — vault-stored Credit attestations; auto-call funder on state trip.** Widens 3.5 scope materially (schema migration, AEAD-sealed attestation storage, vault-layer replay-defense, MVP-3-territory complexity). Auto-debiting without explicit consent is UX-fraught.

**Option C: Two-step — 3.5 ships the state surface + manual `initiate_top_up` API; auto-top-up deferred.** The state machine, balance-check, cost estimate, `RequiresActiveAccount` surface all ship. The actual top-up flow ships as a manual API the CLI / host invokes when it has a Credit attestation in hand (passed as a parameter; NO vault-stored attestation). **Plan-gate strongly recommends Option C** — cleanest MVP-2 scope; matches master plan §5 row 3.5 language ("User pays out-of-band; funder tops up; sync resumes") where "user pays" is explicit, not auto-debited.

**Sub-question:** does 3.5 ship a CLI subcommand (`pangolin top-up <credit-attestation-file>`)? Plan-gate proposes deferring to CLI-V1 batch (per 3.1 / 3.2 / 3.3 / 3.4 precedent) — ship the Rust API only.

**Kelvin's call.**

## Adversarial framing — L-section (load-bearing risks)

### L-balance-staleness

Balance read returns stale data → user sees `Sufficient` and tries to publish → publish succeeds OR fails with insufficient funds depending on fee spike. **Failure mode:** UX disruption only.

**Defense:** per Q-b Option A — synchronous-on-publish freshness check INSIDE the publish path. Cached `RequiresActiveAccount` from background poll (if Q-b Option B / D) is advisory only.

### L-rpc-spoof-balance

Malicious RPC returns fake high balance → client renders `Sufficient` → publish fails on chain.

**Defense:** balance reading is ADVISORY for the state surface. The AUTHORITATIVE failure path is `eth_sendRawTransaction` response (3.3's surface). 3.5 sanity-checks `eth_chainId` before accepting balance.

### L-state-leak-via-label

The `GasBalanceState` enum exposes precise balance values. An attacker watching the host UI / log files learns wallet balance precisely. Combined with address-observability (D-006 mitigation), forms a per-user financial profile.

**Defense:** variant names are boolean-ish (`Sufficient` / `RequiresActiveAccount`). Wei value is in the struct for hosts that EXPLICITLY want to render it; default UX shows label-only. `Debug` impl redacts wei to `"<wei>"` in non-debug builds (similar to `SecretBytes`). Pin: `debug_format_redacts_balance_in_release`.

### L-auto-top-up-DoS (Q-e Option B only)

Attacker triggers repeated `initiate_top_up` calls draining user's paid balance + funder operational capacity.

**Defense:** (a) 3.4 funder R-e rate-limit; (b) per-vault `MIN_TOP_UP_INTERVAL_SECS = 600` rate-limit at client; (c) auto-top-up requires explicit user opt-in (defaults OFF). Moot if Q-e Option C.

### L-credit-attestation-storage (Q-e Option B only)

Vault-stored Credit attestations widen on-disk surface. Mitigated by AEAD-sealing in `credit_attestations` table + contract `nonce[userId]` replay-defense + 3.4 R-g device-binding signature. Plan-gate ENUMERATES this even if storage is deferred — Kelvin MUST sign-off if Option B is chosen.
