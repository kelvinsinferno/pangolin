- **Master plan §0 cardinal principle 4:** "Append-only state. Every credential change is a new immutable revision referencing a parent."
- **Whitepaper §D1 (Revision Log Contracts):**

- **Master plan §3.7 EPIC: Contract (P5):** "P5-1 — `RevisionLogV0.sol` — `publishRevision(bytes32 vaultId, bytes32 accountId, bytes32 parentRevision, bytes32 deviceId, uint8 schemaVersion, bytes encPayload)`. Emits `RevisionPublished` event. **No admin, no owner, no upgrade hooks, no pause.**"

- **DECISIONS.md D-005:** "Contracts must use no Base-specific opcodes and no L2-specific storage tricks. The vault data model must permit future dual-chain readability for migration." (Privacy-chain optionality.)

## Goal

Ship the minimum viable on-chain log: an EVM contract that lets any caller publish an encrypted revision and emits an event consumers can index. **No verification of payload contents.** **No verification of caller identity.** That all comes in v1 (MVP-2 issue 2.1). v0 exists to prove the round-trip — the PoC's chaincli (issue P6) and the Rust chain adapter (issue P7) can both read and write through it.

After this issue:
- The contract is deployed to Base Sepolia (issue P5-4 — separate sub-issue but in the same EPIC).
- The contract's ABI lives in the repo at `contracts/abi/RevisionLogV0.json` for downstream Rust + JS clients.
- Foundry's invariant testing has demonstrated the append-only property under fuzzing.

## Approach

**Hard rule before anything else:** This is a v0 contract. We will deploy v2 if v0 is broken. **No admin keys. No upgrade hooks. No pause. No owner.** This rule is the test that catches every "small convenience addition" that would compromise the design.

### Design

```solidity
// SPDX-License-Identifier: Apache-2.0
pragma solidity ^0.8.24;

contract RevisionLogV0 {
    /// @notice Append-only log of encrypted revisions.
    /// @dev v0: zero verification. v1 (MVP-2 issue 2.1) adds signature
    ///      verification + signer-is-registered-device check. Deploy v2
    ///      to migrate; never upgrade in place.

    event RevisionPublished(
        bytes32 indexed vaultId,
        bytes32 indexed accountId,
        bytes32 indexed parentRevision,
        bytes32 deviceId,
        uint8 schemaVersion,
        uint256 sequence,
        bytes encPayload
    );

    /// @notice Monotonically increasing global sequence number.
    /// @dev Provides total ordering across vaults; v0 readers can
    ///      sort by `sequence` to recover publish order even if block
    ///      timestamps are equal.
    uint256 public nextSequence;

    /// @notice Publish a new encrypted revision.
    /// @dev v0 stores nothing in contract storage beyond `nextSequence`.
    ///      All revision data is emitted as event data, indexable via
    ///      eth_getLogs. This minimizes storage cost and matches the
    ///      whitepaper's "blockchain is a log" framing.
    function publishRevision(
        bytes32 vaultId,
        bytes32 accountId,
        bytes32 parentRevision,
        bytes32 deviceId,
        uint8 schemaVersion,
        bytes calldata encPayload
    ) external returns (uint256 sequence) {
        sequence = nextSequence;
        unchecked { nextSequence = sequence + 1; }
        emit RevisionPublished(
            vaultId,
            accountId,
            parentRevision,
            deviceId,
            schemaVersion,
            sequence,
            encPayload
        );
    }
}
```

### Why this minimal design

- **No `mapping` keyed by vaultId.** Storage writes cost ~20k gas per slot. Events cost ~375 gas + 8 gas/byte. The whitepaper says the chain is a log — events *are* the log. Indexers (and the v0 chaincli + ephemeral indexer in P7/MVP-2 issue 4.2) read via `eth_getLogs` filtered by `topic1 = vaultId`.
- **`nextSequence` is the only storage.** It's the global ordering primitive. Removing it would force readers to rely on `(blockNumber, logIndex)`, which is fine but adds parsing complexity for clients. One slot is cheap.
- **Three indexed topics: `vaultId`, `accountId`, `parentRevision`.** This is the maximum (Solidity caps at 3 indexed event params + the implicit signature topic). Picking these three lets clients filter for "all revisions in this vault" (vaultId), "all revisions for this account" (vaultId + accountId), or "the child of this specific parent" (parentRevision) using just `eth_getLogs`.
- **`deviceId` and `schemaVersion` are not indexed** — they're unindexed event data. Filtering by device or schema is rare and can happen client-side after decoding.
- **`encPayload` is `bytes calldata`** — no upper bound. The base layer doesn't impose a payload size limit beyond block gas. Real-world payloads will be small (a few hundred bytes for a credential JSON+CBOR serialization). Clients that try to publish a 1MB payload will discover that gas alone enforces a soft cap.
- **No constructor.** No initialization parameters. No way to "configure" anything. Maximum simplicity.
- **No `receive` / `fallback`.** Contract cannot accept ETH. (If somehow ETH is sent via `selfdestruct` of another contract, it's stuck. Acceptable.)
- **Solidity 0.8.24+** — for portable EVM compatibility (Cancun-era opcodes available but optional). Compiled with `evm_version = "shanghai"` to ensure broad chain compatibility (Base, Arbitrum, OP, Polygon, etc.). **Important per D-005:** no Base-specific opcodes.

## Public surface

- One external function: `publishRevision(bytes32, bytes32, bytes32, bytes32, uint8, bytes calldata) returns (uint256)`.
- One public storage variable: `nextSequence` (auto-getter `nextSequence()`).
- One event: `RevisionPublished(bytes32 indexed, bytes32 indexed, bytes32 indexed, bytes32, uint8, uint256, bytes)`.

That's the entire surface. **There is no admin. There is no owner. There is no pause. There is no upgrade.**

ABI committed at `contracts/abi/RevisionLogV0.json` (re-generated as part of `forge build` and verified by CI to match the source).

## Success criteria

1. `forge build` succeeds with Solidity 0.8.24, evm_version = "shanghai".
2. `forge test` passes all unit tests (≥10 tests covering happy path + adversarial).
3. `forge test --invariant` passes Foundry invariant tests with ≥10,000 fuzzed runs.
4. Static analysis (`slither` or `forge inspect`) reports zero high-severity findings.
5. Contract bytecode size < 1 KB (sanity check; the contract is trivial).
6. **No admin / owner / pause / upgrade-related bytecode**: assertion via `slither` detector list + a manual review checklist that verifies the contract has no `onlyOwner` modifiers, no `Ownable`/`AccessControl` imports, no proxy patterns, no `selfdestruct`, no `delegatecall`.
7. Gas cost for `publishRevision` with a 256-byte payload is < 50,000 gas (sanity check).
8. ABI file at `contracts/abi/RevisionLogV0.json` matches `forge build` output (CI step verifies).
9. Solidity warning count is zero (`forge build` with `--deny-warnings` flag or equivalent).
10. Cross-chain compatibility check: `forge inspect` confirms no Base-specific opcodes are emitted (per D-005). Verified by setting `evm_version = "shanghai"` and ensuring the build succeeds without falling back to Base-only opcodes.

### Invariant tests (`RevisionLogV0.invariant.t.sol`)

Foundry's invariant runner calls random functions with random args and asserts properties hold across all sequences.

| Invariant | Property |
|---|---|
| `invariant_sequenceMonotonic` | `nextSequence` only ever increases |
| `invariant_eventCountEqualsSequence` | total emitted `RevisionPublished` events == `nextSequence` |
| `invariant_noStorageMutationBesidesSequence` | every storage slot besides slot 0 (`nextSequence`) is zero (probed via `vm.load` in the test) |

Configured for ≥10,000 runs in CI (`forge test --invariant --runs 10000`).

## Failure modes considered

| Failure mode | Mitigation |
|---|---|
| Someone calls `publishRevision` with a malformed `encPayload` (not really decryptable) | v0 has zero content validation by design. The encrypted-payload format is a *client* concern. Garbage inputs cost the publisher gas; no contract-level harm. |
| `nextSequence` overflow | `unchecked` block + `uint256` means overflow at 2^256 calls. Not a real failure mode. |
| Reentrancy via `bytes encPayload` callback | The contract makes no external calls and reads/writes only one storage slot. No reentrancy surface. |
| MEV-style sequence stealing (someone publishes ahead of the user) | Acceptable in v0 — sequence is global, not per-vault. v1 may add per-vault sequencing or a head pointer; v0 doesn't need it. Clients use `(vaultId, parentRevision)` to determine head per account, not the global `sequence`. |
| Contract receives ETH and ETH is stuck | No `payable` functions, so direct sends revert. ETH sent via `selfdestruct` of another contract or via coinbase-block-reward redirection cannot be recovered, but neither is a meaningful loss for a stateless log. Acceptable. |
| Solidity compiler regression | Solc version pinned in `foundry.toml` (`solc = "0.8.24"`); CI fails on version drift. |
| ABI drift between checked-in JSON and source | CI step compares `out/RevisionLogV0.sol/RevisionLogV0.json` against `abi/RevisionLogV0.json`; PR fails if they diverge. |
| Network unavailability during deploy | Out of scope for P5-1 (contract authoring). P5-4 (deploy) has its own retry/verify procedure. |

## Threat enumeration (for THREAT_MODEL.md update)

After P5-1 lands, append the following to THREAT_MODEL.md under "Revision Log v1 contract" (and note the v0 differences):

1. **Adversary publishes garbage to a vault's log to slow sync.** Defense: clients filter by `vaultId` topic; per-vault `eth_getLogs` query is unaffected by other vaults' garbage. Gas cost falls on the adversary. (v1 will require a valid signature; v0 explicitly does not.)
2. **Adversary publishes a fake "next revision" forking a user's account.** Defense: client-side conflict detection (master plan MVP-1 issue 1.6) detects multiple heads. User resolves explicitly per Whitepaper §7.
3. **Adversary tampers with on-chain ciphertext.** Defense: AEAD AAD binds revision metadata (P1-1 design). Tampered ciphertext fails authentication on the client. The contract is *not* the integrity layer; AEAD is.
4. **Chain reorg or network partition.** Defense: clients tolerate reorgs by tracking `(blockNumber, logIndex)` per known revision; on reorg, re-pull from the last known stable block. Out of scope for the contract; in scope for `pangolin-chain` (issue P7).
5. **Permanent contract corruption (storage attack).** Defense: contract has only one storage slot (`nextSequence`); no functions write to other slots. Invariant test asserts this.

## Rollback / abort plan

- If Foundry tests reveal a design flaw (e.g., a way to consume disproportionate gas via a payload trick): document in DEVLOG, propose a fix in this plan, get re-approval. Do not silently patch.
- If `slither` reports a high-severity finding: stop and review with Kelvin.
- If we discover during P5-4 (deploy to Base Sepolia) that the contract behaves differently on L2 vs local anvil: stop, file diagnostic issue, escalate. Most likely cause would be base-specific gas-pricing surprises, which P5-1 doesn't yet exercise.
- If the simple "no storage; events only" approach turns out to break the chaincli or chain adapter (P6 / P7) later: those are separate issues; revisit when they fail. Do not preemptively over-engineer v0.

## Out of scope

- **Signature verification.** v1 (MVP-2 issue 2.1) verifies that `encPayload`'s associated signature is from a registered device key for the vault. v0 has no such check.
- **Device-key registry / authority registry.** That's MVP-2 issue 2.1 + MVP-2 issue 2.2 (Entitlement Registry).
- **Recovery contract.** That's MVP-3 issue 2.2.
- **Gas optimization beyond the obvious.** v0 is intentionally simple; gas is acceptable as long as `publishRevision` < 50k gas for typical payloads.
- **Multi-chain deployment scripts.** Base Sepolia testnet only for v0 (P5-4). Multi-chain comes later if/when we honor the privacy-chain optionality from D-005.
- **Front-end** for issuing transactions. The chaincli (P6) is the test client; the Rust chain adapter (P7) is the production client.
- **EIP-712 domain separation.** Not needed for v0 (no signature verification). v1 will define a domain separator.
