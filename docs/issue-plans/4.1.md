## Open questions for Kelvin (LOAD-BEARING — surface at TOP per §16)

### Q-a · Checkpoint persistence — where does `last_synced_block` live?

`Vault::ingest_chain_revision` is idempotent (hash-match + chain-anchor-match short-
circuits), so a "replay from genesis on every session" sync is *correct* — just slow
on a vault with many published revisions (and even slower on a chain whose tip is
millions of blocks past the deploy block). The question is whether 4.1 caches the
high-water mark to skip the re-fetch.

**Option A: Persist in `.pvf` per-vault.** New `chain_sync_state` table (or new column
on `vaults` / `meta`) keyed by `(vault_id, chain_env)` carrying `last_synced_block`.
§18.7 schema-version bump. Survives restart; persists across machines if the vault is
restored from backup. Trade-off: a tampered `.pvf` could report an unreasonably high
`last_synced_block` (L-checkpoint-corruption) — mitigated by L4 below (monotonic +
sanity-checked against `eth_blockNumber`).

**Option B: In-memory only.** No `.pvf` change; replay from D-017's deploy block on
every session. Trade-off: slow for large vaults (slow-mode by design), but the
"first sync of a fresh client install" cost is bounded by Base Sepolia's actual
revision-event count under D-017 (small as of plan-gate time).

**Option C: Hybrid.** Persist in `.pvf` per Option A, plus a CLI flag / FFI method for
forced re-sync from genesis (`pangolin sync --from-genesis`). Defends against
L-checkpoint-corruption by giving the user an out.

**Plan-gate recommends:** Option C (persist + force-resync). The §4 cluster's
"slow mode" framing explicitly says "slower for large initial syncs" — meaning the
first sync is slow and subsequent syncs are fast (= checkpoint exists). Option B
makes every session slow, undermining the framing. Option C adds the force-resync
escape hatch (cheap; useful for L-checkpoint-corruption + L-rpc-omits-events).

**Kelvin's call.**

### Q-b · Event fetch strategy — HTTP polling vs WebSocket subscription

**Option A: HTTP polling via `eth_getLogs`.** Matches the existing v0
`BaseSepoliaAdapter::pull_since` pattern (`crates/pangolin-chain/src/base_sepolia.rs:457`,
chunked at `LOG_BLOCK_CHUNK = 9_000`). Works with any RPC. The caller invokes
`sync_vault_from_chain` periodically (or on user action); no background task in 4.1
itself. Cheapest implementation; matches 3.3's alloy `ProviderBuilder::new()` pattern.

**Option B: WebSocket subscription to `eth_subscribe("logs")`.** Real-time event push.
Requires a WS-capable RPC (`wss://sepolia.base.org` — Base Sepolia's public RPC does
support it; some private RPCs do not). Adds the `alloy::pubsub::PubSubFrontend` shape +
a long-running task to manage the subscription. Trade-off: better latency; harder to
test hermetically; reconnection-on-disconnect logic is non-trivial.

**Option C: Both — WS preferred, HTTP polling fallback.** Best UX, worst code budget.

**Plan-gate recommends:** Option A for MVP-2. The "slow mode" framing matches periodic-
poll cadence; real-time push is an MVP-3 optimization. Plus: 4.2-4.4's indexer cluster
is the path to real-time event delivery for users who opt in. 4.1 stays
implementation-simple + RPC-portable.

**Kelvin's call.**

### Q-c · Reorg handling — confirmations required before treating events as final

Base Sepolia averages ~2s block times; reorgs are rare but possible (especially for
sequenced L2 chains where the sequencer can re-order blocks pre-finality). Three
plausible postures:

**Option A: Reorg-naive (1 confirmation).** Accept events as soon as they appear in
`eth_getLogs`. Matches 3.3's publish-side posture (`block until 1-conf receipt`).
A subsequent reorg that removes a published event from chain would leave a stale
revision in the local graph. Mitigated by the post-sync canonicalization step in
1.6's `canonical_head` (re-orphaned revisions surface as forks).

**Option B: Reorg-safe with N confirmations.** Wait `N` blocks past the latest before
treating events as final. Sub-question: `N = ?`. Common values: `N = 12` (~24s on Base
Sepolia; matches Ethereum mainnet's "safe" finality heuristic), `N = 30` (~60s; matches
Optimism's "finalized" tag), or `N = 1` for MVP-2 with deferral to MVP-3.

**Option C: Two-stage — accept at 1-conf optimistically, mark "finalized" at N-conf,
rollback if reorg observed.** Most accurate semantics; most code budget. The local
graph would need a "tentative" vs "finalized" status column on `revisions` (or a
companion `chain_finality_status` table). MVP-3 territory.

**Plan-gate recommends:** Option A (1-conf) for MVP-2 — matches 3.3's publish-side
posture; matches v0 `BaseSepoliaAdapter`'s implicit posture; matches the "slow mode is
the baseline; reorg-safe finality is a future hardening" reading of master plan §4.
Worth noting in the doc that this means a reorg on Base Sepolia could leave a
stale-but-idempotent revision in the local graph — recoverable by `pangolin sync
--from-genesis` (Q-a Option C).

**Kelvin's call.**

### Q-d · Device-registration cross-check disposition

D-017's `_verifyAndRegister` already enforces that the recovered signer is either
(a) the first publisher for a fresh `vaultId` (R-b self-bootstrap) or (b) a previously-
registered device. So an event on chain is always already-verified contract-side.
4.1's defense-in-depth question: what does the client do when a *received* event's
signer is not in the local `devices` table?

**Option A: Strict reject.** Treat unknown signers as a chain-side spoof (impossible
under contract semantics — would mean a malicious RPC fabricated an event); reject the
event; surface a `revisions_rejected` count to the host. **Failure mode:** legitimate
multi-device flows where Device B self-bootstraps via 2.1 R-b on a fresh vault — Device
A's client receiving the first sync from D-017 would reject Device B's revisions
because Device B was never locally registered. Breaks multi-device discovery (which is
exactly what the chain READ path exists for).

**Option B: Permissive auto-register.** On every verified event, ensure the recovered
signer is in the local `devices` table; insert a new row if missing. Matches the
contract's on-chain semantics (which registers on first publish). Defense-in-depth
trade-off: a malicious RPC that fakes a `RevisionPublished` event (with a forged sig
that doesn't actually verify) is caught by the per-event verifier BEFORE auto-register
runs; so auto-register only happens for chain-verified events.

**Option C: Hybrid — accept + auto-register, but flag new addresses for user review.**
Like Option B plus a UI surface ("this vault now has a new device — confirm?"). UX
plumbing that mostly belongs in the host; 4.1's job is the data plumbing.

**Plan-gate recommends:** Option B for MVP-2. The contract already enforces registration
on-chain; client-side strict-check breaks multi-device sync (which is the whole point
of the chain read path). Per-event signer recovery is the load-bearing defense; the
device table is downstream of that.

**Kelvin's call.**

### Q-e · API surface — sync, async, both, plus event-shape disposition

**Option A: Async-only on `pangolin-chain`.** Mirrors 3.3 R-d + 3.5 R-a:
`pub async fn sync_vault_from_chain(vault: &mut Vault, rpc_url: &str, env: ChainEnv) ->
Result<SyncReport, ChainSyncError>` lives in `pangolin-chain`. Caller (CLI / FFI host)
hosts the tokio runtime. `Vault` stays sync per the 1.5 / 3.2 / 3.3 doctrine. Trade-off:
the signature takes `&mut Vault`, meaning the `pangolin-chain` crate would gain a
`pangolin-store` dep (BACKWARDS — L7 violation). Resolution: the async fn lives in
`pangolin-store`'s `Vault` impl as `async fn sync_from_chain(...)`, OR it lives in
`pangolin-chain` as `sync_vault_events(rpc_url, env, vault_id, from_block) ->
Result<Vec<RevisionEventV1>, ChainSyncError>` and the caller fans the events into
`Vault::ingest_chain_revision` manually. Plan-gate strongly prefers the second shape —
keeps L7 intact.

**Option B: Sync via `block_on`.** Adds a runtime spin-up per call (3.3 Q-d's known
footgun) + breaks the "Vault is sync" doctrine if it lands on `Vault`. Not recommended.

**Option C: Both.** Doubled surface; not recommended for MVP-2.

**Plan-gate recommends:** Option A's second shape — `pangolin-chain` exposes
`pub async fn pull_signed_revisions_v1(rpc_url, env, vault_id, from_block, until_block)
-> Result<Vec<RevisionEventV1>, ChainError>` (the v1 analogue of
`BaseSepoliaAdapter::pull_since`); the caller (CLI / `apps/cli/src/sync.rs` /
eventual FFI) loops `pull_signed_revisions_v1` → for each event recover-and-verify-
signer → `vault.ingest_chain_revision(...)` → advance checkpoint. The full
`sync_vault_from_chain` orchestration helper lives in **`pangolin-store::Vault`** as
`pub async fn sync_from_chain(&mut self, rpc_url: &str, env: ChainEnv) ->
Result<SyncReport, StoreError>` because it owns both the vault state AND drives the
checkpoint advance — and `pangolin-store` already depends on `pangolin-chain` (the
correct dep direction).

**Sub-question Q-e.1 — event-shape:** Today `pangolin_chain::RevisionEvent` carries
`device_id: [u8; 32]` interpreted as Ed25519 verifying-key bytes (v0 semantics; the v0
adapter even validates it as a canonical Ed25519 key — see
`crates/pangolin-chain/src/base_sepolia.rs:519-526`). The v1 contract emits `deviceId:
bytes32` as a left-padded EVM address PLUS a separate `signer: address` unindexed
field. Two ways to handle:

- **A:** Introduce `RevisionEventV1` as a new type with `device_id: Address` +
  `signer: Address` + `signature: [u8; 65]` fields; v0's `RevisionEvent` stays untouched
  for legacy reads.
- **B:** Widen `RevisionEvent` with optional `v1_signer: Option<Address>` /
  `v1_signature: Option<[u8; 65]>` fields; both v0 + v1 events flow through one type.

Plan-gate strongly recommends **A** — clean-break v0 → v1 at the type level matches
3.1's `SignedRevisionV1`-is-not-`SignedRevision` doctrine; prevents an v0 / v1 mix-up
at the consumer (`Vault::ingest_chain_revision`).

**Sub-question Q-e.2 — verifier input shape:** `recover_signer_v1` could take either
(a) `&SignedRevisionV1` (mirrors 3.1's test-only helper) or (b) the raw
`(fields, signature, env)` tuple. The event-decode path naturally produces (b); 
the publish-roundtrip test in 3.1 naturally produces (a). Plan-gate recommends:
expose BOTH — a primitive `recover_signer_v1_raw(fields: &RevisionFieldsV1, signature:
&[u8; 65], env: ChainEnv) -> Result<Address, ChainError>` AND a convenience
`recover_signer_v1(signed: &SignedRevisionV1, env: ChainEnv) -> Result<Address,
ChainError>` that calls into the primitive. The first is used by the chain-sync
event-decode path; the second is used by the 3.1 round-trip test.

**Kelvin's call.**

## Adversarial framing — L-section (load-bearing risks)

> Per §16, every plan-gate doc for a security-critical issue surfaces the
> load-bearing risks an adversary could leverage from each design choice.

### L-rpc-spoof-events

**What goes wrong:** a malicious RPC returns a `RevisionPublished` event with a forged
signature (a randomly-generated 65-byte blob) for a `vaultId` the user actually owns.
If the client trusted the RPC's filter and skipped the per-event verifier, the forged
event would land in the local graph as a "real" revision the user never published —
poisoning the lineage.

**Failure mode:** the local graph contains attacker-controlled revisions; the user
either (a) sees garbage when they next decrypt + render the vault, or (b) the
canonical-head detector now reports a fork the user didn't create.

**Defense:** L5 — the client-side `recover_signer_v1_raw` verifies the signature for
EVERY event the RPC returned. A forged 65-byte sig cannot recover to a valid Address
under EIP-712 + secp256k1 (the verifier returns `address(0)` or an attacker-controlled
but irrelevant address). Plus L4 — the `eth_getLogs` filter pins the contract address
AND the event signature hash, so the RPC must at minimum produce a syntactically-valid
log; the verifier catches the cryptographic forgery.

**Adversary leverage:** mitigated to "RPC can return zero events" (i.e., the
`L-rpc-omits-events` failure mode, see below).

### L-rpc-omits-events

**What goes wrong:** a malicious RPC silently drops events for a specific `vaultId`,
hiding chain-side revisions the user actually has. The user thinks their lineage is
caught up but in fact has gaps.

**Failure mode:** the local graph is incomplete; subsequent local edits parent at a
stale revision; when the user later syncs with a HONEST RPC, the gap surfaces as a
fork (1.6's `canonical_head` reports multiple heads). Recoverable by re-sync from a
different RPC.

**Defense:** cross-check the contract's on-chain `_nextSequence` global counter
against the highest `sequence` value the client has observed. A gap → either (a) a
true reorg (rare on Base Sepolia) or (b) a hostile RPC. Worth pinning as a sub-check
in the sync loop: after each `eth_getLogs` chunk, query
`contract.nextSequence().call()` (cheap view fn); compare to max observed sequence;
flag the gap.

**Adversary leverage:** with the cross-check, mitigated to "user sees an explicit
'sync incomplete' status". Without the cross-check, the gap is silent until a fork
appears on the next sync.

### L-reorg-rollback

**What goes wrong:** a chain reorg moves a `RevisionPublished` event from one block to
another (Base Sepolia reorgs are possible but rare). The client's checkpoint may
already be advanced past the affected block range, so the next sync misses the
re-positioned event.

**Failure mode:** Q-c-dependent. Under 1-conf naive posture: a stale event sits in
the local graph that no longer exists on chain. Under N-conf safe posture: events are
held tentative until N confirmations; reorgs caught.

**Defense:** Q-c's resolution. Plan-gate recommends 1-conf for MVP-2 with explicit
documentation that reorg-safety is deferred. The `pangolin sync --from-genesis` escape
hatch (Q-a Option C) lets a user force-re-fetch.

**Adversary leverage:** minor; reorgs are a network condition, not an attack.

### L-checkpoint-corruption

**What goes wrong:** a tampered `.pvf` (or in-memory state) reports an unreasonably
high `last_synced_block` so the client never re-fetches old events. Concrete shape:
attacker corrupts the `chain_sync_state.last_synced_block` column to `u64::MAX`; the
next sync starts at `u64::MAX + 1` (overflow → 0 OR cursor > tip → no-op) and the
client reports "caught up" while having processed zero events.

**Failure mode:** the user thinks the client is caught up; in fact has gaps.
Indistinguishable from L-rpc-omits-events from the user's perspective.

**Defense:** L12 verbatim — checkpoint is monotonic (only increases). Plus: before
accepting a new checkpoint value, validate `last_synced_block < current_tip` (where
`current_tip = provider.get_block_number().await`); an out-of-range value → fail with
`ChainError::CheckpointOutOfRange { observed, tip }`. The `.pvf` is AEAD-sealed
per MVP-1, so the corruption surface requires breaking the AEAD seal — which the
threat model already covers under "filesystem-level tamper of a sealed vault is
out-of-scope" (1.5 / 2.1 / 3.1 / 3.3 same boundary). Defense-in-depth: the
sanity-check above + the user-visible `pangolin sync --from-genesis` escape.

**Adversary leverage:** with the sanity-check, mitigated to a per-session re-sync.

### L-large-vault-DoS

**What goes wrong:** a vault with millions of revisions takes forever to sync in slow
mode. This is 4.1's deliberate trade-off; the 4.2-4.4 cluster ships the opt-in
ephemeral indexer for that case.

**Failure mode:** UX failure (a sync that takes hours). Not a security failure.

**Defense:** surface progress via `SyncReport { revisions_pulled, revisions_applied,
revisions_rejected, blocks_scanned, current_cursor }` so the host can show a progress
UI. The sync loop is cancellable (host can drop the future); on cancellation the
checkpoint advances to the last successfully-ingested block (NOT the cursor — only the
block where we actually have all events through completion).

**Adversary leverage:** an attacker who can spam-publish revisions to a victim's
`vaultId` can grow the sync time — but spam requires the attacker to have ETH AND a
device key registered for that vault, which is the same impossibility 2.1 R-b assumes.

### L-rpc-replay-events

**What goes wrong:** a malicious RPC returns the same event twice within a single
`eth_getLogs` chunk (different `tx_hash` / `log_index` but same content hash, OR same
`(tx_hash, log_index)` twice). The client's idempotency check at
`Vault::ingest_chain_revision` (MVP-1 hash-match + chain-anchor-match) handles both.

**Defense:** the existing MVP-1 idempotency machinery. Pin a test that drives the
sync loop with a mock-RPC returning duplicated logs; assert `revisions_applied == 1`
and the second log surfaces as `IngestOutcome::AlreadyPresent`.

**Adversary leverage:** none — idempotency catches it. Worst case is wasted CPU on the
verifier (small).

### L-malicious-vault-id-substitution

**What goes wrong:** a malicious RPC returns events for a DIFFERENT vault than the
client requested (the indexed `topic1` filter notwithstanding — a misbehaving server
could splice in unrelated logs). The client trusts the filter; foreign events land in
the local graph under the requested `vaultId`.

**Defense:** the `eth_getLogs` filter pins `topic1 = vault_id`; the server-side filter
is the first defense. PLUS: at decode time, the client cross-checks
`decoded.vaultId == requested_vault_id` and rejects mismatches (defense-in-depth, same
posture v0's `BaseSepoliaAdapter::pull_since` MED-4 takes for `log.address()`).

**Adversary leverage:** mitigated.

### L-verifier-domain-binding-drift

**What goes wrong:** 4.1's `recover_signer_v1` is built against `BaseSepolia` but
applied to events from a different chain (developer mis-configuration; mock-RPC
returning events from a different deployment). The domain separator mismatches → the
verifier recovers a wrong signer → all events appear forged → the client rejects them
all → no progress.

**Failure mode:** complete sync failure. Recoverable by fixing the env config.

**Defense:** L3 (`eth_chainId` cross-check at construction) + L4 (pinned-address
cross-check before decode). Plus: the verifier itself takes `env: ChainEnv` and
constructs the domain via the same `build_domain` helper 3.1 uses, so a domain-binding
divergence is a `secp256k1_signing.rs` bug catchable at the 3.1 layer.

**Adversary leverage:** none directly — this is a config / self-inflicted concern.

### L-schemaVersion-future-poison

**What goes wrong:** a future contract deploy (v1.1 / v2) emits events with
`schemaVersion > 1`; the 4.1 client doesn't know that schema. Per THREAT_MODEL.md
invariant #11 (§18.7 schema-version ladder), the client must REJECT events with
`schemaVersion > MAX_KNOWN_CLIENT_SCHEMA_VERSION`.

**Defense:** at event-decode time, the verifier checks
`decoded.schemaVersion <= MAX_KNOWN_CLIENT_SCHEMA_VERSION` (= 1 in MVP-2) and rejects
mismatches with `ChainError::UnsupportedSchemaVersion`. Plus: the
`Vault::ingest_chain_revision` path enforces the same check at the storage boundary.

**Adversary leverage:** none — this is the ladder's intended behavior. Documented
in the §18.7 ladder.

## Threat-model touch points

4.1 adds a new entry to `THREAT_MODEL.md`'s per-component table: **"Slow-mode chain sync (read path + v1 verifier)"** — DOCUMENTED (issue 4.1). Per-surface enumeration:

1. **Adversary controls user's RPC.** L-rpc-spoof-events + L-rpc-omits-events + L-rpc-replay-events. Mitigated by per-event signer verifier + sequence-gap cross-check + idempotency at ingest.
2. **Adversary tampers with `.pvf` checkpoint.** L-checkpoint-corruption. Mitigated by monotonic + sanity-checked checkpoint + AEAD-sealed vault file.
3. **Reorg moves a published revision.** L-reorg-rollback. Mitigated by Q-c (1-conf for MVP-2; reorg-safety deferred to MVP-3).
4. **Future schema-version poison.** L-schemaVersion-future-poison. Mitigated by §18.7 ladder enforcement at decode.
5. **Foreign-emitter splicing.** Mitigated by L4 pinned-address cross-check.
6. **Domain-binding drift on the verifier.** L-verifier-domain-binding-drift. Mitigated by L3 + L4 + shared `build_domain` helper with 3.1.

**Invariant additions: none new.** Existing invariants #1, #5, #11 cover 4.1's surface; #11 (§18.7 schema-version ladder) is exercised by the v1 event-decode rejection path.
